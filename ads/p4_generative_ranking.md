# Project 4：生成式排序与下一代范式

> 项目类型：前沿技术探索 | 日期：20260324 | 领域：ads × generative models

## 导言：排序的根本转变

```
【传统排序】= 从固定候选集选择
给定 Top-N 候选，选择 Top-K 并排序
输出：[Ad_1, Ad_2, ..., Ad_K]

【生成式排序】= 直接生成最优排序
不受"候选集"限制，从全库生成
输入：用户上下文
输出：[Ad_best_1, Ad_best_2, ..., Ad_best_K]

【优势】
- 不受候选集限制（可从亿级库生成）
- 冷启动广告自动获得曝光
- 更灵活处理约束
- 端到端优化
```

## Part A：生成式排序（GR）基础

### A1. 核心原理

```
【问题表述】
输入 X = 用户上下文（ID、历史、时间等）
目标 Y = 最优广告序列（[Ad_1, Ad_2, ..., Ad_K]）

【生成方案】
用序列生成模型（Seq2Seq / Transformer）逐步生成：

Step 1：encoder 编码用户上下文 X
  embedding = Encoder(user_context)

Step 2：decoder 逐步生成广告 ID
  token_1 = "Ad_ID_12345"  （最好的广告）
  token_2 = "Ad_ID_67890"  （次好的）
  token_3 = "Ad_ID_11111"  （第三好的）
  ...

Step 3：采样策略
  - Greedy：每步选择概率最高的
  - Beam Search：保留 Top-K 候选，找全局最优
  - Temperature Sampling：控制多样性
```

### A2. 论文案例：WUKONG（Google, 2024）

**Towards a Scaling Law for Large-Scale Generative Ranking**

```
【核心创新】
1. 用大规模生成模型（GPT-scale）
2. 直接生成排序（不经过中间排序层）
3. 在亿级广告库上验证

【结果】
- RPM +2.3%（相比传统排序）
- 新广告曝光 +30%（冷启动改进）
- 推理延迟：~150ms（可接受）

【关键技术】
- 广告 ID 的离散化编码
- 强制解码约束（确保输出有效）
- 与用户兴趣的条件生成
```

## Part B：排列生成与约束

### B1. 排列语言模型（Permutation Language Model）

```
【想法】：排列可以看作 Token 序列

示例：
理想排序 = [Ad_A, Ad_B, Ad_C]
对应 token = ["<ad_A>", "<ad_B>", "<ad_C>"]

目标：
学习 P(排列) = ∏ P(token_i | token_1, ..., token_{i-1})

这就是标准的语言模型！
```

### B2. 约束的编码

```
【方案 1】：Prompt 约束
System Prompt: "You are an ad ranking agent."
User Prompt: "User context: {...}. 
             Constraints: max 2 ads from same advertiser,
             at least 30% diverse topics.
             Rank these ads: [ad1, ad2, ...ad50]"

Output: [ad5, ad20, ad3, ...]

优点：灵活，可动态指定约束
缺点：不够严格，模型可能违反约束

【方案 2】：强制解码
有限状态机（FSM）在生成过程中强制执行约束：
- 当选择了某个广告主，标记为"已选 1 次"
- 生成下一个 token 时，FSM 强制跳过该广告主（如果超过频率上限）

优点：100% 满足约束
缺点：实现复杂

【方案 3】：后处理
生成完整排列后，用约束检查器调整违反约束的部分
"贪心修复"：从违反约束的位置开始，迭代修正
```

### B3. 与评分蒸馏的结合

```
【评分蒸馏】(Score Distillation Sampling, SDS)
思路：用预训练的评分模型指导生成

步骤：
1. 生成器生成排列候选 y
2. 评分模型给出评分 s(y)
3. 反向梯度：∇ 最大化 s(y)，指导生成器学习
4. 迭代，直到收敛

应用于排序：
- 生成模型：学习排列生成
- 评分模型：CTR/CVR 预估、多样性评分、约束检查
- 评分函数：Score = w1×CTR + w2×Diversity + w3×Constraint_satisfaction
```

## Part C：扩散排序（Diffusion for Ranking）

### C1. 核心思想

```
【比喻】：噪声 → 去噪 → 清晰的排序

Step 0：初始化随机排列
y_T = 随机打乱 [ad1, ad2, ..., ad_N]

Step 1-49：逐步去噪
y_{T-1} ~ denoise(y_T, condition)
condition = 用户上下文、评分函数

Step 50：最终排序
y_0 = 清晰的、优化的排列
```

### C2. 数学形式

```
【前向过程】（加噪）
q(y_t | y_0) = 对排列逐步添加噪声
噪声类型：打乱位置、删除广告、插入错误广告

【反向过程】（去噪）
p(y_{t-1} | y_t) = 学习去噪分布
目标：给定 y_t，预测 y_{t-1}

【训练】
学习反向过程，最小化 KL 散度

【推理】
从 y_T（全噪声）开始，迭代去噪到 y_0
```

### C3. 优点 vs 劣势

| 优点 | 劣势 |
|-----|-----|
| 理论基础扎实 | 推理步数多（50-200 步）|
| 灵活处理约束 | 每步都需要模型前向，延迟长 |
| 可迭代改进 | 仍在研究阶段，实战少 |
| 多模态支持（可生成多个最优解） | 需要特殊的模型架构 |

## Part D：多智能体强化学习排序（MARL Ranking）

### D1. 架构

```
【多个 Agent 的协作】
- Agent 1：CTR（收入优化）
- Agent 2：CVR（转化优化）
- Agent 3：多样性（体验优化）
- Agent 4：品牌安全（风控）

【交互过程】
状态 S = 当前展示列表 + 用户反馈
Agent 1：「我想增加高 eCPM 广告」
Agent 2：「我想选可能转化的广告」
Agent 3：「我想确保多样性」
Agent 4：「我要确保品牌安全」

协商 → 最终排列（多目标均衡）
```

### D2. 博弈论基础

```
【纳什均衡】(Nash Equilibrium)
- 每个 agent 的策略都是最优的（给定其他 agent 的策略）
- 没有 agent 想单独改变自己的策略

【MARL 求解】
1. **中心化训练，分散化执行** (CTDE)
   - 训练：所有 agent 共享信息，学习最优策略
   - 执行：每个 agent 只用自己的局部信息

2. **通信机制**
   agent 间可相互通信，协商决策
   
3. **奖励分解**
   全局奖励 = Σ agent 局部奖励
   需要设计合适的奖励分配函数
```

## Part E：工业落地与挑战

### E1. 推理延迟

```
【问题】
生成式排序需要多步生成：
- LLM based GR：150-500ms
- Diffusion Ranking：100-300ms

【要求】
广告排序需要 <100ms（与搜索一样严格）

【解决方案】
1. 模型压缩：蒸馏、量化
2. 批量处理：多个请求一起处理
3. 缓存：热门用户/广告组合预缓存排序结果
4. 异步化：后台预生成，前景返回预存结果
```

### E2. 可解释性与安全性

```
【问题】
用户问「为什么给我看这个广告」？
- 传统排序：分数 = 0.95（有数字）
- GR 排序：模型生成的结果（难以解释）

【方案】
1. 归因分析：逆向分析生成过程，找关键信号
2. 特征重要性：SHAP、LIME
3. 可视化：展示候选、排序过程、最终结果
4. 硬约束保证：（见 Part B2）

【安全性】
- 防止"越狱"（用户通过 prompt 操纵排序）
- 防止歧视（确保相同品质的广告被公平对待）
```

### E3. 数据效率

```
【问题】
生成模型需要大量数据：
- LLM：10B+ tokens
- Diffusion：百万级排序示例

广告系统数据量？
- 日均排序：十亿级
- 但有效的（有点击的）：千万级

【方案】
1. 迁移学习：用 LLM 预训练，然后 finetune
2. 数据增强：合成排序数据
3. 半监督学习：用无标签数据
```

## Part F：现状与未来

### F1. 2024-2025 的现状

```
【学术研究】
多篇论文验证 GR 可行性，但多为离线评估

【工业应用】
- Google、Meta：初步探索，未大规模上线
- 推荐系统（小红书、TikTok）：已有应用
- 广告系统：仍以 LTR 为主，GR 在试验

【瓶颈】
1. 推理延迟（生成式本身是 slow）
2. 成本（模型参数多，GPU 占用高）
3. 复杂性（维护、监控、调参）
```

### F2. 未来方向（2025+）

```
1. 效率突破
   - 更小的模型（蒸馏）
   - 更快的推理（ONNX、TensorRT）
   - 边缘计算（本地排序）

2. 多模态融合
   - 广告内容（图片、视频）理解
   - 用户行为视频（理解真实意图，不止点击）
   
3. 跨域迁移
   - 一个排序模型，多个平台通用
   - 学到通用的"好排序"概念

4. 可信 AI
   - 解释性增强
   - 公平性保证
   - 隐私保护
```

## Part G：决策框架

### G1. 什么时候考虑 GR

✅ **适合 GR**：
- 广告库超大（百万级+）
- 新广告频繁加入（冷启动问题严重）
- 有足够的工程资源（不是小团队）
- 已有成熟的 LTR 基线

❌ **不适合 GR**：
- 才开始做排序（先做 LTR）
- 延迟要求 <50ms（GR 难以满足）
- 数据量不足（难以训练生成模型）
- 团队不懂深度学习

### G2. 实施路线

```
【阶段 1】巩固 LTR 基线（3-6 个月）
- NDCG@10 作为目标指标
- 特征工程、模型调优
- 建立完整的离线评估框架

【阶段 2】探索生成式（3-6 个月）
- 离线数据上训练小规模 GR
- 对比离线 NDCG
- 评估推理延迟（能否接受）

【阶段 3】线上试验（1-3 个月）
- 小流量 A/B 测试
- 关键指标：RPM、CTR、用户留存
- 对比 LTR 基线

【阶段 4】稳定与优化（持续）
- 监控线上效果
- 定期更新模型
- 结合在线反馈（RL）
```
