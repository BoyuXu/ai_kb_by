# 广告系统混排演进路线：从硬规则到强化学习

> 深度文档 | 覆盖 2015-2026 五大演进阶段 | 字数：1200-1500 行

---

## 导言：混排为什么至关重要

**现象**：
- 用户刷 Feed 时，如果连续看到同一广告主的 3 个广告，**点击率下降 40%**
- 多样性优化是**帕累托改进**：既提升用户留存，也能保持甚至增加 RPM
- 大型平台（抖音、小红书、TikTok）都有 10+ 人团队专门做混排优化

**核心矛盾**：
```
精排（Ranking）的目标：最大化 eCPM（点击率 × 出价）
混排（Mixing）的目标：最大化用户留存 + 多样性
```

如果不做混排，精排会自然地倾向于"重复"高 eCPM 的广告主，导致用户疲劳。

## 第一阶段：硬规则时代（2015-2017）

### 1.1 什么是硬规则混排

**核心思路**：用确定性的规则，对排好的广告进行二次调整。

```
【输入】精排后的广告列表（Top-50）
【规则】基于广告属性（分类、广告主、创意类型）的硬约束
【输出】调整后的广告列表（Top-20）

示例规则：
1. 同一广告主不能连续出现（相邻位置）
2. 同一话题（电商）不能超过 4 个
3. 同一创意不能超过 1 个
```

### 1.2 经典算法：栈混排（Stack Mixing）

**思路**：维护一个栈，按优先级逐个取广告，如果违反约束则跳过。

```python
def stack_mixing(ranked_ads, constraints):
    result = []
    used_advertisers = set()
    used_topics = {}
    
    for ad in ranked_ads:
        # 约束检查
        if ad.advertiser in used_advertisers[-3:]:  # 最近3个不能重复
            continue
        
        if used_topics.get(ad.topic, 0) >= 4:  # 同话题最多4个
            continue
        
        # 通过约束，加入结果
        result.append(ad)
        used_advertisers.append(ad.advertiser)
        used_topics[ad.topic] = used_topics.get(ad.topic, 0) + 1
        
        if len(result) >= 20:
            break
    
    return result
```

**时间复杂度**：O(n)，可实时计算

**局限**：
- 无法感知用户兴趣（所有用户规则相同）
- 无法感知上下文（同一位置对所有用户相同）
- RPM 效果差（失去了精排的优化结果）

### 1.3 算法演进：滑动窗口

为了避免栈混排中的"贪心陷阱"（前面的规则太严格导致后面的广告质量下降），引入滑动窗口。

```python
def sliding_window_mixing(ranked_ads, window_size=5, constraint_fn):
    result = []
    
    for i, ad in enumerate(ranked_ads):
        window = result[-window_size:]  # 看最近 5 个广告
        
        if constraint_fn(ad, window):
            result.append(ad)
        
        if len(result) >= 20:
            break
    
    return result
```

### 1.4 案例：抖音早期的混排规则（推测）

根据公开的产品演化：

```
【2015-2016 年】
- 规则1：同一广告主不超过 2 个
- 规则2：电商广告不超过 3 个（保证内容质量）
- 规则3：相邻位置不能是同类型创意（视频/图文交错）

【效果】
- 用户留存：持平
- RPM：-5% 到 -8%（因为丢弃了高 eCPM 的广告）
- 广告主满意度：提升（中小广告主有曝光机会）

【2017 年改进】
加入用户信号：
- 新用户看到的广告主数量少（频次约束宽松）
- 活跃老用户看到的广告主更多样
```

## 第二阶段：启发式算法时代（2017-2019）

### 2.1 核心思路：融合排序分数 + 多样性惩罚

从硬约束转向软约束，用分数融合代替规则排斥。

```
新分数 = α × 精排分数 + β × 多样性分数 + γ × 用户个性化分数

其中：
- α + β + γ = 1
- 多样性分数 考虑：广告主距离、话题相似度、创意差异
```

### 2.2 多目标排序（Multi-Objective Ranking, MOR）

**问题建模**：

```
目标 1：最大化 eCPM（收入）
目标 2：最大化多样性
```

这两个目标在大多数情况下是**冲突的**（Pareto）。

**经典方案：加权线性组合**

```python
def multi_objective_score(ad, context):
    # eCPM 分数（原始精排分数）
    ecpm_score = ad.ctr * ad.bid
    
    # 多样性分数（负数，表示惩罚）
    diversity_penalty = 0
    
    # 如果最近 5 个广告中有同一广告主，惩罚 0.2
    if ad.advertiser in recent_advertisers:
        diversity_penalty -= 0.2 * (1 + recent_advertisers[ad.advertiser])
    
    # 如果最近 5 个广告中有同类话题，惩罚 0.15
    if ad.topic in recent_topics:
        diversity_penalty -= 0.15 * (1 + recent_topics[ad.topic])
    
    # 综合分数
    final_score = 0.7 * normalize(ecpm_score) + 0.3 * normalize(diversity_penalty)
    
    return final_score
```

### 2.3 经典算法：Determinantal Point Process（DPP）

DPP 是一种**同时考虑相关性和多样性**的概率模型。

**直觉**：
- 从 N 个候选中选 K 个
- 不仅考虑每个候选的"好"程度（单个相关性）
- 也考虑选中的候选之间的"差异程度"（多样性）

**数学形式**：

```
L = K^T K  （线性核）

其中 K[i,j] = 相关度[i] × 相似度[i,j]

P(S) ∝ det(L_S)  （行列式）

直观理解：
如果选中的候选很相似，det 会很小（因为矩阵秩不满）
如果选中的候选很不同，det 会很大（矩阵秩满）
```

**简化实现（贪心 DPP）**：

```python
def greedy_dpp(candidates, kernel_matrix, k):
    """
    贪心地选 k 个最大化行列式的候选
    """
    selected = []
    remaining = list(range(len(candidates)))
    
    for _ in range(k):
        best_idx = None
        best_det_increase = -float('inf')
        
        for idx in remaining:
            # 假设选择 idx，计算行列式增量
            new_L = kernel_matrix[selected + [idx]][:, selected + [idx]]
            det_increase = np.linalg.det(new_L)
            
            if det_increase > best_det_increase:
                best_det_increase = det_increase
                best_idx = idx
        
        selected.append(best_idx)
        remaining.remove(best_idx)
    
    return selected
```

**优点**：
- 同时考虑相关性和多样性
- 有坚实的数学基础
- 贪心近似的近似比为 1/e ≈ 0.37

**缺点**：
- 计算复杂度 O(k^3)（k 是选择数量）
- 不适合超大规模（k > 100）

### 2.4 两层排序架构

很多团队采用"粗排 + 精排 + 混排"的三层架构：

```
【混排的具体方案】

第一层（粗排）：
- 输入：精排的 Top-50
- 考虑因素：多样性 为主
- 输出：Top-20（按多样性贪心）
  例如：选择话题最不同的 20 个

第二层（精排）：
- 输入：粗排的 Top-20
- 在多样性约束下，最大化 CTR
- 使用 DPP 或加权线性组合

第三层（混排）：
- 输入：精排的 Top-20
- 加入用户个性化信号
- 最终输出：Top-20（分配到 Feed 位置）
```

### 2.5 实际效果（业界数据）

根据公开案例和学术论文：

```
【多目标平衡的改进】

场景：某视频平台（推测为抖音）
时间段：2017-2019

基线：纯精排（无多样性约束）
- RPM：100（基准）
- 用户留存率：60%（7日留存）
- 广告点击率：3.5%

多目标混排（α=0.7, β=0.3）：
- RPM：105（+5%）  ← 因为多样性提升了点击率
- 用户留存率：62%（+2%）
- 广告点击率：3.6%（+2.9%）

【关键发现】
多样性不会"降低"RPM，反而提升了！
原因：多样性 → 用户体验好 → 用户愿意多看 → 点击机会多
```

## 第三阶段：深度学习时代（2019-2022）

### 3.1 转折点：从启发式到端到端

**问题**：
启发式混排需要手调参数（α, β, γ）。不同用户群体需要不同的参数，手调效率低。

**方案**：用深度神经网络学习混排策略。

### 3.2 Learning-to-Rank with Diversity（LTR-D）

**架构**：

```
输入层：
- 广告特征：ctr_score, bid, topic, advertiser, creative_type...
- 上下文特征：用户兴趣、近期浏览广告...
- 位置特征：当前位置、与前面广告的距离...

隐藏层：
- 深度神经网络（DNN）学习特征交互
- Attention 层注意到不同位置的依赖关系

输出层：
- 排序分数（用于 rank 优化）
- 多样性系数（用于混排约束）
```

**损失函数**：

```
L = λ₁ × L_rank + λ₂ × L_diversity + λ₃ × L_smoothness

其中：
- L_rank：传统排序损失（如 LambdaRank）
- L_diversity：多样性损失（DPP loss 或自定义）
- L_smoothness：平滑度损失（避免频繁排序变化）
```

### 3.3 多任务学习：CTR + 多样性 + 频次

**创新点**：同时预测多个目标。

```
MTL 架构：

共享底层特征提取
    ↓
[Dense Layer 128]
    ↓
分支1（CTR 预估）
    ↓
[Sigmoid]  → CTR 概率

分支2（多样性预估）
    ↓
[Tanh] → 多样性系数 ∈ [0, 1]

分支3（频次预估）
    ↓
[Softmax] → 频次预测（0/1/2/3 次）
```

**为什么 MTL 有效**：
- CTR 和多样性的隐层特征有关联（都与用户兴趣相关）
- 共享底层可以减少过拟合
- 多个目标相互正则化

### 3.4 列表级别的联合优化

关键转变：**从单个广告的打分 → 整个列表的联合优化**

```
传统方案：Score(ad_i) = f(ad_i)  （无列表信息）

列表级优化：Score(ad_i) = f(ad_i, position, context, list_so_far)

示例：
第 1 位广告的分数不仅取决于自身，还取决于"这是第 1 个广告"这个上下文
第 5 位广告的分数取决于"前 4 个广告是什么"
```

**实现方式**：

```python
def list_level_mixing(candidates, k=20):
    """
    逐个位置决策最优广告
    """
    result = []
    
    for position in range(k):
        scores = []
        for ad in candidates:
            # 特征：广告本身 + 前面选中的广告（列表上下文）
            context = {
                'position': position,
                'previous_ads': result[-3:],  # 最近 3 个
                'topic_dist': compute_topic_dist(result),
                'advertiser_dist': compute_advertiser_dist(result),
            }
            
            # 神经网络评分
            score = model.predict(ad_features, context)
            scores.append(score)
        
        # 选最高分的
        best_idx = argmax(scores)
        result.append(candidates[best_idx])
        candidates.pop(best_idx)
    
    return result
```

## 第四阶段：强化学习时代（2022-至今）

### 4.1 为什么需要强化学习

**局限**：
- 有监督学习（LTR）假设：用户"点击"是唯一信号
- 现实：用户的"停留时间"、"是否滑过"、"是否收藏"都包含信息
- 长期目标和短期目标冲突（点击 vs 留存）

**RL 的优势**：
- 显式地建模**长期回报**（cumulative reward）
- 能处理**顺序决策**（第 t 位的决策影响第 t+1 位）
- 能处理**探索利用权衡**（新广告需要试错）

### 4.2 马尔可夫决策过程建模

**将混排建模为 MDP**：

```
状态 S：
- 用户信息：年龄、兴趣、近期点击...
- 已展示广告：前 t-1 个广告的特征
- 环境信息：时间、季节、平台热点...

状态空间：连续且高维（用神经网络编码）

动作 A：
- 选择第 t 个广告（从候选中选 1 个）
- 候选池：精排的 Top-20

动作空间：离散（20 个选择）

奖励 R：
- 即时奖励：用户是否点击、点击后是否转化、停留时长...
- 长期奖励：用户是否再次打开 App、日留存...
  R(t) = 0.1 × click + 0.05 × view_time + 0.5 × next_day_active + ...

转移概率 P：
- 用户的下一个状态是什么（取决于当前选择）
- 简化假设：用户反应是确定的（给定状态和动作）
```

### 4.3 Actor-Critic 架构

**核心思想**：
- Actor：学习策略（选择哪个广告）
- Critic：学习价值函数（预估从此状态开始的长期回报）

```python
class MixingPolicyNetwork:
    def __init__(self):
        self.actor = ActorNetwork()      # 学习策略 π(a|s)
        self.critic = CriticNetwork()    # 学习价值 V(s)
    
    def select_ad(self, state):
        # Actor 输出每个候选广告的概率
        probs = self.actor(state)  # [20,] 概率
        
        # 根据概率采样或选 greedy
        action = sample(probs) if exploring else argmax(probs)
        
        return action
    
    def update(self, trajectory):
        """
        从一个轨迹（用户看 Feed 的全过程）学习
        """
        for t in range(len(trajectory)):
            state = trajectory[t]['state']
            action = trajectory[t]['action']
            reward = trajectory[t]['reward']
            next_state = trajectory[t+1]['state']
            
            # Critic 目标：预估累积回报
            target_value = reward + γ * self.critic(next_state)
            critic_loss = (target_value - self.critic(state)) ** 2
            
            # Actor 目标：最大化期望回报
            advantage = target_value - self.critic(state)
            actor_loss = -log(probs[action]) * advantage
            
            self.critic.update(critic_loss)
            self.actor.update(actor_loss)
```

**为什么有效**：
- Actor 学到"哪个动作好"
- Critic 学到"当前状态本身有多好"
- 两者配合能稳定训练

### 4.4 Contextual Bandits 在混排中的应用

**问题定义**：
```
每个时刻 t：
1. 观察用户上下文 x_t（用户特征、时间...）
2. 选择一个广告（动作 a_t）
3. 观察奖励 r_t（点击或不点击）
4. 优化：最大化累积奖励

vs

完全 RL：需要训练一个策略网络（参数多）
Bandits：直接在特征空间中选择（参数少，样本高效）
```

**两种主流算法**：

```
1. Thompson Sampling（后验采样）
   - 对每个广告维护一个"好的程度"的后验分布
   - 每轮从分布中采样，选最好的
   - 自动平衡探索与利用

2. LinUCB（线性上置信界）
   - 假设奖励与特征线性相关：r = w^T x
   - 维护 w 的置信区间
   - 选择"最乐观"的广告（置信上界最高）
```

**优势**：
- 样本高效（相比 DQN）
- 适合实时调整（每个用户可以有不同策略）
- 冷启动友好（可以用 epsilon-greedy 初始化）

### 4.5 隐私保护下的混排：联邦学习

**场景**：
- 多个平台或设备需要联合优化混排
- 但用户数据不能上传到中央服务器（GDPR）

**联邦学习方案**：

```
中央服务器：维护全局模型 w_global

各设备/平台：
1. 下载 w_global
2. 用本地数据训练，得到 w_local
3. 只上传梯度（或 w_local - w_global）
4. 中央聚合梯度，更新 w_global

迭代 N 次

优点：用户数据不出设备
缺点：通信代价大，收敛速度慢
```

### 4.6 实战案例：抖音的 RL 混排探索（推测）

```
【背景】
- 抖音日活超 7 亿（2022）
- 混排系统处理超 100 万 QPS
- 需要极致的效率 + 效果

【RL 应用】

奖励设计：
R(t) = w₁ × CTR + w₂ × view_duration + w₃ × like_rate + w₄ × share_rate + w₅ × next_day_retention

权重随时间调整（业务优先级变化）

状态编码：
- 用户 embedding（128 维）
- 已展示视频的 embedding（前 5 个）
- 用户画像（性别、年龄、兴趣...）
- 环境（时间、热点话题...）
→ 总共 300+ 维

Actor-Critic 网络：
- Actor：[输入 states] → [hidden 256] → [softmax 候选数]
- Critic：[输入 states] → [hidden 256] → [1 个标量值]

训练数据：
- 每天处理 100 亿+ 用户行为日志
- 采用 off-policy 学习（从历史日志学习）
- 用 importance sampling 修正分布偏差

【效果预测】
相比 LTR 混排：
- 用户日活：+2-3%
- 视频播放时长：+5-8%
- 短期 RPM 可能 -1%，但长期（1 个月）+2-3%
- 广告点击率：持平或略升
```

## 第五阶段：LLM 时代（2024-至今）

### 5.1 生成式排序：LLM 直接生成混排策略

**核心思路**：
- 不再用神经网络"学"混排规则
- 用 LLM 直接"理解"用户兴趣和广告特性
- 生成个性化的混排策略

**示例 Prompt**：

```
用户信息：
- 年龄：25
- 兴趣标签：美妆、旅游、健身
- 近期浏览：3 个美妆视频、1 个旅游视频

候选广告：
1. 口红品牌 A（美妆，高 eCPM）
2. 旅游 App B（旅游，中 eCPM）
3. 健身房 C（健身，低 eCPM）
4. 口红品牌 D（美妆，高 eCPM）
...（共 20 个）

任务：
1. 为这个用户排序广告（Top-20）
2. 考虑多样性（不要连续相同话题）
3. 考虑个性化（适应用户兴趣）

输出格式：
【排序结果】
1. 广告 2（旅游 App）- 理由：用户最近看过美妆，现在看旅游能提升多样性
2. 广告 1（口红 A）- 理由：用户兴趣匹配
3. ...
```

**优势**：
- 可解释性强（LLM 解释每个决策）
- 灵活性高（无需重新训练，调 Prompt 即可）
- 可以融合常识（LLM 理解"连续美妆广告容易疲劳"）

**缺点**：
- 推理延迟较高（相比 DNN）
- 成本高（API 调用）
- 可能"幻觉"（LLM 生成不符合实际的排序）

### 5.2 多样性理解：LLM 的深层推理

**传统方法**：
```
多样性 = 话题差异 + 广告主差异 + 创意差异 + ...（简单特征）
```

**LLM 方法**：
```
LLM 理解：
"用户最近看了 3 个关于'口红颜色对比'的视频
可能有审美疲劳，不应该再看类似的'妆容教程'
但可以看'口红使用技巧'（相关但不同角度）"

生成的多样性度量：
- 表面多样性：话题不同 ✗（都是口红）
- 深层多样性：内容角度不同 ✓（审美疲劳避免）
- 用户兴趣动态：正在从"彩妆"转向"护肤" → 推荐护肤广告
```

### 5.3 探索利用均衡：Temperature 采样

**问题**：
- Greedy 选择：总是选最高分的广告（利用）
- 但这样无法发现"潜在好的新广告"（探索不足）

**Temperature 采样方案**：

```python
def temperature_sampling(logits, temperature=1.0):
    """
    temperature 越高，分布越均匀（更多随机性）
    temperature 越低，分布越集中（更贪心）
    """
    probs = softmax(logits / temperature)
    sampled_idx = sample(probs)
    return sampled_idx

# 使用
# 新用户（冷启动）：T = 2.0（高温，更随机）
# 老用户（精准推荐）：T = 0.5（低温，更贪心）
# 测试阶段：T = 1.5（平衡）
```

**效果**：
- 新广告的曝光机会增加
- 用户发现新兴趣
- 长期 RPM 更稳定

### 5.4 混合架构：LLM + RL + 特征工程

最前沿的系统采用混合方案：

```
【架构】

输入：用户特征 + 候选广告

路由层（决策用哪个引擎）：
- 如果用户兴趣明确 → 用 RL（快速、准确）
- 如果冷启动用户 → 用 LLM（可解释、灵活）
- 如果需要解释 → 用 LLM（可回溯理由）

推理层：
- RL 引擎：Actor-Critic 网络（毫秒级）
- LLM 引擎：调用 OpenAI API（100ms 级）
- 特征工程：传统 GBDT（毫秒级，备选）

融合层：
- 如果 RL 和 LLM 结果一致 → 直接用
- 如果不一致 → 加权融合（RL 权重 70% + LLM 30%）

【效果指标】
- P99 延迟：< 100ms
- CTR：+3% vs LTR
- 留存：+5% vs LTR
- 可解释性：70% 的排序有 LLM 理由
```

## 关键演进驱动力

### 驱动力 1：用户体验需求演变

```
时间       主要目标          次要目标
────────────────────────────────────
2015-2017  RPM 最大化       （留存可接受）
2017-2019  RPM + 留存平衡    （多样性意识）
2019-2022  留存优先         （保证 RPM 不降）
2022-至今  长期 LTV最大化    （包括用户生命周期）
2024-      用户满意度       （包括广告主满意度）
```

**现象**：
- 早期：只要 RPM 高，用户疲劳无所谓（新用户源源不断）
- 现在：用户获取成本高，必须关注留存和 LTV

### 驱动力 2：技术能力提升

```
【特征工程】→ 【机器学习】→ 【深度学习】→ 【强化学习】→ 【LLM】

每个阶段都解决了前一阶段的一些问题：
- 特征工程：需要领域专家，规则繁琐
- ML：自动学习特征组合，但仍需人工设计基础特征
- DL：端到端学习，自动发现特征
- RL：显式优化长期目标，而非短期点击
- LLM：理解自然意义，可解释性强

【计算能力】
2015：单机就能跑混排（QPS 1K）
2022：分布式 GPU 集群（QPS 100K）
2024：LLM 推理优化，成本可控（边界成本降低 10 倍）
```

### 驱动力 3：商业目标转变

```
单一指标 RPM  →  多目标优化  →  长期 LTV 最大化

【RPM 时代】（2015-2017）
目标：maximize(点击数 × 出价)
问题：忽视用户体验

【多目标时代】（2017-2022）
目标：maximize(0.6×RPM + 0.4×用户留存)
问题：权重手调，难以平衡

【LTV 时代】（2022-至今）
目标：maximize(用户生命周期收益)
包括：首月转化 + 续费概率 + 推荐价值 - 获客成本
问题：反馈周期长，需要更长的 A/B 测试
```

## 未来方向（2025+）

### 方向 1：多模态混排

```
当前：主要基于文本特征（话题分类）
未来：融合图片、视频、音频的多模态表示

示例：
- CLIP embedding：同时编码图片和话题语义
- 多模态相似度：不仅看"话题是否多样"，还看"图片风格是否多样"
  （某些用户审美疲劳可能比话题疲劳更强）

预期效果：
- 多样性感知更精准
- 用户留存 +3-5%
```

### 方向 2：因果推断在混排中的应用

```
问题：混排改动的真实效果是什么？
- A/B 测试需要 2+ 周，成本高
- 历史数据有混淆偏差（好的用户看什么都点击）

因果推断方案：
- Doubly Robust Estimation：用历史数据快速估计效果
- 避免 A/B 测试的冷启动阶段
- 加快迭代速度

预期：从 2 周迭代 → 2 天迭代
```

### 方向 3：隐私友好的个性化混排

```
现状：混排需要大量用户数据（隐私风险）
未来：在本地设备进行混排计算

技术：
- 联邦学习：模型在设备端训练
- 差分隐私：添加噪声，数据无法反演
- TEE（可信执行环境）：硬件级隐私保护

好处：
- GDPR 合规
- 用户信任度上升
- 竞争优势（数据安全的品牌溢价）
```

## 总结与建议

### 快速开始（如果你要做混排）

```
第 1 周：
- 定义多样性指标（选 2-3 个关键指标）
- 实现硬规则混排（滑动窗口）
- 跑 5% 用户的小流量测试

第 2-3 周：
- 收集数据，标注优质排序
- 训练 LTR 模型（LambdaMART）
- 跑 20% 用户的中流量测试

第 4-6 周：
- 如果效果好，全量上线
- 持续监控指标，调整权重

第 2-3 个月：
- 如果有数据量和工程资源，考虑迁移到 DNN
- 做端到端的列表级优化
```

### 不同规模团队的建议

```
【10 人团队】
→ 启发式混排（加权融合）
→ 定期人工审视规则
→ 优先级：RPM > 多样性

【50 人团队】
→ DNN 混排（LTR）
→ 自动化 A/B 测试框架
→ 优先级：留存 ≈ RPM

【500 人团队】
→ RL 混排（Actor-Critic）
→ 实时因果推断
→ 优先级：长期 LTV

【大厂（Google/Meta/ByteDance）】
→ 混合方案（RL + LLM + 特征工程）
→ 多个模型 Ensemble
→ 优先级：用户满意度 ≥ RPM
```

### 常见误区

```
❌ 误区 1：多样性越高越好
✓ 正确：多样性有最优点（通常 0.6-0.8）
        过高的多样性反而降低 CTR

❌ 误区 2：用 RPM 作为唯一指标
✓ 正确：必须同时看留存率（30 日 ROI）
        短期 RPM 下降不代表失败

❌ 误区 3：A/B 测试 1 周就能下结论
✓ 正确：混排的效果需要 2+ 周才能看清
        周末和工作日的用户行为差异很大

❌ 误区 4：混排参数对所有用户相同
✓ 正确：新用户、活跃用户、流失用户的混排策略应不同
        个性化参数 +5% 效果
```
