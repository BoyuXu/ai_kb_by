# RTBAgent: LLM as Real-Time Bidding Agent
> 来源：工业论文 | 领域：广告系统 | 学习日期：20260327

## 问题定义
将LLM作为RTB（实时竞价）的出价Agent，挑战：
1. **极端延迟要求**：RTB系统要求<10ms决策，LLM推理通常需要100ms+
2. **大规模并发**：每秒处理数百万次竞价请求
3. **出价推理能力**：LLM需要理解市场竞争、预算约束、用户价值等复杂因素
4. **知识实时性**：LLM训练数据有截止日期，无法感知实时市场变化

**目标**：构建LLM-based RTB Agent，用CoT出价推理实现ROI+10%，同时满足<10ms延迟约束。

## 核心方法与创新点

### 1. 两阶段出价架构
**离线策略生成**（LLM，秒级延迟）：
- LLM分析历史市场数据、竞争态势、预算状态
- 生成出价策略参数（而非具体出价值）：{base_bid, bid_multiplier_curve, budget_pacing_ratio}

**在线执行**（规则引擎，<1ms）：
- 每次请求时，用LLM预生成的策略参数计算具体出价
- `bid = base_bid × f(user_value) × budget_pacing_ratio`

### 2. CoT出价推理（离线）
LLM在离线阶段进行CoT推理，生成策略：
```
当前市场状况分析：
1. 过去1小时竞争对手平均出价：¥1.2（较昨日+15%，需提价）
2. 当前预算使用进度：40%（目标50%，需加速投放）
3. 今日ROI：3.8（高于目标3.0，有提价空间）
→ 策略：提高base_bid 20%，加速系数=1.3，维持ROI≥3.0约束
```

### 3. 实时用户价值函数
在线执行时，用轻量级模型实时估计用户价值：

$$
v_{user} = f_{lightweight}(user\_features) \quad \text{(<1ms)}
$$

最终出价：

$$
b_{final} = \text{base\_bid} \times v_{user} / v_{avg} \times \text{budget\_pacing}
$$

### 4. 策略更新频率
- LLM策略生成：每5-15分钟更新一次
- 在线用户价值函数：实时计算
- 预算pacing参数：每分钟更新

## 实验结论
- **ROI提升**：+10%（相比规则出价方案）
- **延迟**：在线执行<1ms（LLM离线策略生成不在关键路径上）
- **预算达成率**：96%（相比传统方案的88%）
- CoT推理比直接生成策略的ROI高+4%（推理质量更高）

## 工程落地要点
1. **策略参数颗粒度**：策略参数设计要兼顾表达能力和安全性，参数范围要有合理约束
2. **LLM生成失败兜底**：LLM超时或格式异常时，保留上一个有效策略
3. **策略预热**：新广告活动上线时，使用默认策略直到收集足够数据供LLM分析
4. **多广告主隔离**：不同广告主的策略完全隔离，防止信息泄露
5. **策略可解释性**：保存LLM的CoT推理链，供广告主查询"为什么这样出价"

## 面试考点
Q1: RTBAgent如何在保证<10ms延迟的同时使用LLM？
A: 核心方案是"离线策略生成+在线执行"的两阶段架构：LLM不在每次竞价的关键路径上，而是周期性地（每5-15分钟）分析市场数据并生成策略参数。每次真实竞价请求时，只使用LLM预先生成的简单策略参数（几个浮点数），结合轻量级的实时用户价值预测，整个在线执行<1ms。LLM的智慧"离线预计算"，执行时只用轻量逻辑。

Q2: 两阶段架构中，策略更新延迟（5-15分钟）会导致什么问题？
A: (1)市场突变响应慢：竞争对手突然大幅提价时，需要等15分钟才能调整策略；(2)预算节奏问题：广告在短时间内集中投放导致超预算，下一次策略更新才能纠正；(3)实时信号利用率低：实时的用户意图信号无法充分利用。缓解方案：缩短策略更新频率（代价：LLM调用成本增加）；对异常情况触发紧急策略更新（如检测到预算消耗速度异常时立即更新）。

Q3: LLM出价Agent相比DRL（深度强化学习）出价有什么优缺点？
A: LLM出价优点：(1)快速部署，无需复杂的RL训练；(2)可解释，CoT推理链清晰；(3)Zero-shot到新广告类型，泛化性好；(4)结合人类专家知识（通过Prompt）。DRL出价优点：(1)优化精确，可以端到端优化具体指标；(2)更快的在线决策（策略网络轻量）；(3)对历史数据利用更充分。缺点互补：LLM作为高层策略，DRL作为低层执行，结合使用效果最佳。
