## 第 1 部分：排序系统的通用演进阶段

### 1.1 排序系统的三层结构

```
                    用户看到的结果
                          ↑
    ┌─────────────────────┴──────────────────────┐
    │      重排层（Reranking）                     │
    │  目标：多样性、公平性、实时约束              │
    └───────────────────┬────────────────────────┘
                        ↑
    ┌─────────────────────┴──────────────────────┐
    │      精排层（Ranking / Scoring）              │
    │  目标：主要优化目标（CTR/CVR/相关性）        │
    │  模型：DNN、LTR、多目标优化                  │
    └───────────────────┬────────────────────────┘
                        ↑
    ┌─────────────────────┴──────────────────────┐
    │      召回层（Retrieval / Candidate Pool）    │
    │  目标：高效获取候选集（数百 vs 数十万）      │
    │  方法：向量检索、多路召回                    │
    └────────────────────────────────────────────┘
```

### 1.2 四个演进阶段

```
Stage 1: 单目标排序 (2010-2015)
  score = CTR × Bid
  问题：广告轰炸，用户体验差

Stage 2: 多目标融合 (2015-2019)
  score = w1×CTR + w2×CVR + w3×Brand_Safety + ...
  核心：权重平衡，多任务学习（MMOE/PLE）

Stage 3: 约束优化与LTR (2019-2022)
  objective: max(CTR + CVR + ...)
  s.t. Fairness ≥ threshold
       Position_Diversity ≥ threshold
  方法：线性规划、Learning to Rank

Stage 4: 端到端与生成式排序 (2023+)
  从"排序候选"→ "生成排序"
  模型：Transformer decoder、自回归生成
```

## 第 2 部分：各阶段的关键算法

### 2.1 Stage 1：基础排序公式

#### eCPM 排序（最简单）
$$
\text{score} = \text{pCTR} \times \text{pCVR} \times \text{bid}
$$

**特点**：
- 单一得分，全局排序
- O(n log n) 复杂度，极高效率
- 局限：完全忽视用户体验，容易被高价广告主垄断

#### 质量度得分改进
$$
\text{score} = \text{eCPM} \times \text{QualityScore}
$$

其中 QualityScore 综合考虑：
- Landing page 质量
- 广告创意质量  
- 广告主历史表现

### 2.2 Stage 2：多目标排序算法

#### 2.2.1 加权线性组合（业界标配）

$$
\text{score} = \sum_i w_i \times f_i(x)
$$

其中 $f_i$ 为不同目标函数（CTR、CVR、Brand Safety），$w_i$ 为权重。

**权重设置方法**：

1. **静态权重**：基于历史经验
   - CTR: 0.5
   - CVR: 0.3
   - Brand Safety: 0.2
   - 优点：简单、快速
   - 缺点：无法自适应

2. **动态权重**（推荐）
   - 按实时 KPI 调整（如平台收益、用户 CTR）
   - 每小时或每天重新计算
   - 需要在线 A/B 验证

3. **Pareto 前沿**（高级）
   - 多个不同权重组合对应 Pareto 前沿
   - 不同用户群体选择不同前沿点
   - 例：高消费用户看更多商业内容，普通用户内容多些

#### 2.2.2 多任务学习框架

**MMOE（多门控混合专家）**：

```
        输入特征
           ↓
    ┌──────────────┐
    │ Expert 网络群 │ (4-8 个独立网络)
    └──────┬───────┘
           ↓
    ┌──────────────────────────────┐
    │ 多个 Gating 网络               │
    │ - CTR Gating: 从Expert中选择  │
    │ - CVR Gating: 从Expert中选择  │
    │ - FAV Gating: 从Expert中选择  │
    └──────┬───────────┬───────────┬──┘
           ↓           ↓           ↓
        CTR塔        CVR塔       FAV塔
           ↓           ↓           ↓
        pCTR        pCVR        pFAV
```

**核心公式**：

Expert 输出：
$$
e_i = \text{Expert}_i(x)
$$

任务 k 的 Gating：
$$
g_k = \text{Softmax}(W_k x)
$$

任务 k 的融合表示：
$$
h_k = \sum_i g_{ki} \times e_i
$$

**优势**：
- 不同任务选择不同的 Expert 组合
- 缓解多任务间的梯度冲突
- 应用：广告（CTR+CVR）、推荐（CTR+CVR+Fav+Comment）

#### 2.2.3 PLE（渐进分离多任务框架）

在 MMOE 基础上引入**显式的任务私有 Expert**：

```
Layer 1:
  Task A 私有 Expert + Shared Expert
    ↓
  Gating (Task A)
    ↓
  
Task B 私有 Expert + Shared Expert
    ↓
  Gating (Task B)

(逐层堆叠)
```

**适用场景**：
- 任务高度不相关（点击 vs 评论 vs 分享）
- 需要完全隔离任务特定的梯度

## 第 3 部分：Learning to Rank（LTR）范式

### 3.1 三大 LTR 范式

#### 3.1.1 Pointwise：单点预测

$$
L = \sum_i \text{CrossEntropy}(y_i, \hat{y}_i)
$$

**特点**：
- 每条候选独立预测
- 忽视排序顺序关系
- 最简单的方法，但无法建模相对顺序

#### 3.1.2 Pairwise：对比学习

$$
L = \sum_{i,j: y_i > y_j} \text{log}(1 + \exp(-(\text{score}_i - \text{score}_j)))
$$

**特点**：
- 预测一对候选的相对顺序
- 对排序更友好
- 例：RankNet、LambdaRank

#### 3.1.3 Listwise：列表级优化

$$
L = -\log \frac{\exp(s_i)}{\sum_j \exp(s_j)}
$$

**特点**：
- 直接优化列表的排序
- 所有候选一起考虑
- 最符合实际排序任务
- 例：LambdaMART、ListNet、ApproxNDCG

**对比**：

| 范式 | 单点 | 对比 | 列表 |
|------|------|------|------|
| 排序优化 | 弱 | 中 | 强 |
| 实现复杂度 | 低 | 中 | 高 |
| 推荐场景 | ❌ | ⚠️ | ✅ |

### 3.2 LambdaRank 与 LambdaMART

#### LambdaRank 的核心洞察

传统方法需要定义明确的损失函数，但 LTR 的真实目标是 NDCG（归一化折扣累积收益），不可微。

**LambdaRank 的解决方案**：
直接在梯度中融入排序指标。对于候选对 $(i, j)$：

$$
\text{gradient} \propto |\Delta NDCG_{ij}| \times \sigma(\text{score}_i - \text{score}_j)
$$

其中 $|\Delta NDCG_{ij}|$ 是交换 i、j 两个候选导致的 NDCG 变化。

**效果**：
- 自动聚焦高 NDCG 收益的交换对
- 无需显式定义损失函数
- 在搜索排序中广泛应用

#### LambdaMART 进阶

将 LambdaRank 的梯度应用到 GBDT（梯度提升决策树）：

```
初始化：所有文档得分相同

第 1 棵树：
  计算 LambdaRank 梯度
  构造回归树预测梯度
  更新得分

第 2-K 棵树：
  重复上述过程
  
结果：多棵树的累计得分
```

**优势**：
- 比 RankNet（神经网络版本）训练更快
- 可以处理高维稀疏特征
- 实际搜索引擎标配（Google、Bing）

## 第 4 部分：约束优化与公平性

### 4.1 多目标排序中的约束

实际系统排序不是单纯最大化 eCPM，还要考虑多个约束：

#### 约束 1：多样性

```
约束形式：position_diversity ≥ 0.5

含义：前 10 个结果中，广告不超过 5 个
```

实现：
- 在重排层硬约束（推荐）
- 或在排序得分中加入 diversity penalty

#### 约束 2：广告主侧公平

```
约束形式：每个广告主最多展示 N 条广告
         OR 市场份额 ≥ threshold
```

实现：
- 限流（Budget Pacing）
- 位置轮转

#### 约束 3：用户端公平

```
约束形式：不同用户群体的 CTR 差异 ≤ threshold
```

实现：
- Fairness-aware Ranking
- 按用户分层，各层单独优化

### 4.2 线性规划求解

将排序问题建模为整数规划：

$$
\text{maximize} \quad \sum_i \text{score}_i \times x_i
$$

$$
\text{s.t.} \quad \sum_i x_i = K \quad (\text{选择 K 个候选})
$$

$$
\sum_i (\text{is\_ad}_i \times x_i) \leq 5 \quad (\text{广告数量约束})
$$

$$
\text{diversity\_score}(x) \geq 0.5 \quad (\text{多样性约束})
$$

**局限**：
- 求解 NP-hard，实际用贪心近似
- 实时性难以保证

## 第 5 部分：生成式排序（最新范式）

### 5.1 从"排序候选"到"生成排序"

#### 传统范式（2010-2023）
```
Step 1: 召回 (Candidate Retrieval)
  → 获得 100-1000 个候选

Step 2: 排序 (Ranking)
  → 对候选打分排序

Step 3: 重排 (Reranking)
  → 多样性、约束优化
```

#### 生成式范式（2024+）
```
输入：用户特征、上下文

模型：Transformer Decoder (自回归)

输出：直接生成排序序列
  Item_1 → Item_2 → Item_3 → ...
  
优势：
- 端到端，无候选池限制
- 可直接优化多目标
- 可引入强化学习微调
```

### 5.2 生成式排序的技术

#### 方法 1：自回归生成

```python
# Pseudo code
for position in [1, 2, ..., K]:
  # 计算当前位置的概率分布
  prob = model(user_features, prev_items, context)
  # 按概率采样（推理）或取 argmax（训练）
  item = sample(prob) 或 argmax(prob)
  output.append(item)
```

**特点**：
- 同时优化多个位置的物品选择
- 可以建模物品间的互动（同一列表中哪些物品相邻）
- 每个位置的决策依赖前序位置

#### 方法 2：强化学习微调

```
基础模型：SFT (Supervised Fine-tuning) 生成式排序

强化学习奖励：
  r = α × CTR + β × CVR + γ × Diversity - penalty
  
更新：
  使用 PPO 或其他 RL 算法直接优化奖励
```

**优势**：
- 自动权衡多个目标，不需手工设置 w_i
- 可在线学习适应分布变化

### 5.3 对比传统排序

| 维度 | 传统（评分排序） | 生成式 |
|------|-----------------|--------|
| 候选范围 | 固定召回集 | 理论无限 |
| 多目标处理 | 权重组合 | 端到端学习 |
| 位置联动建模 | 无（独立） | 有（自回归） |
| 计算复杂度 | O(n log n) | O(n × K) |
| 微调灵活性 | 较难 | 易（RL） |

## 第 6 部分：领域对比（广告 vs 推荐 vs 搜索）

### 6.1 排序目标对比

| 领域 | 主目标 | 副目标 | 约束 |
|------|--------|--------|------|
| **广告** | 平台收入 (eCPM) | 用户体验、广告主满意度 | 多样性、预算控制 |
| **推荐** | 用户满意度 (CTR) | 商品销售、用户留存 | 新用户冷启、惊喜度 |
| **搜索** | 文档相关性 | 新鲜性、多样性 | 缓存、实时性 |

### 6.2 约束的轻重

- **广告**：约束最多（质量、公平、多样性）
- **推荐**：约束中等（多样性、冷启）
- **搜索**：约束较少（主要是相关性）

### 6.3 模型选择

- **广告**：多任务 + LTR + 硬约束 (MMOE → LambdaMART)
- **推荐**：多任务 + 序列建模 (DIN → SIM)
- **搜索**：LTR 为主 (LambdaMART → Neural LTR)

## 第 7 部分：工程实现细节

### 7.1 线上排序架构

```
┌──────────────────────────────────────┐
│         用户请求                       │
├──────────────────────────────────────┤
│ 1. 特征获取 (Realtime Features)      │
│    - 用户画像                         │
│    - 上下文信息                       │
│    - 实时行为                         │
├──────────────────────────────────────┤
│ 2. 多路召回 (Multi-path Retrieval)   │
│    - 协同过滤                         │
│    - 内容召回                         │
│    - 热点池                           │
├──────────────────────────────────────┤
│ 3. 精排 (Ranking)                    │
│    - 特征交叉 (CTR/CVR)              │
│    - 多任务融合 (MMOE/PLE)           │
│    - 得分融合                         │
├──────────────────────────────────────┤
│ 4. 重排 (Reranking)                  │
│    - 多样性保证                       │
│    - 约束检查                         │
│    - 创意多样化                       │
├──────────────────────────────────────┤
│ 5. 展示 (Delivery)                   │
│    - 日志记录                         │
│    - 实时反馈                         │
└──────────────────────────────────────┘
```

### 7.2 延迟控制

```
总延迟预算：<100ms

特征获取：   5-10ms （缓存）
多路召回：   15-20ms （并行）
精排：       40-50ms （模型预测）
重排：       10-15ms （启发式）
其他：       5ms （网络、序列化）
```
