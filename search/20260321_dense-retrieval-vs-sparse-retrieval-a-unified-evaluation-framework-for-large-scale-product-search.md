# Dense Retrieval vs Sparse Retrieval: A Unified Evaluation Framework for Large-Scale Product Search

> 来源：arxiv | 日期：20260321 | 领域：search

## 问题定义

电商产品搜索领域，密集检索（Dense Retrieval，如 DPR/双塔模型）和稀疏检索（Sparse Retrieval，如 BM25/SPLADE）长期并存，但缺乏统一公平的评估框架：

1. **评估数据集偏差**：现有基准（如 MS MARCO）偏向网页搜索，不代表电商场景（商品标题短、属性词多、无完整句子）
2. **评估维度单一**：大多只看 Recall@K，忽视延迟、内存占用、冷启动能力等工程指标
3. **公平对比困难**：不同方法用不同预训练模型，控制变量难

本文构建针对大规模电商产品搜索的统一评估框架，全面对比密集检索和稀疏检索在多个维度的表现。

## 核心方法与创新点

1. **统一评估维度体系**：
   - **效果指标**：Recall@{10,100,1000}、NDCG@10、MRR
   - **工程指标**：P50/P95/P99 检索延迟、内存占用（GB）、索引构建时间
   - **鲁棒性**：新品冷启动 Recall、罕见词查询召回、跨品类泛化
   - **可解释性**：关键词匹配路径是否可追踪

2. **电商专用测试集构建**：
   - 覆盖：商品标题搜索、属性搜索、品牌搜索、图文混合搜索
   - 长尾查询（频次 <10/天）专项测试集
   - 多语言查询（中英混搜）测试集

3. **Hybrid 方法系统性评估**：
   - 稀疏 + 密集融合（线性组合、倒数排名融合 RRF）
   - 稀疏先召回 + 密集重排
   - 密集先召回 + 稀疏重排
   - 端到端稀疏-密集联合训练（如 SPLADE 系列）

4. **关键发现（empirical）**：
   - 精确匹配查询（如SKU号、品牌+型号）：稀疏检索胜
   - 语义相似查询（如"小众香水"）：密集检索胜
   - 长尾词频查询：密集检索显著胜（BM25 严重退化）
   - 延迟：稀疏 < 密集（倒排索引更快）；内存：稀疏 < 密集（不需要存全量向量）

## 实验结论

- Hybrid（稀疏+密集 RRF 融合）在所有效果指标上优于单一方法：Recall@100 **+12%** vs 纯密集，**+18%** vs 纯稀疏
- 延迟：纯稀疏（BM25）P99 ~10ms，纯密集（FAISS）P99 ~50ms，Hybrid ~60ms（可接受）
- 内存：稀疏索引 1/10 的密集向量存储，显著节省资源
- 冷启动新品：密集检索 Recall@100 比 BM25 高 **+34%**（无点击数据时纯靠语义）

## 工程落地要点

1. **多路召回架构**：不要选"稀疏 vs 密集"，应该同时部署两路，通过 RRF 或学习的融合权重合并结果
2. **索引实时更新**：稀疏倒排索引支持实时增量更新（新品上架即可搜到）；密集 ANN 索引更新代价高（需要全量重建或增量 HNSW 插入），新品有窗口期缺失
3. **查询路由**：可以用简单分类器判断查询类型（精确匹配 vs 语义）路由到不同召回路径，降低不必要的计算
4. **资源弹性**：密集检索 GPU 资源消耗大，需要弹性扩容机制应对搜索流量高峰

## 面试考点

- Q: BM25 和密集检索各自的适用场景？
  A: BM25：精确关键词匹配（SKU号、品牌名）、低延迟要求、资源受限场景、可解释性要求。密集检索：语义理解（同义词/近义词）、冷启动（无点击数据新品）、跨语言检索、长文本理解。工业界几乎都用 Hybrid 取二者之长。

- Q: FAISS 是什么？电商搜索中如何使用？
  A: FAISS（Facebook AI Similarity Search）是高效向量相似度搜索库，支持 IVF（倒排文件索引）、HNSW（分层导航小世界图）等 ANN 算法。电商搜索中用于：(1) 双塔模型的商品向量召回；(2) 图文多模态搜索；(3) 相似商品推荐。关键配置：IVF 的 nlist/nprobe 权衡召回率和速度。

- Q: 什么是 RRF（Reciprocal Rank Fusion）？
  A: 倒数排名融合：将多路检索结果合并的简单有效方法。公式：`score(d) = Σᵢ 1/(k + rank_i(d))`，k 通常取 60。每路结果用排名的倒数（而非分数）贡献融合分，避免不同系统分数尺度不统一的问题。简单、鲁棒、不需要训练。
