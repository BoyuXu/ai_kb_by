# RewriteGen: Autonomous Query Optimization for Retrieval-Augmented LLMs via Reinforcement Learning
> 来源：工业研究论文 | 日期：20260319

## 问题定义
RAG系统中，原始用户查询往往不是检索的最优形式：过短缺乏上下文、过长含无关信息、歧义性强。Query Rewriting（查询改写）可以显著提升检索质量，但监督学习改写需要大量标注数据，且固定改写策略无法适应动态检索环境。RewriteGen用强化学习自动优化查询改写策略。

## 核心方法与创新点
1. **RL框架设计**：
   - 状态：原始查询 + 检索历史 + 当前答案质量
   - 动作：生成改写查询（LLM作为策略网络）
   - 奖励：最终答案质量（EM/F1）- 检索成本惩罚
2. **奖励塑形**：
   - 稀疏奖励（最终答案正确性）+ 稠密奖励（检索到相关文档）
   - 避免"改写但不改善"的无效动作
3. **PPO训练**：用Proximal Policy Optimization稳定训练，防止策略崩溃
4. **多轮改写**：支持基于当前检索结果迭代改写，逐步收敛到最优查询
5. **对比学习辅助**：正样本（改善检索的改写）与负样本（无效改写）对比，加速学习

## 实验结论
- 在Natural Questions、TriviaQA上，EM比固定改写策略提升4%-7%
- 与监督改写相比，RL版本在分布外查询（新领域）泛化更好，约+3%
- 多轮改写比单次改写再提升2%-3%，但增加1-2轮延迟

## 工程落地要点
- RL训练需要大量rollout，建议在离线环境中先训练好策略再部署
- 推理时控制改写轮数（≤3轮），避免延迟过高
- 奖励函数设计关键：需要快速可计算的代理奖励（不能每次都等LLM生成答案）
- 可用A/B实验对比改写策略的端到端效果（而非仅检索指标）

## 面试考点
**Q: Query Rewriting的常见方法有哪些？**
A: （1）HyDE：LLM生成假设答案，用答案embedding检索；（2）Step-Back Prompting：将问题抽象为更通用问题；（3）Sub-query Decomposition：分解为多个子问题；（4）RL改写（本文）。

**Q: 强化学习在NLP中的主要挑战？**
A: （1）动作空间是离散token，维度极高；（2）奖励稀疏（只有最终答案才有奖励）；（3）生成过程不可微（需要REINFORCE或PPO）；（4）训练不稳定，容易奖励hack。

**Q: RAG中如何平衡检索质量与系统延迟？**
A: （1）预检索：query缓存、意图分类（是否需要检索）；（2）检索优化：向量索引（HNSW）、批量检索；（3）后处理：只传必要上下文（Context Compression）。
