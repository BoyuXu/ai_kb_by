# Single-Turn LLM Reformulation Multi-Stage Hybrid Re-Ranking for Tip-of-the-Tongue Retrieval

> 来源：arXiv | 日期：20260317

## 问题定义

**Tip-of-the-Tongue（ToT）检索**：用户明确知道自己要找的东西，但无法用精确词语表达（"就是那个...那个描述某种感觉的歌曲..."）。这类查询通常是**描述性的、模糊的、非规范化的**，与标准关键词检索有本质差异。

挑战：
1. 查询与目标文档在词汇层面几乎没有重叠（Lexical Gap）
2. 查询意图需要语义推理才能理解
3. 传统 BM25 和密集检索对描述性查询效果差

## 核心方法与创新点

1. **单轮 LLM Query 改写（Reformulation）**
   - 使用 LLM（GPT-4 / Llama-3）对用户原始查询进行改写，生成多个候选查询
   - Prompt 设计：要求 LLM 从不同角度改写（关键词化、同义词扩展、意图明确化）
   - 单轮改写（vs 多轮对话）：更高效，延迟可控，适合在线服务

2. **多阶段混合检索**
   - Stage 1：BM25 召回（基于改写后的关键词查询）
   - Stage 2：Dense Retrieval（基于改写后的语义查询，双塔/ColBERT）
   - Stage 3：Hybrid Fusion（RRF / 加权合并多路候选集）

3. **多阶段重排**
   - Cross-Encoder 重排合并后的候选集
   - LLM-as-Reranker：将候选文档和查询一起送入 LLM，要求输出相关性判断
   - Setwise 重排（vs Pointwise/Listwise）：批量比较候选，更高效

4. **ToT 数据集构建**
   - 利用现有 ToT 数据集（LOTR、TREC iKAT）进行评测
   - 人工标注改写质量与检索相关性的相关性

## 实验结论

- LLM 改写 + 混合检索相比直接检索原始查询：Recall@10 提升约 25%
- Cross-Encoder + LLM Reranker 混合相比单一 Cross-Encoder：MRR@10 提升约 8%
- 关键词化改写对 BM25 提升最大；语义改写对密集检索提升最大

## 工程落地要点

1. **改写延迟**：LLM 改写增加约 500ms~2s 延迟，需评估是否可接受；可用更小模型（7B）改写
2. **改写多样性**：生成 3~5 个改写版本，分别检索后 RRF 融合，多样性提升召回
3. **缓存策略**：相似查询的改写结果可缓存（语义去重）
4. **降级策略**：LLM 不可用时降级到规则改写（同义词词典、词性分析扩展）

## 面试考点

- **Q: Tip-of-the-Tongue 检索的特殊挑战是什么？**
  A: 用户查询与目标文档的词汇重叠极低（无法用关键词匹配），查询是描述性的模糊表达，需要模型具备语义推理能力；同时查询通常较长且包含噪声（无关描述）。

- **Q: Query 改写为什么有效？**
  A: 原始 ToT 查询在词汇和语义空间都偏离了目标文档的表达方式；改写将用户意图"翻译"为检索引擎能有效处理的形式，弥合了查询和文档的表达鸿沟。

- **Q: RRF（Reciprocal Rank Fusion）如何工作？**
  A: $\text{RRF}(d) = \sum_r \frac{1}{k + r(d)}$，其中 $r(d)$ 是文档 $d$ 在检索系统 $r$ 中的排名，$k$ 是平滑常数（通常 60）。RRF 不需要各系统分数归一化，对排名靠前的文档给予更高权重。
