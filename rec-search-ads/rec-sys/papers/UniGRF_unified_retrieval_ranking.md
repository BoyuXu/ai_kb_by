# UniGRF: Unifying Retrieval and Ranking with a Single Generative Recommendation Model

> 来源：https://arxiv.org/abs/2504.16454 | 领域：rec-sys | 学习日期：20260401
> 注：ACM SIGIR 2025 收录论文

## 问题定义

传统推荐系统采用**多阶段管道（Multi-Stage Pipeline）**：召回（Retrieval）→ 粗排 → 精排（Ranking），每个阶段独立优化、串行执行。这种设计存在固有缺陷：

**核心问题：阶段间信息损失（Information Loss Between Stages）**

1. **阶段割裂**：召回模型不知道排序模型的偏好，排序模型无法反馈给召回模型
2. **协作缺失**：召回只优化覆盖率，排序只优化精度，两者目标不对齐
3. **信息瓶颈**：召回阶段过滤掉的物品无法被排序阶段"挽救"，即使它们是高质量结果
4. **计算冗余**：两个模型各自维护用户/物品表示，存在大量重复计算

**启发来源**：自然语言处理中 Seq2Seq 生成模型统一了多个 NLP 子任务（翻译、摘要、问答），同样可以用统一的生成式框架统一推荐管道。

## 核心方法与创新点

本文提出 **UniGRF（Unified Generative Recommendation Framework）**，将召回和排序两个阶段整合为单一生成模型。

### 1. 统一生成式框架

**核心思想**：将召回和排序都建模为**序列生成任务**，共享模型参数和物品表示。

**统一目标：**

$$
P(item | user\_history) = \prod_{t=1}^{L} P(token_t | token_{1:t-1}, user\_history)
$$

- **召回视角**：从整个物品库中生成 TopK 候选（Beam Search 生成多条物品 ID 序列）
- **排序视角**：对给定候选集重新打分排序（利用生成概率作为排序分数）

两个任务共享相同的生成模型，差异只在解码策略。

### 2. 排序驱动增强模块（Ranking-Driven Enhancer）

**关键创新**：引入一个增强模块，让排序阶段的精确信号反向提升召回质量，形成**增强闭环（Enhancement Loop）**：

$$
\mathbf{h}_{recall}^{enhanced} = \mathbf{h}_{recall} + \alpha \cdot \text{Attn}(\mathbf{h}_{recall}, \mathbf{h}_{ranking}, \mathbf{h}_{ranking})
$$

- $\mathbf{h}_{recall}$：召回表示
- $\mathbf{h}_{ranking}$：排序表示（更精准的用户偏好信号）
- 交叉注意力让召回表示"借鉴"排序阶段的精细偏好理解

### 3. 梯度引导自适应权重（Gradient-Guided Adaptive Weighter）

多任务学习中的关键挑战：召回和排序任务的梯度可能存在冲突，直接联合优化可能使一个任务的提升损害另一个任务。

**解决方案**：设计梯度自适应权重机制，动态平衡两个任务的优化：

$$
\mathcal{L}_{total} = w_1(t) \cdot \mathcal{L}_{recall} + w_2(t) \cdot \mathcal{L}_{ranking}
$$

$$
w_i(t) = \frac{\exp(-g_i(t) / T_{task})}{\sum_j \exp(-g_j(t) / T_{task})}
$$

其中 $g_i(t)$ 是任务 $i$ 在时刻 $t$ 的梯度范数，梯度小的任务（已相对收敛）权重降低，梯度大的任务（欠拟合）权重提升。这确保两个任务**同步优化（Synchronized Optimization）**。

### 4. 模型无关性（Model-Agnostic）

UniGRF 是一个框架，可以基于任何生成式推荐模型（如 TIGER、P5、LC-Rec）实现：
- 不依赖特定的物品 ID 编码方式
- 不依赖特定的序列编码器架构
- 增强模块和自适应权重可以即插即用

## 实验结论

在多个标准 benchmark 数据集上与 SOTA 方法对比：

**与单阶段生成式推荐方法对比（只做召回）：**

| 方法 | Recall@10 | NDCG@10 |
|------|-----------|---------|
| TIGER | baseline | baseline |
| LC-Rec | +3.2% | +2.8% |
| **UniGRF** | **+8.5%** | **+7.3%** |

**与传统两阶段方法对比：**

| 方法 | Recall@10 | MRR@10 |
|------|-----------|-------|
| DSSM + DCN | baseline | baseline |
| MDSR（多任务联合）| +4.1% | +3.5% |
| **UniGRF** | **+12.3%** | **+10.8%** |

**消融实验发现：**
- 排序驱动增强模块贡献约 35% 的总提升
- 梯度自适应权重贡献约 25% 的总提升
- 统一生成框架本身（vs 分别训练两个生成模型）贡献约 40%

**关键结论：**
- 统一框架显著优于分阶段独立优化
- 排序→召回的增强闭环是核心贡献
- 同步优化（梯度自适应权重）避免了常见的多任务学习梯度冲突问题

## 工程落地要点

### 系统架构对比

**传统架构：**
```
用户特征 → 召回模型（DSSM）→ 200候选 → 排序模型（DeepFM）→ Top20 → 展示
  ↑独立训练                               ↑独立训练
```

**UniGRF 架构：**
```
用户特征 → UniGRF（共享生成模型）
              ↓ 召回任务（Beam Search）    ↓ 排序任务（生成概率打分）
           200候选 ←──增强闭环──→ Top20 → 展示
  ↑联合训练，梯度自适应权重
```

### 关键工程挑战

1. **Beam Search 效率**：生成式召回用 Beam Search，需维护 Trie 树索引，物品库大（>100M）时存在效率瓶颈
2. **增量更新**：物品库动态更新（新品上架、老品下架）时，需增量更新 SID 和 Trie 索引
3. **排序延迟**：生成式排序通过计算生成概率来打分，比传统排序模型（一次前向）慢，需要优化
4. **冷启动物品**：新物品的 SID 需实时生成，可能影响在线服务

### 适用规模

UniGRF 当前最适合：
- 物品库规模 <10M（Beam Search 可接受）
- 有足够训练数据支撑生成式模型（>1亿条交互）
- 延迟容忍度较高的场景（视频/音乐推荐 vs 搜索推荐）

## 常见考点

**Q1: 为什么传统推荐系统不直接用一个模型做召回+排序，而要分两个阶段？**
A: 主要是**效率**原因。召回面对千万级物品库，必须用极快的向量检索（ANN）；排序只处理几百个候选，可以用复杂模型做精细打分。一个统一模型同时满足两个阶段的要求在计算上难以实现——生成式方法的 Beam Search 在大规模物品库上开销极大。UniGRF 的创新在于通过共享表示和增强闭环降低这一代价。

**Q2: UniGRF 的排序驱动增强模块是如何实现"闭环"的？**
A: 排序阶段有更精细的特征（更多特征交叉、更多上下文信息），能更准确地捕捉用户对特定候选的偏好。增强模块通过交叉注意力将排序的精确表示注入召回表示，相当于"用排序的眼光去改善召回的判断"。这是一个软知识蒸馏，不需要额外标签，梯度可以反向传播。

**Q3: 多任务学习中梯度冲突（Gradient Conflict）是什么，如何解决？**
A: 梯度冲突指不同任务的损失梯度方向相反，同时优化会让两个任务互相干扰。解决方法：(1) 梯度归一化（GradNorm）——平衡梯度范数；(2) PCGrad——将冲突任务的梯度投影到不冲突方向；(3) UniGRF 的自适应权重——动态降低已收敛任务的权重，避免其梯度污染欠拟合任务。

**Q4: UniGRF 和 P5（Pretrain, Personalized, Prompt, Predict and Transfer）有何关系？**
A: P5 是将多种推荐任务（评分、解释、序列推荐等）统一为语言模型的 Prompt 格式，用 T5 做 Seq2Seq 生成。UniGRF 更专注于召回和排序这两个核心工业任务，并设计了专门的增强闭环和梯度平衡机制。两者都是"统一生成框架"的代表，但 UniGRF 更接近工业落地。

**Q5: 请解释"同步优化（Synchronized Optimization）"在 UniGRF 中的重要性。**
A: 在联合训练召回和排序时，如果一个任务先收敛而另一个还在快速更新，先收敛任务的梯度会对未收敛任务产生干扰（类似一个人学习快了拖慢整个团队）。同步优化通过自适应权重确保两个任务以相近的速度进步，避免"一快一慢"导致的性能天花板降低。实验表明不做同步优化时 UniGRF 仍优于 baseline，但比做了同步优化差约 25%。
