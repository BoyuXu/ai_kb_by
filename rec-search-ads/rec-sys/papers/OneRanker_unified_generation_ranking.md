# OneRanker: Unified Generation and Ranking for Recommendation
> 来源：工业论文 | 领域：推荐系统 | 学习日期：20260327

## 问题定义
多阶段推荐系统中**Stage Gap**是核心痛点：
- 召回阶段优化Recall，精排阶段优化AUC，两者目标不对齐
- 生成（召回）和排序（精排）使用完全不同的模型架构和训练范式
- 候选集截断导致信息损失，排序模型永远看不到被召回截掉的"潜在好物"
**目标**：设计统一架构同时完成生成（召回）和排序（精排），消除Stage Gap。

## 核心方法与创新点

### 1. 统一生成-排序框架
单一模型同时输出：
- **生成分支**：自回归生成候选item序列（无需传统ANN检索）
- **排序分支**：对给定候选集输出精确排序分数

共享Transformer主干，两个任务协同训练。

### 2. List-wise生成
不同于传统point-wise或pair-wise，直接生成完整推荐列表：

$$
P(r_1, r_2, ..., r_k) = \prod_{i=1}^{k} P(r_i | r_1, ..., r_{i-1}, u, ctx)
$$

List-wise视角能考虑item之间的多样性和互补性。

### 3. 统一损失函数

$$
\mathcal{L} = \lambda_1 \mathcal{L}_{gen} + \lambda_2 \mathcal{L}_{rank} + \lambda_3 \mathcal{L}_{align}
$$

- $\mathcal{L}_{gen}$：自回归语言模型损失（交叉熵）
- $\mathcal{L}_{rank}$：排序损失（ListNet/LambdaRank）
- $\mathcal{L}_{align}$：生成分支和排序分支的表示对齐损失

### 4. 候选集注入机制
排序时将候选集作为prompt条件注入：
```
[User History] + [Candidate Set] → [Ranked List]
```
允许模型在有候选集时做精排，无候选集时做生成式召回。

## 实验结论
- **AUC提升**：+0.8%（精排阶段，相比独立精排模型）
- **GMV提升**：+2.3%（在线A/B测试）
- **Recall提升**：+5%（召回阶段，相比独立双塔模型）
- 消融实验证明生成任务对排序任务有显著正迁移，反之亦然

## 工程落地要点
1. **训练时间**：统一模型训练时间约为独立模型1.5x，但减少了两个模型的维护成本
2. **在线serving架构**：需要支持两种推理模式（生成/排序）的统一服务
3. **候选集大小**：排序模式下候选集作为context，需要限制大小（通常≤200），否则延迟过高
4. **损失权重调优**：λ参数对最终效果影响显著，需要网格搜索
5. **渐进式迁移**：建议先用排序任务初始化，再加入生成任务微调

## 面试考点
Q1: OneRanker如何消除Stage Gap？
A: 通过共享Transformer主干统一生成和排序两个任务，训练信号相互流通。生成任务让模型了解全局item分布，排序任务提供精细的相对偏序监督。两者共享底层用户和item表示，从根本上对齐召回和精排的优化目标。

Q2: List-wise生成相比point-wise排序的优势？
A: (1)多样性：自回归生成天然避免重复，考虑列表内item的互补性；(2)位置偏差：生成顺序隐式编码了展示位置的重要性；(3)上下文感知：第i个推荐结果会考虑前i-1个已推荐item；缺点是推理复杂度高O(k·n)。

Q3: 如何处理训练时生成分支和排序分支的梯度冲突？
A: (1)梯度归一化：对不同任务的梯度按模长归一化；(2)PCGrad：投影冲突梯度到不冲突方向；(3)任务优先级调度：按损失下降速度动态调整学习率；(4)分离底层-共享高层：底层特征提取共享，高层任务头独立。
