# 生成式推荐系统技术全景（2026-03-30）

> 领域：rec-sys | 类型：综合综述 | 覆盖论文：10篇

## 一、技术演进脉络

```
传统推荐 (2015-2020)          生成式推荐 (2022-2026)
─────────────────────────    ──────────────────────────────────
双塔召回 + DIN 精排           统一 Decoder-Only 生成式推荐
独立多任务 (MMOE/PLE)         多任务基础模型 (MTFM, GNOLR)  
固定规则重排                  LLM 推理重排 (GR2, LLM4Rerank)
ID 嵌入为主                   语义 ID + LLM 语义理解 (URM)
```

## 二、核心技术维度

### 2.1 生成式统一架构

**代表工作**：OneRec、UniGRF、OneRec-Think、HSTU（Meta）

核心思路：用 Decoder-Only 自回归模型将召回+排序统一为序列生成任务：

$$P(i_1, i_2, ..., i_K | u) = \prod_{k=1}^K P(i_k | u, i_1, ..., i_{k-1})$$

**物品 ID 离散化方案**：
| 方法 | 原理 | 代表工作 |
|------|------|----------|
| RQ-VAE | 残差向量量化，层次化码本 | UniGRF, OneRec |
| Semantic ID | 聚类后分配语义码 | MFLI |
| Dense ID | 连续 embedding 直接生成 | HSTU |

**工程挑战**：
- Beam Search 候选空间剪枝（物品>千万）
- 生成延迟（~150ms）需 Speculative Decoding 优化
- 新物品实时入库（Semantic ID 增量扩展）

### 2.2 推理增强推荐

**代表工作**：ReasonRec、OneRec-Think、GR2

推理链（Chain-of-Thought）引入推荐的两种路径：

1. **训练时蒸馏**：大 LLM 生成推理链作为 SFT 数据
2. **RL 强化**：以 NDCG/Hit Rate 为 reward，GRPO 训练推理质量

```
用户历史 → [think]分析意图[/think] → 推荐输出
```

**GRPO vs PPO**：
$$\hat{r}_i = \frac{r_i - \bar{r}}{\text{std}(r)}, \quad \text{GRPO 无需 critic 网络}$$

### 2.3 多任务学习创新

**代表工作**：MTFM、GNOLR、DeepMTL2R

**GNOLR 有序多任务**（关键创新）：
$$P(Y > k | x) = \sigma(f_\theta(x) - b_k), \quad b_1 < b_2 < ... < b_K$$

将点击→购买→复购的有序行为链建模为 Ordinal 问题，共享打分函数。

**MTFM Task Prompt**：无需手工 alignment，task token 自动路由专家。

**梯度平衡策略对比**：
| 策略 | 场景 | 成本 |
|------|------|------|
| GradNorm | 梯度量级差异大 | 中 |
| PCGrad | 梯度方向冲突 | 低 |
| UncertaintyWeighting | 任务不确定性高 | 低 |

### 2.4 Scaling Laws

**代表工作**：Kunlun、HSTU

$$\mathcal{L}(N_e, N_d) = A \cdot N_e^{-\alpha} + B \cdot N_d^{-\beta} + C$$

**关键发现**：
- Embedding scaling（$\alpha \approx 0.3$）> DNN scaling（$\beta \approx 0.2$）
- Compute-optimal：70% 算力用于 embedding，30% 用于 DNN
- 数据量翻倍：稳定 3% loss 下降

## 三、📐 关键数学公式

**1. Listwise 排序 ListMLE**：
$$\mathcal{L}_\text{ListMLE} = -\sum_{k=1}^K \log \frac{e^{s_{\pi(k)}}}{\sum_{j=k}^K e^{s_{\pi(j)}}}$$

**2. GNOLR 多任务 Loss**：
$$\mathcal{L} = -\sum_{t=1}^T \sum_i \log P(y_i^{(t)} | x_i; \theta, b^{(t)})$$

**3. Recommendation Scaling Law**：
$$\mathcal{L}(N) = A \cdot N^{-\alpha} + C, \quad N = N_\text{embed} + c \cdot N_\text{DNN}$$

## 四、🎓 面试 Q&A（10题）

**Q1**: 生成式推荐和传统 CTR 排序的本质区别？
> 传统 CTR：逐物品 pointwise 打分，互相独立；生成式：autoregressive 生成有序序列，天然建模物品间依赖，支持长程规划

**Q2**: HSTU 为什么用 SSM 而非 Transformer？
> O(n²) → O(n) 复杂度，支持数万步历史；SSM 递推 $h_t = Ah_{t-1} + Bx_t$ 在推理时 O(1) 更新，完美适配实时推荐

**Q3**: Semantic ID 的构建原理？
> RQ-VAE：将 item embedding 残差量化为多层 codebook token，相似 item 共享前缀，生成式模型可以泛化到新 item

**Q4**: MMOE 和 PLE 的核心差异？
> MMOE：所有任务共享专家池，各任务门控软选择；PLE：增加 task-specific 专家，减少任务间干扰，适合任务差异大的场景

**Q5**: 多任务学习中负迁移的成因和解法？
> 成因：梯度方向冲突，任务 A 的更新损害任务 B；解法：PCGrad（投影消除冲突分量）、MMOE（软路由隔离）、独立专家

**Q6**: 推荐系统中 Scaling Law 的实践意义？
> 指导 compute-optimal 资源分配：embedding >> DNN，避免盲目堆 DNN 层数；预测模型扩展后的性能，降低试验成本

**Q7**: GR2（生成式重排）如何保证输出排列的合法性？
> Constrained Beam Search：decode 时用 prefix tree/token mask，只允许未使用候选 ID 出现，强保证 no repeat + no miss

**Q8**: MTFM 中 Task Prompt 如何自动路由？
> Task token 通过 self-attention 与物品/用户 token 交互，attention weight 决定路由；不同 task token 激活不同特征路径

**Q9**: OneRec-Think 用 GRPO 的 reward 如何设计？
> 主 reward：Hit@k（推荐命中率）+ NDCG@k；辅助 reward：推理链格式合法性、推理一致性（不自相矛盾）

**Q10**: 工业推荐系统从双阶段迁移到生成式的最大挑战？
> ① 延迟：Beam Search 慢于 ANN 检索（~10x）；② 实时物品：新 item 的 Semantic ID 分配；③ 可解释性监控：生成内容难以审查

## 🎯 核心洞察

1. **统一即正道**：OneRec/UniGRF/HSTU 都在往"召回+排序一体化"走，误差传播是工业界迁移的首要动机——不是追求论文 novelty，是真实系统痛点。

2. **CoT 进推荐不是噱头**：OneRec-Think + GR2 证明，推理链能让模型在"生成前思考"，本质是把用户意图建模从隐式（embedding）变显式（文本），可解释性+效果双赢。GRPO 作为训练信号是关键——离散奖励（命中率）的 RL 优化，无需 critic 网络。

3. **Embedding > DNN 的 Scaling 启示**：Kunlun 告诉我们，推荐系统的"大脑"是 embedding table 而非 dense 网络。这和 LLM（transformer 权重为主）截然不同，工程选择应 70% 算力给 embedding。

4. **有序多任务是被低估的创新**：GNOLR 把"点击→购买→复购"的有序性显式建模，比独立二分类头更鲁棒。工业实践中标签天然有序的场景（满意度 1-5 星、停留时长分级）都可以套用。

5. **基础模型范式正在重塑多任务**：MTFM 的 task token 路由方案比 MMOE 门控更优雅，关键创新是 continual task addition（O(1) 接入新任务）——这才是工业落地的核心价值，不是精度。

6. **LLM→推荐的蒸馏路径已成熟**：URM 大模型训练+小模型蒸馏的方案将 LLM 推理时延从 >100ms 降到 <10ms，打通了"语义理解能力"和"在线部署要求"之间的鸿沟。

7. **SSM 替代 Transformer 不是银弹**：HSTU 用 Mamba-style SSM 解决超长序列问题，但牺牲了并行训练效率；对于序列长度适中（<1000步）的场景，标准 Transformer 仍是首选。
