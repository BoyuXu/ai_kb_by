```

────────────────────────────      ────────────────────────────        ────────────────────────────

DIN/DIEN 序列建模                 LLM 语义增强（TagCF）               因果语言模型召回（LinkedIn CLM）
```

## 二、核心技术维度

### 2.1 生成式统一架构

**代表工作**：OneRec、HSTU（Meta）、MTGR（美团）、iRanker

核心思路：用 Decoder-Only 自回归模型将召回+排序统一为序列生成任务：

$$
P(i_1, i_2, ..., i_K | u) = \prod_{k=1}^K P(i_k | u, i_1, ..., i_{k-1})
$$

**物品 ID 离散化方案对比**：
| 方法 | 原理 | 优点 | 代表工作 |
|------|------|------|----------|
| RQ-VAE 语义 ID | 残差向量量化，层次码本 | 语义感知，压缩高效 | UniGRF, OneRec |
| Multi-faceted ID | 多角度索引（品类/价格/风格） | 多维度可检索 | MFLI |
| Dense Embedding | 连续 embedding 直接生成 | 低延迟 | HSTU |
| Causal LM token | 序列自回归预测 | 时序因果建模 | LinkedIn CLM |

**工程挑战**：
- Beam Search 候选空间剪枝（item 池 >千万）
- 生成延迟（~150ms）需 Speculative Decoding 优化
- 新物品实时入库（Semantic ID 增量扩展）

### 2.2 生成式召回：从双塔到可学习索引

**技术演进路径**：
```
随机初始化 → 用户塔/物品塔（双塔）→ ANN 索引（HNSW/IVF）→ 检索
                ↓ 问题：模型与索引目标不一致
可学习索引（MFLI）：端到端联合优化模型 + 索引
因果语言模型（LinkedIn CLM）：自回归建模下一次交互
```

#### MFLI：多面可学习索引

传统 ANN 检索的"目标不一致"问题：模型优化相关性，索引优化向量距离，两者不完全对齐。

**MFLI 联合优化损失**：

$$
L = L_{rec} + \lambda_1 L_{index} + \lambda_2 L_{contrast}
$$

**多面索引（Multi-faceted）核心**：
- 同一个物品拥有多个 facet 编码（品类、价格、风格、使用场景）
- 查询时从不同 facet 维度检索，提升召回覆盖度
- facet 编码在训练时联合学习，与推荐目标对齐

#### LinkedIn 因果语言模型召回

从双塔范式到自回归范式的演进：

$$
P(v_{t+1} | v_1, ..., v_t) = \text{CLM}(v_1, ..., v_t)
$$

**与双塔对比**：
| 维度 | 双塔模型 | CLM 召回 |
|------|----------|----------|
| 序列建模 | 池化/弱序列 | 完整时序因果 |
| 扩展性 | 新物品离线扩展 | 增量解码 |
| 延迟 | 向量相似度 O(1) | 自回归 O(n) |
| 表达能力 | 用户-物品点积 | 全序列依赖 |
| 适用场景 | 大规模召回 | 精准序列推荐 |

**工程解决**：离线预计算物品 token 表示 + 前缀缓存，降低在线延迟。

### 2.3 LLM × 协同过滤融合

**代表工作**：CoLLM、TagCF

**核心问题**：LLM 具备语义理解能力但缺乏协同过滤信号；CF 有协同信号但缺语义。

#### CoLLM：CF Embedding 注入 LLM

**映射网络架构**：

$$
h_{CF \to LLM} = \text{MLP}(e_{CF}), \quad e_{CF} \in \mathbb{R}^{d_{CF}}, \quad h \in \mathbb{R}^{d_{LLM}}
$$

注入位置：在 LLM 的 input token 序列中插入 CF 信号（soft token）：

$$
\text{Input} = [u_\text{text}, u_\text{CF}, i_1^{CF}, i_1^{text}, ..., i_t^{CF}, i_t^{text}]
$$

训练策略：先冻结 LLM，训练映射网络（对齐阶段）；再全参数微调（融合阶段）。

#### TagCF：标签作为 LLM-CF 桥接

LLM 生成结构化标签 → 标签空间作为 LLM 语义和 CF 协同的共同表示：

$$
e_{item} = f_{CF}(\text{Tags}(item)) + \alpha \cdot f_{LLM}(\text{Tags}(item))
$$

优势：标签是可解释的中间表示，可人工审核，避免黑盒融合。

### 2.4 多任务学习创新

**代表工作**：GNOLR、Multi-task MoE

**技术演进**：
```
独立模型 → MMOE（专家混合）→ PLE（任务专属+共享专家）→ GNOLR（有序逻辑约束）
```

#### GNOLR：有序逻辑回归统一多任务

推荐中的任务存在自然有序关系：曝光 → 点击 → 加购 → 购买，每层概率递减。

**GNOLR 有序约束**：

$$
P(Y \geq k | x) = \sigma(f_\theta(x) - b_k), \quad b_1 \leq b_2 \leq ... \leq b_K
$$

嵌套（Nested）结构：高层任务的正样本隐含低层任务的正样本，减少矛盾样本。

**与 ESMM 对比**：
| 维度 | ESMM | GNOLR |
|------|------|-------|
| 一致性保证 | 乘法链（pCTCVR = pCTR × pCVR） | 有序约束（阈值递增） |
| 梯度传播 | 路径长（乘法链梯度消失风险） | 独立优化更稳定 |
| 任务数量 | 适合 2-3 任务 | 支持任意多层有序任务 |

#### MoE 门控网络

$$
y_k = \sum_{i=1}^{N} g_k^{(i)} \cdot f_i(x), \quad g_k = \text{Softmax}(W_k \cdot x)
$$

负迁移机制：门控让不同任务选择性使用不同专家子集，减少参数共享冲突。

### 2.5 Graph-Mamba：图序列混合建模

**背景**：推荐中的用户-物品图既有局部邻居关系，又有长程依赖。Graph Transformer 是 $O(n^2)$，Mamba 是 $O(n)$。

**Graph-Mamba 门控融合**：

$$
h_t = \sigma(W_g) \odot h_\text{mamba} + (1 - \sigma(W_g)) \odot h_\text{local}
$$

- $h_\text{mamba}$：Mamba 捕捉的长程图序列依赖
- $h_\text{local}$：GNN 捕捉的局部邻居聚合

**工业价值**：用户行为序列 >1000 时，Mamba 线性复杂度让长序列建模可行。

### 2.6 排序基础模型：iRanker

**核心理念**：类比 NLP 的 BERT→fine-tune，推荐排序也可以有预训练基础模型。

**iRanker 架构**：
```
预训练（多场景联合）：
  多领域推荐日志 → Universal Ranking Transformer → 通用排序能力

微调（场景适配）：
  场景 Prompt + 少量标注 → 场景专属排序器（快速适配）
```

**Zero-shot 推荐**：新场景通过自然语言 Prompt 描述业务目标，iRanker 无需标注数据直接激活对应能力。

### 2.7 DynamicRAG：推荐重排中的迭代检索

**问题**：传统 RAG 检索是一次性的，检索质量直接决定最终结果。

**DynamicRAG 迭代机制**：

$$
a_t = \text{LLM}(q, D_t, \text{History}}_{\text{t), \quad a}}_{\text{t \in \{\text{refine}}, \text{finalize}\}
$$

- 评估当前检索结果质量
- 若不满意：生成改进的检索 query，重新检索
- 若满意：进入重排/生成阶段
- 终止条件：最大轮数 OR 置信度超阈值

## 四、🎯 核心洞察

1. **从检索到推荐：统一序列生成是大方向，但工程代价在于延迟**。双塔 O(1) 的近似检索难以直接用自回归替代，MFLI 和 CLM 召回分别代表"改进索引"和"改变范式"两条路——前者工程可行性高，后者表达能力强。

2. **LLM × CF 融合的核心矛盾：语义 vs 协同信号**。CoLLM 选择注入（CF→LLM），TagCF 选择桥接（标签作为中间语言）。两者都揭示了同一问题：LLM 的语义理解 ≠ 推荐的协同信号，强行融合会互相稀释，需要显式对齐机制。

3. **GNOLR 的 insight：推荐目标的有序性约束是隐含先验，不显式建模就会浪费**。曝光 → 点击 → 购买的单调性约束（$b_1 \leq b_2 \leq b_K$）把领域知识嵌入模型结构，比单纯多任务 Loss 加权更系统。

4. **iRanker 的价值不是精度，而是快速上线**。新场景从"收集数据→训练模型→上线"缩短为"写 Prompt→微调→上线"，把推荐系统的上线周期从数周压缩到数天。

5. **可学习索引 vs 传统 ANN 的真正差异是优化目标对齐**。传统 HNSW 优化向量近邻精度，MFLI 优化推荐 NDCG。这一差异在长尾物品、多义性查询等场景中差距显著，但增加约 30% 训练时间是代价。

6. **Graph-Mamba 解决的是可扩展性问题而非精度问题**。在序列长度 <200 时，Graph Transformer 精度更高；>1000 时，Mamba 在精度相近的情况下延迟低 3-5 倍。工业场景选型看序列长度分布。

7. **DynamicRAG 的迭代是有代价的**：每轮迭代增加一次检索延迟。实际工业部署中，平均 2.3 轮收敛，延迟增加约 80%，适用于离线重排或容忍高延迟的精排场景。

## 五、🎓 面试 Q&A（15题）

**Q1**: 生成式推荐与传统向量检索的本质区别？
> 传统：编码→ANN 索引→最近邻搜索；生成式：编码→自回归解码 item ID。前者依赖固定索引结构，后者直接生成目标序列。MFLI 的可学习索引是折中方案：保留向量检索的效率，同时让索引对齐推荐目标。

**Q2**: MoE 解决负迁移的核心原理？
> 门控网络让不同任务选择性使用不同专家子集，相当于为每个任务学习定制化的网络子集，减少任务间参数共享的梯度冲突。关键是专家路由的稀疏性（Top-K 路由），强制任务分离。

**Q3**: 为什么 Graph-Mamba 比 Graph Transformer 更适合大规模推荐？
> 推荐场景中用户行为序列动辄数千，Graph Transformer 的 $O(n^2)$ 注意力在这个规模下内存和计算不可承受。Mamba 的 $O(n)$ 线性复杂度让长序列可扩展，门控融合保留了局部邻居的精准捕捉。

**Q4**: CoLLM 中为什么需要映射网络？
> CF Embedding（协同过滤学到的用户/物品向量）和 LLM token Embedding 在维度（$d_{CF}$ vs $d_{LLM}$）和语义空间上都不匹配，映射网络（MLP/线性变换）实现跨空间的维度对齐和语义对齐。

**Q5**: GNOLR 相比 ESMM 的优势？
> ESMM 通过乘法链（pCTCVR = pCTR × pCVR）保证概率一致性，但梯度传播路径长，多任务时容易梯度消失。GNOLR 用有序约束，每层独立优化，阈值参数保证单调性，训练更稳定，支持更多层任务。

**Q6**: iRanker 的 Zero-shot 推荐如何实现？
> 预训练阶段在多场景数据上学习通用排序能力（相关性/点击率/转化率的判断逻辑），新场景通过自然语言 Prompt 描述业务目标，激活预训练中习得的对应能力，无需标注数据。

**Q7**: 可学习索引相比传统索引增加多少训练开销？
> 约 30% 训练时间（需要索引重建 Loss 的反向传播 + 周期性索引重建），但离线训练完全可接受。在线检索阶段与传统 ANN 相同，无延迟增加。

**Q8**: DynamicRAG 的迭代机制会不会导致死循环？
> 设置了双重终止条件：最大迭代轮数（通常 3-5 轮）+ 置信度阈值（当前检索结果质量评分超阈值时提前终止）。实际平均 2.3 轮收敛，工程上安全。

**Q9**: LinkedIn 为什么选择因果语言模型而非双塔？
> 推荐是预测"下一次"交互的任务，天然契合自回归（因果）建模——给定历史序列，预测下一个最可能交互的 item。双塔的点积相似度无法建模序列间的时序因果依赖（用户行为顺序影响偏好）。

**Q10**: MFLI 的多面索引中"facet"具体指什么？
> 同一个商品可以从多个维度被检索：品类（手机壳）、价格区间（50-100元）、风格（简约）、使用场景（户外）。每个维度对应一个 facet 编码，查询时从最匹配的 facet 角度检索，提升长尾物品的召回覆盖。

**Q11**: TagCF 的标签桥接方案有什么工程优势？
> ① 可解释性：标签是人类可理解的中间表示，可人工审核；② 降噪：LLM 生成结构化标签比原始文本更稳定；③ 轻量：不需要把 CF Embedding 注入 LLM，计算开销小；④ 解耦：LLM 和 CF 可独立更新。

**Q12**: 生成式推荐中 Speculative Decoding 如何应用？
> 用小型草稿模型（draft model）快速生成候选 token 序列，大型主模型并行验证和修正，接受率 60-80% 时可获得 2-3x 加速。推荐中 draft model 可以是小双塔或轻量 CLM。

**Q13**: MoE 路由中如何避免专家负载不均？
> 辅助 Loss（Auxiliary Balance Loss）惩罚专家利用率的不均衡：$L_{balance} = \alpha \cdot N \cdot \sum_i f_i \cdot P_i$，其中 $f_i$ 是实际路由比例，$P_i$ 是期望路由概率。

**Q14**: 有序多任务（GNOLR）的阈值参数如何初始化？
> 按实际数据中各级转化率的比例初始化：$b_k = -\log(\frac{P_k}{1-P_k})$（logit 变换），保证初始阶段已满足单调约束，避免早期训练不稳定。

**Q15**: Graph-Mamba 和纯 Mamba 推荐模型的区别？
> 纯 Mamba 把行为序列线性化后建模，忽略了 user-item 的图结构（协同过滤信号）。Graph-Mamba 用 GNN 先捕捉图邻居聚合，再用 Mamba 做长序列全局建模，门控融合两者，既有图拓扑信息又有长程依赖建模。
