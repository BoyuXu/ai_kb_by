# Towards Web-scale Recommendations with LLMs: From Quality-aware Ranking to Candidate Generation
> 来源：https://dl.acm.org/doi/10.1145/3690624.3709413 | 日期：20260320 | 领域：rec-sys

## 问题定义

Explore Further @ Bing 是一个 webpage-to-webpage 推荐产品，通过在搜索结果页面提供相关的网页推荐来增强搜索体验。微软Bing团队面临的核心挑战：

1. **质量评估困难**：如何量化推荐内容的质量？传统点击率指标无法全面反映推荐质量
2. **多目标平衡**：需要同时优化推荐质量和点击率等多个目标
3. **候选生成局限**：传统召回方法难以发现互补性强、新颖度高的候选
4. **Web规模扩展**：方法需要在亿级网页规模下高效运行

## 核心方法与创新点

### 1. RecoDCG - LLM驱动的推荐质量评估指标
- **设计目标**：开发一个由LLM驱动的推荐质量评估指标
- **核心思想**：利用大语言模型的语义理解能力评估推荐结果的相关性和质量
- **作用**：作为离线评估和在线优化的统一质量度量

### 2. Quality-aware Ranking（质量感知排序）
- 利用LLM使排序阶段具备质量感知能力
- 核心技术包括：
  - LLM-based特征提取
  - 质量感知的重排序机制
- 目标：在保证相关性的同时提升推荐质量

### 3. Q' Recall - LLM增强的候选生成
- **创新点**：引入LLM生成互补性强、新颖度高的推荐候选
- **技术特点**：
  - 利用LLM的语义理解和生成能力
  - 产生传统召回路径难以发现的优质候选
  - 增强推荐系统的多样性和惊喜度

### 4. 多目标优化框架
- 同时优化推荐质量和点击率
- 平衡策略：
  - 质量感知排序确保推荐相关性
  - 点击预估模型保证用户参与度

## 工程落地要点

1. **LLM作为评估器**：RecoDCG展示了如何利用LLM构建新的评估指标，替代传统人工标注
2. **多阶段融合**：Q' Recall作为新的召回路径与现有召回（CoClick、CF、Dense Retrieval等）并行
3. **质量-点击平衡**：实践中需要精细调节质量优化与点击优化的权重
4. **推理效率**：LLM在排序和召回阶段的应用需要考虑计算成本，可能需要蒸馏或轻量级模型
5. **迭代验证流程**：离线RecoDCG评估 → 在线A/B测试 → 全量部署的完整流程

## 常见考点

**Q1：RecoDCG与传统DCG有什么区别？为什么需要引入LLM？**
A：传统DCG基于人工标注的相关性等级计算，成本高且难以扩展。RecoDCG利用LLM的语义理解能力自动评估推荐质量，优势包括：(1) 无需人工标注，可大规模扩展；(2) 利用LLM的世界知识评估语义相关性；(3) 可捕捉更深层次的语义关联。挑战在于LLM评估的可解释性和与人类判断的一致性。

**Q2：Q' Recall与传统召回方法（如CF、向量检索）有什么本质区别？**
A：传统召回方法基于历史行为或语义相似度，容易产生同质化推荐。Q' Recall利用LLM的生成能力主动"创造"候选，可以发现：(1) 语义互补但表面不相似的物品；(2) 跨领域的高价值推荐；(3) 具有惊喜度的新颖候选。相当于从"检索已知"扩展到"生成可能"。

**Q3：如何在推荐质量和点击率之间做权衡？**
A：论文采用多目标优化框架，关键策略：(1) 排序阶段引入质量感知信号；(2) 保持点击预估模型作为基础排序因子；(3) 通过A/B测试找到最佳权重组合。实践中发现质量和点击并非完全对立，高质量的推荐往往也能获得不错的点击率。

**Q4：工业界部署LLM-based推荐系统的主要挑战是什么？**
A：主要挑战包括：(1) 推理延迟 - LLM推理慢，需要优化（蒸馏、量化、缓存）；(2) 计算成本 - 大规模调用API成本高昂；(3) 一致性 - LLM输出的不稳定性；(4) 可解释性 - 黑盒模型的调试困难。本文展示了通过架构设计（如Q' Recall作为补充而非替代）可以缓解这些问题。

## 模型架构详解

### 候选编码
- **Item 表示**：Semantic ID（层次化离散编码）或稠密向量 Embedding
- **编码方式**：RQ-VAE（残差量化）/ K-Means 聚类 / 端到端学习的 Token 序列
- **多模态融合**：文本/图片/行为信号的统一表示空间

### 检索机制
- **生成式检索**：自回归解码器逐步生成 Item Token 序列
- **向量检索**：双塔编码 + ANN 索引（HNSW/IVF-PQ）
- **混合召回**：多路检索结果的统一评分与去重

### 训练策略
- **正样本**：用户交互（点击/购买/收藏）
- **负采样**：In-batch Negatives + 难负例挖掘
- **对比学习**：InfoNCE Loss 拉近正样本、推远负样本
- **课程学习**：从简单到困难逐步增加负例难度

## 与相关工作对比

| 维度 | 生成式召回 | 双塔向量召回 | 传统倒排 |
|------|-----------|------------|---------|
| 冷启动 | 好（内容特征） | 中（需行为） | 差 |
| 索引维护 | 无需显式索引 | 需 ANN 索引 | 需倒排表 |
| 推理延迟 | 中（自回归） | 低（一次编码） | 低 |
| 可扩展性 | 亿级 | 亿级 | 百万级 |
| 多模态 | 原生支持 | 需要适配 | 困难 |

## 面试深度追问

- **Q: Semantic ID 的设计思路和优势？**
  A: 将 Item 映射为离散 Token 序列（类似自然语言），使推荐问题转化为序列生成。优势：1) 天然支持自回归生成；2) 层次化结构（粗→细）提升检索效率；3) 避免连续向量的 ANN 近似误差。

- **Q: 生成式召回如何处理新物品？**
  A: 1) 内容特征驱动的 Semantic ID 分配（新物品基于属性分配 Token）；2) 增量学习更新 Codebook；3) 备用的 Content-based 召回通道兜底。

- **Q: 多路召回的融合策略？**
  A: 1) 统一打分：所有通道候选用同一模型重新打分；2) 配额分配：各通道按历史表现分配固定配额；3) 加权融合：考虑通道多样性的加权排序。

- **Q: 如何衡量召回质量？**
  A: 离线：Recall@K, HR@K, NDCG@K。在线：端到端 CTR/GMV 提升 + 召回覆盖率 + 新颖性。注意 K 值要与下游排序的候选集大小匹配。
