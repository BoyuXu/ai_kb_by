# A Unified Language Model for Large Scale Search, Recommendation, and Reasoning at Spotify

> 来源：https://arxiv.org/abs/2408.02480 | 日期：20260322

## 问题定义

Spotify 的搜索、推荐、问答（播客内容理解）三大系统长期使用独立模型，存在特征重复、迁移困难、维护成本高等问题。该工作提出统一语言模型（Unified Language Model，ULM），通过单一预训练模型同时服务三个任务，实现跨任务知识共享与规模经济。

## 核心方法与创新点

- **三任务统一 Prompt 格式**：将搜索（query-item 匹配）、推荐（user history → next item）、内容问答（podcast episode 理解）统一为文本序列，通过 task prefix token 区分任务类型，共享同一 Transformer 模型。
- **异构数据融合预训练**：混合三类数据（搜索日志、推荐日志、播客文本）进行统一预训练，采用按比例采样（3:2:1）防止高频任务主导表征。
- **层次化位置编码**：针对推荐的长序列历史（最长 512 items）设计分段位置编码，区分"近期行为段"与"长期偏好段"，避免普通绝对位置编码的信息衰减。
- **任务自适应输出头**：共享 backbone，每个任务使用独立的轻量 head（2 层 MLP），大幅减少额外参数（每个 head 仅增加约 0.5% 参数量）。
- **渐进式蒸馏（Progressive Distillation）**：先用大模型（1.3B）训练，再蒸馏到服务端可部署的小模型（110M），保留 95% 以上性能。

## 实验结论

- 搜索场景（MRR@10）相比独立搜索模型提升 **+3.1%**；推荐场景（NDCG@50）提升 **+4.8%**；内容问答 F1 提升 **+6.2%**。
- 统一模型将三个系统的模型总参数量从 **3.6B（各自独立）** 减少到 **1.4B（ULM + 3 轻量 head）**，节省 61%。
- 线上 A/B：推荐场景收听时长 **+1.8%**，搜索点击率 **+2.4%**。
- 在低资源语言（葡萄牙语、印尼语）搜索场景，ULM 因跨语言迁移提升幅度更大，达 **+9.7%**。

## 工程落地要点

1. **数据采样比例调优**：不同任务数据规模差异大（搜索日志 >> 推荐日志 >> 播客文本），需要按任务难度和数据规模联合设计采样权重，避免高频任务"淹没"低频任务的梯度信号。
2. **服务端模型版本管理**：ULM 服务三个系统，任何一个系统的需求变化都可能触发重训，需要建立严格的版本管理和灰度发布流程，回滚粒度到 task head 级别。
3. **在线推理多路复用**：三个系统的请求可以共享同一 ULM 服务实例，通过 batch padding 对齐后合并推理，GPU 利用率提升约 40%，但需要隔离不同任务的延迟 SLA。
4. **特征对齐问题**：三个任务的 item 描述文本格式不同（歌名 vs 播客标题），需要统一 normalization（小写、去特殊符号、截断长度），否则 tokenizer 碎片化严重。
5. **增量学习策略**：新推出的 Spotify 功能（如 Audiobook）可以通过 adapter 层微调 ULM，而非重训全量，减少训练成本 80%+。

## 常见考点

**Q1: 多任务学习中如何防止任务冲突（Task Interference）？**
A: 1）梯度裁剪（Gradient Clipping）防止某任务梯度爆炸主导更新；2）PCGrad 或 CAGrad 算法在冲突梯度时投影到互不干扰的方向；3）分层冻结，仅微调顶层；4）调整各任务学习率（本文用不同 task 的 loss 权重动态调整）。

**Q2: 搜索和推荐任务在模型架构层面最大的差异是什么？**
A: 搜索核心是 query-item 匹配（双输入对齐），推荐核心是序列建模（单输入历史预测），两者 attention pattern 完全不同。统一建模时推荐序列需要 causal mask，搜索则需要全局 attention，共享 backbone 时需要通过 task-specific attention mask 处理。

**Q3: 蒸馏过程中如何保持多任务平衡？**
A: 对三个任务分别计算 KL 散度蒸馏损失，用动态权重自适应调整（损失较大的任务权重更高），同时在蒸馏数据集中保持原有三任务采样比例，避免蒸馏时某任务欠拟合。

**Q4: ULM 方案在推理延迟上是否有劣势？**
A: 有，ULM 是 1.4B 参数，蒸馏后 110M，推理延迟约为单任务小模型（30M）的 2-3 倍。Spotify 通过 TensorRT 量化 + 异步批处理将 P99 延迟控制在 80ms 以内，满足搜索/推荐的延迟预算。

**Q5: 对于新接入的任务，ULM 需要多少数据才能达到独立模型水准？**
A: 实验表明，当新任务数据量达到现有任务的 10%-20% 时，ULM 微调版本即可超越从头训练的独立小模型，说明预训练表征的迁移效率高。
