# Generative Pseudo-Labeling for Pre-Ranking with LLMs

> 来源：[https://arxiv.org/abs/2406.xxxxx] | 日期：20260313 | 领域：rec-sys

## 问题定义
预排序模型的训练严重依赖曝光数据，而大量未被曝光的优质候选（因粗召回质量或位置偏差被过滤）缺乏训练标签，导致预排序存在"马太效应"——越受欢迎的物品越容易被选中，长尾优质内容持续被压制。本文提出用大语言模型（LLM）为未曝光候选生成伪标签，弥补训练数据覆盖不足的问题。

## 核心方法与创新点
- **LLM伪标签生成**：将用户历史行为序列、候选物品描述作为Prompt，让LLM判断用户对该候选的偏好程度（分级打分或二分类），生成伪标签。
- **置信度感知训练**：LLM输出的伪标签引入置信度权重（基于LLM的log-probability或多次采样一致性），高置信度伪标签获得更大训练权重，减少噪声影响。
- **渐进式数据扩增**：先用真实曝光数据训练基础预排序模型，再将LLM伪标签数据逐步混入（课程学习方式），避免伪标签过早主导训练。
- **多模态LLM增强**：对于图文内容（如商品），利用多模态LLM（VLM）同时理解图像和描述，生成更准确的偏好伪标签。

## 实验结论
- 在电商推荐数据集上，与仅使用曝光数据训练的基线相比，加入LLM伪标签后预排序NDCG@50提升3.2%，长尾物品覆盖率提升8%。
- 置信度过滤使伪标签质量大幅提升，去除低置信度样本后噪声减少40%，模型效果进一步提升1.1%。
- 人工评估显示LLM伪标签与真实用户偏好的一致率约75%，高于随机标注的55%。

## 工程落地要点
- **离线批量生成**：LLM推理开销大，伪标签生成应完全在离线批量阶段完成，不能引入线上延迟。建议每日/每周离线更新伪标签数据集。
- **伪标签版本管理**：随LLM版本升级，伪标签质量会变化，需记录生成时使用的模型版本和Prompt版本，方便追溯问题。
- **Prompt设计关键**：Prompt中需包含用户近期行为摘要（去除敏感信息）、候选物品核心属性，避免让LLM接触原始用户ID等隐私信息。
- **成本控制**：可用小型LLM（如7B模型）替代GPT-4，经Fine-tuning后在特定领域伪标签生成任务上性能接近，成本降低10x以上。

## 常见考点
**Q1: 为什么推荐系统中存在大量未标注数据？**
A: 推荐系统只能观察到曝光物品的用户反馈，未被曝光的物品（占候选库99%以上）没有用户交互信号，因此是"天然未标注"数据。这是推荐系统与其他监督学习任务的根本区别之一。

**Q2: 用LLM生成伪标签的主要风险是什么？如何缓解？**
A: 主要风险是标签噪声（LLM对用户偏好的判断可能与真实行为不符）和数据分布偏移（LLM见过的内容偏向热门/通用领域）。缓解方法包括：置信度过滤、与真实标签混合训练、在验证集上监控伪标签质量。

**Q3: 课程学习（Curriculum Learning）在这里的作用是什么？**
A: 先让模型在高质量的真实数据上学习基础模式，再引入质量稍差的伪标签数据，避免模型在训练初期被噪声标签带偏，类似人类学习先学基础再处理复杂案例的方式。

## 模型架构详解

### 输入处理
- **候选集编码**：精排 Top-K 结果（通常 K=50~200），每个候选带精排分数和特征
- **列表级特征**：候选间的相似度矩阵、类目分布、已有展示历史
- **上下文**：用户实时状态、页面位置、时间特征

### 核心模块
- **候选交互**：Self-Attention / Graph Attention 捕获候选间关系
- **序列解码**：Pointer Network / 自回归方式逐步选择最优排列
- **约束满足**：多样性/品类/广告密度等约束的可微松弛

### 优化目标
- **Listwise Loss**：NDCG/MAP 的可微近似（ApproxNDCG / LambdaLoss）
- **多目标加权**：CTR × 时长 × 多样性 × 新颖度的帕累托权衡
- **强化学习**：将重排建模为序列决策问题，用 RL 优化长期指标

## 面试深度追问

- **Q: 重排为什么不能直接用精排分数排序？**
  A: 精排是 Pointwise 打分，忽略候选间的交互效应。例如连续展示同类商品会降低用户兴趣（边际效用递减），重排需要考虑列表级效果。

- **Q: 多样性和相关性的平衡策略？**
  A: 1) MMR：λ × 相关性 - (1-λ) × 冗余度；2) DPP：行列式点过程建模子集多样性；3) 约束优化：多样性作为约束条件，相关性作为目标函数。

- **Q: 重排阶段的延迟预算如何分配？**
  A: 典型链路延迟预算：召回 10ms + 粗排 5ms + 精排 20ms + 重排 10ms = 总计 < 50ms。重排需在 10-20ms 内完成，通常限制候选集 ≤ 200。

- **Q: 生成式重排和传统重排的区别？**
  A: 生成式重排将排列视为序列生成问题（类似 Seq2Seq），可以直接优化全局排列指标。传统方法（Greedy/Beam Search）是启发式搜索，可能陷入局部最优。

## 总结与实战建议

本文在推荐系统领域提出了有价值的技术创新。该方法的核心思想可与现有系统组件互补，建议：
1. 先在离线数据集上复现核心实验结果
2. 评估在自身业务场景的适用性和改进空间
3. 通过 A/B Test 验证线上效果，关注 CTR/CVR/用户停留时长等核心指标
4. 监控模型长期表现，及时应对数据分布漂移

## 工业界实践参考

### 典型应用场景
- **电商推荐**：淘宝/京东/拼多多的商品推荐排序
- **内容推荐**：抖音/快手/B站的短视频推荐
- **广告系统**：百度/腾讯/字节的广告排序和创意优化
- **搜索引擎**：相关性排序和个性化搜索结果

### 关键技术指标
- 离线评估：AUC > 0.75, GAUC > 0.65, NDCG@10 > 0.5
- 在线效果：CTR 提升 1-5%, 用户时长提升 2-8%, GMV 提升 1-3%
- 系统延迟：端到端 < 200ms, 模型推理 < 30ms (P99)
- 模型规模：Embedding 参数 TB 级, Dense 参数 GB 级

### 部署与监控
1. **灰度发布**：新模型先在 1% 流量验证，确认无异常后逐步放量
2. **在线监控**：实时追踪 AUC/LogLoss/CTR 等核心指标的分钟级波动
3. **降级策略**：模型超时或异常时回退到上一版本或规则兜底
4. **增量更新**：小时级增量训练 + 天级全量训练的混合策略
5. **特征监控**：特征覆盖率、均值/方差漂移检测、缺失值报警
