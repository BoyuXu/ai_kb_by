# Not All Candidates are Created Equal: Heterogeneity-Aware Pre-ranking in Recommender Systems

> 来源：arXiv | 日期：20260317

## 问题定义

大规模推荐系统通常采用多阶段漏斗架构：召回 → 粗排（Pre-ranking）→ 精排（Ranking）→ 重排。粗排阶段需要在极低延迟（<10ms）内从数千候选中筛选出数百个，传统做法是用轻量级双塔模型对所有候选一视同仁打分，忽略了不同候选之间的**异质性**（Heterogeneity）——即不同类型候选的最优评分策略各不相同。

## 核心方法与创新点

1. **候选异质性建模**
   - 将候选池按特征空间分组（如内容类型、发布时间、领域标签），识别不同子群的分布差异
   - 引入 Group-Aware 表征学习：对每个候选组学习独立的轻量 adapter，在共享主干基础上进行 group-specific 微调

2. **异质感知打分机制**
   - 使用混合专家（MoE-lite）结构替换单一评分头，每个专家负责特定候选子群
   - 路由权重由候选的 group embedding 决定，保持端到端可训练

3. **训练优化**
   - 采用 group-balanced 负采样策略，避免头部候选压制尾部候选的梯度信号
   - 引入 intra-group 排序损失（ListNet 变体），强化同组内相对顺序学习

4. **工程约束下的轻量化**
   - 参数量控制与标准双塔相当（<10M），推理仅增加约 5% 延迟
   - 动态路由可在 FPGA/TensorRT 上 kernel fusion

## 实验结论

- 在某大型工业数据集上，NDCG@50 提升 1.8%，相比全精排效果损失从 8.2% 降至 4.1%
- 召回率（Recall@200 → Precision@50）提升约 2.3%
- 头部/尾部候选的排序一致性显著改善，长尾内容曝光提升约 15%

## 工程落地要点

1. **候选分组策略**：建议离线根据候选特征聚类确定 K（通常 4~8 组），避免在线动态聚类增加延迟
2. **模型服务部署**：将 group-id 作为额外输入，路由逻辑在预处理阶段完成，评分服务无分支
3. **特征对齐**：粗排与精排的候选特征体系需统一，避免 group 分配漂移
4. **AB 测试**：注意异质性带来的组间流量变化，需分层评估各候选类型的指标

## 常见考点

- **Q: 粗排为什么要做异质性建模？**
  A: 不同类型候选的特征分布、用户偏好信号差异显著，统一评分函数会导致某些类型候选系统性低估或高估，异质建模可让每类候选用更匹配其特性的打分策略。

- **Q: MoE 在粗排中如何保持低延迟？**
  A: 使用少量专家（2~4个）+ 轻量路由，避免 top-k 路由的不确定性；路由前置在特征预处理，评分阶段并行执行多个轻量专家头。

- **Q: 如何解决组间不平衡的训练问题？**
  A: Group-balanced 采样 + 组内 ListNet 损失；也可用 group-weighted loss 手动平衡各组梯度贡献。

## 模型架构详解

### 输入处理
- **候选集编码**：精排 Top-K 结果（通常 K=50~200），每个候选带精排分数和特征
- **列表级特征**：候选间的相似度矩阵、类目分布、已有展示历史
- **上下文**：用户实时状态、页面位置、时间特征

### 核心模块
- **候选交互**：Self-Attention / Graph Attention 捕获候选间关系
- **序列解码**：Pointer Network / 自回归方式逐步选择最优排列
- **约束满足**：多样性/品类/广告密度等约束的可微松弛

### 优化目标
- **Listwise Loss**：NDCG/MAP 的可微近似（ApproxNDCG / LambdaLoss）
- **多目标加权**：CTR × 时长 × 多样性 × 新颖度的帕累托权衡
- **强化学习**：将重排建模为序列决策问题，用 RL 优化长期指标

## 面试深度追问

- **Q: 重排为什么不能直接用精排分数排序？**
  A: 精排是 Pointwise 打分，忽略候选间的交互效应。例如连续展示同类商品会降低用户兴趣（边际效用递减），重排需要考虑列表级效果。

- **Q: 多样性和相关性的平衡策略？**
  A: 1) MMR：λ × 相关性 - (1-λ) × 冗余度；2) DPP：行列式点过程建模子集多样性；3) 约束优化：多样性作为约束条件，相关性作为目标函数。

- **Q: 重排阶段的延迟预算如何分配？**
  A: 典型链路延迟预算：召回 10ms + 粗排 5ms + 精排 20ms + 重排 10ms = 总计 < 50ms。重排需在 10-20ms 内完成，通常限制候选集 ≤ 200。

- **Q: 生成式重排和传统重排的区别？**
  A: 生成式重排将排列视为序列生成问题（类似 Seq2Seq），可以直接优化全局排列指标。传统方法（Greedy/Beam Search）是启发式搜索，可能陷入局部最优。

## 总结与实战建议

本文在推荐系统领域提出了有价值的技术创新。该方法的核心思想可与现有系统组件互补，建议：
1. 先在离线数据集上复现核心实验结果
2. 评估在自身业务场景的适用性和改进空间
3. 通过 A/B Test 验证线上效果，关注 CTR/CVR/用户停留时长等核心指标
4. 监控模型长期表现，及时应对数据分布漂移

## 工业界实践参考

### 典型应用场景
- **电商推荐**：淘宝/京东/拼多多的商品推荐排序
- **内容推荐**：抖音/快手/B站的短视频推荐
- **广告系统**：百度/腾讯/字节的广告排序和创意优化
- **搜索引擎**：相关性排序和个性化搜索结果

### 关键技术指标
- 离线评估：AUC > 0.75, GAUC > 0.65, NDCG@10 > 0.5
- 在线效果：CTR 提升 1-5%, 用户时长提升 2-8%, GMV 提升 1-3%
- 系统延迟：端到端 < 200ms, 模型推理 < 30ms (P99)
- 模型规模：Embedding 参数 TB 级, Dense 参数 GB 级

### 部署与监控
1. **灰度发布**：新模型先在 1% 流量验证，确认无异常后逐步放量
2. **在线监控**：实时追踪 AUC/LogLoss/CTR 等核心指标的分钟级波动
3. **降级策略**：模型超时或异常时回退到上一版本或规则兜底
4. **增量更新**：小时级增量训练 + 天级全量训练的混合策略
5. **特征监控**：特征覆盖率、均值/方差漂移检测、缺失值报警

### 未来趋势
- 大模型与推荐的深度融合（LLM as Recommender / LLM-augmented Features）
- 生成式推荐（从判别式到生成式范式转变）
- 多模态统一表示（文本/图片/视频/行为的联合编码）
- 因果推断在推荐去偏中的更广泛应用
