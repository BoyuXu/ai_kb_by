## 一、粗排在推荐链路中的定位

### 1.1 四阶段漏斗架构

```
用户请求
    │
    ▼
┌──────────────────────────────────────────────────┐
│  召回 (Recall)          百万 → 万级              │
│  多路召回：协同/向量/图/规则                     │
│  模型：DSSM、双塔、TDM     延迟 ~1ms/路         │
└────────────────────┬─────────────────────────────┘
                     ▼
┌──────────────────────────────────────────────────┐
│  ★ 粗排 (Pre-Ranking)    千~5000 → 百级         │
│  轻量模型：双塔 / COLD / 蒸馏小模型              │
│  延迟约束：< 10ms（含网络开销 < 15ms）           │
└────────────────────┬─────────────────────────────┘
                     ▼
┌──────────────────────────────────────────────────┐
│  精排 (Fine Ranking)      百 → 十级              │
│  重模型：DCN-V2 / DIN / 多任务 (MMoE/PLE)       │
│  延迟约束：< 50ms                                │
└────────────────────┬─────────────────────────────┘
                     ▼
┌──────────────────────────────────────────────────┐
│  重排 (Re-Ranking)        十 → 最终展示          │
│  多目标融合 + 多样性 + 业务规则                  │
└──────────────────────────────────────────────────┘
```

### 1.2 粗排的核心约束

| 约束维度 | 具体要求 | 原因 |
|---------|---------|------|
| **延迟** | < 10ms（P99 < 15ms） | 候选量大（1000-5000），需在总预算内 |
| **算力** | 单次请求 < 精排 1/10 FLOPs | GPU/CPU 资源有限，粗排吞吐量要求高 |
| **特征** | 不能使用交叉特征 / 实时统计特征 | 特征计算成本高，无法对每个候选逐一计算 |
| **模型** | 参数量 < 精排 1/5~1/10 | 推理速度正比于参数量 |

### 1.3 为什么粗排需要蒸馏

精排模型（Teacher）具备的优势，粗排（Student）天然缺失：

| 精排有 | 粗排没有 | 差距来源 |
|--------|---------|---------|
| 交叉特征（user×item 统计） | 只有 user/item 独立特征 | 特征空间不一致 |
| 深层网络（DCN-V2, Transformer） | 浅层 MLP / 双塔 | 模型容量差距 |
| 长序列建模（SIM, DIEN） | 简单序列池化 | 行为序列信息损失 |
| 多任务联合建模 | 单任务或少任务 | 目标信息不足 |

**蒸馏的价值**：让粗排小模型学到精排大模型的「排序知识」，在不增加推理成本的前提下，大幅提升粗排质量。

工业实测数据：
- 蒸馏 vs 不蒸馏：AUC 提升 0.3-0.8pp
- 蒸馏粗排 vs 精排：AUC 差距从 2-3pp 缩小到 0.3-0.5pp
- 延迟：粗排 5-8ms vs 精排 30-50ms（20-40x 加速）

## 二、粗排蒸馏的核心方法

### 2.1 Logit 蒸馏（Response-Based Distillation）

**原理**：精排模型的 soft label（软标签）包含丰富的「暗知识」——不仅告诉粗排「这个 item 会被点击」，还告诉「这个 item 和哪些 item 的点击概率接近」。

**核心公式**：

$$
\mathcal{L}_{logit} = (1-\alpha) \cdot \mathcal{L}_{CE}(y, \hat{p}_s) + \alpha \cdot T^2 \cdot \text{KL}\left(\sigma\!\left(\frac{z_t}{T}\right) \bigg\| \sigma\!\left(\frac{z_s}{T}\right)\right)
$$

- $z_t, z_s$：精排/粗排的 logits
- $T$：温度参数（T>1 软化分布，放大暗知识）
- $\alpha$：蒸馏损失权重（工业常用 0.3-0.7）
- $y$：真实标签（硬标签）

**对于 CTR 二分类场景**（sigmoid 输出），更常用 MSE loss：

$$
\mathcal{L}_{MSE} = \frac{1}{N} \sum_{i=1}^{N} \left( \sigma(z_t^{(i)}) - \sigma(z_s^{(i)}) \right)^2
$$

**温度 T 的作用与选择**：

| T 值 | 分布特征 | 适用场景 |
|------|---------|---------|
| T=1 | 原始尖锐分布 | 不推荐（暗知识弱） |
| T=2~5 | 平衡暗知识与信号强度 | CTR 粗排蒸馏常用起点 |
| T=10+ | 趋近均匀分布 | 类间关系迁移，但信噪比下降 |

**KL 散度 vs MSE loss 对比**：

| 维度 | KL 散度 | MSE Loss |
|------|---------|----------|
| 适用场景 | 多分类（softmax 输出） | 二分类（sigmoid 输出） |
| 暗知识利用 | 完整分布匹配 | 只匹配点估计 |
| 数值稳定性 | 需要温度缩放 | 天然稳定 |
| 工业偏好 | 多目标粗排 | 单目标 CTR 粗排 |

**优点**：实现简单，不需要额外架构设计；工业界最成熟的方案。

**缺点**：只学到精排的「最终输出」，丢失了中间过程的知识；绝对分数蒸馏在排序任务中不一定最优。

**代表论文**：Hinton et al., "Distilling the Knowledge in a Neural Network", 2015

**代表工业案例**：阿里 COLD、百度凤巢、几乎所有粗排系统的 baseline。

### 2.2 特征蒸馏（Feature-Based Distillation）

**原理**：不仅学精排的输出，还学精排的中间层表示（embedding / hidden states），让粗排在特征空间上对齐精排。

**核心公式**：

$$
\mathcal{L}_{feat} = \sum_{l=1}^{L} \left\| f_s^{(l)}(x) \cdot W_l - f_t^{(l')}(x) \right\|_2^2
$$

- $f_s^{(l)}, f_t^{(l')}$：粗排第 $l$ 层 / 精排第 $l'$ 层的中间表示
- $W_l$：投影矩阵（适配维度差异，如粗排 64 维 → 精排 256 维）

**FitNets / Hint Learning 在粗排的应用**：

FitNets（Romero et al., 2015）提出「Hint Layer」的概念：选择教师的某个中间层作为 Hint，引导学生中间层对齐。

在粗排蒸馏中的具体实践：
1. 选择精排的 Embedding 聚合层或最后一个 Hidden Layer 作为 Hint
2. 粗排添加一个轻量投影层（linear projection）对齐维度
3. 训练目标：$\mathcal{L} = \mathcal{L}_{task} + \beta \cdot \mathcal{L}_{hint}$

**粗排场景的特殊挑战**：
- 精排和粗排的特征集不同，直接对齐 Embedding 层不现实
- 解决方案：对齐**最终隐层**（特征交叉之后的语义表示），而非原始 Embedding

**组合使用**：

$$
\mathcal{L}_{total} = \alpha \cdot \mathcal{L}_{logit} + \beta \cdot \mathcal{L}_{feat} + (1-\alpha-\beta) \cdot \mathcal{L}_{hard}
$$

工业实践中 logit 蒸馏 + 特征蒸馏组合效果优于单一方法（AUC 额外提升 0.1-0.2pp）。

**优点**：利用了教师的中间层知识，信息量更大。

**缺点**：需要额外的投影层；精排和粗排架构差异大时对齐困难；增加训练复杂度。

### 2.3 特权特征蒸馏（Privileged Features Distillation, PFD）

**核心思想**：精排可以使用「特权特征」（粗排无法获取的特征），如 user-item 交叉统计特征、实时特征。蒸馏的目标是让粗排在**没有这些特征**的情况下，学到这些特征带来的排序增益。

**特权特征的分类**：

| 特权特征类型 | 示例 | 为什么粗排不能用 |
|-------------|------|-----------------|
| user×item 交叉统计 | 用户对该品类的历史 CTR | 需要逐候选计算，O(N) 开销 |
| 实时行为特征 | 用户最近 5 分钟行为 | 实时特征服务延迟高 |
| 长序列特征 | 用户最近 1000 次行为 | 序列建模计算量大 |
| 交叉 Embedding | user_emb × item_emb 拼接后 DNN | 破坏双塔独立性 |

**Rocket Launching（Zhou et al., "Rocket Launching: A Universal and Efficient Framework for Training Well-performing Light Net", AAAI 2018）**：

核心方案：教师（Booster）和学生（Rocket）共享底层 Embedding，联合训练。关键技巧是 **gradient block**——阻止 mimic loss 的梯度回传到教师，保证教师持续提升。

$$
\mathcal{L} = \mathcal{L}_{light} + \lambda \cdot \mathcal{L}_{booster} + \mu \cdot \text{MSE}(z_{light}, \text{sg}(z_{booster}))
$$

其中 $\text{sg}(\cdot)$ 是 stop-gradient 操作。

关键创新：
1. **共享 Embedding**：教师和学生共享底层 Embedding 表，特权特征的梯度间接提升共享 Embedding 的质量
2. **Booster-Rocket 架构**：教师（Booster）训练完成后「脱落」，只部署学生（Rocket）
3. **Gradient Block**：阻止 mimic loss 梯度回传到教师，避免教师被拉向学生容量水平

**Google RKME（Representation Knowledge Mining and Extraction）**：

思路：精排使用交叉特征产生的高质量 Embedding 表示，通过蒸馏迁移到粗排的独立 Embedding 空间。

**实时特征 vs 离线特征的 Gap**：

```
精排训练数据：
  user_feature + item_feature + cross_feature + realtime_feature → label

粗排训练数据：
  user_feature + item_feature → soft_label (来自精排打分)

Gap 来源：
  1. 特征维度不同（精排 200+ 维，粗排 50-80 维）
  2. 特征分布不同（精排的交叉特征是非线性组合）
  3. 特权特征的信息无法通过 soft label 完全传递
```

**PFD（Xu et al., "Privileged Features Distillation at Taobao Recommendations", KDD 2020）**：

淘宝提出的特权特征蒸馏方案。核心洞察：这不是模型压缩问题，而是**信息不对称**问题——教师看到了学生永远看不到的特征。

$$
\mathcal{L}_{PFD} = \mathcal{L}_{CE}(y_S, y_{true}) + \alpha \cdot \text{KL}(p_T(x_r, x_p) \| p_S(x_r))
$$

其中 $x_p$ 是特权特征，$x_r$ 是常规特征。

**效果**：CTR 任务 +5.0%，CVR 任务 +2.3%，无额外推理延迟。

**后续理论分析**：NeurIPS 2022（Amazon）"Toward Understanding Privileged Features Distillation in Learning-to-Rank" 从理论上解释了 PFD 为什么有效。

**优点**：直接解决粗排缺少关键特征的问题；Rocket Launching 的共享 Embedding 机制简洁有效；PFD 的信息不对称视角独特。

**缺点**：共享 Embedding 可能导致教师和学生互相干扰；特权特征信息通过 soft label 传递的效率有上限。

### 2.4 排序蒸馏（Ranking Distillation）

**核心思想**：粗排的核心目标是排序（ranking），而非预测绝对分数（scoring）。因此，蒸馏排序关系（pairwise / listwise）比蒸馏绝对 logit 更契合粗排的任务本质。

#### 2.4.1 Pairwise 排序蒸馏

**核心公式**：

$$
\mathcal{L}_{pair} = \sum_{(i,j): s_t^{(i)} > s_t^{(j)}} \max\left(0, -\left(s_s^{(i)} - s_s^{(j)}\right) + m\right)
$$

或使用 BPR 风格的软版本：

$$
\mathcal{L}_{pair-soft} = -\sum_{(i,j): s_t^{(i)} > s_t^{(j)}} \log \sigma\left(s_s^{(i)} - s_s^{(j)}\right)
$$

- $s_t^{(i)}$：精排给 item $i$ 的分数
- $s_s^{(i)}$：粗排给 item $i$ 的分数
- 目标：粗排保持精排的相对排序

#### 2.4.2 Listwise 排序蒸馏

**ListNet Distillation**：

$$
\mathcal{L}_{list} = -\sum_{i=1}^{N} P_t(i) \cdot \log P_s(i)
$$

其中排序概率：

$$
P_t(i) = \frac{\exp(s_t^{(i)} / \tau)}{\sum_{j=1}^{N} \exp(s_t^{(j)} / \tau)}, \quad P_s(i) = \frac{\exp(s_s^{(i)} / \tau)}{\sum_{j=1}^{N} \exp(s_s^{(j)} / \tau)}
$$

$\tau$ 类似温度参数，控制排序概率的平滑度。

#### 2.4.3 Ranking Distillation（Tang & Wang, KDD 2018）& RankDistil（Reddi et al., AISTATS 2021, Google）

**Ranking Distillation（KDD 2018）**核心创新：学生从教师的排序列表和真实标签同时学习。

$$
\mathcal{L}_{RD} = -\sum_{k \in R_T \setminus R_S} \pi_k \cdot \log \sigma(r_k^S)
$$

其中 $R_T$ 是教师 Top-K 推荐集，$\pi_k = (K+1-\text{rank}(k)) / \sum_j j$ 是排名加权重要性。

**RankDistil（AISTATS 2021, Google Research）** 进一步提出 Top-K 保序蒸馏框架，建立了排序蒸馏与 listwise/pairwise ranking loss 之间的理论联系。

- 惩罚学生将教师 Top-K 的 item 排到 K 之外
- $\pi_k$ 按教师排名位置递减加权（Top-1 的 item 被排错惩罚最大）

#### 2.4.4 为什么排序蒸馏比 Logit 蒸馏更适合粗排

| 维度 | Logit 蒸馏 | 排序蒸馏 |
|------|-----------|---------|
| 优化目标 | 绝对分数一致 | 相对排序一致 |
| 与粗排任务的匹配度 | 中（粗排关心排序，不关心绝对分数） | 高（直接优化排序一致性） |
| 对教师校准的依赖 | 高（教师分数必须准确） | 低（只需排序正确） |
| 对样本空间差异的鲁棒性 | 低（不同样本空间分数尺度不同） | 高（排序是相对关系） |

**工业实践**：排序蒸馏通常作为 logit 蒸馏的补充，组合使用效果最佳：

$$
\mathcal{L}_{total} = \alpha \cdot \mathcal{L}_{logit} + \beta \cdot \mathcal{L}_{rank} + (1-\alpha-\beta) \cdot \mathcal{L}_{hard}
$$

**优点**：直接优化排序目标；对教师分数校准要求低；鲁棒性强。

**缺点**：Pairwise 计算量 O(N²)；Listwise 需要 list 级别的 batch 构造；工程实现比 logit 蒸馏复杂。

### 2.5 样本蒸馏 / 数据增强

**核心思想**：利用精排模型的打分能力，为粗排构造更好的训练数据。

#### 2.5.1 精排打分作为 Soft Label

最简单也最常用的数据增强方式：

```
传统粗排训练：
  (user, item, click_label) → 训练粗排

蒸馏增强训练：
  (user, item, teacher_score) → 训练粗排
  其中 teacher_score = σ(精排 logit)
```

精排 soft label 的优势：
1. **更细粒度**：硬标签只有 0/1，soft label 可以是 0.73 vs 0.68，包含排序信息
2. **正则化效果**：soft label 作为平滑标签，防止粗排过拟合
3. **标签去噪**：精排模型的预测比原始 click label 更准确（消除了 position bias 等噪声）

#### 2.5.2 负样本策略

| 策略 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| **全量负样本** | 曝光未点击全部作为负样本 | 数据量大 | 含大量 easy negative |
| **精排淘汰样本** | 精排打低分但粗排打高分的样本 | hard negative，信息量大 | 需要额外的精排打分流程 |
| **随机负采样** | 从全库随机采样 | 简单 | 太容易，对粗排提升有限 |
| **粗排困难负样本** | 粗排排名高但精排排名低的样本 | 针对性强 | 需要 online 收集 |

**精排淘汰样本蒸馏**（最有价值的方案）：

$$
\mathcal{D}_{hard} = \{(u, i) : \text{rank}_{coarse}(i) \leq K_{coarse}, \text{rank}_{fine}(i) > K_{fine}\}
$$

即：粗排认为好但精排淘汰的样本，是粗排最需要学习的「错误案例」。

**优点**：不需要修改模型架构；通用性强；可以与任何蒸馏方法组合。

**缺点**：需要精排模型对全量候选打分（离线成本高）；数据时效性问题。

### 2.6 在线蒸馏 / 联合训练（Online / Co-Distillation）

**代表论文**：Anil et al., "Large Scale Distributed Neural Network Training Through Online Distillation", ICLR 2018（Google）

**核心思想**：精排和粗排同时训练，梯度互传，避免传统离线蒸馏的时效性问题。

#### 2.6.1 Co-Distillation 架构

```
┌────────────────────────────────────────────────┐
│  Joint Training Framework                       │
│                                                 │
│  ┌──────────────┐      ┌──────────────┐        │
│  │  精排 Teacher │      │  粗排 Student │        │
│  │  (大模型)     │      │  (小模型)     │        │
│  └──────┬───────┘      └──────┬───────┘        │
│         │     ┌────────┐     │                  │
│         ├────►│ KL Loss │◄───┤                  │
│         │     └────────┘     │                  │
│         │     ┌────────┐     │                  │
│         └────►│梯度互传 │◄───┘                  │
│               └────────┘                        │
│                                                 │
│  精排 loss = L_task + λ₁·KL(p_s ∥ p_t)         │
│  粗排 loss = L_task + λ₂·KL(p_t ∥ p_s)         │
└────────────────────────────────────────────────┘
```

#### 2.6.2 核心公式

**精排侧**（可选，双向蒸馏时使用）：

$$
\mathcal{L}_{teacher} = \mathcal{L}_{task}^{t} + \lambda_1 \cdot \text{KL}(p_s^{sg} \| p_t)
$$

**粗排侧**：

$$
\mathcal{L}_{student} = \mathcal{L}_{task}^{s} + \lambda_2 \cdot \text{KL}(p_t^{sg} \| p_s)
$$

其中 $sg$ 表示 stop gradient（阻止梯度回传到对方）。

#### 2.6.3 工业实践中的挑战

| 挑战 | 原因 | 解决方案 |
|------|------|---------|
| 训练不稳定 | 两个模型互相影响，容易震荡 | 交替训练 + 渐进增大蒸馏权重 |
| 计算成本高 | 需要同时维护两个模型的前向+反向 | 异步更新：精排每 N 步更新一次粗排的 target |
| 模型耦合 | 精排和粗排训练 pipeline 耦合 | 解耦方案：精排定期 checkpoint → 粗排异步蒸馏 |

**实际工业中更常见的是半在线蒸馏**：精排模型每天/每小时更新 checkpoint → 粗排用最新 checkpoint 的 soft label 重训。这比完全离线蒸馏时效性更好，又避免了联合训练的复杂度。

**优点**：消除教师-学生训练时间差；理论上信息传递最充分。

**缺点**：工程复杂度高；训练不稳定；资源消耗大；工业落地案例相对较少。

## 三、工业实践案例

### 3.1 阿里 COLD

**论文**：COLD: Towards the Next Generation of Pre-Ranking System（DLP-KDD 2020）

**核心设计思想**：

> 粗排不需要精排级别的复杂度，但需要比双塔更强的表达能力。

COLD 三大核心创新：

| 创新点 | 具体方案 | 目的 |
|--------|---------|------|
| **算力感知** | 可配置的计算复杂度（调节网络深度/宽度） | 在延迟预算内最大化效果 |
| **SENET 特征交叉** | Squeeze-and-Excitation 替代 DCN | 轻量级交叉，效果接近 DCN |
| **Hash Embedding** | 特征 ID 哈希到固定大小的 Embedding 表 | 减少参数量 10x+ |

**COLD 架构**：

```
Input Features
    │
    ▼
Embedding Layer (Hash Trick: id % table_size)
    │
    ▼
SENET Feature Recalibration
  ┌──────────────────────────────────┐
  │ Squeeze:  z = mean(x, dim=-1)   │
  │ Excite:   s = σ(W₂·ReLU(W₁·z)) │
  │ Reweight: x' = s ⊙ x           │
  └──────────────────────────────────┘
    │
    ▼
Lightweight DNN [512→256→128]
    │
    ▼
Output: pCTR
```

**COLD + 蒸馏**：COLD 模型本身就使用精排 soft label 作为训练目标之一。

$$
\mathcal{L}_{COLD} = \alpha \cdot \text{MSE}(p_{teacher}, p_{student}) + (1-\alpha) \cdot \text{BCE}(y, p_{student})
$$

**效果**：
- 粗排 AUC 提升 2.5pp（对比传统双塔）
- 在线 CTR 提升 6.1%
- 延迟：8ms（满足粗排约束）

### 3.2 快手粗排蒸馏

**核心方案**：双塔 + 多层次蒸馏 + 多样性蒸馏

快手粗排系统的蒸馏策略：

1. **基础蒸馏**：精排 soft label → 粗排双塔模型
2. **序列蒸馏**：通过 TWIN（Two-view Interest Network）结构，将精排的序列建模能力蒸馏到双塔中
3. **多目标蒸馏**：精排多个 head（CTR/CVR/时长）的 soft label 同时蒸馏
4. **多样性蒸馏**：CDM（Contextual Distillation Model, KDD 2024）将 MMR 重排的多样性知识蒸馏到粗排模型

**TWIN 双塔序列建模**（快手创新）：

```
User Tower:
  ┌─────────────────────┐
  │ Short-term: DIN     │ ← 最近 50 条行为
  │ Long-term:  Target  │ ← 最近 1000 条行为
  │            Attention │   (用 target item 做 query)
  └──────────┬──────────┘
             │
  User Embedding (离线计算/缓存)

Item Tower:
  ┌─────────────────────┐
  │ Item Features       │
  │ + Side Information  │
  └──────────┬──────────┘
             │
  Item Embedding (离线计算/缓存)

Score = cos(user_emb, item_emb) → ANN 检索
```

核心突破：在保持双塔独立性（支持 ANN 检索）的前提下，引入用户行为序列建模。

### 3.3 美团粗排双塔 + 蒸馏

**核心方案**：增强双塔 + NAS 搜索 + 蒸馏

美团在粗排领域有三篇重要工作：

1. **DAT（Dual Augmented Two-tower, DLP-KDD 2021）**：通过 Adaptive-Mimic 机制在双塔间创建增强向量，在保持双塔推理效率的同时引入交叉交互。在线 A/B：CTR +4.17%，GMV +3.46%
2. **AutoFAS（KDD 2022）**：用 NAS 同时搜索粗排最优特征和网络架构，ranking model-guided reward 确保粗排逼近精排。**首个联合搜索特征+架构的粗排工作**
3. **RankTower（AdKDD 2024）**：以蒸馏 loss 作为双塔粗排的主训练目标，Multi-Head Gated Network + Gated Cross-Attention 增强双塔内的 user-item 交互

$$
\mathcal{L}_{cascade} = \mathcal{L}_{task} + \lambda_1 \cdot \text{KL}(p_{fine} \| p_{mid}) + \lambda_2 \cdot \text{KL}(p_{mid} \| p_{coarse})
$$

级联蒸馏的优势：教师和学生的容量差距被中间模型桥接，避免了大跨度蒸馏的信息损失。

### 3.4 Google Two-Tower + Distillation

**核心方案**：偏差纠正双塔 + 辅助蒸馏

Google 在搜索和推荐中大规模使用双塔模型，有两篇关键论文：

**Sampling-Bias-Corrected Two-Tower（Yi et al., RecSys 2019, YouTube）**：

解决 batch 内负采样的流行度偏差：

$$
p(y|x) = \frac{e^{s(x,y)/\tau} / p_y^{\beta}}{\sum_j e^{s(x,y_j)/\tau} / p_{y_j}^{\beta}}
$$

其中 $p_y$ 是流式频率估计器计算的 item 频率。

**Auxiliary Distillation（Khani et al., RecSys 2024, Google DeepMind）**：

识别在线排序蒸馏的三大挑战：(1) 教师偏差泄漏 (2) 教师配置搜索成本 (3) 标签共享基础设施。

核心方案：**辅助蒸馏**——为硬标签和软标签使用独立的 task logits，防止噪声叠加。

效果：2x 教师 → +0.42% engagement；4x 教师 → +0.85%；多任务选择性蒸馏 → +1.13%。

### 3.5 百度凤巢粗排

**核心方案**：MOBIUS 匹配层 CTR 蒸馏

**MOBIUS（Fan et al., "Towards the Next Generation of Query-Ad Matching in Baidu's Sponsored Search", KDD 2019）**：

打破传统匹配-排序边界，在匹配层就联合优化相关性和 CTR：
1. **Teacher-Student for 匹配层 CTR**：原始匹配模型（Teacher）筛选相关性，神经点击网络（Student）在匹配层预测 CTR
2. **Active Learning 负样本**：用教师标注的不相关 query-ad pair 作为困难负样本训练学生
3. **ANN 检索部署**：Student 支持 ANN 高效检索

**COPR（Zhao et al., "Consistency-Oriented Pre-Ranking for Online Advertising", CIKM 2023）**：

解决粗排-精排排序不一致问题。chunk-based sampling + rank alignment + delta-NDCG weighting，显式优化 eCPM 排序一致性。效果：CTR +12.3%，RPM +5.6%（淘宝展示广告验证）。

百度凤巢广告粗排的独特需求：

1. **多目标蒸馏**：CTR + CVR + 出价相关性 → 粗排统一模型
2. **eCPM 一致性约束**：粗排的 eCPM 排序需要与精排一致
3. **实时特权特征蒸馏**：广告的实时竞价特征（bid、budget 状态）属于特权特征

广告粗排与推荐粗排的关键区别：

| 维度 | 推荐粗排 | 广告粗排 |
|------|---------|---------|
| 优化目标 | CTR / 用户体验 | eCPM = CTR × CVR × bid |
| 候选量 | 1000-5000 | 可达万级（广告库大） |
| 实时性 | 小时级更新 | 分钟级更新（预算消耗快） |
| 特权特征 | user×item 交叉 | + 实时出价、预算、竞争环境 |

## 四、粗排蒸馏的关键挑战

### 4.1 Teacher-Student 容量差距

**问题**：精排几百维特征 + 深层网络（~50M 参数），粗排几十维 + 浅层网络（~5M 参数），容量差距 10x+。

**解决方案**：

| 方案 | 原理 | 适用场景 |
|------|------|---------|
| **渐进式蒸馏** | 精排 → 中排 → 粗排 | 容量差距 > 5x |
| **中间教师桥接** | 训练中间大小的教师 | 精排太大无法直接蒸馏 |
| **选择性蒸馏** | 只蒸馏 Top-K 样本的排序 | 关注头部排序质量 |
| **课程蒸馏** | 从易到难逐步增加蒸馏难度 | 学生初期学习困难 |

### 4.2 样本空间不一致

**问题**：精排只看到粗排筛选后的候选（几百），但粗排需要处理全量候选（几千）。精排的「好坏判断」只在它看到的候选中成立。

```
粗排看到的候选：{A, B, C, D, E, F, G, H, I, J, ...}  (1000+)
                          │
                     粗排筛选 Top-200
                          │
精排看到的候选：{A, B, C, E, G, ...}  (200)
                          │
                     精排排序
                          │
精排 soft label 只覆盖这 200 个候选
```

**解决方案**：
1. **全量候选打分**：离线让精排对全量候选打分（计算成本高但效果最好）
2. **负样本扩充**：将粗排淘汰但精排可能给高分的候选加入训练
3. **纠偏训练**：对精排未覆盖的候选使用硬标签，覆盖的用 soft label

### 4.3 特征空间不一致

**问题**：精排有交叉特征（user_category_pref × item_category），粗排只有独立特征。蒸馏的 soft label 中隐含了交叉特征的信息，但粗排无法完全吸收。

**解决方案**：
1. **特权特征蒸馏（PFD）**：专门针对这个问题设计（见 2.3 节）
2. **预计算交叉特征**：将部分交叉特征预计算并存储（牺牲实时性换取特征质量）
3. **隐式交叉学习**：粗排通过 SENET/FiBiNET 等轻量交叉结构，学习部分交叉信号

### 4.4 在线更新频率不匹配

**问题**：精排可能每天更新，粗排可能每周更新。蒸馏 soft label 基于旧版精排模型，但新版精排已经变化。

| 更新频率 | 精排 | 粗排 |
|---------|------|------|
| 全量重训 | 天级 / 周级 | 周级 |
| 增量更新 | 小时级 | 天级 |
| Soft label | 实时生成 | 离线批量生成 |

**解决方案**：
1. **高频蒸馏**：精排每次更新后，立即重新生成 soft label 并触发粗排增量训练
2. **EMA 教师**：使用精排的指数移动平均版本作为教师，减少波动
3. **在线蒸馏**：精排和粗排联合训练（见 2.6 节），彻底消除时间差

### 4.5 多目标蒸馏的冲突

**问题**：精排通常是多任务模型（CTR + CVR + 时长），粗排蒸馏多个目标时可能冲突。

$$
\mathcal{L}_{multi} = \sum_{k=1}^{K} w_k \cdot \text{KL}(p_t^{(k)} \| p_s^{(k)})
$$

不同任务的蒸馏梯度方向可能冲突（类似多任务学习中的梯度冲突问题）。

**解决方案**：使用 PCGrad 或 GradNorm 平衡多目标蒸馏的梯度。

## 六、面试考点

### Q1: 为什么不直接用小精排代替粗排蒸馏？

**A**: 小精排 = 直接用小模型 + 精排全量特征训练。问题是：
1. **特征约束**：粗排不能用交叉特征（延迟和算力约束），小精排如果用全量特征就不满足粗排的延迟要求
2. **独立训练 vs 蒸馏**：同样的小模型，独立训练的效果远差于蒸馏。因为蒸馏提供了软标签的「暗知识」——精排对每个样本的置信度、样本间的相对关系，这些信息在硬标签中是丢失的
3. **样本空间**：精排训练数据只覆盖粗排筛选后的候选，小精排在未见样本上泛化能力弱；而蒸馏可以通过全量打分扩展覆盖

**实测对比**：蒸馏小模型 AUC 比同架构独立训练高 0.5-1.0pp。

### Q2: 排序蒸馏 vs Logit 蒸馏，什么时候选哪个？

**A**:

| 选择 Logit 蒸馏 | 选择排序蒸馏 |
|-----------------|-------------|
| 教师模型校准好（分数可靠） | 教师分数不校准但排序准确 |
| 需要绝对分数（如广告 eCPM） | 只关心排序质量 |
| 简单实现优先 | 愿意承受更高工程复杂度 |
| 单目标（CTR） | 多目标融合后的排序 |

**最佳实践**：两者组合。Logit 蒸馏提供绝对分数信号，排序蒸馏补充相对关系信号。

$$
\mathcal{L} = \alpha \cdot \mathcal{L}_{logit} + \beta \cdot \mathcal{L}_{rank} + (1-\alpha-\beta) \cdot \mathcal{L}_{hard}
$$

工业常用：$\alpha=0.3, \beta=0.2$。

### Q3: COLD 的设计思想是什么？为什么不直接缩小精排模型？

**A**: COLD 的核心思想是**算力感知**（Computing power cost-aware）：

1. **不是简单缩小精排**：精排缩小后仍然需要交叉特征计算，延迟瓶颈在特征而非模型
2. **SENET 替代 DCN**：SENET 只做特征重标定（field 级权重），不做显式交叉，计算量 << DCN 的交叉层
3. **Hash Embedding**：精排有数十亿级 ID Embedding 表，COLD 用 hash trick 压缩到固定大小，牺牲微小精度换取巨大的参数量减少
4. **可配置计算量**：COLD 的网络深度/宽度可以根据实时 QPS 动态调整——高峰期用浅网络，低峰期用深网络

这体现了粗排的本质：**不是精排的缩小版，而是一种专门设计的、在算力约束下最大化排序质量的模型**。

### Q4: 粗排蒸馏中如何处理样本空间不一致？

**A**: 样本空间不一致是粗排蒸馏的核心挑战之一。精排只对粗排 Top-K 的候选打分，但粗排需要对全量候选排序。

解决方案按效果排序：
1. **全量离线打分**（最佳）：让精排模型离线对全量候选打分，生成完整 soft label。成本高但效果最好
2. **困难负样本挖掘**：专门收集「粗排排高、精排排低」的样本，加大权重训练
3. **混合标签训练**：精排覆盖的候选用 soft label，未覆盖的用硬标签
4. **纠偏权重**：对精排覆盖和未覆盖的样本使用不同的 loss 权重

$$
\mathcal{L} = \sum_{i \in \mathcal{D}_{covered}} w_c \cdot \mathcal{L}_{soft}^{(i)} + \sum_{j \in \mathcal{D}_{uncovered}} w_u \cdot \mathcal{L}_{hard}^{(j)}
$$

### Q5: 特权特征蒸馏（PFD）和普通 logit 蒸馏有什么区别？

**A**: 关键区别在于**信息传递路径**：

- **Logit 蒸馏**：精排输出 → soft label → 粗排学习。特权特征的信息被「压缩」到一个分数里
- **PFD**：精排因特权特征产生的 Embedding/梯度 → 通过共享参数或额外对齐 loss → 粗排学习。特权特征的信息通过更丰富的通道传递

Rocket Launching 是 PFD 的经典实现：教师和学生共享底层 Embedding 表，特权特征的梯度在训练中提升了共享 Embedding 的质量，学生间接受益。

PFD 特别适合的场景：
1. 精排和粗排的特征差距非常大（粗排缺少 50%+ 的关键特征）
2. 特权特征对排序质量的贡献显著（如 user-item 交叉统计占 AUC 增益的 30%+）
3. 简单 logit 蒸馏已经到瓶颈，需要更深层次的信息传递

### Q6: 如何评估粗排蒸馏的效果？

**A**: 粗排蒸馏的评估不能只看 AUC，需要从多个维度：

| 指标 | 含义 | 重要性 |
|------|------|--------|
| **NDCG@K（与精排比）** | 粗排排序与精排排序的一致性 | ★★★★★ |
| **Recall@K（召回率）** | 精排 Top-K 中有多少被粗排保留 | ★★★★★ |
| **AUC** | 粗排模型本身的排序能力 | ★★★★ |
| **P99 延迟** | 线上推理延迟 | ★★★★★ |
| **在线 CTR/CVR** | A/B 实验效果 | ★★★★★ |

**最关键的指标是 Recall@K**：精排 Top-50 中有多少被粗排的 Top-200 保留？这直接决定了粗排是否「漏掉」了好的候选。

$$
\text{Recall@K}_{coarse \to fine} = \frac{|\text{Top-K}_{fine} \cap \text{Top-M}_{coarse}|}{|\text{Top-K}_{fine}|}
$$

工业标准：Recall@50 (粗排 Top-200) > 95%。

### Q7: 在线蒸馏 vs 离线蒸馏，实际工业中怎么选？

**A**:

| 维度 | 离线蒸馏 | 在线蒸馏 |
|------|---------|---------|
| 工程复杂度 | 低（pipeline 解耦） | 高（精排粗排耦合） |
| 时效性 | 小时~天级延迟 | 实时 |
| 训练稳定性 | 高 | 低（互相影响） |
| 资源消耗 | 正常 | 2x+（同时训练两个模型） |
| 效果 | 好 | 理论更好，实际提升有限 |

**结论**：95%+ 的工业粗排系统使用离线蒸馏（或半在线：精排小时级更新 → 触发粗排增量训练）。在线蒸馏的额外收益通常不值得工程复杂度的代价。

## 七、PyTorch 伪代码实现

### 7.1 完整粗排蒸馏训练框架

```python
import torch
import torch.nn as nn
import torch.nn.functional as F

class PreRankDistillationLoss(nn.Module):
    """粗排蒸馏损失：Logit + Ranking + Hard Label"""

    def __init__(self, temperature=4.0, alpha=0.3, beta=0.2):
        super().__init__()
        self.T = temperature
        self.alpha = alpha    # logit 蒸馏权重
        self.beta = beta      # 排序蒸馏权重

    def logit_distill(self, s_logits, t_logits):
        """Response-based distillation (MSE for binary CTR)"""
        s_prob = torch.sigmoid(s_logits)
        t_prob = torch.sigmoid(t_logits)
        return F.mse_loss(s_prob, t_prob)

    def rank_distill(self, s_logits, t_logits):
        """Pairwise ranking distillation (BPR-style)"""
        # 按教师分数排序，构造 pairwise
        batch_size = s_logits.size(0)
        if batch_size < 2:
            return torch.tensor(0.0, device=s_logits.device)

        # 随机采样 pair
        idx_i = torch.randint(0, batch_size, (batch_size,))
        idx_j = torch.randint(0, batch_size, (batch_size,))

        # 教师认为 i > j 的 pair
        t_diff = t_logits[idx_i] - t_logits[idx_j]
        mask = (t_diff > 0).float()

        # 学生也应该 i > j
        s_diff = s_logits[idx_i] - s_logits[idx_j]
        loss = -mask * F.logsigmoid(s_diff) - (1 - mask) * F.logsigmoid(-s_diff)

        return loss.mean()

    def forward(self, s_logits, t_logits, labels):
        # Hard label loss
        hard_loss = F.binary_cross_entropy_with_logits(s_logits, labels)
        # Logit distillation
        logit_loss = self.logit_distill(s_logits, t_logits)
        # Ranking distillation
        rank_loss = self.rank_distill(s_logits, t_logits)

        total = (1 - self.alpha - self.beta) * hard_loss \
                + self.alpha * logit_loss \
                + self.beta * rank_loss

        return total, {
            'hard': hard_loss.item(),
            'logit': logit_loss.item(),
            'rank': rank_loss.item()
        }
```
