## 技术问答（项目相关技术提问部分）

### Q1: 大模型在推理时分哪两个阶段，每个阶段的特点是什么？

**Prefill（预填充）阶段：**
- 一次性处理全部 input tokens，计算所有 token 的 KV Cache
- **计算密集型（Compute-bound）**：大量矩阵乘法并行执行，GPU 算力是瓶颈
- 延迟 = TTFT（Time To First Token）
- 可高度并行，算力利用率高

**Decode（解码）阶段：**
- 逐 token 自回归生成，每步只算 1 个新 token
- **访存密集型（Memory-bound）**：每步都要读取完整 KV Cache，但只做很少计算
- 瓶颈在显存带宽（HBM bandwidth），GPU 算力严重浪费
- Batch 越大越能摊薄访存开销（continuous batching 的动机）

**延伸思考：** 这也是 Prefill/Decode 分离调度（Splitwise, DistServe）的底层逻辑——两阶段 Arithmetic Intensity 差异巨大，适合不同硬件配置。

### Q2: 给了一小段代码，如果在推理时出现了 OOM，可能是什么原因？如何排查？

**常见原因：**
1. **KV Cache 爆显存**：长序列 + 大 batch，KV Cache 大小 = $2 \times L \times n_h \times d_k \times s \times b \times \text{dtype\_bytes}$。seq_len=4096, 70B 模型单条就要几十 GB
2. **模型权重 + 激活 + KV Cache 总和超限**：权重本身就占大量显存，留给 KV Cache 的空间不足
3. **动态 batch 突增**：continuous batching 下新请求涌入导致瞬时显存飙升
4. **显存碎片化**：反复分配释放导致虽有剩余但无法分配连续块

**排查方法：**
- `nvidia-smi` / `torch.cuda.memory_summary()` 看显存分配明细
- 打印 KV Cache 的实际占用大小，计算理论值对比
- 检查 max_batch_size 和 max_seq_len 配置是否合理
- 检查是否有显存泄漏（用 `torch.cuda.memory_allocated()` 在推理前后对比）
- 降低 batch size / 序列长度 / 使用 GQA / 量化 KV Cache 来缓解

### Q3: 输入 shape 为 (b,s,h)，详细计算 MHA 的计算量，包括 self-attention 和 MLP 部分

设 $b$ = batch, $s$ = seq_len, $h$ = hidden_dim, $n_h$ 个头, $d_k = h / n_h$。

**Self-Attention 部分：**
- QKV 投影：$3 \times 2bsh^2 = 6bsh^2$
- Attention Score $QK^T$：$2bs^2h$
- Score $\times$ V：$2bs^2h$
- Output 投影：$2bsh^2$
- **合计：$8bsh^2 + 4bs^2h$**

**MLP 部分（标准 FFN，中间维度 4h）：**
- 第一层 $[h, 4h]$：$8bsh^2$
- 第二层 $[4h, h]$：$8bsh^2$
- **合计：$16bsh^2$**

> 若用 SwiGLU（LLaMA 等），中间维度取 $8h/3$，gate+up+down 三投影，总量约 $16bsh^2$

**单层 Transformer 总计 = $24bsh^2 + 4bs^2h$**

当 $s \ll h$ 时 $4bs^2h$ 可忽略，主导项是 $24bsh^2$。

### Q4: 介绍下 vLLM 的 PagedAttention 原理

**核心思想：** 借鉴 OS 虚拟内存的分页机制管理 KV Cache。

**问题背景：** 传统为每个请求预分配 max_seq_len 的连续显存给 KV Cache，导致内部碎片（实际远短于 max）+ 外部碎片（不连续空洞）。

**做法：**
1. **KV Cache 分块（Block）：** 固定大小 block（如 16 tokens 的 KV）
2. **Block Table（页表）：** 逻辑→物理 block 映射
3. **非连续存储：** 物理 block 分散在显存任意位置
4. **按需分配：** 生成新 token 时才分配新 block
5. **Copy-on-Write：** beam search 共享前缀 KV block，修改时才复制

**收益：** 显存浪费从 60-80% 降到 <4%，吞吐量提升 2-4x。

→ 参考：[[mock-interview-llm.md#Q10]] | [[synthesis/llm/01_LLM推理优化全景.md]]

### Q5: 讲下 Flash Attention 的原理

**核心思想：** 通过 **tiling（分块）+ kernel fusion** 减少 HBM 访存。

**传统 Attention 的问题：**
- 需要 materialize 完整 $S = QK^T$（$s \times s$），写回 HBM
- 多次 HBM 读写：QK乘→写回→读回 softmax→写回→读回乘V
- **O(s²)** 的 HBM 读写是瓶颈

**Flash Attention 做法：**
1. **分块（Tiling）：** Q、K、V 按 seq 维度分小块，放进 SRAM
2. **Online Softmax：** 关键创新——逐块计算 softmax，维护 running max + running sum，修正之前结果
3. **Kernel Fusion：** QK乘法、softmax、AV乘法融成一个 CUDA kernel，中间结果全在 SRAM
4. **不存储 S 矩阵：** 反向传播时重算（recomputation），用计算换显存

**复杂度：** HBM 访存 $O(s^2) \to O(s^2d/M)$（M=SRAM 大小），显存 $O(s^2) \to O(s)$，实际加速 2-4x。

→ 参考：[[mock-interview-llm.md#Q1]] | [[synthesis/llm/01_LLM推理优化全景.md]]
