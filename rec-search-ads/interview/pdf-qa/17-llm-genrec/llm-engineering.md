# LLM 推荐系统的工程实践与优化

## 1. 推理延迟优化

### 1.1 模型优化

```
手段              原理                      加速比    精度损失
模型量化(INT8)    降低权重/激活值精度         2-4x     微小
模型量化(INT4)    更激进的精度降低            4-8x     需QAT
模型剪枝          移除冗余注意力头/层         1.5-2x   中等
知识蒸馏          大模型→小模型知识迁移       10-50x   可控
分组查询注意力    GQA替代多头注意力           1.3-1.5x 极小
动态退出          简单样本提前退出浅层         1.5-3x   几乎无
```

模型量化详解：
```
FP32 → FP16: 几乎无损，显存减半，推荐作为默认选项
FP16 → INT8: 需校准数据，体积减半，适合推理部署
INT8 → INT4: 较大损失，需量化感知训练(QAT)补偿
推荐组合: 权重INT8 + 激活FP16 = 效果和速度的最佳平衡
```

### 1.2 工程优化

```
手段              原理                      效果
vLLM             PagedAttention管理KV Cache  吞吐提升2-3x
动态Batching     聚合请求提升GPU利用率       吞吐提升1.5-3x
KV Cache         缓存已计算的Key/Value        延迟降低30-50%
持续批处理       不等batch满就开始推理        延迟降低40%
投机解码         小模型猜测+大模型验证        加速2-3x
流式输出         边生成边返回                 首Token延迟降低
```

vLLM 核心技术：
- PagedAttention：将 KV Cache 分页管理，解决内存碎片
- 持续批处理：不同请求的不同阶段可以在同一batch中
- 前缀缓存：相同系统 Prompt 的 KV Cache 共享

### 1.3 模型结构优化

```
Flash Attention: 优化注意力计算的IO效率，减少HBM访问
  → 训练和推理都能加速
  → 已成为标配

GQA (Grouped Query Attention):
  → 多个Query头共享一组KV头
  → KV Cache 减少 4-8x
  → Llama2/3, Qwen 等已采用

MoE (Mixture of Experts):
  → 稀疏激活，每次只用部分参数
  → 推荐领域潜力大：不同用户群激活不同Expert
```

## 2. 百亿级模型上线路径

### 2.1 完整流程

```
阶段1: 蒸馏
  千亿教师模型 → 百亿学生模型 (保留90%+效果)
  蒸馏方式: 模仿教师的输出分布(Soft Labels) + 中间层特征

阶段2: 剪枝
  分析学生模型 → 识别冗余注意力头/FFN维度
  结构化剪枝: 硬件友好，直接减少计算量

阶段3: 量化
  FP32 → INT8 (GPTQ/AWQ)
  量化感知训练缓解精度损失

阶段4: 部署
  vLLM/TensorRT-LLM + 动态batching
  多卡推理(tensor parallelism)
```

### 2.2 效果-成本权衡

```
                效果保留    推理延迟    部署成本
原始千亿模型    100%        ~5s         极高(多机多卡)
蒸馏百亿模型    92-95%      ~500ms      高(单机多卡)
剪枝+量化       88-92%      ~100ms      中(单卡)
蒸馏到7B+INT8   85-90%      ~50ms       低(单卡)
```

## 3. 在线推理服务架构

### 3.1 核心组件

```
API网关
  → 请求路由、身份认证、限流熔断、日志记录
  ↓
特征服务中心
  → 实时特征获取(Redis: 点击序列/用户画像/物品属性)
  → 特征拼接与序列化(多源特征→结构化Prompt)
  ↓
LLM推理服务(GPU集群)
  → vLLM/TensorRT-LLM
  → 持续批处理+PagedAttention
  → 量化推理(INT8)
  ↓
输出解析与后处理
  → 结果解析(正则/JSON解析提取item_id列表)
  → 去重、多样性控制、强插逻辑
  → 业务规则校验
  ↓
缓存层
  → Redis多级缓存(用户维度/请求维度)
  → 避免重复LLM推理计算
```

### 3.2 服务治理

```
降级策略:
  LLM超时(>200ms) → 自动切换到规则重排/传统排序模型
  GPU负载>80% → 降级到轻量模型
  全部不可用 → 返回热门榜单/缓存结果

健康检查:
  每30s探测LLM服务健康状态
  unhealthy节点自动摘除,恢复后自动加入

流量控制:
  令牌桶限流 → 保护GPU不被打满
  优先级队列 → VIP用户优先处理
```

## 4. 增量更新与持续学习

### 4.1 增量更新策略

```
全量重训太贵 → PEFT/LoRA适配器增量训练

触发条件:
  - 新数据量达阈值(如100万新样本)
  - 在线指标下降(CTR/CVR连续3天低于基线)
  - 定时(每日/每周)

双Buffer模型更新:
  Buffer A: 当前服务模型
  Buffer B: 后台加载新LoRA适配器
  验证通过 → 原子切换A↔B → 旧Buffer回收
```

### 4.2 LoRA 在推荐场景的应用

```
冻结LLM主体 → 加LoRA适配器(低秩矩阵)
  → 用推荐数据(曝光-点击对)微调
  → 0.1%-1%参数量达到full finetune 90%+效果

优势:
  - 多任务: 不同场景(搜索/推荐/广告)各挂一个LoRA
  - 快速迭代: 新适配器训练时间小时级
  - 回滚简单: 切回旧适配器即可
```

### 4.3 在线学习 vs 批量更新

```
维度         在线学习                    批量更新
更新频率     秒/分钟级                   天/周级
实时性       极高,捕捉即时兴趣           延迟高
稳定性       易受噪声/攻击影响           稳定
实现复杂度   高(流处理+在线训练)         低(离线训练)
推荐实践     实时特征更新比模型更新收益大 主流方案
```

混合策略（工业界最佳实践）：
- 批量更新训练大的基准模型（每日/每周）
- 在线学习在基准模型上做轻量微调（FTRL等）
- 实时特征更新（分钟级）比模型更新收益更大

## 5. A/B 测试与效果评估

### 5.1 LLM 推荐的评估指标体系

```
层级          指标                        说明
内容质量      推荐理由通顺率              LLM生成质量
              幻觉率                      编造不存在的item比例
              重复率                      同一理由重复使用比例
交互深度      CTR                         点击率
              停留时长                    用户参与度
              对话轮数(对话式推荐)        用户满意度代理
列表质量      覆盖率                      推荐item覆盖catalog的比例
              多样性(ILS)                 列表内item间的平均距离
              新颖性                      推荐长尾item的比例
语义匹配      query-item语义相似度        意图满足程度
              意图满足率                  LLM理解用户意图的准确度
长期留存      7日/30日留存                长期影响
              NPS(净推荐值)              用户推荐意愿
```

### 5.2 LLM 特有的评估维度

传统指标(CTR/CVR)不足以评估 LLM 推荐：
- 可解释性：推荐理由是否合理准确
- 多样性：LLM 强语义匹配可能导致信息茧房
- 事实准确性：生成内容是否有幻觉
- 新颖性和惊喜度：是否推荐了用户意想不到但喜欢的内容

评估方法：
```
自动评估: 幻觉检测(item_id存在性校验) + 多样性指标(ILS)
LLM-as-Judge: 用另一个LLM评估推荐理由的质量(0-5分)
人工评估: 标注员评估推荐相关性+解释合理性(成本高但最准)
```

## 6. 偏差与幻觉的工程解决方案

### 6.1 位置偏差

```
问题: LLM倾向推荐Prompt中靠前/靠后的item (Primacy/Recency)

解决:
1. 候选列表随机打乱后输入LLM
2. 多次推理取共识(投票/交集)
3. 位置去偏校准: 统计各位置的偏差 → 后处理校正
```

### 6.2 流行度偏差

```
问题: LLM预训练数据中热门item出现频率高 → 倾向推荐热门

解决:
1. Prompt中显式加入长尾item的曝光
2. 后处理: 控制热门/冷门item比例
3. 对比学习去偏: 拉远热门和冷门物品的Embedding
```

### 6.3 幻觉问题（三层防线）

```
数据层: RAG检索真实item信息注入Prompt
  → LLM只能从提供的真实候选中选择

模型层: Grounded generation
  → constrained decoding约束输出必须来自候选集
  → JSON schema强制输出格式

后处理层: 输出校验
  → item_id存在性检查
  → 属性一致性验证(推荐理由中的属性 vs 实际属性)
  → 不通过的item用规则替补
```

## 7. 数据流一致性与实时特征

### 7.1 Kafka + Flink 实时特征管道

```
用户行为(点击/购买) → Kafka → Flink实时处理
  → 实时特征计算(最近10min点击品类分布/实时兴趣向量)
  → 写入在线特征库(Redis)
  → 推荐服务实时读取最新特征

Flink保证:
  - 精确一次状态一致性(两阶段提交)
  - 事件时间处理(处理乱序事件)
  - 检查点机制(故障快速恢复)
```

### 7.2 特征一致性问题

```
问题: 离线训练特征 vs 在线推理特征可能不一致
  → 离线用T-1天全量数据计算, 在线用实时窗口计算
  → 分布差异导致模型效果下降

解决:
  - 特征平台统一管理: 训练和推理共用同一套特征计算逻辑
  - 特征日志回放: 在线推理时记录实际使用的特征,用于离线训练
  - PSI监控: 定期检测线上特征分布 vs 训练特征分布的偏移
```

## 8. 模型漂移、高可用与安全

### 8.1 模型漂移应对

三种漂移：突然漂移(热点事件) / 渐进漂移(用户群变化) / 周期性漂移(工作日vs周末)

应对：在线学习实时更新(FTRL) + 多模型集成动态选择 + PSI监控自动触发重训

### 8.2 高可用容灾

```
多级降级:
  L1: LLM重排超时 → 规则重排  L2: 精排故障 → 粗排结果
  L3: 召回故障 → 热门/缓存    L4: 全部故障 → 运营兜底

多级缓存: 本地(100ms TTL) → Redis(5min) → 向量数据库
微服务: 召回/粗排/精排/重排独立部署, gRPC通信, K8s编排
熔断: 连续3次超时自动熔断, 10s探测恢复, 半开放小流量
```

### 8.3 安全与隐私

- Prompt 用户数据脱敏(ID哈希/敏感行为不入Prompt)
- Prompt 注入防护(输入过滤 + 系统/用户Prompt隔离 + 输出审核)
- 合规：GDPR/个保法(用户删除权) + 算法备案

## 面试高频问题

Q: LLM做推荐最大的工程挑战？
A: 推理延迟和成本。核心策略：蒸馏到小模型 + 离线预计算 + 结果缓存 + 量化部署(INT8) + vLLM推理优化。线上LLM只用于重排阶段的小候选集。

Q: 百亿模型怎么上线？
A: 四步流水线：蒸馏(千亿→百亿/7B) → 剪枝(去冗余头/层) → 量化(INT8/GPTQ) → 部署(vLLM+动态batching)。实际推荐场景中蒸馏到7B+INT8就够用。

Q: 在线学习 vs 批量更新怎么选？
A: 工业界主流是混合策略：批量更新训练基准模型(每日/每周) + 在线学习做轻量微调。关键洞察：实时特征更新(分钟级)比模型更新的收益更大。

Q: LLM推荐怎么保证高可用？
A: 多级降级是核心。LLM超时→规则重排→缓存结果→运营兜底。同时多级缓存减少LLM调用频次，健康检查+熔断防止级联故障。

Q: 数据流一致性怎么保证？
A: 特征平台统一管理训练和推理的特征计算逻辑；用Flink精确一次语义保证实时特征不丢不重；PSI监控特征分布偏移。
