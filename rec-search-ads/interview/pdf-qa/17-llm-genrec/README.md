## 大语言模型与生成式推荐

LLM 在推荐系统中的角色：从特征增强器到端到端推荐引擎。核心矛盾是 LLM 的强语义理解能力 vs 推理延迟/成本约束。

深度子文件：
- [llm-recall-ranking.md](llm-recall-ranking.md) -- LLM在召回/精排/重排全链路应用、三种排序范式(Pointwise/Pairwise/Listwise)、Embedding增强、序列推荐融合
- [llm-engineering.md](llm-engineering.md) -- 推理优化(量化/vLLM/蒸馏)、百亿模型上线、在线服务架构、增量更新、A/B测试、偏差与幻觉、高可用容灾

## 1. LLM + 推荐的基本范式

**三种融合模式**

```
模式          | 思路                        | 延迟  | 效果  | 成本
特征增强      | LLM 离线生成特征，传统模型在线 | 低    | 中高  | 低
重排器        | 传统模型召回排序，LLM 做重排   | 中    | 高    | 中
端到端        | LLM 直接做推荐                | 高    | 最高  | 高
```

实践中最常用：特征增强（离线 LLM 提取语义 Embedding → 注入精排模型），兼顾效果与成本。

**RAG 增强推荐**
- 问题：LLM 训练数据静态，缺少实时 item 信息
- 方案：检索候选 item 信息 → 注入 Prompt → LLM 基于实时上下文推荐
- 流程：用户 query → 向量检索 top-K item → 构造 Prompt（用户画像 + item 描述）→ LLM 生成推荐 + 理由
- 关键：检索质量决定上限，LLM 决定下限

**结构化输出约束**
- LLM 输出不可控 → 需要约束为推荐系统可解析的格式
- 方法：JSON schema 约束 + constrained decoding + 后处理校验
- 示例 Prompt："请从以下候选中选择5个最适合用户的，以JSON数组格式输出 item_id 和推荐理由"

## 2. 偏差与幻觉

LLM 推荐的三大风险：

**位置偏差**
- LLM 倾向于推荐 Prompt 中靠前/靠后的 item（Primacy/Recency bias）
- 解决：候选列表随机打乱 + 多次推理取共识 + 位置去偏校准

**流行度偏差**
- LLM 预训练数据中热门 item 出现频率高 → 倾向推荐热门
- 解决：Prompt 中加入长尾 item 曝光 + 后处理提升长尾比例 + 对比学习去偏

**幻觉问题**
- LLM 可能编造不存在的 item 或虚假属性
- 解决：三层防线 —
  1) 数据层：RAG 检索真实 item 信息注入
  2) 模型层：Grounded generation，约束输出必须来自候选集
  3) 后处理层：输出校验（item_id 存在性检查 + 属性一致性验证）

## 3. 对话式推荐

四模块架构：
```
用户输入 → [意图解析] → [状态管理] → [推荐引擎] → [回复生成]
                |              |              |              |
           NLU/LLM解析    多轮偏好追踪    候选检索+排序    自然语言回复
           slot filling   belief state    个性化过滤      解释+追问
```

关键设计：
- 意图识别：推荐请求 / 信息查询 / 偏好修正 / 闲聊，LLM 可统一处理
- 多轮偏好追踪：对话历史 → 用户偏好 state（品类、价格区间、风格等 slot）→ 增量更新
- 主动追问策略：当偏好信息不足时主动问 "你更看重性价比还是品牌？"
- 推荐多样性：对话中避免反复推荐同类，需要探索-利用平衡

## 4. LLM + 知识图谱 + 多模态

**LLM + GNN 融合**
- LLM 擅长语义理解，GNN 擅长图结构建模，互补
- 融合方式：
  1) LLM as Feature Extractor：LLM 生成节点文本 Embedding → 输入 GNN
  2) GNN as Structure Encoder：GNN 学图结构 → 与 LLM 语义 Embedding 拼接
  3) 联合训练：LLM + GNN 端到端，但成本极高

推荐场景示例：用户-商品知识图谱 → GNN 学习结构关系 → LLM 补充商品描述语义 → 融合后精排

**多模态特征对齐**
- 问题：文本/图片/视频在不同特征空间
- 方案：CLIP 式对比学习，将多模态统一到共享向量空间
- 推荐应用：商品图片 + 标题文本 → 统一 Embedding → 多模态召回

**LLM 增强 Item 侧特征**
- 用 LLM 从商品描述中提取结构化标签（风格、场景、人群）
- 用 LLM 生成商品卖点摘要（替代人工运营文案）
- 用 LLM 补充缺失属性（类目预测、品牌识别）

**LoRA 轻量级协同训练**
- 全量微调 LLM 成本太高 → LoRA 冻结主体、只训低秩适配器
- 三阶段流程：1) 预训练 LLM 2) LoRA 适配推荐任务（对齐用户行为信号） 3) 与下游模型联合微调
- 参数量：LoRA 仅增加 0.1%-1% 参数，显存降 3-4x

## 5. 工程实践

**推理优化**
```
手段           | 原理                        | 加速比
量化 INT8/FP16 | 降低精度                    | 2-4x
KV Cache       | 缓存已计算的 Key/Value       | 1.5-2x
PagedAttention | KV Cache 分页管理减少碎片    | 1.3-1.8x
投机解码       | 小模型猜测 + 大模型验证       | 2-3x
动态 Batching  | 聚合请求提升 GPU 利用率      | 1.5-3x
模型蒸馏       | 大模型知识迁移到小模型        | 10-50x
```

**百亿模型上线路径**
1. 蒸馏：100B teacher → 7B student（保留 90%+ 效果）
2. 剪枝：去掉冗余注意力头和层
3. 量化：FP16 → INT8（GPTQ/AWQ）
4. 部署：vLLM/TensorRT-LLM + 动态 batching

**在线服务架构**
```
API Gateway → 特征中心（Redis）→ LLM 推理（GPU 集群）→ 输出解析 → 结果缓存
                                        |
                                   模型版本管理（双 buffer 热切换）
```

关键：结果缓存（相似 query 命中缓存避免重复推理） + 异步预计算（离线批量生成推荐理由）

**增量更新**
- 全量重训太贵 → PEFT/LoRA 适配器增量训练
- 双 buffer 模型更新：新适配器加载到备用 buffer → 原子切换 → 旧 buffer 回收
- 触发条件：新数据量达阈值 or 在线指标下降 or 定时（每日/每周）

**A/B 测试指标体系**
```
层级       | 指标
内容质量   | 推荐理由通顺率、幻觉率、重复率
交互深度   | 点击率、停留时长、对话轮数
列表质量   | 覆盖率、多样性(ILS)、新颖性
语义匹配   | query-item 语义相似度、意图满足率
长期留存   | 7日留存、30日留存、NPS
```

## 6. LLM 在召回阶段

**语义 Embedding 召回**
- LLM 生成 item 文本 Embedding → 建 ANN 索引（Faiss/Milvus）
- 用户 query → LLM Embedding → ANN 检索 top-K
- vs Item2Vec：LLM Embedding 语义更丰富，冷启动效果好；但维度高、计算贵

**混合召回架构**
```
行为召回（协同过滤/Swing/ItemCF）──┐
                                    ├──→ 融合排序器 → 精排
语义召回（LLM Embedding + ANN）──┘
```
- 行为召回捕捉用户历史偏好，语义召回捕捉 query 意图和 item 内容
- 融合策略：加权合并 or 学习型融合（小模型学两路召回的最优权重）

**用户兴趣摘要**
- 用 LLM 将用户行为序列总结为自然语言兴趣描述
- Prompt："用户最近浏览了[item列表]，总结其核心兴趣偏好，不超过3个关键词"
- 摘要作为召回 query 的增强信号

**工程优化**
- Embedding 预计算：离线批量生成 item Embedding，增量更新新 item
- 向量检索加速：PQ 量化 + HNSW 索引，百万级 item <10ms 检索
- 蒸馏小模型：LLM Embedding → 蒸馏到轻量 Encoder（BERT-base 级别），在线使用

## 7. LLM 在精排阶段

**两种接入方式**
```
方式1（特征增强）：LLM 离线生成语义特征 → 注入传统精排模型（DeepFM/DIN）
  → 延迟不增加，效果提升 2-5%

方式2（LLM 重排器）：传统模型粗排 top-50 → LLM Listwise 重排
  → 效果最好，但延迟高（需控制候选数量）
```

**LLM Embedding 增强精排**
- 提取：LLM 编码 item 标题/描述 → 取 [CLS] 或 mean pooling 作为语义 Embedding
- 对齐：LLM Embedding 与业务 Embedding 维度对齐（线性投影 or 对比学习）
- 融合：拼接 or 门控融合（`gate = sigmoid(W * [llm_emb; biz_emb])`, `output = gate * llm_emb + (1-gate) * biz_emb`）

**LLM vs 传统精排模型对比**
```
维度         | DeepFM/DIN        | LLM 精排
特征理解     | 交叉特征为主       | 语义理解+交叉
冷启动       | 差                | 好（语义泛化）
推理延迟     | <10ms             | 100ms-1s
可解释性     | 特征重要性         | 自然语言解释
迭代成本     | 低                | 高
```

**分布漂移与冷启动**
- LLM Embedding 来自预训练语料，与推荐场景有分布偏差
- 解决：领域适配微调（推荐语料 continue pretrain） + 在线特征分布监控（PSI 告警）
- 新 item 冷启动：LLM 从标题/描述直接生成 Embedding，无需行为数据

**混合排序框架（LLM + GBDT/DNN）**
```
离线层：LLM 生成语义特征 + 传统特征工程
准实时层：Flink 窗口特征 + 用户最新行为序列
在线层：轻量 DNN/GBDT 融合所有特征 → 实时打分
```

## 8. 排序模型设计

三种排序范式：

**Pointwise**
- 每个 item 独立打分：`score = LLM("用户画像:{profile}, 商品:{item}, 请打1-10分")`
- 优点：简单，可并行
- 缺点：忽略 item 间比较关系
- 损失：MSE 或 BCE

**Pairwise**
- 两两比较：`LLM("用户更可能点击A还是B？")`
- 优点：捕捉相对偏好
- 缺点：O(n^2) 比较次数
- 损失：BPR loss = -log(sigmoid(s_pos - s_neg))

**Listwise**
- 整体排序：`LLM("请将以下N个商品按用户可能的点击概率排序")`
- 优点：全局最优排列
- 缺点：候选数受 context window 限制（通常 20-50 个）
- 损失：ListNet (top-1概率分布KL散度) 或 LambdaRank

实践选择：
```
场景           | 推荐范式   | 原因
精排 top-50    | Listwise   | 候选少，全局排序效果最好
粗排 top-500   | Pointwise  | 候选多，需要并行高效
重排 top-20    | Listwise   | 候选最少，可加业务约束到 Prompt
```

## 9. 特征工程中的 LLM

**vs 传统方法对比**
```
方法         | 优势                    | 劣势
TF-IDF       | 快、可解释              | 无语义，稀疏
Word2Vec     | 捕捉语义相似            | 无上下文，OOV 问题
BERT         | 上下文理解              | 领域适配需微调
LLM          | 零样本理解、多任务复用   | 成本高、延迟大
```

**冷启动特征增强**
- 新 item：LLM 从标题/描述生成类目预测、标签提取、质量评分
- 新用户：LLM 从注册信息/首次行为推断兴趣画像
- 效果：冷启动 item CTR 提升 15-30%（行业经验）

**长文本结构化提取**
- 输入：商品详情页（500-2000字）
- 输出：JSON 格式（品牌、材质、适用人群、核心卖点、使用场景）
- 方法：Few-shot Prompt + JSON schema 约束 + 后处理校验

## 10. 前沿与伦理

**可解释性挑战**
- LLM 是黑箱 → 生成式推荐难以解释 "为什么推荐这个"
- 解决方向：Chain-of-Thought 推理 + 归因分析（哪些输入影响了输出）
- 幻觉式解释：LLM 生成的推荐理由听起来合理但可能与实际模型决策无关

**公平性与伦理**
- 偏差放大：LLM 预训练偏差 + 推荐马太效应 = 双重偏差
- 信息茧房：LLM 强语义匹配 → 更容易推荐同质内容 → 需要主动注入探索
- 隐私风险：Prompt 中注入用户行为数据 → 需要脱敏处理

**因果推理 + LLM**
- 从相关推荐到因果推荐：不只是 "看了A的人也看B"，而是 "因为A所以喜欢B"
- 反事实推理："如果用户没有看过这部电影，还会喜欢这个演员吗？"
- LLM 作为因果推理引擎：Prompt 构造因果问题 → LLM 推理 → 去混淆

## 面试高频考点速查

```
Q: LLM 做推荐最大的工程挑战是什么？
A: 推理延迟和成本。解决：蒸馏小模型 + 离线预计算 + 结果缓存 + 量化部署。

Q: LLM 推荐的幻觉问题怎么解决？
A: 三层防线：RAG 注入真实信息 + 约束生成在候选集内 + 后处理校验。

Q: LLM 适合放在推荐链路的哪个位置？
A: 性价比最高是离线特征增强；效果最好是 Listwise 重排；端到端成本最高。

Q: LLM Embedding 和 Item2Vec 有什么区别？
A: Item2Vec 基于行为共现（谁和谁一起被点击），LLM Embedding 基于语义理解（内容相似）。
   互补使用效果最好。

Q: 如何解决 LLM 推荐的位置偏差？
A: 候选列表随机打乱 + 多次推理取交集/投票 + 校准层去偏。

Q: LoRA 在推荐场景怎么用？
A: 冻结 LLM 主体 → 加 LoRA 适配器 → 用推荐数据（曝光-点击对）微调 →
   0.1% 参数量达到 full finetune 90%+ 效果。

Q: Listwise 排序的 Prompt 怎么设计？
A: "用户画像:{profile}。请将以下商品按点击概率从高到低排序，输出 item_id 列表：{items}"
   关键：控制候选数（20-50）、明确输出格式、加入排序依据要求。
```
