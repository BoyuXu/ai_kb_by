# LLM 在召回-排序-重排全链路的深度应用

## 1. LLM 在推荐系统中的五大作用

```
作用              机制                          核心价值
深度语义特征      编码用户行为/物品内容          高质量Embedding，缓解冷启动
序列建模          将行为序列视为"语言"          捕捉动态兴趣和意图
生成式推荐        推荐定义为Token生成任务        端到端，可生成解释
可解释性          生成自然语言推荐理由            增强用户信任
知识推理          理解复杂查询和物品关系          互补性/替代性识别
```

## 2. LLM 在召回阶段

### 2.1 语义 Embedding 召回

```
离线:
  物品标题/描述 → LLM → 取[CLS]或mean pooling → 物品语义Embedding
  → 建ANN索引(Faiss/Milvus/HNSW)

在线:
  用户query/行为序列 → 轻量Embedding模型(蒸馏后) → 查询向量
  → ANN检索 top-K
```

vs Item2Vec：
- Item2Vec 基于行为共现（谁和谁一起被点击）
- LLM Embedding 基于语义理解（内容相似）
- LLM 冷启动效果好（无需行为数据），但维度高、计算贵
- 最佳实践：两路互补

### 2.2 混合召回架构

```
行为召回(协同过滤/Swing/ItemCF) ─┐
                                   ├→ 融合排序器 → 精排
语义召回(LLM Embedding + ANN)  ─┘
```

融合策略：
- 简单：加权合并（行为召回权重0.6 + 语义召回权重0.4）
- 进阶：学习型融合（小模型学两路召回的最优权重）
- 去重：两路重叠物品只保留一份，取较高分

### 2.3 用户兴趣摘要增强召回

```
Prompt: "用户最近浏览了[item列表]，总结其核心兴趣偏好，不超过3个关键词"
输出: "运动鞋、跑步装备、户外运动"
→ 关键词作为召回query的增强信号
→ 也可直接编码为向量做语义检索
```

### 2.4 工程优化

```
策略            说明                        效果
Embedding预计算  离线批量生成,增量更新新item  消除在线LLM推理
向量检索加速     PQ量化+HNSW索引             百万级item <10ms
蒸馏小模型       LLM Embedding→蒸馏到BERT-base 在线使用,效果保留90%+
多级缓存         热门item Embedding缓存Redis  减少向量库查询
```

## 3. LLM 在精排阶段

### 3.1 两种接入方式

方式1 — 特征增强（推荐）：
```
离线: LLM编码item标题/描述 → 语义Embedding
在线: 语义Embedding作为精排模型(DeepFM/DIN)的额外特征
效果: CTR提升2-5%, 延迟不增加
```

方式2 — LLM 重排器：
```
传统模型粗排 top-50 → LLM Listwise重排
LLM("用户画像:{profile}, 请将以下商品按点击概率排序: {items}")
效果最好, 但延迟高(需控制候选数量)
```

### 3.2 LLM Embedding 对齐与融合

提取：
```
物品标题 → LLM → [CLS]向量 或 mean pooling → 语义Embedding (d=768/4096)
```

对齐（LLM空间 → 推荐空间）：
```
方案1: 线性投影  v_aligned = W * v_llm + b
方案2: 对比学习  拉近同物品的LLM和业务Embedding
方案3: 适配器    2层MLP将LLM维度投影到业务维度
```

融合：
```
方案1: 拼接      v_final = [v_llm_aligned; v_biz]
方案2: 门控融合  gate = sigmoid(W * [v_llm; v_biz])
                 v_final = gate * v_llm + (1-gate) * v_biz
方案3: 交叉注意力 (效果最好但最复杂)
```

### 3.3 分布漂移与冷启动

- LLM Embedding 来自预训练语料，与推荐场景有分布偏差
- 解决：领域适配微调（推荐语料 continue pretrain）
- 在线监控：PSI (Population Stability Index) 告警分布漂移
- 冷启动优势：新 item 无需行为数据，LLM 从标题/描述直接生成 Embedding

## 4. LLM 增强物品侧特征

### 4.1 直接抽取法

```
物品标题 → LLM生成结构化特征描述(核心功能/适用场景/卖点/风格)
  → Sentence-BERT编码为固定维度向量
  → 作为物品的增强语义特征
```

Prompt 设计：
```
"请为以下商品生成一个结构化的特征描述，包括：
核心功能、适用场景、主要卖点和风格特点。
商品信息: [商品标题]"
```

### 4.2 间接增强法

```
物品原始文本(标题+描述)
  → 微调后的LLM
  → 取最后一层[CLS]输出向量 或 所有token均值
  → 作为深度语义Embedding
```

### 4.3 特征融合应用

```
增强特征 → 与原有特征(ID Embedding/类别/价格/统计特征)融合
  → 输入排序模型(DeepFM/DIN)联合训练
```

### 4.4 LLM 作为生成式特征引擎

```
输入: 用户历史行为 + 当前候选物品
LLM生成:
  - "推荐理由": 自然语言解释
  - "匹配度分数": 0-10分
  → 这些生成结果本身作为新特征输入排序模型
```

优势：特征具有强解释性
挑战：LLM推理成本高 → 用小型开源模型(Qwen/ChatGLM) + 自有算力微调

## 5. 三种排序范式

### 5.1 Pointwise

```
每个item独立打分:
score = LLM("用户画像:{profile}, 商品:{item}, 请打1-10分")
```
- 优点：简单，可并行处理
- 缺点：忽略 item 间比较关系
- 损失：MSE 或 BCE
- 适用：粗排 top-500（候选多，需并行高效）

### 5.2 Pairwise

```
两两比较:
LLM("用户更可能点击A还是B？")
```
- 优点：捕捉相对偏好
- 缺点：O(n^2) 比较次数
- 损失：BPR loss = -log(sigmoid(s_pos - s_neg))
- 适用：中等规模候选

### 5.3 Listwise

```
整体排序:
LLM("请将以下N个商品按用户点击概率从高到低排序")
```
- 优点：全局最优排列
- 缺点：候选数受 context window 限制（20-50个）
- 损失：ListNet (top-1概率KL散度) 或 LambdaRank
- 适用：精排 top-50 / 重排 top-20

### 5.4 实践选择

```
场景          范式       原因
粗排 top-500  Pointwise  候选多,需并行高效
精排 top-50   Listwise   候选少,全局排序效果最好
重排 top-20   Listwise   候选最少,可加业务约束到Prompt
```

## 6. LLM 集成到多阶段推荐架构

### 6.1 完整方案设计

```
召回阶段 (LLM as Embedding Generator)
  离线: LLM为物品生成语义Embedding → 存入向量数据库
  在线: 轻量化Embedding模型(蒸馏后)编码用户查询 → ANN检索
  核心: 预计算+向量化检索保证效率

排序阶段 (LLM as Feature Generator)
  离线: LLM对用户行为序列做深度分析,生成富语义特征
    - 用户长期兴趣概括
    - 用户意图识别("寻求性价比高的电子产品")
    - 用户-候选商品多维匹配度评分
  在线: 排序模型(DeepFM/DIN)使用预先生成的LLM特征
  核心: LLM不直接参与在线推理,完全规避延迟问题

重排阶段 (LLM as Direct Reranker)
  在线: 候选集已缩小至top-K(10-20个)
  LLM对精排后的列表做优化:
    - 多样性打散(避免同质化商品扎堆)
    - 理由生成(每个推荐项生成解释性文案)
    - 敏感信息过滤
    - 新颖性探索
  核心: 异步处理,先返回默认排序,再流式更新推荐理由
```

### 6.2 效率与效果的平衡策略

```
策略          说明
特征预计算    所有LLM特征离线批量处理,线上无感
蒸馏+量化     召回和特征生成用蒸馏小模型+INT8量化
多级缓存      高频用户/商品的LLM结果缓存Redis
服务降级      LLM超时自动降级到规则重排,保证可用性
超时控制      重排阶段LLM调用设严格超时+vLLM推理优化
```

## 7. LLM 与序列推荐模型融合

### 7.1 语义增强的序列 Transformer 框架

```
模块1: 行为序列 + 文本信息获取
  用户行为序列: [item1, item2, ..., itemN]
  每个item收集: 物品ID + 标题 + 描述

模块2: LLM语义编码
  每个item的文本 → 冻结/轻量化LLM → 语义嵌入向量
  (离线预计算,在线读取)

模块3: 序列建模
  物品ID Embedding + LLM语义嵌入 → 拼接/加权/门控
  → 增强的物品表示序列 → Transformer编码器
  → 自注意力建模序列中的依赖关系

模块4: 预测
  Transformer最后位置输出 → 预测层 → 下一个物品
  端到端训练(LLM部分可冻结以节省计算)
```

### 7.2 核心价值

- 物品ID Embedding 捕捉协同过滤信号
- LLM语义Embedding 捕捉内容理解
- 二者融合 = 协同信号 + 语义信号，互补增强

## 面试高频问题

Q: LLM 在推荐链路的最佳位置？
A: 性价比最高是离线特征增强（LLM离线生成语义Embedding/特征 → 注入精排模型），延迟不增加效果提升2-5%。效果最好是Listwise重排但延迟高。端到端成本最高不推荐。

Q: LLM Embedding 和 Item2Vec 有什么区别？
A: Item2Vec基于行为共现（谁和谁一起被点击），LLM Embedding基于语义理解（内容相似）。互补使用效果最好：行为召回覆盖热门+高频共现，语义召回覆盖长尾+冷启动。

Q: Listwise排序的Prompt怎么设计？
A: "用户画像:{profile}。请将以下商品按点击概率从高到低排序，输出item_id列表：{items}"。关键：控制候选数(20-50)、明确输出格式、加入排序依据要求。

Q: LLM增强物品特征的两种方法？
A: 直接法：LLM生成结构化描述 → 句子编码器转向量。间接法：物品文本直接输入微调LLM → 取隐藏层输出作Embedding。直接法更灵活可解释，间接法语义更深但需微调。

Q: 如何保证LLM推荐的效率？
A: 核心原则是"LLM离线生成，在线轻量使用"。所有LLM特征预计算缓存；蒸馏到小模型做在线查询编码；重排阶段异步调用+超时降级；结果多级缓存避免重复计算。
