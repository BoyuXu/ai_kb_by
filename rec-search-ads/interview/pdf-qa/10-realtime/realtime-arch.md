# 实时推荐系统架构与特征工程

## 1. 实时推荐 vs 离线推荐

### 1.1 核心区别

```
              离线推荐                    实时推荐
兴趣捕捉      历史长期兴趣（天/周级）       即时意图（秒/分钟级）
模型更新      批量重训（天/周）             在线学习（分钟/秒）
特征时效      T+1 离线特征                 实时流式特征
典型场景      用户兴趣稳定的场景            新闻/短视频/电商实时场景
用户体验      静态推荐列表                  动态响应行为变化
```

### 1.2 实时性的三个层级

```
秒级实时：
  对用户单次会话内行为的响应
  例：用户刚浏览手机壳 → 信息流立即出现手机配件
  技术：在线推理 + 实时特征拼接

分钟级近实时：
  热点挖掘和趋势捕捉
  例：突发新闻 → 推荐系统快速分发
  技术：Flink 窗口聚合 + 近线模型更新

小时级准实时：
  快速迭代模型
  例：每小时增量训练排序模型
  技术：增量训练 + 模型热加载
```

### 1.3 实时推荐的业务价值

电商场景：
- 用户购买意图短暂且冲动，实时响应直接关联 GMV
- 实时行为辅助清库存和长尾商品曝光策略

资讯/短视频场景：
- 内容价值随时间衰减极快
- 实时追踪用户关注点演变，推荐深度报道形成消费闭环

## 2. 流式数据处理架构

### 2.1 Lambda 架构

```
                ┌─── 批处理层（Batch Layer）───┐
原始数据 ───→   │   离线全量计算，保证准确性      │──→ 服务层
                │   技术：Spark/Hive，天级更新    │    （合并
                └──────────────────────────────────┘     两路
                ┌─── 速度层（Speed Layer）────┐         结果）
原始数据 ───→   │   实时流处理，保证时效性      │──→
                │   技术：Flink，秒/分钟级更新  │
                └──────────────────────────────┘

优点：容错性好，批处理可修正实时误差
缺点：维护两套代码逻辑，一致性保证困难
```

### 2.2 Kappa 架构

```
原始数据 ──→ 消息队列（Kafka，持久化）──→ 流处理层（Flink）──→ 服务层
                      ↑
              需要回溯时重放历史消息流

优点：架构简单，代码只有一套
缺点：大规模回溯开销大
```

### 2.3 工业实践：Lambda 变体为主流

```
大厂多用 Lambda 变体：
  流批一体思想：用同一套特征定义 + 不同的计算引擎
  Flink 同时处理实时流和微批量
  统一特征平台管理离线和实时特征的计算与存储
```

## 3. 实时特征工程

### 3.1 实时特征类型

```
用户短期兴趣特征：
  最近 N 次点击的物品 ID 序列
  最近 M 分钟的行为统计（点击数、停留时长均值）
  当前会话的品类点击分布

物品实时统计特征：
  最近 1 小时的曝光/点击/收藏/购买次数
  实时热度分（热度飙升检测）

用户-物品交叉实时特征：
  用户最近 30 分钟对候选物品所属品类的浏览时长/点击次数

全局实时热门趋势：
  最近 5 分钟热度飙升最快的物品列表

上下文特征：
  时间（工作日/周末、上午/下午/深夜）
  地理位置、设备类型、网络环境
```

### 3.2 Flink 实时特征计算流程

```
Step 1: 数据源接入
  用户行为事件（点击/浏览/购买）→ Kafka 消息队列
  数据格式：{user_id, item_id, action, category, timestamp}

Step 2: 流式计算（Flink）
  keyBy(user_id) → 按用户分区
  滑动窗口（如 window=10min, slide=1min）→ 窗口内聚合
  计算：品类点击分布、点击率、行为序列

  示例代码逻辑：
  stream
    .keyBy(event -> event.userId)
    .window(SlidingEventTimeWindows.of(Time.minutes(10), Time.minutes(1)))
    .aggregate(new CategoryDistributionAggregator())
    → 输出：{user_id, window_end, feature: {"tech": 0.6, "sports": 0.4}}

Step 3: 写入在线特征存储
  Redis/Tair，key=user:{userId}
  多个特征组织为 Hash/Protobuf
  设置 TTL（如 30 分钟），用户不活跃后自动过期

Step 4: 线上推理时拉取
  推荐服务收到请求 → 从 Redis 拉取实时特征
  与离线特征拼接 → 送入排序模型推理
```

### 3.3 Flink 关键技术点

```
事件时间 vs 处理时间：
  推荐使用事件时间（Event Time）+ 水印（Watermark）
  水印 = 特殊时间戳，表示该时间前的事件都已到达
  保证窗口计算的准确性

状态管理：
  键控状态（Keyed State）：per-user 的特征统计
  ValueState：存储用户最近点击序列
  MapState：存储多个统计指标
  状态后端（RocksDB）：持久化到磁盘，支持大状态

允许延迟（Allowed Lateness）：
  窗口触发后仍接受迟到数据，触发更新计算
  例：allowedLateness(Time.minutes(1))

精确一次语义：
  Flink + Kafka 两阶段提交 → 确保数据不丢不重
  故障恢复：从最近检查点（Checkpoint）恢复状态
```

### 3.4 实时特征 vs 离线特征对比

```
              实时特征                    离线特征
数据新鲜度    秒/分钟级                    小时/天级
计算延迟      低延迟流处理                  批处理
覆盖度        近期活跃用户                  全量历史数据
稳定性        可能抖动或缺失               清洗后稳定
典型特征      最近 10 次点击序列            30 天品类偏好
适用场景      捕捉即时兴趣和突发趋势       刻画长期兴趣和静态属性
```

## 4. Feature Store（特征平台）

### 4.1 解决的核心问题

训练-推理特征一致性（Training-Serving Skew）：
```
训练时用特征 X_train（离线计算）
推理时用特征 X_serve（在线拉取）
如果两者定义或计算逻辑不一致 → 模型效果大打折扣

Feature Store 的解决方案：
  统一特征定义（声明式，一处定义多处使用）
  统一计算管道（同一逻辑的流/批两种执行）
  统一存储接口（离线读 HDFS，在线读 Redis）
```

### 4.2 Feature Store 架构

```
特征定义层：
  声明式定义特征（名称、类型、计算逻辑、数据源）
  版本管理、血缘追踪

离线存储：HDFS/Hive
  存历史全量特征，供模型训练

在线存储：Redis/DynamoDB
  存最新特征值，供在线推理
  毫秒级读取延迟

物化管道：
  离线计算 → 写入离线存储
  流式计算 → 同时写入在线+离线存储（特征双写）

特征服务 API：
  统一读取接口，屏蔽存储细节
```

### 4.3 关键能力

```
时间旅行（Point-in-time Join）：
  训练时严格用事件发生前的特征值
  防止特征穿越（Feature Leakage）
  例：预测 T 时刻的 CTR 时，只能用 T 之前的特征

特征血缘追踪：
  从原始数据到最终使用的全链路可视化
  便于排查特征异常和审计

特征共享与复用：
  跨团队、跨模型复用特征定义
  避免重复计算，降低成本

特征监控：
  特征值分布漂移检测（PSI）
  特征服务可用性和 P99 延迟监控
  异常告警和自动降级
```

### 4.4 业界产品

```
开源：Feast（Google 开源，轻量级）
商业：Tecton（Feast 核心团队创立）
大厂自研：字节 ByteFeature、阿里 PAI-Feature Store
```

## 5. 实时推荐系统整体架构

### 5.1 核心模块及协作

```
用户请求
    ↓
┌─────────────┐
│  API 网关     │ ← 认证、限流、路由
└─────┬───────┘
      ↓
┌─────────────┐     ┌──────────────┐
│  召回模块     │ ←── │ 实时特征存储   │ ← Flink 计算
│  (多路召回)   │     │ (Redis/Tair)  │
└─────┬───────┘     └──────────────┘
      ↓ 候选集（数百~数千）
┌─────────────┐     ┌──────────────┐
│  排序模块     │ ←── │ 在线特征拼接   │ ← 离线+实时特征
│  (精排/粗排)  │     │              │
└─────┬───────┘     └──────────────┘
      ↓ 排序后列表
┌─────────────┐
│  重排模块     │ ← 去重、打散、多样性、商业规则
└─────┬───────┘
      ↓
推荐结果返回用户
      ↓
用户行为日志 → Kafka → Flink → 特征更新 / 模型增量训练
```

### 5.2 各模块的实时性要求

```
召回：
  实时向量召回（用户实时兴趣向量 → Faiss/Milvus 检索）
  实时热门召回（最近 1 小时热门物品）
  实时兴趣标签召回（最近 10 分钟点击品类 → 倒排索引）

排序：
  融合实时特征的特征集合
  模型可通过在线学习实时更新
  延迟预算：10-50ms（模型推理部分）

重排：
  去重（过滤本会话已曝光物品）
  打散（避免同类型商品连续出现）
  新物品强插（探索策略）
  商业逻辑调整（高利润商品位置）
```

### 5.3 延迟预算分配

```
端到端 P99 延迟预算：50-200ms

典型分配：
  特征查询：20-50ms（Redis 批量读取）
  模型推理：10-50ms（TF Serving / Triton）
  排序逻辑：10-20ms（业务规则）
  网络通信：10-30ms（服务间 RPC）

超时降级策略：
  方案 A：缓存上次推荐结果返回
  方案 B：降级到简化模型（轻量级模型预热缓存）
  方案 C：返回热门榜单兜底
```

## 6. 高可用与容灾设计

### 6.1 微服务架构

```
拆分原则：召回、排序、重排各自独立服务
部署：Docker + Kubernetes，自动扩缩容
服务发现：Consul/Nacos
通信：gRPC（高性能、低延迟）
```

### 6.2 容灾策略

```
服务降级：
  实时特征不可用 → 自动切换到离线特征
  复杂模型超时 → 降级到简化模型
  全部不可用 → 返回热门榜单

熔断机制：
  下游服务错误率 > 阈值 → 自动熔断，快速失败
  防止级联故障

多活部署：
  多机房部署服务集群
  全局负载均衡，单机房故障自动切换

缓存策略：
  多级缓存：本地缓存 + Redis 分布式缓存
  防止缓存击穿（热点 key + 互斥锁）
```

### 6.3 监控告警

```
系统指标：QPS、P99 延迟、错误率
业务指标：CTR/CVR 实时趋势
模型指标：预测分布、AUC 趋势
特征指标：特征服务可用性、数据流吞吐

告警分级：
  P0（立即处理）：服务不可用、核心指标暴跌
  P1（30分钟内）：模型漂移、特征异常
  P2（当天处理）：性能退化、资源水位高
```

## 7. 性能优化全链路

### 7.1 特征计算优化

```
Flink 预聚合和特征预计算
结果存入 Redis（键值查询 O(1)）
异步加载 + 旁路缓存（避免在关键路径上做复杂查询）
特征重要性分析 → 只实时化 ROI 高的特征
```

### 7.2 模型推理优化

```
模型轻量化：
  剪枝、量化（INT8/FP16）、知识蒸馏
  减少模型体积和计算量

高性能推理引擎：
  TensorRT（NVIDIA GPU 优化）
  ONNX Runtime（通用加速）

模型分级策略：
  高价值用户 → 复杂模型
  长尾用户 → 轻量模型
  热门物品 → 预计算并缓存推理结果

硬件加速：
  GPU 推理：吞吐高，适合批量请求
  CPU 推理：延迟低，适合单请求快速响应
```

### 7.3 系统级优化

```
异步与并行：
  召回多路并发（多路同时查询，取并集）
  特征查询与召回并行

连接池复用：减少 Redis/RPC 连接开销
批量请求：合并多个特征查询为一次 Pipeline
Protocol Buffers：高效序列化，减少网络传输
```

## 8. 数据一致性保障

### 8.1 流处理一致性

```
Flink 精确一次语义：
  Kafka 事务 + 两阶段提交
  确保数据不丢不重

状态一致性：
  Checkpoint 定期快照状态到 HDFS
  故障恢复从最近成功的 Checkpoint 恢复
```

### 8.2 特征一致性

```
训练-推理一致性：
  Feature Store 统一特征定义
  特征双写：实时特征同时写入在线存储和回溯日志
  特征版本号：训练时记录使用的特征版本

模型参数一致性：
  参数服务器架构中，一致性哈希保证分片
  最终一致性（允许短暂不一致，提升可用性）
```

## 9. 面试高频问题速查

Q: Lambda vs Kappa 架构如何选？
A: Lambda 容错好但维护两套代码；Kappa 简单但大规模回溯贵。大厂多用 Lambda 变体 + 流批一体思想。

Q: 如何保证实时推荐系统的特征一致性？
A: Feature Store 统一特征定义 + 流批统一计算逻辑 + 特征版本管理 + 特征双写。

Q: 实时推荐系统的性能瓶颈在哪？
A: 特征计算（Flink 预聚合解决）、模型推理延迟（量化/蒸馏/TensorRT 解决）、网络通信（gRPC + Protobuf 解决）。

Q: 如何设计实时特征工程流程？
A: 行为事件 → Kafka → Flink keyBy+窗口聚合 → Redis 在线存储 → 推荐服务拉取拼接 → 模型推理。

Q: 实时推荐系统如何容灾？
A: 微服务拆分 + 服务降级（实时→离线→热门兜底）+ 熔断机制 + 多活部署 + 多级缓存 + 模型自动回滚。
