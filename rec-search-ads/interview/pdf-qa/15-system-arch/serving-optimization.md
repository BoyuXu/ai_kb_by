## 推荐系统高性能服务与优化

推荐系统从模型到上线，核心挑战是在严格延迟约束（P99 < 50ms）下完成召回-粗排-精排-重排全链路。本文覆盖模型部署、推理加速、特征工程、全链路监控、云原生架构五大模块。

---

## 1. 模型序列化与部署

核心概念：将训练好的模型转为高效推理格式，通过模型服务框架对外提供 gRPC/REST 接口。

主流方案对比：

```
框架           | 格式              | 特点
TF Serving     | SavedModel        | TF 生态原生，版本管理好，gRPC 高效
Triton         | ONNX/TRT/多格式   | 多框架通吃，动态 batching，GPU 利用率高
TorchServe     | .mar              | PyTorch 原生，自定义 handler 灵活
ONNX Runtime   | .onnx             | 跨框架标准格式，CPU 推理优化强
```

部署要点：
- 模型版本管理：蓝绿部署 / 金丝雀发布，灰度切流验证
- 模型热更新：双 buffer 机制（加载新模型到备用 buffer → 原子切换 → 旧 buffer 回收）
- A/B 测试基础设施：流量分桶一致性哈希，确保同一用户始终命中同一实验组

面试要点：为什么不直接用 Python Flask 部署？→ GIL 限制并发，无 batching，无 GPU 调度，生产环境必须用专业框架。

**模型格式转换链路**
```
PyTorch (.pt) → ONNX (.onnx) → TensorRT (.trt)    # GPU 推理最优
PyTorch (.pt) → TorchScript (.pt) → TorchServe     # PyTorch 生态
TF (.pb) → SavedModel → TF Serving                  # TF 生态
任意框架 → ONNX → ONNX Runtime                       # CPU 推理最优
```

转换注意事项：
- 动态 shape 支持：batch_size 和序列长度需要声明为动态维度
- 算子兼容性：自定义算子可能不被 ONNX/TRT 支持，需注册 custom op 或改写
- 精度验证：转换后必须用测试集比对输出差异（通常要求 max_diff < 1e-5）

## 2. 推理加速

四大加速手段：

**量化（Quantization）**
- 训练后量化（PTQ）：FP32 → INT8/FP16，精度损失 <1%，推理速度 2-4x
- 量化感知训练（QAT）：训练时模拟量化误差，精度更高但成本大
- 混合精度：Embedding 层保持 FP32，全连接层用 FP16
- 关键公式：`q = round(x / scale) + zero_point`，scale = (max-min) / (2^bits - 1)

**剪枝（Pruning）**
- 非结构化剪枝：按权重绝对值剪（稀疏矩阵，需硬件支持）
- 结构化剪枝：整行/整列/整个注意力头剪（直接减小模型维度）
- 迭代剪枝：训练 → 剪 → 微调 → 再剪，逐步压缩

**知识蒸馏（Distillation）**
- Teacher-Student 框架：大模型 soft label 指导小模型
- 蒸馏损失 = alpha * KL(soft_teacher, soft_student) + (1-alpha) * CE(hard_label, student)
- 温度 T 控制 softmax 平滑度，T 越大类间关系信息越丰富

**推理引擎优化**
- TensorRT：算子融合（Conv+BN+ReLU）、kernel 自动调优、FP16/INT8 校准
- 动态 Batching：等待窗口内聚合请求，提升 GPU 利用率（Triton 原生支持）
- vLLM PagedAttention：KV Cache 分页管理，减少显存碎片（LLM 场景）

加速手段组合顺序：蒸馏 → 剪枝 → 量化（先减知识冗余 → 减结构冗余 → 减精度冗余）

## 3. 实时特征工程

全链路特征流：

```
数据源(MySQL/日志/Kafka)
  → Flink 实时流处理（窗口聚合/特征计算）
    → 特征存储（Redis: 毫秒级读取 / Cassandra: 大规模存储）
      → 特征服务（gRPC 批量查询 + 本地缓存）
```

特征一致性问题：
- 训练-推理偏差（Training-Serving Skew）：训练用离线特征，推理用实时特征
- 解决：特征日志回放（将线上实时特征写入日志 → 离线训练直接用）
- Feature Store 统一管理：Feast / 自研，一次定义、训练推理共用

实时特征计算模式：
- 滑动窗口聚合：近 30 分钟点击率、近 1 小时曝光数
- 序列特征：用户最近 N 次行为 ID 列表（存 Redis sorted set）
- 交叉特征：实时计算 user-item 交叉统计量

面试要点：如何保证特征 P99 延迟 < 5ms？→ 预计算 + Redis 本地缓存 + 批量查询合并 + 异步降级（特征缺失时用默认值）

**特征存储选型**
```
存储            | 适用场景                  | 延迟     | 容量
Redis Cluster   | 用户实时特征、热数据       | <1ms     | TB 级
Cassandra       | 用户历史特征、大规模       | 5-10ms   | PB 级
DynamoDB        | Serverless、低运维         | 5ms      | 无限
本地缓存(Caffeine)| 超热数据（全局统计量）   | <0.1ms   | GB 级
```

最佳实践：三级缓存 — 本地缓存（全局热特征）→ Redis（用户级热特征）→ Cassandra（冷特征），缓存未命中率 < 1%。

## 4. 全链路监控

三大支柱：Metrics + Logging + Tracing

**指标监控（Prometheus + Grafana）**
- 业务指标：CTR/CVR/GMV 实时曲线，按实验组分桶
- 系统指标：QPS、P50/P95/P99 延迟、GPU 利用率、队列积压
- 模型指标：AUC 在线估算、特征分布漂移检测（PSI/KL散度）
- 告警规则：P99 > 阈值连续 3 分钟 → PagerDuty

**分布式追踪（SkyWalking / Jaeger）**
- 每个请求一个 trace_id，经过召回/粗排/精排/重排各阶段打 span
- 慢请求分析：找到瓶颈阶段（通常是特征查询或模型推理）
- 采样策略：全量太贵，1% 随机 + 100% 慢请求 + 100% 错误请求

**模型质量监控**
- 数据漂移检测：每小时计算特征分布 PSI，PSI > 0.25 → 告警
- 在线 AUC 估算：收集曝光-点击对，滑动窗口计算
- 模型过期检测：训练数据最新日期 vs 当前日期，超过阈值触发重训

面试要点：线上 CTR 突然下降 20% 排查步骤？→ 1) 检查部署变更（模型/配置） 2) 检查数据源（特征缺失/延迟） 3) 检查流量分布变化（新用户占比） 4) 检查竞品/外部事件

## 5. 云原生与服务治理

**K8s 弹性伸缩**
- HPA：基于 CPU/GPU 利用率 或 自定义指标（QPS、队列长度）
- VPA：自动调整 Pod 资源 request/limit
- 混合策略：白天峰值 HPA 扩容 + 夜间 VPA 缩资源节省成本

**Service Mesh（Istio / Linkerd）**
- 流量管理：金丝雀发布精确到百分比，按 header 路由到实验组
- 熔断（Circuit Breaker）：下游 5xx 率 > 50% → 熔断 → 快速失败 → 降级策略
- 限流（Rate Limiting）：保护核心服务，令牌桶/漏桶算法
- 超时控制：召回 10ms + 粗排 5ms + 精排 20ms + 重排 5ms = 总 40ms budget

**eBPF 可观测性**
- 内核级网络/系统调用追踪，无需修改应用代码
- 用于定位网络延迟、CPU 调度、锁竞争等底层问题

降级策略优先级：
```
1. 缓存兜底（返回上次推荐结果）
2. 简化模型（跳过精排，用粗排结果直接返回）
3. 热门推荐（跳过个性化，返回全局热门）
4. 静态推荐（运营预配置的固定列表）
```

## 6. 全链路延迟优化实战

优化策略按 ROI 排序：
```
策略                 | 收益      | 成本
特征预计算+缓存      | 高（-10ms）| 低
模型量化 FP16        | 高（-15ms）| 低
动态 Batching        | 中（-5ms） | 低
并行召回多路         | 高（-20ms）| 中
模型蒸馏             | 高（-30ms）| 高
架构拆分微服务        | 中         | 高
```

经典面试场景 — P99 从 80ms 优化到 40ms：
1. 分布式追踪定位瓶颈阶段（特征查询 30ms）
2. 特征预计算 + Redis pipeline 批量查询（30ms → 8ms）
3. 多路召回并行化（串行 25ms → 并行 10ms）
4. 精排模型 FP16 量化 + TensorRT（20ms → 12ms）
5. 重排轻量化（规则 + 小模型代替大模型）

## 面试高频考点速查

```
Q: 推理加速方案怎么选？
A: 先蒸馏（精度损失最小），再结构化剪枝，最后量化。三者可叠加。

Q: 训练推理不一致怎么解决？
A: Feature Store 统一管理 + 特征日志回放 + 上线前 shadow mode 对比。

Q: 模型上线后效果下降怎么排查？
A: 数据漂移(PSI) → 特征缺失 → 代码bug → 流量变化 → 外部因素，逐层排除。

Q: 熔断和降级的区别？
A: 熔断是自动检测故障并切断请求，降级是主动降低服务质量换取可用性。

Q: 为什么推荐系统延迟要求这么高？
A: 每增加 100ms 延迟，CTR 下降约 1%。用户体验 + 业务收入直接相关。

Q: 多路召回如何并行化？
A: 各路召回独立 RPC 并行请求 → CompletableFuture/asyncio 收集 → 合并去重排序。
   设置每路超时（如 8ms），超时返回空结果不阻塞其他路。

Q: Feature Store 解决什么问题？
A: 核心解决训练-推理特征不一致。统一特征定义、计算逻辑、存储，
   训练和推理使用同一份特征，消除 Training-Serving Skew。

Q: 如何判断模型需要重训？
A: 三个信号：1) PSI>0.25(数据漂移) 2) 在线AUC持续下降 3) 训练数据年龄超阈值。
   任一触发即自动启动重训 pipeline。
```
