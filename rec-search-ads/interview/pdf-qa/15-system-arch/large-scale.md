# 大规模推荐系统架构设计

## 1. 系统总览：多阶段漏斗架构

```
全量物品库（百万~亿级）
    |
    v
[召回层] -- 多路并行，毫秒级，筛出千级候选集
    |         协同过滤 / 向量召回 / 热门召回 / GNN召回
    v
[排序层] -- 精排模型，精准打分排序
    |         Wide&Deep / DIN / DeepFM / MMoE
    v
[重排层] -- 业务规则 + 多样性优化
    |         去重 / 多样性打散 / 运营插入 / 公平性
    v
[混排层] -- 多业务线结果融合
    |         流量分配 / Bandit 动态调整
    v
最终推荐列表
```

## 2. 召回策略设计

### 2.1 多路召回架构
同时使用多种召回策略，覆盖不同方面的用户兴趣：

协同过滤召回：
- Item-CF：找用户历史兴趣物品的相似物品
- User-CF：找相似用户喜欢的物品
- 优点：利用群体智慧，效果稳定
- 缺点：冷启动问题，需要足够的交互数据

向量召回（Embedding-based）：
- 双塔模型：用户塔 + 物品塔，分别编码为低维向量
- 训练：正样本内积大，负样本内积小（对比学习）
- 在线服务：预计算所有物品 Embedding，用 ANN 检索最近邻
- 负采样策略：随机负采样、难例挖掘、流行度加权采样

热门召回：
- 基于全局/分群统计，召回近期热门物品
- 作为保底策略，保证基础体验和新颖性
- 简单高效，适合所有场景

GNN 召回：
- 构建用户-物品-标签异构图
- 通过多层消息传递聚合邻居信息
- 优势：融合多源异构信息、缓解冷启动
- 挑战：十亿级图的训练和推理效率

### 2.2 向量索引（ANN）
当候选集达到亿级时，暴力计算不可行，需要近似最近邻搜索：

HNSW（分层可导航小世界网络）：
- 结构：多层跳表，底层全数据，上层稀疏子集
- 查询：从顶层贪婪搜索，逐层进入下一层
- 优势：查询效率极高，召回率优异
- 代价：内存占用高，索引构建时间长

IVF（倒排索引 + 聚类）：
- K-Means 聚类为粗粒度簇，记录每簇内的向量
- 查询：先找最近的 nprobe 个簇，再在簇内精细搜索
- 优势：内存可控
- 代价：性能依赖聚类质量

实际选择：
- 延迟敏感 + 资源充足 -> HNSW
- 内存敏感 + 候选集极大 -> IVF 或 IVF-HNSW 复合索引
- 工业工具：Faiss, Hnswlib, Milvus, SPTAG

### 2.3 多路召回流量分配
- 基于 MAB 框架动态调整各路召回的候选集比例
- 定义价值函数：价值 = w1*CTR + w2*留存率 + w3*多样性
- Thompson Sampling 采样决定各路分配比例
- 安全机制：设置每路最低/最高比例，防止某路被完全关闭

## 3. 数据收集与特征工程

### 3.1 特征类型

用户特征：
- 画像属性：年龄、性别、地域、职业
- 行为特征：历史点击/购买/收藏序列
- 统计特征：购买总金额、活跃度、品类偏好
- 实时兴趣：当前会话行为序列

物品特征：
- 基础属性：品类、品牌、价格
- 内容特征：TF-IDF/BERT 文本向量、CNN 图像向量
- 统计特征：历史 CTR、购买转化率、近期曝光量

上下文特征：
- 时间、设备、网络、位置
- 推荐场景（首页/详情页/搜索结果）

交叉特征：
- 用户-物品交叉：用户对该品类的历史偏好
- FM/DeepFM/DCN 自动学习特征交叉

### 3.2 序列特征构建
- 将用户历史行为按时间排序构建行为序列
- 物品 Embedding -> GRU/Transformer 编码 -> 动态兴趣向量
- GRU：计算效率高，适合中短序列
- Transformer：长序列建模能力强，可并行计算
- 双塔模型中用序列编码作为用户兴趣表示

### 3.3 类别型特征处理
- 独热编码：适合低基数特征（如性别），高基数会维度爆炸
- Embedding：将高基数 ID（用户ID/物品ID/品类）映射到低维稠密向量
- 特征哈希：固定大小向量空间，可能有哈希冲突
- 目标编码：用目标变量（如 CTR）的统计量作为特征值

### 3.4 实时特征 Pipeline

```
用户行为日志
    |
    v  Kafka 消息队列
    |
    v  Flink 流处理引擎
    |    - 实时统计：滑动窗口内用户行为计数、物品实时 CTR
    |    - 序列特征：用户最近 N 次交互序列
    |    - 关联计算：异步 IO 或维表关联
    v
Redis/ScyllaDB (在线特征存储)
    |
    v  特征服务 API（统一查询接口）
    |    - 本地缓存 -> 分布式缓存 -> 底层存储
    v
推荐模型推理
```

关键挑战与解决：
- 正确性与一致性：Flink 事件时间语义 + 水位线 + 状态后端
- 低延迟高可用：集群化部署 + 客户端分片 + 多级缓存
- 特征回溯一致性：特征注册中心记录时间戳，训练时按时间回查历史特征值
- Lambda vs Kappa：Lambda 分离实时/离线特征，Kappa 全用流处理统一

## 4. 排序模型

### 4.1 模型演进

```
逻辑回归（LR）
  -> 因子分解机（FM）：自动二阶特征交叉
  -> GBDT + LR：梯度提升树学非线性 + LR学组合
  -> Wide & Deep：记忆性（Wide）+ 泛化性（Deep）
  -> DeepFM：FM 自动交叉 + DNN 高阶交叉
  -> DCN：显式可控的高阶交叉网络
  -> DIN：注意力机制动态建模用户兴趣
  -> DIEN：GRU 建模兴趣演化过程
  -> MMoE：多任务学习共享专家网络
```

### 4.2 Wide & Deep
- Wide 部分：广义线性模型 + 手工交叉特征 -> 记忆（记住频繁共现模式）
- Deep 部分：多层 DNN + Embedding -> 泛化（发现新的特征交互）
- 联合训练：两部分输出合并到同一输出层

### 4.3 DIN（Deep Interest Network）
- 核心创新：局部激活单元（注意力机制）
- 对不同候选物品，动态计算用户历史行为的注意力权重
- 用户兴趣向量 = 历史行为 Embedding 的注意力加权和
- 优势：精准捕捉用户兴趣的动态性和多样性
- 挑战：线上服务计算量大（每个候选物品都要重算用户向量）

### 4.4 多任务学习（MMoE / ESSM）

MMoE（Multi-gate Mixture-of-Experts）：
- 共享多个专家网络 + 每个任务独立的门控网络
- 门控网络学习如何动态组合专家输出
- 共通知识通过共享专家学习，任务特有知识通过门控选择性吸收
- 有效缓解任务间梯度冲突

ESSM（Entire Space Multi-Task Model）：
- 引入辅助任务：点击转化率（CTCVR）
- 建模链条：曝光 -> 点击 -> 转化
- 概率依赖关系作为结构化约束，天然缓解任务冲突

损失函数与梯度冲突处理：
- 不确定性加权：按任务损失方差的倒数动态调权
- PCGrad：梯度投影消除冲突分量
- 实践：结构改进 + 梯度操作结合使用

### 4.5 特征交叉模型对比

```
模型        交叉方式          交叉阶数    计算复杂度    适用场景
FM          隐向量内积        二阶        O(kn)       计算敏感/高维稀疏
DeepFM      FM + DNN         低阶+高阶   O(kn+DNN)   通用在线排序
DCN         显式交叉网络      可控高阶    O(d^2*L)    需精确控制交叉深度
```

### 4.6 冷启动排序方案
- 特征工程：引入丰富的上下文/内容特征弥补行为缺失
- 多任务学习：辅助任务（如注册信息预测）提供额外监督信号
- 元学习（MAML）：学习好的参数初始化，少量交互后快速适应
- 迁移学习：在老用户/老物品上预训练，冻结中间层后微调

## 5. 重排与多样性优化

### 5.1 重排阶段的目标
优化整个列表的整体效用，而非单个物品的预测分数：
- 保证多样性：避免同类内容过于集中
- 控制新鲜度：时效性内容优先
- 处理运营需求：固定位置插入运营内容
- 去重策略：过滤已读/已购物品
- 公平性：保证不同供给方获得合理曝光

### 5.2 重排策略

基于规则的多样性打散：
- 连续 N 个物品中不允许出现同一品类/作者/品牌
- 实现简单，效果稳定
- 最基础且广泛使用的方法

MMR（Maximal Marginal Relevance）：
- 贪心迭代选物品，每步最大化：lambda * 相关性 - (1-lambda) * 与已选物品最大相似度
- lambda 控制相关性 vs 多样性的平衡
- lambda=1 纯相关性排序，lambda=0 纯多样性

DPP（Determinantal Point Process）：
- 用行列式衡量物品集合质量：det(L_S)
- 核矩阵 L 的对角线 = 物品质量，非对角线 = 物品相似度
- 天然地将质量和多样性统一建模
- 计算复杂度较高，适合候选集规模中等的场景

多目标加权融合：
- 综合分数 = w1*CTR + w2*多样性 + w3*新颖性 + w4*公平性
- 权重通过离线分析 + A/B 测试确定
- 帕累托前沿搜索最优权重组合

强化学习重排：
- 将重排建模为序列决策问题
- 状态：用户上下文 + 已选物品列表
- 动作：下一个要加入列表的物品
- 奖励：长期用户满意度（留存/LTV）

### 5.3 多目标重排优化框架
1. 定义量化指标：多样性（品类方差/Embedding 距离）、新颖性（流行度倒数）、公平性（曝光比例）
2. 构建优化函数：
   - 约束优化法：max GMV s.t. 多样性 >= theta_d, 公平性 >= theta_f
   - 线性标量化：Score = w1*GMV + w2*多样性 + w3*新颖性 + w4*公平性
3. 目标权衡：
   - 上下文自适应权重（新用户侧重探索，老用户侧重 GMV）
   - 分层优化（先保公平性硬约束，再权衡其他目标）
   - 在线学习动态调整权重

## 6. 评估体系

### 6.1 离线评估指标

准确率类：
- AUC：整体排序能力，不依赖阈值，但对 Top-K 不敏感
- GAUC：按用户分组计算 AUC 再加权平均，更贴近实际
- NDCG：考虑位置衰减的排序质量，关注列表头部

排序类：
- MRR（Mean Reciprocal Rank）：首个正确推荐的排名倒数的均值
- Precision@K / Recall@K / F1@K
- Hit Rate：至少命中一个感兴趣物品的用户比例

多样性/新颖性类：
- 多样性：列表内物品的不相似度（ILS）
- 新颖性：推荐非热门/长尾物品的程度
- 覆盖率：被推荐物品占总物品库的比例
- 基尼系数：物品曝光分布的公平性

### 6.2 在线评估（A/B 测试）

核心指标：
- 点击率（CTR）：即时吸引力
- 人均曝光点击次数：覆盖效果
- 留存率（次日/7日/30日）：长期价值
- GMV/转化率：商业价值

实验设计要点：
- 分层随机化：按用户活跃度、新老用户等关键维度分层后随机分流
- 样本量计算：显著性水平 0.05、统计功效 0.8、最小可检测效应
- 实验周期：覆盖完整用户活跃周期（含工作日和周末）
- 多重检验校正：同时观察多指标时用 Bonferroni 校正或 FDR 控制

辛普森悖论防范：
- 必须进行分组/分群分析，不能只看整体指标
- 检查各关键用户群体中实验效果是否一致
- 整体与局部结论相反时，深入探查混杂变量

### 6.3 离线与在线指标的不一致性
- 常见现象：离线 AUC 提升但在线 CTR 无变化/下降
- 原因：
  - 离线评估无法模拟用户实时反馈闭环
  - 曝光偏差导致离线评估数据有偏
  - 系统延迟/多样性等工程因素离线不可见
- 应对：先离线筛选 -> 少数候选投入 A/B 测试 -> 以在线结果为准

## 7. 系统性能与可扩展性

### 7.1 性能指标

```
指标        含义                        推荐系统要求
QPS         每秒查询数                  万级以上
P99 延迟    99分位响应时间              < 100ms
吞吐量      单位时间处理数据量          支撑全量特征计算+模型推理
```

三者关系：系统资源固定时 QPS 提升通常导致延迟增加

### 7.2 服务降级与熔断

服务降级：
- 实时用户画像不可用 -> 降级为离线画像或全局热门
- 深度排序模型超时 -> 降级为轻量线性模型或规则排序
- 某路召回失败 -> 关闭该路，依赖其他召回保证结果输出

熔断机制：
- 下游服务失败率达阈值 -> 熔断器打开 -> 停止发送请求
- 休眠后半开状态 -> 尝试少量请求 -> 成功则关闭熔断
- 应用场景：特征服务、外部数据源、下游模型服务

### 7.3 缓存策略

Cache-Aside（旁路缓存）：
- 读：先查缓存 -> 未命中则查后端存储 -> 结果写入缓存
- 写：先更新后端存储 -> 删除缓存
- 最通用的模式

多级缓存：
- L1：推荐服务本地缓存（Guava Cache），极低延迟
- L2：分布式缓存（Redis Cluster），全量热点数据
- L3：后端存储（HBase/DynamoDB）

缓存一致性：
- TTL 过期 + 主动更新的混合机制
- 版本号比对触发更新
- 先更新数据库再删除缓存，防止脏数据
- 缓存穿透防护：缓存空值 + 短 TTL
- 缓存雪崩防护：TTL 加随机偏移

### 7.4 水平扩展方案

接入层：
- 负载均衡（Nginx/HAProxy）
- API 网关：鉴权、限流、路由

应用服务层：
- 微服务拆分：召回/排序/重排独立服务
- 无状态设计：所有状态存外部存储
- 每个服务独立水平扩展

数据层：
- Redis Cluster 分片存储特征
- 特征数据库/模型存储选用分布式 NoSQL
- 关系型数据库读写分离 + 分库分表

基础设施：
- Docker + Kubernetes 容器化部署
- 自动伸缩（HPA）基于 CPU/内存/QPS 指标
- 弹性伸缩应对流量洪峰

### 7.5 分布式一致性保障
- 特征一致性：统一特征存储 + 版本号/时间戳机制
- 请求上下文透传：确保召回/排序/重排基于同一份上下文
- 模型版本一致：AB测试平台 + 影子部署灰度发布

### 7.6 P99 延迟排查方法论
```
第一步：确认影响范围（全局 vs 局部，与发布/运营活动关联）
第二步：检查外部依赖（特征服务/模型服务/缓存的 P99 和错误率）
第三步：分析服务内部（流量/代码变更/数据热点/日志异常）
第四步：深入模型层（特征分布漂移/模型 Profiling 瓶颈算子）
第五步：应急措施（回滚/降级/扩容/长期优化）
```

## 8. 前沿技术与趋势

### 8.1 联邦学习推荐
- 核心：用户数据留在本地设备，只上传模型更新
- 流程：中央服务器下发模型 -> 客户端本地训练 -> 上传梯度/权重 -> 联邦平均聚合
- 挑战：通信开销、统计异质性、冷启动效果差
- 隐私保护：同态加密、差分隐私增强安全

### 8.2 GNN 在推荐中的应用
- 用户-物品交互图上的消息传递和邻居聚合
- 优势：高阶连通性、融合异构信息、缓解冷启动
- 挑战：十亿级图的计算与存储、实时性、可解释性
- 与多阶段架构结合：GNN 嵌入作为高质量召回源

### 8.3 多模态推荐
- 融合文本/图像/音频/视频特征
- 融合策略：早期融合（特征拼接）、晚期融合（预测融合）、中期融合（交互模块）
- 应用场景：短视频推荐、电商商品推荐
- 挑战：模态缺失鲁棒性、融合策略动态性、效率与效果平衡

### 8.4 因果推理推荐
- 解决传统模型因依赖相关性而产生的偏差问题
- 应用：
  - 消除流行度偏差：IPS 加权 / 双重机器学习
  - 消除位置偏差：因果图 + 干预
  - 反事实评估：离线估计新策略效果
  - 公平性推荐：确保不同群体获得公平曝光
- 工程落地挑战：计算复杂度高、可解释性要求高

## 9. AB 测试系统架构

### 9.1 核心组件
```
API 网关（入口）
    |
    v
AB 调度服务（流量分配）
    |    - 分层正交实验设计
    |    - 配置中心动态管理
    v
推荐服务（按实验标记执行不同策略）
    |
    v
日志收集（Kafka -> 数据仓库）
    |
    v
数据分析平台（指标计算、统计显著性检验）
```

### 9.2 分层正交实验
- 将流量划分为多个层（如召回层、排序层）
- 不同层的实验互不干扰，可正交
- 同一用户同时参与多个不同层的实验
- 分配必须保证均匀性和一致性（同一用户始终在同一组）

### 9.3 长期评估方案
- 长期 AB 测试（8-12 周）
- 队列分析（Cohort Analysis）
- 中断实验（关闭个性化推荐对比）
- 用户生命周期价值追踪
- 综合健康指数：加权融合留存率、多样性、创作者生态等
