# 指标体系设计与效果分析

## 1. 推荐系统评估指标体系

### 1.1 用户行为指标（漏斗模型）

```
曝光 → 点击 → 互动 → 转化 → 留存
 |       |       |       |       |
CTR    CVR    停留时长  GMV   次日/7日留存
```

CTR（点击率）= 点击数 / 曝光数
- 业务含义：衡量推荐的吸引力和精准度
- 局限：过度优化导致标题党、低质内容

CVR（转化率）= 转化数 / 点击数
- 业务含义：衡量推荐的深层价值（购买/完播/注册）
- 局限：延迟转化难捕捉，受价格等外部因素影响

### 1.2 用户体验指标

```
用户活跃度：日均访问次数、会话时长
用户留存率：次日/7日/30日留存
人均阅读深度/观看时长：内容消费质量
负反馈率：不感兴趣点击率，是算法安全性的重要信号
用户满意度：调研/NPS（净推荐值）
```

### 1.3 平台生态指标

```
基尼系数：衡量曝光分配公平性（越小越均匀，0=完全公平）
内容多样性：推荐列表中品类/标签的覆盖度
新品冷启动成功率：新物品获得初始曝光和点击的比例
创作者留存率：内容供给侧的生态健康
长尾物品覆盖率：被推荐物品占总物品库的比例
```

### 1.4 商业指标

```
GMV（商品交易总额）：电商场景的终极指标
ARPU（每用户平均收入）：用户价值
广告收入/eCPM：广告场景核心
LTV（用户生命周期价值）：长期商业价值
```

## 2. 指标体系构建原则

### 2.1 North Star Metric（北极星指标）

定义：统揽全局的单一核心指标，代表产品的长期健康。
- 短视频：人均观看时长
- 电商：GMV 或月活买家数
- 资讯：日均内容消费条数

### 2.2 主指标 + 护栏指标体系

```
主指标（OEC, Overall Evaluation Criterion）：
  实验的核心决策依据，必须预先定义
  例：排序实验 → CTR 或人均时长

护栏指标（Guardrail Metrics）：
  不能变差的底线指标
  例：用户留存率、崩溃率、负反馈率
  即使主指标提升，护栏显著恶化 → 不上线

辅助观察指标：
  帮助理解实验效果的补充指标
  不作为决策依据，避免多重检验问题
```

### 2.3 指标间的权衡关系

核心矛盾：
- CTR vs 停留时长：标题党提升 CTR 但降低停留
- CTR vs 多样性：集中推热门提升 CTR 但损害生态
- 短期 GMV vs 长期留存：促销推荐提升短期收入但消耗用户信任

量化权衡方法：
```
方法一：综合得分
  score = alpha * norm(GMV提升) + beta * norm(多样性提升)
  alpha, beta 由业务战略决定

方法二：帕累托前沿
  多组实验散点图，X轴=GMV提升，Y轴=多样性提升
  前沿上的点 = 无法在不损害另一指标的情况下改善

方法三：约束优化
  max CTR, s.t. 多样性 >= threshold
```

## 3. 统计显著性检验

### 3.1 假设检验基础

```
零假设 H0：实验组和对照组无差异
备择假设 H1：实验组优于对照组

第一类错误（alpha）：H0 为真但拒绝（假阳性）
第二类错误（beta）：H1 为真但未拒绝（假阴性）

统计功效 = 1 - beta = P(拒绝 H0 | H1 为真)
```

### 3.2 t 检验实践

双样本 t 检验步骤：
```
1. 计算两组指标均值 mu_t, mu_c 和标准差 s_t, s_c
2. 计算 t 统计量：
   t = (mu_t - mu_c) / sqrt(s_t^2/n_t + s_c^2/n_c)
3. 查找 p 值：p = P(|T| > |t|)
4. 判断：p < alpha → 拒绝 H0 → 差异显著
```

### 3.3 结果不显著的诊断

实验跑完但 p 值不显著，可能原因：
1. 样本量不足 → 计算实际功效，考虑延长实验
2. 效应太小 → 真实提升可能小于 MDE
3. 方差太大 → 考虑 CUPED 降方差或更精确的指标
4. 实现 bug → 检查分流、埋点、指标计算管道
5. 确实无效 → 接受结果，避免 HARKing（事后假设）

### 3.4 置信区间解读

```
95% 置信区间 [0.2%, 1.8%] 的含义：
  如果重复实验无穷次，95% 的置信区间会包含真实效应
  不是"真实效应有 95% 概率落在此区间"

实践价值：
  区间全在 0 以上 → 显著正效果
  区间包含 0 → 不显著
  区间下界 > MDE → 不仅显著，且效应有业务意义
```

## 4. 指标冲突分析框架

### 4.1 CTR 提升但停留时长下降

系统性分析思路：
```
Step 1: 数据可信度校验
  → 分流均匀？埋点准确？观察期稳定？

Step 2: 指标拆解与用户分群
  → 拆解到新老用户/高低活/品类维度
  → 定位是全体还是特定群体的问题

Step 3: 因果链条分析
  假设 A：标题党效应 → 检查详情页跳出率、有效播放率
  假设 B：信息密度提升 → 检查任务完成率、满意度
  假设 C：推荐多样性降低 → 检查基尼系数、品类覆盖率
  假设 D：排序策略影响 → 检查点击位置分布和滑动深度

Step 4: 长期影响评估
  → 关注次日/7日留存、长期活跃度
  → CTR 提升若以牺牲留存为代价 → 不可接受

Step 5: 决策建议
  → 标题党 → 回滚 + 优化模型目标
  → 效率提升 → 可推广 + 持续监控
  → 多样性下降 → 调整损失函数引入多样性约束
```

### 4.2 GMV vs 多样性冲突

设计实验量化权衡：
```
1. 多组 A/B：A组(当前) vs B/C/D组(不同多样性策略)
2. 运行 2-4 周，观察 GMV 和多样性变化
3. 绘制帕累托前沿图
4. 引入长期指标（留存、LTV）作为最终判断依据
5. 考虑用户分群策略：不同用户类型用不同权衡点
```

## 5. 长期效果评估

### 5.1 长短期效果矛盾

典型案例：
- 促销推荐：短期提升 GMV，长期培养价格敏感性
- 标题党：短期提升 CTR，长期损害信任
- 过度个性化：短期提升点击，长期形成信息茧房

### 5.2 长期评估方法

长期 Holdout 实验：
- 保留 1-5% 用户长期不接受新策略
- 持续对比实验组和 holdout 组的留存、LTV
- 成本：小部分用户体验落后

代理指标（Surrogate Metrics）：
- 用短期可观测指标预测长期效果
- 例：7日留存率作为 LTV 的代理
- 需要历史数据验证代理关系的稳定性

用户生命周期价值建模：
- 建立 LTV 预测模型
- 在 A/B 测试中对比两组的预测 LTV 分布
- 适合电商等可量化收入的场景

### 5.3 留存队列分析

```
方法：按用户首次暴露时间分组（cohort）
追踪每个 cohort 的留存曲线
对比实验组和对照组的衰减速率

优势：
  可以发现短期实验无法捕捉的长期趋势
  区分"一次性效果"和"持续改进"
```

## 6. Interleaving 实验

原理：将两个策略的推荐结果交错展示给同一用户。

Team Draft Interleaving：
```
算法 A 排序列表：a1, a2, a3, a4, ...
算法 B 排序列表：b1, b2, b3, b4, ...

交错展示：b1, a1, a2, b2, b3, a3, ...
（两个算法轮流选择，每次选自己列表中排名最高的未选项）

统计：用户更多点击来自 A 还是 B
```

优势：
- 所需样本量远小于传统 A/B（同一用户内消除个体差异）
- 适合快速筛选排序模型

局限：
- 只能做两两比较
- 适合排序模型对比，不适合全链路改动
- 展示偏差需特殊处理

## 7. 因果推断与推荐

### 7.1 观测数据的偏差来源

```
位置偏差：用户倾向点击排在前面的物品（与质量无关）
选择偏差：只观测到被推荐的物品，未曝光物品无法评估
流行度偏差：热门物品获得更多曝光和交互，马太效应
```

### 7.2 纠偏方法

IPW（逆倾向得分加权）：
- 对每个样本按曝光概率的倒数加权
- 低曝光概率的样本获得更高权重
- 缺点：极端倾向得分导致方差爆炸

Doubly Robust：
- 结合 IPW 和回归调整
- 只要其中一个模型正确，估计就是一致的
- 工业界推荐使用

因果森林 / DML（Double Machine Learning）：
- 用于异质处理效应估计
- 可以发现"哪些用户群体受益最大"
- 个性化实验决策的基础

## 8. 实验文化与流程

### 8.1 工业界最佳实践

```
实验前：
  1. 明确假设和预期效果
  2. 预定义主指标和护栏指标
  3. 计算样本量和实验时长
  4. 配置实验分流和监控

实验中：
  1. 定期检查数据质量（分流均匀、埋点正常）
  2. 不基于中间结果做决策
  3. 监控护栏指标的实时告警

实验后：
  1. 统计检验（t检验/卡方检验）
  2. 分群分析（新老用户/高低活/地域）
  3. 业务解读（因果而非相关）
  4. 决策：主指标显著提升 + 护栏不恶化 → 上线
```

### 8.2 常见反模式

```
Cherry Picking：只报告显著的指标
HARKing：看到结果后编造假设
p-hacking：调整分析方法直到显著
Early Stopping：看到显著就提前停止
Ignoring Guardrails：忽视护栏指标的恶化
```

## 9. 推荐系统多样性评估

```
列表级多样性（ILD, Intra-List Diversity）：
  定义：列表内物品两两相似度的均值取反
  ILD = 1 - avg(sim(i,j)) for all pairs i,j in list
  衡量单个用户的推荐多样性

系统级多样性：
  定义：不同用户推荐结果的差异度
  衡量系统是否给不同用户推荐了不同内容

覆盖率（Coverage）：
  定义：被推荐物品数 / 总物品库大小
  低覆盖率 = 长尾物品得不到曝光 = 信息茧房风险
```

## 10. 面试高频问题速查

Q: 如何评估推荐系统的多样性？
A: 三个层面：ILD（列表内两两相似度取反）、系统级多样性（不同用户结果差异度）、覆盖率（推荐物品占总物品库比例）。

Q: CTR 和 CVR 为什么常被用作核心指标？
A: CTR 衡量吸引力（漏斗入口），CVR 衡量转化能力（漏斗出口），两者处于用户决策漏斗的关键路径，高敏感度且可快速收集。

Q: 只看 CTR/CVR 的局限？
A: 点击率陷阱（标题党）、极端保守推荐（只推最安全的商品）、忽视长期留存和生态健康。必须引入护栏指标和长期指标综合评估。

Q: CUPED 的原理？
A: 用实验前数据作为协变量，回归掉与处理无关的方差。y_cuped = y - theta * (x_pre - E[x_pre])。通常减少 50%+ 方差。

Q: 如何处理 A/B 测试中指标矛盾？
A: 数据校验 → 指标拆解分群 → 因果链条假设验证 → 长期影响评估 → 综合决策。核心是判断矛盾来自标题党/效率提升/多样性下降中的哪一种。
