# ANN 近似最近邻检索与工程选型

## 核心问题

当物品库达到亿级规模时，精确最近邻搜索不可行（线性扫描 O(N)）。ANN (Approximate Nearest Neighbor) 通过预处理构建索引结构，牺牲少量精度换取查询速度的数量级提升。

推荐系统向量召回的核心要求：
- 召回率 Recall@K > 95%
- P99 延迟 < 10ms
- 吞吐量 > 10K QPS
- 支持增量更新

---

## 主流 ANN 算法详解

### HNSW (Hierarchical Navigable Small World)

```
核心思想: 构建分层可导航小世界图

结构:
  Layer L (顶层): 极少节点，跳跃连接，覆盖全局
  Layer L-1: 更多节点
  ...
  Layer 0 (底层): 所有节点，密集连接

  每个节点以概率 1/M_L 被提升到更高层
  类似跳表(Skip List)的分层思想

搜索过程:
  1. 从顶层的入口点开始
  2. 在当前层贪婪搜索，找到局部最近邻
  3. 以此为入口进入下一层
  4. 重复直到底层
  5. 在底层精确搜索 ef 个候选，返回 Top-K

关键参数:
  M: 每层每个节点的最大连接数
     M 大 → 召回率高，但内存和构建时间增加
     通常 M = 16-64

  efConstruction: 构建时搜索的候选集大小
     值越大索引质量越高，构建越慢
     通常 200-500

  efSearch: 查询时搜索的候选集大小
     值越大召回率越高，查询越慢
     通常 64-256
```

```
优点:
  - 查询速度极快 (亚毫秒级)
  - Recall@10 可达 99%+
  - 对向量分布不敏感
  - 支持增量添加节点（无需重建）

缺点:
  - 内存占用大: 需存储图结构 + 原始向量
    128维float32向量: 原始512B + 图结构约1KB = 每向量~1.5KB
    10亿向量 ≈ 1.5TB 内存
  - 索引构建时间长
  - 不支持高效删除

适用: 向量库 < 1亿，对精度要求高，内存充足
```

### IVF (Inverted File Index)

```
核心思想: 先聚类，后搜索

构建:
  1. 对全量向量用 K-Means 聚类，得到 nlist 个聚类中心 (Voronoi胞腔)
  2. 每个向量分配到最近的聚类中心
  3. 构建倒排索引: 聚类ID → [属于该聚类的向量列表]

搜索:
  1. 计算查询向量与所有聚类中心的距离
  2. 选择最近的 nprobe 个聚类
  3. 在这些聚类内精确搜索

关键参数:
  nlist: 聚类数量，通常 sqrt(N) 到 4*sqrt(N)
  nprobe: 搜索时探测的聚类数，nprobe/nlist 越大越精确越慢

优点: 索引构建快，内存友好
缺点: 精度受聚类质量影响，对数据分布敏感
```

### PQ (Product Quantization)

```
核心思想: 向量压缩，用编码近似原始距离

流程:
  1. 将 D 维向量切分为 M 个子空间，每个子空间 D/M 维
  2. 对每个子空间独立做 K-Means，得到 K 个聚类中心（码本）
  3. 每个子向量用其最近聚类中心的ID编码（1字节=256个码字）
  4. 原始向量 → M 字节的编码

  例: 128维向量，M=8 个子空间
      原始: 128 * 4 = 512 字节
      PQ编码: 8 字节 (压缩64倍!)

距离计算:
  预计算查询向量与所有码字的距离表 → O(K*M)
  然后通过查表求和近似原始距离 → O(M)

优点: 极致内存压缩
缺点: 量化误差导致召回率损失
```

### IVF-PQ (组合方案)

```
结合 IVF 的粗筛 + PQ 的压缩:

构建:
  1. K-Means 聚类 → IVF 倒排索引
  2. 计算残差: residual = vector - cluster_center
  3. 对残差做 PQ 编码（残差比原始向量更集中，量化误差更小）

搜索:
  1. 找 nprobe 个最近聚类
  2. 在这些聚类中用 PQ 近似距离计算
  3. 返回 Top-K

参数:
  nlist: 聚类数 (通常 1024-65536)
  nprobe: 搜索聚类数 (通常 8-128)
  M: PQ 子空间数 (通常 8-64)
  nbits: 每子空间量化位数 (通常 8，即256个码字)

优点:
  - 内存效率极高，适合十亿级向量库
  - 10亿128维向量: 约 10-20GB (vs HNSW 的 1.5TB)

缺点:
  - 召回率受聚类和量化双重影响
  - 需要精调多个参数
  - 索引构建需全量数据，更新不灵活
```

### ScaNN (Google)

```
核心改进: 各向异性量化 (Anisotropic Quantization)

传统PQ: 最小化量化误差 ||x - q(x)||^2
ScaNN: 最小化内积误差 |<query, x> - <query, q(x)>|

关键洞察:
  - 推荐系统用内积/余弦相似度，不是欧氏距离
  - 对内积目标做针对性量化，在同等压缩率下内积近似精度更高

性能: 在多个 benchmark 上 Recall-QPS 曲线优于 Faiss
```

## 算法全景对比

```
算法       内存占用    构建速度    查询速度    召回率     增量更新
──────────────────────────────────────────────────────────────
HNSW       高          慢         极快       极高(99%+)  支持
IVF        中          中等       快         中(90-95%) 需重聚类
PQ         极低        中等       中         中         需重训码本
IVF-PQ     低          中等       快         中高       部分支持
ScaNN      低          中等       极快       高         部分支持
```

```
向量库规模          推荐方案
──────────────────────────────────────────────────
< 100万             HNSW (精度优先)
100万 - 1亿         HNSW 或 IVF-HNSW
1亿 - 10亿          IVF-PQ (一级) + HNSW (二级精筛)
> 10亿              分布式 IVF-PQ + 分片
```

## 工业级向量索引架构

### 双索引架构（工业标配）

```
基础索引 (Base Index):
  - 定期全量重建 (天级/周级)
  - 覆盖全量历史物品
  - 索引质量高(完整数据聚类/建图)

增量索引 (Delta Index):
  - 高频增量更新 (分钟级)
  - 只包含新增/更新的物品向量
  - 可用 HNSW (支持增量添加)

在线检索:
  - 同时查询两个索引
  - 结果合并去重
  - 按相似度排序返回 Top-K
```

### 索引发布流程

```
1. 离线构建新索引
2. 影子部署: 新索引部署到验证环境
3. 流量回放: 用真实请求对比新旧索引结果
4. 质量验证: 召回重合度 > 阈值
5. 蓝绿发布 / 热切换
6. 监控指标: 召回率、延迟、错误率
7. 自动 fallback: 异常时切回旧索引
```

### 用户向量实时更新

```
用户 Embedding 随行为实时变化:

方案一（在线计算）:
  用户最近行为 → Flink/Kafka 实时流处理
  → 用户塔模型推理 → 新用户向量
  → 发布到 Redis/特征库
  → 在线服务实时获取

方案二（增量更新）:
  定时(每小时)批量重算活跃用户向量
  发布到特征库

物品向量: 变化较慢，天级批量更新即可
```

## 向量召回效果排查（5步法）

```
线上 AB 测试效果下降时的系统排查:

第一步: 服务层
  - 检查接口响应时间、错误率、超时情况
  - 确认 topK 参数、搜索半径是否被修改
  - 检查服务资源(CPU/内存/网络)

第二步: 索引层
  - 确认索引版本是否最新
  - 检查索引参数(HNSW 的 efConstruction/M)
  - 对比新旧索引下同一查询的召回差异

第三步: 模型层
  - 离线评估: 在固定测试集上对比新旧模型
  - 检查模型是否收敛/过拟合
  - 检查超参数/模型版本

第四步: 数据层
  - 检查训练数据: 特征缺失/分布漂移/标签泄露
  - 训练样本 vs 在线特征一致性
  - PSI 指标监控特征漂移

第五步: 实验层
  - 检查 AB 分流逻辑和指标计算
  - 排除外部因素(营销活动/竞争对手)
```

## 面试高频问题

```
Q1: HNSW 和 IVF-PQ 怎么选？
A: 看向量规模和资源约束:
   - 向量 < 1亿 + 内存充足 → HNSW (精度高/增量友好)
   - 向量 > 1亿 → IVF-PQ (内存效率高)
   - 工业实践: IVF-PQ 一级召回(亿→万) + HNSW 二级精筛(万→千)

Q2: PQ 的压缩比怎么计算？
A: 128维float32: 原始 128*4=512字节
   PQ(M=8,nbits=8): 8字节
   压缩比 = 512/8 = 64倍
   更激进: M=4 → 4字节 → 128倍压缩

Q3: 向量索引多久更新一次？
A: 双索引架构:
   基础索引: 天级全量重建
   增量索引: 分钟级增量更新
   用户向量: 实时(Flink)或小时级批量
   物品向量: 天级

Q4: 如何保证索引更新时服务不中断？
A: 蓝绿发布:
   1. 新索引先在影子环境验证
   2. 流量回放确认召回质量
   3. 热切换到新索引
   4. 异常自动 fallback 到旧索引
   5. 增量索引数据在全量发布后重置

Q5: 向量维度怎么选？
A: 64-256维:
   - 维度越高表达力越强，但ANN检索越慢/内存越大
   - 128维是常见选择(平衡性能和效果)
   - 也可用 PCA/自编码器降维后再建索引

Q6: ANN 检索的离线评估怎么做？
A: 在测试集上:
   1. 精确 KNN 得到 ground truth Top-K
   2. ANN 检索得到 Approx Top-K
   3. Recall@K = |ANN结果 ∩ KNN结果| / K
   4. 同时记录 QPS 和延迟
   5. 画 Recall-QPS 曲线选择最优参数
```
