## 故事框架：背景 → 痛点 → 方案 → 效果

### 标准 3 分钟 Elevator Pitch 模板

```
[1句背景] 
我在 [公司/产品] 做推荐系统，面临 [具体业务问题]。

[2句痛点]
传统方法是 [现状]，但 [为什么有问题]。具体来说，[量化痛点]。

[2句方案]
我们采用了 [算法/方法名] 的思路，核心是 [1句本质]。
[具体做法]。

[1句效果]
结果是 [核心指标提升]，还有副作用是 [隐藏收益或限制]。

[可选加分项] 这个案例的深层启示是：[核心思维转变]。
```

## 📐 核心公式与原理

### 1. 推荐系统漏斗

$$
\text{全量} \xrightarrow{召回} 10^3 \xrightarrow{粗排} 10^2 \xrightarrow{精排} 10^1 \xrightarrow{重排} \text{展示}
$$

- 逐层过滤，平衡效果和效率

### 2. CTR 预估

$$
pCTR = \sigma(f_{DNN}(x_{user}, x_{item}, x_{context}))
$$

- 排序核心：预估用户点击概率

### 3. 在线评估

$$
\Delta metric = \bar{X}_{treatment} - \bar{X}_{control}
$$

- A/B 测试量化策略效果

## 5 个常见错误

### ❌ 错误 1：堆砌技术细节

**错误讲法**
> "我用了 ESMM 多任务学习框架，定义了两个损失函数 L_CTR 和 L_CVR，然后加了一个 pCTCVR 的约束项，用 Adam 优化器，学习率 0.001，batch size 64..."

**正确讲法**
> "问题是 CTR/CVR 模型有选择偏差——我们只能学到被展示的广告数据。我采用了 ESMM 的多任务思路：同时学 CTR 和 CVR，用它们的乘积去约束全空间的转化率。这样就能纠正样本偏差。效果是 CTR AUC +8%。"

**为什么**
- 面试官不在乎实现细节，而是你发现问题的逻辑
- 技术细节会显得你在背论文，不是真正理解

---

### ❌ 错误 2：只说数字，不说原因

**错误讲法**
> "我们从 AUC 0.72 提升到 0.78。"

**正确讲法**
> "我们从 AUC 0.72 提升到 0.78。关键是发现了**样本偏差是比特征工程更大的瓶颈**。传统做法拼命加特征，我们反而简化模型、改变损失函数定义，效果更好。"

**为什么**
- 数字是结果，面试官想听的是你的思维过程
- "为什么能提升" > "提升了多少"

---

### ❌ 错误 3：说自己做的，不说为什么重要

**错误讲法**
> "我负责优化 Feed 的多样性排序。"

**正确讲法**
> "我们发现用户在单一内容类型的堆积下留存下降很快。我选择用 DPP（行列式点过程）在排序后进行多样性重排，平衡相关性和多样性。最终留存 +8%、RPM +5%。"

**为什么**
- 面试官想知道你的判断力：为什么选择这个问题？为什么这个方案？
- 单纯的技术成就不足以展示思维深度

---

### ❌ 错误 4：忽视工程和权衡

**错误讲法**
> "理论上 DPP 能带来最优的多样性。"

**正确讲法**
> "DPP 理论最优，但计算复杂度 O(n³)。在线上要求 P99 < 100ms 的约束下，我们用贪心近似，牺牲 1% 的多样性指标换来 10 倍的速度。这是实际工程的必要权衡。"

**为什么**
- 工程决策比算法创新更能体现高级工程师的素质
- 面试官想看你是否能在理想和现实之间做出聪明的选择

---

### ❌ 错误 5：讲自己不了解的细节

**错误讲法**
> "我用了 GraphSAGE，通过 GCN 层来聚合邻居..."
（但你其实没有深入理解 GCN 的原理）

**正确讲法**
> "我用了图神经网络（GraphSAGE）来处理用户-朋友-内容的关系。核心思想是：朋友的兴趣往往比用户的人口统计特征更有预测力。通过聚合邻近节点，冷启动用户留存 +25%。"

**为什么**
- 面试官一追问就会发现你的认识深度
- 宁可少说一些你不确定的细节，也别被问穿

## 30 个案例的分类和选择策略

### 🎯 我应该选哪 3-5 个案例重点准备？

**选择标准**
1. **你有亲身经历** 或 **深入理解**（不要虚）
2. **故事完整性** 强（背景 → 痛点 → 方案 → 效果都清晰）
3. **难度递进** 的组合（简单 + 中等 + 高难，展示不同角度）
4. **行业匹配** 和面试公司相关（百度搜索 → 选搜索案例；抖音推荐 → 选推荐案例）

---

### 📊 按难度分类

#### 简单案例（易讲、高覆盖）

```
11. 冷启动用户留存（GraphSAGE）
  → 核心：图结构比特征更有信息量
  → 难度：低（图神经网络概念简单）
  → 适用：几乎所有推荐系统面试

12. 冷启动内容曝光（Thompson Sampling）
  → 核心：探索与利用的权衡
  → 难度：中（多臂老虎机略抽象，但直观）
  → 适用：推荐、内容平台

24. 实时搜索索引（Kafka + Flink）
  → 核心：实时性的隐形价值
  → 难度：低（架构直观，不涉及算法）
  → 适用：任何系统面试（通用工程能力）
```

#### 中等难度案例（有深度、需要练习）

```
1. CTR 预估升级（ESMM）
  → 核心：样本偏差 > 特征工程
  → 难度：中高（涉及因果推断思想）
  → 追问陷阱：CVR 延迟偏差怎么处理？IPW 的原理？
  → 适用：算法/数据驱动面试

2. 自动竞价系统（拉格朗日对偶）
  → 核心：问题重新定义的力量
  → 难度：中高（涉及优化论）
  → 追问陷阱：PID 控制和对偶优化什么关系？
  → 适用：算法优化、系统设计面试

3. Feed 混排多样性（DPP）
  → 核心：列表级优化 > 逐项优化
  → 难度：中（DPP 的数学稍复杂）
  → 追问陷阱：DPP 和推荐的不同之处？为什么用行列式？
  → 适用：推荐系统、排序面试
```

#### 高难度案例（展示思维深度）

```
7. 跨域归因（Shapley 值）
  → 核心：因果推断的实战应用
  → 难度：高（Shapley 值很抽象）
  → 追问陷阱：Shapley 值的计算复杂度？与 SHAP 的关系？
  → 建议：如果你深入研究过，讲这个；否则避免

29. 因果推断在营销（PSM）
  → 核心：观测数据也能做因果推断
  → 难度：高（因果推断理论）
  → 追问陷阱：混淆变量如何处理？匹配的假设是什么？
  → 建议：只有你真正有实战经验才讲

14. 个性化排序（多目标学习）
  → 核心：帕累托前沿的思想
  → 难度：中（概念清晰，但理解需要）
  → 追问陷阱：多目标如何权衡？权重如何确定？
  → 适用：推荐、系统设计面试
```

## 进阶：如何应对追问

### 常见追问和应答思路

#### 追问 1：为什么不用其他方法？

**问题**
> "为什么用 ESMM，不用另外的多任务框架？"

**好答案**
> "ESMM 的特别之处在于：它用了一个**全空间可观测的约束**（pCTCVR）来纠正 CTR/CVR 的偏差。这个约束的妙处是它来自 CTR × CVR 的数学关系，不需要额外标注数据。其他多任务框架通常只是把多个任务的损失加起来，没有考虑任务之间的因果关系。所以 ESMM 更elegant。"

**关键**
- 不只是说"我选择的更好"
- 要说"为什么更好"的本质原因
- 对比时聚焦差异点，不是堆砌优点

---

#### 追问 2：你遇到的最大挑战是什么？

**问题**
> "在实现 DPP 多样性的时候，最难的地方是什么？"

**好答案**
> "最大的挑战不是算法本身，而是**线上 latency 约束**。DPP 的精确算法是 O(n³)，对于一个有 5000 个候选的排序列表，在 100ms 的 latency budget 下根本跑不动。我最后用了贪心近似，牺牲了 1% 的多样性指标换来了 10 倍的速度。这个权衡是通过 A/B 测试验证的——用户体验反而更好（可能是因为 latency 的减少抵消了多样性的微小下降）。"

**关键**
- 展示真实的工程困境（不是"算法完美"的故事）
- 解决方案体现了权衡和判断力
- 有数据支持（A/B 测试）

---

#### 追问 3：如果重新做一遍，你会怎样改进？

**问题**
> "如果让你重新做一遍 CTR 预估项目，你会做哪些不同的选择？"

**好答案**
> "有几个方面：
> 1. **数据层面**：当初我们没有很好地处理延迟反馈（delayed feedback）。如果重来，我会从一开始就用 IPW 加权补偿延迟。这可能能多提升 2-3% 的 AUC。
> 2. **模型层面**：现在我理解了**样本偏差来自两个地方**——既来自选择偏差（ESMM 解决了），也来自 position bias（不同位置的点击率不同）。加上 position bias 模型可能还能提 1-2%。
> 3. **系统层面**：我们的特征工程投入太大相对收益太小。回过头看，花同样的时间投在因果推断上，ROI 会更高。"

**关键**
- 展示你的反思和进步（不是固步自封）
- 指出了原有方案的limitations（保持谦虚）
- 有清晰的改进方向（不是模糊的"更努力"）

---

#### 追问 4：这个技术能迁移到其他场景吗？

**问题**
> "DPP 多样性的思路能用在搜索排序上吗？"

**好答案**
> "可以，但需要改进。在推荐中，多样性意味着不同**类别/风格**的内容混合。但在搜索中，多样性意味着不同**意图**的搜索结果（如 Apple：既有公司、也有苹果水果、也有人名）。所以：
> 1. DPP 的相似度矩阵定义需要改变（从推荐的内容相似度 → 搜索的意图相似度）
> 2. 搜索中可能需要加一个**意图识别**的上游步骤
> 3. 评估指标也不同（推荐看留存，搜索看 satisfaction 和 diversity aware CTR）
> 
> 总的来说，**思想可以迁移，细节需要重新设计**。"

**关键**
- 不是简单的"yes/no"，而是条件性的思考
- 指出改进的地方（展示问题敏感度）
- 说明了不同领域的本质差异

## 讲故事的节奏和语气

### 速度节奏

```
背景（30 秒）：设置舞台，让听众理解问题的重要性
  → "在推荐系统中，用户冷启动一直是难点..."

痛点（45 秒）：深入问题，展示现状的局限性
  → "传统做法是...但这样有个问题..."

方案（60 秒）：讲清楚核心思路，重点不在细节
  → "我们改变了思路，采用...的方法..."

效果（20 秒）：量化成果，但也说限制
  → "结果是提升了...，同时我们发现...的启示"

总计：2.5-3 分钟
```

### 语气建议

**避免的语气**
```
❌ 居高临下：听起来像在教别人
❌ 不确定：经常说"应该"、"可能"、"我不太确定"
❌ 过度谦虚：把所有功劳都归给"团队"
❌ 过度自信：听起来自大，禁不起追问
```

**推荐的语气**
```
✅ 诚实坦率：说自己真实的思考过程和局限
✅ 好奇心：展现对问题的热情和深入思考
✅ 谦逊但有自信：承认团队贡献，但清晰表述自己的角色
✅ 故事感：像在讲一个"我们怎样解决问题"的故事，而不是背诵
```

## 准备检查清单

```
对于你选择的每个案例，都要能回答以下问题：

□ 背景：这个问题在业务中为什么重要？
□ 数据：当时的数据规模、特征数、模型复杂度怎样？
□ 痛点：传统方法具体哪里不行？有量化的对比吗？
□ 方案：为什么选这个方案，不选别的？
□ 实现：有什么tricky的工程细节吗（不用讲得太细）？
□ 效果：结果怎样？不仅仅是主指标，还有什么意外收获？
□ 限制：方案有什么缺点或适用范围？
□ 思维：这个案例教会你什么？
□ 迁移：这个思路能用到其他问题吗？

能清晰回答这 8 个问题 → 你已经准备好讲故事了
```

## 最后的建议

1. **不要硬背案例**，要内化成自己的思考
2. **练习讲几遍**，但不要像背稿子，要像讲故事
3. **在 3 分钟内讲完**，争取 2.5 分钟，给面试官追问的时间
4. **期待追问**，不是恐惧追问（追问意味着面试官在听，感兴趣）
5. **团队贡献很重要，但清晰表述自己的角色**（不要模糊 ownership）

- YouTube: "How to Tell a Great Story" by Jim Signorelli

### Q1: 面试项目介绍的 STAR 框架？
**30秒答案**：Situation（背景）→Task（任务）→Action（方案）→Result（结果）。关键：量化结果（AUC +0.5%, 线上 CTR +2%），突出个人贡献，准备 follow-up 追问。

### Q2: 算法面试如何展现系统性思维？
**30秒答案**：①先说全局架构再说细节；②主动分析 trade-off；③提及工程约束（延迟/资源）；④讨论 A/B 测试验证；⑤对比多种方案优劣。

### Q3: 面试中遇到不会的问题怎么办？
**30秒答案**：①诚实说不了解具体细节；②从已知相关知识推导思路；③说明学习路径（"我会从 XX 论文入手了解"）。比胡编强 100 倍。

### Q4: 简历中项目经历怎么写？
**30秒答案**：①每个项目 3-5 行；②突出方法创新点和业务效果；③用数字量化（AUC/CTR/时长提升 X%）；④技术关键词匹配 JD；⑤按相关度排序而非时间顺序。

### Q5: 如何准备系统设计面试？
**30秒答案**：①准备推荐/搜索/广告各一个完整系统设计；②每个系统能说清召回→排序→重排全链路；③准备 scalability 方案（如何从百万到亿级）；④准备 failure mode 和降级方案。

### Q6: 八股文和实际项目经验如何结合？
**30秒答案**：八股文提供理论框架，项目经验证明落地能力。面试时：先用八股文回答「是什么/为什么」，再用项目经验回答「怎么做/效果如何」。纯八股文没有竞争力。

### Q7: 面试中如何展示 leadership？
**30秒答案**：①描述自己在项目中的角色和贡献；②说明如何推动跨团队协作；③展示主动发现问题并推动解决的案例；④分享技术方案选型的决策过程。

### Q8: 被问到不会的论文怎么办？
**30秒答案**：①说清楚自己了解的相关工作；②从论文标题推断可能的方法（如 xxx for recommendation 可能是把 xxx 技术迁移到推荐）；③承认不了解但表达学习意愿。

### Q9: 算法岗面试的常见流程？
**30秒答案**：①简历筛选→②一面（算法基础+项目）→③二面（系统设计+深度追问）→④三面（部门 leader，考察思维+潜力）→⑤HR 面→Offer。每轮约 45-60 分钟。

### Q10: 如何准备不同公司的面试？
**30秒答案**：①字节：重工程实现+大规模系统+实际效果；②阿里：重业务理解+电商场景+系统设计；③腾讯：重算法深度+创新性+论文理解；④快手/小红书：重内容推荐+短视频场景+多模态。

## 📐 核心公式直观理解

### STAR 结构量化公式

$$
\text{Impact} = \Delta\text{Metric} \times \text{DAU} \times \text{Revenue}_{per}_{\text{user}}
$$

**直观理解**：面试讲项目必须量化。"AUC 提升 0.5%"没感觉，但"AUC 提升 0.5% → CTR 提升 2% → 日均收入增长 50 万"让面试官立刻理解项目价值。从模型指标→线上指标→业务指标的链路要讲清楚。

### A/B 测试显著性

$$
n \geq \frac{2(z_{\alpha/2} + z_\beta)^2 \hat{p}(1-\hat{p})}{\delta^2}
$$

**直观理解**：要验证"0.5% 的 AUC 提升是真的"需要多少样本？效果越小（$\delta$ 小）需要的样本越多。面试中常被问"你的实验跑了多久、多少流量"——答不上来说明没有统计意识。

### 项目复杂度评估

$$
\text{ROI}_{project} = \frac{\text{线上收益} \times \text{持续时间}}{\text{人力投入} \times \text{机器成本}}
$$

**直观理解**：面试官关心"这个项目值不值得做"。低投入高回报的项目（改一个特征 AUC 涨 1%）比高投入低回报的项目（重写系统 AUC 涨 0.3%）更有说服力。讲项目时先说 ROI 最高的。
