# 🔍 搜索方向 50题

## 精选深度题（含口语化参考答案）

**Q1. 你们线上用的是 BM25 还是语义检索？怎么融合的？**

> 说实话，纯 BM25 在长尾 query 上很容易挂。我们的做法是双路召回——BM25 负责精确词匹配，双塔或者 BERT-bi-encoder 负责语义召回，然后用 RRF（Reciprocal Rank Fusion）或者学一个小的 LTR 模型来做融合。BM25 的权重在精确词 query 上会给高一些，语义那路在改写 query 或者口语化 query 上更有效。难点在于两路召回数量不一样，分数不在同一个尺度上，所以要做归一化或者直接 RRF 绕开这个问题。

**Q2. Query 理解你们做到什么程度？**

> 这块是搜索质量的天花板。我们做的有这几层：第一是意图识别，比如这个 query 是导航型（找特定页面）、信息型（了解知识）还是事务型（要买东西）；第二是实体识别和 linking，比如"苹果"到底是水果还是 Apple 品牌；第三是 query 改写，同义词扩展、纠错、口语转书面。最难的是歧义处理，比如"Java"在不同上下文下完全不同，这时候要么靠用户历史，要么做多路召回然后在排序层融合。

**Q3. 相关性模型和排序模型有什么区别，你们怎么训练的？**

> 相关性解决的是"这个结果跟 query 有没有关系"，排序解决的是"有关系的结果里哪个更好"。相关性是 prerequisite。训练相关性模型一般用人工标注的 query-doc 对，4级或5级相关性标注，然后用 pairwise 或 listwise loss 来训。排序模型更复杂，要把 CTR、停留时长、用户反馈这些线上信号都加进去，而且要注意 position bias——因为靠前的结果就是天然点击率高，不处理这个的话模型就学歪了，要做 propensity 校正。

**Q4. 你怎么看待 BERT 在搜索里做 reranker 的局限性？**

> 最大的局限是速度。BERT-cross-encoder 精度很好，但是每个 query-doc 对都要跑一遍，线上 latency 根本撑不住。所以它只能做 reranker，在精排之后再过一遍 top-K，K 一般就几十个。另一个问题是训练数据，cross-encoder 需要 query-doc pair，但相关性标注成本很高。还有就是长文档处理，BERT 有 512 token 限制，长文档要切片然后 aggregate，会损失一些信息。所以工程上一般是 bi-encoder 做召回，cross-encoder 做精排 top 层，这是个 quality-latency 的 tradeoff。

**Q5. 搜索里怎么做个性化？会引入哪些风险？**

> 个性化最直接的做法是把用户历史行为作为特征加到排序模型里——点击过的 query、停留过的文档类型、品类偏好这些。但有几个风险：第一是 filter bubble，用户只看到他历史喜欢的东西，探索性降低；第二是新用户冷启动，没有历史数据怎么个性化；第三是隐私，用户可能不知道自己的行为在影响结果。所以一般会做一个探索-利用的 balance，保留一定比例的非个性化结果，同时在 UI 上给用户感知——"根据你的偏好"之类的。

**Q6. 向量检索（ANN）在搜索里怎么做工程落地？**

> 主要就是选 ANN 库——FAISS、Milvus、Elasticsearch 的 HNSW 这几个。FAISS 适合离线批量检索，HNSW 适合在线低延迟。工程上要注意几点：一是 index 更新，文档库是动态的，新文档怎么加进去，HNSW 支持增量但 IVF 类的不支持，需要定期重建；二是向量维度和 recall 的 tradeoff，维度越高精度越好但内存和速度都差；三是量化，FP32 换 INT8 或者 PQ 量化，可以大幅减少内存，但精度会掉一点，要在 recall@k 上测一测影响多大。

**Q7. 搜索结果的 bad case 你们怎么 debug 的？**

> 先分层——是召回问题还是排序问题。召回问题就是 top-N 里根本没有相关文档，这时候要看 query 分析是不是跑偏了，或者 index 里就没有这个内容。排序问题是相关文档召回了但排名靠后，这要看特征——相关性分给多少、CTR 预估多少、有没有 position bias 没处理好。我们线上有一个 trace 工具，可以把某个 query-doc 对的所有特征和分数都打出来，方便逐层分析。还有就是看 query 改写有没有改错，有时候同义词扩展会扩歪。

**Q8. 电商搜索和通用搜索最大的区别是什么？**

> 最大的区别是目标不一样。通用搜索的目标是相关性和用户满意度，电商搜索还要考虑 GMV、商品的商业价值。这就导致排序模型里要融入商业因素——商品的转化率、利润率、商家竞价。还有就是 query 类型不同，电商 query 大量是品类词（"运动鞋"）、属性词（"红色连衣裙 M 码"），需要结构化的属性理解，而不只是文本匹配。另外电商里 filter 很重要，价格区间、品牌、评分这些 facet 搜索，在通用搜索里基本没有。

# 📣 广告方向 50题

## 精选深度题（含口语化参考答案）

**Q1. CTR 预估模型从 LR 到 DNN 再到 Transformer，每一步解决了什么问题？**

> LR 最大的问题是特征交叉要手工做，工程师得手动组合"用户年龄×商品类目"这种特征，成本很高而且靠经验。FM 解决了二阶交叉自动化的问题，但是三阶以上还是不行。DNN 来了之后理论上可以学任意阶交叉，但实际上 DNN 对低阶交叉的学习效率不如 FM——因为 DNN 里低阶特征会被高维空间稀释掉。所以后来有了 DeepFM、DCN 这种显式+隐式交叉结合的结构。Transformer 的好处是 attention 可以捕捉特征之间的动态相关性，不是所有特征都同等重要，但计算量大，工程上要做各种优化。

**Q2. 广告竞价里 GSP 和 VCG 的区别，为什么 Google 最终切到了 VCG？**

> GSP 是广义二价拍卖，你出价最高拿第一位，但只付第二名的价格。听起来公平，但其实不是 truthful 的——也就是说出价等于真实价值不一定是最优策略，广告主有动机去策略性出价。VCG 是 truthful 的，出真实价值是最优策略，但是计算复杂度高，而且实际上 VCG 在多 slot 的情况下实现起来很复杂。Google 切 VCG 是因为他们发现 GSP 下广告主的出价行为很难预测，导致收入不稳定，VCG 虽然理论上单次收入可能低一点，但整体生态更健康。

**Q3. eCPM 排序，为什么不能直接用出价×CTR，还要加 quality score？**

> 纯 eCPM = bid × CTR，问题是出价高的广告不一定用户体验好。比如一个很夸张的标题可能 CTR 很高，但用户点进去体验差、转化率低，广告主浪费了钱，平台长期也受损。Quality Score 引入了相关性、落地页质量、历史账户表现等因素，实际上是对 eCPM 做了修正，让优质广告能用更低的出价竞得好位置，让劣质广告即使出价高也排不上去。这是平台在短期收入和长期生态之间做 tradeoff。

**Q4. 预算控制和 pacing 怎么做？**

> 最简单的是 throttling——随机丢弃一定比例的竞价机会来控制消耗速度。但这个不精准，因为流量不均匀，早上流量少晚上流量多，如果按比例 throttle 的话，预算可能晚上才消耗完，错过了早高峰的好流量。更好的做法是 feedback control，实时监控消耗速度，动态调整出价或者参与率。还有就是把预算分配问题建模成一个控制问题，比如 PID 控制器，根据实际消耗速度和目标速度的偏差来调整。

**Q5. 广告里的 CVR 预估为什么难，有什么特有的挑战？**

> CVR 的训练样本是"点击了但没转化"和"点击了且转化"，问题是转化信号延迟——用户今天点击，可能三天后才下单，这期间这条样本算正还是负？如果等太久，模型更新不及时；如果不等，会有大量假负例。另外 CVR 有 SSB（Sample Selection Bias）问题——只有点击的样本才有 CVR 标签，但实际上我们想预估的是曝光后的转化率，这两个分布不一样。阿里的 ESMM 就是为了解决这个问题，把 CVR 和 CTR 联合训练，用全量曝光样本。

# 🎯 推荐方向 50题

## 精选深度题（含口语化参考答案）

**Q1. 召回为什么要多路，每路解决什么问题？**

> 单路召回肯定有盲区。比如你只用 i2i（item-to-item 协同过滤），那新用户没有历史行为就完全没法召回了。你只用 u2i 向量召回，那热门 item 的质量不一定高，而且新 item 冷启动很难。所以实际上都是多路并行：u2i 向量召回（用户兴趣）、i2i 向量召回（关联 item）、u2tag2i（用户标签→商品标签匹配）、热门召回（保底）、实时行为召回（刚看过的关联品）……这些路各有侧重，最后合并去重，送进粗排。

**Q2. 你怎么看 Two-Tower 模型的局限性，有什么改进？**

> 双塔最大的问题是 user 和 item 完全分开编码，相互之间没有信息交流，所以特征交叉只能靠最后的 dot product，表达能力有限。改进思路有几个：一是 late interaction，比如 ColBERT 思路，在 item 侧多保留几个 embedding，最后做 MaxSim；二是把部分 user 特征加到 item 侧做 pre-filtering；三是训练时用更强的 hard negative，让模型学得更好；四是用蒸馏，用 cross-encoder 的预测来监督 bi-encoder 的训练。

**Q3. 推荐系统里 Embedding 怎么训练？负样本怎么选？**

> Embedding 训练通常是矩阵分解或者 graph embedding 或者行为序列建模。负样本选择是最关键的——随机负样本（从全量 item 里随机选）太简单，模型学不到有效的判别能力，因为随机 item 和正样本差异太大；但 hard negative（跟正样本很像但没被点击的）如果采得太激进，会引入噪音。一般做法是混合采样：大部分随机负，一小部分 hard negative（比如同类目下没点击的、曝光了但跳过的）。还有 in-batch negative，就是把同一个 batch 里其他样本的 item 当负样本，这个实现简单效果也不错。

**Q4. 推荐系统的 Bias 有哪几种，怎么处理？**

> 主要有这几种：位置偏置（头部位置 CTR 天然高，不代表 item 质量好）、流行度偏置（热门 item 被曝光多，数据更多，模型更偏向推热门）、曝光偏置（没曝光的 item 没有负反馈，但可能是因为根本没机会曝光）、确认偏置（用户持续看某类内容是因为我们一直推，不代表他真的喜欢）。处理方式：位置偏置用 propensity score 校正或者训练时把位置当特征加入；流行度偏置在 item embedding 里做 popularity debiasing；曝光偏置用 ULRMF 之类的显式建模。

**Q5. 推荐系统里怎么做实时性？**

> 分几层：特征实时性——用户最近 5 分钟的行为要实时更新，这依赖 feature store 的流式写入（Kafka + Redis）；模型实时性——全量模型训练慢，但可以用增量更新，比如每小时跑一次增量训练，或者用在线学习；还有就是 session-level 的实时，用 GRU 或者 transformer 对用户当前 session 行为建模，体现"当下兴趣"。最关键的权衡是实时性和稳定性——实时模型容易过拟合刚刚发生的突发事件，要做一些平滑。

**Q6. 为什么推荐系统里要做 re-ranking，不直接用精排结果？**

> 精排是对每个 item 单独打分，没有考虑 item 之间的关系。re-ranking 主要解决两个问题：多样性（精排 top-20 可能全是同类商品，用户体验差）和上下文感知（展示顺序会相互影响，比如第一个 item 很突出，后面的会被影响）。多样性可以用 MMR、DPP（行列式点过程）来做；上下文感知用 listwise 的模型，把候选集合作为整体输入来打分。re-ranking 的成本比精排高，因为要考虑组合，一般只在最后几十个 item 里做。

**Q7. 冷启动问题，新用户和新 item 分别怎么解决？**

> 新用户（没有历史行为）：短期靠人口统计特征（年龄/地区/设备）做粗略兴趣判断，然后用探索策略快速收集行为信号，比如强制推几个类目的 item，根据停留和点击快速更新画像。也可以用注册时的兴趣选择做 bootstrapping。长期用 meta-learning，让模型有能力从少量行为快速适应。新 item（没有曝光历史）：靠 item 内容特征（文本、图片、类目）的 embedding 来找相似 item，用相似 item 的历史做迁移；另外要做流量扶持，给新 item 一定比例的保底曝光机会，让它能积累数据。
