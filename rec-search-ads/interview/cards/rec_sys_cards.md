## L1 概念卡（What）—— 10张

### L1-01 什么是推荐系统全链路漏斗？

**Q:** 推荐系统为什么要分召回、粗排、精排、重排这几层？

**A（30秒版）:**
推荐系统面对亿级候选物品，直接跑复杂模型成本极高。漏斗分层是**精度与效率的平衡术**：
- **召回**：亿级 → 万级，快速过滤，用轻量模型（双塔、CF）找"有可能感兴趣的"
- **粗排**：万级 → 千级，初步排序，用中等复杂度模型筛选
- **精排**：千级 → 百级，复杂模型精细打分，追求精度极致
- **重排**：百级 → 十级，考虑多样性、业务规则、上下文，最终曝光列表

**记忆锚点:** 🏠 像招聘流程——海投简历（召回）→ HR初筛（粗排）→ 技术面（精排）→ 最终offer决策（重排）。每层淘汰大量候选，越往后越精细。

### L1-02 什么是协同过滤（CF）？

**Q:** 协同过滤的核心思想是什么？UserCF 和 ItemCF 有什么区别？

**A（30秒版）:**
协同过滤（Collaborative Filtering）核心思想：**"物以类聚，人以群分"**
- **UserCF**：找到与你行为相似的人，把他们喜欢的推给你。适合兴趣变化快、社交性强的场景（新闻）
- **ItemCF**：找到与你历史交互物品相似的物品推给你。适合物品关系稳定的场景（电商）
- 不需要知道物品内容（content-agnostic），只靠"协作"行为矩阵

**记忆锚点:** 👥 UserCF = "和你一起买过相同东西的人还买了什么" → 亚马逊早期；ItemCF = "买了A的人还买了B" → 购物车推荐。

### L1-03 什么是双塔模型（DSSM）？

**Q:** 双塔模型的结构是什么？为什么叫"双塔"？

**A（30秒版）:**
双塔模型（DSSM / Two-Tower）：**用户塔**和**物品塔**各自独立编码，最后内积计算相似度。
- 用户塔：输入用户特征（历史行为、人口属性）→ DNN → 用户Embedding（d维向量）
- 物品塔：输入物品特征（ID、内容、标签）→ DNN → 物品Embedding（d维向量）
- 相似度：`score = u · v`（内积）或 `cos(u, v)`
- 物品Embedding可**离线预计算**，在线只需实时算用户Embedding + ANN检索

**记忆锚点:** 🗼 两座塔各自高高矗立，最后用"距离远近"衡量缘分——塔越近，越配对。

---

### L1-04 什么是矩阵分解（MF）？

**Q:** 矩阵分解如何解决协同过滤的稀疏性问题？

**A（30秒版）:**
将用户-物品交互矩阵 R（极度稀疏）分解为两个低秩矩阵：
`R ≈ U × V^T`
- U（m×k）：用户潜在因子矩阵
- V（n×k）：物品潜在因子矩阵
- k（隐向量维度）远小于 m、n，通过学习隐空间中的语义表示克服稀疏性
- 经典算法：SVD、ALS（交替最小二乘）、BPR（贝叶斯个性化排序）

**记忆锚点:** 🧩 把稀疏的"人×物品"大矩阵，压缩成"人的口味基因"×"物品风格基因"——用少量基因解释所有偏好。

---

### L1-05 什么是 HNSW？

**Q:** HNSW 是什么？在推荐系统中解决什么问题？

**A（30秒版）:**
HNSW（Hierarchical Navigable Small World）：一种**近似最近邻（ANN）检索**算法，用于在海量向量中快速找到最相似的 Top-K 向量。
- 解决问题：双塔模型产出用户Embedding后，需要在亿级物品Embedding中找最近邻，暴力枚举O(n)太慢
- 核心：构建多层图结构，高层稀疏（长距离跳跃）→低层密集（精细搜索）
- 时间复杂度：O(log n)，精度可达 95%+
- 工业实现：Faiss（Facebook）、Milvus、Proxima（阿里）

**记忆锚点:** 🗺️ 像地图导航——先看全国地图找大概方向（高层图），再缩小到市区找具体路线（低层图），比逐条路比较快得多。

---

### L1-06 什么是 DIN（深度兴趣网络）？

**Q:** DIN 相比普通 DeepCTR 模型解决了什么问题？

**A（30秒版）:**
DIN（Deep Interest Network，阿里2018）解决：用户兴趣多样，**不同目标物品激活不同兴趣**的问题。
- 传统模型：把历史行为序列 sum/avg pooling → 丢失了与目标物品的相关性
- DIN：对历史行为序列做**注意力加权**，目标物品和历史物品越相关，权重越高
- 本质：让模型"聚焦"于当前目标物品最相关的历史兴趣，而非所有历史的平均

**记忆锚点:** 👀 看电影推荐时，系统关注的是你看过的"科幻电影"历史而非"爱情电影"——DIN让模型有了选择性"注意力"。

### L1-07 什么是 MMR（最大边际相关）？

**Q:** MMR 算法如何在推荐中实现多样性？

**A（30秒版）:**
MMR（Maximal Marginal Relevance）：重排时平衡**相关性**和**多样性**的贪心算法。
- 每次从候选集中选下一个物品，选择：`argmax[λ·相关性 - (1-λ)·与已选集合最大相似度]`
- λ=1：纯相关性排序；λ=0：纯多样性；λ∈(0,1)：权衡
- 贪心逐步构建最终列表，时间复杂度 O(n·k)

**记忆锚点:** 🎨 调色板原则——每次加一种颜色时，选"又好看又和已有颜色不撞色"的，避免全是蓝色或全是红色。

---

### L1-08 什么是 DPP（行列式点过程）？

**Q:** DPP 如何比 MMR 更优雅地建模多样性？

**A（30秒版）:**
DPP（Determinantal Point Process）：用**行列式**衡量集合的多样性质量。
- 核矩阵 L：`L_{ij} = r_i · r_j · sim(i,j)`（相关性 × 物品间相似度）
- 集合 Y 的概率：`P(Y) ∝ det(L_Y)`（行列式越大，集合既相关又多样）
- 行列式几何意义：向量张成的超平行体体积——向量越"各不相同"，体积越大
- DPP vs MMR：DPP 有全局最优保证，MMR 是贪心近似

**记忆锚点:** 📐 行列式=向量组"张开"的空间体积。推荐的好列表 = "又高质量又各向延伸"，DPP用数学精确描述了这一点。

### L1-09 什么是 GAUC？

**Q:** 为什么推荐系统用 GAUC 而不是普通 AUC？

**A（30秒版）:**
GAUC（Group AUC）：对每个用户分别计算 AUC，再以用户曝光/点击量加权平均。
- 普通 AUC 问题：不同用户活跃度差异大，少数活跃用户主导全局AUC，掩盖其他用户体验
- GAUC：`GAUC = Σ(w_u × AUC_u) / Σw_u`，权重 w_u = 用户点击数或曝光数
- 更能反映**用户级别的排序质量**，是工业界推荐系统的核心离线评估指标

**记忆锚点:** 📊 期末考试平均分 vs 按班级加权平均——GAUC是后者，防止某个"学霸班"掩盖其他班成绩差的问题。

### L1-10 什么是 GNN 召回（图神经网络召回）？

**Q:** GNN 在召回中解决了什么问题？和双塔有什么区别？

**A（30秒版）:**
GNN 召回（如 PinSage、EGES）：将**用户-物品交互图**的结构信息引入 Embedding 学习。
- 双塔局限：用户和物品 Embedding 独立学习，没有利用图中"谁和谁共现"的高阶信息
- GNN：通过图上邻居聚合（Message Passing），把物品的邻居物品信息融入自身表示
- 例：商品A和商品B经常被同一批人买→在图上是近邻→GNN让A、B的Embedding更相近
- PinSage（Pinterest）：GraphSAGE变体，生产级别的大规模图召回

**记忆锚点:** 🕸️ 你的朋友圈影响你——GNN让每个物品"听取"经常和它一起被购买的物品的"意见"，更好地理解自己是谁。

## L2 原理卡（How）—— 9张

### L2-01 矩阵分解的优化目标是什么？

**Q:** BPR 损失函数是什么？为什么比 MSE 更适合推荐？

**A（原理版，含公式）:**

**BPR（Bayesian Personalized Ranking）损失：**
```
L_BPR = -Σ_{(u,i,j)} ln σ(r̂_ui - r̂_uj) + λ||Θ||²
```
- u：用户，i：正样本（有交互），j：负样本（无交互）
- `r̂_ui = p_u · q_i`（用户隐向量与物品隐向量内积）
- 直接优化**相对排序**（正样本分数 > 负样本分数），比 MSE 优化绝对评分更适合 Top-K 推荐
- σ = sigmoid 函数

**MSE vs BPR 对比：**
- MSE：`L = Σ(r_ui - r̂_ui)²`，优化评分值精度，但推荐不需要精确预测评分，只需排序正确
- BPR：优化排序对的正确性，更符合推荐任务本质

**一句话记忆:** BPR直接说"正样本要比负样本分高"，MSE说"分数要接近真实值"——推荐关心的是前者。

### L2-02 双塔模型的训练目标是什么？

**Q:** 双塔模型如何训练？In-batch Negative Sampling 是什么？

**A（原理版，含公式）:**

**训练目标（InfoNCE / Softmax Loss）：**
```
L = -Σ_i log[ exp(u_i · v_i / τ) / Σ_j exp(u_i · v_j / τ) ]
```
- u_i：用户i的Embedding；v_i：用户i正样本物品的Embedding
- τ：温度系数（控制分布平滑度）
- 分母中 j 遍历 batch 中所有物品（**批内负采样，In-batch Negative**）

**In-batch Negative Sampling 原理：**
- 把 batch 中其他用户的正样本物品当作负样本
- 优点：无需额外采样，天然高效；缺点：**采样偏差**（热门物品被采样为负样本概率更高）
- 修正：快手等提出频率修正 `score -= log(P(v_j))`

**相似度计算：**
```
sim(u, v) = u · v  (内积，考虑向量模长)
或
sim(u, v) = u · v / (||u|| · ||v||)  (余弦相似度)
```

**一句话记忆:** 双塔训练 = "把同一batch里别人的正样本当你的负样本"，用对比学习拉近正样本、推远负样本。

### L2-03 DIN 的注意力机制如何计算？

**Q:** DIN 的注意力权重公式是什么？

**A（原理版，含公式）:**

**DIN 注意力权重：**
```
e_i = a(v_a, e_i, v_a ⊙ e_i, v_a - e_i)
```
- v_a：目标物品（Ad）的 Embedding
- e_i：第i条历史行为物品的 Embedding
- `v_a ⊙ e_i`：元素乘积（交互特征）
- `v_a - e_i`：差值（距离特征）
- a(·)：小型 MLP，输出标量权重

**用户兴趣表示：**
```
v_u = Σ_i a_i · e_i   (加权求和，而非简单 sum pooling)
```
其中 `a_i = Softmax(e_i)` 或直接使用原始权重（DIN用 DICE 激活代替 PReLU）

**DICE 激活函数：**
```
f(s) = p(s) · s + (1-p(s)) · α·s
p(s) = sigmoid((s - E[s]) / √(Var[s] + ε))
```
数据驱动的自适应激活，比 PReLU 更灵活。

**一句话记忆:** DIN注意力 = "目标物品和历史物品做4维交互特征 → MLP打分 → 加权历史行为"。

### L2-04 DeepFM 如何统一 FM 和 DNN？

**Q:** DeepFM 的结构和 Wide&Deep 的区别是什么？

**A（原理版，含公式）:**

**Wide&Deep（Google 2016）：**
```
输出 = σ(W_wide · [x, φ(x)] + W_deep · a^(lf) + b)
```
- Wide 部分：LR（记忆能力，需人工特征交叉）
- Deep 部分：DNN（泛化能力）
- **问题**：Wide 部分需要人工特征工程

**DeepFM（华为2017）：**
```
输出 = sigmoid(y_FM + y_DNN)
y_FM = w_0 + Σ_i w_i·x_i + Σ_i<j <v_i, v_j>·x_i·x_j
y_DNN = DNN(embedding concat)
```
- FM 部分自动学习二阶特征交叉（无需手工）
- DNN 部分学习高阶交叉
- **关键**：FM 和 DNN **共享同一套 Embedding**，参数更高效
- 不需要 Wide&Deep 中的人工特征交叉

**一句话记忆:** DeepFM = 把 Wide&Deep 的 Wide 部分换成 FM，并共享 Embedding，自动二阶交叉 + DNN 高阶交叉。

### L2-05 HNSW 的构建和检索原理？

**Q:** HNSW 多层图是如何构建的？查询时如何导航？

**A（原理版，含公式）:**

**构建过程：**
- 每个节点插入时，随机分配到层 l（层越高概率越低）：`l = ⌊-ln(uniform(0,1)) · m_L⌋`
- 在每一层，该节点与最近的 M 个邻居相连
- 高层：连接少、覆盖范围大（长程跳跃）；低层（第0层）：连接多、覆盖局部（精细搜索）

**查询过程（从最高层开始）：**
```
for layer in [L, L-1, ..., 0]:
    在当前层执行贪心搜索，找到最近邻节点
    以该节点为入口点进入下一层
在第0层：扩大搜索范围（ef_search 参数），返回 Top-K
```

**关键参数：**
- M：每层最大邻居数（影响精度和索引大小）
- ef_construction：构建时动态候选集大小（影响索引质量）
- ef_search：查询时动态候选集大小（影响查询精度/速度）

**时间复杂度：** O(log N) 查询，O(N log N) 构建

**一句话记忆:** HNSW = 多层"跳表"思想 + 小世界图（六度分隔）—— 高层大步跳，低层精细找。

### L2-06 DIEN 如何建模兴趣演化？

**Q:** DIEN 相比 DIN 做了什么改进？GRU 如何建模时序兴趣？

**A（原理版，含公式）:**

**DIEN（Deep Interest Evolution Network，阿里2019）两阶段：**

**阶段1 - 兴趣抽取层（Interest Extractor Layer）：**
```
h_t = GRU(e_t, h_{t-1})   ← 对历史行为序列建模
辅助损失: L_aux = -Σ log σ(h_t · e_{t+1}) - Σ log(1 - σ(h_t · e_t'))
```
e_{t+1}：下一个真实行为（正样本）；e_t'：负样本 → 监督 GRU 在每步学习兴趣状态

**阶段2 - 兴趣演化层（Interest Evolving Layer）：**
```
AUGRU (Attention Update GRU)：
u_t' = a_t · u_t   ← 注意力分数调制更新门
```
- a_t：当前目标物品对历史时刻 t 的注意力权重（同 DIN）
- 用注意力权重调制 GRU 的更新门，目标相关的历史步骤更新更多

**DIN vs DIEN：**
| 对比维度 | DIN | DIEN |
|---------|-----|------|
| 时序性 | 忽略时序 | GRU建模演化 |
| 序列长度 | 固定历史 | 考虑兴趣漂移 |
| 训练信号 | 主任务 | 辅助损失 + 主任务 |

**一句话记忆:** DIEN = DIN + GRU兴趣演化 + AUGRU注意力门控，用时序建模"今天的你和昨天的你不一样"。

### L2-07 COLD 粗排框架如何实现高效精排？

**Q:** 阿里 COLD 框架的核心思路是什么？

**A（原理版，含公式）:**

**COLD（Computing power cost-aware Online and Lightweight Deep pre-ranking，阿里2020）：**

**核心问题：** 精排模型太重，粗排若直接用精排，延迟不可接受；若用轻量模型，精度下降

**COLD 解决方案：**
```
特征工程层：SE-block（Squeeze-and-Excitation）自动特征选择
                ↓
            轻量级 DNN（少层少参数）
                ↓
            并行化推断（多路并行）
```

**SE-block 特征选择：**
```
s = Sigmoid(W_2 · ReLU(W_1 · GlobalAvgPool(feature)))
output = feature × s   ← 通道注意力，自动选重要特征
```

**系统级优化：**
- 并行计算多个候选物品的 Embedding
- 使用 SIMD 指令向量化计算
- 工程目标：在响应时间约束下最大化模型复杂度

**粗排 vs 精排取舍：**
| 维度 | 粗排 | 精排 |
|-----|------|------|
| 模型复杂度 | 低（ms级） | 高（百ms级） |
| 特征交叉 | 有限 | 充分 |
| 候选数量 | 万→千 | 千→百 |
| 精度要求 | 相对排序 | 绝对精度 |

**一句话记忆:** COLD = 用SE-block自动筛特征 + 并行推断，在有限算力内把精排能力"移植"到粗排。

### L2-08 DPP 的数学原理是什么？

**Q:** DPP 核矩阵如何构建？行列式如何衡量多样性？

**A（原理版，含公式）:**

**DPP 核矩阵分解：**
```
L_{ij} = q_i · φ(s_i, s_j) · q_j
```
- q_i：物品 i 的相关性得分（精排分）
- φ(s_i, s_j)：物品 i、j 的相似度核（如 cos 相似度）
- 矩阵形式：`L = diag(q) · S · diag(q)`，S 为相似度矩阵

**选集概率：**
```
P(Y ⊆ Z) ∝ det(L_Y)
```
det(L_Y)：子集 Y 的核矩阵行列式

**行列式的几何意义：**
```
det([v1, v2, ..., vk]) = 向量v1...vk 张成的 k 维平行体体积
```
- 若 v1≈v2（物品相似）→ 行列式 ≈ 0 → 概率低
- 若 v1⊥v2（物品多样）→ 行列式大 → 概率高

**工业近似（贪心 MAP 推断）：**
```
Y* = argmax det(L_Y)  ← NP-hard，用贪心近似
每步选择 argmax_i det(L_{Y∪{i}})
```
时间复杂度：O(k²n)，k为最终集合大小，n为候选数

**一句话记忆:** DPP行列式 = 向量体积 = 多样性，选出的物品集合像"一组互相垂直的向量"，既大（相关）又散（多样）。

### L2-09 特征交叉方法的演进？

**Q:** FM、FFM、DCN、xDeepFM 在特征交叉上各有什么创新？

**A（原理版，含公式）:**

**FM（因子分解机）：**
```
y = w_0 + Σ_i w_i·x_i + Σ_{i<j} <v_i, v_j>·x_i·x_j
```
用隐向量内积表示二阶交叉，O(kn) 复杂度

**FFM（Field-aware FM）：**
```
y = w_0 + Σ_{i<j} <v_{i,f_j}, v_{j,f_i}>·x_i·x_j
```
每个特征对不同 field 学习不同隐向量，更精细但参数量 O(nfk)

**DCN（Deep & Cross Network，Google2017）：**
```
Cross层: x_{l+1} = x_0 · x_l^T · w_l + b_l + x_l
```
显式建模多阶特征交叉，第 l 层包含 x_0 的 l+1 阶交叉

**xDeepFM（交叉压缩网络，微软2018）：**
```
CIN（Compressed Interaction Network）：
X^k_{h,*} = Σ_{i=1}^{H_{k-1}} Σ_{j=1}^{m} W_{ij}^{k,h} · (X^{k-1}_{i,*} ⊙ X^0_{j,*})
```
在向量级别（而非bit级别）做特征交叉，保留Embedding空间结构

**一句话记忆:** FM用向量内积做二阶交叉 → DCN用Cross层做显式多阶 → xDeepFM在Embedding维度做向量级高阶交叉。

### L3-01 召回策略如何选型？

**Q:** 面对不同业务场景，应该选择哪种召回策略组合？

**A（决策框架）:**

```
业务场景决策树：
├── 物品有内容信息？
│   ├── 是 → 可用内容召回（TF-IDF / BERT Embedding）
│   └── 否 → 纯 CF 或双塔
│
├── 用户行为稀疏（冷启动）？
│   ├── 是 → 基于内容/热门/标签/知识图谱
│   └── 否 → CF / 双塔 / GNN
│
├── 实时性要求？
│   ├── 极高（<10ms）→ 基于预计算Embedding + ANN
│   └── 中等 → 在线特征召回
│
└── 个性化深度？
    ├── 浅 → ItemCF / 热门
    └── 深 → 双塔 + GNN + 序列召回
```

**多路召回融合策略：**
- 并行多路（CF + 双塔 + GNN + 热门）：覆盖度最大
- 分数校准：各路召回分数不可比，需归一化到同一空间
- 权重分配：可学习（训练粗排时联合优化）

**工业实践口诀:** 多路并行保覆盖，轻量粗排保质量，不要迷信单一召回。

**决策口诀:** 冷启动找内容，有行为上双塔，图结构用GNN，最终多路融合 = 召回不翻车。

### L3-02 何时用 DIN，何时用 DIEN？

**Q:** DIN 和 DIEN 的选型决策是什么？

**A（决策框架）:**

**核心问题：** 用户兴趣是否随时间演化？历史是否越新越重要？

```
选择决策：
├── 历史行为序列短（<50）且兴趣相对稳定 → DIN 足够
├── 历史行为序列长（>100）且有明显时序规律 → DIEN
│   ├── 例：电商购物路径（浏览→加购→下单）
│   └── 例：内容消费（短视频的兴趣漂移）
├── 系统资源受限（粗排/移动端） → DIN（更轻量）
└── 追求极致精排效果 → DIEN 或 BST（Transformer序列）
```

**工程取舍：**
| 对比 | DIN | DIEN |
|-----|-----|------|
| 训练复杂度 | O(T·d) | O(T·d) + GRU |
| 推理延迟 | 低 | 中 |
| 兴趣粒度 | 粗（加权平均） | 细（时序状态） |
| 适用场景 | 兴趣稳定 | 兴趣演化明显 |

**决策口诀:** 序列短用DIN，序列长且有时序规律上DIEN，算力富裕上Transformer序列建模。

### L3-03 离线指标高，但线上效果差，怎么分析？

**Q:** 离线 AUC 高但在线 CTR 不涨，可能的原因和解决方向是什么？

**A（决策框架）:**

**常见原因分析：**
```
离线→在线 Gap 原因：
1. 特征穿越（Feature Leakage）
   └── 离线训练时用了未来信息，线上无法获取
   
2. 样本分布差异（Distribution Shift）
   └── 离线样本来自历史曝光（有偏），线上是新分布
   
3. 线上线下特征不一致（Feature Gap）
   └── 离线有某特征，线上实时获取失败或延迟

4. 评估方式不合理
   └── AUC衡量排序，但真正的目标是某特定位置的CTR

5. 模型陈旧（Model Staleness）
   └── 离线测试集时间和线上时间差大，分布漂移
```

**排查清单：**
- [ ] 检查特征穿越（时间戳对齐）
- [ ] GAUC 是否和 AUC 趋势一致？
- [ ] 线上线下特征一致性监控（Feature Monitor）
- [ ] A/B 实验流量是否足够，统计显著性？

**决策口诀:** 先排穿越，再查分布，看特征一致性，最后看统计显著性——别轻信单一指标。

### L3-04 如何设计冷启动策略？

**Q:** 新用户和新物品冷启动分别怎么解决？

**A（决策框架）:**

**新用户冷启动：**
```
用户信息维度 → 策略选择
├── 有注册信息（年龄/性别/地区）→ 基于人口属性的规则召回
├── 有设备信息（机型/系统）→ 基于设备画像
├── 有引导行为（注册时兴趣选择）→ 基于兴趣标签
├── 有首次行为（前3次点击）→ 快速I2I相似推荐
└── 完全冷启动 → 热门多样性列表 + 快速收集反馈
```

**新物品冷启动：**
```
物品信息维度 → 策略选择
├── 有内容信息（标题/图片/标签）→ 内容Embedding相似推送
├── 有作者信息 → 利用作者历史粉丝推送
├── 流量扶持 → 赛马机制（小流量测试 → 指标好扩量）
└── 延迟Embedding更新 → 用侧信息（Side Info）代替ID Embedding
```

**双端冷启动终极方案：**
- 预训练 + 少样本微调（Meta-Learning）
- 用内容特征做物品 Embedding 的先验初始化

**决策口诀:** 新用户看"身份"做假设，新物品靠内容打头阵，都用赛马快速验证，别一刀切热门列表。

### L3-05 粗排和精排的工程取舍？

**Q:** 在系统资源约束下，如何在粗排和精排之间做取舍？

**A（决策框架）:**

**核心约束矩阵：**
```
┌──────────┬──────────────┬──────────────┬──────────────┐
│          │  粗排        │  精排        │  取舍原则    │
├──────────┼──────────────┼──────────────┼──────────────┤
│ 响应时间 │ <10ms        │ <100ms       │ 粗排更严格   │
│ 候选数量 │ 万→千        │ 千→百        │ 粗排量大     │
│ 特征丰富度│ 受限         │ 充分         │ 精排更丰富   │
│ 模型复杂度│ 简单(双塔等) │ 复杂(DIN等)  │ 各司其职     │
│ 更新频率 │ 相对稳定     │ 频繁更新     │ 精排更敏感   │
└──────────┴──────────────┴──────────────┴──────────────┘
```

**设计决策：**
- 粗排精度不足：考虑 COLD 框架（在时延约束内最大化模型复杂度）
- 精排延迟过高：特征预计算 + 模型蒸馏 + 异步预测
- 极端情况：合并粗排精排（全量精排，需要极高算力）

**决策口诀:** 粗排保速度广撒网，精排保精度细筛选，COLD是两者中间的工程艺术。

### L3-06 如何选择负采样策略？

**Q:** 不同负采样策略的适用场景和陷阱是什么？

**A（决策框架）:**

**负采样策略对比：**
```
策略          | 原理                  | 优点         | 陷阱
─────────────────────────────────────────────────────────
随机负采样    | 从全量物品随机采样    | 简单         | 假负样本问题
热度加权      | 按热度^α 采样(α<1)   | 减少假负样本  | α难调
批内负采样    | 同批次其他正样本      | 高效         | 热门物品被过多采样
Hard Negative | 召回结果中未点击      | 训练更难     | 引入噪声
曝光未点击    | 真实曝光未点击        | 最接近线上   | 数量有限
```

**假负样本问题：**
用户实际上喜欢物品 j，但 j 被当作负样本 → 训练信号错误

**推荐策略：**
- 双塔召回训练：批内负采样 + 热度频率修正
- 精排训练：曝光未点击 + Hard Negative（ETA策略）
- 避免把用户历史正样本采样为负样本

**决策口诀:** 召回训练批内负采样但修正热度偏差，精排训练优先用真实曝光负样本 + 少量Hard Negative。

### L4-01 召回率下降排查

**场景描述:** 某天早上监控告警，推荐系统召回率（Recall@100）从95%下降到78%，系统负责人需要快速定位原因。

**Q:** 召回率突然下降10%以上，如何系统性排查？

```
多路召回监控 → 分路统计：
- CF召回率：X%
```

5. **候选集过滤规则**：是否有业务规则变更导致大量物品被过滤？

**第三步：快速恢复**
- 回滚到上一个稳定版本
- 增大热门兜底比例
- 临时降低候选集过滤阈值

**第四步：长期改进**
- 建立分路召回监控看板
- 关键数据管道加延迟告警

### L4-02 为新用户设计冷启动策略

**场景描述:** 产品经理反馈，新用户（注册后7天内）的次日留存率明显低于老用户，认为推荐效果差是主因。

**Q:** 如何从算法侧设计一套完整的新用户冷启动策略？

**A（结构化回答）:**

**阶段1：注册即刻（T=0，0条行为）**
- 收集注册信息（年龄、性别、地区、职业）

**阶段4：第4-7天（T=4d~7d，>100条行为）**
- 逐渐过渡到常规个性化策略

- 实时特征流（用户行为 < 1min 内进入模型）
- 梯度更新频率提高（新用户模型更新更频繁）
- 冷启动专用模型 vs 统一模型（根据数据量决策）

### L4-03 多样性与相关性如何平衡？

**场景描述:** 用户反馈推荐越来越"窄"，总是推同类内容（过滤气泡），但如果加太多多样性，CTR 下降，产品经理和算法工程师产生分歧。

**Q:** 如何在业务和算法两个层面解决推荐多样性问题？

**A（结构化回答）:**

**度量多样性：**
```
Intra-List Diversity (ILD) = 1 - (1/K(K-1)) Σ_{i≠j} sim(i,j)
Coverage = |推荐物品集合| / |全量物品|
Novelty = 推荐物品的平均 -log(popularity)
```

**算法侧方案（按复杂度排序）：**
1. **MMR（最简单）**：重排时贪心加多样性约束，λ 调节
2. **DPP（更优雅）**：行列式过程，全局多样性保证
3. **类目强制打散**：硬规则，相邻物品类目不同
4. **Slot-based 重排**：每个位置有类目/风格约束

**业务侧方案：**
- 用户主动选择多样性偏好（"多给我推不同内容"按钮）
- 长短期兴趣分离建模（短期点击 + 长期收藏）
- 曝光去重：已曝光物品降权

**效果评估：**
- 不能只看 CTR，需要同时监控 ILD、用户留存（7天/30天）
- 多样性的收益是长期的，短期 CTR 可能略降

### L4-04 推荐系统实时性优化

**场景描述:** 电商大促期间，用户的购买兴趣变化极快（刚买完运动鞋不应再推运动鞋），但当前系统特征延迟约30分钟，无法及时响应。

**Q:** 如何将推荐系统特征实时性从30分钟降到1分钟内？

**A（结构化回答）:**

**问题分析：**
```
当前架构（T+30min）：
用户行为 → Kafka → Flink处理 → 写入HBase → 模型读取

目标架构（T+1min）：
用户行为 → Kafka → 实时流处理 → 在线特征存储 → 实时模型
```

**实时化改造方案：**

**1. 实时行为特征（<10s）：**
- 用户最近 N 条行为序列实时写入 Redis
- 推理时实时拼接（不走离线特征流）

**2. 实时统计特征（1-5min）：**
- Flink 计算滑动窗口统计（近5分钟点击数、类目偏好）
- 写入低延迟 KV 存储（Redis/Tair）

**3. 模型端改造：**
- 实时特征服务（Feature Store）统一管理
- 模型参数在线更新（Online Learning / FTRL）

**4. 缓存与容灾：**
- 实时服务降级兜底：缓存最近1小时特征
- 特征缺失时用离线特征替代，加缺失标记特征

**关键指标：** 特征延迟 P99、特征命中率、召回率变化

## L5 陷阱卡（Trap）—— 5张

### L5-01 AUC 高就代表推荐效果好？

**Q（陷阱）:** 离线测试 AUC=0.82，上线后 CTR 却没有提升，你认为模型没问题，可以继续推进。这个思路有什么问题？

**A:**
**AUC ≠ 推荐效果好**，多个陷阱叠加：

1. **全局 AUC vs GAUC**：少数活跃用户的高 AUC 拉高全局，实际上大多数普通用户体验没改善
2. **AUC 衡量排序，不衡量位置**：AUC 高说明相对排序准，但排名第1和第10对用户的价值截然不同
3. **测试集分布 ≠ 线上分布**：测试集来自历史曝光，线上有新物品、新用户行为
4. **业务目标错位**：你的 AUC 目标是 CTR，但产品真正关心的可能是 GMV、留存率

**正确做法：**
- 同时看 GAUC（用户级 AUC）
- 关注 NDCG@K（位置敏感指标）
- AUC 提升是必要条件，不是充分条件
- 最终以 A/B 实验的在线指标为准

**面试金句:** "AUC 提升是做 A/B 实验的门票，不是进宫的钥匙。"

---

### L5-02 双塔模型用余弦相似度还是内积？

**Q（陷阱）:** 双塔模型召回，应该一律用余弦相似度，因为余弦相似度归一化了向量长度，更稳定。

**A:**
**不是绝对的！** 内积 vs 余弦各有适用场景：

**内积的优势（工业主流）：**
- 向量模长携带信息：流行度高的物品 Embedding 模长往往更大，内积会自然偏向热门，符合"热门物品受欢迎"的直觉
- ANN 检索（HNSW / Faiss）的内积模式效率高
- 微信看一看、YouTube DNN 等均用内积

**余弦的适用场景：**
- 需要消除流行度偏差时
- 物品 Embedding 维度差异大时

**真正的陷阱：**
- 训练时用余弦，检索时用内积（不一致！）→ 检索结果不准确
- 温度系数 τ 的设置和相似度计算方式强相关，换了相似度要重调 τ

**面试金句:** "余弦还是内积不重要，训练和推理要一致才重要，否则离线和在线必然有 Gap。"

---

### L5-03 负采样越难越好？

**Q（陷阱）:** Hard Negative 挖掘越难越好，我让模型每次都学习"最难"的负样本，这样模型会更强。

**A:**
**过难的负样本会损害训练！**

**假负样本问题（False Negative）：**
- "最难"的负样本可能是用户实际上喜欢但没机会曝光的物品（采样偏差）
- 把这些当负样本训练，等于告诉模型"用户不喜欢他其实喜欢的东西"→ 梯度方向错误

**实验现象：**
- Hard Negative 比例过高 → 训练 loss 下降很快，但离线 AUC 不升甚至下降
- 模型学到了"与训练分布对齐"而非"与真实用户偏好对齐"

**正确做法（ETA策略，阿里2022）：**
- 混合策略：70% 随机负样本 + 30% Hard Negative
- Hard Negative 来源：召回阶段排名靠前但未点击（更可信的负样本）
- 避免从历史正样本库中采样负样本

**面试金句:** "Hard Negative 是双刃剑，太软学不到判别力，太硬引入假负样本——比例是关键，黄金比例 7:3。"

---

### L5-04 序列长度越长，DIN 效果越好？

**Q（陷阱）:** 用户历史行为序列越长，DIN 能关注到更多历史信息，所以序列越长越好，应该把所有历史都加进去。

**A:**
**序列过长反而有害！**

**工程问题：**
- Attention 计算是 O(T) 的，T 过大显著增加推理延迟
- 训练样本内存占用线性增长

**效果问题：**
- 非常早期的行为（1年前）和当前意图相关性低
- 噪声行为（误点击）比例随序列增长而增加，稀释有效信号
- 注意力机制在超长序列上计算精度下降（数值稳定性）

**工业解决方案：**
- **SIM（Search-based Interest Model，阿里2020）**：先用快速检索从超长序列中找最相关的 Top-K 行为，再做精细 Attention
- **ETA（Efficient Target Attention）**：用局部敏感哈希（LSH）快速检索相关历史
- 实践：序列长度一般控制在 50-200 之间，超长用检索先筛

**面试金句:** "序列不是越长越好，是越相关越好——SIM用检索把'万里挑一'做到毫秒内，比直接上万步Attention聪明得多。"

---

### L5-05 重排用 DPP 就一定比 MMR 好？

**Q（陷阱）:** DPP 有严格的数学基础，一定比 MMR 这种启发式方法效果好，生产环境直接换 DPP 就行。

**A:**
**DPP 不一定胜过 MMR！** 理论美≠工程优

**DPP 的实际问题：**
1. **计算复杂度高**：核矩阵的 MAP 推断是 NP-hard，贪心近似的复杂度 O(k²n) 在候选集大（n>1000）时很慢
2. **核矩阵构建难**：相关性 q_i 和相似度 φ(s_i,s_j) 如何定义、量纲如何统一，调参复杂
3. **可解释性差**：行列式是黑盒，MMR 的 λ 参数业务侧更易理解和调控
4. **A/B 实验不一定赢**：多样性效果是长期指标（留存），短期 CTR 可能显著下降，推动困难

**MMR 的优势：**
- O(nk) 低复杂度，实时性好
- λ 参数直观，业务侧易于干预
- 工程上更稳定

**正确策略：**
- 小候选集（<200）：DPP 可行
- 大候选集（>500）：DPP 近似或 MMR
- 看 A/B 实验数据，不要先入为主

**面试金句:** "DPP 是数学家的最爱，MMR 是工程师的朋友——生产环境先用 MMR 快速验证多样性收益，再决定是否换 DPP。"

### 卡片：Mamba SSM vs Transformer 序列推荐

**核心机制**：
- Mamba 用**选择性状态空间**（SSM）替代 Self-Attention
- 状态转移：`h_t = exp(Δ_t A) · h_{t-1} + Δ_t·B_t·x_t`
- `Δ_t` 是输入自适应参数（大 = 关注当前，小 = 保持历史）

**关键对比**：

| | Transformer | Mamba |
|--|------------|-------|
| 复杂度 | O(n²) | O(n) |
| 推理显存 | O(n) KV Cache | O(1) 固定状态 |
| 序列上限 | ~4096 | 理论无限 |
| 选择性遗忘 | ❌（全局注意力） | ✅（Δ_t 控制） |

**推荐场景适配原因**：用户历史稀疏有价值信号，大部分是噪声 → Mamba 选择性遗忘天然契合。

**工业落地（MambaRec）**：延迟 280ms → 45ms，CTR +6.8%，季节性品类 GMV +23%

**面试一句话**：Mamba 不只是"更快的 Transformer"，是**选择机制**让它特别适合推荐中"值得记住的信号稀疏"的场景。

---

### 卡片：HSTU ReLU Pointwise Attention（Meta）

**改了什么**：把 `softmax(QKᵀ/√d)` 改成 `ReLU(QKᵀ)`

**为什么改**：
- softmax 在超长序列（推荐常见 2000-5000 token）下浪费算力：大量权重趋近 0 但还是要全局归一化
- ReLU 自然过滤负相关（输出 0），无需归一化，天然稀疏

**收益**：推理速度提升 3-5×，支持更长序列，允许单模型统一召回+排序。

**代价**：注意力权重不再是概率分布，训练稳定性更敏感（需 LayerNorm + 梯度裁剪）

**与 FlashAttention 区别**：FlashAttention 是 IO 优化（减少 HBM 读写），仍是 O(n²)；HSTU 是**算法级**改变，实际 FLOP 减少。

---

### 卡片：MTGR 双流融合（美团外卖）

**为什么不直接用纯 HSTU**：HSTU 序列建模忽略 side features（价格、品类、上下文），这些在 DLRM 框架中通过特征交叉被充分利用。

**双流设计**：
```
行为序列 → [HSTU 序列流] → h_seq
结构化特征 → [DLRM 特征流] → h_feat
→ Adaptive Gating → h_final
```

**Adaptive Gating**：`g = σ(W·concat(h_seq, h_feat))`，冷启动用户特征流权重自动上升。

**渐进式训练**：① 预训 HSTU ② 固定序列流训特征流 ③ 联合 fine-tune —— 防止两路相互干扰

**效果**：GAUC +2.88pp，GMV +2.1%

### 卡片：生成式推荐 Semantic ID 演进

| 方案 | 核心思路 | 问题 |
|------|---------|------|
| 随机 Item ID | 整数编号 | 无语义性 |
| RQ-VAE（TIGER）| 离散码本量化 | 量化误差 + 碰撞 + 梯度断裂 |
| Differentiable Semantic ID | Gumbel-Softmax 可微量化 | 端到端但仍有离散化 |
| UniGRec Soft Identifiers | 直接用连续向量，ANN 检索定位 | 存储大，但无碰撞无量化误差 |

**UniGRec 关键设计**：自回归生成下一个物品的连续向量 → FAISS ANN 检索最近邻定位物品

**Recall@10 对比**：Soft ID 比 RQ-VAE 高 5.8%，碰撞率从 3.2% → 0%

### 卡片：Test-Time Compute 推荐（PROMISE）

**核心思想**：推荐不是"一次贪心解码"，而是 Beam Search 探索多个候选列表，用 PRM（过程奖励模型）评估。

**PRM vs ORM**：
- ORM：只看最终列表质量 → 无法定位是哪步选错
- PRM：每步打分 → 密集监督信号，学习效率更高，效果好 2.3%

**Beam Search K=8 ≈ Best-of-64 效果**（PROMISE 实验），Recall@10 +9.1%。

**工业延迟方案**：
1. 离线大算力 Beam Search → 缓存 100 候选，在线轻量 PRM 重排（<5ms）
2. 分级：高价值用户 Beam Search，普通用户贪心解码
3. PRM 蒸馏 → 1-2 层轻量 PRM，<3ms

### 卡片：O1 Embedder（推理增强检索）

**核心改进**：在编码 query 前先生成"内部推理链"，拼接后再编码。

$$e_q = \text{Encoder}(\text{Think}(q) \oplus q)$$

**与传统 Query 扩展的区别**：
- 传统：先改 query 文字，再编码 → 改变了 query 表面形式
- O1 Embedder：内部推理，query 文字不变 → 完全兼容已有 ANN 索引，无需重建

**适用场景**：知识型 query（"比铝轻的常见金属"），不适合高频简单 query（延迟代价大）
