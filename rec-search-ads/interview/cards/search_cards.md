## L1 概念卡（What）

### L1-01 · 倒排索引

**Q：什么是倒排索引（Inverted Index）？**

**A：**
倒排索引是信息检索中最核心的数据结构，由"词项→文档列表"的映射构成。

```
正排索引（Forward Index）：        倒排索引（Inverted Index）：
doc1 → ["苹果", "手机", "价格"]    "苹果" → [doc1, doc3, doc7]
doc2 → ["苹果", "水果", "甜"]      "手机" → [doc1, doc2, doc5]
doc3 → ["苹果", "iOS", "发布"]     "iOS"  → [doc3, doc6]
```

- **词项（Term）**：索引的基本单元（词/短语）
- **倒排列表（Posting List）**：包含该词的文档 ID + 词频 + 位置
- **优点**：O(1) 查找词项，快速求交集（AND查询）
- **应用**：Elasticsearch / Lucene 核心数据结构

---

### L1-02 · TF-IDF

**Q：TF-IDF 是什么？各部分含义？**

**A：**
TF-IDF = **词频（TF）× 逆文档频率（IDF）**，衡量词对文档的重要程度。

| 组成 | 公式 | 含义 |
|------|------|------|
| TF | f(t,d) / |d| | 词 t 在文档 d 中出现的频率 |
| IDF | log(N / df(t)) | 包含词 t 的文档越少，IDF 越高 |
| TF-IDF | TF × IDF | 高频但不普遍 → 重要 |

- **高 TF-IDF**：词在该文档中频繁出现，但在全语料中罕见（专有名词、关键词）
- **低 TF-IDF**："的"、"是"等停用词，到处都有，IDF≈0
- **局限**：无法处理同义词、语序信息

---

### L1-03 · BM25

**Q：BM25 是什么？相比 TF-IDF 有哪些改进？**

**A：**
BM25（Best Match 25）是 TF-IDF 的改进版，是工业搜索中使用最广泛的词袋模型。

**两大核心改进**：
1. **TF 饱和**：词频贡献有上限，避免词频很高时得分无限增长（参数 k₁）
2. **文档长度归一化**：长文档不应因词频绝对值高而获利（参数 b）

**应用**：Elasticsearch 默认排序算法、作为召回阶段基线

---

### L1-04 · Dense Retrieval（稠密检索）

**Q：什么是 Dense Retrieval？双塔模型结构如何？**

**A：**
Dense Retrieval 用神经网络将 Query 和文档编码为稠密向量，通过向量相似度检索。

```
双塔结构（Bi-Encoder）：

Query: "苹果手机"              Document: "iPhone 15 发布"
       ↓ Query Tower                    ↓ Doc Tower
   [BERT-like]                      [BERT-like]
       ↓                                 ↓
  q_vec ∈ R^d              ←ANN→    d_vec ∈ R^d
            cosine_similarity(q_vec, d_vec)
```

- **优点**：文档向量离线计算，在线仅需一次 Query 编码 + ANN 查询
- **代表**：DPR（Facebook）、ANCE（微软）
- **训练**：对比学习，正样本（相关文档）vs 负样本（不相关文档）

---

### L1-05 · FAISS / HNSW 向量检索

**Q：向量近邻检索（ANN）有哪些常用方法？**

**A：**

| 方法 | 全称 | 原理 | 特点 |
|------|------|------|------|
| FAISS-Flat | 精确暴力搜索 | 全量计算 | 精确但慢，小数据用 |
| FAISS-IVF | 倒排文件索引 | 聚类后只搜邻近桶 | 速度快，精度略降 |
| HNSW | 分层可导航小世界图 | 图结构多层跳跃 | 速度与精度均衡，工业常用 |
| PQ | 乘积量化 | 向量压缩后搜索 | 节省内存 |

- **HNSW 优点**：无需训练，增量插入友好，工业部署首选
- **FAISS 优点**：GPU 加速，批量检索高效

---

### L1-06 · Learning to Rank（LTR）

**Q：Learning to Rank 是什么？三种范式分别是？**

**A：**
LTR 是将机器学习用于文档排序的框架，学习目标是排序质量而非点预测。

| 范式 | 输入 | 损失函数视角 | 代表算法 |
|------|------|------------|---------|
| **Pointwise** | 单文档特征 | 回归/分类损失 | RankSVM(点), 线性回归 |
| **Pairwise** | 文档对 | 对的相对顺序 | RankNet, LambdaRank |
| **Listwise** | 整个候选列表 | 列表排序质量 | ListNet, LambdaMART |

---

### L1-07 · MMR（最大边际相关性）

**Q：MMR 是什么？解决什么问题？**

**A：**
MMR（Maximal Marginal Relevance）是重排阶段的多样性算法，避免搜索结果同质化。

```
MMR(dᵢ) = λ · Sim(dᵢ, Q) - (1-λ) · max_{dⱼ∈S} Sim(dᵢ, dⱼ)
```

- **第一项**：与 Query 的相关性（越高越好）
- **第二项**：与已选文档集合 S 的最大相似度（越低越好）
- **λ**：平衡相关性与多样性的超参数（λ=1时退化为纯相关性排序）
- **应用**：新闻聚合、电商搜索结果去重

---

### L1-08 · NDCG

**Q：NDCG 是什么？为什么用它评估搜索？**

**A：**
NDCG（Normalized Discounted Cumulative Gain，归一化折损累积增益）是搜索排序的核心评估指标。

- **Gain**：文档的相关性得分（0-3分）
- **Discount**：排名越靠后，权重越低（log₂(rank+1)）
- **Cumulative**：对整个列表求和
- **Normalized**：除以理想排序的 DCG（IDCG）

**优点**：考虑了相关性程度（多级标注）+ 位置偏差（靠前更重要）

---

### L1-09 · MRR 和 MAP

**Q：MRR 和 MAP 分别评估什么？**

**A：**

**MRR（Mean Reciprocal Rank，平均倒数排名）**：
- 关注第一个相关文档的位置
- 适合：问答检索、单一正确答案场景

**MAP（Mean Average Precision，平均精确率均值）**：
- 对所有相关文档的 Precision@k 求平均
- 适合：文档检索（多个相关文档）

| 指标 | 适合场景 | 关注点 |
|------|---------|--------|
| NDCG | 排序质量（多级相关） | 全列表质量 |
| MRR | 单一答案检索 | 第一个相关文档位置 |
| MAP | 多文档检索 | 所有相关文档位置 |

---

### L1-10 · 召回-排序两阶段架构

**Q：工业搜索为什么用两阶段（召回+排序）？**

**A：**

```
千万级文档库
     ↓ 召回（Recall）
  千级候选集（BM25 + 向量检索）
     ↓ 粗排（Coarse Ranking）
  百级候选集（轻量模型）
     ↓ 精排（Fine Ranking）
  十级结果（Cross-Encoder / 复杂特征模型）
     ↓ 重排（Re-ranking）
  最终展示结果（MMR / 业务规则）
```

**核心矛盾**：精度 vs 延迟
- 召回：高覆盖率，低延迟（ANN检索 <10ms）
- 精排：高精度，计算量大（只对少量候选排序）
- 两阶段分工解决了"既要精准又要快速"的工程矛盾

## L2 原理卡（How）

### L2-01 · BM25 完整公式推导

**Q：BM25 完整公式是什么？每个参数如何影响得分？**

**A：**

```
BM25(q, d) = Σᵢ IDF(qᵢ) × TF_sat(qᵢ, d)

IDF(qᵢ) = log[ (N - n(qᵢ) + 0.5) / (n(qᵢ) + 0.5) ]

TF_sat(qᵢ, d) = f(qᵢ,d) × (k₁ + 1)
                ─────────────────────────────────────────
                f(qᵢ,d) + k₁ × (1 - b + b × |d| / avgdl)
```

**参数说明**：

| 参数 | 典型值 | 含义 |
|------|--------|------|
| N | 语料总文档数 | 计算 IDF 分母 |
| n(qᵢ) | 含词 qᵢ 的文档数 | 词的稀有程度 |
| f(qᵢ,d) | 词 qᵢ 在文档 d 中的原始词频 | 基础 TF |
| k₁ | 1.2 ~ 2.0 | TF 饱和控制：k₁→∞ 退化为 TF，k₁=0 退化为 binary |
| b | 0.75 | 长度归一化强度：b=0 无归一化，b=1 完全归一化 |
| avgdl | 平均文档长度 | 归一化基准 |

**TF 饱和效果**：
```
TF=1 → 得分≈0.85  (k₁=1.2)
TF=5 → 得分≈1.56
TF=∞ → 得分→ k₁+1 = 2.2  （上限）
```

**为什么 IDF 加 0.5 平滑**：避免 n(qᵢ)=N 时 IDF=0（全文档都包含该词，如停用词）

### L2-02 · NDCG 计算方法

**Q：NDCG 如何计算？给出完整步骤和公式。**

**A：**

**Step 1：定义相关性得分（Relevance Grade）**
```
3 = 完全相关，2 = 高度相关，1 = 一般相关，0 = 不相关
```

**Step 2：计算 DCG@K**
```
DCG@K = Σᵢ₌₁ᴷ  relᵢ / log₂(i + 1)

或使用指数版（更常用）：
DCG@K = Σᵢ₌₁ᴷ  (2^relᵢ - 1) / log₂(i + 1)
```

**Step 3：计算 IDCG@K（理想排序的DCG）**
```
IDCG@K = 将所有文档按相关性从高到低排序后的 DCG@K
```

**Step 4：NDCG@K = DCG@K / IDCG@K**

**示例**：
```
系统返回：rel = [3, 2, 0, 1, 2]（排名1-5）
DCG@5 = 3/log₂(2) + 2/log₂(3) + 0/log₂(4) + 1/log₂(5) + 2/log₂(6)
       = 3/1 + 2/1.585 + 0 + 1/2.322 + 2/2.585
       = 3 + 1.262 + 0 + 0.431 + 0.774 = 5.467

理想排序：rel = [3, 2, 2, 1, 0]
IDCG@5 = 3 + 1.262 + 2/2 + 1/2.322 + 0 = 3 + 1.262 + 1 + 0.431 = 5.693

NDCG@5 = 5.467 / 5.693 = 0.960
```

### L2-03 · LambdaRank 核心思想

**Q：LambdaRank 如何将排序指标（NDCG）融入梯度下降？**

**A：**

**背景**：NDCG 不可导，无法直接用梯度下降优化。

**RankNet 基础**：学习文档对的偏序关系
```
P(dᵢ > dⱼ) = σ(sᵢ - sⱼ)  （sigmoid 函数）
Loss = -log P(dᵢ > dⱼ)（交叉熵）
λᵢⱼ = ∂Loss/∂sᵢ = -σ(sⱼ - sᵢ)
```

**LambdaRank 关键创新**：在 RankNet 的梯度上乘以 |ΔNDCG|
```
λᵢⱼ_Lambda = λᵢⱼ_RankNet × |ΔNDCG(i,j)|
```

- **|ΔNDCG(i,j)**：交换文档 i 和 j 排名后，NDCG 的变化量
- **效果**：NDCG 影响大的文档对，梯度更大，优先修正
- **无需显式计算NDCG**：只用 |ΔNDCG| 作为梯度缩放因子

**LambdaMART**：将 LambdaRank 的梯度用 GBDT（MART）拟合
```
LambdaMART = LambdaRank（梯度） + MART（GBDT学习器）
```

### L2-04 · MRR 和 MAP 计算

**Q：MRR 和 MAP 的计算公式是什么？**

**A：**

**MRR（Mean Reciprocal Rank）**：
```
MRR = (1/|Q|) × Σ_q  1/rank_q

其中 rank_q = 第 q 个 Query 的第一个相关文档排名
```

示例：
```
Q1: 第1个相关文档在位置 1 → 1/1 = 1.0
Q2: 第1个相关文档在位置 3 → 1/3 = 0.333
Q3: 第1个相关文档在位置 2 → 1/2 = 0.5
MRR = (1+0.333+0.5)/3 = 0.611
```

**MAP（Mean Average Precision）**：
```
AP(q) = (1/R_q) × Σₖ P(k) × rel(k)

其中 R_q = 相关文档总数，P(k) = Precision@k，rel(k) = 第k位是否相关
MAP = (1/|Q|) × Σ_q AP(q)
```

示例（Q1，3个相关文档，排名结果 rel=[1,0,1,0,1]）：
```
P@1=1/1=1.0(相关), P@3=2/3(相关), P@5=3/5(相关)
AP = (1.0 + 2/3 + 3/5) / 3 = (1.0 + 0.667 + 0.6) / 3 = 0.756
```

### L2-05 · HNSW 索引原理

**Q：HNSW 如何实现高效近邻搜索？**

**A：**
HNSW（Hierarchical Navigable Small World）构建多层图结构：

```
Layer 2: ●────────●                 （少量节点，长距离跳跃）
          \       /
Layer 1:  ●──●──●──●               （中层，中等连接）
          / \/  \/ \
Layer 0: ●─●─●─●─●─●─●─●           （底层，所有节点，精细搜索）
```

**搜索过程**：
1. 从顶层（最稀疏层）入口节点开始
2. 贪心搜索：跳到与 Query 最近的邻居
3. 到达当前层最优节点后，下沉到下一层
4. 底层精细搜索，返回 top-K 结果

**关键参数**：
- `M`：每个节点最大连接数（M=16常用）
- `ef_construction`：建图时候选集大小（精度 vs 建图速度）
- `ef_search`：搜索时候选集大小（精度 vs 查询速度）

**优点**：查询复杂度 O(log N)，无需训练，支持增量更新

### L2-06 · 稀疏 vs 稠密检索对比

**Q：稀疏检索和稠密检索的核心差异？什么时候用哪个？**

**A：**

| 维度 | 稀疏检索（BM25/SPLADE） | 稠密检索（DPR/ANCE） |
|------|----------------------|-------------------|
| 表示 | 高维稀疏向量（词袋） | 低维稠密向量（128-768维） |
| 词汇匹配 | ✅ 精确词匹配 | ❌ 词汇不匹配问题 |
| 语义理解 | ❌ 无法处理同义词 | ✅ 语义相似可召回 |
| 冷启动 | ✅ 无需训练 | ❌ 需大量标注训练 |
| 新词/OOV | ✅ 直接索引新词 | ❌ 需更新模型/索引 |
| 延迟 | ✅ 极低（倒排索引） | 需 ANN（HNSW等）|
| 可解释性 | ✅ 词匹配可追踪 | ❌ 黑箱 |

**SPLADE（语义稀疏）**：稀疏 + 语义的折中
- 用 BERT 生成稀疏向量（大多数维度为0）
- 兼顾倒排索引的效率 + 语义扩展能力

**混合检索（推荐）**：
```
final_score = α × BM25_score + (1-α) × dense_score
```

### L2-07 · Query 意图识别

**Q：搜索中 Query 意图识别如何实现？有哪些主要意图类型？**

**A：**

**主要意图类型**（TREC Web Track 分类）：
```
导航型（Navigational）：找特定网站，如"百度官网"
信息型（Informational）：获取信息，如"BM25是什么"
事务型（Transactional）：完成动作，如"下载Python"
商业型（Commercial）：购买意图，如"苹果手机价格"
```

**实现方式**：
1. **规则 + 关键词**：高精度，低覆盖，可解释
2. **分类模型（FastText/TextCNN）**：快速，离线训练
3. **BERT Fine-tune**：准确率高，延迟稍高
4. **LLM 零样本**：泛化强，成本高

**下游影响**：
- 导航型 → 优先返回官网
- 事务型 → 优先返回操作入口
- 信息型 → 优先返回知识卡片/详细文档

### L2-08 · Cross-Encoder 精排原理

**Q：Cross-Encoder（精排）和双塔（召回）的区别？为什么精排更准？**

**A：**

```
双塔（Bi-Encoder）：                  Cross-Encoder（精排）：
Query ─→ Encoder → q_vec             [Query, Document] 拼接
Doc   ─→ Encoder → d_vec             ─────────────────────
cosine(q_vec, d_vec) = score         → BERT → [CLS] → score
```

**关键差异**：

| 维度 | 双塔 | Cross-Encoder |
|------|------|--------------|
| 交互方式 | 向量内积（独立编码） | 全注意力交叉 |
| 精度 | 中等 | 高 |
| 延迟 | 低（离线Doc编码） | 高（在线联合编码） |
| 可扩展性 | ✅ 亿级文档 | ❌ 只能百级候选 |
| 使用阶段 | 召回 | 精排 |

**Cross-Encoder 更准的原因**：Query 和 Document 的每个 token 都能互相 attention，捕捉细粒度交互信号（如 Query 中"不好"对 Doc 的否定关系）。

### L2-09 · Query 扩展与改写

**Q：Query 扩展有哪些方法？各自适用场景？**

**A：**

**伪相关反馈（PRF）**：
1. 用原始 Query 检索 top-K 文档
2. 提取高频词加入 Query
3. 重新检索
- 缺点：可能引入噪音（Query Drift）

**同义词扩展**：
- 词典法（WordNet、行业词表）
- 词向量近邻（word2vec 近义词）
- 应用：搜索"手机" → 扩展"智能手机、mobile、cellphone"

**Query 改写（Rewriting）**：
- NMT（神经机器翻译风格）：seq2seq 模型生成标准形式
- LLM 改写：更灵活，可处理上下文
- 场景：口语化 → 规范化，如"想买个不贵的包" → "平价女包"

**Back-Translation**：
- 原始 Query → 翻译成其他语言 → 翻译回来
- 获得等价但措辞不同的 Query，增强多样性

### L2-10 · 上下文感知重排

**Q：上下文感知重排（Context-aware Reranking）如何实现？**

**A：**
考虑已选文档集合和用户上下文，动态调整文档得分。

**方法 1：MMR（见L1-07）**

**方法 2：Setwise Reranking**
- 将候选集作为整体输入模型，输出整体最优排列
- 代表：PRP（Passage Reranking with Permutation）

**方法 3：用户会话上下文**
```
本次搜索结合：
- 历史搜索词（最近 N 次）
- 历史点击（购买、收藏）
- 地理位置、时间
→ 个性化重排
```

**方法 4：LLM 重排（2024+）**
```
Prompt: "以下是用户搜索'XXX'后返回的10个文档，
请按相关性从高到低重新排序：
[doc1], [doc2], ..., [doc10]"
→ GPT-4o 输出排列
```
延迟高，适合离线评估或低 QPS 场景。

## L3 决策卡（Why）

### L3-01 · 为什么 BM25 在工业界仍被广泛使用？

**Q：神经检索（Dense Retrieval）这么强，BM25 为什么还没被取代？**

**A：**

**BM25 的不可替代优势**：

1. **零训练成本**：无需标注数据，新领域直接可用
2. **极低延迟**：倒排索引查询 <5ms，向量检索 ANN 通常 10-50ms
3. **精确匹配**：产品编号、专有名词、新词 → BM25 100%召回
4. **可解释**：每个词的贡献清晰可查，便于调试
5. **内存友好**：稀疏索引远比稠密向量库占用小

**Dense Retrieval 的短板**：
- 词汇不匹配问题（Vocabulary Mismatch）
- 新词/长尾词召回差
- 训练数据质量敏感
- ANN 有精度损失

**最优实践**：BM25 + Dense Retrieval 混合检索，取长补短

---

### L3-02 · 为什么 Listwise 比 Pointwise 更好？

**Q：Learning to Rank 中，为什么 Listwise 方法通常优于 Pointwise？**

**A：**

**Pointwise 的核心问题**：
- 每个文档独立训练，忽略文档间的相对关系
- 同一Query下，文档的相关性标注本质上是相对的
- 不同Query难度不同（Query归一化问题）

**Pairwise 的改进与局限**：
- ✅ 考虑文档对的相对顺序
- ❌ 忽略文档在列表中的位置（position-sensitive）
- ❌ 文档对数量是 O(n²)，训练开销大

**Listwise 的优势**：
- ✅ 直接优化列表级别的排序指标（NDCG）
- ✅ 考虑位置信息（LambdaRank中位置越靠前权重越大）
- ✅ LambdaMART 在工业LTR中长期占据SOTA

**实践结论**：
- 数据量少/冷启动 → Pointwise（简单有效）
- 对排序质量要求高 → LambdaMART（工业标准）
- 新范式 → BERT-based Listwise（精排阶段）

---

### L3-03 · 为什么搜索要两阶段而不是 End-to-End？

**Q：为什么不直接用一个强大的模型对全量文档排序（End-to-End）？**

**A：**

**核心约束：在线延迟 < 200ms**

```
暴力 Cross-Encoder 全量排序：
1000万文档 × 50ms/doc = 500,000秒 ❌ 不可行

两阶段方案：
召回 1000万 → 1000候选（ANN, <10ms）
精排 1000候选（Cross-Encoder, ~50ms）
总计 <100ms ✅
```

**为什么不用蒸馏把 Cross-Encoder 变快？**
- 蒸馏后精度会下降
- 精排模型仍需处理候选集，候选太大还是慢
- 两阶段允许不同阶段使用不同策略（规则/统计/神经网络）

**工程权衡**：
- 增加召回阶段数量（多路召回）提升覆盖率
- 在精排前加轻量粗排（GBDT）减少精排压力

---

### L3-04 · 为什么用 NDCG 而不是 Accuracy 评估搜索？

**Q：搜索评估为什么不直接用准确率（Accuracy）？**

**A：**

**Accuracy 的问题**：
1. **不考虑位置**：第1位和第10位相关文档贡献相同，但用户行为差异巨大
2. **二元相关性**：现实中文档有"完全相关/一般相关/不相关"多级别
3. **不考虑列表质量**：只看是否召回，不看排的好不好

**NDCG 的优势**：
1. **位置折扣**：log₂(rank+1) 使第1位权重远大于第10位
2. **多级相关性**：支持0-3多级相关性标注
3. **归一化**：NDCG∈[0,1]，跨Query可比较
4. **与用户行为相关**：NDCG 高的系统通常 CTR 也更高

---

### L3-05 · 为什么需要 Query 理解？

**Q：搜索引擎为什么要在检索前做 Query 理解？**

**A：**

**用户 Query 的特点**：
- 简短（平均2-3个词）
- 模糊（"苹果"=水果/手机/公司？）
- 口语化（"想找个好看的衣服"）
- 含噪音（错别字、缺词）

**Query 理解的价值**：

| 问题 | Query 理解解法 | 效果 |
|------|--------------|------|
| 歧义 | 意图识别 + 上下文消歧 | 精准返回用户真实意图 |
| 拼写错误 | Query 纠错（Edit Distance + LM） | 减少召回失败 |
| 同义词/OOV | Query 扩展 | 提升召回率 |
| 口语化 | Query 改写 | 规范化便于检索 |
| 实体识别 | NER | 匹配知识图谱、触发特殊展示 |

**没有 Query 理解的代价**：
- 用户搜"iPhone"，不能召回标注为"苹果手机"的商品
- 用户搜"苹果"，系统不知道返回水果还是手机
- 召回率和精准率双双下降

---

### L3-06 · 为什么需要重排（Re-ranking）？

**Q：精排模型已经很好了，为什么还需要重排阶段？**

**A：**

**精排的局限**：
1. **同质化**：精排倾向于返回最相关但内容相似的文档（如同一品牌多款手机）
2. **无业务意识**：纯相关性排序忽略多样性、新颖性、公平性
3. **单一目标**：精排优化相关性，但最终目标是用户满意度（多目标）

**重排解决的问题**：

| 问题 | 重排策略 |
|------|---------|
| 结果同质化 | MMR 多样性重排 |
| 商业目标（广告/推广） | 规则插入/商业位置 |
| 个性化 | 用户画像加权重排 |
| 时效性 | 新内容加分 |
| 公平性 | 品类/来源均衡 |

**设计哲学**：重排是精排结果的"润色"，在相关性基础上叠加业务策略。

## L4 场景卡（Scene）

### L4-01 · 场景：搜索召回率低如何排查？

**Q：线上发现搜索召回率下降，如何系统排查？**

**A：**

**排查思路（从上到下）**：

```
Step 1: 确认问题范围
├── 全量召回率下降 → 系统级问题
└── 特定类目/词下降 → 局部问题

Step 2: Query 理解层排查
├── 分词是否正确？（新词未识别？）
├── 意图识别是否误判？
└── Query 改写是否引入噪音？

Step 3: 索引层排查
├── 文档是否成功入库？（检查索引日志）
├── 新文档的更新延迟？
└── 分词策略与索引时是否一致？

Step 4: 检索层排查
├── BM25：停用词过多？k₁/b参数是否合适？
├── 向量检索：向量模型是否更新？ANN参数（ef_search）是否过低？
└── 多路召回策略：某路是否失效？

Step 5: 评估层排查
├── Recall@K 计算方式是否改变？
└── Ground Truth 标注是否更新？
```

**快速定位工具**：
- 采样 bad case → 人工分析 → 定位哪个模块失效
- A/B 实验对比 → 确认变更带来的影响

---

### L4-02 · 场景：用户搜"苹果"如何处理歧义？

**Q：用户搜索"苹果"，如何判断是水果、手机还是公司？**

**A：**

**多级消歧策略**：

**Level 1：上下文推断（用户历史）**
```
用户最近搜过"iPhone价格" → 大概率是手机
用户在生鲜平台 → 大概率是水果
```

**Level 2：Query 上下文**
```
"苹果 手机" → 手机
"苹果 好吃" → 水果  
"苹果 股票" → 公司
```

**Level 3：实体链接（Entity Linking）**
- NER 识别"苹果"为歧义实体
- 结合上下文向量，链接到知识图谱节点（Apple Inc. / 水果）

**Level 4：多意图混合召回**
```
置信度不够时：
- 展示多意图结果（前3位手机，插入水果，补充新闻）
- 搜索框提示"您是否想搜索：苹果手机 / 苹果水果"
```

**Level 5：用户反馈学习**
- 用户点击了水果 → 负样本标记手机类
- 在线学习更新意图权重

---

### L4-03 · 场景：新词/OOV 如何处理？

**Q：用户搜索的词在词表中不存在（OOV/新词），如何保证召回？**

**A：**

**OOV 问题的危害**：
- 分词把新词切散（"ChatGPT"→ "Chat" + "G" + "PT"）
- 向量模型对新词输出随机/低质量向量
- BM25 直接找不到匹配文档

**解决方案**：

**1. 动态词表更新**
- 实时监控新词（基于 PMI 点互信息 + 频次）
- 定期更新分词词典（jieba/HanLP 自定义词典）

**2. 字符级/子词模型（Subword）**
- BPE（Byte Pair Encoding）、WordPiece → 新词拆成已知子词
- BERT 使用 WordPiece，天然处理 OOV

**3. 字符级 Fallback**
- 新词用字符级 n-gram 索引
- "ChatGPT" → 索引 "Cha", "hat", "atG", "tGP", "GPT"

**4. 向量模型的 OOV 处理**
- 用子词向量平均作为新词向量
- 定期全量重训模型（捕获新词语义）

**5. Query 扩展 + 用户反馈**
- 监控新词的点击数据
- 将新词加入同义词表

---

### L4-04 · 场景：电商搜索"便宜手机"如何处理？

**Q：用户搜索包含价格意图的模糊词（"便宜"、"性价比"），搜索系统如何处理？**

**A：**

**挑战**：
- "便宜"不是可检索的词项
- 不同用户对"便宜"定义不同

**处理流程**：

**Step 1：意图识别**
- 识别"手机"为商品实体，"便宜"为价格属性限定
- 触发价格过滤逻辑

**Step 2：Query 改写**
```
"便宜手机" → "手机 price:range:[0, 1500]"（基于用户画像定义价格区间）
```

**Step 3：个性化价格阈值**
- 分析用户历史购买价格区间
- 大学生用户 → price < 2000
- 高端用户 → price < 8000

**Step 4：排序特征融入**
- 在精排特征中加入 price_score = 1 - (price / max_price)
- 性价比得分 = 评分 / 价格（归一化）

**Step 5：结果展示**
- 价格由低到高展示
- 添加"性价比推荐"标签

### L4-05 · 场景：长文档 vs 短文档搜索公平性

**Q：搜索系统中长文档和短文档如何公平竞争？**

**A：**

**问题**：
- 长文档词频绝对值高，BM25 倾向于排高（未归一化时）
- 短文档信息密度高，反而可能被压制

**BM25 的解决方案**：
- 参数 b 控制长度归一化（b=0.75 是工业默认值）
- `|d|/avgdl` 使得长文档的 TF 得分被压缩

**场景化策略**：

| 场景 | 长度偏好 | 策略 |
|------|---------|------|
| 问答检索 | 短文档（直接答案） | b 调大，强化长度归一化 |
| 论文/报告搜索 | 长文档（全面内容） | b 调小，弱化长度归一化 |
| 新闻搜索 | 均衡 | 默认 b=0.75 |

**密度归一化**：
```
score_normalized = BM25(q, d) / sqrt(|d|)
```
对短小精悍的文档更友好。

## L5 陷阱卡（Trap）

### L5-01 · 陷阱：稠密检索一定比稀疏检索好？

**Q：Dense Retrieval 效果一定比 BM25 好吗？**

**A：❌ 不一定！**

**Dense Retrieval 表现差的场景**：

1. **精确关键词匹配**：
   - 搜索"型号 XZ-2000"→ BM25 精确召回，Dense 可能返回"XZ-1000"
   - 搜索编号、ISBN、专有名词 → BM25 胜出

2. **领域数据稀缺**：
   - Dense 模型需要大量领域标注数据微调
   - 零样本时，DPR 在专业领域可能不如 BM25

3. **新词/长尾词**：
   - "元宇宙"、"AIGC" 等新概念，模型训练时未见过
   - BM25 直接索引，Dense 向量质量差

4. **极短文档**：
   - 文档只有几个词，Dense 向量信息不足
   - BM25 的词匹配更可靠

**结论**：
```
精确匹配场景 → BM25 可能更好
语义理解场景 → Dense 更好
生产最优 → BM25 + Dense 混合检索
```

---

### L5-02 · 陷阱：NDCG 和 AUC 的区别？

**Q：NDCG 和 AUC 都是排序指标，可以互相替代吗？**

**A：❌ 不能替代，适用场景完全不同！**

| 维度 | NDCG | AUC |
|------|------|-----|
| 适用任务 | 搜索排序（列表） | 二分类（CTR预估）|
| 相关性 | 多级（0/1/2/3） | 二元（点击/不点击）|
| 位置感知 | ✅ 排名越高权重越大 | ❌ 无位置概念 |
| 查询感知 | ✅ 按 Query 计算后平均 | ❌ 全局计算 |
| 含义 | 系统返回列表的质量 | 正负样本区分能力 |

**具体陷阱**：
- 一个 CTR 模型 AUC=0.8，但 NDCG 可能很低（因为它可能把所有高概率的同类内容堆在前面）
- 搜索模型 NDCG 高，不代表 AUC 高（因为搜索关注列表整体质量，不是点击概率精度）
- **搜索用 NDCG/MRR，推荐/广告点击率预估用 AUC**

---

### L5-03 · 陷阱：向量检索 ANN 一定比精确检索快？

**Q：HNSW 等 ANN 算法一定比暴力向量检索快吗？**

**A：❌ 小规模数据时不一定！**

**暴力搜索（Flat）vs HNSW**：

```
数据量 1000 条：
- Flat: 1000次内积 → <1ms
- HNSW: 图构建 + 查询 → 反而有额外开销

数据量 100万 条：
- Flat: 100万次内积 → ~100ms
- HNSW: 图跳跃搜索 → ~5ms ✅ HNSW 胜出
```

**陷阱场景**：
- 刚上线的系统，索引数据量小 → 用 Flat 更简单且快
- HNSW 的 ef_search 设太高 → 实际上接近暴力搜索，ANN 加速失效
- GPU 加速时，Flat（矩阵乘法）可能比 HNSW（图遍历）更快

**结论**：ANN 的收益在数据规模 > 10万 时才明显，小数据用 Flat 就行。

---

### L5-04 · 陷阱：LambdaRank 直接优化 NDCG？

**Q：LambdaRank 真的"优化 NDCG"吗？**

**A：❌ 严格来说，LambdaRank 不是直接优化 NDCG！**

**误解来源**：LambdaRank 用 |ΔNDCG| 缩放梯度，感觉像在优化 NDCG

**实际情况**：
- LambdaRank 没有显式的损失函数
- 梯度是"手工设计"的，不是从某个 NDCG 损失函数求导得来的
- 论文作者（Burges等）承认：梯度对应的隐式目标函数尚不明确

**为什么还有效**？
- 实验表明，用 |ΔNDCG| 缩放梯度确实使 NDCG 提升
- 属于"启发式但有效"的工程方案
- 后续工作（LambdaLoss，ICML 2019）才给出了严格的理论证明

**正确理解**：
```
LambdaRank ≈ "近似优化 NDCG 的 Pairwise 方法"
不是 = "NDCG 的梯度下降"
```

---

### L5-05 · 陷阱：召回率越高越好？

**Q：搜索系统是不是召回率（Recall）越高越好？**

**A：❌ 召回率需要与精确率（Precision）权衡！**

**高召回低精确的代价**：
- 返回大量不相关结果 → 用户需要更多时间筛选
- 精排、重排压力增大 → 系统延迟升高
- 用户体验下降（搜"苹果手机"召回出大量水果）

**召回率的正确定位**：
- 召回阶段的目标：**保证相关文档不被漏掉**（高 Recall@1000）
- 排序阶段的目标：**把最相关的放在前面**（高 NDCG@10）
- 两者是不同阶段的不同目标

**工程实践**：
```
召回率 @ 1000 → 99%+（几乎不漏相关文档）
NDCG @ 10    → 尽量高（前10位质量高）
```

**过度提升召回的陷阱**：
- 扩大 ANN ef_search → 延迟从 5ms 增加到 50ms，召回率仅提升 0.5%
- 不合理的 Query 扩展 → 召回率高但引入大量噪音，精排效果变差
