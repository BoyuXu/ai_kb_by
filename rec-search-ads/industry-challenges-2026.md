# 广告/推荐/LLM 行业挑战调研报告 2026

> 调研时间：2026-04-03 | 来源：arXiv 论文 + 工业界实践分析 + 公开技术分享
> 目的：指导 ai-kb 内容优先级，聚焦真实痛点而非教科书知识

---

## 一、推荐系统当前难题

### 难题 1：超长序列建模 vs 实时延迟约束

**现象**：用户历史行为动辄 10K+ 条（活跃用户在抖音/淘宝的年度行为），但线上 P99 延迟要求 < 10ms。

**根因**：Transformer 的 self-attention 是 $O(n^2 d)$，序列长度翻倍延迟增加 4 倍。

**当前解法**：
- **ETA（End-to-End Target Attention）**：Hash 技巧将全序列压缩为固定大小的 sketch，再做 attention，复杂度降至 $O(n)$
- **HSTU（Meta，2023）**：用 causal masking 的 Transformer 替换整个推荐 pipeline（召回到排序），trillion 参数但 batch 效率极高
- **SIM（Search-based Interest Model）**：从全序列中检索 Top-K 相关行为，只对这 K 条做精细建模

**仍未解决**：
- 真实时序动态（用户兴趣漂移）在序列中无法被位置编码捕捉（100天前 vs 昨天的权重问题）
- 冷启动用户只有 10 条行为，长序列模型完全退化
- 多模态行为混合（视频+搜索+购买）的统一序列表示

---

### 难题 2：离线评估与在线效果的不一致

**现象**：AUC 提升 0.1% 的模型，线上 CTR 可能没有变化甚至下降。"AUC++, online-"是行业公开的痛点。

**根因**：
- 离线训练数据有曝光偏差（只有被展示的商品才有标签）
- 测试集的时序泄露（用未来数据评估过去模型）
- 离线 AUC 主要衡量排序能力，但线上看的是绝对点击量（与头部流量分布有关）

**当前解法**：
- **反事实评估（OPE）**：用 Inverse Propensity Score 估计"如果换了策略会怎样"
- **Replay 评估**：用历史日志做仿真，但需要足够的覆盖度
- **Interleaving 实验**：两个策略交替展示给同一用户，比 A/B 更敏感但实现复杂

**仍未解决**：
- 没有普适的离线指标能替代 A/B 实验
- Interleaving 评估成本仍高，无法快速迭代
- 长期用户满意度（LTV、留存）的离线估计几乎不可能

---

### 难题 3：多目标优化的 Pareto 困境

**现象**：推荐系统需要同时优化 CTR、CVR、时长、多样性、新颖性、用户满意度……这些目标往往互相矛盾。

**根因**：
- CTR 优化 → 标题党/猎奇内容 → 用户疲劳 → 长期流失
- 多样性提升 → 短期 CTR 下降
- 业务方各自有 KPI，导致目标函数本身有政治性

**当前解法**：
- **MMOE/PLE**：多任务学习，各目标共享 + 独立，但需要人工设置 loss 权重
- **Pareto 前沿优化（PMOCO）**：学习 Pareto 前沿，推理时根据当前业务偏好选点
- **约束优化（Constrained Optimization）**：优化主目标，满足其他目标的约束条件

**仍未解决**：
- 权重如何设置仍靠人工调参，需要大量 A/B 实验才能找到"最优平衡"
- 长期目标（用户健康度、平台生态）无法量化进入 loss
- 不同用户群体的目标权重应该不同，但动态权重实现复杂

---

### 难题 4：LLM 推荐的延迟 vs 质量权衡

**现象**：GPT-4 级别的推荐理解能力极强，但单次推理 500ms-2s，而推荐系统要求 10ms 内返回。

**根因**：LLM 自回归生成是串行的，每个 token 生成一次大矩阵乘法，本质上是显存带宽瓶颈。

**当前解法**：
- **离线预计算**：对用户/物品的 LLM 特征离线跑批，在线查表
- **知识蒸馏**：用 GPT-4 标注数据训练小模型（7B→1B），小模型做在线推理
- **LLM 作为召回增强**：LLM 生成搜索 query 和 item 关键词扩展，召回层利用，排序层不用 LLM

**仍未解决**：
- 实时个性化（根据当前对话历史动态更新偏好）无法做到 LLM 级别质量
- 蒸馏后小模型的理解能力损失难以量化
- 多模态（图片+文字+视频）的实时 LLM 理解几乎不可能

---

### 难题 5：推荐系统的"回音壁效应"与干预

**现象**：协同过滤会不断强化用户已有偏好，导致信息茧房。平台知道这个问题但难以在不损害 CTR 的情况下解决。

**根因**：优化 CTR 本质上是优化"用户喜欢看的"而非"用户应该看的"，这是目标函数设计问题。

**当前解法**：
- **多样性约束（MMR/DPP）**：重排时强制引入多样性
- **Exploration-Exploitation**：$\epsilon$-greedy 或 Thompson Sampling 引入探索
- **因果推断去偏**：IPS 权重修正曝光偏差，给冷门内容更多机会

**仍未解决**：
- 多样性定义本身有争议（类型多样性 vs 观点多样性 vs 来源多样性）
- 强制多样性会降低短期指标，业务方不愿接受
- 用户自选择偏好（主动搜索特定内容）vs 被动推荐偏好的区别很难建模

---

## 二、广告系统当前难题

### 难题 1：隐私计算与广告精准度的矛盾

**现象**：Chrome 废弃第三方 Cookie（2024 实施）、iOS ATT 框架、GDPR 严格执法 → 跨站用户追踪失效，广告定向精准度下降 30-50%。

**根因**：
- 数字广告的精准性依赖跨平台用户 ID 拼接（用户在 A 网站买了某商品，在 B 网站看到相关广告）
- Cookie 废弃后，这个 ID 链接断裂

**当前解法**：
- **Google Topics API**：浏览器本地计算用户兴趣话题，只暴露泛化话题（不暴露具体 URL）
- **Federated Learning（联邦学习）**：数据不出本地，模型梯度聚合
- **Cohort-based Targeting**：对用户群体而非个体定向（Google FLoC 的理念）
- **First-party Data 回归**：广告主依赖自有数据（CRM + 官网行为）

**仍未解决**：
- Topics API 的粒度（300个话题）远不如 Cookie 精准（10000+ 维兴趣向量）
- 联邦学习通信开销大，梯度逆向工程可能泄露信息
- 小广告主没有 first-party data，精准度损失更严重

---

### 难题 2：First-Price Auction 生态稳定性

**现象**：2019-2022 年间广告交易市场从 Second-Price 切换到 First-Price Auction，但所有 DSP 都开始用 ML bid shading（压低出价），导致平台收入不稳定。

**根因**：在 FPA 中，出真实估值是"亏损策略"——需要学习 clearing price 分布并出略低于估值的价格。但所有竞标者都这么做 → 价格不断下压 → 平台收入下降 → 平台提高 reserve price → 竞标者调高出价 → 来回振荡。

**当前解法**：
- **Soft Floor（底价）**：平台设置动态 reserve price，防止价格崩溃
- **Auction Landscape Modeling**：竞标者用 ML 估计 clearing price 的概率分布，出 $E[v - \text{clearing price}]$ 最大化的价格
- **Google 的 Unified Pricing**：统一底价规则减少"拍卖博弈"

**仍未解决**：
- 博弈论意义上的均衡尚未被实践证明达到（所有 DSP 的 bid shading 相互博弈导致无稳定均衡）
- LLM-based 实时出价策略（如 RTBAgent）是否会加剧振荡还不清楚

---

### 难题 3：多触点归因的因果性

**现象**：用户看了 5 次广告后购买，到底哪次广告贡献最大？Last-click 归因给最后一次，First-click 归因给第一次，Data-Driven Attributtion（DDA）用 Shapley 值分配，但没有一种是"因果正确"的。

**根因**：
- 真正的因果问题需要反事实：如果没有第 3 次广告曝光，用户还会购买吗？
- 但反事实在现实中不可观测（你不能同时有和没有看到广告）

**当前解法**：
- **Geo-based Experiments**：用地理分割做"有广告区域 vs 没有广告区域"的准实验
- **Shapley Value Attribution**：所有子集的边际贡献平均，满足公理化性质
- **Uplift Modeling**：直接估计广告的增量转化（不是整体转化）

**仍未解决**：
- Geo-based 实验受外部因素干扰（节假日、竞品活动）
- Uplift 模型的 ground truth 本质上不可观测（双重反事实）
- 线下转化（用户看了广告，去线下店购买）几乎无法归因

---

### 难题 4：广告 LLM 创意生成的合规与质量

**现象**：LLM 生成广告标题/文案速度快、成本低，但容易产生：违规声明（"最好"、"第一"）、品牌调性不符、内容不实（幻觉）、文化敏感问题。

**根因**：
- LLM 优化的是语言流畅性，不是广告法合规性
- 品牌调性是隐性知识，难以完全注入 prompt
- 规模化生成（一次生成 10000 条创意）时人工审核不可行

**当前解法**：
- **Rule-based 后处理**：违禁词库 + 正则过滤（治标不治本）
- **RLHF 合规微调**：用合规/违规标注数据做 LoRA 微调
- **LLM-as-judge 自检**：生成后用另一个 LLM 检查合规性（成本翻倍）
- **CTR-conditioned Generation**：以历史高 CTR 创意为参照，引导生成风格

**仍未解决**：
- 多语言广告（日语、泰语、阿拉伯语）的合规性 LLM 能力不均
- 实时生成（用户画面出现时即时生成个性化文案）延迟无法满足
- 法律责任归属：AI 生成的违规广告谁负责？

---

### 难题 5：AutoBidding 的系统稳定性

**现象**：所有广告主都用 RL-based AutoBidding 后，系统可能陷入振荡——你出高了，我出低；你出低了，我出高。传统单个广告主优化没问题，但多 agent 同时 RL 时出现博弈不稳定。

**根因**：RL 的 stationary environment 假设被破坏——你的 agent 优化时，其他 agent 也在变化，环境非平稳。

**当前解法**：
- **平台侧约束（Mechanism Design）**：平台不是旁观者，可以通过拍卖机制设计引导均衡
- **Opponent Modeling**：对竞争对手的策略建模，纳入决策
- **Regret Minimization**：不追求 Nash 均衡，追求最小化后悔值（No-regret 算法）

**仍未解决**：
- 实际竞价场景有几百万广告主，Opponent Modeling 无法 scale
- RL 收敛到 Nash 均衡需要足够多轮对局，广告场景时间有限
- 平台利益 vs 广告主利益的对齐问题没有完美解

---

## 三、LLM / Agent 当前难题

### 难题 1：幻觉（Hallucination）的根治

**现象**：LLM 在知识密集型任务中自信地生成错误信息，且无法可靠地自知"我不知道"。

**根因**：
- 自回归训练目标（最大化下一 token 的概率）与"说真话"目标不对齐
- 训练数据中的错误信息被记忆
- 长 context 中的事实往往被"平均"处理，细节丢失

**当前解法**：
- **RAG（检索增强生成）**：用外部知识库提供实时事实，减少对参数记忆的依赖
- **Chain-of-Thought**：强制逐步推理，使错误更容易暴露
- **RLVR（可验证奖励 RL）**：对有标准答案的任务（数学、代码）用 RL 训练，显著减少该类幻觉
- **Self-RAG**：生成时主动判断是否需要检索，并反思检索结果的相关性

**仍未解决**：
- 主观任务（创意写作、法律建议）的幻觉无法用 RLVR 解决（没有标准答案）
- RAG 的检索准确率有上限，检索失败时 LLM 仍会幻觉
- 幻觉的"置信度校准"问题：模型说"我确定"时不可信

---

### 难题 2：长 Context 的真实理解能力

**现象**：GPT-4 支持 128K token 上下文，但实验表明在 context 中间部分的信息往往被"遗忘"（"Lost in the Middle"问题）。

**根因**：
- 注意力机制对远距离依赖的衰减（position encoding 的局限）
- 训练时 long context 样本少，模型没有充分学习如何利用长 context

**当前解法**：
- **ROPE（Rotary Position Embedding）+ 外推**：通过频率缩放支持超出训练长度的 context（LLaMA 3系列）
- **KV Cache 压缩**：选择性保留重要 token 的 KV，丢弃不重要的（StreamingLLM、SnapKV）
- **RAG 替代 Long Context**：用检索代替把所有文档塞进 context

**仍未解决**：
- 即使技术上支持 128K，模型对 context 中间信息的注意力仍然不足
- Long context prefill 的推理成本是 $O(n^2)$，128K context 一次推理需要数秒
- 多文档推理时的跨文档推断能力（需要综合 doc1 + doc5 的信息）不稳定

---

### 难题 3：Agent 的多步推理可靠性

**现象**：单步任务 LLM 成功率 95%，但 10 步任务成功率降至 $0.95^{10} \approx 60\%$，20步降至 36%。复杂 Agent 流程频繁失败。

**根因**：
- 错误会级联传播（step 3 出错 → step 4-10 全基于错误结果）
- Tool use 的错误处理不完善（API 失败、数据格式错误）
- LLM 没有真正的"记忆"，Context 太长时早期信息被遗忘

**当前解法**：
- **ReAct（Reasoning + Acting）**：每步先推理再行动，显式化中间步骤
- **Reflection / Self-Correction**：每步后自检，发现错误时回溯
- **Hierarchical Planning**：高层规划 + 低层执行分离，降低错误传播
- **Checkpoint + Retry**：关键步骤设置检查点，失败时从最近检查点重试

**仍未解决**：
- 自我纠错（Self-Correction）本质上用同一个模型检查自己，发现不了系统性错误
- 多 Agent 协作时，Agent 间通信协议尚未标准化（A2A、MCP 还在竞争）
- 长任务的 Context Window 管理：任务进行到一半 context 满了怎么办？

---

### 难题 4：推理成本与性能的权衡

**现象**：GPT-4 级别的推理能力已经很好，但 API 调用成本 $0.01-0.03/千 token，高频系统（推荐/搜索）完全无法负担。

**根因**：
- 参数量 = 推理速度瓶颈（显存带宽限制，每个 token 需要读取所有参数）
- 长 context + 大 batch = 显存溢出
- 云端 API 包含服务商利润，实际成本高于自部署

**当前解法**：
- **INT4 量化（AWQ/GPTQ）**：4-bit 量化将模型压缩 4×，推理速度提升 1.5-2×，精度损失 < 1%
- **投机解码（Speculative Decoding）**：小模型快速草稿，大模型验证，实测 3-4× 加速
- **MoE（混合专家）**：激活参数量远小于总参数（DeepSeek-V3: 671B 总参数，每次激活 37B）
- **蒸馏 + LoRA**：用 GPT-4 输出蒸馏出 7B/13B 小模型，成本降低 100×

**仍未解决**：
- 推理和训练成本的绝对值仍在快速下降（Moore's Law for LLM），但"够好够便宜"何时到来尚不确定
- 量化在数学推理等精密任务上有可感知的质量下降
- 边缘设备部署（手机端实时推断）的能耗问题

---

### 难题 5：RLVR vs RLHF — 后训练范式之争

**现象**：DeepSeek-R1 用 RLVR（可验证奖励）在数学/代码上超越了 RLHF 模型，但 RLHF 在主观任务（写作、对话）上仍更优。哪种后训练范式是未来？

**根因**：
- RLVR 依赖"可验证的真值"（数学答案对/错），但大多数真实任务没有明确真值
- RLHF 依赖人类偏好标注，成本高且主观性强

**当前解法**：
- **GRPO（Group Relative Policy Optimization）**：组内相对评分，比 PPO 更稳定高效
- **Process Reward Model（PRM）**：不只看最终答案，对中间推理步骤打分
- **Constitutional AI（Claude 的 CAI）**：用原则而非人工标注做 RLHF
- **DAPO（DeepSeek）**：去除 clip，允许更大策略偏移

**仍未解决**：
- 如何为主观任务设计"可验证奖励"（是否可以让 LLM 自动评分）
- RLVR 的"发散"问题：过度优化奖励会产生 reward hacking
- 后训练数据质量 vs 数量的 scaling law 尚不清晰

---

## 四、横向共性问题

### 共性 1：Evaluation（评估）是一切问题的基础

三个领域都有评估不可靠的问题：
- 推荐：AUC 与在线效果不一致
- 广告：归因模型不因果
- LLM：自动评估（LLM-as-judge）被 gaming

**根本矛盾**：真正的评估需要反事实（你换了策略效果如何），但反事实不可观测。

---

### 共性 2：数据偏差无处不在

| 偏差类型 | 推荐 | 广告 | LLM |
|---------|------|------|-----|
| 选择偏差 | 只有被曝光的物品有标签 | 只有被点击的广告有 CVR 标签 | 只有高质量文本进入训练集 |
| 确认偏差 | 推荐强化用户已有偏好 | 高点击广告获得更多展示 | 预训练偏向常见观点 |
| 时序偏差 | 近期行为权重高于历史 | 季节性导致数据分布漂移 | 训练数据截止日期问题 |

---

### 共性 3：LLM 引入后的"新旧系统融合"之痛

2024-2026 是传统模型（DNN/GBM）与 LLM 混用的过渡期，系统复杂度急剧上升：
- 两套推理基础设施（GPU 集群 + CPU 集群）
- 两套特征工程（数值特征 + 文本特征）
- 两种延迟量级（10ms + 500ms）
- 两种评估框架

**没有公司完成了彻底的 LLM 替换**，都是 patch 式集成。

---

### 共性 4：ScalingLaw 的边际效益递减

2023-2024 年"更大的模型、更多的数据"是规律。2025-2026 年：
- 推荐：Wukong 验证了 ScalingLaw，但 10× 参数量只带来有限增益，边际成本不合算
- 广告：CTR 模型 AUC 从 0.80 → 0.82，再往上推进极难
- LLM：GPT-4 → GPT-5 的飞跃，但在特定任务上已趋近天花板（MMLU 接近满分）

**下一步突破可能不在 Scale，而在架构创新和数据质量**。
