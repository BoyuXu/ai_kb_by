## 广告系统方向（10 个）

### 1. CTR 预估模型升级 —— ESMM 多任务学习

**业务背景**
广告点击率（CTR）预估是竞价广告系统的核心。然而，线上用户的行为数据存在样本选择偏差：只有被展示的广告才能被点击，系统看不到未展示广告的反馈。传统模型在这种截断数据上训练，导致 CTR 预估精度下降（AUC 仅 0.72）。

**痛点**
- 样本选择偏差（Selection Bias）：模型只学到"被展示的广告"的数据分布，无法泛化到未展示的广告
- 转化率（CVR）预估更差：因为转化数据更稀疏，偏差问题更严重

**算法创新：ESMM（Entire Space Multi-Task Model）**
ESMM 的核心思想是同时学习 CTR 和 CVR，利用 CTR × CVR ≈ pCTCVR（转化率）这个数学关系。pCTCVR 是全空间可观测的（无偏），用它作为约束来纠正 CTR 和 CVR 的偏差。

```
Loss = L(CTR) + L(CVR) + L(pCTCVR)
pCTCVR_pred = CTR_pred × CVR_pred
```

**效果指标**
- CTR AUC：0.72 → 0.78（+8.3%）
- 点击率提升：+12%（相同曝光下）
- 转化预估覆盖率：+20%（能预测更多延迟转化）

**技术栈**
- TensorFlow / PyTorch（多任务学习框架）
- 特征工程：DIN（Deep Interest Network）处理用户行为序列
- 样本采样：负采样优化训练速度

**核心洞察**
样本偏差比特征工程的收益更大。加再多特征也无法克服选择偏差；只有改变损失函数的定义（ESMM），才能真正解决问题。

### 2. 自动竞价系统（AutoBidding）—— 对偶优化

**业务背景**
广告平台的广告主需要设置出价策略以优化 ROI。传统方式是广告主手工调整出价，效率低、反馈慢。智能竞价系统需要在给定 ROI 目标（如 target_CPA = 50 元）的约束下，自动优化出价，最大化转化量。

**痛点**
- 广告主无法精准判断市场出价变化，导致出价过高或过低
- CPA 不稳定：有时达到目标 CPA，有时超支
- 人工调整周期长，市场反应慢

**算法创新：拉格朗日对偶优化**
问题形式化为约束优化：最大化转化量，约束条件为 CPA ≤ target_CPA。使用拉格朗日对偶法求解，将约束问题转化为无约束问题，通过交替优化乘数 λ（Lagrange multiplier）和出价参数。

```
L(bid, λ) = conversions(bid) - λ × (CPA(bid) - target_CPA)
```

**效果指标**
- CPA 达标率：65% → 92%（接近 100% 可行）
- 日均转化量：+18%（在达标 CPA 约束下）
- 平台收入：+7%（广告主转化增加 → 点击增加 → CPM 收入增加）
- 收敛速度：<24 小时达到新的平衡点

**技术栈**
- 优化算法：梯度下降 + 对偶上升（Dual Ascent）
- 实时竞价数据：PPC（Pay-Per-Click）日志
- 模型：CTR/CVR 预估模型 + CPA 预测模型

**核心洞察**
将营销问题从"回归预测"重新定义为"约束优化"，变换问题的本质（参数化），比堆积特征工程效果更显著。

### 3. Feed 混排多样性 —— DPP（Determinantal Point Process）

**业务背景**
信息流（Feed）中既有来自广告主的付费广告，也有推荐内容。直接用点击率排序会导致同类广告/内容堆积，用户疲劳感强。需要在保持相关性的前提下，增加列表的多样性。

**痛点**
- 贪心排序（按 CTR 降序）导致前几位都是类似的内容（如都是手机广告）
- 用户疲劳：重复看到相似内容，点击率和停留时长快速下降
- 多样性与相关性往往是矛盾的

**算法创新：DPP（行列式点过程）**
DPP 用一个矩阵的行列式来定义"多样性"。选中的项目子集越"不相似"，行列式越大。通过贪心采样算法快速找到多样性最大的列表。

```
P(S) ∝ det(L_S)  // L_S 是相似度矩阵 L 的子矩阵
贪心：每次选择让 det 增长最快的项目
```

**效果指标**
- 用户留存（7 日留存）：+8%
- 多样性指标（Entropy）：+75%（内容种类更均匀分布）
- RPM（每千次展示收入）：+5%（虽然单个广告 CTR 略降，但广告量增加）
- 计算延迟：<100ms（贪心算法）

**技术栈**
- 相似度矩阵：用户兴趣向量 + 内容编码向量（CLIP/BERT）
- 贪心采样：O(N²) 复杂度（N = 候选数量）
- 工程：Triton（推理引擎）实时计算

**核心洞察**
局部最优（逐项最大化）≠ 全局最优（列表级优化）。列表级优化（列表的多样性）需要显式建模项目间的相互作用。

### 4. 广告主预算控制（Pacing）—— PID 控制器

**业务背景**
广告主为一次营销活动预设日预算（如每天 1000 元）。如果系统不加控制，高峰期会快速耗尽预算，导致后期无法投放。需要平滑控制出价乘数，使预算均匀消耗。

**痛点**
- 高峰期快速耗尽：上午 10-12 点是黄金时段，出价高，快速花完一天的预算
- 欠消耗（Underspend）：后期预算剩余，无法利用
- 频繁调整出价导致成本抖动

**算法创新：PID 控制器**
借鉴控制论的 PID（Proportional-Integral-Derivative）算法。实时监控预算消耗速率，与理想消耗速率对比，调节出价乘数。

```
error_t = ideal_pace_t - actual_pace_t  // 理想vs实际消耗速率差
multiplier_t = K_p × error_t + K_i × ∑error + K_d × (error_t - error_{t-1})
```

**效果指标**
- 预算消耗曲线平滑度：+90%（方差减少 90%）
- 欠消耗率：<2%（几乎完全消耗预算）
- 均匀消耗达成率：85%（每个小时消耗都接近理想值）
- CPA 稳定性：提升 35%（出价调整更平缓）

**技术栈**
- 实时控制：每 5 分钟计算一次消耗速率，调整出价乘数
- 数据源：实时竞价日志（毫秒级）
- 参数调优：历史数据学习最优 K_p, K_i, K_d

**核心洞察**
工程问题（消耗控制）往往比算法问题的商业价值更大。一个好的工程方案能直接减少客户投诉和欠消耗损失。

### 5. 展示频次控制 —— 贝叶斯优化

**业务背景**
同一用户在一段时间内看到同一广告的次数叫"频次"。频次太高（如一天看 5 次）导致用户反感、点击率下降；频次太低则无法留下印象。需要为每个用户找到最优频次曲线。

**痛点**
- 固定频次上限（如 3 次/天）不够灵活，无法应对不同用户的需求
- 用户反感率高：40% 的用户反感过高频次，导致屏蔽或卸载
- 频次与转化率的关系非线性，难以直接优化

**算法创新：贝叶斯优化**
用贝叶斯优化（Bayesian Optimization）探索每个用户群体的最优频次。先用高斯过程（Gaussian Process）建模"频次→转化率"的关系，然后通过采集函数（Acquisition Function）选择下一个要尝试的频次值。

```
f(frequency) ~ GP(μ, σ)  // 不确定的函数
选择使 EI(frequency) 最大的 frequency  // Expected Improvement
```

**效果指标**
- 用户反感率：-40%（从 40% → 24%）
- ROI：+15%（同样的展示预算，转化率提升）
- 最优频次被正确识别的概率：78%（vs 随机 25%）
- 计算成本：低（GP 预测快）

**技术栈**
- 贝叶斯优化库：Botorch / Optuna
- 分层探索：不同用户群（年龄/性别/地域）分别优化
- 反馈循环：每天收集 1000+ 用户的频次-转化数据，更新 GP

**核心洞察**
最优参数（频次）不是固定的，需要动态、个性化调整。贝叶斯优化通过主动探索和反馈，找到高效的参数空间。

### 6. 广告创意优化 —— 多臂老虎机

**业务背景**
一个广告主可能有 100+ 个创意（不同的文案、图片组合）。手工选版（A/B 测试）效率低，无法快速发现高转化的创意。需要自动化机制来探索和利用好创意。

**痛点**
- 创意众多：手工测试每个创意需要时间
- 冷启动问题：新创意没有数据，排序靠后
- 创意性能差异大：最好的创意可能比平均创意 CTR 高 3 倍

**算法创新：Thompson Sampling（多臂老虎机）**
将"选择创意"建模为多臂老虎机问题。每个创意是一个"臂"，点击数是"奖励"。Thompson Sampling 通过贝叶斯更新，自动平衡"探索"（尝试不确定的创意）和"利用"（推荐确定高转化的创意）。

```
θ_i ~ Beta(α_i, β_i)  // 第 i 个创意的 CTR 分布
选择使 E[θ_i] 最大的创意 i
看到点击 → α_i++；看到未点击 → β_i++
```

**效果指标**
- 最佳创意被推荐的概率：95%（vs 随机 1/100）
- 探索效率：新创意在 500 次展示内就能判断优劣
- 创意替换频率：每 2 天自动替换一次（vs 人工手动）
- 整体 CTR 提升：+8%（持续用更好的创意）

**技术栈**
- Thompson Sampling 框架：Vowpal Wabbit / Contextual Bandit
- 分层探索：按用户属性分层，每个群体单独探索
- 黑名单机制：表现极差的创意快速下线

**核心洞察**
平衡探索和利用是多臂问题的核心。Thompson Sampling 用概率方式自动平衡，无需手工调参。

### 7. 跨域归因 —— Shapley 值多触点模型

**业务背景**
用户的转化路径往往涉及多个触点：可能先看到 Facebook 广告（未转化），再看到 Google 搜索广告（点击但未转化），最后通过直接访问完成购买。如何公平分摊这三个触点的"功劳"？

**痛点**
- 单点归因不准：只给最后一次点击计功劳，忽视了之前的品牌建设
- 广告主预算分配不当：高估了某个渠道的转化效果
- 跨域数据整合困难：Facebook、Google、自有站点的数据来自不同源

**算法创新：Shapley 值多触点归因**
借用博弈论的 Shapley 值，公平分配转化的"功劳"。每个接触点的归因值 = 它加入路径前后的期望收益差。

```
φ_i = (1/n!) × ∑ [v(S ∪ {i}) - v(S)]  // S 是接触点的子集
```

**效果指标**
- 归因准确性：与因果推断方法（IPW）的误差 <5%
- 预算重分配效率：+25%（识别高效渠道后的预算分配优化）
- 广告主满意度：+30%（能看到每个渠道的真实贡献）
- 计算时间：<1 秒/用户（优化后）

**技术栈**
- Shapley 值计算：SHAP 库 / TreeSHAP（快速版本）
- 跨域数据链接：用户 ID 映射（First-Party Cookie）
- 特征：用户接触时间序列、广告主维度、渠道维度

**核心洞察**
因果推断在营销中是必需的，不只是学术问题。Shapley 值提供了一种数学上公正的分配方案。

### 8. 低频转化优化 —— 生成对抗网络（GAN）

**业务背景**
某些产品的转化频率极低：房产、汽车、保险等。一个用户可能一年才转化一次。导致转化数据极其稀缺，无法训练高质量的转化预估模型。

**痛点**
- 数据稀疏：转化样本只占所有点击的 0.5%
- 模型过拟合：用稀少的正样本训练模型，容易过拟合
- 迁移学习困难：不同产品的用户特征差异大

**算法创新：GAN 生成虚拟数据**
用 GAN 生成虚拟的"用户→广告→转化"交互数据。生成器学习真实数据的分布，生成逼真的虚拟数据；判别器区分虚拟和真实数据。用虚拟数据扩大训练集，提升模型泛化能力。

```
min_G max_D E_real[log D(x)] + E_fake[log(1-D(G(z)))]
生成器：z -> fake user-ad-conversion tuple
判别器：判断是真实还是虚拟数据
```

**效果指标**
- 转化预估 RMSE：-30%（虚拟数据扩充后）
- 转化预测覆盖率：+50%（能预测更多样的用户-产品对）
- 模型稳定性：+40%（过拟合减少）
- 虚拟数据占比：30%（每 1 份真实数据生成 0.43 份虚拟）

**技术栈**
- GAN 框架：WGAN（Wasserstein GAN，训练更稳定）
- 网络结构：MLP for tabular data（不是图像）
- 评估：Inception Score 衡量虚拟数据质量

**核心洞察**
生成数据在数据稀疏的场景（低频转化、长尾产品）有巨大价值。关键是确保生成数据的"真实性"。

### 9. 实时竞价（RTB）优化 —— 强化学习（DQN）

**业务背景**
实时竞价（Real-Time Bidding）发生在毫秒级：用户访问页面 → 广告平台收到竞价请求 → 100ms 内决定出多少价格来赢这次竞价。需要在实时条件下做最优决策。

**痛点**
- 决策速度要求极高：<50ms 需要给出决策（包括网络延迟）
- 不确定性大：不知道对手会出什么价格
- 目标多样：既要赢高价值的竞价，又要控制成本

**算法创新：DQN（Deep Q-Network）**
用强化学习学习最优竞价策略。状态：用户特征、广告特征、当前时间等；动作：出价金额（离散化）；奖励：赢率和成本的综合评分。DQN 学一个深度网络 Q(s, a) 来估计每个动作的长期价值。

```
Q(s, a) ≈ 赢率 × 预期价值 - 出价金额
DQN loss = (r + γ max_a' Q(s', a') - Q(s, a))²
```

**效果指标**
- 竞价赢率：+15%（相同预算下）
- 单次竞价成本：-8%（更智能的价格决策）
- 端到端延迟：<50ms（包括推理）
- ROI 提升：+12%（赢了更多高价值竞价）

**技术栈**
- RL 框架：TensorFlow / PyTorch + OpenAI Gym
- 网络结构：CNN/MLP（快速推理）
- 离线训练：用历史竞价日志训练，定期上线新策略

**核心洞察**
强化学习处理"实时序列决策"比传统优化更自然。DQN 自动学会权衡赢率和成本。

### 10. 品牌安全检测 —— 文本分类 + 主动学习

**业务背景**
广告可能出现在不适合的内容旁边，损伤品牌形象：如奢侈品广告出现在暴力新闻旁。需要实时检测内容的安全性，自动过滤不安全页面。

**痛点**
- 内容多样化：日均新增 100 万+ 页面，手工审核不可行
- 标注成本高：需要专家标注"安全"vs"不安全"
- 分类边界模糊：有些内容处于灰色地带（如政治讨论）

**算法创新：BERT 分类 + 主动学习**
用 BERT 预训练模型做文本分类，识别不安全内容。同时用"主动学习"选择最有信息量的样本给人工标注，而不是随机采样，从而用更少的标注成本达到更高的精度。

```
BERT_output = [CLS] token embedding
logits = Linear(BERT_output)  // 分类层
不确定性采样：选择 max(p_safe, p_unsafe) 最接近 0.5 的样本
```

**效果指标**
- 安全内容覆盖率：98%（只有 2% 的安全内容被误判）
- 不安全内容检出率：94%（只有 6% 的不安全内容漏检）
- 标注成本：-60%（主动学习减少标注量）
- 模型迭代周期：从月度 → 每周

**技术栈**
- 模型：BERT（预训练）→ 微调
- 采样策略：不确定性采样 + 多样性采样
- 数据更新：每周汇总新标注数据，重新微调

**核心洞察**
主动学习在标注数据有限的场景下效果显著。聪明地选择标注样本，比随机标注节省 60% 的成本。

### 11. 冷启动用户留存 —— 图神经网络（GNN）

**业务背景**
新注册用户没有历史行为数据，传统推荐算法无法提取特征，只能推荐热门内容。导致冷启动用户的 7 日留存仅 28%，远低于活跃用户的 65%。

**痛点**
- 新用户无历史行为：协同过滤无法工作
- 推荐都是热门内容：用户看不到个性化内容，留存差
- 冷启动是推荐系统的通用难题

**算法创新：GraphSAGE（图神经网络）**
将用户、内容、社交关系建图：用户 A-聊天-用户 B（朋友）、用户 B-点赞-内容 C。用 GraphSAGE 在这个图上传播信息：新用户的特征 = 聚合朋友的特征 + 朋友点赞的内容特征。

```
h_u = AGGREGATE({h_friend | friend ∈ N(u)})
h_u = h_u ⊕ h_u_profile  // ⊕ 是拼接
推荐 = 新用户向量 · 内容向量
```

**效果指标**
- 7 日留存：28% → 35%（+25%）
- 冷启动用户的推荐多样性：+40%（不再只是热门）
- 点击率：+18%（个性化程度提升）
- 图构建成本：低（社交关系已有，内容交互实时更新）

**技术栈**
- GNN 框架：PyTorch Geometric / DGL
- 图数据：用户-用户（社交）、用户-内容（交互）
- 聚合函数：Mean / LSTM / Attention
- 推理：GPU 推理，<100ms 延迟

**核心洞察**
图结构比用户特征本身更有信息量。新用户虽然无自己的数据，但他的朋友的数据可以代理他的兴趣。

### 12. 冷启动内容曝光 —— 探索利用（Exploration-Exploitation）

**业务背景**
新发布的内容没有反馈数据，排序算法无法给它分配流量，导致新内容曝光少、反馈少、排序靠后——一个死循环。需要主动分配探索流量给新内容。

**痛点**
- 冷启动死循环：无数据 → 无法排序 → 无曝光 → 无数据
- 发现好内容能力弱：即使发布了优质新内容，也难以被发现
- 流量浪费：如果探索分配不当，会浪费大量流量在垃圾内容上

**算法创新：Thompson Sampling（多臂老虎机）**
同样的道理应用到内容冷启动：每个新内容是一个"臂"，点击数是"奖励"。Thompson Sampling 自动分配探索流量，同时利用已验证的高质量内容。

```
新内容的 CTR ~ Beta(α, β)，初始 α=β=1（均匀分布）
每天：采样一个新内容作为"探索对象"，分配 5% 的流量
收到反馈后，更新 Beta 参数
```

**效果指标**
- 新内容首周曝光：3x（vs 传统排序）
- 发现好内容的能力：+50%（更多优质新内容被识别）
- 流量浪费率：<8%（大部分探索流量用在最终有转化的内容上）
- 探索期后的排序精度：无损（不再探索的内容排序准确）

**技术栈**
- 多臂框架：Vowpal Wabbit / Contextual Bandit
- 特征提取：新内容的冷特征（标签、分类、作者等）
- 实时更新：每小时更新 Beta 参数

**核心洞察**
适当的"浪费"（探索）能避免更大的浪费（长期无法发现好内容）。这是一个探索-利用的权衡问题。

### 13. 转化率预估（CVR）—— 样本加权

**业务背景**
推荐系统不仅要优化点击率（CTR），还要优化后续转化（如购买、点赞等）。但转化往往有延迟：用户今天点击，可能 3 天后才转化。这导致模型无法学到"延迟转化"的信号。

**痛点**
- 延迟偏差（Delayed Feedback Bias）：模型的训练数据中，很多点击后来转化了，但在训练时被标记为"未转化"
- 转化预估准确率低：AUC 仅 0.58
- 无法预测长期价值

**算法创新：样本加权（Weight 纠正延迟）**
对训练样本进行加权，考虑观测窗口的长度。如果观测窗口短（如只看 1 天），那些还没转化的点击，实际上可能最终会转化。给这些样本一个权重补偿。

```
weight_i = 1 / (1 - click_to_conversion_rate)  // 如果平均 20% 点击最终转化，权重 = 1.25
loss_i = weight_i × cross_entropy(y_true, y_pred)
```

**效果指标**
- CVR 预估准确率（AUC）：0.58 → 0.78（+34%）
- 转化预测覆盖率：+20%（能预测更多延迟转化）
- 模型重新训练周期：加速 30%（收敛更快）
- 排序效果提升：+15%（优先推荐最终会转化的内容）

**技术栈**
- 样本构造：收集完整观测窗口的数据（如 30 天）
- 权重计算：基于历史转化延迟分布估计权重
- 模型：任何判别模型（LR / GBDT / DNN）都可以用加权版本

**核心洞察**
延迟是转化预估的隐藏杀手。显式处理延迟偏差，比增加特征的收益更大。

### 14. 个性化排序 —— 多目标排序（MOR）

**业务背景**
推荐系统有多个目标：用户看点击率、平台看停留时长、内容创作者看互动数。单一排序模型无法同时优化所有目标。不同用户的偏好也不同：有人喜欢快速翻看（优化点击数），有人喜欢深度沉浸（优化停留时长）。

**痛点**
- 单一目标优化导致侧重不当：过度优化点击率会导致内容肤浅
- 用户满意度不均：快速翻看的用户满意，深度阅读的用户不满
- 目标冲突：高点击内容往往停留时长短（如标题党）

**算法创新：多目标学习 + 帕累托前沿**
不追求单一最优解，而是学习帕累托前沿（Pareto Frontier）：多个目标的权衡集合。对每个用户，根据他的偏好选择帕累托前沿上的一个点。

```
目标 1：最大化点击率
目标 2：最大化停留时长
帕累托前沿：无法同时提高两个目标的点的集合

用户偏好参数 w ∈ [0,1]
score = w × CTR_pred + (1-w) × duration_pred
```

**效果指标**
- 用户满意度（综合评分）：+8%
- 内容消费时长：+12%（不只是点击数）
- 创作者收入：+10%（互动数增加）
- 模型复杂度：+30%（多目标需要多个任务分支）

**技术栈**
- 多任务学习框架：MMoE（Multi-gate Mixture of Experts）
- 目标：CTR、停留时长、互动、分享等
- 用户偏好：从历史行为推断，或从显式反馈获取

**核心洞察**
单一指标优化往往是幸存者偏差。多目标学习才是真实的用户价值。

### 15. 序列建模 —— Transformer 推荐

**业务背景**
用户的行为是序列：今天看体育，明天看电影，后天又看体育。这个序列有长期依赖：某个体育事件（如世界杯）可能影响后续 30 天的行为。传统的 CNN/RNN 难以捕捉这种长期依赖。

**痛点**
- RNN 梯度消失：长期依赖在 RNN 中会导致梯度指数衰减
- CNN 无法处理可变长序列
- 难以理解用户的"意图演变"：为什么用户从看体育突然改看电影

**算法创新：Transformer（Self-Attention）**
Transformer 用 Self-Attention 直接连接序列中远距离的位置，学习长期依赖。每个位置的表示是其他所有位置的加权平均，权重由注意力机制学习。

```
Attention(Q, K, V) = softmax(QK^T / √d_k)V
Q = W_q × x_i  // 当前位置的查询向量
K = W_k × x_j  // 其他位置的键向量（j = 1..T）
V = W_v × x_j  // 其他位置的值向量
```

**效果指标**
- 点击率（CTR）：+18%（相比 RNN）
- 用户留存（30 日）：+6%
- 长期依赖捕捉能力：显著提升（能识别用户兴趣周期）
- 模型参数量：减少 40%（相比 LSTM）

**技术栈**
- Transformer 架构：BERT4Rec（推荐领域的改进版）
- 位置编码：Sinusoidal 或 Learnable
- 优化：Adam + 学习率预热

**核心洞察**
序列模型比 CNN/RNN 更容易优化，且能学到更复杂的依赖关系。Self-Attention 是理解序列结构的更好方式。

### 16. 多样性与相关性平衡 —— 再排序（Re-ranking）

**业务背景**
推荐系统的排序模型通常优化相关性（用户是否点击）。但如果推荐都是相关性最高的内容，列表会显得单调，用户看腻。需要在保持相关性的前提下，增加列表的多样性。

**痛点**
- 相关性最高的内容往往很相似：都是同一类别、同一风格
- 用户疲劳：重复看到相似的推荐
- 多样性和相关性往往矛盾

**算法创新：两层排序（粗排 + 精排）**
第一层用模型排序（优化相关性），第二层用规则/算法再排序（增加多样性）：
- 粗排：用排序模型产生 Top-100
- 精排：用多样性算法（如 MMR、DPP）从 Top-100 中选择 Top-10，确保多样性

```
精排算法：贪心选择
S = []
for i in range(10):
    j = argmax_j [λ × relevance(j) - (1-λ) × similarity(j, S)]
    S.append(j)
```

**效果指标**
- 相关性：-2%（可接受的权衡）
- 用户留存：+7%（多样性带来的收益）
- 列表多样性：+45%（内容种类更均衡）
- 列表覆盖率：+60%（推荐了更多不同的内容）

**技术栈**
- 粗排模型：任何排序模型（LR/GBDT/DNN）
- 精排算法：MMR（Maximum Marginal Relevance）或 DPP
- 参数 λ：权衡相关性和多样性（通常 0.4-0.6）

**核心洞察**
列表级优化 > 逐项优化。考虑列表中项目间的相互作用，才能得到最优的用户体验。

### 17. 内容理解增强 —— 多模态学习

**业务背景**
推荐系统通常只用用户的显式行为（点击、收藏）来理解内容。但内容本身很丰富：图片的视觉特征、文字的语义、音频的情感。如果主动提取这些特征，能更好地理解内容，改进冷启动。

**痛点**
- 仅用隐式信号（点击）无法准确理解内容本质
- 新内容冷启动困难：无点击反馈，无法提取特征
- 内容的"真实"特征被隐藏

**算法创新：CLIP（Contrastive Learning 对比学习）**
CLIP 是 OpenAI 的多模态模型：用对比学习在大规模图文数据上训练，学到图像和文本的对齐向量表示。对任何新内容，直接用 CLIP 提取特征，无需等待反馈。

```
图像编码：I_enc = ImageEncoder(image)
文本编码：T_enc = TextEncoder(text)
对比损失：max(sim(I_enc, T_enc)) - min(sim(I_enc, T_wrong))
```

**效果指标**
- 冷启动内容的推荐精度：+22%（主动特征提取）
- 新颖内容曝光：+35%（CLIP 特征能识别新类别）
- 特征覆盖率：98%（几乎所有内容都能有特征）
- 计算成本：低（CLIP 模型轻量化后可实时推理）

**技术栈**
- 多模态模型：CLIP / ALIGN / LiT
- 特征提取：在线推理或离线计算
- 推荐器：用 CLIP 特征 + 用户向量进行匹配

**核心洞察**
主动特征提取 > 被动反馈。不依赖用户反馈，直接从内容本身理解它，冷启动问题得到根本解决。

### 18. 用户分群与个性化 —— K-means 聚类 + 分群模型

**业务背景**
推荐系统用一个全局排序模型服务所有用户。但不同用户的偏好差异大：有人看新闻，有人看视频，有人看电商。用一个模型无法满足所有人。

**痛点**
- 千人一面：所有用户看到类似的推荐
- 模型无法学到"用户间差异"：全局模型的参数对所有用户相同
- 个性化推荐效果有上限

**算法创新：聚类 + 分群模型**
先用 K-means 聚类，将用户分成 K 个群（如 5 群）。然后为每个群单独训练一个排序模型。推荐时，先识别用户所在群，再用该群的模型排序。

```
用户特征：年龄、性别、地域、历史行为
聚类：K-means(用户特征, K=5)
对每个群：训练独立的排序模型
```

**效果指标**
- 群 1（新闻爱好者）的满意度：+5%
- 群 2（视频爱好者）的满意度：+7%
- 群 3（电商爱好者）的满意度：+8%
- 总体满意度提升：+6%（不同群体都受益）

**技术栈**
- 聚类算法：K-means / KMeans++
- 群数量选择：用轮廓系数（Silhouette Coefficient）评估
- 模型训练：为每个群单独训练排序模型（LR/GBDT/DNN）

**核心洞察**
复杂度权衡：N 个群 × 1 个模型 vs 1 个超大模型。前者可解释性更好，维护成本更低。

### 19. 点击反馈延迟处理 —— Inverse Probability Weighting（IPW）

**业务背景**
用户点击后需要打开页面（延迟 2-5 秒），如果我们立即用这次点击更新模型，那么还没等用户看到内容，页面就已经加载好了，反馈信息不完整。需要纠正这个延迟偏差，加快模型更新频率。

**痛点**
- 反馈延迟：无法实时更新模型，只能批量处理
- 模型更新周期长：每天一次，无法快速适应用户兴趣变化
- 冬眠效应：热点话题火了 1 天就冷了，模型却还在推荐过时内容

**算法创新：IPW（Inverse Probability Weighting）**
用因果推断的 IPW 方法，纠正延迟偏差。核心思想：给那些"反馈更新更快"的样本更高的权重，来补偿那些"反馈仍在更新"的样本。

```
反馈延迟分布：p(延迟 > t | 特征)
权重 = 1 / p(延迟已观测 | 特征)
带权损失：loss_i = weight_i × cross_entropy(y_true, y_pred)
```

**效果指标**
- 模型更新频率：1 天 → 1 小时（加速 24 倍）
- 排序精度：+4%（更及时的反馈）
- 热点话题的推荐延迟：6 小时 → 1 小时
- 计算成本：低（权重计算简单）

**技术栈**
- 反馈数据：收集完整的延迟分布
- 权重估计：用逻辑回归估计反馈延迟概率
- 在线学习框架：支持加权样本的增量学习

**核心洞察**
因果推断处理时间延迟很有效。显式建模延迟，而不是假装不存在。

### 20. 内容生命周期优化 —— 指数衰减模型

**业务背景**
不同内容的生命周期不同：热点新闻前 1 天爆火，然后迅速冷却；长篇内容可能保持热度一周；学习资料的热度平稳。需要根据内容类型调整推荐权重。

**痛点**
- 热点内容快速衰落后还在推荐：浪费了用户时间
- 长尾内容曝光不足：一直被热点压制
- 生命周期被忽视：排序模型无法学到时间效应

**算法创新：指数衰减模型**
用指数衰减函数建模内容热度随时间的变化：`热度(t) = 初始热度 × exp(-λ × t)`。其中 λ 是衰减速率，可根据内容类型学习。

```
score_final = score_model × exp(-λ_c × (now - publish_time))
λ_c：第 c 类内容的衰减系数（从数据学习）
```

**效果指标**
- 热点内容峰值曝光：+25%（前 1-2 天集中推荐）
- 长尾内容覆盖率：+15%（热点冷却后，长尾获得更多曝光）
- 内容消费"新鲜度"：+40%（用户看更多新发布的内容）
- 用户满意度：+5%（推荐内容时间敏感性提高）

**技术栈**
- 模型：在排序分数上应用指数衰减
- 参数 λ：每个内容类别单独学习（从历史数据回归）
- 实时计算：简单，<1ms

**核心洞察**
内容生命周期是时间的函数，需要显式建模。忽视时间维度，推荐系统会不断重推陈旧内容。

### 21. 搜索相关性排序 —— Learning-to-Rank（LTR）

**业务背景**
搜索系统需要对候选文档排序。BM25（基于关键词频率的启发式排序）是经典方法，但效果有限（NDCG@10 仅 0.65）。需要学习一个排序函数，优化用户搜索满意度。

**痛点**
- BM25 的排序效果上限：基于词频统计，无法理解语义
- 搜索用户满意度低：前 10 结果中，只有 6.5 个是用户想要的
- 无法利用用户反馈（点击、停留时长）

**算法创新：LambdaMART（Learning-to-Rank）**
LambdaMART 是 Pairwise Learning-to-Rank 方法：优化排序对（i 比 j 更相关）的相对顺序，而不是预测绝对的相关性分数。损失函数是 NDCG 的导数（Lambda），直接优化排序指标。

```
Pairwise 损失：max(0, 1 - s_i + s_j) + ∆NDCG(i, j)
s_i, s_j：文档 i, j 的排序分数
∆NDCG：交换 i, j 的 NDCG 变化
```

**效果指标**
- NDCG@10：0.65 → 0.78（+20%）
- 用户搜索满意度：+12%（更多相关结果在前面）
- Precision@1：+25%（第一个结果更可能是用户想要的）
- 模型排序延迟：<50ms

**技术栈**
- LTR 框架：LightGBM / XGBoost + rank loss
- 特征工程：查询-文档相关性（BM25、TF-IDF）、文档质量、点击热度等
- 用户反馈收集：隐式反馈（点击）、显式反馈（评分）

**核心洞察**
Listwise 损失 > Pairwise > Pointwise。直接优化排序指标（NDCG），比预测中间量（相关性分数）效果更好。

### 22. 搜索查询理解 —— 意图分类

**业务背景**
查询"苹果"可能指水果、苹果手机或苹果公司。同一查询的意图不同，应该返回不同的搜索结果。需要自动理解查询意图，然后根据意图调整搜索结果。

**痛点**
- 查询歧义性高：10% 的查询有多重意图
- 简单的关键词匹配无法区分：搜"苹果"返回的既有水果又有手机
- 用户满意度低：无法找到自己想要的信息

**算法创新：BERT 意图分类 + 多信号融合**
用 BERT 做意图分类，同时融合多个信号（用户历史搜索、时间、位置、设备）来消消歧。

```
输入：[CLS] 查询 [SEP] 用户历史搜索 [SEP]
输出：意图分类（4 个类别：水果、手机、公司、其他）
```

**效果指标**
- 意图分类准确率：94%（vs 简单基线 60%）
- 搜索结果满意度：+18%（更符合用户意图）
- 用户转化率：+22%（用户更快找到想要的）
- 模型推理延迟：<50ms

**技术栈**
- 模型：BERT + 微调层
- 训练数据：手工标注的查询-意图对，或从用户点击行为推断
- 特征：查询、用户历史、时间、地理位置、设备类型

**核心洞察**
查询歧义性很高，需要多信号融合。仅靠查询文本本身无法判断用户意图。

### 23. 语义搜索 —— 向量检索（Dense Retrieval）

**业务背景**
关键词匹配的局限：查询"便宜的手机"和文档"低价 iPhone"在词汇上无重叠，但语义相关。需要用语义相似性检索，而不仅是词汇匹配。

**痛点**
- 关键词匹配的召回率有上限：很多语义相关的文档被遗漏
- 短查询的检索效果差：查询很短，关键词信号不足
- 无法理解查询的深层意思

**算法创新：DPR（Dense Passage Retrieval）双塔架构**
DPR 训练两个编码器（BERT）：一个编码查询，一个编码文档。两者都映射到同一个向量空间，用向量相似性（如余弦相似度）检索。

```
q_vec = QueryEncoder(query)  // 768 维向量
doc_vec = DocumentEncoder(doc)  // 768 维向量
相似度 = cos(q_vec, doc_vec)
检索：找 Top-K 最相似的文档
```

**效果指标**
- 召回率@100：+35%（vs BM25）
- 搜索精度（MRR@10）：+8%
- 检索延迟：<100ms（向量相似性计算快）
- 索引大小：✓ 可接受（每文档一个 768 维向量）

**技术栈**
- 编码器：BERT 或其他预训练 LLM
- 向量数据库：Faiss / Milvus / Weaviate（快速近似最近邻搜索）
- 训练：对比学习（Contrastive Learning），正文档相似度高，负文档相似度低

**核心洞察**
语义相似性 > 关键词匹配。向量表示能捕捉深层的语义关系。

### 24. 实时搜索 —— 流式索引更新

**业务背景**
新内容发布后，搜索不到，因为索引还没更新。传统的全量索引更新周期是 30 分钟或 1 小时，导致新内容不可搜索。对于新闻、突发事件等，这个延迟是无法接受的。

**痛点**
- 索引延迟：新内容发布 30 分钟后才能被搜索到
- 用户体验差：错过了新闻的热点时间窗口
- 新内容曝光不足

**算法创新：Kafka + Flink 流式索引更新**
用消息队列（Kafka）收集新内容的发布事件，用流处理引擎（Flink）实时更新 Elasticsearch 索引。降低索引延迟到 <5 分钟。

```
内容发布系统 → Kafka Topic: new-content
Flink 消费 → 实时索引 → Elasticsearch
延迟：内容发布 → Elasticsearch 可搜索 < 5 分钟
```

**效果指标**
- 新内容可搜索延迟：30min → 5min（6 倍加速）
- 新闻搜索体验评分：+40%
- 实时搜索的点击率：+25%（用户能及时搜到新闻）
- 系统吞吐量：支持 10000+ 内容/分钟

**技术栈**
- 消息队列：Kafka（高吞吐、可靠）
- 流处理：Flink（低延迟、有状态处理）
- 搜索引擎：Elasticsearch（支持实时索引更新）

**核心洞察**
实时性往往是用户体验的隐形指标。对于时效性强的内容，搜索延迟直接影响体验。

### 25. 搜索排序个性化 —— 用户特征融合

**业务背景**
搜索排序是通用的：同一查询返回给所有用户相同的结果。但不同用户的需求不同：查询"iPhone"，有人要买手机（电商需求），有人要查参数（资讯需求）。需要根据用户背景个性化排序。

**痛点**
- 排序无法适应用户的个性偏好
- 同一查询不同用户看到的结果相同，满意度不同
- 无法利用用户的上下文信息（位置、历史搜索等）

**算法创新：用户特征融合**
在排序模型中融合用户特征：用户历史搜索、点击模式、位置、设备等。让模型学到"对这类用户，这个结果更相关"。

```
排序分数 = score(query, doc) + user_bias(user, query, doc)
user_bias = f(用户历史搜索, 用户点击模式, 用户位置, ...)
```

**效果指标**
- 点击率：+15%（更符合用户个性需求）
- 用户多样性需求满足：+22%（不同用户看到不同结果）
- 用户搜索满意度：+10%
- 计算成本：+20%（多了用户特征）

**技术栈**
- 排序模型：LTR（Learning-to-Rank）框架
- 用户特征：历史搜索序列、点击热度、地理位置、设备
- 特征存储：Redis（低延迟访问用户特征）

**核心洞察**
个性化排序的收益比通用排序更大。用户背景信息能显著改进搜索体验。

## NLP & LLM 方向（3 个）

### 26. 自动广告创意生成 —— 条件文本生成

**业务背景**
写高转化率的广告文案是广告主的瓶颈。一个活动需要数百个创意，手工写很低效。同时，什么样的文案转化率高不好预测，需要通过 A/B 测试。

**痛点**
- 创意生成成本高：手工写每个创意需要时间
- 高转化创意稀缺：即使写了很多，大部分转化率平庸
- 快速迭代困难：测试新创意的周期长

**算法创新：GPT-2 微调生成**
用 GPT-2 在高转化创意的语料上微调，学到"好创意"的文风。给定商品、用户群等条件，生成新的创意文案。

```
输入：[商品: iPhone] [用户群: 学生] [风格: 幽默]
输出：生成创意文案（如 "学生党福音：iPhone 分期 0 利息，一年 12 期，月付比咖啡便宜！"）
微调损失：cross_entropy(预测文案, 真实文案)
```

**效果指标**
- 生成文案的点击率：接近人工水平（误差 <10%）
- 创意生成成本：-70%（自动化替代人工初稿）
- 创意多样性：+40%（生成的创意风格更多样）
- 每次活动的创意数量：100 → 500+（可以快速生成更多候选）

**技术栈**
- 模型：GPT-2 或更大的 LLM（GPT-3.5）
- 训练数据：历史高转化创意
- 微调：LoRA（Low-Rank Adapter）加速微调
- 推理：实时生成，<2 秒生成一个创意

**核心洞察**
生成模型在创意方向能替代人工初稿。虽然质量不如人工，但成本低 70%，迭代快，对整体效果有利。

### 27. 评论情感分类 —— 预训练模型迁移学习

**业务背景**
平台需要快速判断用户评论的情感（正/负/中性），用于产品反馈、质量控制等。传统的手工规则（关键词匹配）效果差，无法理解讽刺、复杂表达等。

**痛点**
- 标注成本高：从零开始标注 10000 条样本，成本大
- 规则无法覆盖：讽刺、俚语、方言等难以用规则判断
- 模型泛化能力差：特定领域的评论用通用模型准确率低

**算法创新：BERT 预训练 + 微调**
BERT 在大规模无标签文本上预训练，学到了语言的通用表示。只需用少量标注数据（<1000）微调最后的分类层，就能达到 92% 的准确率。

```
BERT 预训练：在 100G 文本上自监督学习
微调：用 500 条标注评论，微调分类层
微调损失：cross_entropy(情感预测, 真实标签)
```

**效果指标**
- 准确率：92%（vs 规则基线 65%）
- 标注成本：<1000 条（vs 从零开始 10000 条）
- 成本节省：90%（预训练节省了 9000 条标注）
- 模型推理延迟：<50ms

**技术栈**
- 预训练模型：BERT / RoBERTa / ALBERT
- 微调框架：HuggingFace Transformers
- 标注工具：众包（Amazon MTurk）或内部团队

**核心洞察**
预训练模型能大幅降低标注需求。迁移学习的威力在于，用通用知识节省特定任务的成本。

### 28. 对话式推荐 —— 多轮对话理解

**业务背景**
传统推荐系统是"一问一答"：用户输入查询 → 系统返回结果。但用户的需求往往复杂、需要多次沟通才能表达清楚。对话式推荐允许多轮交互，逐步理解用户需求。

**痛点**
- 一次搜索无法满足：用户的需求模糊，需要多次筛选
- 用户步数多：要点 5 次才能找到想要的
- 用户感受差：不如人工客服那样高效

**算法创新：对话状态追踪 + 意图识别**
系统维护一个"对话状态"：已确定的用户需求（如"预算 1000 元"、"需要快递"）。每轮对话，用 NLU（自然语言理解）提取用户的新约束，更新状态，然后推荐符合条件的商品。

```
对话状态 = {预算: 1000-2000, 颜色: 黑色, 品牌: Apple}
用户说："我要黑色的"
意图识别 → 提取约束 {颜色: 黑色}
更新状态 → 推荐符合条件的商品
```

**效果指标**
- 用户搜索步数：5 → 2（大幅减少交互次数）
- 转化率：+20%（对话式推荐更易达成购买）
- 用户满意度：+15%
- 人工客服工作量：-40%（对话系统替代了部分工作）

**技术栈**
- NLU：意图识别 + 槽位提取（用 BERT 或规则）
- 对话管理：追踪对话状态，生成下一步问题
- 推荐引擎：基于当前状态推荐商品
- 对话系统框架：Rasa / Botfront

**核心洞察**
交互式推荐能提升用户体验和转化率。多轮对话比一次性搜索更符合用户的决策过程。

## 创新方向（2 个）

### 29. 因果推断在营销中的应用 —— 倾向评分匹配（PSM）

**业务背景**
广告平台想评估某个优化（如推荐算法升级）的效果。A/B 测试是黄金标准，但成本高（需要一周、百万用户）。能否用观测数据（看过但未参加实验的用户）也做因果推断？

**痛点**
- A/B 测试成本高：每次 1 周 × 百万用户 × 机会成本
- 无法频繁测试：新算法多，无法一一测试
- 观测数据被浪费：有很多历史数据，但没用上

**算法创新：倾向评分匹配（PSM，Propensity Score Matching）**
从观测数据中找"相似的用户对"（一个接触了新算法，一个没有），通过匹配这样的对，估计处理效应（Treatment Effect）。核心是估计用户被分配到"新算法"组的倾向（Propensity Score），用它作为平衡条件。

```
倾向评分：P(treatment=1 | 用户特征) ∈ [0,1]
匹配：找倾向评分相近的用户对（treated vs control）
处理效应估计：ATE = E[Y_1 - Y_0 | matched pairs]
```

**效果指标**
- 评估精准度：接近 A/B 测试（bias <5%）
- 实验成本：-80%（不需要实验周期，用历史数据）
- 评估周期：从 1 周 → 1 天（实时评估）
- 可以评估更多优化

**技术栈**
- 因果推断库：DoWhy / CausalML
- 倾向评分估计：逻辑回归 / GBDT
- 匹配算法：1:1 最近邻匹配、核匹配（Kernel Matching）

**核心洞察**
观测数据也能做因果推断。通过精心的倾向评分匹配，在没有 RCT 的情况下，也能估计因果效应。

### 30. 图神经网络在推荐中的应用 —— 知识图谱补全

**业务背景**
推荐系统的特征稀疏：新用户没有历史数据，新内容没有反馈。传统方法无法为新实体生成特征，导致冷启动推荐效果差。

**痛点**
- 冷启动精度低：新用户/新内容无法精准推荐
- 特征缺失：新实体缺少特征向量
- 无法利用实体间的关系

**算法创新：GNN 知识图谱补全**
构建一个知识图（Knowledge Graph）：节点是用户/内容/属性/类别，边是关系（用户-点赞-内容、内容-属于-类别等）。用 GNN（如 GraphSAGE）在图上聚合邻近节点的信息，补全缺失的节点特征。

```
节点特征传播：
h_user = AGGREGATE({h_friend, h_liked_content})
h_new_content = AGGREGATE({h_creator, h_category, h_similar_content})

新用户的推荐 = 新用户向量 · 内容向量
```

**效果指标**
- 冷启动推荐精度：+25%（补全的特征很有用）
- 模型泛化能力：+15%（图结构提供了归纳偏好）
- 长尾用户/内容的推荐改进最明显（+35%）
- 图计算延迟：<100ms（可实时推理）

**技术栈**
- 知识图谱：用户-内容-属性关系，构成异构图
- GNN 框架：PyTorch Geometric / DGL
- 聚合函数：Mean / LSTM / Attention over neighbors
- 图数据库：Neo4j / ArangoDB（存储和查询）

**核心洞察**
显式图结构能大幅改进冷启动。新实体虽然无自己的数据，但他们的邻近节点的数据可以代理它。
