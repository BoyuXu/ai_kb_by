## 二、数据流设计

### 2.1 离线流水线

```
【T-1 日离线流水线】（每天凌晨1~5点运行）

1. 商品向量化（凌晨1:00~2:00，约60分钟）
   商品数据库（MySQL）
   → 获取昨天新增/更新的商品（约1万条）
   → BGE-M3 批量向量化（批大小256，GPU加速）
   → 生成768维向量

2. CTR权重计算（凌晨2:00~2:30）
   点击日志（Kafka → ClickHouse）
   → 汇总每条创意的曝光和点击（过去7天滑动窗口）
   → 贝叶斯平滑（Beta先验参数每周更新）
   → 计算加权向量：v_weighted = v * sqrt(ctr_smooth / mean_ctr)

3. Milvus索引更新（凌晨2:30~3:30）
   → 增量插入新向量（平均约1万条/天，< 1分钟）
   → 每周全量重建（周日凌晨3:00~5:00，约2小时）

4. CTR模型（XGBoost/LightGBM）每周重训（周六凌晨）
   → 使用过去30天标注数据
   → 训练时间约30分钟（10GB数据）
   → 蓝绿部署，新模型通过离线评估后才切换
```

### 2.2 在线流水线

```python
# 在线请求处理（目标P99 < 100ms）
async def process_creative_request(request: CreativeRequest) -> CreativeResponse:
    
    # 【阶段1: 检索层】并行化处理（目标 < 30ms）
    async with asyncio.TaskGroup() as tg:
        # 1a. 向量化
        embed_task = tg.create_task(embed_query(request))
        # 1b. 获取CTR缓存（Redis，命中率~70%）
        ctr_task = tg.create_task(get_cached_k(request.category))
    
    query_embedding = await embed_task
    cached_k = await ctr_task
    
    # 1c. Milvus粗检索
    raw_results = await milvus_client.search(query_embedding, top_k=100)
    
    # 1d. 动态K决策（使用缓存K或实时计算）
    optimal_k = cached_k or compute_dynamic_k(raw_results)
    
    # 1e. BGE-Reranker精排
    top_k_creatives = await reranker.rerank(
        request.query, raw_results[:50], optimal_k
    )
    
    # 【阶段2: 生成层】异步处理（目标 < 8s，用户异步等待）
    # 检查是否有预生成的CoT中间步骤（缓存24小时）
    cot_context = await get_cached_cot_context(request.product_id)
    
    if not cot_context:
        # 没有缓存：全流程CoT（耗时3~8s）
        cot_context = await llm_client.cot_generate(
            request, top_k_creatives
        )
    else:
        # 有缓存：只跑Step4（文案生成，耗时1~2s）
        pass
    
    raw_creatives = await llm_client.generate_creatives(
        cot_context, top_k_creatives,
        n_samples=5, temperature=1.1
    )
    
    # 【阶段3: 过滤层】后处理（目标 < 20ms）
    # 多样性检查 + MMR选取 + 合规过滤
    final_creatives = await postprocess_creatives(
        raw_creatives, request
    )
    
    return CreativeResponse(creatives=final_creatives)
```

## 三、核心技术选型对比

| 组件 | 候选方案 | 最终选择 | 选择理由 |
|------|---------|---------|---------|
| **向量数据库** | Milvus / Faiss / Pinecone / Weaviate | **Milvus 2.3** | 支持实时增量更新、标量+向量联合查询（`WHERE category=XX`）、开源自部署（数据隐私）、成熟的分布式方案 |
| **Embedding模型** | BGE-M3 / text-embedding-3 / bge-large-zh | **BGE-M3** | 多语言支持、768维平衡效果/内存、本地部署低延迟（3ms） |
| **重排器** | BGE-Reranker / Cohere Rerank / MonoT5 | **BGE-Reranker-v2** | 本地部署无API费用、中文效果优秀、兼容ONNX加速 |
| **LLM** | GPT-4o / Claude-3.5-Sonnet / Qwen-72B | **主：Claude-3.5-Sonnet；降级：Qwen-72B** | Claude质量最优；Qwen-72B本地降级避免API依赖 |
| **消息队列** | Kafka / RabbitMQ / RocketMQ | **Kafka** | 高吞吐（点击日志场景），已有基础设施 |
| **时序监控** | ClickHouse / TimescaleDB / InfluxDB | **ClickHouse** | 与现有广告数据栈统一，列式存储聚合查询极快 |
| **缓存** | Redis / Memcached | **Redis** | 支持TTL、Hash、Sorted Set（K值缓存、CoT缓存） |
| **归因计算** | SHAP / LIME / 自研 | **TreeSHAP（树模型）+ DeepSHAP（DNN）** | TreeSHAP精确高效；按模型类型选择最优方法 |

## 四、资源估算（精确版）

### 4.1 存储资源

```
向量索引规模：
  - 创意总量：100万条
  - 向量维度：768（float32 = 4字节）
  - 向量存储：100万 × 768 × 4 ≈ 3GB
  - HNSW索引开销（约4x向量大小）：12GB
  - 元数据（文案文本、CTR等）：约5GB
  - 总内存需求：约20GB → 配置2台32GB内存服务器（主备）
```

## 六、系统级面试问答（10个高频问题）

**Q1：如何做灰度上线？**

A：五步灰度策略。(1) **流量分桶**：按广告主ID的哈希值分桶，1%→5%→20%→50%→100%，每个阶段观察24~48小时；(2) **分层灰度**：先对低风险场景（长尾品类、小预算广告主）灰度，再扩大到核心场景；(3) **核心指标监控**：CTR（主）、审核通过率、P99延迟、系统报错率，任何指标下降超过阈值（如CTR -3%）自动回滚；(4) **对照组保持**：灰度期间保留10%流量给旧系统，用于持续对比；(5) **功能开关**：所有关键组件（动态K、CoT、MMR）独立开关，可单独回滚某个模块而不影响整体系统。

**Q2：如何处理LLM API宕机？**

A：三级降级策略。第一级（API延迟高但未宕机）：启用请求超时（3s），超时后用本地Qwen-72B处理，用户无感知；第二级（Claude API完全宕机）：切换到本地Qwen-72B全量处理（提前在本地部署，专为降级备用），可能有轻微质量下降，但服务不中断；第三级（本地GPU也崩溃）：返回预生成的"备用文案"——从数据库捞取该品类历史高CTR创意前5条直接返回，零延迟但无个性化。同时触发PagerDuty告警，运维介入。关键：永远不要返回空结果，降级链路必须完整测试。

**Q3：如何评估整个系统的ROI？**

A：ROI = (收益增量 - 成本) / 成本。收益来源：(1) CTR提升→更多点击→更多广告收入（广告平台视角）；(2) 广告主成本节省（人工写稿2小时→AI 1分钟，每条节省 ¥500）；(3) 广告主留存率提升（好工具降低流失）。成本：GPU服务器（约¥50万/月）+ API费用 + 研发维护。量化方法：大规模A/B测试，观察6周，主指标CTR，次指标广告主日活跃度。例：若系统使平台广告日收入从1亿提升到1.08亿（+8%），而系统成本为300万/月，则月收益约2400万，ROI = (2400-300)/300 = 700%。

**Q4：如果CTR不升反降，如何排查？**

A：分层排查，避免过早下结论。第一步（数据验证）：确认A/B分组是否正确（是否有组间污染），检查实验组和对照组的流量分布是否均衡（品类分布、广告主规模）；第二步（模块定位）：逐模块比较——检索层（实验组检索结果的语义相关性是否下降？）、生成层（生成文案的CTR预估分是否低于对照组？）、过滤层（MMR是否过度牺牲CTR换多样性？）；第三步（失败归因）：对实验组的低CTR创意跑SHAP，找出主要负贡献特征；第四步（假设检验）：根据归因结果，单独关闭某个模块（如临时禁用动态K，回退到固定K=10），观察是否恢复。

**Q5：如何确保生成内容合规？**

A：四层合规防线。(1) **Prompt约束层**：System Prompt注入平台规范（字数、禁用词、不允许夸大词汇）和示例，让LLM自我约束；(2) **规则过滤层**：生成后用AC自动机扫描违禁词库（平台提供，约5万词），匹配到则重写或屏蔽，延迟 < 1ms；(3) **合规API层**：调用平台官方审核API（部分平台提供pre-check接口），获取结构化违规原因，并重新生成对应段落；(4) **人工审核兜底**：对于敏感品类（医疗保健、金融）自动加入人工审核队列，人工确认后才交付。同时维护合规通过率指标，低于95%时触发告警，提示更新违禁词库。

**Q6：系统如何扩展到10倍体量？**

A：每个层的扩展策略不同。**检索层（瓶颈：Milvus）**：Milvus天然支持分片（sharding），将100万创意按品类哈希分片到10个节点，查询时路由到对应节点，线性扩展；**生成层（瓶颈：LLM）**：(1) 批处理合并——同品类的并发请求合并一次LLM调用；(2) 增加Qwen-72B实例（GPU水平扩展）；(3) 扩大预生成缓存覆盖率；**监控层（瓶颈：ClickHouse写入）**：ClickHouse集群增加副本，开启异步写入，批量插入（每10秒一批）。整体目标：单机处理1000 QPS，10倍体量用10节点水平扩展，成本线性增长，无架构重写。

**Q7：如何处理多语言广告？**

A：多语言支持需在三个层面改造。(1) **检索层**：将BGE-M3替换为或配合多语言模型（如multilingual-e5-large），它支持100+语言，跨语言相似度有效；或建立分语言索引，en/zh/ja各自独立；(2) **生成层**：在CoT的System Prompt中注入目标语言约束（"请用日语输出"），Claude-3.5-Sonnet多语言能力强；(3) **合规层**：每个语言市场的违禁词库单独维护，不能共用。额外挑战：不同文化的文案风格差异巨大（日本广告偏含蓄，美国广告偏直接），Few-shot示例必须本地化，不能直接翻译中文示例。

**Q8：如何做持续的模型迭代？**

A：建立持续学习（Continual Learning）的完整闭环。(1) **数据飞轮**：广告上线后的真实CTR数据每日回流，补充到训练集（使用时间权重，近期数据权重更高）；(2) **CTR模型迭代**：每周自动触发模型重训任务，离线评估指标（AUC、NDCG）提升则自动发布；(3) **RAG索引迭代**：高CTR创意自动进入"精选池"，在检索时给予额外权重（定期人工审核精选池质量）；(4) **Prompt迭代**：通过归因分析发现某类失败模式后，A/B测试改进的Prompt版本，效果好则合并；(5) **评估标准迭代**：每季度和业务方对齐核心指标，避免"优化了错误的目标"。

**Q9：与竞品系统相比优劣势？**

A：竞品系统分为两类：(1) **直接Prompt LLM**（如直接用ChatGPT生成）——优势：我们系统有历史高效创意的RAG支撑，生成更有"实战依据"；CTR加权确保参考质量；多样性监控避免广告疲劳。(2) **传统模板系统**——优势：我们的CoT能理解模糊诉求，不依赖预设模板，覆盖更多场景；缺点：延迟比模板系统高（3s vs 50ms），需要异步化补偿。核心差异化：我们是**可解释的效果驱动系统**——不仅生成创意，还能告诉广告主为什么这条文案好/为什么那条会失败（SHAP归因），这是所有竞品系统都没有的能力。

**Q10：如何说服老板投入资源做这个项目？**

A：用数据说话，分三个论点。(1) **痛点量化**：调研显示广告主每条人工创意平均耗时2.5小时，成本约¥300；我们的系统将成本压缩到¥0.5（API费用），**节省99.8%的制作成本**，ROI极高。(2) **效果验证**：先做小规模POC（Proof of Concept），选5个愿意配合的广告主，用动态K+CTR加权检索做内部测试（最低成本：无需LLM，只需RAG），2周内展示CTR数据对比；如果RAG阶段就能提升5%，则完整系统的ROI论证成立。(3) **竞争视角**：竞品平台已开始推AI创意功能（字节、阿里均有布局），若不跟进，2年内可能丧失高质量广告主；先发优势在于数据飞轮——越早做，积累的高CTR创意库越大，竞争护城河越深。关键：用3页PPT，而非20页，第一页讲ROI数字，第二页讲竞争格局，第三页讲最小可行路径。
