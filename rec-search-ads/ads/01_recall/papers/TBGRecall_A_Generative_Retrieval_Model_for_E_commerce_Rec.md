# TBGRecall: A Generative Retrieval Model for E-commerce Recommendation Scenarios
> 来源：https://arxiv.org/abs/2508.11977 | 领域：ads | 日期：20260323

## 问题定义
电商推荐中，传统向量召回难以处理语义复杂性（如"妈妈的生日礼物"这类抽象需求）。TBGRecall提出基于tokenized beam-search的生成式召回模型，专门针对电商推荐场景优化。

## 核心方法与创新点
- 层级物品编码：基于商品类目树构建层级token，beam search按层级剪枝
- 语义感知tokenization：融合商品文本描述的语义信息到token设计
- 用户意图感知生成：根据上下文（浏览历史、搜索词）条件化生成
- 高效beam search：类目树剪枝使搜索空间从O(V)降至O(log V)

## 实验结论
在淘宝/天猫电商数据集，TBGRecall的Recall@200比DIN等传统召回提升约8%；beam search利用类目树剪枝，速度比全量搜索快约50x；长尾类目召回提升最显著（约15%）。

## 工程落地要点
- 商品类目树需要定期维护，新类目的引入需要更新token结构
- 生成式召回和向量召回可以互补，构成多路召回体系
- 电商场景需要考虑库存（缺货商品不应被召回）和地区限制

## 常见考点
1. **Q: 电商推荐的核心与普通推荐的差异？** A: 商品价格/库存约束、购买周期性（不会反复买同款）、品类相关性复杂
2. **Q: 类目树在生成式召回中的作用？** A: 提供层级约束，减少搜索空间，同类目商品共享前缀token
3. **Q: 生成式召回如何处理商品实时更新（上下架）？** A: 定期重建token索引、实时过滤下架商品、在线更新策略
4. **Q: 多路召回如何融合？** A: 归一化分数后加权融合、学习式融合（Learn-to-rank）、规则合并去重
5. **Q: 如何处理电商推荐中的"探索与利用"平衡？** A: ε-greedy、UCB、汤普森采样，对新商品增加探索流量

## 模型架构详解

### 索引构建
- **离线阶段**：将全量候选 Item 编码为向量/Token序列，构建 ANN 索引
- **编码器选择**：双塔（用户塔+物品塔）/ 生成式（自回归生成Item ID）

### 检索策略
- **向量检索**：FAISS / ScaNN / HNSW 近似最近邻
- **生成式检索**：Beam Search 生成 Top-K Item ID Token序列
- **混合检索**：向量召回 + 倒排召回 + 生成式召回的多路融合

### 训练方法
- 对比学习：正样本（点击/购买）vs 负样本（随机/难负例）
- In-batch Negatives：同 batch 内其他样本作为负例
- 课程学习：从简单负例到困难负例逐步提升难度

## 面试深度追问

- **Q: 双塔模型的负采样策略有哪些？**
  A: 1) 随机负采样；2) In-batch Negatives（同 batch 内互为负例）；3) 难负例挖掘（前一轮模型的 Top-K 非正例）；4) 混合策略（简单+困难按比例混合）。

- **Q: 生成式召回相比向量召回的优劣？**
  A: 优势：无需构建 ANN 索引、天然支持多模态、生成过程可解释；劣势：推理延迟较高（自回归逐步生成）、训练复杂度高、不易增量更新。

- **Q: 如何评估召回模型的效果？**
  A: 离线：Recall@K、NDCG@K、Hit Rate；在线：召回对下游排序的贡献（End-to-End GMV/CTR 提升）。注意 Recall@K 的 K 选择要和线上候选集大小一致。

- **Q: 冷启动物品如何召回？**
  A: 1) Content-based：利用商品文本/图片特征计算相似度；2) 跨域迁移：从有行为的域迁移 Embedding；3) 探索流量：分配小比例流量给新物品收集反馈。


## 工业界应用案例

- **YouTube**：深度双塔模型用于视频候选召回
- **淘宝**：TDM（Tree-based Deep Model）树索引召回
- **Spotify**：利用双塔 + 对比学习实现音乐推荐召回

### 关键指标参考
- 召回率（Recall@1000）：通常 > 80% 覆盖后续排序的正样本
- 索引构建时间：百万级别 < 10 分钟，亿级 < 2 小时
- 在线检索延迟：< 10ms（含 ANN 检索 + 后过滤）

### 部署注意事项
1. 索引更新频率：至少日级更新，热门物品小时级更新
2. 多路召回融合：不同召回通道的结果需要统一打分后截断
3. 过滤逻辑：召回后需过滤已曝光/不可用/违规物品
