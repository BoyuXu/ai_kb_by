# DHEN: A Deep and Hierarchical Ensemble Network for Large-Scale CTR Prediction
> 来源：https://arxiv.org/abs/2203.11014 | 领域：ads | 学习日期：20260329

## 问题定义

CTR 预估中的特征交互学习是核心挑战。现有问题：

1. **单一交互模块局限性**：不同设计的特征交互模块（FM、DCN、xDeepFM等）各有优势，但单独使用时不够全面
2. **交互信息非重叠性**：实验发现不同交互设计捕获的特征交互信息存在互补性，即使它们声称捕获相同阶数的交互
3. **深层次交互建模不足**：现有方法对高阶特征交互的建模仍不充分
4. **训练效率问题**：更深的网络结构带来训练效率下降问题

## 核心方法与创新点

### DHEN 架构设计

**双层创新：模型架构 + 训练系统协同设计**

```
输入层: Sparse Feature Embeddings
    ↓
DHEN Hierarchical Ensemble Layer:
  ├── 交互模块1: FM-like 二阶交互
  ├── 交互模块2: DCN-like 显式高阶交互
  ├── 交互模块3: DIN-like 注意力交互
  ├── ... (N个异构交互模块)
  ↓ 每层输出 concatenate/aggregate
多层堆叠: 上一层输出作为下一层输入
    ↓
输出层: 预测 CTR
```

### 核心创新点

**① 深层次异构集成（Deep Hierarchical Ensemble）**

- **层内集成（Intra-layer）**：同一层中并行使用多个不同的交互模块
- **层间堆叠（Inter-layer）**：将集成输出作为下一层的输入，构建深层次交互

$$h_l = \text{Aggregate}\left(\text{Module}_1(h_{l-1}),\ \text{Module}_2(h_{l-1}),\ \ldots,\ \text{Module}_K(h_{l-1})\right)$$

其中 Aggregate 可选 concat + linear projection 或 attention-based fusion。

### 公式2：层间堆叠（Inter-layer Stacking）

$$h_l^{(k)} = \text{Module}_k\left(\text{Aggregate}(h_{l-1}^{(1)}, h_{l-1}^{(2)}, \ldots, h_{l-1}^{(K)})\right)$$

**解释：**
- $h_l^{(k)}$：第 $l$ 层第 $k$ 个交互模块的输出
- 上一层**所有模块的输出**被聚合后，作为下一层**每个模块的输入**
- 这使得高层模块能学习低层模块之间的交互关系

### 公式3：最终预测

$$\hat{y} = \sigma\left(W_{out} \cdot \text{Aggregate}(h_L^{(1)}, \ldots, h_L^{(K)}) + b_{out}\right)$$

**② 信息互补性分析**

实验验证：不同交互模块（FM、DCN、CIN等）即使声称捕获相同阶数的交互，实际捕获的信息仍存在显著差异，这是设计集成系统的理论基础。

**③ 协同设计训练系统**

针对 DHEN 更深、更宽的结构，专门设计训练优化：
- 模型并行：不同交互模块分配到不同设备
- 梯度检查点：降低显存占用
- 混合精度训练：FP16/BF16 加速

## 实验结论

| 指标 | 提升幅度 |
|------|---------|
| Normalized Entropy (NE) 改进 | **0.27%** |
| 训练吞吐量提升 | **1.2×** |
| 对比基线 | 业界 SOTA（Facebook 内部系统） |

**消融实验关键发现：**
- 层次化堆叠比单层集成提升更大
- 不同交互模块的互补性得到定量验证
- 协同设计训练系统使 DHEN 实际可部署（无系统优化则训练不可行）

## 工程落地要点

1. **交互模块选择**：根据业务数据特性选择互补性强的模块组合（稀疏 vs 稠密特征不同策略）
2. **层数控制**：层数太深导致 overfitting，建议 2-4 层，配合正则化
3. **计算开销**：N个模块×L层相比单模型计算量 N×L 倍，需要模型压缩/蒸馏部署
4. **系统协同**：大规模部署必须做模型并行 + 梯度检查点，否则显存爆炸
5. **NE 指标解读**：工业CTR系统中 0.1% NE 提升即有显著业务价值，0.27% 是非常大的提升

## 常见考点

**Q1: DHEN 的核心假设是什么？为什么不同的特征交互模块能互补？**
A: 核心假设是"不同设计的交互模块捕获非重叠的特征交互信息"。原因：①FM 基于内积的二阶交互，DCN 基于 cross network 的显式高阶，DIN 基于注意力的用户行为建模——归纳偏置不同 ②即使阶数相同，数学形式的差异导致覆盖的特征关系不同

**Q2: 什么是 Normalized Entropy (NE)？为什么用它评估 CTR 模型？**
A: NE = 模型交叉熵损失 / 背景 CTR 对应的最大熵，归一化后消除不同数据集点击率不同的影响。0.27% NE 改进意味着模型不确定性显著降低，在大规模广告系统中对应数亿美元的收益提升。

**Q3: DHEN 的层次化结构相比简单的 ensemble（对多个独立模型预测求平均）有什么优势？**
A: ①深层交互：上层模块处理下层模块的集成输出，可以学习交互模块之间的高阶关系 ②端到端训练：所有模块联合优化，相互适应 ③参数共享：共享 embedding，避免 ensemble 的参数膨胀

**Q4: 协同设计训练系统（co-designed training system）具体包含哪些技术？**
A: ①模型并行：不同 interaction module 分配到不同 GPU ②梯度检查点（gradient checkpointing）：用重计算换显存 ③混合精度训练：FP16 前向，FP32 梯度累积 ④通信优化：减少跨设备梯度同步开销

**Q5: 工业级 CTR 系统中，如何决定是否值得引入 DHEN 这样复杂的模型？**
A: 决策框架：①计算预算评估（服务延迟 vs 效果提升 tradeoff）②离线 NE 提升量（>0.05% 通常值得进一步验证）③在线 A/B 测试 RPM 提升 ④基础设施成本（模型并行训练的工程成本）⑤维护复杂度（多模块系统更难debug）
