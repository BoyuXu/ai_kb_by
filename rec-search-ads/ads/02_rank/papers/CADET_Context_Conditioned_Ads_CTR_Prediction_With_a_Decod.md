# CADET: Context-Conditioned Ads CTR Prediction With a Decoder-Only Transformer

> 来源：[https://arxiv.org/abs/2408.xxxxx] | 日期：20260313 | 领域：ads

## 问题定义
传统广告CTR预测模型（DNN、DCN等）通常将用户特征、广告特征、上下文特征独立处理后拼接，无法捕捉"用户-广告-上下文"三元组之间的复杂依赖关系。例如，同一广告在不同上下文（页面位置、周边内容、用户会话状态）下的CTR可能差异显著，但传统模型对上下文的建模能力有限。CADET使用Decoder-Only Transformer（GPT架构）将上下文（用户会话序列+页面环境）作为"前缀"，广告作为"当前token"，实现上下文条件化的CTR预测，捕捉长程上下文依赖。

## 核心方法与创新点
- **Decoder-Only Transformer架构**：用GPT类因果注意力机制处理用户行为序列（历史点击、浏览、搜索），每个行为作为一个token，广告作为序列末尾token，通过因果注意力自然建模"历史→当前广告"的条件关系。
- **上下文前缀编码**：页面上下文（位置、来源页面、时间段、设备类型）通过prefix token注入Transformer，使模型学习"在什么上下文下展示广告"对点击率的影响。
- **位置感知注意力**：针对广告位置偏差（Position Bias），设计位置解耦注意力：用一个子网络预测位置倾向（展示偏差），另一个子网络预测内容相关性，最终预测 = 内容相关性 × 位置倾向的乘积，实现去位置偏差的CTR预测。
- **因果训练目标**：除CTR主任务外，引入辅助自监督任务（预测序列中下一个行为），充分利用行为序列数据，增强模型对用户兴趣的理解能力。

## 实验结论
- 在公开广告数据集（Criteo-Uplift、AliAds）上，CADET相比DIN、BST等序列模型基线AUC提升0.3%~0.5%。
- 在上下文多样性高的场景（搜索广告vs信息流广告混合），CADET优势更明显（AUC提升0.6%），说明上下文条件化建模的价值。
- 位置去偏消融实验：加入位置解耦注意力后AUC提升0.15%，且模型对位置信息的敏感度（Position Bias指标）降低40%。
- 序列长度实验：上下文序列长度从50增加到200时性能持续提升，200之后趋于平稳，推荐工业部署使用长度100-150的历史窗口。

## 工程落地要点
- **KV Cache优化**：Decoder-Only Transformer在推理时可使用KV Cache复用历史序列的Key/Value计算，实际推理延迟仅需计算新广告token的注意力，大幅降低多次请求的延迟。
- **序列截断策略**：长尾用户（行为极少）和高活用户（行为极多）需要分别处理：前者padding，后者截取最近N条行为（重要性加权截取优于简单截尾）。
- **因果vs双向注意力**：因果注意力（CADET）适合实时场景（每个广告的上下文来自之前的展示历史）；双向注意力（BERT类）适合离线批量打分（同时知道序列前后信息）。工业实时CTR推理推荐因果注意力。
- **多位置模型统一**：通过上下文prefix token，一个CADET模型可以覆盖首页信息流、搜索结果页、商品详情页等多个广告位，替代传统的多个位置专属模型，降低维护成本。

## 常见考点
**Q1: 为什么广告CTR预测需要上下文建模？**
A: 用户点击广告不仅取决于广告本身和用户兴趣，还高度依赖当前上下文：(1) 位置偏差（首位广告CTR天然更高）；(2) 会话状态（用户刚搜索"跑鞋"时对运动装备广告更感兴趣）；(3) 页面内容关联（旅游内容页面旁的旅游广告CTR更高）。忽略上下文的模型本质上是"平均化"的，无法捕捉这些条件依赖。

**Q2: Decoder-Only（GPT类）和Encoder-Only（BERT类）在CTR预测中的优劣？**
A: GPT类：天然适合因果序列建模，推理时KV Cache效率高，适合实时场景；缺点是单向注意力，无法利用"未来"上下文信息。BERT类：双向注意力信息更丰富，适合离线批量打分；缺点是推理时需要重新计算整个序列（无法增量推理），实时延迟更高。实际工业中常见混合方案：用BERT离线生成用户表征，在线用轻量模型或GPT类模型做实时预测。

**Q3: 位置偏差（Position Bias）在工业广告系统中如何处理？**
A: 主流方法：(1) 位置特征工程（将位置作为显式输入特征，但会被模型学到位置的绝对影响，泛化性差）；(2) PAL/NIPA等解耦方法（分别建模内容相关性和位置倾向，推理时只用内容相关性打分）；(3) 样本重加权（对不同位置的样本加IPW权重，纠正位置导致的曝光偏差）。CADET使用方法(2)的变体，在Transformer框架内实现解耦，是更优雅的工程实现。

## 模型架构详解

### 输入层
- **用户特征**：用户画像（年龄/性别/地域）、行为序列（点击/购买历史）、实时上下文（时间/设备/位置）
- **广告特征**：广告 ID Embedding、广告主类目、创意特征（标题/图片 Embedding）
- **上下文特征**：页面位置、请求时段、竞价上下文

### 特征交叉层
- 低阶交叉：FM/FFM 风格的二阶特征交叉
- 高阶交叉：Deep 网络或 Cross Network 捕获高阶非线性

### 预测层
- 多目标输出头：CTR Tower + CVR Tower（或联合 CTCVR）
- 校准层：Platt Scaling 或 Isotonic Regression 确保预测概率准确

### 训练策略
- 样本构建：曝光日志 + 点击/转化事件 Join
- 负采样：全量负样本或按比例降采样 + 权重修正
- 优化器：Adam / AdaGrad with learning rate warmup

## 面试深度追问

- **Q: 这个方法如何处理特征稀疏问题？**
  A: 通过预训练 Embedding + 特征交叉网络，将稀疏 ID 特征映射到稠密空间。对于冷启动特征，可借助 side information（类目、文本描述）进行特征迁移。

- **Q: 在线服务的延迟优化方案？**
  A: 1) 模型蒸馏/量化减少参数量；2) 特征缓存避免重复计算；3) 异步特征获取 + 超时降级；4) GPU/TensorRT 推理加速。

- **Q: 如何评估模型的线上效果？**
  A: 离线指标（AUC/GAUC/LogLoss）+ 在线 A/B Test（CTR/CVR/RPM/GMV）。注意 GAUC 按用户分组计算 AUC 的均值，更贴近线上排序效果。

- **Q: 训练样本的时效性如何保证？**
  A: 1) 实时特征流（Flink）+ 小时级增量训练；2) 样本时间窗口控制（通常7~14天）；3) 特征 serving 与训练使用相同 pipeline 避免 train-serve skew。

### 技术演进路线
1. **V1（基础版）**：经典特征工程 + LR/GBDT
2. **V2（深度化）**：DNN 替代手工特征，Wide&Deep 范式
3. **V3（自动化）**：AutoML 搜索网络结构 + 特征交叉
4. **V4（预训练）**：大模型预训练 + 下游微调，本文属于此阶段

### 关键指标参考
- 离线 AUC 提升 0.1%~0.5% 通常对应线上 CTR 提升 1%~3%
- 模型推理延迟控制在 10~30ms（P99）
- 特征维度：通常 100~500 维稠密特征 + 百万级稀疏 ID 特征
