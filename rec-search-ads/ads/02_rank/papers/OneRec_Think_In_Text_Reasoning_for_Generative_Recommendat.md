# OneRec-Think: In-Text Reasoning for Generative Recommendation
> 来源：https://arxiv.org/abs/2510.11639 | 领域：ads | 日期：20260323

## 问题定义
快手OneRec系统的升级版，在生成式推荐中引入文本推理（In-Text Reasoning），让模型在生成推荐列表的同时产生可解释的推理过程，提升推荐质量和可解释性。

## 核心方法与创新点
- 文本内推理：推理步骤直接嵌入生成序列（<think>...推理...</think><rec>物品列表</rec>）
- 推理质量监督：用用户满意度信号反向监督推理链的质量
- 压缩推理：相比完整CoT，仅保留关键推理步骤，控制token开销
- 端到端联合优化：推理和推荐的联合损失，确保推理服务于推荐

## 实验结论
快手线上实验：OneRec-Think相比无推理的OneRec，用户留存率+0.4%，新用户留存提升更显著（+1.2%）；推理链长度与推荐质量呈倒U型关系，最优约50-100 token。

## 工程落地要点
- 推理token增加约30%的计算开销，需要硬件扩容或推理优化
- 推理内容可作为推荐解释展示给用户，提升信任度
- 需要过滤不良推理内容（涉及隐私、偏见等），添加内容安全层

## 常见考点
1. **Q: 推荐系统中推理（reasoning）的价值？** A: 处理复杂多义意图、提升长尾推荐、提供可解释性、增强用户信任
2. **Q: 推理长度对效果的影响？** A: 适中推理最优（倒U型），过短无法深入分析，过长引入噪声和干扰
3. **Q: 如何评估推理链的质量？** A: 与推荐结果的相关性、用户满意度相关性、人工评估、LLM-as-judge
4. **Q: 新用户（冷启动）为何更受益于推理？** A: 新用户行为数据少，推理可以引入常识先验弥补数据不足
5. **Q: 推荐解释的展示策略？** A: 简短摘要（1-2句）展示给用户、可折叠详情、个性化详细程度
