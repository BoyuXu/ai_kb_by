# 广告特征工程（Ad Feature Engineering）

> 来源：工程实践 | 日期：20260317

## 问题定义

广告 CTR/CVR 预估模型的输入特征质量直接决定上限。广告系统特征工程有三大挑战：
1. **多源异构**：用户特征、广告特征、上下文特征来自不同系统，格式各异
2. **实时性要求**：部分特征需近实时更新（用户最近行为）
3. **高维稀疏**：广告/用户 ID 空间亿级，大量 ID 出现频次低

## 核心特征体系

### 用户特征

| 特征类型 | 示例 | 实时性 |
|---------|------|--------|
| 人口属性 | 年龄、性别、城市、设备 | 低频更新 |
| 历史行为 | 最近 7/30 天点击/购买类目 | 日更 |
| 实时行为 | 最近 1h 搜索词、点击 | 秒级 |
| 消费能力 | 历史消费金额分位 | 日更 |
| 兴趣标签 | 系统分配的兴趣 ID 列表 | 日更 |

### 广告特征

| 特征类型 | 示例 |
|---------|------|
| 广告 ID | Creative ID、Campaign ID、广告主 ID |
| 内容特征 | 标题词、图片类别、视频时长 |
| 统计特征 | 历史 CTR、展示量、点击量（需平滑） |
| 竞争特征 | 出价、质量分、账户历史 |

### 上下文特征

- 位置/时间：小时、星期、季节
- 场景：App 页面、feed 位置、设备类型
- 请求上下文：Search Query（搜索广告）、Page Category（展示广告）

## 核心技术

1. **统计特征平滑（CTR Smoothing）**

$$
\hat{CTR} = \frac{\text{click} + \alpha \cdot \mu}{\text{impression} + \alpha}
$$

-

$$
\alpha
$$

为平滑参数，

$$
\mu
$$

为全局平均 CTR
   - 低展示量广告 CTR 估计更保守，避免过拟合

2. **特征交叉**
   - 手动交叉：用户性别 × 广告类目、小时 × 设备
   - 自动交叉：FM/DeepFM/DCN 自动学习

3. **序列特征**
   - 用户行为序列：[item_id_1, item_id_2, ...] → DIN/DIEN attention
   - Target Attention：关注与当前广告相关的历史行为

4. **特征哈希（Feature Hashing）**
   - 高维稀疏 ID 通过哈希映射到固定大小 embedding table
   - 注意哈希冲突率，通常桶大小 10x 特征基数

5. **实时特征（Online Features）**
   - 用 Flink/Spark Streaming 计算窗口统计
   - Redis/Aerospike 存储实时特征，毫秒级 lookup

## 工程落地要点

1. **特征一致性（Training-Serving Skew）**：离线训练特征和在线服务特征必须完全一致，这是生产最常见问题
2. **特征存储分层**：批处理特征（HDFS/Hive）+ 实时特征（KV 存储）+ 请求级特征（实时计算）
3. **缺失值处理**：Embedding 需要特殊的缺失 token，数值特征用均值或 0 填充并加 is_missing 标志
4. **特征监控**：监控特征分布漂移（KL 散度），及时发现上游数据问题

## 常见考点

- **Q: 广告 CTR 历史统计特征为什么需要平滑？**
  A: 新广告或低展示量广告的历史 CTR 方差大（少量点击/未点击就会使比例极端），不平滑会导致模型对这些广告的特征学习过拟合。平滑后低置信度的统计值向全局先验收缩。

- **Q: Training-Serving Skew 的常见原因？**
  A: 1) 离线用 Label 时的特征与服务时特征计算逻辑不同；2) 实时特征在训练时用离线近似，但服务时用实时值；3) 特征版本不对齐（训练时的特征快照与当时的线上特征不同）。

- **Q: 如何处理高基数 ID 特征？**
  A: 1) 低频 ID 映射到 UNK token；2) 特征哈希到固定大小 embedding table；3) 频率截断（只保留出现次数>N的 ID）；4) 分层 embedding（粗粒度 + 细粒度 ID 组合）。

## 模型架构详解

### 输入层
- **用户特征**：用户画像（年龄/性别/地域）、行为序列（点击/购买历史）、实时上下文（时间/设备/位置）
- **广告特征**：广告 ID Embedding、广告主类目、创意特征（标题/图片 Embedding）
- **上下文特征**：页面位置、请求时段、竞价上下文

### 特征交叉层
- 低阶交叉：FM/FFM 风格的二阶特征交叉
- 高阶交叉：Deep 网络或 Cross Network 捕获高阶非线性

### 预测层
- 多目标输出头：CTR Tower + CVR Tower（或联合 CTCVR）
- 校准层：Platt Scaling 或 Isotonic Regression 确保预测概率准确

### 训练策略
- 样本构建：曝光日志 + 点击/转化事件 Join
- 负采样：全量负样本或按比例降采样 + 权重修正
- 优化器：Adam / AdaGrad with learning rate warmup


## 与相关工作对比

| 维度 | 本文方法 | 传统方法 | 优势 |
|------|---------|---------|------|
| 特征交叉 | 自动化/高阶 | 手工/低阶 | 减少特征工程，捕获复杂模式 |
| 多任务学习 | 联合优化 | 独立模型 | 共享知识，缓解数据稀疏 |
| 在线适应 | 增量更新 | 全量重训 | 更快响应分布漂移 |
| 样本效率 | 全空间/去偏 | 有偏子集 | 更准确的概率估计 |


## 面试深度追问

- **Q: 这个方法如何处理特征稀疏问题？**
  A: 通过预训练 Embedding + 特征交叉网络，将稀疏 ID 特征映射到稠密空间。对于冷启动特征，可借助 side information（类目、文本描述）进行特征迁移。

- **Q: 在线服务的延迟优化方案？**
  A: 1) 模型蒸馏/量化减少参数量；2) 特征缓存避免重复计算；3) 异步特征获取 + 超时降级；4) GPU/TensorRT 推理加速。

- **Q: 如何评估模型的线上效果？**
  A: 离线指标（AUC/GAUC/LogLoss）+ 在线 A/B Test（CTR/CVR/RPM/GMV）。注意 GAUC 按用户分组计算 AUC 的均值，更贴近线上排序效果。

- **Q: 训练样本的时效性如何保证？**
  A: 1) 实时特征流（Flink）+ 小时级增量训练；2) 样本时间窗口控制（通常7~14天）；3) 特征 serving 与训练使用相同 pipeline 避免 train-serve skew。
