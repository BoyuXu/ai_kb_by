# HiGR: Efficient Generative Slate Recommendation via Hierarchical Planning

> 来源：arxiv | 领域：ads | 学习日期：20260328

## 问题定义

**Slate 推荐**（列表级推荐）：同时推荐一组商品/广告，而非单个。挑战在于：
1. **组合爆炸**：候选池大小 N，选 K 个的组合数为 $C_N^K$，暴力搜索不可行
2. **列表内依赖**：列表中各 item 应当多样且互补，不能独立选择
3. **生成效率**：自回归生成方式逐个输出 item，对于长列表延迟高

HiGR 提出**层次规划**（Hierarchical Planning）加速生成式 Slate 推荐，在保持列表质量的同时大幅提升效率。

## 核心方法与创新点

### 两阶段层次规划

**第一阶段：类别/主题规划（Coarse Planning）**

不直接生成 item ID，先生成"槽位规划"（slot plan）：
$$Plan = (c_1, c_2, ..., c_K)$$
其中 $c_k$ 是第 k 个位置应该放什么类别/主题。

**第二阶段：槽位级 item 生成（Fine Generation）**

在类别约束下为每个槽位选择最优 item：
$$i_k = \arg\max_{i \in \mathcal{C}_{c_k}} score(i | plan, i_1, ..., i_{k-1})$$

### 层次 Token 设计

用分层 codebook 表示 item：
- Level 1：粗粒度类别 token（~100 个）
- Level 2：细粒度 item token（数百万个）

生成序列：$[c_1, c_2, ..., c_K, i_1, i_2, ..., i_K]$

类别 token 先生成，为后续 item 生成提供高层约束。

### 高效推理

由于类别 token 数量少（100 vs 百万），第一阶段 beam search 极快；第二阶段在类别约束下的候选空间大幅缩小，总体效率提升显著。

$$\text{效率提升} \approx \frac{|\mathcal{I}|}{|\mathcal{C}| + K \cdot |\mathcal{I}_{per\_cat}|}$$

## 实验结论

- 在 Amazon、Taobao 数据集上，Recall@K 和 NDCG@K 优于 TIGER、P5 等生成式推荐基线
- 推理速度比朴素自回归生成快 3-5x（主要来自第一阶段的快速规划）
- 层次规划显著提升列表多样性（Coverage 指标提升约 15%）
- 类别约束使生成结果更符合用户在类别层面的偏好

## 工程落地要点

1. **Codebook 构建**：类别 token 可用 K-means 聚类 item embedding 得到，或直接用业务类目树
2. **类别粒度选择**：太粗（10 类）多样性不足；太细（10000 类）规划效率低，通常 50-200 类合适
3. **位置感知**：不同槽位（首位、末位）的重要性不同，可以在 plan 生成时加入位置嵌入
4. **候选集过滤**：每个类别下的 item 候选用向量检索预过滤，而非全量遍历
5. **在线/离线混合**：Plan 可以提前离线生成缓存，item 选择做在线细粒度匹配
6. **多样性约束**：在 item 生成阶段加入 Maximal Marginal Relevance 约束，防止同质化

## 面试考点

**Q：为什么 Slate 推荐不能简单地独立选 K 个最优 item？**
A：独立选择忽略了 item 间的互补性和多样性。例如：用户对运动鞋感兴趣，同时推 5 双运动鞋价值低；而推 1 双鞋 + 1 件运动服 + 1 双袜子的组合用户体验更好。同时，列表内的曝光竞争效应（展示效果相互影响）也需要联合建模。

**Q：HiGR 的层次规划如何提升多样性？**
A：第一阶段显式规划类别分布，如 $[时尚, 运动, 家居, 电子, 美食]$，天然保证类别多样性；若不用规划，自回归生成容易陷入局部最优，连续选择同一类别的高分 item。

**Q：生成式推荐（如 HiGR）和传统召回+排序范式相比有什么根本区别？**
A：传统范式：召回（ANN 检索）→ 粗排 → 精排 → 重排，各阶段独立优化；生成式推荐：端到端将 item ID 视为 token，用序列模型直接生成推荐列表，天然建模列表级依赖，但对 item 词表规模敏感（百万 ID → token 空间过大）。
