# LLM as Explainable Re-Ranker for Recommendation System

> 来源：arxiv | 领域：ads | 学习日期：20260328

## 问题定义

传统推荐系统的重排（Re-ranking）阶段：
1. 缺乏可解释性：为什么这个商品/广告排在前面，用户和运营人员无法理解
2. 依赖手工设计的多目标融合规则，泛化能力差
3. 难以融合语义信息（用户意图、商品描述的语义对齐）

LLM 作为重排器的核心思路：利用 LLM 的语言理解和推理能力，对候选集进行重排，同时输出**可解释的排序理由**。

## 核心方法与创新点

### 框架设计

1. **召回阶段**：传统双塔/协同过滤召回 Top-K 候选（K=100~200）
2. **粗排阶段**：传统 CTR 模型打分，筛选 Top-N（N=10~20）
3. **LLM 精排阶段**：将候选列表格式化为 prompt，让 LLM 重排并解释

### Prompt 设计

```
User Profile: [用户历史行为摘要]
Query/Context: [当前搜索词或场景]
Candidate Items:
1. [商品标题 + 关键属性]
2. ...
N. ...

Task: Rank these items based on relevance and user preference.
Output format: Ranked list + brief explanation for top-3 items.
```

### 可解释性输出

LLM 不只输出排序，还输出：
- **整体排序依据**：用户偏好分析
- **Top items 的具体理由**：为什么这个商品更匹配
- **潜在不匹配说明**：标注哪些候选可能不相关

### 效率优化

由于 LLM 推理延迟高（100ms+），工程上采用：
1. **缓存复用**：相似 query 的排序结果缓存
2. **异步预排**：在用户浏览上一页时预计算下一页排序
3. **蒸馏部署**：用 LLM 排序结果训练小模型，线上用小模型

## 实验结论

- 在电商推荐数据集上，LLM 重排的 NDCG@10 比传统 listwise 排序提升约 5-8%
- 可解释性用户调研：73% 用户认为 LLM 给出的排序理由"有帮助"
- 蒸馏后小模型保留 LLM 重排约 85% 的效果增益，延迟从 200ms 降至 15ms
- GPT-4 优于 GPT-3.5，但 GPT-3.5 经过 fine-tuning 可接近 GPT-4 效果

## 工程落地要点

1. **候选集大小权衡**：LLM 输入 token 有限，候选集不超过 20 条（过多效果反而下降）
2. **商品描述精简**：每个候选项描述控制在 50 token 以内，突出关键属性
3. **用户画像摘要**：不能直接拼入原始历史（token 太多），需要先用规则/模型生成摘要
4. **输出解析**：LLM 输出需要 robust 解析逻辑，处理格式不规范的情况
5. **安全过滤**：LLM 的解释内容需要过滤敏感词和不当内容
6. **成本控制**：LLM API 调用成本高，只对高价值流量（高意图用户）启用精排

## 面试考点

**Q：LLM 重排相比传统 listwise 排序（LambdaMART 等）的核心优势是什么？**
A：1) 语义理解：LLM 能理解用户意图和商品描述的语义对齐，传统方法依赖手工特征；2) 零样本泛化：LLM 对新品类/新场景无需重新训练；3) 可解释性：可输出自然语言排序理由，传统黑盒模型无法做到。

**Q：LLM 重排的主要工程挑战是什么？如何解决？**
A：最大挑战是延迟（LLM 推理 100-500ms，远超排序阶段预算 10-20ms）。解决方案：1) 用 LLM 生成排序结果训练小模型蒸馏；2) 预计算缓存；3) 只对精选流量启用；4) 异步预排。

**Q：如何评估 LLM 重排的可解释性质量？**
A：1) 人工标注：标注员评估解释是否准确、有帮助；2) 用户调研：线上实验比较有/无解释时的点击率和满意度；3) 一致性评估：同一 query 多次调用 LLM，解释是否一致；4) 对抗测试：给 LLM 错误的用户偏好信息，看排序是否相应变化。
