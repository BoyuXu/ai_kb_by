# Part 1: 问题定义与业务理解

## 1.1 混排要解决什么问题？

### 本质矛盾

推荐系统和广告系统有着截然不同的优化目标：

```
推荐系统优化目标：最大化用户长期留存和平台时长
  → 选择用户最感兴趣的内容，保持 Feed 新鲜度和多样性

广告系统优化目标：最大化广告收入（RPM）
  → 选择 eCPM 最高的广告，追求每次展示的商业价值
```

如果不做混排，直接将精排结果呈现给用户，会发生什么？

**灾难性场景**：用户刷 Feed 时，前 5 条全是高 eCPM 的电商广告，第 6 条是一条自然内容，第 7-10 条又是广告。数据表明：

- 连续看到 3 个同类广告 → 点击率下降 40%
- Feed 前 5 条全是广告 → 7 日留存率下降 15-20%
- 长期用户流失 → 广告库存（可售流量）反而减少

**核心洞见**：混排不是在收入和体验之间"二选一"，而是找到一个**帕累托改进**——适当的混排策略既能提升用户体验，又能维持甚至增加长期收入。

### 三方利益博弈

| 利益方 | 核心诉求 | 量化指标 | 冲突点 |
|--------|---------|---------|--------|
| **用户** | 看到有用/有趣的内容，不被广告打扰 | CTR gap（广告 vs 自然内容点击率差距）、留存率、广告跳过率 | 广告多→体验差→流失 |
| **广告主** | 投放有效，ROI 达标，曝光公平 | ROI、CVR、曝光量、CPA | 曝光受限→ROI下降→预算转移 |
| **平台** | 最大化长期商业价值 | RPM、LTV、DAU、广告填充率 | 短期RPM↑ vs 长期DAU↓ |

### 关键指标体系

```
【收入指标】
RPM (Revenue Per Mille)：千次展示收入，最核心的商业指标
  RPM = (总广告收入 / 总展示次数) × 1000

eCPM (effective Cost Per Mille)：单条广告的有效千次展示价格
  eCPM = pCTR × pCVR × bid

【体验指标】
CTR gap = |CTR_organic - CTR_ad|
  理想值越小越好，说明广告质量接近自然内容

Ad Load（广告负载率）= 广告展示数 / 总展示数
  通常控制在 10%-35%

Ad Fatigue Index = weighted_sum(跳过率, 反感率, 连续广告后CTR衰减)
  衡量用户对广告的疲劳程度

【长期指标】
7日/30日留存率
用户 LTV (Life Time Value)
广告主续费率
```

## 2.2 混排层的定位与上下游依赖

### 混排 vs 重排——面试必须澄清的概念

```
重排 (Reranking):
  定义: 在同一类候选集内部调整顺序
  输入: 精排输出的 Top-N 广告（或 Top-N 自然内容）
  目标: 在单一候选集内优化多样性、频控、位置偏差
  举例: 广告重排——50个广告中选20个并排好序

混排 (Mixing):
  定义: 将不同来源的候选集合并为一个统一序列
  输入: 自然内容序列 + 广告序列（可能还有直播、电商等）
  目标: 在多来源混合中平衡商业价值和用户体验
  举例: 30条自然内容 + 20条广告 → 最终25条Feed
```

面试中很多人混淆这两个概念。本方案聚焦的是**混排**——它在重排之后，是最接近用户的一环。

### 上下游数据流

```
【上游依赖】
1. 推荐精排分数 (relevance score / engagement score)
   - CTR预估、完播率预估、互动率预估
   - 通常是一个 0-1 的概率值

2. 广告 eCPM
   - eCPM = pCTR × pCVR × bid
   - 量纲是"元/千次展示"，与推荐分数量纲完全不同

3. 用户画像
   - 实时特征: 最近N次浏览、当前session上下文
   - 离线特征: 兴趣标签、用户价值分层

4. 系统约束
   - 广告主预算（当天剩余）
   - 频控（用户级/广告主级/素材级）
   - 合规（某些品类在特定时段不展示）

【下游产出】
最终 Feed 序列 + 每条内容的位置 + 展示原因标注

【反馈闭环】
用户行为 → 日志收集 → 特征更新 → 模型训练 → 下次请求
```

## 2.3 统一打分的核心挑战——量纲对齐

这是混排最关键的工程问题之一：

```
推荐分数: relevance_score ∈ [0, 1]  (概率值)
广告分数: eCPM ∈ [0, 100+]  (金额)
```

如果直接比较，广告永远排在前面。解决方案：

**方案A: 虚拟 eCPM（最常用）**

为自然内容赋予一个"虚拟eCPM"，代表展示这条内容的长期商业价值：

$$
\text{virtual\_eCPM} = \text{engagement\_score} \times \text{user\_LTV\_factor} \times \text{retention\_value}
$$

其中 `retention_value` 反映"展示这条内容对用户留存的贡献"，通常通过因果推断离线估计。

**方案B: 分位数归一化**

$$
\text{norm\_score}(x) = \text{Percentile}(x, \text{历史分布})
$$

将推荐分数和广告 eCPM 都映射到 [0, 1] 的分位数空间。

**方案C: 学习转换函数**

用一个小的 MLP 学习 `f(relevance_score, user_context) → unified_score`，使得在统一分数空间中排序的结果最大化长期 LTV。

**工业界现状**：大多数公司使用方案A或B起步，成熟后迁移到方案C。

# Part 3: 混排方案演进

## 3.1 V1: 硬规则插入（2015-2017）

### 核心思路

最简单的混排：每N条自然内容插入1条广告，按用户分层调整N值。

```python
def rule_based_mixing(organic_list, ad_list, config):
    """
    硬规则混排：每 N 条自然内容插入 1 条广告
    """
    result = []
    ad_idx = 0
    insert_interval = config['insert_interval']  # 例如 N=4

    for i, content in enumerate(organic_list):
        result.append(content)

        # 每 N 条内容后插入一条广告
        if (i + 1) % insert_interval == 0 and ad_idx < len(ad_list):
            result.append(ad_list[ad_idx])
            ad_idx += 1

    return result
```

### 用户分层调N值

```
用户类型          │ 插入间隔 N │ 广告比例  │ 设计逻辑
──────────────────┼───────────┼──────────┼─────────────────
新用户 (< 3天)    │ N=8       │ ~11%     │ 保护体验，防流失
活跃用户          │ N=4       │ ~20%     │ 容忍度高，可变现
高价值用户(VIP)   │ N=6       │ ~14%     │ 减少打扰，提升满意度
流失风险用户      │ N=10      │ ~9%      │ 尽量挽回，少推广告
```

### 优缺点分析

| 维度 | 评价 |
|------|------|
| **延迟** | O(n)，极快，<1ms |
| **效果** | RPM -5%~-8%（相比纯eCPM排序），但留存持平 |
| **可维护性** | 简单直观，新人一天上手 |
| **核心缺陷** | 无法感知用户兴趣（所有用户同一规则）；无法感知上下文（不知道前面展示了什么）；无法全局优化（只是机械插入） |

### 为什么大厂都从这里起步？

1. **风险可控**：规则明确，出了问题一眼就能定位
2. **快速上线**：不需要训练模型，纯工程实现
3. **建立基线**：为后续的学习方案提供 baseline 指标
4. **积累数据**：在规则阶段积累混排日志，为训练学习模型提供样本

> **面试话术**：面试讲到这里，可以补一句"我们从V1起步但很快发现了天花板：RPM损失严重且无法个性化，所以第二个月就开始做V2"。

## 3.2 V2: eCPM 统一排序（2017-2019）

### 核心思路

给自然内容赋予虚拟 eCPM，与广告在同一分数空间排序。

$$
\text{score}_{ad} = \text{pCTR} \times \text{pCVR} \times \text{bid}
$$

$$
\text{score}_{organic} = \text{pCTR}_{organic} \times \text{engagement\_value} \times \text{retention\_multiplier}
$$

### 虚拟 eCPM 怎么算？

这是V2方案的核心技术点：

```
【方法1: 留存价值估计】

virtual_eCPM = pCTR × estimated_engagement × user_retention_value

其中:
- pCTR: 自然内容的点击率预估（推荐系统已有）
- estimated_engagement: 点击后的互动价值（停留时长、点赞、收藏的加权）
- user_retention_value: 该用户的长期价值估计
  = DAU概率 × 平均session数 × 平均广告曝光数 × 平均eCPM

举例:
- 一个高活跃用户的 retention_value = 0.85 × 5 × 8 × ¥0.5 = ¥17/天
- 一条高质量内容对留存的贡献 ≈ +0.2% 留存概率
- virtual_eCPM ≈ 0.002 × ¥17 × 1000 = ¥34 CPM

这个值和广告 eCPM（通常 ¥10-¥50）在同一量级，可以直接比较
```

```
【方法2: 历史对比法（更简单）】

virtual_eCPM = percentile_mapping(organic_score)

将自然内容的 relevance score 映射到广告 eCPM 的分位数分布上
例如: organic_score=0.9 → 映射到广告 eCPM 的 P90 = ¥35
```

### organic 高估/低估的后果

| 情况 | 现象 | 后果 |
|------|------|------|
| **高估 organic** | 自然内容分数偏高 | Feed 中广告太少 → RPM 大幅下降 → 广告主曝光不足 |
| **低估 organic** | 广告分数偏高 | Feed 中广告太多 → 用户体验差 → 留存下降 → 长期RPM也降 |
| **恰到好处** | 广告比例 15-25% | RPM +5%，留存 +2%，CTR持平或微升 |

**工业实践**：虚拟eCPM的系数通常通过 AB 测试网格搜索确定，每周根据业务目标微调。

### 效果数据

```
基线: V1 硬规则混排

V2 eCPM统一排序:
  RPM: +5% (因为高质量广告和内容交错→点击率提升)
  7日留存: +2% (多样性提升→用户不疲劳)
  广告CTR: +3% (广告出现在更合适的位置)
  P95延迟: 3ms (排序本身很快)
```

## 3.3 V3: 多目标约束优化（2019-2022）

### 核心思路

从简单的分数排序升级为带约束的优化问题：

$$
\max_{S} \quad \sum_{i \in S} \text{score}_i
$$

$$
\text{s.t.} \quad \frac{|\{i \in S : \text{type}(i) = \text{ad}\}|}{|S|} \leq \alpha \quad \text{(广告比例约束)}
$$

$$
\quad \quad |\{i \in S : \text{advertiser}(i) = a\}| \leq N_a, \forall a \quad \text{(频控约束)}
$$

$$
\quad \quad H(\text{topic\_dist}(S)) \geq \tau \quad \text{(多样性约束, Shannon Entropy)}
$$

$$
\quad \quad \text{consecutive\_ads}(S) \leq 2 \quad \text{(连续广告数约束)}
$$

### Lagrange 松弛求解

将约束转化为惩罚项：

$$
\mathcal{L}(S, \lambda) = \sum_{i \in S} \text{score}_i + \lambda_1 \left(\alpha - \frac{|S_{ad}|}{|S|}\right) + \lambda_2 \left(H(S) - \tau\right) + \ldots
$$

对偶变量 $\lambda_k$ 的更新规则：

$$
\lambda_k \leftarrow \lambda_k + \eta \cdot \text{violation}_k
$$

- 约束被违反 → $\lambda_k$ 增大 → 惩罚加重 → 下次排序自动调整
- 约束满足 → $\lambda_k$ 保持或减小

### 工业实现：在线 vs 离线

```
【在线求解（实时 Lagrangian）】
优点: 实时适应变化（广告主预算消耗、用户行为变化）
缺点: 收敛需要多次迭代，延迟较高
适用: QPS < 10万，约束维度 < 5

实现方式:
  每次请求时:
  1. 取当前 λ 值（全局共享，定期更新）
  2. 计算 Lagrangian 分数
  3. 贪心选择 Top-K
  4. 记录约束违反情况
  背景线程每分钟更新 λ

【离线预计算（Look-up Table）】
优点: 在线延迟极低（查表 O(1)）
缺点: 无法实时适应
适用: 大规模 QPS，约束相对稳定

实现方式:
  离线:
  1. 按用户分群（VIP/活跃/新客...）
  2. 每个分群预计算最优 λ 组合
  3. 写入配置中心
  在线:
  1. 查用户分群
  2. 读取对应 λ
  3. 计算分数，排序
```

### Pareto 优化

当约束阈值不好设定时，可以求解 Pareto 前沿：

$$
\min_{\theta} \sum_{k=1}^{K} \lambda_k \mathcal{L}_k(\theta), \quad \lambda \in \Delta^K
$$

**MGDA（Multiple Gradient Descent Algorithm）**：

$$
d^* = \arg\min_d \max_k \nabla_\theta \mathcal{L}_k \cdot d
$$

找到所有目标梯度的"最小包络方向"，保证每个目标都不退化。

**工业实践**：线下训练多组 Pareto 最优模型 → 线上通过调整权重在 Pareto 前沿上滑动 → AB 测试选择最优平衡点。

### 效果数据

```
基线: V2 eCPM统一排序

V3 约束优化:
  RPM: +3% (约束自动找到更优平衡点)
  7日留存: +4% (多样性约束直接保证)
  广告CTR: +2% (连续广告约束减少疲劳)
  P95延迟: 8-15ms (Lagrangian计算)

关键改进:
  连续3+条广告出现率: 从12%降至0%
  话题多样性Shannon Entropy: 从0.55提升到0.72
  广告主覆盖率: 从35%提升到60%
```

## 3.4 V4: RL 序列决策（2022-2024）

### 为什么需要 RL？

V3 的约束优化有一个根本局限：**只看当前页面，不看长期影响**。

```
约束优化的视角:
  "这一页Feed里，广告比例不超过25%，多样性够高就行"

RL的视角:
  "这一页展示什么，会影响用户明天还来不来。
   今天多展示一条用户感兴趣的自然内容，
   可能用户明天多刷3分钟，多看5条广告，
   长期收益 >> 今天少展示一条广告的损失"
```

### MDP 建模

$$
\text{State } s_t = (\underbrace{\text{user\_profile}}_{\text{兴趣/画像}}, \underbrace{\text{displayed\_list}_{<t}}_{\text{已展示序列}}, \underbrace{\text{session\_context}}_{\text{时间/设备/实时特征}})
$$

$$
\text{Action } a_t = \text{选择第 } t \text{ 个位置展示广告还是自然内容（以及具体哪一条）}
$$

$$
\text{Reward } r_t = w_1 \cdot \text{click} + w_2 \cdot \text{view\_time} + w_3 \cdot \text{engagement} + w_4 \cdot \text{ad\_revenue}
$$

$$
\text{目标: } \max_\pi \mathbb{E}\left[\sum_{t=0}^{T} \gamma^t r_t\right]
$$

其中 $\gamma \in [0.95, 0.99]$ 是折扣因子，$T$ 是一次 session 的长度。

### Actor-Critic 架构详解

```python
class MixingPolicyNetwork(nn.Module):
    """混排策略网络: Actor-Critic"""

    def __init__(self, state_dim=512, action_dim=50):
        super().__init__()
        # 状态编码器：融合用户特征 + 已展示序列 + 候选特征
        self.state_encoder = nn.Sequential(
            nn.Linear(state_dim, 256),
            nn.ReLU(),
            nn.Linear(256, 128)
        )
        # Actor: 输出每个候选的选择概率
        self.actor = nn.Sequential(
            nn.Linear(128, 64),
            nn.ReLU(),
            nn.Linear(64, action_dim),
            nn.Softmax(dim=-1)
        )
        # Critic: 估计当前状态的长期价值
        self.critic = nn.Sequential(
            nn.Linear(128, 64),
            nn.ReLU(),
            nn.Linear(64, 1)
        )

    def forward(self, state):
        encoded = self.state_encoder(state)
        policy = self.actor(encoded)      # π(a|s)
        value = self.critic(encoded)       # V(s)
        return policy, value
```

### State 设计的关键细节

```
【用户表征 (128维)】
- 用户 Embedding (从推荐系统迁移, 64维)
- 实时兴趣向量 (session内点击序列的平均embedding, 32维)
- 用户分群特征 (VIP等级/活跃度/流失风险, 16维)
- 人口统计 (年龄段/性别/城市级别, 16维)

【已展示序列 (256维)】
- 最近5条展示内容的embedding均值 (64维)
- 类型统计: 已展示广告数/自然内容数/各品类数量 (32维)
- 用户反馈: 最近5条的点击/跳过/停留时长 (32维)
- 多样性状态: 当前话题分布的Shannon Entropy (16维)
- 疲劳状态: 连续广告/连续同品类计数 (16维)
- 时间衰减: 距离上一条广告的位置距离 (16维)
- 预算状态: 广告剩余预算百分比 (16维)
- 其他上下文: session内已浏览条数/总停留时长 (48维)

【候选特征 (128维)】
- 候选embedding (64维)
- 候选类型 (广告/自然/直播, 8维)
- 候选质量分 (CTR/eCPM/relevance, 24维)
- 与已展示序列的差异度 (32维)
```

### Reward 设计的工业权衡

Reward 设计是 RL 混排的核心难题：

```
【短期信号 (即时反馈)】
- click: 0.1 (用户点击了)
- view_time: 0.001 × seconds (停留时长)
- like: 0.05 (点赞)
- share: 0.08 (分享)
- ad_revenue: 0.001 × eCPM (广告收入)

【长期信号 (延迟反馈)】
- next_day_active: 0.5 (次日还来)
- 7day_retention: 1.0 (7日还活跃)
- LTV_increase: 用户生命周期价值变化

【负向信号】
- skip: -0.02 (快速划过)
- dislike: -0.1 (点了反感)
- uninstall: -2.0 (卸载App)
```

**核心挑战**：

1. **Reward 延迟**：次日留存信号要等24小时，7日留存要等7天
   - 解决：用 Reward Model 预估长期收益，作为即时奖励的补充
2. **Reward 稀疏**：大多数展示没有明确反馈（既没点击也没反感）
   - 解决：用停留时长作为隐式反馈的 proxy
3. **多目标 Reward 权重**：$w_1, w_2, w_3, w_4$ 怎么定？
   - 解决：AB 测试 + 贝叶斯优化搜索最优权重组合

### 离线 RL vs 在线 RL

| 维度 | 离线 RL | 在线 RL |
|------|---------|---------|
| **数据来源** | 历史日志（off-policy） | 实时交互（on-policy） |
| **安全性** | 高（不影响线上用户） | 低（探索可能损害体验） |
| **样本效率** | 低（需要 IS 修正） | 高（直接从当前策略采样） |
| **实施难度** | 中（需要 IS/DICE 修正） | 高（需要安全探索机制） |
| **工业首选** | ✅ 先用离线 RL 训练初始策略 | 后续小流量在线微调 |

**Importance Sampling 修正**：

$$
\mathbb{E}_{\pi_{new}}[r] \approx \mathbb{E}_{\pi_{old}}\left[\frac{\pi_{new}(a|s)}{\pi_{old}(a|s)} \cdot r\right]
$$

离线日志是用旧策略 $\pi_{old}$ 收集的，需要通过重要性采样修正到新策略 $\pi_{new}$ 下的期望。

### 实际落地难点

```
【难点1: Exploration 安全性】
问题: RL需要探索（尝试非最优动作），但在广告场景中探索 = 损失收入
方案:
  - ε-constrained exploration: 探索幅度不超过当前最优策略的 ε%
  - 安全约束: 任何探索动作的 eCPM 不低于候选池中位数
  - 灰度隔离: 只在1%流量上探索，99%流量仍用当前最优策略

【难点2: Sample Efficiency】
问题: 用户一次session可能只有20-30次决策，训练一个好的策略需要海量数据
方案:
  - 大规模离线预训练 + 小规模在线微调
  - 将RL策略初始化为V3的约束优化结果（warm start）
  - 用模拟器扩增训练数据

【难点3: Reward 延迟】
问题: 次日留存等长期指标无法实时获得
方案:
  - 训练一个 Reward Model 预估长期价值
  - 短期奖励(即时) + Reward Model 预估(长期) 的加权组合
  - 定期用真实留存数据校准 Reward Model

【难点4: 非平稳环境】
问题: 用户偏好、广告主预算、季节性都在变化
方案:
  - 定期重训练（每天/每周）
  - 在线增量学习
  - 元学习（MAML）快速适应
```

### 效果数据

```
基线: V3 约束优化

V4 RL序列决策:
  RPM: +2% (RL学到了更好的广告插入时机)
  7日留存: +3% (长期优化目标直接体现)
  用户session时长: +8% (RL优化了序列的整体吸引力)
  P95延迟: 15-25ms (Actor网络推理)

关键发现:
  RL学到的策略vs约束优化的差异:
  1. RL倾向在session开始时少展示广告（用户需要"热身"）
  2. RL学会了在用户engagement高峰期插入广告（此时CTR最高）
  3. RL会在session快结束时适当增加广告（用户即将离开，不影响留存）
```

## 3.5 V5: 生成式混排（2024-至今，最新探索方向）

### 核心思路

不再用评分+选择的范式，而是让模型直接**生成**最优的混排序列。

$$
\pi^* = \arg\max_\pi P_{\text{LLM}}(\pi \mid \text{candidates}, \text{user}, \text{constraints})
$$

### 三种技术路线

**路线A: CoT推理链排序（GR2, 2025）**

$$
P(\text{rank} | q, u, \mathcal{C}) = P(\text{reasoning} | q, u) \cdot P(\text{rank} | \text{reasoning}, \mathcal{C})
$$

先生成推理链（为什么要这样排），再基于推理链生成排列。

```
输入:
  用户: 25岁女性，最近浏览: 美妆(3次)、旅游(1次)、健身(1次)
  候选: [口红A(美妆,eCPM=45), 旅游AppB(旅游,eCPM=30), 健身房C(健身,eCPM=15), ...]
  约束: 广告≤25%, 连续广告≤2

LLM 推理链:
  "用户最近密集浏览美妆内容，可能已经审美疲劳。
   旅游和健身是潜在兴趣但曝光不足。
   策略: 先用旅游内容打破美妆疲劳→穿插1条美妆广告→再用健身内容..."

输出排列:
  [旅游内容1, 口红A广告, 健身内容1, 旅游AppB广告, 美妆内容2, ...]
```

训练方式：SFT 预训练 + RLVR（可验证奖励强化学习）微调。GR2 使用 DAPO 训练，NDCG@5 超越 SOTA +1.3%。

**路线B: 层次规划生成（HiGR, 2024, 腾讯）**

$$
\text{Slate Planner: 先生成类别序列} \rightarrow \text{Item Generator: 再在每个类别中选具体item}
$$

```
第一步 (Slate Planner):
  输入: 用户+上下文
  输出: [美妆, 旅游, 广告, 美妆, 健身, 广告, ...]  (类别序列)

第二步 (Item Generator):
  对每个位置，在对应类别的候选中选择最优item
  可以并行执行 → 推理速度5×提升
```

关键技术：CRQ-VAE（Contrastive Residual Quantized VAE）构建 Semantic ID，将 item 映射为层次化的离散 token。

**路线C: 端到端序列生成（EGA-v2, 2025）**

$$
P(a_1, a_2, ..., a_K | u, \mathcal{C}) = \prod_{k=1}^{K} P(a_k | a_{<k}, u, \mathcal{C})
$$

直接自回归生成广告序列，用 Semantic ID 表示每个 item。

### 端到端 vs 分层 Pipeline 的 Trade-off

| 维度 | 端到端生成式 | 分层Pipeline（传统） |
|------|------------|-------------------|
| **全局优化能力** | 强——一次性考虑整个列表 | 弱——每层独立优化，信息损失 |
| **多样性** | 天然好——生成过程避免重复 | 需要额外的 DPP/MMR 后处理 |
| **可解释性** | CoT推理链可读 | 各层分数可审计 |
| **推理延迟** | 高——自回归逐步生成 | 低——各层并行 |
| **大候选集** | 差——需要先筛选到小候选集 | 好——多级过滤天然处理 |
| **工业成熟度** | 低——仍在探索阶段 | 高——几十年积累 |

### 当前工业实践

```
最前沿的系统采用混合架构:

  传统Pipeline (粗排→精排→重排)
       ↓
  输出 Top-50 候选
       ↓
  ┌────────────────────────┐
  │   路由层 (Router)       │
  │                        │
  │  用户兴趣明确 → RL混排   │  (延迟低，准确)
  │  冷启动用户 → LLM混排    │  (理解力强，灵活)
  │  需要解释 → CoT混排      │  (可追溯原因)
  └────────────────────────┘
       ↓
  融合层: 如果多个策略结果一致→直接用
         如果不一致→加权融合 (RL 70% + LLM 30%)
```

### 开放问题

1. **推理成本**：LLM 每次推理的成本是传统 DNN 的 100-1000 倍
2. **Hallucination**：LLM 可能生成不存在的 item ID
3. **Likelihood Trap**：自回归 MLE 训练目标与用户偏好不对齐，导致生成高概率但低多样性的同质列表。ConGRATS（快手）用图结构多路径解码解决，GR2 用可验证奖励 RL 解决
4. **实时性**：新广告上架到被模型感知的延迟（EGA-v2 的动态 Token 方案已将此降至 <5 分钟）

# Part 4: 关键技术深潜

## 4.1 多目标优化技术栈

### 模型结构层面

**MMoE (Google 2018)**：

$$
h^{(t)}(x) = \sum_{k=1}^{K} g_k^{(t)}(x) \cdot f_k(x)
$$

$$
g^{(t)}(x) = \text{softmax}(W_g^{(t)} \cdot x)
$$

每个任务有独立门控，选择性使用不同 Expert。但所有 Expert 对所有任务开放，任务差异大时仍有干扰。

**PLE (腾讯 2020)**：

在 MMoE 基础上增加：
- **私有 Expert**：每个任务独占
- **共享 Expert**：所有任务共用
- 多层渐进式抽取

一句话区别：MMoE 是软分离（通过 gate 隐式分配），PLE 是显式分离（硬性划分私有/共享）。

### 损失函数层面

**不确定性加权 (Kendall 2018)**：

$$
\mathcal{L} = \sum_{k=1}^{K} \frac{\mathcal{L}_k}{2\sigma_k^2} + \log\sigma_k
$$

$\sigma_k$ 是可学习参数——任务不确定性高时 $\sigma_k$ 大，权重自动降低。

### 梯度操纵层面

**PCGrad**：当两个任务梯度方向冲突时，将梯度投影到对方的正交方向：

$$
g_i' = g_i - \frac{g_i \cdot g_j}{\|g_j\|^2} g_j \quad \text{if } g_i \cdot g_j < 0
$$

**MGDA**：找所有任务梯度的最小包络方向：

$$
d^* = \arg\min_d \max_k \nabla_\theta \mathcal{L}_k \cdot d
$$

### 约束优化层面

**Lagrange 对偶 + PID 控制**（广告系统最常用）：

$$
\lambda_k \leftarrow \lambda_k + K_p \cdot e_k + K_i \cdot \sum e_k + K_d \cdot \Delta e_k
$$

其中 $e_k$ 是约束违反量。PID 控制器让 $\lambda$ 的更新更平滑，避免振荡。

## 4.2 用户疲劳建模

### 连续广告对 CTR 的衰减函数

经验公式（工业界常用）：

$$
\text{CTR}_{adj}(i) = \text{CTR}_{base}(i) \times e^{-\beta \cdot \text{consecutive\_ad\_count}}
$$

其中 $\beta \approx 0.3$，即连续第2条广告的CTR衰减约26%，第3条衰减约59%。

更精细的建模：

$$
\text{fatigue}(i) = \alpha_1 \cdot \text{same\_advertiser\_count} + \alpha_2 \cdot \text{same\_category\_count} + \alpha_3 \cdot \text{consecutive\_ad\_count}
$$

$$
\text{CTR}_{adj}(i) = \text{CTR}_{base}(i) \times \sigma(-\text{fatigue}(i))
$$

### 多维度疲劳

```
【广告主级疲劳】
同一广告主7天内展示超过3次 → CTR衰减30%
同一广告主同一session展示2次 → CTR衰减50%

【品类级疲劳】
同一品类连续展示3次 → 跳过率从20%升至45%
同一品类在Feed中占比>40% → 7日留存下降2%

【创意级疲劳】
同一素材7天内展示超过1次 → CTR衰减40%
（用户记得"看过这个广告"→忽略）

【时间维度】
广告疲劳在session内累积，但跨session有衰减
衰减半衰期 ≈ 2天（用户2天后对广告的记忆减半）
```

## 4.3 广告比例动态化

### 按用户价值分层

```python
def get_dynamic_ad_ratio(user):
    """
    根据用户价值动态确定广告比例
    """
    base_ratio = 0.20  # 默认20%

    # VIP用户：减少广告，保护体验
    if user.vip_level >= 2:
        return base_ratio * 0.5  # 10%

    # 新用户：极低广告比例
    if user.age_days < 3:
        return 0.05  # 5%

    # 流失风险用户：减少广告
    if user.churn_probability > 0.7:
        return base_ratio * 0.4  # 8%

    # 活跃且高容忍度用户：适度增加
    if user.dau_ratio > 0.8 and user.ad_skip_rate < 0.15:
        return base_ratio * 1.5  # 30%

    # 根据实时session状态微调
    if user.session_ad_count > 5:
        return base_ratio * 0.7  # session内已看较多广告，降低

    return base_ratio
```

| 活跃用户 | 30% | 25% | 55% | 主力变现群体 |
| 普通用户 | 40% | 20% | 25% | 平衡体验和收入 |
| 低活跃 | 15% | 8% | 4% | 保护体验，防流失 |

## 4.4 频控与去重策略

### 三级频控体系

```
Level 1: 广告主级频控
  规则: 同一广告主，7天内展示上限 N 次
  N 的设定: 根据广告主品类
    - 电商: N=5 (用户可能反复比价)
    - 品牌: N=3 (品牌记忆足够)
    - 游戏: N=7 (高频用户可能转化)

  实现: Redis Hash
    key: user:{uid}:advertiser_freq
    field: advertiser_id
    value: count:timestamp

Level 2: 素材级频控
  规则: 同一创意素材，7天内展示上限 1 次
  原因: 用户对"见过的广告"有明确记忆，重复展示的CTR极低

  实现: Bloom Filter (节省内存)
    估算: 1亿用户 × 平均50条已见素材 = 50亿对
    Bloom Filter误判率1%时约需6GB内存

Level 3: 品类级频控
  规则: 当前Feed中同一品类广告不超过 M 条
  M 的设定:
    - 电商广告: M=2
    - 信息流广告: M=3
    - 品牌广告: M=1

  实现: 在线计数器 (per-request scope)
```

### 频控的软硬结合

```
硬频控 (Hard Cap): 绝对不能违反
  - 合规要求的频控（如医疗广告24小时最多1次）
  - 广告主合同约定的频控

软频控 (Soft Cap): 尽量遵守，必要时可突破
  - 品类级频控（广告库存不足时可适度放宽）
  - 实现方式: 超频后不是直接过滤，而是对分数施加惩罚
    score_adjusted = score × decay_factor^(excess_count)
```

## 4.5 位置偏差校正

### Position Bias 在混排中的影响

用户注意力随位置指数衰减：

$$
P(\text{click} | \text{item}, \text{pos}) = P(\text{relevant} | \text{item}) \times P(\text{examine} | \text{pos})
$$

$$
P(\text{examine} | \text{pos}) \approx \frac{1}{\log_2(\text{pos} + 1)} \quad \text{(DCG 假设)}
$$

或更通用的：

$$
P(\text{examine} | \text{pos}) = e^{-\lambda \cdot \text{pos}} \quad (\lambda \approx 0.1\text{-}0.3)
$$

### 混排中位置偏差的特殊性

```
问题: 广告通常被放在用户注意力高的位置(前几条)
     → 广告CTR被高估（不是广告好，是位置好）
     → 模型学到"广告应该排前面"的偏差
     → 恶性循环：越放前面→越被点击→模型越认为应该放前面

解决方案:

1. IPW (Inverse Propensity Weighting):
   训练时对每个样本加权 w = 1 / P(examine|pos)
   高位置的点击权重降低，低位置的点击权重提升

2. Position Feature:
   将位置作为模型输入特征
   推理时用统一的位置值（如pos=1）消除位置影响

3. PAL (Position-Aware Learning):
   训练两个模型：
   - 位置无关的相关性模型 P(relevant|item)
   - 位置模型 P(examine|pos)
   推理时只用相关性模型排序
```

### 混排层的位置偏差处理

```python
def position_debias_mixing(candidates, position_model):
    """
    位置校正后的混排
    """
    debias_scores = []
    for candidate in candidates:
        # 原始分数（包含位置偏差）
        raw_score = candidate.score

        # 估计位置效应（在训练集平均位置下的CTR）
        position_effect = position_model.predict(candidate.avg_historical_position)

        # 去偏分数 = 原始分数 / 位置效应
        debiased_score = raw_score / position_effect

        debias_scores.append(debiased_score)

    # 用去偏分数排序
    sorted_indices = argsort(debias_scores, reverse=True)

    # 排好序后，再乘回实际位置的效应（用于最终eCPM计算）
    result = []
    for rank, idx in enumerate(sorted_indices):
        actual_position_effect = position_model.predict(rank + 1)
        final_score = debias_scores[idx] * actual_position_effect
        result.append((candidates[idx], final_score))

    return result
```

## 4.6 实时性工程约束

### 混排延迟预算分配

```
总延迟预算: <100ms (端到端)
  ├── 网络传输: 10-20ms
  ├── 推荐Pipeline: 40-50ms
  │     ├── 召回: 5-10ms
  │     ├── 粗排: 5-10ms
  │     ├── 精排: 20-30ms
  │     └── 重排: 5-10ms
  ├── 广告Pipeline: 20-30ms
  │     ├── 广告召回: 5ms
  │     ├── 广告精排: 10-15ms
  │     └── 竞价: 5ms
  └── 混排层: <20ms  ← 关键约束
        ├── 特征加载: 3-5ms
        ├── 分数计算: 3-5ms
        ├── 约束检查: 2-3ms
        ├── 排序/选择: 3-5ms
        └── 结果封装: 1-2ms
```

### 延迟优化技巧

```
1. 特征预计算
   不在请求时计算用户特征，而是维护实时特征存储
   用户每次行为 → 异步更新特征 → 请求时直接读取
   效果: 特征加载从40ms降至3ms

2. 候选预过滤
   在混排前先做快速过滤:
   - 硬频控过滤 (Bloom Filter, O(1))
   - 预算耗尽的广告主过滤
   候选从50→30，减少后续计算量

3. 评分缓存
   对近期请求的评分结果做LRU缓存
   同一用户短时间内多次请求（如翻页）可以复用部分评分

4. 模型量化
   Actor网络从FP32→INT8
   推理延迟从15ms降至5ms，精度损失<0.1%

5. 并行计算
   多个候选的评分可以batch parallel
   使用SIMD指令加速向量运算
```

# Part 5: AB实验与效果验证

## 5.1 混排AB实验设计

### 分流粒度

```
推荐AB的分流粒度: 通常按用户ID hash分桶

混排AB的特殊性:
  - 用户级分流 ✓ (主流方案)
    user_bucket = hash(user_id) % 100
    实验组: bucket 0-9 (10%流量)
    对照组: bucket 10-99 (90%流量)

  - 请求级分流 ✗ (不推荐)
    问题: 同一用户不同请求看到不同策略→体验不一致
    而且无法评估留存等长期指标

  - 广告主级分流 ✗ (不推荐)
    问题: 不同广告主的流量会相互影响
    例: 广告主A在实验组曝光减少→A的预算剩余多→
    影响对照组中其他广告主的竞价
```

### 样本量计算

$$
n = \frac{(z_{\alpha/2} + z_\beta)^2 \cdot 2\sigma^2}{\delta^2}
$$

```
参数解释:
- α = 0.05 (显著性水平), z_α/2 = 1.96
- β = 0.2 (统计功效80%), z_β = 0.84
- σ: RPM的标准差 (从历史数据估计)
- δ: 最小可检测效应 (MDE)

举例:
  想检测RPM +2%的变化
  RPM均值=¥50, σ=¥15
  δ = ¥50 × 2% = ¥1
  n = (1.96 + 0.84)² × 2 × 15² / 1² ≈ 3528
  即每组至少需要3528个用户-天的数据

  如果每天10万DAU，1%流量 = 1000用户
  至少需要运行 3528/1000 ≈ 4天
```

### 推荐实验时长

```
最短: 7天 (覆盖工作日+周末的差异)
推荐: 14天 (覆盖两个完整周，消除周效应)
长期指标验证: 28-30天 (留存指标需要足够窗口)

原因:
1. 工作日vs周末的用户行为差异很大
2. 广告主预算周期通常是周为单位
3. Novelty Effect: 新策略头几天可能有虚假提升
4. 长期指标(留存)需要至少2周才能看到稳定趋势
```

## 5.3 常见陷阱

### 陷阱1: 辛普森悖论

```
现象: 整体RPM提升了2%，但分人群看:
  - 活跃用户: RPM -1%
  - 低活跃用户: RPM +8%

原因: 实验组中低活跃用户占比更高（因为分桶不均匀），
     而低活跃用户的RPM基数低，提升幅度大但绝对值小。
     整体均值被低活跃用户的高增长率拉上去了。

解决:
  - 分层分析 (stratified analysis)
  - 对每个用户群体单独计算效果
  - 用加权平均代替简单平均
```

### 陷阱2: Novelty Effect

```
现象: 实验头3天RPM +5%，然后逐渐回落到+1%

原因:
  - 新的混排策略改变了Feed结构
  - 用户对"新鲜"的布局更愿意点击
  - 适应后回归正常水平

解决:
  - 实验至少跑14天
  - 剔除前3天的数据
  - 关注尾部稳定值而非峰值
```

### 陷阱3: 广告主预算耦合

```
现象: 实验组RPM +3%，但一个月后广告主A投诉曝光减少

原因:
  - 实验组改变了广告展示策略
  - 广告主A在实验组的曝光减少
  - 但A的日预算不变 → A的预算在对照组消耗更多
  - 导致对照组的其他广告主竞争减少 → 对照组RPM略升
  - 实验组真实效果被低估

解决:
  - 监控广告主维度的曝光/消耗变化
  - 使用 Budget-isolated 分流（给实验组和对照组分配独立预算池）
  - 或在评估时修正预算溢出效应
```

### 陷阱4: 季节性干扰

```
现象: 11月做的实验RPM +10%，但12月全量上线后RPM只+3%

原因:
  - 11月双十一前广告主集中投放，预算充足
  - 广告库存高 → 混排优化空间大
  - 12月广告主预算收紧，优化空间变小

解决:
  - 同期对比（实验组vs对照组，而非实验vs历史）
  - 在不同季节重复验证
  - 使用Diff-in-Diff消除时间趋势
```

## Q3: eCPM 和虚拟 eCPM 有什么区别？虚拟 eCPM 高估/低估会怎样？

```
eCPM (广告):
  = pCTR × pCVR × bid
  代表: 展示这条广告的即时商业价值
  单位: 元/千次展示

虚拟eCPM (自然内容):
  = engagement_score × user_LTV_factor × retention_value
  代表: 展示这条内容的长期间接商业价值
  本质: 将"留存/时长/活跃度"等非直接收入指标货币化

高估organic → Feed广告太少 → RPM大幅下降
低估organic → Feed广告太多 → 用户流失 → 长期RPM也降
关键: 虚拟eCPM不是一个精确值，而是通过AB测试找到的最优操作点
```

## Q4: 多目标优化在混排中怎么做？MMoE 和 PLE 区别是什么？

```
混排的多目标:
  - 收入 (RPM): maximize
  - 用户体验 (留存/时长): maximize
  - 多样性 (Shannon H): ≥ threshold
  - 广告负载 (Ad Load): ≤ threshold

三种处理范式:
  1. 线性加权: score = Σ w_i × f_i (简单但权重难调)
  2. 约束优化: max f1 s.t. f_i ≥ θ_i (工业主流)
  3. Pareto优化: 找Pareto前沿 (理论最优但计算复杂)

MMoE vs PLE:
  MMoE: 所有Expert共享 → Gate soft routing → 任务差异大时仍有干扰
  PLE: 私有Expert + 共享Expert → 显式隔离 → 减少负迁移
  工业选择: 目标数<4用MMoE够了，>4或目标差异很大用PLE
```

## Q8: 位置偏差(Position Bias)在混排中如何影响和纠偏？

```
影响:
  广告通常放在高位置 → CTR被高估 → 模型学到"广告应排前面"
  → 恶性循环

纠偏方法:
  1. IPW: 训练时对高位置样本降权 w=1/P(examine|pos)
  2. Position Feature: 模型输入位置特征，推理时用固定位置值
  3. PAL: 分离相关性模型和位置模型，推理只用相关性

混排特殊性:
  广告和自然内容的位置偏差曲线可能不同
  （用户对广告的位置衰减更快，因为"广告盲区效应"）
  需要分别建模 P(examine|pos, type=ad) 和 P(examine|pos, type=organic)
```

## Q9: 广告比例(Ad Load)应该怎么动态调整？

## Q10: 混排系统的降级方案怎么设计？


## Q11: 生成式混排(V5)与传统混排的核心区别是什么？工业落地的挑战？

```
核心区别:
  传统: 评分→排序→约束后处理 (分阶段pipeline)
  生成式: 直接输出最终序列 (端到端)

优势:
  1. 全局优化: 一次性考虑整个列表，避免贪心次优
  2. 天然多样性: 生成过程中隐式避免重复
  3. 可解释性: CoT推理链给出排序理由
  4. 灵活性: 通过prompt调整策略，无需重新训练

挑战:
  1. 推理延迟: LLM推理比DNN慢10-100倍
     → 解决: 蒸馏到小模型 / 分层并行解码(HiGR)
  2. Hallucination: 可能生成不存在的item
     → 解决: Constrained Beam Search / Semantic ID体系
  3. Likelihood Trap: MLE训练→同质化列表
     → 解决: RLVR(GR2) / 图结构多路径解码(ConGRATS)
  4. 成本: 每次请求的计算成本高100倍
     → 解决: 只在部分请求(如冷启动用户)使用LLM混排
```

## Q12: 如何处理推荐和广告的"时间尺度冲突"？

```
广告系统: 按天/小时优化（广告主预算按天消耗）
推荐系统: 按周/月优化（留存是长期指标）

小时级 (在线):
```

## Q13: DPP (Determinantal Point Process) 在混排中怎么用？

```
核心思想:
  P(S) ∝ det(L_S)
  选择一组item S的概率正比于核矩阵L的子矩阵行列式
  det大 = 高质量且多样 → 概率高

核心公式:
  L[i,i] = quality[i]²  (对角线: item质量)
  L[i,j] = quality[i] × quality[j] × similarity[i,j]  (非对角线: 相似度)

在混排中的应用:
  将广告和自然内容统一编码为embedding
  构造L矩阵: quality用归一化分数, similarity用cosine相似度
  贪心近似选K个: 每步选使det增量最大的item

工程挑战:
  精确求解O(n³)太慢 → 贪心近似O(n²k)
  n=50, k=20时: ~50000次运算，约5ms
  近似比有理论保证: ≥1/e ≈ 0.37

工业实践:
  通常DPP只用在重排层(单一候选集内多样性)
  混排层更多用约束优化(可以处理异构候选的量纲差异)
  但DPP思想可以嵌入约束优化的多样性项中
```

## Q14: 混排系统怎么处理广告主的竞争关系？

```
场景: 广告主A(可口可乐)和B(百事可乐)是直接竞品
  如果同一页Feed同时展示A和B的广告 → 两家都不满意

解决方案:

1. 竞争关系图
   维护广告主之间的竞争关系Graph
   边权重表示竞争强度(0=无竞争, 1=直接竞品)
   来源: 广告主自行标注 + 品类自动推断

2. 排他约束
   同一Feed中，竞争度>0.8的广告主只能出现1个
   实现: 在约束优化中增加排他约束
   ∀(a,b) ∈ CompetitorPairs: x_a + x_b ≤ 1

3. 轮替展示
   A和B按照eCPM比例轮替展示
   如A的eCPM=40, B的eCPM=30 → A展示57%, B展示43%
   保证长期公平性

4. 品类级隔离
   更通用的方案: 同品类广告在Feed中间隔≥3个位置
   实现简单，覆盖面广
```

## 附录C: 推荐+广告一体化趋势（OMTM）

### One Model Two Markets

传统架构中推荐和广告是完全独立的两套系统，但最新趋势是统一建模：

$$
\text{score} = f_{\text{relevance}}(\text{user}, \text{item}) + g_{\text{commercial}}(\text{bid}, \text{budget}, \text{context})
$$

### Bid-Aware 建模

将商业信号（出价、预算）作为显式输入而非隐式 bias：

- 传统：模型只看用户和内容特征，出价在排序后才介入
- Bid-Aware：模型同时看到用户/内容/出价/预算，学习"在出价X下这条广告对用户Y的价值"

### 核心挑战

| 挑战 | 描述 | 应对方案 |
|------|------|---------|
| 目标冲突 | 用户满意度 vs 商业收入 | 约束优化 + Pareto |
| 分布偏移 | 高出价广告偏多，扭曲训练分布 | 反事实校正 + 重采样 |
| 用户体验保护 | 需要硬约束兜底 | 兴趣阈值 + 广告比例硬限 |

### 统一模型的参数共享

$$
\text{底层共享 (User/Item Embedding)} + \text{任务特定头 (CTR/CVR/Engagement)} + \text{门控融合 (动态权重)}
$$

优势：
- 推荐和广告共享用户表征 → 冷启动改善
- 统一训练 → 参数效率高
- 端到端优化 → 避免pipeline级联误差
