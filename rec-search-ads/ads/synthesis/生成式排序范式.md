## 📐 核心公式与原理

### 1. 传统排序 vs 生成式排序

**传统排序（Ranking from Pool）**

$$
\text{Score}(i) = f_\theta(\text{features}_i)
$$

$$
\text{Output} = \text{argsort}(\{\text{Score}(1), ..., \text{Score}(N)\})
$$

- 限制：只能从 N 个候选中选择
- 特点：模型看完全部候选后做决策

**生成式排序（Generative Ranking）**

$$
\text{Sequence} = [Ad_1, Ad_2, ..., Ad_K] \sim P_\theta(\cdot | \text{user\_context})
$$

- 特点：逐步生成，不受候选集限制
- 原理：条件概率链式分解

$$
P(\text{seq}) = \prod_{t=1}^{K} P(\text{Ad}_t | \text{Ad}_{<t}, \text{context})
$$

### 2. Seq2Seq 架构（标准实现）

**编码器（Encoder）**：用户上下文编码

$$
\text{user\_embedding} = \text{Encoder}(\text{user\_id}, \text{history}, \text{context})
$$

**解码器（Decoder）**：逐步生成广告 ID

$$
P(\text{Ad}_t | \text{context}, \text{Ad}_{<t}) = \text{softmax}(\text{Decoder}(\text{Ad}_{<t}, \text{user\_embedding}))
$$

### 3. 采样策略

**贪心解码（Greedy Decoding）**

$$
\text{Ad}_t = \arg\max_{\text{Ad}} P(\text{Ad} | \text{context}, \text{Ad}_{<t})
$$

- 快速，确定性输出
- 风险：局部最优，可能陷入次优排列

**Beam Search**
- 保留每步的 Top-B 候选，全局搜索最优
- 权衡速度和最优性：B=3~5 较常见

**Temperature Sampling**

$$
P(\text{Ad}) = \frac{\exp(\text{logit}(\text{Ad}) / T)}{\sum_{\text{Ad}'} \exp(\text{logit}(\text{Ad}') / T)}
$$

- T 越小越接近贪心；T 越大越随机
- 用于增加生成多样性

## 🎯 核心洞察

### 1. **范式转变：从 Ranking 到 Generative**

| 维度 | 传统排序 | 生成式排序 |
|-----|---------|-----------|
| **输入** | 固定 N 个候选 | 用户上下文 |
| **搜索空间** | 大小 = N! 或 (N choose K) | 大小 = 库规模^K（指数级） |
| **冷启动** | 新广告无排序特征 → 排名靠后 | 直接从上下文生成 → 自动获得曝光 |
| **长尾效果** | 只有进入候选才有机会 | 任何商品都可能被生成 |
| **推理延迟** | ~10ms（排序快） | ~100-200ms（生成慢） |
| **模型规模** | 中等（GBDT 或浅层 DNN） | 大型（Seq2Seq Transformer） |

### 2. **冷启动问题的根本解决**

传统排序的冷启动困境：
```
新广告 → 无历史特征（无点击记录）
       → CTR 预估为 0
       → 排名垫底
       → 无曝光机会
       → 无法积累数据
       → 永久冷启动恶性循环
```

生成式排序的突破：
```
新广告 → 从用户上下文直接生成
      → 基于"用户喜欢这类广告"而非"这个广告受欢迎"
      → 每个新广告都有 P(Ad | context) > 0
      → 获得初始曝光
      → 积累反馈数据
      → 快速优化
```

### 3. **突破候选集限制的意义**

- **传统 Top-50 召回 + 排序**：只考虑 50 个广告，其他 N-50 个无机会
- **生成式排序**：理论上可从全库（百万级）生成，实际通过采样限制
- **结果**：长尾广告获得 30%+ 的曝光增长（Google Wukong 实验）

### 4. **关键技术：强制有效性约束**

生成式模型面临的挑战：
```
【问题】
P(seq) 可能生成重复广告或无效 ID
例如：[Ad_123, Ad_123, Ad_456]  // 重复
      [Ad_999999, Ad_123]      // 无效 ID

【解决】
强制解码约束：
- 维护已生成集合 {Ad_1, Ad_2, ...}
- Decoder 输出时，将已生成广告的概率置为 0
- 确保每个位置只生成有效的"新"广告
```

### 5. **Online-Offline 协作的新机制**

- **离线**：用大规模历史数据训练 Seq2Seq 模型
- **在线**：实时获取用户最新上下文（所在页面、最近点击等）
- **反馈**：点击 / 停留秒数 → 动态调整模型权重或 RL 微调
- **闭环**：周期性用新数据重训 Seq2Seq，精化表示

## 🔧 技术实现路线

### A. 基础 Seq2Seq + Attention

```
【架构】
User Context → Encoder（LSTM/Transformer）→ user_emb [B, D]
                                           ↓
                      Decoder（LSTM + Attention）← Ad Embedding Table
                                           ↓
                      Output: Attention Weights [T, N]
                                           ↓
                      Sampled Ad Sequence [T]

【特点】
- 标准 NLP 架构，易于实现
- 注意力机制自动学习 "给哪些广告更高概率"
- 推理快（~50-100ms）
```

### B. Transformer 解码器（WUKONG 方案）

```
【架构】
input: [user_emb, Ad_emb_1, Ad_emb_2, ...]

Decoder Layers:
  - Self-Attention：学习广告间的关联（例如品类互补）
  - Cross-Attention：与用户表示关联
  - Feed-Forward

Output: 序列概率分布

【优势】
- 并行生成可行（相比 LSTM）
- 捕捉长距离依赖（例如"前 5 个广告影响第 10 个的生成"）
- 可扩展到大模型（GPT-scale）

【挑战】
- 推理延迟更长（100-200ms）
- 内存占用大
- 需要大规模预训练
```

### C. 广告 ID 的离散化编码

传统方案：广告 ID → One-hot（维度 = 库规模，百万级）❌ 不可行

**WUKONG 方案**：多层次编码
```
Ad_ID_12345
  ↓
Layer 1: 品类（电商、美妆、房产...）→ 100 维
Layer 2: 子品类 → 200 维
Layer 3: 广告主 ID → 256 维
Layer 4: Ad 内容特征（标题、图片 embedding） → 512 维

最终 Embedding = Concatenate + Linear Projection → 256 维

好处：
- 参数可学习，不是固定 one-hot
- 利用离散特征的层次结构
- Embedding 维度 << 库规模
```

## 🚀 工业案例：Google WUKONG（2024）

### 背景

Google 在广告库超过 1 亿的场景下实验生成式排序，对标传统的"召回 Top-50 + 精排"流程。

### 核心创新

1. **大规模 Seq2Seq Transformer**：
   - 编码器：8 层 Transformer，512 维
   - 解码器：8 层 Transformer，位置编码支持到 K=100
   - 参数量：~500M

2. **分层广告编码**：
   - 品类、广告主、内容特征的混合表示
   - 冷启动新广告：从内容特征自动继承表示

3. **两阶段生成**：
   - 第一阶段：快速生成粗排（K=100，Greedy）
   - 第二阶段：精排前 20 个（Beam Search）

### 结果

| 指标 | 提升 |
|-----|------|
| **RPM** | +2.3% |
| **新广告曝光** | +30% |
| **用户点击率** | +1.1% |
| **推理延迟** | 150ms（可接受） |

### 关键成功因素

- 离线用 100B+ 展示数据训练
- 在线 A/B 测试 + 流量分层（10% → 50% → 100%）
- 定期重训（周级别），适应季节性和新广告流入

## 📊 生成式排序 vs 传统排序对比

| 维度 | 传统排序 | 生成式排序 | 混合方案 |
|-----|---------|-----------|---------|
| **召回精度** | 高（Top-50 已过滤） | 中（全库生成） | 高（两阶段） |
| **冷启动效果** | 差 | 优 | 好 |
| **推理成本** | 低（10ms） | 高（150ms） | 中（50ms） |
| **实现复杂度** | 低 | 高 | 中 |
| **适用场景** | 通用 | 冷启动频繁、长尾丰富 | 大规模工业应用 |

## 🛡️ 挑战与解决方案

### 挑战 1：推理延迟

**问题**：生成式模型 150-200ms，可能超过系统预算。

**解决方案**：
```
① 分层生成：第一层快速（Greedy），第二层精化（Beam）
② 缓存：预计算热门用户的表示
③ 模型蒸馏：用小模型蒸馏大模型，加速推理
④ 批处理：离线生成热门 Feed，在线组装
```

### 挑战 2：生成重复/无效广告

**问题**：生成器可能输出相同 ID 或库中不存在的 ID。

**解决方案**：
```
① 强制去重：维护已生成集合，Decoder 输出时掩码
② 约束解码：只允许有效 ID 的概率 > 0
③ 后处理：生成后检查有效性，无效则拒绝
```

### 挑战 3：离线-在线偏差

**问题**：训练数据分布 vs 线上真实流量分布差异。

**解决方案**：
```
① 定期重训：周级别用最新数据重训
② 在线学习：Bandit 算法在线调整采样策略
③ IPS Weighting：用倾向分估计修正偏差
```

### 挑战 4：模型监控与调试

**问题**：Seq2Seq 黑盒，难以理解"为什么生成这个广告"。

**解决方案**：
```
① 注意力可视化：看 Decoder 的注意力权重
② 特征重要性：各层 Embedding 对输出的贡献
③ 反事实分析：改变输入，观察输出变化
```
