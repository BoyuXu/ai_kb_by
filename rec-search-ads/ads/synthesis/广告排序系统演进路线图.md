## 导言：为什么这份演进路线很重要

广告系统的排序（Ranking）经历了 15+ 年的演进，从简单的**分数排序** → **多目标权衡** → **约束优化** → **端到端学习** → **生成式排序**。

这个演进历程**不是单纯的技术升级**，而是在三个相互冲突的目标间寻找平衡：
1. **平台收入**（最大化 RPM/eCPM）
2. **用户体验**（避免广告轰炸、保证内容质量）
3. **实时性**（<100ms 的排序延迟）

每个阶段的核心问题与解决方案值得深入理解。


## 📐 核心公式与原理

### 1. eCPM 排序
$$
eCPM = pCTR \times pCVR \times bid
$$
- 广告排序的核心公式

### 2. 质量得分
$$
Q = \alpha \cdot pCTR + \beta \cdot relevance + \gamma \cdot landing\_quality
$$
- 综合质量影响排序和计费

### 3. ROI 约束
$$
ROI = \frac{revenue}{cost} \geq target
$$
- 广告主的核心约束条件

## Phase 1：简单排序（2010-2015）

### 1.1 核心设计：线性得分排序

**目标函数**：
```
Score(ad) = CTR × Bid
排序：按 Score 从高到低排列
```

**算法特点**：
- ✅ 简单高效，O(n log n)
- ✅ 易于理解和维护
- ❌ 没有用户体验约束
- ❌ 容易被游戏化（恶意高出价）

### 1.2 典型问题：广告轰炸

**场景**：
- 用户打开 Feed，前 5 个全是广告
- 用户体验极差，点击率反而下降
- 平台流量开始流失

**根本原因**：
排序只考虑 eCPM，没有任何多样性或用户体验约束。

### 1.3 衍生工作

- **竞价机制改进**：GSP（广义第二价格）vs VCG（Vickrey-Clarke-Groves）
  - 激励相容性改进，防止虚报
  - 但仍然无法解决"全是广告"的问题

---

## Phase 2：多目标融合（2015-2019）

### 2.1 核心转变：从单目标到多目标

**新的目标函数**：
```
Score(ad) = w1 × CTR + w2 × CVR + w3 × Brand_Safety + w4 × User_Happiness
           w1 + w2 + w3 + w4 = 1
```

**目标的来源**：
- **CTR / CVR**：广告主利益（能否转化）
- **Brand Safety**：平台方利益（广告主满意度）
- **User Happiness**：最终用户利益（体验不被轰炸）

### 2.2 技术进展：多任务学习

**多模型并行预估**：
```
         Input Features
              ↓
    ┌─────────┴─────────┐
    ↓                   ↓
  CTR Model          CVR Model        Brand Safety Model
  (DeepFM)           (ESMM)             (Text Classifier)
    ↓                   ↓                   ↓
  P(Click)          P(Convert)         Safety Score
    ↓                   ↓                   ↓
    └─────────┬─────────┘─────────┬──────┘
              ↓                    ↓
        Hard Parameter Sharing    Multi-Task Learning
        (简单共享 Embedding)      (负迁移问题！)
```

**关键论文与技术**：
- **Wide & Deep Learning**（Google 2016）：记忆 + 泛化的平衡
- **MMOE（Multi-gate Mixture of Experts）**（Google 2018）：任务隔离，缓解负迁移
- **PLE（Progressive Layered Extraction）**（腾讯 2020）：进一步隔离专家，改进 MMOE
- **ESMM（Entire Space Multi-Task Model）**（阿里 2018）：CTR+CVR 的统一框架，解决样本偏差

### 2.3 重大问题：权重如何设定？

```
Score = 0.4 × CTR + 0.3 × CVR + 0.2 × Brand_Safety + 0.1 × User_Happiness

Q: 这些权重从哪里来？
A: 通常是**人工经验 + A/B 测试迭代**

问题：
- 权重随着市场变化需要不断调整
- 不同广告类型可能需要不同权重
- 无法自动适应实时场景变化
```

### 2.4 可观察的真实效果

- **CTR/CVR 预估精度提升** → 排序更准确
- **用户体验有所改善** → 但仍然容易被游戏化（例如"软文"广告很好看但实际是广告）
- **仍然没有硬约束** → "最多展示 30% 广告"这样的约束无法强制

## Phase 3：约束优化排序（2019-2023）

### 3.1 核心突破：从权重融合到约束优化

**新的问题表述**：
```
maximize:   CTR × CVR × Bid  （主目标：广告收入）

subject to:
  - RPM ≥ threshold_rpm      （收入下限）
  - Diversity ≥ 0.3          （至少 30% 非广告）
  - Brand_Safety ≥ 0.8       （品牌安全评分下限）
  - Ad_Frequency ≤ 5/day     （用户看到同一广告最多 5 次）
  - Topic_Diversity ≥ 0.4    （话题多样性）
  
限制：Runtime < 100ms         （实时性保证）
```

这是一个**约束优化问题**，不再是加权组合！

### 3.2 求解方法：拉格朗日松弛 + KKT 条件

**基本思想**：
将约束条件转化为**惩罚项**，迭代优化

```
L(x, λ, μ) = f(x) - Σ λᵢ gᵢ(x) - Σ μⱼ hⱼ(x)

其中：
- f(x)：主目标函数（最大化 eCPM）
- g(x)：不等式约束（RPM ≥ 阈值）
- h(x)：等式约束（多样性 = 目标值）
- λ, μ：对偶变量（在迭代中学习）
```

**算法**：
1. **初始化**：λ = [0, 0, ...], μ = [0, 0, ...]
2. **原问题优化**：给定 λ、μ，用贪心或 DP 找最优排序
3. **对偶问题优化**：更新 λ、μ，使约束得到满足
4. **迭代**：重复 2-3 直到收敛

### 3.3 工业案例：Meta（Facebook）的 MOEF

**MOEF（Multi-Objective Exploration Framework）**：
- 论文：Mohan et al., 2021，Meta
- 核心：用**贪心约束排序** + **实时反馈**循环优化

```
【Ranking Pipeline】
  
  Candidates (Top 1000)
      ↓
  [Coarse Ranking]
      ↓
  [Fine Ranking with Multi-Task CTR/CVR/CVR2C models]
      ↓
  [Constraint-based Reranking]  ← 约束优化
    • 检查多样性约束
    • 调整广告频率
    • 应用帕累托最优
      ↓
  [Online Feedback Loop]  ← 用户实际点击反馈
    • 更新约束权重
    • 调整下一轮排序
```

**关键突破**：
- 约束不再是"刚性的"，而是**动态的、可学习的**
- 用户反馈直接影响排序策略

### 3.4 多样性的定义与衡量

```
【话题多样性】
Topic Diversity = (不同话题数) / (总广告数)
示例：Top 10 广告中有 8 个不同话题 → 多样性 = 0.8

【来源多样性】
Source Diversity = (不同广告主数) / (总广告数)
示例：避免同一广告主垄断 Feed

【新颖性】
Novelty = 1 - (用户已看过此话题的比例)
示例：用户之前看过"电商"，再看同类广告新颖性低

【频率控制】
Frequency Cap = max(看到同一广告的次数)
示例：同一广告每天最多 3 次
```

### 3.5 可观察的真实效果

- **用户体验明显改善** → Feed 中广告分布更合理
- **收入仍然保持** → 约束优化保证 RPM 下限
- **减少用户流失** → 多样性约束有效降低广告疲劳
- **模型规模爆炸** → 需要 10+ 个子模型预估各种目标/约束

**问题**：
- 约束定义仍需手工调试
- 不同场景、不同用户群体可能需要不同约束
- 线上离线数据分布不一致（online-offline gap）

## Phase 4：学习排序 + 重排层（2023-2025）

### 4.1 核心创新：端到端学习排序过程

**关键认识**：
```
【Before】排序层 = 固定的规则 + 启发式
【After】排序层 = 可学习的神经网络 + 强化学习
```

**关键技术：LTR（Learning To Rank）**

#### A. 排序学习的三种范式

```
【Pointwise LTR】
输入：单个广告的特征
输出：该广告的评分
损失：回归或分类损失
问题：不考虑排序的相对顺序

【Pairwise LTR】
输入：两个广告的特征对 (ad_i, ad_j)
输出：ad_i 应该排在 ad_j 之前的概率
损失：Pairwise 排序损失（RankNet, LambdaRank）
优势：考虑了相对顺序
问题：O(n²) 对的数量，计算复杂

【Listwise LTR】
输入：整个候选广告列表
输出：最优的排列顺序
损失：Listwise 排序损失（LambdaMART, ListNet）
优势：直接优化排序指标（NDCG, MAP）
问题：计算复杂，但效果最好
```

#### B. LambdaMART：工业标准算法

**LambdaMART 的核心思想**：
```
【传统梯度增强】
∇ Loss = ∂Loss/∂pred_i  （普通梯度）

【LambdaMART】
∇ Loss = λᵢⱼ = (|NDCG_ij|) / (1 + exp(σ × (pᵢ - pⱼ)))
         （排序特定的梯度）

λᵢⱼ 的含义：
- 当 pᵢ < pⱼ 但 i 排名应高于 j 时，λᵢⱼ 很大（强信号）
- 当 pᵢ > pⱼ 且 i 排名本就高时，λᵢⱼ 很小（弱信号）
```

**为什么 LambdaMART 在 2024 年仍是工业标准**：
- ✅ 直接优化 NDCG/MAP 等排序指标
- ✅ 梯度增强树天然支持特征交叉
- ✅ 推理快，占用内存少
- ✅ 容易添加排序约束
- ❌ 不如深度学习灵活
- ❌ 难以捕捉复杂的特征交互

### 4.2 强化学习排序（RL Ranking）

**动机**：用户点击后再露出，应该直接优化用户反馈

```
【MDP 定义】
状态 S：已展示的广告列表 + 用户上下文
动作 A：从候选集中选择下一个广告
奖励 R：用户是否点击（+1 点击，0 无点击）
策略 π：给定状态 S，选择动作 A 的概率分布

【目标】
maximize: E[Σ γᵗ Rₜ]  （折扣累计奖励）
```

**关键挑战**：
1. **探索-利用权衡**：现在最好的排序是否最优？需要探索其他排序以找到更好的
2. **序列决策**：前面展示什么广告会影响后续用户反馈
3. **样本效率**：需要大量线上交互才能学会最优策略

**工业解决方案**：
- **Contextual Bandit**：简化版 RL，每次只预测下一个广告（不考虑序列）
- **Policy Gradient（A3C）**：用神经网络学习策略
- **Thompson Sampling**：基于贝叶斯的探索策略

### 4.3 重排层（Reranking）的正式定义

**重排层的输入和输出**：
```
输入：精排后的 Top-50 广告（已有初步排序）
输出：最终展示的 20 个广告（应用约束和优化）

【重排层要解决的问题】
1. 多样性调整：避免话题、来源重复
2. 用户反馈融入：基于用户的实时反馈调整
3. 多约束优化：同时满足多个硬约束
4. 序列效应：考虑广告的展示顺序对转化的影响
```

**重排的三类算法**：

| 算法类别 | 代表 | 优势 | 劣势 |
|---------|------|------|------|
| **启发式** | 贪心约束、循环法则 | 简单快速 | 无法学习，效果有上限 |
| **学习排序** | LambdaMART on reranking list | 直接优化指标 | 离线训练，online-offline gap |
| **强化学习** | Policy Gradient for reranking | 直接优化在线反馈 | 探索不足，样本效率低 |

### 4.4 实际效果与问题

**可观察的效果**：
- **NDCG 提升 5-10%**（排序指标直接改善）
- **用户停留时间 +8%**（更好的多样性 → 更多浏览）
- **广告点击率 +3-5%**（虽然竞争激烈，但好的排序仍能提升）

**遗留问题**：
- **Online-Offline Gap**：离线排序模型和线上实际点击的差异
- **冷启动新广告**：新广告没有历史反馈，如何排序？
- **计算成本**：每个用户都需要运行排序模型，<100ms 很紧张
- **模型复杂度**：10+ 个预估模型 + 多个排序模型，维护成本高

## Phase 5：生成式排序与多智能体（2025+）

### 5.1 范式转变：从"排序"到"生成"

**关键问题**：
```
【传统排序的瓶颈】
给定 K 个候选（例如 Top-50），从中选择 N 个（例如 20 个）并排序
这是一个 O(K!) 的组合优化问题

如果 K=50，N=20，可能的排列数 = C(50,20) × 20! ≈ 10^30
无法穷举，必须用启发式或贪心

【新思路：生成式排序】
为什么不直接生成最优排序，而不是从固定候选集中选择？
```

### 5.2 生成式排序（Generative Ranking, GR）

**核心思想**：用序列生成模型直接生成最优排序

```
【模型架构】
用户上下文 + 广告库
    ↓
[编码器] 编码用户和广告信息
    ↓
[解码器 (Transformer)] 逐步生成排序
  - Token 1: 生成第 1 个最好的广告 ID
  - Token 2: 生成第 2 个最好的广告 ID
  - ...
  - Token N: 生成第 N 个最好的广告 ID
    ↓
最优排序序列

【优点】
- 不受"Top-K 候选"的限制，可从全库（亿级）生成
- 端到端优化排序质量
- 更灵活处理约束（约束编码为 prompt token）

【缺点】
- 推理成本高（需要 N 次解码，每次 100ms）
- 难以保证最优性
- 幻觉问题（生成虚假广告 ID）
```

**工业论文**：
- "Generative Recommendation for Large-Scale Advertising"（头部平台，2023）
- 报告生成式排序在亿级广告库上的实现
- RPM +2.3%，冷启动广告曝光 +30%

### 5.3 扩散排序（Diffusion-based Ranking）

**新的想法**：用扩散模型生成排序

```
【扩散模型排序的直觉】
1. 从随机排列开始（全是噪声）
2. 逐步去噪，变成高质量排序
3. 每一步受到"排序质量评分函数"的指导

【数学形式】
x_T ~ N(0, I)           （初始化：随机排列）
x_{t-1} ~ p(x_{t-1}|x_t) （去噪步骤）
...
x_0                      （最终排序）

评分函数：∇ log p(y|x) 指导去噪方向
y = "好的排序"（用 CTR/CVR 和约束定义）
```

**优点**：
- 理论基础扎实（已在图像生成、文本生成中验证）
- 容易加入约束（编码为条件指导信号）
- 可迭代改进（多步去噪）

**缺点**：
- 推理时间长（50-200 步去噪）
- 实时性堪忧（超过 100ms 要求）
- 仍在探索阶段，工业应用有限

### 5.4 多智能体强化学习排序（MARL Ranking）

**想法**：用多个 RL agent 协作生成排序

```
【Agent 设计】
- Agent 1：最大化收入（CTR × Bid）
- Agent 2：最大化用户体验（多样性）
- Agent 3：最大化品牌安全
- ...

【协作机制】
每个 agent 控制一个"排序位置"，逐步协商出最优排序
例如：
  Agent 1 想把高 eCPM 广告放第 1 位
  Agent 2 想把多样的广告放第 1 位
  → 协商出折中方案（例如放第 2 位高 eCPM，第 1 位多样）
```

**优点**：
- 目标明确，易于理解
- 支持异构目标（不同 agent 优化不同目标）
- 可扩展（多目标 → 加 agent）

**缺点**：
- 协作算法复杂（博弈论、纳什均衡）
- 收敛困难
- 工业应用还在研究阶段

### 5.5 Phase 5 的可观察趋势

- **LLM 赋能排序**：用大语言模型理解"排序意图"（例如 "展示品牌广告优先" 编码为 prompt）
- **实时个性化排序**：不再用全局模型，而是为每个用户 finetune 排序策略
- **成本与效果的权衡**：是否值得用复杂排序算法换取 2-3% 的 RPM 提升？

## 对比总结：各 Phase 的关键指标

| 维度 | Phase 1 | Phase 2 | Phase 3 | Phase 4 | Phase 5 |
|-----|--------|--------|--------|---------|---------|
| **排序目标** | 单目标 (eCPM) | 加权多目标 | 约束优化 | 学习排序 | 生成排序 |
| **约束处理** | 无 | 软约束 (权重) | 硬约束 (拉格朗日) | 学习约束权重 | 编码为生成条件 |
| **多样性** | ❌ | 权重很小 | ✅ 独立约束 | ✅ 学到多样性 | ✅ 自动优化 |
| **实时反馈** | ❌ | ❌ | 初步尝试 | ✅ (RL) | ✅ 完整闭环 |

### Q1: 广告系统的全链路延迟约束是什么？
**30秒答案**：端到端 <100ms：召回 <10ms，粗排 <20ms，精排 <50ms，竞价 <10ms。关键优化：模型蒸馏/量化、特征缓存、异步预计算。

### Q2: 广告和推荐的核心技术差异？
**30秒答案**：①校准要求不同（广告需绝对概率，推荐只需排序）；②约束不同（广告有预算/ROI 约束）；③更新频率不同（广告更高频）；④数据不同（广告有竞价日志）。

### Q3: 广告系统的数据闭环怎么做？
**30秒答案**：展示日志→点击/转化回收→特征构建→模型训练→线上服务。关键：①归因窗口设置（7天/30天）；②延迟转化处理；③样本权重修正；④在线学习增量更新。

### Q4: 广告系统如何处理数据稀疏问题？
**30秒答案**：①多任务学习（用 CTR 辅助 CVR）；②数据增广（LLM 生成/对比学习）；③迁移学习（从相似领域迁移）；④特征工程（高阶交叉特征增加信号密度）。

### Q5: 隐私计算对广告系统的影响？
**30秒答案**：三方 Cookie 消亡后：①联邦学习（多方数据联合建模不出域）；②差分隐私（加噪保护用户数据）；③安全多方计算；④First-party Data 价值提升。挑战：效果和隐私的 trade-off。

### Q6: 广告 CTR 模型的在线 A/B 怎么做？
**30秒答案**：分流：按用户 hash 分桶，保证同一用户始终在同一组。核心指标：CTR、CVR、RPM（千次展示收入）、广告主 ROI。时长：至少 7 天（覆盖周效应）。注意：广告有预算效应，需要同时监控广告主消耗。

### Q7: 广告特征工程有哪些核心特征？
**30秒答案**：①用户画像（年龄/性别/兴趣标签）；②广告属性（品类/出价/预算/素材质量）；③上下文（时间/设备/位置）；④交叉统计（用户×品类的历史 CTR）；⑤实时特征（最近 N 次曝光/点击）。

### Q8: 广告模型的样本构建有什么特殊之处？
**30秒答案**：①曝光不等于展示（广告被加载但用户可能没看到）；②延迟转化（点击后数天才转化）；③竞价日志（不仅有展示结果，还有出价/竞争信息）；④样本加权（不同位置的曝光权重不同）。

### Q9: 自动出价和手动出价的效果对比？
**30秒答案**：数据显示自动出价通常比手动出价提升 15-30% ROI。原因：①实时调整能力（秒级 vs 天级）；②全局优化（考虑跨时段预算分配）；③数据驱动（比人工经验更精准）。但冷启动期手动出价更稳定。

### Q10: 广告系统的降级策略？
**30秒答案**：①模型服务不可用：回退到规则排序（按出价×历史 CTR）；②特征服务延迟：用缓存特征替代实时特征；③预算系统故障：按历史消耗速度限流；④全系统故障：展示自然内容，不展示广告。
