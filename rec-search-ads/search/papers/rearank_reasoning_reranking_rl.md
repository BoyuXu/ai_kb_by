# REARANK: Reasoning Re-ranking Agent via Reinforcement Learning
> 来源：https://arxiv.org/abs/2505.20046 | 领域：search | 学习日期：20260401

## 问题定义

在信息检索的重排阶段，大型语言模型（LLM）已展现出强大潜力，但现有方法面临两个核心挑战：

1. **推理与重排割裂**：大多数 LLM 重排方法直接输出排序结果，没有显式的推理步骤，导致模型难以处理需要深度理解的复杂查询
2. **标注数据稀缺**：高质量的重排标注数据极其昂贵，传统 SFT 方法需要大量标注样本

**REARANK** 提出了一种基于强化学习的 LLM 列表式推理重排代理，核心思路是：
- **先推理，再重排**：在给出最终排序前，模型先生成明确的推理过程
- **极少标注样本**：仅需 **179 个标注样本**，通过 RL 大幅提升性能
- 在 Qwen2.5-7B 上实现与 GPT-4 相当甚至超越的性能

## 核心方法与创新点

### 整体架构：列表式推理重排

```
输入：query Q + 候选文档列表 [d1, d2, ..., dk]
         ↓
[推理阶段] 分析每个文档与查询的相关性、对比不同文档
         ↓
[重排阶段] 输出文档排序 [di, dj, ..., dk]（listwise 格式）
```

REARANK 采用 **listwise** 方法一次性对所有候选文档排序（而非 pointwise 逐一打分），能捕捉文档间的相对关系。

### 强化学习训练框架

REARANK 的关键创新在于设计了专门用于重排任务的 RL 奖励函数：

**奖励设计**：

$$
R = \alpha \cdot \text{nDCG@10}(\hat{\pi}, \pi^*) + \beta \cdot \text{Format}_{	ext{Reward}}
$$

- **主奖励**：预测排序与金标准排序之间的 nDCG@10 差异
- **格式奖励**：确保输出格式正确（有效的排序列表）
- 通过 GRPO（Group Relative Policy Optimization）或 PPO 进行策略优化

### 数据增强策略

仅有 179 个标注样本的情况下，REARANK 通过以下方式扩充训练信号：

1. **负样本挖掘**：从 BM25 检索结果中挖掘难负样本
2. **排列增强**：对同一查询生成多种文档排列，计算各排列的奖励
3. **推理轨迹生成**：用强模型（如 GPT-4）生成初始推理轨迹用于冷启动

### 与 SFT 方法的对比

| 维度 | SFT 重排 | REARANK (RL) |
|------|---------|-------------|
| 标注需求 | 大量（数万条） | 极少（179条） |
| 推理过程 | 隐式/无 | 显式可见 |
| 泛化能力 | 依赖标注分布 | 强（RL 探索） |
| 复杂查询 | 弱 | 强 |

## 实验结论

### 核心结果

**基础模型**：Qwen2.5-7B → REARANK-7B

| 基准 | GPT-4 | REARANK-7B | 提升 |
|------|-------|-----------|------|
| MSMARCO DL19 | 73.1 | 72.8 | ≈同等 |
| MSMARCO DL20 | 71.4 | 71.9 | +0.5 |
| BRIGHT（推理密集）| 28.4 | 29.7 | **+1.3** |
| BEIR 平均 | 56.2 | 55.8 | ≈同等 |

**关键发现**：
- REARANK-7B 在 in-domain 和 out-of-domain 均与 GPT-4 相当
- 在 **BRIGHT 推理密集基准**上超越 GPT-4，验证 RL 对推理能力的增强
- 相比无推理的基线模型，REARANK 在复杂查询上提升显著（+5-10% nDCG）
- 仅 179 个样本的 RL 训练效果超过数千样本的 SFT 训练

### 消融实验

- 去除推理步骤 → nDCG 下降 3.2%（验证显式推理的价值）
- 去除数据增强 → 性能不稳定（验证增强策略的必要性）
- SFT 冷启动 → 无冷启动收敛更慢（验证冷启动的有效性）

## 工程落地要点

### 推理效率优化

**挑战**：listwise 推理需要处理多个文档，token 消耗大

**优化策略**：
1. **分层重排**：先用轻量模型粗排 Top-100，再用 REARANK 精排 Top-20
2. **推理长度控制**：设置 max_reasoning_tokens 上限，避免过度思考
3. **批量并行**：对不同 query 的候选文档列表并行推理

### 训练数据收集建议

```python
# 最小数据需求：179 个高质量标注
# 推荐格式：
{
  "query": "...",
  "positive_docs": [...],  # 相关文档
  "negative_docs": [...],  # 不相关文档（BM25 hard negatives）
  "gold_ranking": [...]    # 金标准排序
}
```

### RL 训练注意事项

1. **奖励稳定性**：nDCG 是稀疏奖励，需要结合格式奖励平滑训练曲线
2. **KL 散度惩罚**：防止策略偏离参考模型过远（避免"奖励黑客"）
3. **探索温度**：推理生成阶段使用较高温度（T=0.8-1.0），排序输出阶段用低温度（T=0.1）

### 部署架构

```
用户查询 → 初始检索（BM25/Dense）→ Top-100
→ REARANK listwise 推理 → 最终 Top-10
→ 推理链可选择性展示给用户
```

## 常见考点

**Q1：REARANK 为什么只需要 179 个标注样本就能超越大量 SFT 训练的模型？**
A：RL 的关键优势在于通过奖励信号（nDCG）探索和自我改进，不依赖于标注样本的覆盖率。模型可以自主探索不同排序策略，优化目标（nDCG）直接对齐最终评测指标，避免了 SFT 的标注偏差问题。

**Q2：listwise vs pointwise vs pairwise 重排的区别与适用场景？**
A：
- Pointwise：逐文档打分，忽略文档间关系，效率高但准确度低
- Pairwise：文档两两比较，准确但 O(n²) 复杂度
- Listwise（REARANK）：整体考虑所有文档，捕获文档间相对关系，准确度最高但计算量最大

**Q3：REARANK 的 RL 奖励函数如何设计？为什么选 nDCG 而不是 MAP 或 MRR？**
A：nDCG@10 考虑了位置权重（越靠前越重要）和分级相关性，与实际用户行为模式最接近。MAP 对每个位置权重相等，MRR 只关注第一个相关文档，都不如 nDCG 全面。

**Q4：在实际系统中如何平衡 REARANK 的推理质量和延迟？**
A：通过分层架构解决：(1) 初始检索用高效近似方法（ANN）取 Top-100；(2) 用轻量模型粗排到 Top-20；(3) 仅对 Top-20 使用 REARANK 精排。对延迟要求极高的场景，限制推理 token 数量（budget forcing）。
