# A Survey of Model Architectures in Information Retrieval

> 来源：arXiv 2502.14822 | 领域：search | 学习日期：20260402

## 问题定义

信息检索领域的模型架构经历了从 BM25 到神经检索的演进，当前存在多种架构范式（稀疏检索、密集检索、后期交互、交叉编码器、生成式检索）。本综述系统梳理各架构的设计理念、优劣和适用场景。

## 核心方法与创新点

1. **架构分类体系**：
   - **稀疏检索**：BM25、SPLADE、DeepCT — 可解释性强、可用倒排索引
   - **密集检索**：DPR、E5、GTE — 语义匹配能力强、需 ANN 索引
   - **后期交互**：ColBERT、ColBERTv2 — 精细 token 级匹配、延迟与精度平衡
   - **交叉编码器**：MonoBERT、RankT5 — 最高精度、延迟最高
   - **生成式检索**：DSI、GENRE — 端到端生成文档标识符
2. **效率-效果 Pareto 分析**：量化各架构在不同资源约束下的最优前沿
3. **混合检索趋势**：分析稀疏+密集混合检索的协同效应
4. **LLM 嵌入趋势**：总结 LLM-based 嵌入模型（E5-Mistral、GTE-Qwen）的架构特点

## 实验结论

- 混合检索（稀疏+密集）在 BEIR 上平均比单一检索高 5-8%
- 后期交互在精度和效率的 Pareto 前沿上占优
- LLM-based 嵌入模型在跨域零样本检索上显著优于传统双塔

## 工程落地要点

- **架构选择指南**：低延迟优先选稀疏/密集，高精度优先选交叉编码器，平衡选后期交互
- **混合检索**：建议稀疏+密集混合作为默认 baseline
- **索引成本**：密集检索需要 ANN 索引（FAISS/Milvus），考虑存储和构建成本

## 面试考点

1. **Q：对比稀疏检索和密集检索的优劣？**
   A：稀疏检索基于词匹配，可解释性强、精确匹配好，但语义理解弱；密集检索基于语义嵌入，语义匹配强但可能丢失精确匹配信号。
2. **Q：ColBERT 为什么叫"后期交互"？**
   A：query 和 document 分别编码为 token 级表征，交互（MaxSim）发生在编码之后，兼顾了独立编码的效率和 token 级交互的精度。
3. **Q：生成式检索的局限性？**
   A：文档集合变化时需要重新训练（无法增量更新），且对大规模文档集合的 Scaling 能力待验证。
