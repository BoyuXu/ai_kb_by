# LegalMALR: Multi-Agent Query Understanding LLM-Based Reranking for Chinese Statute Retrieval

> 来源：arXiv | 日期：20260317

## 问题定义

**中国法律条文检索**：给定法律查询（案情描述、法律问题），从数万条法律条文中精确检索相关法条。特殊挑战：
1. **专业术语**：法律语言与日常语言差异大，模型需要理解法律专业表达
2. **长文本法条**：部分法条含多款多项，结构复杂
3. **语义精确性**：法律语境中细微措辞差异意义迥异（"应当" vs "可以"）
4. **查询理解复杂性**：用户提供的案情描述需要转化为法律语言再匹配法条

## 核心方法与创新点

1. **多 Agent 查询理解（Multi-Agent QU）**
   - Agent 1（法律概念提取）：识别查询中的法律要素（主体、行为、法律关系）
   - Agent 2（案由分类）：判断案件类型（刑事/民事/行政），确定适用法律范围
   - Agent 3（查询扩展）：生成相关法律术语和同义表达，扩充检索覆盖
   - 多 Agent 输出融合为结构化查询表示

2. **LLM-Based Reranking**
   - 使用 Qwen/ChatGLM 等中文 LLM 作为重排器
   - Pointwise 模式：对每个（查询, 法条）对生成相关性分数
   - Listwise 模式：让 LLM 对 Top-K 法条直接排序（效果更好但计算慢）

3. **法律专用微调**
   - 构建法律案例-法条标注数据集（来自法院判决书）
   - 在通用 LLM 基础上做 SFT + DPO 对齐（正确法条 > 错误法条）

4. **层次化检索**
   - 先检索法律文本（刑法/民法/...）→ 再检索章节 → 再检索条款
   - 减少搜索空间，提升精度

## 实验结论

- 中国法律法条检索基准：MAP@10 提升约 8%（vs BM25 基线）
- 多 Agent QU 相比单一查询：Recall@20 提升约 12%
- LLM Listwise 重排相比 Cross-Encoder：MRR@5 提升约 3%，但延迟增加 5x

## 工程落地要点

1. **法律知识增强**：将法律知识图谱（罪名体系、法律关系）作为 LLM prompt 背景知识
2. **Multi-Agent 并行化**：三个 Agent 可并行执行，合并输出，控制总延迟
3. **法条更新**：法律修订时需及时更新索引，条文版本管理是工程难题
4. **用户隐私**：法律查询可能含敏感案情，需要私有化部署，避免调用外部 LLM API

## 面试考点

- **Q: 法律检索为什么比通用检索难？**
  A: 1) 词汇鸿沟：用户用口语描述，法条用专业法律语言；2) 精确性要求：法律语境中措辞差异意义大，模糊匹配可能引入错误法条；3) 长文本理解：法条含复杂条件逻辑（"如果...则...除非..."）；4) 领域专业性：通用模型缺乏法律知识。

- **Q: Multi-Agent 查询理解相比单一查询改写的优势？**
  A: 单一查询改写是"单视角"扩展，可能引入偏差；Multi-Agent 从多个专业角度（法律要素、案由、术语）分别处理，各 Agent 专注自身擅长的子任务，最终融合更全面。类似多专家会诊 vs 单一医生诊断。

- **Q: Listwise vs Pointwise LLM 重排的权衡？**
  A: Listwise 让 LLM 整体考虑候选集排序，可以比较候选间相对质量，效果更好；Pointwise 对每个候选独立打分，忽略候选间关系，但延迟低（可批量化）。实际中常用 Pointwise 配合大模型少量 Listwise 精排。
