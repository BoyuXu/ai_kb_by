# 向量检索基础设施全景 (2026-04-14)

> 基于4个工具/系统综合：FAISS, Milvus, Qdrant, OpenSearch LTR

## 1. 技术演进脉络

### ANN索引算法演进
- **IVF (倒排文件)**: 分区搜索，适合大规模数据
- **HNSW (分层导航小世界图)**: 高召回率，内存占用大
- **CAGRA (GPU图索引)**: NVIDIA cuVS专用，4.7x快于CPU HNSW

### 向量数据库代际
| 代际 | 代表 | 核心特性 |
|------|------|---------|
| 第一代 | FAISS (库) | 纯向量搜索，无持久化 |
| 第二代 | Milvus/Qdrant (数据库) | 持久化+过滤+水平扩展 |
| 第三代 | Milvus 2.6 | 自动embedding+混合检索+分析 |

## 2. 核心技术对比

| 特性 | FAISS | Milvus 2.6 | Qdrant | OpenSearch |
|------|-------|-----------|--------|-----------|
| 类型 | 库 | 数据库 | 数据库 | 搜索引擎+插件 |
| 语言 | C++/Python | Go/C++ | Rust | Java |
| GPU加速 | cuVS(8x) | 支持 | 无 | 无 |
| 稀疏向量 | 有限 | BM25 | 支持 | 原生BM25 |
| 混合检索 | 手动 | 原生 | 原生 | LTR插件 |
| 十亿级 | 支持 | 支持 | 支持 | 支持 |
| 部署 | 嵌入式 | K8s/Cloud | K8s/Cloud | K8s/Cloud |

## 3. 选型建议

- **纯性能优先**: FAISS + GPU (cuVS)
- **云原生全功能**: Milvus 2.6 (自动embedding, 混合检索)
- **Rust性能+简洁**: Qdrant
- **已有ES/OS生态**: OpenSearch + LTR插件
- **推荐系统场景**: FAISS做底层 + Milvus/Qdrant做服务层

## 4. 面试考点

1. **HNSW原理**: 多层跳表+贪心搜索，如何平衡精度和速度？
2. **量化方法**: PQ/SQ/OPQ各自特点和适用场景
3. **混合检索**: 稠密向量+稀疏向量(BM25)如何融合？RRF vs 加权?
4. **向量DB选型**: 给定场景如何选择？考虑QPS、延迟、数据规模、功能需求
5. **GPU加速**: cuVS在向量检索中的加速原理
