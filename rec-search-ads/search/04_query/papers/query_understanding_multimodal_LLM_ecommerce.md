# Query Understanding with Multimodal LLM for E-commerce Search
> 来源：工业论文 | 领域：搜索 | 学习日期：20260327

## 问题定义
电商搜索的Query理解面临：
1. **图文混合查询**：用户上传商品图片+文字描述进行搜索（"找和这个一样款式的但是蓝色的"）
2. **意图模糊**：同一个查询可能有多种意图（"苹果"→手机/水果/品牌）
3. **属性提取难**：从Query中提取结构化属性（颜色、尺寸、风格）用于过滤

**目标**：使用多模态LLM理解图文混合查询，提升搜索点击率+6%。

## 核心方法与创新点

### 1. 多模态Query表示
融合文本和图片Query：

$$
Q_{mm} = \text{CrossAttn}(\text{TextEncoder}(q_{text}), \text{VisionEncoder}(q_{image}))
$$

用Cross-Attention融合文本和图片特征，文本作为Query，图片作为Key/Value（用图片信息增强文本理解）。

### 2. 意图分类
对Query进行多标签意图分类：
- 导航类（找特定店铺/品牌）
- 信息类（了解某产品）
- 交易类（直接购买）
- 图片搜索类（以图搜图）

$$
P(\text{intent}) = \text{softmax}(W_i \cdot Q_{mm} + b_i)
$$

### 3. 结构化属性提取
从多模态Query中提取商品属性，用于精准过滤：
```
Query: "找这个款式的红色连衣裙"（附图）
→ 属性：{类目: 连衣裙, 颜色: 红色, 风格: [根据图片识别]}
```

LLM以结构化JSON格式输出属性，与商品库属性字段对齐。

### 4. 查询改写（Query Rewriting）
用LLM将模糊查询改写为精确查询：
```
原始: "便宜但好看的夏天裙子"
改写: ["连衣裙 夏季 轻薄", "雪纺连衣裙 夏", "夏季连衣裙 50元以内"]
```

多版本改写后做多路召回，合并结果。

## 实验结论
- **搜索点击率（CTR）**：+6%（相比纯文本Query理解）
- **图片搜索效果**：Recall@20 +25%（多模态融合vs纯图片）
- **属性过滤精度**：+15%（结构化属性提取比关键词匹配更准确）
- 意图分类准确率：92%（5类意图）

## 工程落地要点
1. **图片Query前处理**：用户上传图片质量参差不齐，需要自动去噪、裁剪、质量过滤
2. **延迟控制**：多模态LLM推理比纯文本慢3-5x，需要异步处理或模型轻量化
3. **属性本体（Ontology）对齐**：提取的属性名必须与商品库属性字段精确匹配（颜色:红 vs 颜色:红色vs Color:Red）
4. **隐私保护**：用户上传的图片可能包含隐私信息（人脸等），需要在模型推理前过滤
5. **冷启动**：新上线商品的图片属性需要离线预处理，确保多模态搜索能召回到

## 常见考点
Q1: 电商搜索的多模态Query理解与图片搜索（以图搜图）有什么区别？
A: 以图搜图：查询是纯图片，找视觉上相似的商品，侧重视觉特征匹配（颜色、形状、纹理）。多模态Query理解：图片+文字组合，需要理解用户意图中的文字修改部分（"一样款式但蓝色"），侧重语义理解。多模态理解更难：需要LLM理解"一样款式"指图片中的款式，"但蓝色"是文字修改条件，两者之间有对比关系。

Q2: Query Rewriting对搜索效果有什么风险？
A: (1)语义漂移：改写后的Query可能偏离用户原意（"便宜的手机"→"预算1000元的手机"，但用户可能指500元）；(2)召回过广：多路改写合并后候选集太大，排序压力增大；(3)召回偏窄：改写过于具体（"夏季雪纺连衣裙"），错过了某些合法结果。缓解方案：保留原始Query的召回路，与改写Query召回结果做RRF（Reciprocal Rank Fusion）融合。

Q3: 多模态LLM在工业搜索中的主要工程挑战？
A: (1)GPU资源：多模态推理需要大量GPU，成本是纯文本的3-5倍；(2)延迟：用户对搜索响应时间敏感（<200ms），多模态LLM难以满足；(3)图片存储：用户上传的查询图片临时存储和处理需要专门的基础设施；(4)安全检测：图片输入增加了内容安全风险（色情、违禁品），需要内容审核前置；(5)效果评估：多模态搜索的离线评估需要带图片标注的查询-文档对。
