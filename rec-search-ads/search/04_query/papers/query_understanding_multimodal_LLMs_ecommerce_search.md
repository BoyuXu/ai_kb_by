# Query Understanding with Multi-Modal LLMs for E-Commerce Search
> 来源：https://arxiv.org/abs/2601.xxxx（占位符URL，基于主题知识分析） | 领域：search | 学习日期：20260329

## 问题定义

电商搜索的 Query Understanding（QU）面临独特挑战：

1. **多模态查询的兴起**：用户不仅输入文字，也会上传商品图片（"找相似款"）或图文混合查询，传统纯文本 QU 无法处理
2. **电商语义复杂性**：电商查询短且歧义多（"苹果"可能是水果/手机/品牌）；专业术语多（SKU、SPU、品类、属性）
3. **意图理解的多粒度需求**：需要同时完成查询分类（品类路由）、属性抽取（颜色/尺码/材质）、意图识别（导航型/事务型/信息型）、查询改写（拼写纠错/同义词扩展）
4. **LLM 在 QU 中的应用挑战**：LLM 理解语义能力强，但推理速度慢，在搜索 QPS 要求下难以直接应用

**核心目标**：构建基于多模态 LLM 的电商搜索查询理解框架，提升搜索的语义理解能力，同时满足生产级延迟要求。

## 核心方法与创新点

### 1. 多模态 Query Encoder
- **视觉-语言融合**：将查询文本与用户上传的参考图片共同编码，通过多模态 LLM（如 LLaVA/InternVL 类架构）生成统一的查询 representation
- **图文跨模态对齐**：通过对比学习将文本查询与视觉 token 对齐，实现"图片找同款"和"文字+图片联合搜索"
- **轻量化蒸馏**：将大型 multimodal LLM 蒸馏为小型快速模型，满足在线推理延迟要求（< 50ms）

### 2. 多任务 QU 输出头
基于统一的多模态 backbone，同时预测：
- **品类分类**（Category Classification）：将查询路由到正确商品品类树节点
- **属性抽取**（Attribute Extraction）：提取颜色、尺码、品牌、材质等结构化属性
- **查询规范化**（Query Normalization）：统一别名和方言表达（"耐克" → "Nike"）
- **意图识别**（Intent Detection）：购买意向强弱分类

### 3. 电商知识注入
- **商品知识图谱增强**：将电商 knowledge graph（品牌-品类-属性关系）注入 LLM fine-tuning，提升专业术语理解
- **用户行为对齐**：用购买/点击日志中的用户行为信号作为额外监督，将"模型认为的相关性"与"用户实际行为"对齐
- **多语言/多地区适配**：在全球化电商平台上处理多语言查询和地域性表达差异

### 4. 推理优化架构
- **分阶段推理**：简单查询（关键词查询）用轻量规则/小模型处理；复杂/图文混合查询才调用多模态 LLM
- **预计算 Cache**：高频查询的 QU 结果缓存，减少重复计算
- **流水线并行**：QU 各子任务并行执行

## 工程落地要点

1. **分级处理架构**：99% 的简单文本查询用轻量 NLU 模型（< 5ms）；1% 的图文混合或复杂查询路由到多模态 LLM（20-50ms），避免全量 LLM 推理的成本
2. **蒸馏是关键**：将 7B+ 多模态 LLM 蒸馏到 300M-1B 小模型，保留 90%+ 的能力，推理速度提升 10-20x
3. **多任务学习降本**：一个模型同时处理分类/抽取/改写多个 QU 任务，避免多个独立模型的维护成本
4. **用户行为作为弱监督**：点击/购买日志是最易获取的对齐信号，无需昂贵的人工标注
5. **视觉 Token 压缩**：图片输入的 visual token 数量是推理成本主要来源，通过 token compression（如 token merging）减少视觉 token 到 64-128 个，大幅降低推理延迟
6. **商品品类树作为约束**：将品类分类的输出空间约束在已有品类树节点上，既提升准确率又保证输出合法性

## 常见考点

**Q1：电商搜索 QU 相比通用搜索 QU 有哪些独特挑战？**
> A：1) 查询极短且歧义多（"白色"可能是颜色属性，也可能是品牌）；2) 需要输出结构化属性（颜色/尺码/品牌），而非仅分类；3) 商品世界观（品类树/属性体系）是私有的，通用 LLM 没有这些知识；4) 视觉搜索占比高（以图搜图），必须处理多模态；5) 对延迟要求极高（电商搜索 P99 通常要求 < 100ms）。

**Q2：为什么要将 LLM 蒸馏为小模型而非直接用 LLM 做 QU？**
> A：电商搜索每秒处理数万次查询，7B LLM 单次推理需要 200-500ms 且 GPU 资源消耗巨大。蒸馏后的 300M-1B 模型推理时间 < 10ms，可以在 CPU/小 GPU 上运行，成本降低 10-50x，同时保留 90%+ 的性能。生产系统通常是蒸馏模型 + fallback 到大模型（仅用于复杂 case）的组合。

**Q3：如何用图文混合输入提升电商搜索的商品召回？**
> A：用户上传参考图片时，多模态 LLM 提取图片中的视觉属性（风格/颜色/款式）并与文字描述融合，生成联合 query embedding。检索时与商品图文 embedding 做近邻搜索。相比纯文字查询，多模态查询能更精准描述用户想要的具体款式，特别是在服装、家居等视觉依赖性强的品类中召回准确率提升显著。

**Q4：如何设计多模态 QU 的训练数据？**
> A：1) 图文对构造：从商品详情页抓取"商品图片 + 商品名/属性"对作为正样本；2) 用户搜索日志：用户上传图片后的点击/购买序列作为监督信号；3) 合成数据：用文生图模型（如 DALL-E/Stable Diffusion）根据查询文本生成参考图片；4) 电商 Knowledge Graph：品类-属性-品牌关系图谱用于 SFT 训练中的知识注入；5) 人工标注少量高质量数据用于 QU 子任务（分类/抽取）的微调。

**Q5：如何衡量 QU 模块的改进对下游搜索整体效果的贡献？**
> A：1) 分层分析：分别衡量 QU 在品类分类/属性抽取/意图识别上的准确率；2) 组件分析：单独关闭 QU 某个子模块，观察下游检索指标（NDCG/MAP/Recall@k）的变化；3) 在线 A/B 实验：最终 ground truth 是 GMV/CTR/用户满意度等业务指标，通过严格 A/B 实验验证，避免 "metrics 提升但 GMV 没变" 的情况；4) 覆盖率分析：QU 改进对哪类查询（长尾/新兴/图文）的覆盖率有最大提升。
