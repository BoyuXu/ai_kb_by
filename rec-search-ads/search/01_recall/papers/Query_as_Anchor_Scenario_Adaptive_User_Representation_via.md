# Query as Anchor: Scenario-Adaptive User Representation via Large Language Model for Search

> 来源：arxiv | 日期：20260321 | 领域：search

## 问题定义

搜索系统中的个性化依赖用户历史行为构建用户表示（User Representation）。传统方法面临的核心问题：

1. **场景不自适应**：同一用户在不同搜索场景（工作搜索 vs 娱乐搜索）下需求截然不同，但固定的用户 embedding 无法捕获这种动态性
2. **历史行为噪声**：用户历史包含大量无关行为，简单 Pooling 会引入噪声
3. **跨领域泛化差**：用户在 A 领域的行为对 B 领域的搜索几乎没有贡献（协同信号弱）

"Query as Anchor"的核心洞察：当前搜索查询是理解用户在此刻此场景下需求的最强信号，应以查询为锚点来选择和权衡历史行为。

## 核心方法与创新点

1. **查询锚定的历史注意力**：
   - 用查询 embedding 作为 Query，历史行为 embedding 作为 Key/Value
   - Attention score 反映历史行为与当前查询的相关性
   - 公式：`user_repr = Softmax(Q_query × K_history^T / √d) × V_history`
   - 与当前查询无关的历史行为被自动降权

2. **LLM 增强的场景理解**：
   - 用 LLM 分析查询所属场景（购物/工作/娱乐/学习）
   - 根据场景类型激活不同的历史行为子集（场景过滤）
   - LLM 作为 zero-shot 场景分类器，无需额外标注

3. **多粒度用户表示**：
   - 会话级表示：当前 Session 的行为（捕获即时意图）
   - 中期表示：近 30 天行为（捕获阶段性偏好）
   - 长期表示：全量历史（捕获稳定偏好）
   - 三层表示通过查询锚定注意力动态加权融合

4. **轻量化在线推理**：
   - LLM 推理离线完成（对查询进行场景分类，结果缓存）
   - 在线只做轻量 Attention 计算

## 实验结论

- 在电商搜索数据集上：相比 BERT 双塔模型，NDCG@10 **+6.8%**，MRR **+5.3%**
- 冷启动用户（历史 <5 次）：查询锚定方法比全局偏好表示 **+12%**（查询本身信号更强）
- 场景自适应相比单一用户 embedding：跨场景准确率 **+9.1%**（工作/娱乐区分正确率提升）
- 在线 A/B：搜索 CTR **+3.2%**，转化率 **+2.7%**

## 工程落地要点

1. **查询缓存**：高频查询（"手机"、"裙子"）的场景分类结果可预计算并缓存，减少 LLM 推理量
2. **注意力计算效率**：用户历史可能很长，需要预先用 ANN 对历史行为做粗召回（Top-100 相关历史），再做精细注意力
3. **场景分类的误分类处理**：LLM 偶尔场景分类错误，需要监控场景分布并设置兜底（分类置信度低时用全历史）
4. **历史行为存储**：按场景分类预存储历史行为（按场景建索引），在线快速按场景检索相关历史

## 常见考点

- Q: 搜索个性化和推荐个性化有什么区别？
  A: 推荐是主动发现（用户无明确意图），个性化权重更高；搜索有明确查询意图，相关性是第一优先级，个性化是在相关性满足的基础上的排序调整（相关且符合偏好 > 相关但不符合偏好 > 不相关）。过度个性化搜索会产生"回音壁"问题。

- Q: Query as Anchor 方法相比传统用户 embedding 的优势？
  A: 传统用户 embedding 是静态的（或只做简单聚合），无法随查询变化动态调整关注的历史。Query as Anchor 用查询作为动态过滤器，从历史中"挑"出与当前需求相关的部分，是一种 context-aware 的个性化方式，更符合人类真实的搜索行为。

- Q: 如何在搜索系统中处理冷启动用户的个性化？
  A: (1) 查询本身作为主信号（无需历史）；(2) 人口统计学特征（年龄/性别/地区）的群体画像；(3) 当前 Session 内的行为迁移（Session-aware 个性化）；(4) 上下文特征（时间、设备、位置）；(5) 兴趣问卷主动采集（新用户引导）。
