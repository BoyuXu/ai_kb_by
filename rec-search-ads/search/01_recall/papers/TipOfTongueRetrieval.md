# 单轮 LLM 改写驱动的多阶段混合重排序：舌尖上的检索

> 来源：Single-Turn LLM Reformulation Powered Multi-Stage Hybrid Re-Ranking for Tip-of-the-Tongue Retrieval | 日期：20260318 | 领域：search

## 问题定义

**Tip-of-the-Tongue (TOT) 检索**：用户知道想找什么，但无法准确表达，只能用模糊描述（"那个 90 年代的电影，主角是个超级英雄，但没有超能力，用蝙蝠作为符号…"）。与普通搜索不同，TOT 查询：
- 语义模糊、描述冗长、关键词稀疏。
- 需要跨模态理解（描述人物、情节、风格）。
- 传统 BM25 关键词匹配效果差；标准 dense retrieval 也因查询噪声而表现不佳。

## 核心方法与创新点

1. **单轮 LLM 查询改写**：用 LLM（GPT-4/Llama）将用户的模糊描述改写为多个高质量检索查询：
   - 精确查询（直接提取关键词：蝙蝠侠, Batman）
   - 语义扩展查询（超级英雄, 暗黑风格, 哥谭市）
   - 否定查询（排除无关候选）
   - 仅需一轮 LLM 调用（单轮），避免多轮对话的延迟和错误累积。

2. **多阶段混合检索**：
   - 稀疏检索（BM25）：处理精确查询，高 Recall。
   - 稠密检索（bi-encoder）：处理语义扩展查询，捕捉语义相似性。
   - 多查询结果 RRF（Reciprocal Rank Fusion）合并：简单有效的无参数融合。

3. **跨编码器重排序**：将 RRF 合并后的 top-100 候选用 cross-encoder 精排，综合原始查询 + 所有改写查询的信息对候选打分。

## 实验结论

- 在 TREC 2023 TOT Track 数据集，MRR@10 从 BM25 基线的 0.31 提升到 0.58（+87%）。
- 单轮 LLM 改写 vs 多轮对话改写：效果相近（差约 2%），但速度快约 4x。
- RRF 融合 vs Learned Fusion：RRF 在 TOT 场景效果更稳定（Learned Fusion 容易过拟合到域内数据）。

## 工程落地要点

- **改写 Prompt 设计**：Prompt 需引导 LLM 生成多样化查询（不同粒度、正/负样本），避免所有改写语义相似（退化为单查询）。
- **查询数量控制**：LLM 改写生成 5-10 个查询为宜，过多导致计算成本上升和噪声增加。
- **RRF 参数**：RRF 中的 k 参数（默认 60）对 TOT 场景相对不敏感，无需大量调参。
- **LLM 调用缓存**：相同或相似的 TOT 查询可以缓存 LLM 改写结果（语义哈希 key），降低 API 成本。

## 常见考点

**Q: 什么是 TOT 检索？与普通搜索的本质区别？**
- TOT：用户明确知道目标但无法准确表达，需要系统"猜"出用户意图；普通搜索：用户有明确关键词。TOT 更依赖语义理解和跨模态联想。

**Q: 查询改写（Query Reformulation）有哪些方法？**
- 规则/模板：同义词替换、查询扩展（Word2Vec 近邻）；模型：seq2seq 改写模型（T5 微调）；LLM：few-shot/zero-shot prompt 生成多样化查询。

**Q: RRF（Reciprocal Rank Fusion）的原理？**
- 对每个文档，计算其在各个排名列表中的排名倒数之和：`score = Σ 1/(k + rank_i)`，k=60 避免头部文档权重过大；无需学习参数，对多路检索融合稳定有效。
