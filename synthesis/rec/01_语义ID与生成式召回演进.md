## 一、召回范式演进全景

```
CF (ItemCF/UserCF)
  → MF (矩阵分解)
    → DSSM/双塔 (深度语义匹配)
      → 图召回 (PinSage/GraphSAGE/EGES)
        → 序列召回 (GRU4Rec/SASRec/BERT4Rec)
          → 生成式召回 (TIGER/GLIDE/OneRec)

物品表示演进:
Random ID → Learned Embedding → Discrete Semantic ID (RQ-VAE) → Soft ID (UniGRec)
```

### 关键转折点

| 阶段 | 代表工作 | 核心变化 |
|------|---------|---------|
| 协同过滤 | ItemCF/UserCF | 基于共现统计 |
| 向量化 | MF/DSSM | 隐向量内积匹配 |
| 双塔工业化 | YouTube DNN/DSSM | User+Item 独立编码，ANN 检索 |
| 图增强 | PinSage/EGES | 引入图结构关系 |
| 序列建模 | SASRec/SIM | 用户行为时序建模 |
| **生成式召回** | TIGER/GLIDE/OneRec | 自回归生成物品 token，统一召回与排序 |

## 二、Semantic ID 核心技术

### 2.1 RQ-VAE 编码原理

Semantic ID 的核心是 Residual Quantization（残差量化），将物品 embedding 编码为多层离散 token：

$$
\text{SemanticID}(x) = (c_1, c_2, \ldots, c_L), \quad c_l = \arg\min_{j \in [K]} \|\mathbf{r}_{l-1} - \mathbf{e}_j^{(l)}\|^2
$$

残差更新：$\mathbf{r}_l = \mathbf{r}_{l-1} - \mathbf{e}_{c_l}^{(l)}$

**层次化语义**：第1层编码粗粒度类别（如"电子产品"），后续层逐步细化（如"手机→iPhone→iPhone 16 Pro"）。相似物品共享 SID 前缀，支持 beam search 高效生成。

### 2.2 SID 设计演进

| 方案 | 代表工作 | 优势 | 劣势 |
|------|---------|------|------|
| 原始 ID (Random Int) | DLRM | 简单 | 无语义，冷启动差 |
| Learned Embedding | SASRec/DIN | 端到端学习 | 无层次结构 |
| 离散 SID (RQ-VAE) | TIGER, LC-Rec | 有语义层次，支持生成 | 量化误差，碰撞 |
| SID + Reasoning | Reasoning SID | 推理增强，冷启动 +7.2% | 推理 token 增加延迟 |
| 变长 SID | Variable-Length SID | 信息密度自适应 | 实现复杂 |
| Soft ID (连续) | UniGRec | 无量化误差，碰撞率 0% | 存储开销大 (10-50GB) |
| Prefix N-gram | Meta Ads | 高效前缀匹配 | 精度略低 |

### 2.3 四类特征标记方法（2025中文综述）

| 标记类型 | 代表方法 | 适用场景 |
|---------|---------|---------|
| 语义描述 | P5 | 冷启动、可解释性 |
| 向量量化 (VQ) | VQ-Rec | 计算受限场景 |
| 残差量化 (RQ) | TIGER/LC-Rec | 大规模系统（工业主流） |
| LLM 标记 | IDGenRec | 跨域迁移、零样本泛化 |

RQ 是工业最主流选择——Hash 冲突率降至 7.86%，平衡精度与效率。

## 三、生成式召回核心架构

### 3.1 生成式推荐基本范式

$$
P(\text{item} | \text{user history}) = \prod_{l=1}^{L} P(\text{SID}_l | \text{SID}_{<l}, \text{user history})
$$

用户历史行为构成输入序列，模型自回归生成下一步推荐物品的 Semantic ID token。

### 3.2 HSTU — 生成式推荐奠基工作 (Meta)

Pointwise Aggregated Attention（去掉 softmax，用 silu 替代）：

$$
\mathbf{O} = \text{silu}(Q) \cdot \text{silu}(K)^\top \cdot V \cdot M_{\text{causal}}
$$

- 避免全局归一化，GPU 矩阵乘法直接实现，长序列加速 3-5 倍
- 首次验证推荐领域 Scaling Law：1.5B → 1.5T 参数性能持续提升未饱和
- 统一召回与排序为单一生成式模型

### 3.3 GLIDE/Spotify — 工业部署标杆

首个在 Spotify 生产环境部署的 SID-based 生成式检索系统：
- **Catalog Grounding**：避免幻觉，保证生成 ID 是真实 podcast
- **用户级个性化**：soft prompt 注入长期偏好
- **实验结果**：百万用户 A/B 测试，非习惯性播客 +5.4%，新节目发现 +14.3%

### 3.4 OneRec-V2 — Lazy Decoder (快手)

OneRec-V1 中 97.66% 计算消耗在上下文编码。Lazy Decoder 改进：

$$
\text{LazyBlock}(x) = \text{FFN}(\text{CausalSelfAttn}(\text{CrossAttn}(x, K_{\text{static}}, V_{\text{static}})))
$$

上下文一次性处理为静态 KV pair，decoder 惰性复用。计算减 94%，成功 scale 到 8B 参数。A/B 测试：App Stay Time +0.467%。

### 3.5 NEO — 统一搜推理 (Spotify)

$$
P(\text{next\_token} | \text{context}) = \text{softmax}(\mathbf{h} \cdot \mathbf{E}_{\text{valid}}^T)
$$

Constrained decoding 限制 token 空间为 catalog 有效子集。单模型端到端完成搜索+推荐+推理，10M+ item catalog。

### 3.6 UniGRec — 连续空间生成

用 Soft Identifiers（连续向量序列）替代离散 SID：
- 消除量化误差，碰撞率 0%（vs TIGER 3.2%）
- Recall@10 比 TIGER 提升 5.8%
- 挑战：千万级物品约 10-50GB 存储

### 4.2 十大工业召回经验

1. **微信看一看**：多兴趣召回 + 时效性衰减
2. **快手双塔**：负采样优化（混合负采样比随机负采样 +3.2%）
3. **Pinterest PinSage**：图卷积 + Random Walk 采样，20亿 pin 上线
4. **YouTube DNN**：候选生成 + 排序两阶段，Example Age 特征消除分布偏移
5. **EGES (阿里)**：Side Information 图嵌入，冷启动效果显著

7. **SIM (阿里)**：超长序列（万级行为）两阶段检索
8. **HNSW 向量检索**：召回率 >99%@Top100，延迟 <1ms
9. **分数校准**：多路召回分数对齐（温度缩放/Platt Scaling）
10. **在线 A/B**：多路召回比例动态调整

### 4.3 双塔模型四条改进道路

1. **特征交互增强**：塔内加深交叉（SENet/DSSM 变体）
2. **负样本优化**：困难负例、混合负采样、batch 内负采样
3. **多兴趣建模**：MIND（多头）/ ComiRec（动态路由）
4. **对齐与蒸馏**：精排模型蒸馏召回模型，减少级间 gap

## 五、生成式推荐跨域统一视角

### 5.1 推荐与广告的技术收敛

| 维度 | 推荐系统 | 广告系统 | 共同趋势 |
|------|---------|---------|---------|
| 核心范式 | HSTU → 生成式推荐 | EGA-v2/GPR → 生成式广告 | 判别式→生成式 |
| 物品表示 | Semantic ID (RQ-VAE) | Semantic ID (RQ codes) | 离散化 token |
| 模型架构 | Transformer decoder | Transformer decoder | 自回归生成 |
| 扩展范式 | PROMISE test-time scaling | GRAD diffusion 出价 | 生成+概率建模 |

### 5.2 PROMISE — Test-Time Scaling

$$
\hat{y} = \arg\max_{y^{(i)}, i \in [N]} \sum_{t=1}^{T} \text{PRM}(y_{<t}^{(i)}, y_t^{(i)}, c)
$$

推理阶段生成多个候选推荐列表，用 Process Reward Model 逐步评估。Best-of-64 策略 Recall@10 提升 9.1%。

### 5.3 Agentic RecSys — 未来方向

从固定 pipeline 到自适应推荐 Agent：
- **感知**：MTGR 自适应 gating 根据用户状态调整
- **规划**：Semantic ID Reasoning 的推理机制
- **反思**：PROMISE 的 PRM 评估推荐质量
- **核心瓶颈**：LLM 推理延迟 100ms-1s vs 推荐 SLA <10ms

## 六、面试高频考点

**Q1: Semantic ID 解决了什么问题？为什么不用原始 ID？**
A: 原始 ID 无语义，无法泛化到未见物品。SID 通过 RQ-VAE 编码实现：(1) 层次化语义（第1层=大类，后续=细粒度）；(2) 相似物品共享前缀；(3) 支持自回归生成。代价是需要 RQ-VAE 预训练且码本需定期重建。

**Q2: 生成式推荐 vs 传统推荐的根本区别？**
A: 传统方法是 ranking/retrieval（$P(y|\mathbf{x}, \text{item})$），给定候选打分。GenRec 是序列生成（$P(\text{item}|\mathbf{x})$），直接生成推荐。优势：统一多阶段 pipeline，消除信息损失。代价：推理延迟更高。

**Q3: 离散 Semantic ID vs Soft ID，如何选？**
A: 物品库 <1M 且追求极致精度：Soft ID（碰撞率 0%，Recall +5.8%）。物品库 >10M 且需要高效搜索：离散 SID（与 beam search 天然兼容）。未来可能出现混合方案：粗层离散做高效搜索，细层连续做精确匹配。

**Q4: 如何避免生成式推荐的 hallucination？**
A: (1) Constrained Decoding（NEO）：限制 token 空间为 catalog 有效子集；(2) Semantic ID Grounding（GLIDE）：所有生成映射到已注册 SID；(3) ANN Verification：embedding 空间验证物品存在性。

**Q5: 冷启动问题在生成式推荐中为什么更复杂？**
A: Item cold-start 需要新 item 的 SID（码本中无对应编码），User cold-start 缺历史序列无法自回归生成。关键发现（Cold-Start GenRec）：标识符设计是关键，非模型规模。文本 SID 改善 item 冷启但损害其他场景，分层语义 SID 更鲁棒。

**Q6: HSTU 的 Pointwise Aggregated Attention 为什么优于 softmax？**
A: 效率层面：去掉 softmax 后可分解为矩阵乘法链，推理加速 3-5 倍。效果层面：推荐行为 token 分布与自然语言不同——异构性高且稀疏，softmax 归一化反而抹平重要信号差异。

**Q7: 统一搜推理模型（NEO）的核心技术挑战？**
A: (1) 异构任务 loss 平衡；(2) SID 与自然语言的 modality alignment；(3) 10M+ item 的 constrained decoding 效率；(4) 生产环境延迟约束。

**Q8: 双塔模型的核心局限？**
A: User 和 Item 在最后才做内积交互，无法建模细粒度特征交叉。解法：(1) 塔内加深交叉；(2) 多向量表示；(3) 蒸馏精排知识。
