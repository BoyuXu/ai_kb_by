# 异质感知预排序：Not All Candidates are Created Equal

> 来源：Not All Candidates are Created Equal: Heterogeneity-Aware Pre-ranking in Recommender Systems | 日期：20260318 | 领域：rec-sys

## 问题定义

在工业级推荐系统的多阶段漏斗中，预排序（Pre-ranking）处于召回之后、精排之前，需要从数千候选中快速筛选出数百条。传统预排序把所有候选统一用轻量模型打分，但候选来自不同召回源（协同过滤、语义向量、规则等），其质量分布存在显著异质性。高质量候选被轻量打分低估，导致精排曝光机会浪费；低质量候选占坑，系统整体效率下降。

## 核心方法与创新点

1. **候选异质性建模**：将候选按来源（召回通道）分组，为不同组设计差异化的特征空间和打分头（group-specific scoring head），允许模型对不同质量分布的候选采用不同决策边界。

2. **双塔 + 轻量交叉**：主干仍是向量化双塔（保证推理速度），在 top-K 候选上追加一个轻量交叉网络对高潜力候选做精细化评估，而不对全量候选做昂贵交叉。

3. **质量感知的截断策略**：将 quota 分配从均匀截断改为按召回源质量动态分配，确保高质量通道候选不被低质量通道稀释。

4. **训练目标改造**：引入 group-aware softmax，对组内候选做对比学习，避免跨组负样本干扰。

## 实验结论

- 在内部大规模推荐平台上，相比统一打分基线，CTR +2.1%，GMV +1.4%（A/B test）。
- NDCG@50 在离线评估提升约 3.2%，候选质量方差显著降低。
- 计算延迟增加 < 5ms（通过只对 top-K 做交叉实现），满足预排序延迟要求。

## 工程落地要点

- **召回通道标记**：需在候选传递时携带来源标签，否则无法区分组；下游系统改造成本需评估。
- **配额动态调整**：各通道配额需定期根据线上反馈重新标定，建议每周更新一次。
- **冷启动**：新召回通道上线时缺少历史分布，可先用最相似的已有通道初始化，逐步迁移。
- **特征一致性**：预排序与精排共享 user/item embedding，更新频率需对齐，避免特征穿越。

## 面试考点

**Q: 预排序和精排的核心区别是什么？为什么需要预排序？**
- 精排模型重、特征复杂，只能处理数百条；预排序用轻量模型处理数千条，两者是速度-精度的 trade-off。

**Q: 候选异质性问题怎么解决？**
- 按来源分组，用 group-specific head 和 group-aware 对比学习分别建模不同质量分布。

**Q: 双塔模型如何兼顾速度和效果？**
- User/Item 独立 forward，离线缓存 Item embedding，在线只做 User embedding + dot product，复杂度 O(N)；在 top-K 上追加交叉层在不破坏整体延迟的前提下提升精度。
