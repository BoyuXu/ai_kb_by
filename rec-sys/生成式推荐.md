## 核心框架

### 生成式推荐 vs 判别式推荐

```
判别式推荐（传统）：
  给定 user + item → 预测 score → ranking

生成式推荐（GR）：
  给定 user context → 直接生成 item_token 序列 → 召回结果
```

### GR 技术路线

```
1. Item ID 化（量化）：
   item embedding → 离散 semantic tokens (RQ/VQ/Codebook)
   
2. 序列生成模型：
   user context → Transformer Decoder → 自回归生成 token 序列 → item
   
3. 推理加速：
   Beam Search + KV Cache + 商业约束剪枝
```

## 关键技术组件

### 1. 量化方法

| 方法 | 原理 | 优缺点 |
|------|------|--------|
| VQ (Vector Quantization) | 单层量化，距离最近的 codebook vector | 简单，但精度损失大 |
| RQ (Residual Quantization) | 多层残差量化，逐层量化残差 | 精度高，支持层次检索 |
| PQ (Product Quantization) | 分段量化 | 向量压缩，不适合生成 |

## 工程落地要点

### 在线推理架构

```
用户请求（含行为序列）
      ↓
Token 化（离线预计算 item tokens）
      ↓
Transformer Decoder 自回归生成（KV Cache 加速）
      ↓
Beam Search（beam_size=20~50）
      ↓
候选 item 列表（召回结果）
      ↓
精排/重排
```

### 系统挑战

1. **延迟**：自回归生成的 token 步数 × beam size × Transformer 层数，延迟比双塔高 3~5x
2. **Codebook 版本管理**：item token 更新需要版本一致性
3. **商业目标**：纯 CTR 优化的 GR 可能牺牲 eCPM，需要在 beam search 中引入商业约束

## 2026-03-12 更新

### 本次新增内容

**1. GR 正式进入工业广告系统**
- Generative Recommendation for Large-Scale Advertising（2026-03-04）
- 亿级广告库 GR 的工程化方案：分布式 RQ 量化 + eCPM beam search
- 关键成果：RPM +2.3%，P99 延迟 < 30ms

**2. COBRA：稀疏+稠密级联生成**
- arXiv 2503.02453（2026-03-04）
- 创新：先生成稀疏 semantic ID，再以其为条件生成稠密向量（BeamFusion）
- 解决了 GR 信息损失问题，同时保留了精确向量检索的优势

**3. S-GRec：LLM 语义监督 GR**
- arXiv 2502.XXXXX（2026-02-25）
- 核心：用 LLM 的语义知识提供更强的监督信号，改善行为日志弱监督的 GR 问题
- 适合 UGC 平台（内容语义丰富，行为日志质量参差不齐）

### 技术趋势判断

**2026 年 GR 的核心演进方向：**
1. 从召回扩展到精排（One-Stage GR）
2. LLM 语义 + 行为协同信号深度融合
3. 广告商业目标约束的端到端优化
