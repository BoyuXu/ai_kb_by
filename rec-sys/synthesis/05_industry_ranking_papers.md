## 1. 京东推荐算法精排技术实践

### 📌 核心贡献

京东精排实践系统阐述了电商场景下从特征工程到在线服务的全链路优化方案。核心创新在于**多场景共享底座+场景适配器**架构，既保证了跨场景迁移学习，又兼顾了各场景的特异性。同时提出用户近30分钟实时行为聚合的意图感知机制。

### 💡 技术亮点

| 创新点 | 说明 |
|--------|------|
| **多场景共享底座** | 首页、详情页、购物车共享底层Embedding，上层场景适配器学习场景特异性 |
| **实时意图特征** | Flink实时聚合近30分钟浏览/点击/加购行为，刻画即时购物意图 |
| **价格敏感度建模** | 用户历史促销响应率作为特征，区分价格敏感/品质优先用户 |
| **序列长度自适应** | 根据用户活跃度动态调整序列长度（活跃用户200+，沉默用户50） |

### 🔧 工程落地难点

| 难点 | 解决方案 |
|------|----------|
| **实时特征一致性** | 训练时回溯Flink状态，确保训练/在线特征来源一致 |
| **多场景样本不均衡** | 首页样本量是详情页10倍+，采用按场景采样率动态调整 |
| **冷启动用户意图** | 相似用户意图迁移（基于User2Vec的最近邻） |
| **大促期间分布漂移** | 大促前一周启动模型热更新，双模型并行A/B测试 |

### 🎤 面试引用

> "在电商精排中，我认为最关键的是**用户实时意图的捕捉**。参考京东的实践，我们通过Flink实时聚合用户近30分钟的行为作为动态特征。同时采用多场景共享底座+场景适配器的架构，既保证了跨场景迁移学习，又兼顾了各场景的特异性。"

**面试追问应对：**
- Q: 多场景如何防止负迁移？
- A: 通过场景适配器只学习上层的场景特异性，底层Embedding共享保证迁移；同时不同场景的损失按样本量加权。

## 2. 贝壳CVR转化率预估模型实践

### 📌 核心贡献

贝壳CVR实践针对房产领域**转化周期长、样本稀疏、正负样本极度不平衡**的特点，提出了延迟反馈建模与ESMM结合的方案。核心创新是将CVR分解为「带看率×带看后转化率」，通过中间目标增加监督信号，解决了延迟归因难题。

### 💡 技术亮点

| 创新点 | 说明 |
|--------|------|
| **ESMM + 延迟反馈** | 将点击到转化的延迟时间分桶建模，辅助CVR任务学习 |
| **房源属性继承** | 用户浏览房源A后成交房源B，房源A的特征作为正样本学习 |
| **多目标分解** | CVR分解为「带看率 × 带看后转化率」，降低预估难度 |
| **房源生命周期** | 新上房源、在售房源、下架房源分别用不同塔学习 |

### 🔧 工程落地难点

| 难点 | 解决方案 |
|------|----------|
| **转化延迟长达数月** | 点击后7天内成交的作为确定正样本，7天后成交的按延迟概率加权 |
| **假阴性样本多** | 引入Fake Negative Detection模块，识别高概率延迟转化样本 |
| **房源A浏览B成交** | 浏览房源的特征作为辅助监督信号，通过属性继承网络传递 |
| **成交样本极少** | 带看作为中间目标增加监督信号，采用GradNorm平衡多任务损失 |

### 🎤 面试引用

> "房产CVR的核心难点是**转化周期长导致的延迟反馈问题**。我们参考贝壳的方案，一方面使用ESMM架构共享点击任务的表征，缓解转化样本稀疏；另一方面对延迟时间进行分桶建模，调整CVR预估。同时把CVR分解为「带看率×带看后转化率」，通过中间目标增加监督信号。"

**面试追问应对：**
- Q: 延迟反馈和样本选择偏差(SSB)有什么区别？
- A: SSB是样本空间问题（只观察到曝光样本），延迟反馈是时间窗口问题（转化未发生时点击被误判为负样本）。两者常同时存在，需要分别处理。

## 3. 张俊林：推荐系统排序Embedding建模

### 📌 核心贡献

张俊林老师从工业视角系统梳理了推荐排序中Embedding的核心作用，提出**"Embedding是推荐系统的血液"**这一观点。核心贡献在于总结了千万级到亿级ID特征的Embedding设计范式，以及序列特征从Mean Pooling到Multi-Interest的演进路径。

### 💡 技术亮点

| 创新点 | 说明 |
|--------|------|
| **ID特征Embedding压缩** | 千万级ID用Hash+冲突处理，亿级ID用高频单独Embedding+低频共享 |
| **多值特征Pooling策略** | 用户点击类目序列：Mean Pooling → Attention Pooling → Multi-Interest |
| **Embedding维度自适应** | 高频特征用高维(64/128)，低频特征用低维(8/16)，总参数量可控 |
| **预训练Embedding迁移** | Item2Vec预训练作为初始化，Fine-tune时学习率差异化 |

### 🔧 工程落地难点

| 难点 | 解决方案 |
|------|----------|
| **Embedding表过大** | 高频单独Embedding + 低频Hash共享，冲突特征辅助区分 |
| **Embedding更新延迟** | 天级全量更新 + 小时级增量更新热加载 |
| **冷启动ID无Embedding** | 用属性均值初始化（新品用同类目均值） |
| **多值特征变长** | Padding + Mask + 实际长度输入，避免Padding干扰 |

### 🎤 面试引用

> "Embedding是推荐系统的血液，它的质量直接决定了模型天花板。在工业实践中我关注三点：一是**高频低频分离**，避免长尾ID浪费参数；二是**序列特征多兴趣建模**，单一向量难以表达用户多面性；三是**预训练+微调**，用Item2Vec等先验知识加速收敛。"

**面试追问应对：**
- Q: Hash冲突怎么处理？
- A: 高频ID单独保留，低频ID Hash共享；同时加入冲突标识特征（如Hash前的ID签名），让模型学习区分冲突ID。

## 4. 快手KDD2022 D2Q观看时长预估

### 📌 核心贡献

快手D2Q(Discrete Distributed Quality)提出将**观看时长预估转化为分桶分类问题**，解决了传统回归方法对长视频预估不准、离线在线指标不一致的难题。核心创新在于Ordinal Regression建模桶间顺序关系，并通过加权损失反映业务对不同时长区间的偏好。

### 💡 技术亮点

| 创新点 | 说明 |
|--------|------|
| **时长分桶离散化** | 将连续时长映射到K个桶(0-3s, 3-10s, 10-30s...)，转为多分类 |
| **Ordinal Regression** | 桶间有顺序关系，用累积概率建模 P(时长>桶k边界) |
| **Weighted LogLoss** | 不同桶赋予不同权重，反映业务对不同时长区间的偏好 |
| **Position Debias** | 引入位置作为特征，同时用IPW(逆概率加权)消除位置偏置 |

```
┌─────────────────────────────────────────────────────────────────────┐
│                    D2Q 观看时长预估框架                               │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  传统回归方案的问题                                                   │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │  MSE Loss: 对长尾分布不友好，长视频梯度小，难以学习           │ │
│  │  MAE Loss: 梯度不平滑，训练不稳定                             │ │
│  │  实际业务关心: 是否能看完、完播率、平均观看时长                 │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                      │
│  D2Q: 离散化分桶 + Ordinal Regression                                │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                               │ │
│  │  原始时长: 45秒                                               │ │
│  │      │                                                        │ │
│  │      ▼                                                        │ │
│  │  分桶: [0-3s] [3-10s] [10-30s] [30-60s] [60s+]                │ │
│  │          0      0       0        1       1    (累积标签)      │ │
│  │                                                               │ │
│  │  模型输出: P(>3s)  P(>10s)  P(>30s)  P(>60s)                  │ │
│  │              │        │        │        │                     │ │
│  │              └────────┴────────┴────────┘                     │ │
│  │                         │                                     │ │
│  │                         ▼                                     │ │
│  │              期望时长 = Σ P(>桶k) × 桶k中位数                 │ │
│  │                                                               │ │
│  │  损失函数: Weighted Cross Entropy (高价值桶权重更高)          │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                      │
│  Position Debias (位置去偏)                                          │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │                                                               │ │
│  │  问题: 位置1的视频天然CTR高，不一定是质量好                   │ │
│  │                                                               │ │
│  │  方案1: 位置作为特征输入 (模型学习位置效应)                   │ │
│  │  方案2: IPW (Inverse Propensity Weighting)                    │ │
│  │         - 估计P(点击|曝光,位置)                               │ │
│  │         - 样本权重 = 1 / P(点击|位置)                         │ │
│  │                                                               │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

### 🔧 工程落地难点

| 难点 | 解决方案 |
|------|----------|
| **分桶边界选择** | 按业务目标分桶(如3s完播、完播率)，非均匀划分 |
| **长短视频不平衡** | 加权损失，长视频桶权重更高，或单独建模 |
| **在线推理期望计算** | 预计算各桶边界对应时长，线上查表+加权求和 |
| **位置偏置随时间变化** | 定期重训练位置倾向模型，防止偏置估计过时 |

### 🎤 面试引用

> "在快手短视频场景，观看时长预估面临长尾分布问题。我们采用D2Q方案，将时长分桶转化为Ordinal Regression问题，预测累积概率P(时长>边界)。相比MSE回归，离散化方案更关注业务关心的完播节点，同时通过加权损失让模型更关注长视频质量。"

**面试追问应对：**
- Q: Ordinal Regression和普通多分类有什么区别？
- A: 普通多分类桶间独立，Ordinal Regression利用桶间顺序关系，预测累积概率P(>边界)，天然满足单调性约束，更符合时长预估的物理意义。

## 5. 阿里新一代Rank技术

### 📌 核心贡献

阿里新一代Rank技术体系涵盖了从特征、模型到训练范式的全面升级。核心创新在于提出**粗精排一致性优化**和**全域特征一致性**方案，通过精排蒸馏指导粗排、特征分布对齐等手段，解决了超大规模推荐场景中的目标不一致和分布漂移问题。

### 💡 技术亮点

| 创新点 | 说明 |
|--------|------|

| **因果推断去偏** | 用Causal Forest估计曝光到点击的因果效应，消除选择偏置 |

**面试追问应对：**
- Q: 因果推断去偏和传统的位置去偏有什么区别？
- A: 位置去偏处理的是展示位置带来的偏置，因果推断去偏处理的是曝光机制本身的选择偏置（用户只能看到系统决定曝光的内容）。两者互补，因果推断更全面但计算成本更高。

## 8. Life-long兴趣建模：SIM模型

### 📌 核心贡献

阿里SIM(Search-based Interest Model)提出了**两阶段长序列建模**范式，通过General Search Unit(GSU)快速筛选相关子序列，Exact Search Unit(ESU)精细化建模。核心贡献是首次实现了数万级行为序列的工业级落地，突破了DIN/DIEN的序列长度瓶颈。

### 💡 技术亮点

| 创新点 | 说明 |
|--------|------|
| **GSU硬搜索** | 按Category/Brand匹配筛选，O(N)复杂度，毫秒级响应 |
| **GSU软搜索** | 向量相似度Top-K检索(FAISS)，语义匹配更全面 |
| **ESU精确建模** | Multi-Head Attention建模筛选后子序列，复杂度O(m²) |
| **长短期分离** | 短期用DIN，长期用SIM，分别建模即时兴趣和长期偏好 |

**面试追问应对：**
- Q: Hard Search和Soft Search如何选择？
- A: 工业落地建议两者结合：Hard Search速度快做初筛，Soft Search语义召回更全面但依赖向量质量。也可根据候选物品属性是否完整动态选择。

## 9. TWIN双塔序列模型

### 📌 核心贡献

TWIN(TWo-view Interest Network)创新性地将**双塔结构与时序建模结合**，解决了粗排阶段无法使用序列特征的问题。核心贡献是通过分离短期和长期兴趣提取，在保持双塔在线效率的同时，引入了用户行为序列的丰富信号。

### 💡 技术亮点

| 创新点 | 说明 |
|--------|------|
| **双塔+序列融合** | 用户塔内部做序列建模，输出向量后仍可用ANN检索 |
| **短期DIN Attention** | 最近50个行为用候选物品做Attention，提取即时兴趣 |
| **长期Target Attention** | 更长序列用Target Attention聚合，关注与候选相关的历史 |
| **兴趣融合门控** | 学习短期/长期兴趣的融合权重，适应不同用户类型 |

## 10. COLD级联精排

### 📌 核心贡献

阿里COLD(Computing power cost-aware Online and Lightweight Deep ranking)提出了**面向算力感知的轻量级深度排序**。核心贡献是在粗排阶段使用类精排的网络结构(SENET特征交互+轻量DNN)，通过哈希技巧和控制复杂度，实现了粗排效果的显著提升。

### 💡 技术亮点

| 创新点 | 说明 |
|--------|------|
| **SENET特征重标定** | 用Squeeze-Excitation网络学习特征重要性，轻量级交叉 |
| **特征哈希降参** | Hash Trick将千万级ID映射到百万级Embedding表 |
| **算力感知的NAS** | 根据在线QPS约束，搜索最优网络结构 |
| **精排蒸馏指导** | 粗排学习精排的软标签，提升排序一致性 |

| **精排蒸馏稳定性** | 教师模型固定版本，防止教师频繁变化 |
| **粗排召回率** | 保持足够大的Top-K(如5000)，让精排有空间优化 |

**面试追问应对：**
- Q: COLD和双塔模型相比优势在哪？
- A: 双塔表达能力受限（只能内积），COLD引入SENET实现轻量级特征交叉，表达能力更强。同时保持可控的复杂度，通过NAS在效果和延迟间找到平衡点。

## 面试组合拳：精排问题应答策略

### 问题1：你们精排模型是怎么迭代的？

**答题框架：**
1. **基础建模**：DIN/DCN作为baseline，解决特征交叉和序列建模
2. **长序列优化**：引入SIM两阶段建模，从200扩展到5000+
3. **多目标优化**：MMOE/PLE处理CTR/CVR/停留时长多目标
4. **一致性优化**：粗精排蒸馏对齐，特征实时化保证训练在线一致

### 问题2：长序列建模怎么做工程落地？

**答题框架：**
1. **先讲问题**：DIN复杂度O(N²)，长序列不可行
2. **SIM方案**：GSU快速筛选+ESU精细建模，复杂度降到O(N)+O(m²)
3. **索引构建**：Hard Search用倒排，Soft Search用FAISS
4. **工程细节**：向量漂移监控、天级更新、冷启退化成Category匹配

### 问题3：粗排和精排如何保持一致性？

**答题框架：**
1. **问题定义**：粗排选出的Top-K，精排再排序，两级目标不一致导致信息损失
2. **特征层**：粗排分数作为精排特征，让精排"看到"粗排的判断
3. **蒸馏层**：精排作为教师，粗排学习软标签，KL散度约束
4. **评估层**：粗排评估用精排Top-K的召回率，而非绝对的CTR相关性
