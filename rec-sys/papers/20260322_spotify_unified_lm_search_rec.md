# A Unified Language Model for Large Scale Search, Recommendation, and Reasoning at Spotify

> 来源：工业界（Spotify） | 日期：20260322 | 领域：推荐系统

## 问题定义

Spotify 的搜索、推荐、问答等任务由独立模型服务，导致知识碎片化、维护成本高、跨任务协同差。核心问题：如何用统一的语言模型同时支持搜索、推荐和推理任务？

## 核心方法与创新点

- **统一序列化框架**：将搜索（query→items）、推荐（user history→items）、推理（question→answer）统一表示为 text-to-text 任务
- **多任务预训练**：
  - 音乐/播客内容的语言建模（MLM + CLM）
  - 搜索点击数据的序列学习
  - 用户行为序列的推荐信号
- **混合专家路由（MoE-style）**：不同任务使用不同 soft-prompt 前缀，共享底层 Transformer 参数
- **稀疏检索集成**：将生成式推荐输出的 token 序列与 ANN 检索结合，兼顾准确性和效率
- **Reasoning 能力**：支持自然语言查询如"帮我找适合跑步的播客"，通过 Chain-of-Thought 推理返回结果

## 实验结论

- 搜索：NDCG@10 较独立模型提升 6%
- 推荐：Recall@50 提升 9%，冷启动场景提升 18%
- 多任务训练带来的正迁移显著，尤其是长尾 query 和新用户场景
- 单一统一模型维护成本降低 60%（相比维护 3 个独立模型）

## 工程落地要点

- **模型规模**：使用 ~1B 参数 Transformer，推理时用 INT8 量化 + TensorRT 部署
- **任务标识符**：每个任务有独特的前缀 token（[SEARCH], [REC], [REASON]），共享模型在不同任务间无缝切换
- **数据混合比例**：搜索:推荐:推理 ≈ 4:4:2，根据业务优先级调整
- **在线服务**：用 KV-cache 加速，批量推理 QPS 满足 Spotify 实时需求（<100ms p99）

## 面试考点

1. **Q：多任务学习中如何避免任务间的负迁移？**
   A：使用任务特定的 soft-prompt 前缀；调整数据采样比例；监控各任务指标，若某任务下降则增大其采样权重

2. **Q：为什么统一模型对冷启动效果更好？**
   A：搜索和推理任务提供了丰富的内容语义信号，弥补了新用户/新 item 交互数据不足的缺陷

3. **Q：大规模部署统一模型的主要挑战？**
   A：延迟（单模型替代多个轻量模型后变慢）、多任务 QPS 峰值的资源调度、模型更新时各任务的版本对齐
