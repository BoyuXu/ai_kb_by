# 从评论到对话：零样本 LLM 对话推荐系统

> 来源：From Reviews to Dialogues: Active Synthesis for Zero-Shot LLM-based Conversational Recommender System | 日期：20260318 | 领域：rec-sys

## 问题定义

对话式推荐系统（CRS）需要大量多轮对话训练数据（用户提需求 → 系统询问澄清 → 给出推荐）。然而收集此类数据成本极高，且多数平台只有评论数据（单向、非对话格式）。本文解决：**如何在没有对话训练数据的情况下，利用现有评论数据构建零样本 LLM CRS？**

## 核心方法与创新点

1. **主动合成（Active Synthesis）**：用 LLM（GPT-4）将用户评论数据自动转化为多轮对话格式。关键设计：
   - 从评论中提取用户偏好信号（情感、属性偏好）作为"用户意图种子"。
   - 按照 CRS 对话模板（询问喜好 → 澄清 → 推荐 → 反馈）生成合成对话。
   - 主动采样：优先合成覆盖长尾用户/物品的对话，缓解流行度偏差。

2. **零样本推理**：合成的对话不用于微调 LLM 本身，而是作为 few-shot prompt 样例，在推理时提供上下文指导 LLM 的推荐决策。

3. **检索增强的示例选择**：根据当前对话状态的语义相似度，从合成对话库中检索最相关的 k 个示例作为 in-context examples，而非随机采样。

## 实验结论

- 在 ReDial 和 INSPIRED 数据集上，零样本设置下比 ChatRec 等基线 Recall@10 提升约 8-12%。
- 相比有监督的 KBRD 和 UNICORN，仅差约 2-3%，但无需任何人工标注对话数据。
- 主动采样 vs 随机采样：长尾物品推荐准确率提升约 15%。

## 工程落地要点

- **合成数据质量控制**：需对 LLM 生成对话做过滤（规则过滤 + 语义一致性检查），去除逻辑不通的对话。
- **评论 → 对话 Prompt 工程**：Prompt 中需明确指定对话风格、澄清轮数、推荐触发条件，避免对话过短或过长。
- **检索库构建**：可用 FAISS 索引合成对话的意图向量，在线检索延迟 < 10ms。
- **成本控制**：合成阶段用 GPT-4，但推理阶段可降级到 GPT-3.5 或开源 LLM，节省 API 成本。

## 面试考点

**Q: 对话推荐系统和传统推荐系统的核心区别？**
- 传统推荐是单次请求-响应；CRS 是多轮交互，系统会主动澄清用户意图，动态调整推荐策略。

**Q: 如何解决 CRS 训练数据稀缺问题？**
- 数据合成（利用 LLM 从评论/行为数据生成对话）、零样本/few-shot 推理（用检索增强的 in-context learning 替代微调）。

**Q: In-context Learning 在推荐中的局限性？**
- Prompt 长度限制（示例数量有限）、示例质量影响大、缺乏持续学习能力、推理成本高（每次请求都要走 LLM）。
