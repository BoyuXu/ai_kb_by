# GEMs: Breaking the Long-Sequence Barrier in Generative Recommendation with a Multi-Stream Decoder

> 来源：arxiv | 日期：20260321 | 领域：rec-sys

## 问题定义

生成式推荐（Generative Recommendation）将用户历史行为序列输入语言模型，自回归生成目标物品的 Semantic ID。然而，当用户历史行为很长（如电商用户数年的购买记录，可能有数千个物品），每个物品用多个 token 表示时，输入序列总长度会爆炸性增长：

- 设用户历史 N 个物品，每个物品 K 个 Semantic ID token，则输入长度为 O(N×K)
- Transformer 的注意力复杂度 O(n²)，长序列推理延迟无法接受
- 现有方法截断历史（只取最近 N 个），丢失了长期偏好信息

GEMs（Generative Efficient Multi-stream）提出多流解码器架构，在不增加推理延迟的前提下建模完整长序列用户历史。

## 核心方法与创新点

1. **多流解码架构（Multi-Stream Decoder）**：
   - 将用户历史拆分为多个"流"（Stream）：短期流（近期行为）、长期流（历史偏好摘要）、类目流（物品类别偏好）
   - 每个流独立 Transformer 编码，最后通过 Cross-Attention 融合
   - 解码时注意力只在各流的压缩表示上计算，而非原始长序列

2. **层次化历史压缩**：
   - 长期历史先通过 Pooling/Attention 压缩为固定长度的 Memory Token（如 64 个 token 表示 1000 条历史）
   - 类似于 LongFormer 的 global token 概念，但针对推荐场景设计

3. **异步多粒度建模**：
   ```
   Short Stream: [item_{t-K}, ..., item_{t-1}]  → 捕获最近兴趣漂移
   Long Stream: compress([item_{t-N}, ..., item_{t-K}]) → 捕获稳定偏好
   Category Stream: [cat_1_freq, cat_2_freq, ...]  → 捕获类目级偏好
   ```
   三流输出通过门控机制（Gating）动态加权融合。

4. **训练效率**：
   - 长期流和类目流可以离线预计算，在线推理只需计算短期流 + 融合
   - 显著降低在线推理的有效序列长度

## 实验结论

- 在 Amazon Books 和内部电商数据集上，GEMs 相比截断为 50 条历史的基线：HR@10 **+9.2%**，NDCG@10 **+7.8%**
- 相比使用完整历史（长序列）的普通 Transformer：推理延迟降低 **68%**，性能持平甚至略优（因为压缩过滤了噪声）
- 在行为序列 >500 条的重度用户群体上提升尤为显著：HR@10 **+15%**
- 参数量仅增加约 12%（相比单流基线），计算开销可控

## 工程落地要点

1. **流的异步预计算**：长期流和类目流天然可以 T+1 离线更新（用户今日偏好摘要），在线只需实时更新短期流，大幅降低推理延迟
2. **内存管理**：多流融合需要缓存各流的 KV Cache，需要评估显存开销；可以用 PagedAttention 按需分配
3. **冷启动处理**：新用户只有短期流，长期流用类目热门或人口统计学先验初始化
4. **流的粒度调优**：短期流长度（K）和长期流压缩比例是关键超参，需要根据业务场景（时效性 vs. 稳定性）调整

## 面试考点

- Q: 长序列建模的主要挑战是什么？推荐系统中常见的解决方案？
  A: 主要挑战是 O(n²) 注意力复杂度和内存开销。推荐领域常见方案：(1) 截断+最近邻（简单但丢失信息）；(2) 层次化 RNN/Transformer；(3) 记忆网络（Memory Network）将历史压缩为固定大小记忆；(4) 线性注意力（如 Linformer）；(5) 多流/多粒度分治（如 GEMs）。

- Q: 推荐系统中的"兴趣漂移"问题如何建模？
  A: 兴趣随时间变化，短期兴趣和长期兴趣需要分开建模。常见方法：(1) 时间衰减权重；(2) 会话内/跨会话区分建模；(3) 注意力机制动态融合不同时期偏好；(4) 多流架构如 GEMs。

- Q: 序列推荐中 Semantic ID 和整数 ID 各有什么优劣？
  A: 整数 ID：训练简单，内存占用小，但无法泛化到新物品，冷启动差。Semantic ID：有语义，冷启动好，可复用语言模型知识，但序列长度增加（每个物品多个 token），引入量化误差，RQ-VAE 训练也更复杂。
