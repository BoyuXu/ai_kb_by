# A Unified Language Model for Large Scale Search, Recommendation, and Reasoning at Spotify

> 来源：arxiv / Spotify Research | 日期：20260321 | 领域：rec-sys

## 问题定义

Spotify 的搜索（Search）、推荐（Recommendation）和推理（Reasoning）三大核心任务长期由独立模型服务，带来以下问题：
- **维护成本高**：三套模型、三套特征工程、三套推理链路，工程复杂度呈倍数增长
- **知识孤岛**：搜索积累的用户查询理解能力无法共享给推荐，推荐的协同过滤信号无法增强搜索
- **多任务泛化弱**：用户同时有搜索和发现需求，单任务模型无法联合优化

本文提出 Unified Language Model（ULM），用统一的 LLM 框架统一处理三个任务，通过任务标记（Task Token）区分不同场景，实现知识共享和多任务联合优化。

## 核心方法与创新点

1. **统一序列格式**：
   - 将搜索、推荐、推理统一表示为"用户上下文序列 → 目标 token"的语言建模任务
   - 搜索：`[User History] [Query: "jazz playlist"] → [Item: Coltrane Greatest Hits]`
   - 推荐：`[User History] [RECOMMEND] → [Item: Miles Davis]`  
   - 推理：`[User History] [EXPLAIN why] → [Text: "Because you liked..."]`

2. **层次化 Semantic ID**（与生成式检索结合）：
   - 物品用 Semantic ID token 序列表示，模型同一套词表处理内容和物品 ID
   - 减少词表碎片化，提升罕见物品的泛化

3. **多任务联合训练**：
   - 三个任务数据混合训练（比例经过调参）
   - 搜索任务帮助模型理解用户意图语义
   - 推荐任务提供协同过滤偏好模式
   - 推理任务增强模型的解释能力，间接提升可解释推荐质量

4. **工业级推理优化**：
   - 使用 vLLM + PagedAttention 服务推理
   - 在线请求做批处理（Dynamic Batching）
   - 模型量化至 INT8 后性能损失 <1%

## 实验结论

- 统一模型相比各自独立模型：搜索 NDCG@10 **+3.1%**，推荐 HR@20 **+5.7%**，推理质量人工评分 **+12%**
- 参数量：统一 7B 模型优于三个独立 1B 模型（总参数 3B），验证参数效率提升
- 冷启动场景（新用户 <5 次交互）统一模型提升最显著：推荐 Recall@50 **+18%**
- 训练数据总量约 500B token，在 Spotify 内部语料上预训练 + 任务微调

## 工程落地要点

1. **任务采样比例**：三个任务数据量差异巨大（推荐数据远多于推理），需要做温度采样（Temperature Sampling）平衡，避免大任务压制小任务
2. **延迟 SLA**：LLM 推理天然比双塔慢，对延迟敏感场景需要做 Speculative Decoding 或蒸馏小模型
3. **多任务冲突监控**：联合训练时某个任务指标突然下降往往意味着梯度冲突，需要设置任务独立的评估 dashboard
4. **增量更新**：物品库每天新增大量内容，需要高效的增量微调（LoRA/Adapter）而非全量重训

## 面试考点

- Q: 多任务学习中如何处理任务间的梯度冲突？
  A: 常见方法：(1) PCGrad：投影冲突梯度到不冲突的方向；(2) 任务权重自适应调整（如 GradNorm）；(3) 任务专属 Adapter 层，共享 backbone；(4) 多任务采样比例调参。实践中最简单有效的是分层参数共享 + 采样比例调优。

- Q: 为什么统一模型在冷启动场景提升最大？
  A: 冷启动用户历史少，协同过滤信号弱。统一模型通过搜索和推理任务学到了更强的语义理解能力，能从少量历史中提取更丰富的用户意图，相当于有了更强的语义先验。

- Q: 如何评估推荐系统的推理质量（可解释性）？
  A: 通常结合自动指标（ROUGE、BERTScore）和人工评估（相关性、准确性、说服力三个维度）。工业界还会做 A/B 测试看是否提升用户对推荐的接受率（点击率、转化率）。
