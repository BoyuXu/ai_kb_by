## L1 概念卡（What）

### L1-01｜什么是 A/B 测试？

**Q：** A/B 测试的本质是什么？解决什么问题？

**A：**
A/B 测试本质是**受控实验（Controlled Experiment）**，通过随机将用户分配到实验组（B）和对照组（A），在其他条件不变的情况下，只改变一个变量，然后统计推断两组指标差异是否为**真实效果**，而非随机波动。

- **解决问题**：因果推断（Causality），而非相关性
- **核心假设**：SUTVA（稳定单元处理值假设）—— 各用户的结果不互相影响
- **应用**：推荐算法上线、UI改版、广告策略验证

---

### L1-02｜什么是假设检验？

**Q：** A/B 测试中的原假设 H0 和备择假设 H1 分别是什么？

**A：**
| 假设 | 含义 |
|------|------|
| H0（原假设） | 实验组与对照组无差异（Δ = 0） |
| H1（备择假设） | 实验组与对照组存在真实差异（Δ ≠ 0） |

- **检验逻辑**：在 H0 为真的前提下，观察到当前数据或更极端数据的概率（p-value）
- **决策**：若 p < α，则拒绝 H0，认为有显著效果
- **注意**：拒绝 H0 ≠ 证明 H1 为真，只是 H0 的可能性极低

---

### L1-03｜什么是 p-value？

**Q：** p-value 的定义是什么？如何正确解读？

**A：**
**p-value** = 在 H0 为真的条件下，观察到当前或更极端结果的概率。

- **正确解读**：p < 0.05 表示"如果无真实效果，出现这么大差异的概率 < 5%"
- **错误解读 ❌**：
  - "p < 0.05 说明效果一定存在" → 还可能是 Type I Error
  - "p = 0.03 比 p = 0.04 效果更强" → p-value 不衡量效果大小
  - "p > 0.05 说明无效果" → 可能是样本量不足（Type II Error）

---

### L1-04｜Type I Error 和 Type II Error 是什么？

**Q：** 两类错误的定义和后果分别是什么？

**A：**
| 错误类型 | 又称 | 定义 | 控制参数 | 后果 |
|----------|------|------|----------|------|
| Type I Error | 假阳性（误报） | H0 为真却拒绝了 H0 | α（显著性水平） | 上线无效甚至有害的改动 |
| Type II Error | 假阴性（漏报） | H1 为真却未能拒绝 H0 | β（功效 = 1-β） | 错过真实有效的改进 |

- 工业实践：α 通常设 0.05，Power（1-β）通常设 0.8 或 0.9
- 两者存在权衡：降低 α → 增大 β，需增加样本量来同时控制

---

### L1-05｜什么是统计功效（Power）？

**Q：** 统计功效的含义是什么？影响因素有哪些？

**A：**
**统计功效 Power = 1 - β**，即"当真实效果存在时，实验能正确检测到的概率"。

- Power = 0.8 表示：若真实 MDE 存在，有 80% 概率得出显著结论
- **影响因素**：
  - ↑ 样本量 N → Power ↑
  - ↑ 真实效果大小 δ → Power ↑
  - ↓ 指标方差 σ² → Power ↑
  - ↑ 显著性水平 α → Power ↑（但 Type I Error 也↑）

---

### L1-06｜什么是 MDE？

**Q：** 最小可检测效果（MDE）的含义和作用？

**A：**
**MDE（Minimum Detectable Effect）** = 在给定 α、β、样本量下，实验能以期望功效检测到的最小真实效果大小。

- **作用**：帮助实验设计者确定：这次实验能检测到多小的提升？
- **工业意义**：推荐系统 CTR 提升 0.1% 可能值数百万营收，MDE 要设计得足够小
- **与样本量关系**：MDE ↓ → 需要更多样本（敏感性 ↑，成本 ↑）
- **举例**：若当前 CTR = 5%，MDE 设为 0.5%（相对提升 10%），则 δ = 0.0025

---

### L1-07｜什么是 SRM（Sample Ratio Mismatch）？

**Q：** SRM 是什么？为什么实验中必须先检查 SRM？

**A：**
**SRM（Sample Ratio Mismatch）** = 实际流量分配比例与预期不符。

- **例子**：设计 50:50 分桶，实际 A 组 48%，B 组 52%
- **危害**：分配不随机 → 两组用户系统性不同 → 实验结论完全无效
- **常见原因**：
  - 流量分割代码 Bug
  - Bot/爬虫流量分布不均
  - 缓存/预加载导致部分用户未被正确分桶
  - 重定向逻辑问题
- **检验方法**：卡方检验（Chi-square test）
- **规则**：SRM 不通过 → 停止分析，先排查原因

---

### L1-08｜什么是 CUPED？

**Q：** CUPED 的核心思想是什么？

**A：**
**CUPED（Controlled-experiment Using Pre-Experiment Data）** = 利用实验前协变量减少指标方差，提升实验敏感度的方差缩减技术。

- **核心**：用历史数据（如实验前 CTR）解释当前指标的一部分方差，去掉这部分后方差更小
- **调整公式**：`Y_cuped = Y - θ × (X - X̄)`
  - Y = 实验期指标，X = 实验前协变量（同一用户历史数据），θ = Cov(Y,X)/Var(X)
- **效果**：方差减少 30%-70%，相当于样本量扩大数倍
- **使用方**：Microsoft ExP、Airbnb、Netflix、字节跳动

---

### L1-09｜什么是 AA 测试？

**Q：** AA 测试的目的和做法是什么？

**A：**
**AA 测试** = 将用户分为两组，但两组都使用相同策略（无任何差异），验证实验系统的随机性和可靠性。

- **目的**：
  1. 验证分桶系统是否正确随机（两组基线应无显著差异）
  2. 检查实验平台的 Type I Error 率（长期 AA 测试中，显著率应 ≈ α）
  3. 收集方差估计，为后续实验的样本量计算提供基础
- **AA 不通过怎么办**：检查分桶逻辑、哈希函数、数据管道、历史偏差

---

### L1-10｜什么是 Holdout 实验？

**Q：** Holdout Set 的作用是什么？和普通 A/B 测试有什么区别？

**A：**
**Holdout Set（保留对照组）** = 长期保留一个不接受任何新策略的用户群，作为衡量整体系统累积效益的基准线。

- **区别**：普通 A/B 测试是短期、单次，Holdout 是长期持续
- **解决问题**：
  - 多个实验同时上线，无法归因各自贡献
  - 节假日、季节性因素干扰短期实验
  - 评估整个算法迭代周期的累积效益
- **代价**：Holdout 用户无法享受最新优化，有业务损失

## L2 原理卡（How）

### L2-01｜样本量公式推导与使用

**Q：** A/B 测试样本量公式是什么？各参数如何取值？

**A：**
**双尾检验样本量公式**（每组）：

```
N = 2 × (z_{α/2} + z_β)² × σ² / δ²
```

| 参数 | 含义 | 常用取值 |
|------|------|----------|
| z_{α/2} | 显著性水平对应 z 值 | 1.96（α=0.05） |
| z_β | 功效对应 z 值 | 0.84（Power=80%）；1.28（Power=90%） |
| σ² | 指标方差（需预估） | 来自历史数据或 AA 测试 |
| δ | MDE（期望检测到的最小差值） | 业务决定 |

**实际计算示例**：
- CTR 均值 5%，σ² ≈ p(1-p) = 0.0475
- 期望检测 0.5% 绝对提升，δ = 0.005
- N = 2 × (1.96+0.84)² × 0.0475 / 0.005² ≈ **59,584 人/组**

**调优技巧**：σ² 可通过 CUPED 缩减，等效放大样本量

---

### L2-02｜p-value 的计算过程

**Q：** 双样本 z 检验中，p-value 如何计算？

**A：**
**步骤**：

1. 计算检验统计量：
```
z = (Ȳ_B - Ȳ_A) / √(σ_A²/n_A + σ_B²/n_B)
```

2. 在标准正态分布中查 z 对应的双尾概率：
```
p-value = 2 × P(Z > |z|) = 2 × (1 - Φ(|z|))
```

3. 判断：p < α（如 0.05）则拒绝 H0

**置信区间**（同等价）：
```
CI = (Ȳ_B - Ȳ_A) ± z_{α/2} × SE
若 CI 不包含 0 → 显著
```

**注意**：大样本可用 z 检验；小样本用 t 检验；比例类指标（CTR）可直接用 z 检验

---

### L2-03｜CUPED 方差缩减公式详解

**Q：** CUPED 如何计算调整后的指标？θ 怎么求？方差减少多少？

**A：**
**CUPED 调整指标**：
```
Y_cuped = Y - θ × (X - X̄)
```

**θ 的最优解**（最小化方差）：
```
θ* = Cov(Y, X) / Var(X)
```

**调整后方差**：
```
Var(Y_cuped) = Var(Y) × (1 - ρ²)

其中 ρ = Corr(Y, X)，即实验期指标与历史指标的相关系数
```

**方差减少率**：
```
减少比例 = ρ²
```

**示例**：若 ρ = 0.7（历史 CTR 与当前 CTR 相关性 0.7）
- 方差减少 = 0.7² = 49%
- 等效样本量扩大约 2× → 相当于免费多做了一倍实验

**工程实现**：
- X 通常取实验前 7/14/30 天的同指标均值
- 注意 X 必须与实验分配无关（实验前数据）

---

### L2-04｜Bonferroni 校正原理

**Q：** 多重比较问题是什么？Bonferroni 如何校正？

**A：**
**多重比较问题（Multiple Testing Problem）**：
- 同时检验 k 个指标，每个 α=0.05，至少一个误报的概率 = 1-(1-0.05)^k
- k=10 时：误报概率 ≈ 40%（远高于 5%）

**Bonferroni 校正**：
```
校正后显著性水平 α' = α / k
```
- k=10，α=0.05 → α' = 0.005
- 优点：简单保守，控制 FWER（Family-Wise Error Rate）
- 缺点：过于保守，降低 Power（漏报增多）

**更优替代**：
- **BH 校正（Benjamini-Hochberg）**：控制 FDR（False Discovery Rate），比 Bonferroni 更宽松
- **分层优先级**：将指标分为"主指标"和"辅助指标"，只对主指标严格控制 α

### L2-05｜Cookie Day（入组日）分析方法

**Q：** 什么是 Cookie Day 分析？如何用它判断新奇效应？

**A：**
**Cookie Day / Day-of-Enrollment 分析**：

**方法**：
```
按用户的"入组日"（进入实验的日期）分层，
分别计算每个入组日期队列的实验效果，
绘制"实验效果 vs 入组后天数"曲线
```

**解读**：
| 效果曲线形态 | 可能原因 | 判断 |
|-------------|---------|------|
| 早期高、后期衰减 | 新奇效应（Novelty Effect） | 等效果稳定再下结论 |
| 早期低、后期上升 | 学习效应（Learning Effect） | 需更长实验时间 |
| 稳定无明显趋势 | 真实稳定效果 | 可信 |

**稳定期判断**：
- 观察效果曲线连续 N 天（通常 7 天）无显著趋势变化
- 建议实验时长 ≥ 1-2 周，覆盖完整用户行为周期

---

### L2-06｜正交分层实验原理

**Q：** 实验平台中的"层（Layer）"和"正交"是什么？

**A：**
**分层实验架构**：

```
流量总池
├── Layer 1（推荐召回层）
│   ├── Exp A（新召回策略）
│   └── Exp B（对照）
├── Layer 2（排序层）
│   ├── Exp C（新排序模型）
│   └── Exp D（对照）
└── Layer 3（重排层）
    └── Exp E（多样性策略）
```

**正交性**：
- 不同 Layer 的用户分桶相互独立（通过不同 Hash Key 实现）
- 同一用户可同时参与不同 Layer 的不同实验
- **条件**：两个 Layer 的实验在统计上无交互效应

**互斥（Mutex）**：
- 同一 Layer 内，同一用户只能属于一个实验组
- 用于测试有直接冲突的策略（如两种完全不同的排序逻辑）

### L2-07｜SRM 检测方法

**Q：** 如何用卡方检验检测 SRM？

**A：**
**卡方检验 SRM**：

```
χ² = Σ (O_i - E_i)² / E_i

O_i = 实际各组流量数
E_i = 按预期比例计算的期望流量数
```

**示例**：
- 预期 50:50，总样本 10000
- 实际：A 组 4800，B 组 5200
- E_A = E_B = 5000
- χ² = (4800-5000)²/5000 + (5200-5000)²/5000 = 8 + 8 = 16
- p-value = P(χ²(1) > 16) ≈ 0.00006 → **严重 SRM**

**SRM 排查路径**：
1. 检查流量分割代码（哈希函数是否有 Bug）
2. 检查是否有 Bot 流量
3. 检查日志是否有丢失（数据管道问题）
4. 检查是否有缓存/CDN 导致用户未正确暴露

---

### L2-08｜统计功效与样本量的关系

**Q：** 如何在样本量不足的情况下提升实验功效？

**A：**
**提升 Power 的四条路径**：

| 方法 | 原理 | 代价 |
|------|------|------|
| ↑ 样本量 N | 直接增加 | 时间/流量成本 |
| ↑ 实验 α | 降低门槛 | Type I Error 增大 |
| ↓ 方差 σ² | CUPED、分层采样 | 工程复杂度 |
| ↑ MDE δ | 只测大效果 | 错过小但真实的提升 |

**工业推荐做法**：
- **首选 CUPED**：零成本方差缩减，推荐系统中 ρ 通常 0.5-0.8
- **延长实验**：如需 2× 样本量，延长实验时间 2×
- **重新评估 MDE**：是否确实需要检测如此小的效果？

---

### L2-09｜新奇效应 vs 学习效应

**Q：** 新奇效应和学习效应有什么区别？如何区分并处理？

**A：**
| 效应 | 定义 | 曲线特征 | 处理方式 |
|------|------|----------|----------|
| 新奇效应（Novelty Effect） | 用户因新鲜感短期活跃，后续回归正常 | 实验初期高，逐渐下降 | 延长实验，看稳定期效果 |
| 学习效应（Learning Effect） | 用户需要适应新功能，初期效果低于真实水平 | 实验初期低，逐渐上升 | 同样延长实验，等收敛 |

**识别技巧**：
- Cookie Day 分析：画出"新用户 vs 老用户"的效果对比
  - 若新用户效果显著高于老用户 → 新奇效应（老用户已"审美疲劳"）
  - 若新用户效果低于老用户 → 学习效应（老用户已习惯）
- 对新功能（UI 改版、新交互）尤其需要警惕新奇效应

---

### L2-10｜Holdout 实验设计与长期收益评估

**Q：** Holdout 实验如何设计？如何评估累积收益？

**A：**
**Holdout 设计**：
```
总流量 = 实验流量(95%) + Holdout 流量(5%)
Holdout 组：永远不接受任何新策略，作为绝对基准
```

**评估累积收益**：
```
累积收益 = (Holdout 用户指标 - 当前全量用户指标) / Holdout 用户指标
```

**典型应用场景**：
- 算法团队每季度对比：过去 N 次迭代的总体 CTR/GMV 提升
- 排除季节性因素（Holdout 组和当前组同时经历相同季节）

**注意事项**：
- Holdout 期间用户体验损失（商业成本），通常 5% 流量最多 3 个月
- Holdout 组需严格管控，任何实验都不能侵入

## L3 决策卡（Why）

### L3-01｜为什么需要随机分组？

**Q：** A/B 测试为什么必须随机分组？不随机会有什么问题？

**A：**
**随机分组的作用**：消除选择偏差（Selection Bias），使实验组和对照组在所有可观测和不可观测变量上统计等价。

**不随机的后果（反例）**：
- 把"活跃用户"放实验组 → 实验组本来就更好 → 虚假正效果
- 按地区分组 → 不同地区用户特征不同 → 混淆因素

**随机化的保证**：
- 任何可能影响指标的混淆变量，在两组间期望分布相同
- 正是这一点，使 A/B 测试可以做"因果推断"而非"相关性分析"

---

### L3-02｜为什么要先做 AA 测试？

**Q：** 为什么每个实验平台都应该定期做 AA 测试？

**A：**
**AA 测试的三个价值**：

1. **验证系统随机性**：两组相同策略，若指标显著差异 → 分桶或数据管道有问题
2. **校准 Type I Error 率**：运行 100 次 AA 测试，显著次数应 ≈ 5 次（α=0.05）；若 >> 5% 说明 p-value 计算有偏
3. **估计方差基准**：为真实 A/B 实验的样本量计算提供 σ² 参考

**何时做 AA**：
- 新建实验平台必做
- 大规模系统变更（分桶策略调整）后必做
- 实验结果异常时作为诊断工具

---

### L3-03｜为什么需要 CUPED 而不是直接加样本量？

**Q：** CUPED 相比增加样本量有什么优势？局限性是什么？

**A：**
**优势**：
- **零流量成本**：不需要额外用户，只用现有用户的历史数据
- **快速效果**：ρ=0.7 时，方差减少 49%，实验时长可缩短近一半
- **工程可行**：推荐系统本身就有历史行为数据，实现成本低

**局限性**：
- 协变量 X 必须与实验分配无关（必须是实验前数据）
- 若 ρ 很低（历史数据相关性弱），收益有限
- 新用户没有历史数据，无法使用 CUPED

**何时优先用 CUPED**：
- 用户行为稳定（ρ 高）的场景：老用户 CTR、购买行为
- 实验流量受限（资源紧张）时

---

### L3-04｜为什么不能提前停止实验（Peeking Problem）？

**Q：** Peeking Problem 是什么？为什么危害很大？

**A：**
**Peeking Problem（窥探问题）**：
- 实验过程中频繁查看结果，一旦 p < 0.05 就立即停止
- 本质：多次检验同一数据，每次检验都消耗 α

**为什么危害大**：
- 若每天查看，连续 20 天，相当于做了 20 次检验
- 实际 Type I Error 率 >> 5%（可高达 30%-50%）

**解决方案**：
1. **预先设定样本量**：计算好后不提前停止
2. **Sequential Testing（序贯检验）**：Always Valid Inference（AVI），允许随时查看，但调整 α 的消耗方式
3. **Bonferroni 校正时间点**：若计划看 k 次，每次用 α/k

---

### L3-05｜为什么正交分层比互斥实验更高效？

**Q：** 正交实验和互斥实验在流量利用上有什么本质区别？

**A：**
**互斥实验**：
- 同一流量只能用于一个实验
- 10 个实验需要 10× 流量
- 适用：改动有直接冲突的策略

**正交分层实验**：
- 每个 Layer 独立使用全部流量
- 10 个 Layer 各自使用 100% 流量（前提是 Layer 间正交）
- **正交条件**：Layer 间无交互效应（实验 A 的效果不依赖实验 B 的处理）
- 适用：推荐系统不同阶段（召回/排序/重排）的独立优化

**工业实践**：字节跳动/美团/阿里的实验平台均采用多层正交架构，每天可并行运行数千个实验

---

### L3-06｜为什么护栏指标（Guardrail）必须同步监控？

**Q：** Guardrail Metrics 的作用是什么？如何设置？

**A：**
**Guardrail Metrics（护栏指标）**：
- 定义：实验必须满足的底线指标，即使目标指标提升，护栏指标劣化也应拒绝上线
- **常见护栏**：
  - 页面加载时延（P99 < 200ms）
  - 崩溃率（Crash Rate < 0.1%）
  - 广告加载成功率
  - 核心用户留存率

**为什么必要**：
- 目标指标（CTR）和系统质量可能存在权衡
- 没有护栏：优化 CTR 可能导致页面加载变慢、用户体验下降
- 护栏失效 = 上线了一个"虽然提升了点击但毁了用户体验"的策略

**设置原则**：护栏阈值基于历史基线 ± 合理容忍范围，提前声明，不可事后调整

## L4 场景卡（Scene）

### L4-01｜场景：实验跑了3天，CTR高了15%，能上线吗？

**Q：** 实验组 CTR 提升 15%，实验仅跑了 3 天，p < 0.01，应该怎么判断？

**A：**
**不能直接上线，需要完整 checklist 排查**：

**Step 1：检查 SRM**
```
实际流量比是否符合预期？用卡方检验验证。
SRM 未通过 → 结果无效，停止分析。
```

**Step 2：评估实验时长是否足够**
- 3 天可能未覆盖完整用户行为周期（如周末/工作日差异）
- 推荐：至少 1-2 周，最好覆盖 1 个完整业务周期

**Step 3：排查新奇效应**
- CTR 提升 15% 异常大 → 高度怀疑新奇效应
- 做 Cookie Day 分析：看不同入组日用户的效果是否稳定

**Step 4：检查多重比较**
- 是否同时看了多个指标，只报告了最好的那个？

**Step 5：检查护栏指标**
- 时延、崩溃率等是否正常？

**结论**：3 天 + 15% 提升 → 大概率是新奇效应或系统问题，建议延长到 2 周再看

---

### L4-02｜场景：AA 测试不通过怎么办？

**Q：** AA 测试发现两组 CTR 有显著差异（p = 0.02），下一步怎么做？

**A：**
**排查路径（由浅到深）**：

1. **检查分桶逻辑**：
   - Hash Key 是否唯一且均匀分布？
   - 是否存在用户 ID 复用、哈希碰撞问题？

2. **检查数据管道**：
   - 日志收集是否有丢失？
   - A 组和 B 组的日志处理逻辑是否一致？

3. **检查历史偏差**：
   - 是否在实验开始前就对两组做了不同处理？
   - 是否有 Cookie 迁移、用户账号合并等操作？

4. **检查 Bot 流量**：
   - 爬虫/机器人流量是否在两组分布不均？

5. **重新随机化**：
   - 换一个 Hash Key 重新分桶，再做一次 AA 测试

**规则**：AA 不通过 → 当前实验系统不可信，不可启动任何正式 A/B 实验

---

### L4-03｜场景：如何排查 SRM？

**Q：** 发现实验组/对照组比例是 47:53，原本设计 50:50，如何排查？

**A：**
**SRM 排查 SOP**：

```
第一步：确认 SRM 严重程度
└── 卡方检验 p < 0.001 → 严重 SRM，必须修复

第二步：定位发生阶段
├── 分配阶段（Assignment）：检查哈希函数
├── 曝光阶段（Exposure）：检查埋点是否遗漏
└── 日志阶段（Logging）：检查数据管道丢失

第三步：分维度拆解
├── 按设备类型（iOS/Android/PC）
├── 按地区
├── 按新老用户
└── 找到 SRM 集中的子群体

第四步：常见原因
├── 重定向逻辑（实验组多一次跳转导致流失更多）
├── 缓存命中率差异（实验组缓存 miss 更多）
├── Bot 流量（通过 IP 黑名单过滤后重新验证）
└── Cookie 失效（实验组因新功能导致 Cookie 重置）
```

---

### L4-04｜场景：推荐系统中如何处理网络效应？

**Q：** 社交推荐场景，用户 A（实验组）关注了用户 B（对照组），实验结果可信吗？

**A：**
**问题**：SUTVA（稳定单元处理值假设）被违反 —— 实验组用户的行为影响了对照组

**影响**：
- 实验组用户看到了新推荐 → 行为改变 → 内容创作/社交互动影响对照组
- 对照组被"污染"，实际接受了部分实验处理
- 结果：低估实验效果（污染稀释了差异）

**解决方案**：

| 方案 | 原理 | 适用 |
|------|------|------|
| 以社区/图聚类分桶 | 同一社交图聚类的用户放同一组 | 强社交平台 |
| 地理隔离实验 | 不同城市/地区作为实验单元 | 本地化服务 |
| 时间分割 | A 时间段用策略 A，B 时间段用策略 B | 无个体识别场景 |
| 双边市场设计 | 买家/卖家分层分桶 | 电商/直播平台 |

---

### L4-05｜场景：实验显示 CTR↑ 但 DAU↓，应该上线吗？

**Q：** 实验组 CTR 提升 3%，但 7 日留存下降 1%，如何决策？

**A：**
**指标冲突决策框架**：

1. **判断主次指标**：
   - 若北极星指标是"DAU/留存" → 留存下降 = 实验失败
   - 若北极星指标是"GMV/CTR" → 看业务价值权衡

2. **量化业务影响**：
   ```
   CTR +3% 带来的短期收益 vs 留存 -1% 导致的长期 DAU 损失
   估算 LTV（用户生命周期价值）影响
   ```

3. **深入分析原因**：
   - CTR ↑ + 留存 ↓ → 典型"标题党"信号：推荐了用户会点但不满意的内容
   - 检查满意度指标（收藏、完播率、负反馈率）

4. **结论**：
   - 若留存下降统计显著 + 原因是内容质量问题 → **不应上线**
   - 护栏指标（留存）红线 → 无论目标指标多好都不上线

## L5 陷阱卡（Trap）

### L5-01｜陷阱：p < 0.05 说明效果一定显著？

**Q：** p = 0.03，是否可以直接宣布"实验有显著效果"并上线？

**A：**
**❌ 错误！p < 0.05 只是"统计显著"，≠ "实践显著"**

**三个常见陷阱**：

| 陷阱 | 说明 | 正确做法 |
|------|------|---------|
| 效果量太小 | 样本 100 万，CTR +0.001%，p=0.001，但业务价值为零 | 同时报告效果量（Effect Size）和置信区间 |
| 多重比较未校正 | 看了 20 个指标，找到 1 个 p=0.03 | Bonferroni/BH 校正，或预先声明主指标 |
| SRM 未检查 | p=0.03 但存在严重 SRM | 先过 SRM 检验再看显著性 |

**正确汇报格式**：
```
CTR 提升 0.5%（95% CI: 0.2%-0.8%），p=0.003
SRM 检验通过（p=0.42），实验时长 14 天
结论：统计显著 + 业务有意义 + 系统可信
```

---

### L5-02｜陷阱：可以提前停止实验吗（Peeking Problem）？

**Q：** 实验跑了 5 天，p 值已经 < 0.01，为什么不能直接停止？

**A：**
**❌ 绝对不能！这是 Peeking Problem / Optional Stopping**

**为什么危害极大**：
```
模拟实验：真实无效果（H0 为真），每天查看 p 值
- 第 1 天看：Type I Error ≈ 5%
- 第 5 天看：累计 Type I Error ≈ 22%
- 第 20 天看：累计 Type I Error ≈ 64%
```

**本质**：数据收集过程中重复检验，相当于用同一数据"做了多次实验"

**正确做法**：
1. **预先注册（Pre-registration）**：实验前声明样本量、检验时间、主指标
2. **Sequential Testing**：如 mSPRT、Always Valid Inference，允许随时停止但控制 Type I Error
3. **Bayesian A/B Testing**：贝叶斯框架天然支持提前停止

**记住**：5 天 p < 0.01，很可能只是随机波动的峰值，继续等往往会回到 > 0.05

---

### L5-03｜陷阱：多个指标同时提升一定是好事？

**Q：** CTR、CVR、GMV 三个指标同时显著提升，这个实验一定成功吗？

**A：**
**❌ 不一定！需要警惕以下情况**：

1. **多重比较膨胀**：3 个指标，α 未校正，总体误报率 ≈ 14%
2. **指标之间高度相关**：CTR、CVR、GMV 往往正相关，"三个提升"可能来自同一个效果，信息量不如表面多
3. **遗漏了负向指标**：
   - 护栏指标（页面加载时延）恶化了吗？
   - 用户负反馈率上升了吗？
   - 7 日 / 30 日留存下降了吗？

4. **SRM 或 Peeking**：三个指标都提升也不能说明系统没问题

**正确做法**：
- 预先声明一个主指标（北极星指标）
- 辅助指标仅作为参考，单独报告 Effect Size
- 必须同时检查护栏指标

---

### L5-04｜陷阱：实验显著 = 可以直接全量上线？

**Q：** 实验 p < 0.05，是否意味着可以立刻全量发布？

**A：**
**❌ 还差最后一公里！上线前完整 checklist**：

| 检查项 | 说明 |
|--------|------|
| ✅ SRM 检验 | 实验组/对照组比例正确 |
| ✅ 实验时长充足 | ≥ 1-2 周，覆盖业务周期 |
| ✅ 护栏指标 | 所有 Guardrail 指标无恶化 |
| ✅ 新奇效应排查 | Cookie Day 分析无衰减趋势 |
| ✅ 效果量合理 | 提升幅度在预期范围内（≠ 异常大） |
| ✅ 多重比较已校正 | 主指标事先声明，有校正方法 |
| ✅ 全量风险评估 | 小流量显著 ≠ 全量一定好（方差估计可能不准） |

**全量策略**：通常先 10% → 50% → 100% 分阶段放量，观察是否有异常

---

### L5-05｜陷阱：分层实验中的"层污染"问题

**Q：** 不同 Layer 的实验真的完全独立吗？什么情况下正交假设会被破坏？

**A：**
**正交假设可能被破坏的场景**：

| 场景 | 说明 |
|------|------|
| 交互效应 | 排序策略 A 的效果，在召回策略 B 存在时会增强/减弱 |
| 共享资源竞争 | 两个 Layer 都消耗同一资源（如 GPU），相互影响延迟 |
| 相同用户特征改变 | Layer 1 改变了用户行为，影响了 Layer 2 的输入特征 |
| 日志串联错误 | 不同 Layer 的日志被错误聚合 |

**检测方法**：
- **交互检验**：将两个 Layer 的实验组合（A1B1、A1B2、A2B1、A2B2），检验交互项是否显著
- **如果发现交互**：两个实验不可正交，需改为互斥（Mutex）

**工业实践**：推荐系统各阶段（召回/粗排/精排/重排）之间通常有交互，跨阶段实验需谨慎声明正交假设
