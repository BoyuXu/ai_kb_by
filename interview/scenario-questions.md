## 场景1：新用户冷启动

### 📌 背景分析

新用户冷启动是推荐系统中最经典的难题。核心矛盾：**系统对用户一无所知，但需要立即提供个性化体验**。

**业务影响**：
- 新用户留存率直接决定产品增长曲线
- 首屏点击率（CTR）在新用户首 session 是成熟用户的 30%-50%
- 新用户如果前3次推荐都不满意，流失率高达70%

**技术难点**：
- 无历史行为数据（点击、收藏、购买等）
- 用户画像缺失（兴趣标签、人口属性）
- 无法计算相似用户（User-CF失效）

---

### 🧠 完整解题思路

#### 1. 注册信息利用（Immediate）

**客户端被动采集**：
```
- 设备信息：品牌/型号 → 消费能力推断
- 地理位置：城市级别 → 内容地域偏好
- 网络环境：WiFi/4G → 视频/图文策略
- 安装渠道：App Store/应用宝 → 用户质量分层
- 时段：注册时间 → 作息规律
```

**主动引导填写**：
```
- 兴趣选择：3-5个主题标签（音乐/体育/游戏）
- 身份选择：学生/职场/宝妈 → 人群画像
- 社交授权：微信/QQ登录 → 社交图谱导入
```

**推荐策略**：基于注册信息走**规则召回 + 热门兜底**

#### 2. 多路召回设计

```
┌─────────────────────────────────────────────────────────┐
│                    新用户召回架构                          │
├─────────────────────────────────────────────────────────┤
│  1. 热门召回：全站Top100，按品类/地域加权                   │
│  2. 运营配置：人工精选池，新用户专属内容                     │
│  3. 相似人群：同设备/同地域/同注册渠道用户的热门             │
│  4. 内容冷启：新发布优质内容，给新用户探索                   │
│  5. 多样性保证：类别打散，避免同质化                         │
└─────────────────────────────────────────────────────────┘
```

#### 3. 实时反馈闭环（最关键）

**短序列建模**：
- 记录用户前N次点击（通常N=5~10）
- 使用 Session-based 模型（NARM、SR-GNN）实时捕捉即时兴趣
- 实时更新召回的 I2I 相似度

**探索-利用平衡（EE问题）**：
```python
# ε-greedy 简化版
if random() < ε:  # ε随交互次数递减
    item = random_select(candidates)  # 探索
else:
    item = model.predict_best(candidates)  # 利用
```

#### 4. 模型层面的冷启方案

**Meta-Learning（元学习）**：
- MAML、Reptile 等算法，学习"如何快速适应新用户"
- 少量交互即可快速收敛

**对比学习增强**：
- 将冷启动问题转化为：学习「用户注册特征」到「用户偏好」的映射
- 预训练阶段用全量用户，推理阶段只用注册特征

**ID特征的特殊处理**：
```python
# 冷启用户ID embedding用全局平均初始化
user_emb = embedding_lookup(user_id) if user_id in vocab else global_user_emb

# 或者直接用属性特征，不用ID
user_feature = concat([age_emb, gender_emb, city_emb, device_emb])
```

#### 5. 工程架构

```
用户注册
    │
    ▼
┌───────────────┐
│ 冷启动判断器   │ ← 交互次数 < 阈值(如5次)
└───────┬───────┘
        │
    ┌───┴───┐
    ▼       ▼
冷启动分支   常规分支
    │
    ▼
┌─────────────────────────────────┐
│ 实时Feature Store读取注册信息     │
│ 触发：热门召回 + I2I + 运营配置   │
└─────────────────────────────────┘
        │
        ▼
┌─────────────────────────────────┐
│ 轻量级排序模型（减少特征依赖）     │
│ 特征：content_features + context │
│ 弱化：user_history_features      │
└─────────────────────────────────┘
```

## 场景2：物品冷启动

### 📌 背景分析

新物品（New Item）冷启动的核心矛盾：**好内容需要曝光才能被发现，但没有曝光就没有数据证明自己**。

**业务影响**：
- 创作者生态：新创作者如果内容得不到曝光会流失
- 内容 freshness：用户需要看到新内容，否则平台老化
- 长尾分发：冷启是打破"马太效应"的关键

**技术难点**：
- 无点击数据，无法进入常规召回池
- 内容侧特征充足，但交互侧特征缺失
- 需要快速判断内容质量（"测款"）

---

### 🧠 完整解题思路

#### 1. 内容理解先行

**多模态内容分析**：
```
┌────────────────────────────────────────────┐
│           新物品内容理解pipeline             │
├────────────────────────────────────────────┤
│  文本：标题/描述 → BERT提取语义向量           │
│  图像：封面图 → ResNet/ViT提取视觉特征        │
│  视频：关键帧+音频 → 多模态融合               │
│  标签：分类模型打标签（类目/风格/质量分）      │
│  创作者：作者历史表现加权（老作者信任度更高）   │
└────────────────────────────────────────────┘
```

**质量初筛**：
- 标题党/低俗内容过滤（先审后发/先发后审）
- 原创度检测（避免搬运）
- 基础质量分（清晰度、完整度）

#### 2. 冷启流量分配机制

**保量机制（流量扶持）**：
```python
# 新物品获得基础曝光quota
def get_exposure_quota(item_age, author_level, content_quality):
    """
    item_age: 物品上架时间（小时）
    author_level: 作者等级（新人/普通/KOL）
    content_quality: 内容质量分
    """
    base_quota = 1000 if item_age < 24 else 500
    author_boost = 2.0 if author_level == "new_creator" else 1.0
    quality_boost = content_quality / 50  # 质量好的多给
    
    return base_quota * author_boost * quality_boost
```

**流量池分级**：
```
Level 0（冷启动池）：0-100曝光 → 判断CTR是否>阈值
        ↓
Level 1（测试池）：100-1000曝光 → 判断完播率/互动率
        ↓
Level 2（潜力池）：1000-10000曝光 → 进入常规推荐
        ↓
Level 3（热门池）：全量推荐
```

#### 3. 基于内容的召回策略

**内容相似召回（Content-based）**：
```
新物品内容特征 ──→ 相似热门物品 ──→ 推给喜欢热门物品的用户
```

**实现方式**：
- 向量召回：用内容向量找相似Item的受众
- 标签召回：相同标签的Item，共享用户群
- 知识图谱：实体链接，扩展相关用户

#### 4. 探索性曝光（Exploration）

**Thompson Sampling / UCB**：
```python
# UCB: Upper Confidence Bound
# 对不确定性高的物品给予额外奖励
def ucb_score(click_rate, exposure_count, total_rounds):
    exploitation = click_rate
    exploration = sqrt(2 * log(total_rounds) / exposure_count)
    return exploitation + exploration

# 新物品exposure_count小，exploration项大，优先曝光
```

**Bandit算法应用**：
- 上下文Bandit（LinUCB）：结合用户特征做个性化探索
- 多臂老虎机：平衡探索新物品 vs 利用已知好物品

#### 5. 冷启评估指标

```
短期指标（小时级）：
- 冷启曝光率：新物品获得曝光的比例
- 冷启点击率：新物品的CTR vs 大盘CTR
- 达标率：进入下一流量池的比例

中期指标（天级）：
- 冷启转化成功率：从Level 0 → Level 3的比例
- 新作者留存：获得正向反馈的作者比例

长期指标（周级）：
- 内容生态健康度：头部/腰部/尾部内容分发比例
- 创作者满意度：新创作者投稿意愿
```

#### 6. 特殊策略

**新作者保护**：
- 独立流量池，不与头部作者直接竞争
- 冷启失败后的二次机会（3天后重新进入冷启池）

**社交冷启**：
- 先推给作者粉丝（如果有）
- 推给相似作者的粉丝

**跨域迁移**：
- 同一物品在不同场景冷启（Feed vs 搜索 vs 相关推荐）
- 一个场景表现好，数据迁移到其他场景

**一句话总结**：内容理解做基础，分级保量给机会，EE平衡探潜力。

## 场景3：黑产/刷量检测

### 📌 背景分析

黑产刷量是推荐系统的"毒瘤"，直接影响：**广告主利益、平台收入、用户体验、数据质量**。

**常见刷量类型**：
```
┌─────────────────────────────────────────────────────────┐
│                    刷量行为分类                          │
├─────────────────────────────────────────────────────────┤
│  1. 机器刷量：脚本/模拟器自动点击、曝光                    │
│  2. 众包刷量：真人兼职平台，人工完成点击任务               │
│  3. 协议刷量：直接伪造请求，绕过客户端                     │
│  4. 诱导点击：隐藏按钮、误触设计                          │
│  5. 刷榜刷粉：批量关注/点赞，提升账号权重                  │
└─────────────────────────────────────────────────────────┘
```

**损失评估**：
- 广告主：CPM/CPC计费场景，刷量直接导致资金损失
- 平台：推荐模型被污染，用户体验下降
- 创作者：虚假数据干扰内容策略

---

### 🧠 完整解题思路

#### 1. 刷量识别特征体系

**设备维度**：
```
- 设备指纹：IMEI、MAC、Android ID 异常（重复、伪造）
- 设备型号集中度：同一机型大量账号
- 设备环境：模拟器检测（传感器缺失、CPU信息异常）
- Root/越狱检测
```

**IP维度**：
```
- IP聚集度：大量点击来自同一IP/网段
- IP信誉库：代理IP、IDC机房IP黑名单
- 地理位置跳变：1分钟内从"北京"跳到"深圳"
- IP与GPS不匹配
```

**行为维度**：
```
- 点击频率：固定间隔点击（机器人特征）
- 行为轨迹：无滑动直接点击、点击位置固定
- 页面停留：点击后秒关（无真实浏览）
- 转化链路：只点击不转化（无下单/下载）
```

**账号维度**：
```
- 注册时间：新注册账号异常活跃
- 社交关系：僵尸粉特征（关注多互动少）
- 行为模式：只点击广告，不看正常内容
```

#### 2. 检测模型架构

**规则引擎（第一层，实时）**：
```python
rules = {
    "high_frequency": "同设备1分钟点击>10次",
    "ip_cluster": "同IP下账号数>5",
    "click_without_view": "曝光时间<100ms就点击",
    "simulator": "设备指纹命中模拟器库",
    "proxy_ip": "IP命中代理池",
}
# 命中规则直接拦截或降权
```

**机器学习模型（第二层，准实时）**：
```
特征：设备特征 + 行为序列 + 历史统计
模型：GBDT/XGBoost（可解释性强，方便运营审核）
输出：风险分（0-1）
阈值：>0.9标记高风险，人工复核
```

**图神经网络（第三层，离线）**：
```
构建用户-设备-IP异构图
异常子图检测：发现团伙作案
Louvain聚类：识别黑产社群
```

#### 3. 对抗刷量的工程手段

**前端防护**：
```javascript
// 行为验证码（无感/滑动）
// 点击热区分析：正常点击在Item中心，刷量可能随机分布
// 埋点增强：加密传输、防篡改
```

**后端校验**：
```python
# 曝光校验：必须真正渲染才计数
# 点击防重：同一用户短时间内重复点击只算1次
# 链路校验：曝光→点击→转化的时间间隔合理性
```

**智能限流**：
```python
def smart_rate_limit(device_id, user_id, action):
    """
    动态限流：正常用户宽松，可疑用户严格
    """
    risk_score = get_risk_score(device_id, user_id)
    base_limit = 100  # 正常用户每小时100次点击
    
    if risk_score > 0.8:
        return base_limit * 0.1  # 严重限流
    elif risk_score > 0.5:
        return base_limit * 0.5
    return base_limit
```

#### 4. 应对策略（发现后的处理）

**实时处理**：
```
- 直接拦截：请求返回空结果或默认值
- 标记降权：点击不计入计费/推荐模型
- 延迟计费：先扣费，确认正常后结算
```

**事后处理**：
```
- 数据回刷：识别出的刷量从历史数据中剔除
- 模型修正：用clean数据重新训练
- 资金追回：对广告主进行赔付
```

**对抗升级**：
```
- 黑产会进化（换设备、换IP、模拟真实行为）
- 建立红蓝对抗机制，持续迭代策略
- 与行业内共享黑产情报
```

#### 5. 业务层面的平衡

**误杀控制**：
```
- 网吧/公共WiFi场景：同IP多用户是正常行为
- 羊毛党 vs 真实优惠用户：如何区分？
- 策略灰度：新策略先小流量，观察误杀率
```

**分级处置**：
```
疑似（0.5-0.7）：正常服务，但数据不计入模型
可疑（0.7-0.9）：限流，人工审核
确认（>0.9）：封号，关联账号连坐
```

## 场景4：负反馈处理

### 📌 背景分析

用户主动表达「不感兴趣」是高价值信号，比「曝光未点击」的置信度更高。

**负反馈类型**：
```
┌─────────────────────────────────────────────────────────┐
│                    负反馈分类                            │
├─────────────────────────────────────────────────────────┤
│  1. 内容级："不感兴趣"（对该Item的负反馈）                │
│  2. 作者级："屏蔽此作者"                                 │
│  3. 类别级："减少此类内容"（如"减少美食"）                │
│  4. 原因级："内容重复""质量差""不喜欢UP主"等              │
│  5. 隐式负反馈：快速滑动、短停留、举报                    │
└─────────────────────────────────────────────────────────┘
```

**处理难点**：
- 如何避免"过度反应"？（用户可能只是对当前Item不爽，不代表不喜欢整个类别）
- 如何平衡探索与利用？（负反馈后完全不推可能导致信息茧房）
- 如何设计衰减机制？（用户的兴趣会变化，昨天的负反馈今天可能失效）

---

### 🧠 完整解题思路

#### 1. 负反馈的分层处理

**即时生效层（在线）**：
```python
# 同一session内立即生效
if user_click_dislike(item_id):
    # 1. 该Item立即从推荐队列移除
    current_queue.remove(item_id)
    
    # 2. 相似Item降权（但不完全过滤）
    for similar_item in get_similar_items(item_id, top_k=10):
        similar_item.score *= 0.5  # 降权50%
```

**短期生效层（实时Feature Store）**：
```python
# 写入用户实时画像，影响当天推荐
dislike_record = {
    "user_id": user_id,
    "item_id": item_id,
    "author_id": item.author_id,
    "category": item.category,
    "tags": item.tags,
    "timestamp": now(),
    "reason": reason,  # 用户选择的不喜欢原因
}
write_to_feature_store(dislike_record)
```

**长期生效层（模型训练）**：
```python
# 作为负样本参与模型训练
# 注意：负反馈样本权重比普通曝光未点击更高
train_sample = {
    "user": user_features,
    "item": item_features,
    "label": 0,  # 明确负样本
    "weight": 3.0,  # 加权
}
```

#### 2. 不同维度的负反馈策略

**Item维度**：
```
- 该Item永不展示（或N天内不展示）
- 相似Item降权（用I2I相似度控制影响范围）
```

**Author维度**：
```
- 屏蔽作者：该作者所有内容过滤
- 降权作者：该作者内容排序分降低
- 软屏蔽：减少频率，但不完全过滤（给用户后悔机会）
```

**Category/Tag维度**：
```python
def apply_category_penalty(user_id, disliked_category):
    """
    类别级负反馈需要更谨慎，避免过度反应
    """
    # 检查该类别历史正反馈情况
    history_clicks = get_category_click_count(user_id, disliked_category)
    
    if history_clicks > 10:
        # 该用户历史上喜欢过此类别，可能只是不喜欢当前Item
        penalty_factor = 0.8  # 轻微降权
    else:
        # 确实不喜欢该类别
        penalty_factor = 0.3  # 大幅降权
    
    return penalty_factor
```

#### 3. 负反馈的衰减机制

```python
def calculate_dislike_weight(dislike_timestamp, current_timestamp):
    """
    负反馈权重随时间衰减
    """
    hours_passed = (current_timestamp - dislike_timestamp) / 3600
    
    # 指数衰减
    decay_factor = exp(-hours_passed / 24)  # 24小时半衰期
    
    return max(decay_factor, 0.1)  # 最小保留10%影响

# 使用示例
total_penalty = sum(
    calculate_dislike_weight(d.time, now()) * d.base_weight 
    for d in user_dislike_history
)
```

#### 4. 误操作恢复机制

```python
# 允许用户"撤销"不喜欢
if user_undo_dislike(item_id):
    remove_from_dislike_list(item_id)
    
    # 恢复该类别的推荐权重
    category = get_item_category(item_id)
    restore_category_weight(user_id, category)
```

**自动恢复策略**：
```
- 用户连续点击同类正反馈后，自动降低负反馈权重
- 周期性（如30天）清理历史负反馈记录
```

## 场景5：推荐结果变差定位

### 📌 背景分析

线上推荐效果突然下降是算法工程师最头疼的问题。需要快速定位根因，减少业务损失。

**常见表现**：
```
- CTR/CVR突然下降（小时级/天级）
- 用户反馈增多（"推荐的都是什么鬼"）
- 核心指标报警（DAU下降、留存下降）
```

**排查难点**：
- 推荐系统链路长（数据→特征→模型→排序→展示），问题可能出在任意环节
- 线上环境复杂（AB实验、灰度发布、外部因素）
- 需要快速定位，业务压力巨大

---

### 🧠 完整解题思路

#### 1. 问题确认（先确认是不是真有问题）

- 数据口径是否变化？（埋点是否变更？）

步骤2：排除统计误差
- 样本量是否足够？（小流量波动是正常的）
- 是否是周期性波动？（工作日vs周末，白天vs晚上）
- 同比/环比对比

步骤3：确认影响范围

## 场景6：双塔模型线上线下不一致

### 📌 背景分析

双塔模型离线AUC很高，但线上CTR没提升，这是推荐系统经典难题。

**常见表现**：
```
- 离线AUC：0.85 → 线上CTR：持平或下降
- 离线NDCG：大幅提升 → 线上停留时长：无变化
- 训练loss正常收敛 → 线上推理结果异常
```

**不一致根因分类**：
```
1. 数据层面：样本选择偏差、特征穿越、数据泄露
2. 特征层面：离线在线特征不一致、特征缺失处理不同
3. 模型层面：模型导出错误、推理代码bug、精度损失
4. 评估层面：离线评估指标与线上业务指标不一致
```

---

### 🧠 完整解题思路

#### 1. 数据层面排查

**样本选择偏差（Sample Selection Bias）**：
```
问题：离线训练用「曝光」样本，线上是「全库」候选

解决方案：
- 加入随机曝光样本（一定比例的探索流量）
- 使用IPS（Inverse Propensity Score）加权
- 考虑用因果推断方法（如DR、Doubly Robust）
```

**特征穿越（Data Leakage）**：
```python
# 错误示例：用未来信息预测过去
# 比如：用「点击后停留时长」作为特征预测「是否点击」
# 这实际上是不可获取的

features = {
    "user_age": user.age,
    "item_price": item.price,
    # "dwell_time_after_click": item.avg_dwell_time,  # ❌ 穿越特征
}
```

**训练/推理数据分布不一致**：
```
检查项：
- 时间窗口：训练和推理的数据周期是否一致？
- 用户分布：训练是否包含所有用户群？
- 物品分布：新物品在训练中的比例是否合理？
```

#### 2. 特征层面排查

**离在线特征一致性检查**：
```python
# 建立特征一致性检查机制
def feature_diff_checker():
    """
    对比离线和在线特征值
    """
    test_cases = load_test_cases()  # 预定义的测试用例

```

**常见不一致原因**：
```
1. 时间戳处理：离线用T-1，在线用实时
2. 空值填充：离线填0，在线填-1
3. 编码方式：离线用LabelEncoder，在线直接hash
4. 特征拼接：离线字段顺序和在线不一致
5. 归一化：离线用全局统计，在线用局部统计
```

**特征重要性分析**：
```python

```

**模型导出检查**：

```

- 向量相似度计算：在线用FAISS近似搜索 vs 离线精确计算
```

#### 4. 评估层面排查

**离线指标 vs 线上业务指标**：
```
问题：AUC高不代表CTR高

原因：
1. AUC衡量排序能力，但线上还有位置偏差、展示策略影响
2. 离线评估是全库排序，线上是召回后排序
3. 离线用历史数据，线上是实时分布

解决方案：
- 离线模拟线上流程（召回→排序→展示）
- 使用和业务指标更相关的评估指标（如GAUC、NDCG）
- 离线预估CTR，与线上对比校准
```

**位置偏差（Position Bias）**：
```python

```
