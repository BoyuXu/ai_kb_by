## 题目1：从0设计一个短视频推荐系统（类抖音）

### 1.1 需求拆解

#### 业务场景
- 用户滑动浏览短视频，每次曝光1个视频
- 核心目标：最大化用户观看时长、互动率（点赞/评论/分享）、DAU留存

#### 量级估算
| 指标 | 估算值 | 说明 |
|------|--------|------|
| DAU | 3亿 | 抖音级别 |
| 人均日刷视频 | 200条 | 深度用户更多 |
| 日曝光总量 | 600亿 | 3亿 × 200 |
| 峰值QPS | 100万 | 按2倍日平均 |
| 候选视频池 | 1亿+ | 日新增千万级 |
| 用户特征存储 | PB级 | 行为日志海量 |

#### 核心挑战
1. **极致延迟**：单次请求<200ms（端到端）
2. **海量候选**：从亿级视频筛选
3. **实时性**：新视频、新行为秒级感知
4. **多样性**：避免信息茧房

### 1.3 每层模型选型与理由

#### 召回层：多路召回架构

| 召回通道 | 模型/算法 | 适用场景 | 选型理由 |
|---------|----------|---------|---------|
| **协同过滤** | ItemCF / UserCF | 热门偏好 | 简单有效，捕捉共现关系 |
| **向量召回** | DSSM / YoutubDNN | 语义匹配 | 泛化能力强，支持超长序列 |
| **图召回** | PinSAGE / LightGCN | 高阶关系 | 挖掘社交/内容关系图谱 |
| **热门/运营** | 规则策略 | 冷启动/热点 | 补充新内容曝光 |
| **关注/社交** | 时间序排序 | 社交关系 | 强信号，优先保障 |

**向量召回详细设计**：
```python
# 双塔结构 (DSSM variant)
class TwoTowerRecall:
    def user_tower(self, user_features):
        # 输入: 用户画像 + 长期兴趣 + 短期行为序列
        seq_emb = TransformerEncoder(user_behavior_sequence)
        profile_emb = DenseLayer(user_profile)
        return concat([seq_emb, profile_emb])
    
    def item_tower(self, video_features):
        # 输入: 视频ID + 作者信息 + 内容标签 + 封面图
        id_emb = Embedding(video_id)
        tag_emb = TagEmbedding(tags)
        visual_emb = ResNet50(cover_image)
        return concat([id_emb, tag_emb, visual_emb])
    
    # 内积计算相似度，ANN检索 (HNSW/FAISS)
    def similarity(self, user_vec, item_vec):
        return cosine_similarity(user_vec, item_vec)
```

**为什么选双塔而非单塔？**
- 双塔：用户/物品向量独立计算，可离线预存，检索快
- 单塔：交互早，特征融合充分，但只能逐个打分，太慢
- 召回层追求速度，双塔是标准选择

#### 粗排层：轻量双塔/DNN

```
输入特征 (精简版):
- 用户侧: 用户ID emb + 年龄/性别/地域 + 活跃时间段
- 视频侧: 视频ID emb + 类目emb + 作者等级 + 历史统计CTR
- 交叉: 用户-类目偏好 + 用户-作者偏好

模型: 3-4层DNN (256→128→64→1)
损失: 加权BCE (按观看时长加权)
```

#### 精排层：多目标深度学习

**为什么需要多目标？**
- 单一CTR预估会导致"标题党"问题
- 抖音核心优化: 观看时长 + 完播率 + 互动(点赞/评论/转发)

**MMoE架构 (Multi-gate Mixture-of-Experts)**：
```
                         ┌─────────┐
    ┌───────────────────►│ 专家1   │
    │                    │ (CTR)   │
    ├───────────────────►│ 专家2   │
    │   共享层           │ (时长)  │
输入 ───► MMoE ─────────►│ 专家3   │──► Gate ──► 任务1 (pCTR)
    │   网络             │ (完播)  │      │
    ├───────────────────►│ 专家4   │      └──► 任务2 (pWatchTime)
    │                    │ (互动)  │           任务3 (pFinish)
    └───────────────────►│ 专家5   │           任务4 (pLike)
                         └─────────┘
```

**特征工程 (精排层)**：
| 特征类别 | 具体特征 | 处理方式 |
|---------|---------|---------|
| 用户画像 | 年龄/性别/地域/设备 | Embedding |
| 用户行为 | 近7天观看序列(1000+) | DIN/Transformer注意力 |
| 视频特征 | ID/标签/作者/时长 | Embedding + Dense |
| 上下文 | 时间/网络/位置 | One-hot/Emb |
| 交叉特征 | 用户标签×视频标签 | 模型自动学习 |
| 统计特征 | 视频历史CTR/完播率 | 分桶/归一化 |

**DIN (Deep Interest Network) 核心思想**：
```python
# 传统方法: 用户行为序列简单Pooling
user_interest = mean([emb_vid1, emb_vid2, ..., emb_vidN])

# DIN: 候选视频与历史行为做注意力
attention_weights = softmax(
    candidate_video_emb · [emb_vid1, emb_vid2, ...]
)
user_interest = sum(attention_weights * [emb_vid1, emb_vid2, ...])
```

#### 重排层：业务规则与多样性

```
输入: 精排层输出的Top 50视频
处理:
1. 多样性控制 (MMR算法)
   - 相似视频去重 (视频内容相似度)
   - 作者打散 (同一作者间隔>5)
   - 标签多样性 (类目分布均衡)

2. 新鲜度加权
   - 新视频流量扶持 (Explore)
   - 用户已看相似内容降权

3. 业务策略
   - 广告插入 (不影响用户体验位置)
   - 运营内容加权
   - 低质/违规内容过滤

输出: 最终展示的Top 10-20视频
```

**MMR (Maximal Marginal Relevance)**：
```
MMR = λ · Relevance(i) - (1-λ) · max Sim(i, j)
       └─精排分数─┘         └─与已选视频最大相似度─┘
```

### 2.2 模型架构选型

#### 模型演进路线
```
LR (2010) → FM/FFM (2012) → Wide&Deep (2016) → DeepFM (2017) 
         → DCN (2017) → DIN (2018) → DIEN (2019) → AutoInt (2019)
         → MMOE (2018) → PLE (2020) → 大模型时代
```

#### 工业界主流架构: DeepFM + DIN

```
Input Layer
    │
    ├─── 稀疏特征 (Sparse) ─── Embedding Layer
    │                              │
    ├─── 连续特征 (Dense) ────────┤
    │                              ▼
    │                        Feature Embedding
    │                              │
    ▼                              ▼
┌──────────────────────────────────────────────────────────┐
│                      FM Component                         │
│  y_FM = w₀ + Σwᵢxᵢ + ΣΣ<vᵢ,vⱼ>xᵢxⱼ                        │
│  └─一阶─┘  └────二阶交叉────┘                             │
└──────────────────────────────────────────────────────────┘
    │
    ▼
┌──────────────────────────────────────────────────────────┐
│                     Deep Component                        │
│  Input → Dense(512) → Dense(256) → Dense(128) → Dense(1) │
│              ↓           ↓          ↓                    │
│            BN+ReLU     Dropout     ...                   │
└──────────────────────────────────────────────────────────┘
    │
    ▼
Output: pCTR = sigmoid(y_FM + y_Deep)
```

**为什么选DeepFM？**
| 优势 | 说明 |
|-----|------|
| 端到端 | 无需手工特征交叉 |
| 低阶+高阶 | FM处理二阶交叉，DNN处理高阶非线性 |
| 工业验证 | 大量公司生产环境使用 |
| 推理高效 | 一次前向传播，<10ms |

#### 序列建模: DIN (Deep Interest Network)

```
用户行为序列: [ad_A, ad_B, ad_C, ad_D, ...]
                      │
                      ▼
              ┌───────────────┐
              │  注意力机制    │
              │               │
候选广告 ─────►│  attention    │────► 加权兴趣向量
  emb        │  = f(ad, ad_i) │
              │               │
              └───────────────┘
                      │
                      ▼
              与候选广告相关的兴趣表达
```

**DIN核心公式**：
```
v_u = Σᵢ a(vᵢ, v_a) · vᵢ

其中: a(vᵢ, v_a) = softmax(MLP([vᵢ, v_a, vᵢ-v_a, vᵢ⊙v_a]))
      └───── 权重 ────┘  └───────── 输入包括: emb, 差值, 逐元素积 ─────┘
```

#### 多任务学习: ESMM/MMoE

**CVR预估的延迟反馈问题**：
```
问题: 点击→转化 可能有数天延迟
      直接训练CVR模型，Label不准确

ESMM (Entire Space Multi-Task Model) 解法:

        pCTR          pCVR
用户 ──────► 点击 ──────► 转化
      │                    │
      └───── pCTCVR ───────┘
      
pCTCVR = pCTR × pCVR

损失函数: L = L_CTR + L_CTCVR
         (使用全部曝光样本训练CTR，用点击样本的转化作为CTCVR监督)
```

### 2.4 延迟反馈问题处理

#### 问题定义
```
点击时间 ───────────────────────────────────────► 转化时间
    │                                              │
    │         延迟窗口 (几分钟 ~ 几天)               │
    │                                              │
    ▼                                              ▼
   可立即获得                                       延迟获得
   训练时Label=0?                                  真实Label=1
```

#### 解决方案

**1. 等待窗口法 (Waiting Window)**
```
做法: 只使用 延迟窗口外的样本 训练
      延迟窗口内的样本 暂不使用

优点: Label准确
缺点: 样本利用率低，模型时效性差
适用: 延迟短的场景 (<1小时)
```

**2. 重要性采样法 (Importance Sampling)**
```
对延迟样本加权，纠正分布偏差

原始分布: p(y=1|x)
观测分布: p_obs(y=1|x) = p(y=1|x) × p(observed|y=1,x)

权重: w = 1 / p(observed|y=1,x)

实现: 预测"样本是否已观察到转化"的概率模型
```

**3. 多任务学习 (ESMM)**
```
不直接建模CVR，而是建模CTCVR = CTR × CVR

损失: L = L_CTR + α × L_CTCVR

优势: CTCVR可以在曝光时立即确定Label (转化与否)
      无需等待转化发生
```

**4. 延迟模型 (Funnel-based)**
```
建模转化延迟分布:
p(conversion|click, delay=t) = λ × exp(-λt)

在线预测时，对近期点击进行"预期转化"修正:
expected_conversion = observed + expected_later
```

### 3.1 Query理解

#### 整体流程
```
用户Query: "iphone 15 pro max 256g 深空黑"
    │
    ├──► 预处理
    │    └── 繁简转换、大小写统一、去停用词
    │
    ├──► 分词
    │    └── iphone | 15 | pro | max | 256g | 深空黑
    │
    ├──► 词权重
    │    └── iphone(高) | 15(高) | pro(中) | max(中) | 256g(中) | 深空黑(低)
    │
    ├──► 意图识别
    │    └── 类目: 手机数码 > 手机 > 智能手机
    │        属性: 品牌=Apple, 型号=iPhone15PM, 容量=256GB, 颜色=深空黑
    │        意图: 购买意图 (高)
    │
    └──► Query改写
         ├── 纠错: ipone → iphone
         ├── 扩展: 同义词 (手机/移动电话/Cell Phone)
         └── 丢词: 非核心词降级
```

#### 分词技术
| 方法 | 代表 | 适用场景 |
|-----|------|---------|
| 词典分词 | 最大匹配、MMSEG | 垂直领域、词典完善 |
| 统计分词 | HMM、CRF | 通用场景 |
| 深度学习 | BERT分词、BiLSTM-CRF | 复杂边界、新词发现 |

#### 意图识别
```python
# 多任务分类
class IntentClassifier:
    def __init__(self):
        self.bert = BertModel()
        
    def forward(self, query):
        cls_emb = self.bert(query).pooler_output
        
        # 类目预测 (多级)
        cate1 = softmax(Linear(cls_emb, num_cate1))  # 一级类目
        cate2 = softmax(Linear(cls_emb, num_cate2))  # 二级类目
        cate3 = softmax(Linear(cls_emb, num_cate3))  # 叶子类目
        
        # 意图强度
        intent = sigmoid(Linear(cls_emb, 1))  # 0-1购买意图
        
        # 属性抽取 (序列标注)
        attrs = CRF(self.bert(query).last_hidden_state)
        
        return {
            'category': (cate1, cate2, cate3),
            'intent': intent,
            'attributes': attrs
        }
```

#### Query改写
```
1. 拼写纠错
   "雅诗兰代小综瓶" → "雅诗兰黛小棕瓶"
   方法: 编辑距离 + 语言模型困惑度

2. 同义词扩展
   "手机壳" → "手机保护套" / "phone case"
   方法: 语义相似度模型 + 人工词表

3. 丢词/降权
   "iphone 15 pro max 在哪里买便宜"
   → 核心: "iphone 15 pro max"
   → 意图词: "便宜" (触发促销加权)

4. 类目归一
   "苹果手机" → 类目ID: 50012345
```

### 3.2 召回：BM25 + 向量召回

#### 整体召回架构
```
Query ──────────────────────────────────────►
    │
    ├──► 文本召回 (BM25) ───┐
    │   └── 倒排索引         │
    │                        ├──► 融合排序 ──► TopK
    ├──► 向量召回 (Dense) ───┤
    │   └── ANN索引          │
    │                        │
    └──► 图召回 (GNN) ───────┘
        └── 知识图谱
```

#### BM25原理
```
BM25(q, d) = Σ IDF(qᵢ) · [ f(qᵢ,d) · (k₁+1) ] / [ f(qᵢ,d) + k₁ · (1-b+b·|d|/avgdl) ]
              └─词权重─┘   └───────── TF项 (词频饱和) ─────────────┘

其中:
- IDF(qᵢ) = log( (N - n(qᵢ) + 0.5) / (n(qᵢ) + 0.5) )
- k₁: 控制词频饱和度 (通常1.2-2.0)
- b: 控制文档长度归一化 (通常0.75)
```

**BM25特点**：
| 优点 | 缺点 |
|-----|------|
| 可解释性强 | 词汇不匹配问题 |
| 无需训练 | 无法捕捉语义相似 |
| 高效 | 对长尾Query效果差 |

#### 向量召回 (Dense Retrieval)

**双塔模型架构**：
```
Query Tower                     Doc Tower
    │                               │
    ▼                               ▼
┌─────────┐                   ┌─────────┐
│  BERT   │                   │  BERT   │
│ Encoder │                   │ Encoder │
└────┬────┘                   └────┬────┘
     │                             │
     ▼                             ▼
  [CLS] token emb              [CLS] token emb
     │                             │
     ▼                             ▼
  MLP Projection              MLP Projection
     │                             │
     ▼                             ▼
   q_vector                      d_vector
     │                             │
     └─────────► 相似度计算 ◄───────┘
                   │
                   ▼
              Score = cos(q, d)
```

**训练目标 (Contrastive Loss)**：
```
L = -log( exp(sim(q,d⁺)/τ) / Σ exp(sim(q,dᵢ)/τ) )
    
其中: d⁺ 是正样本 (点击的doc)
      dᵢ 包含负样本 (in-batch negatives)
      τ 是温度系数
```

### 3.3 排序：Learning to Rank (LTR)

#### LTR三类方法
```
1. Pointwise (单点法)
   └── 把排序问题转化为回归/分类问题
   └── 预测: 每个doc的相关性分数
   └── 代表: GBDT回归、DNN回归
   └── 缺点: 忽略文档间顺序关系

2. Pairwise ( pairwise法)
   └── 预测: 文档对的相对顺序
   └── 代表: RankSVM、LambdaRank
   └── 缺点: 不直接优化NDCG等评估指标

3. Listwise (列表法) ⭐ 工业界主流
   └── 预测: 整个列表的排序
   └── 直接优化NDCG、MAP等
   └── 代表: LambdaMART、ListNet
```

#### 工业界实践：GBDT + LR → Deep Model

**演进路径**：
```
2014: Facebook GBDT + LR
      └── GBDT做特征组合，LR做最终分类
      
2016: Wide & Deep (Google)
      └── 人工特征 + DNN自动学习
      
2018+: 纯深度模型
      └── DIN → DIEN → BST
```

**深度排序模型架构**：
```
Input Features
    │
    ├── 离散特征 ───► Embedding
    │
    ├── 连续特征 ───► 分桶/归一化
    │
    ├── 序列特征 ───► DIN注意力 ────┐
    │    (Query-Item交互)            │
    │                               │
    └── 交叉特征 ───► FM/DCN ───────┤
                                   ▼
┌──────────────────────────────────────────────────────────────┐
│                    Deep Network                               │
│  Dense(512) → BN → ReLU → Dropout →                          │
│  Dense(256) → BN → ReLU → Dropout →                          │
│  Dense(128) → BN → ReLU → Dropout → Dense(1)                 │
└──────────────────────────────────────────────────────────────┘
                              │
                              ▼
                       Sigmoid → pRelevance
```

#### 多目标优化
```
搜索场景的多目标：
├── 相关性: Query-Doc匹配度
├── 点击率: pCTR
├── 转化率: pCVR (电商)
├── 停留时长: Dwell Time (内容)
└── 满意度: 用户反馈信号

模型: Shared-Bottom / MMoE / PLE

最终分数: Score = α·rel + β·pCTR + γ·pCVR + δ·quality
          (超参通过离线NDCG + 在线AB调优)
```

### 3.4 相关性 vs 个性化平衡

#### 核心矛盾
```
相关性 (Relevance)              个性化 (Personalization)
    │                                │
    ▼                                ▼
"Query说的是什么"              "这个用户想要什么"
    │                                │
文本匹配度                       用户画像匹配
通用性好                         千人千面
结果权威                         发现惊喜
    │                                │
    └─────────── 如何平衡 ───────────┘
```

#### 融合策略

**1. 分层融合**
```
第一层: 强相关性过滤
        └── 相关性分数 < 阈值，直接过滤
        
第二层: 个性化重排序
        └── 相关性达标的基础上，个性化调整
        
Score = f(relevance) × g(personalization)
        │
        └── f: 非线性变换，低相关性大幅降权
```

**2. 特征层面融合**
```
模型输入:
├── Query特征 (文本语义)
├── Doc特征 (内容质量)
├── 交互特征 (Query-Doc匹配)
├── 用户特征 (偏好画像) ◄── 个性化信号
└── 上下文特征 (时间/位置)

模型自动学习各特征权重
```

**3. 多路召回融合**
```
召回层:
├── 文本召回 (强相关性) ──── 权重高
├── 向量召回 (语义匹配) ──── 权重中
├── 个性化召回 (用户历史) ─── 权重低 (需相关性校验)
└── 热门召回 (兜底) ──────── 权重低

融合: RRF (Reciprocal Rank Fusion)
      score = Σ 1/(k + rankᵢ)
```

#### 个性化程度控制
```
Query类型分析:
┌─────────────────┬────────────────────────────────┐
│ Query类型       │ 个性化策略                     │
├─────────────────┼────────────────────────────────┤
│ 导航类          │ 低个性化，优先官方结果         │
│ "淘宝官网"      │                                │
├─────────────────┼────────────────────────────────┤
│ 事实类          │ 低个性化，权威结果优先         │
│ "姚明身高"      │                                │
├─────────────────┼────────────────────────────────┤
│ 交易类          │ 高个性化，偏好匹配             │
│ "连衣裙"        │ 风格/价格/品牌偏好             │
├─────────────────┼────────────────────────────────┤
│ 内容类          │ 中个性化，兴趣相关             │
│ "机器学习教程"  │ 难度/形式/来源偏好             │
└─────────────────┴────────────────────────────────┘
```

## 面试技巧总结

### 答题框架 (STAR-R)
```
S - Scenario (场景): 明确业务场景和目标
T - Trade-off (权衡): 讨论设计取舍
A - Architecture (架构): 分层设计，由粗到细
R - Reasoning (理由): 每个选择都有数据/理论支撑
R - Robustness (鲁棒性): 考虑异常、扩展、监控
```

### 常见追问
| 问题 | 答题要点 |
|-----|---------|
| 如何评估推荐效果？ | 离线AUC/GAUC → 在线AB → 长期留存 |
| 模型效果差怎么办？ | 特征工程 → 模型调优 → 数据质量 → 业务理解 |
| 如何处理冷启动？ | 内容冷启 → 探索机制；用户冷启 → 热门+兴趣探针 |
| 如何保证实时性？ | 在线特征服务 + 流式模型更新 |
| 推荐去重怎么做？ | 用户短期/长期历史 + 相似内容检测 |
