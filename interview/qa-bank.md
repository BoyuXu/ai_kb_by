### Q1 | 难度: 初级 | Mamba SSM 基础

**Q: 请解释 Mamba（选择性状态空间模型）相比 Transformer 的核心优势，以及为什么它特别适合推荐系统的超长用户行为序列建模？**

**A:**

**直接结论：** Mamba 以 O(n) 线性复杂度替代 Transformer 的 O(n²) 注意力计算，其「选择性遗忘」机制与推荐场景「用户历史稀疏有价值信号」的本质天然契合，是超长序列建模从工程不可行到生产可用的关键跨越。

**Mamba 核心机制 — 选择性状态空间（SSM）：**

传统 SSM 的状态转移方程：
```
h_t = A·h_{t-1} + B·x_t     # 状态更新（h 为隐状态）
y_t = C·h_t                  # 输出
```

Mamba 的改进：让 A、B、C **依赖于输入 x_t**（Selection Mechanism）：
```
Δ_t, B_t, C_t = Linear(x_t)   # 参数动态化
h_t = exp(Δ_t A)·h_{t-1} + Δ_t·B_t·x_t
```

- **Δ_t 大**：模型"专注"于当前输入，大量更新隐状态
- **Δ_t 小**：模型"遗忘"当前输入，保持历史状态不变

**与 Transformer 的对比：**

| 维度 | Transformer | Mamba |
|------|------------|-------|
| 计算复杂度 | O(n²) 注意力矩阵 | O(n) 顺序扫描 |
| 序列长度上限 | ~4096（显存瓶颈） | 理论无限 |
| 历史信息利用 | 全局注意力（平等对待所有位置） | 选择性记忆（动态权衡保留/遗忘） |
| 并行训练 | 完全并行 | 并行扫描（Parallel Scan） |
| 推理显存 | O(n) KV Cache | O(1) 固定大小状态 |
| 适合场景 | 短序列语义理解 | 长序列时序建模 |

**推荐场景的天然契合：**

用户行为序列特征：「大部分不相关，少量关键信号稀疏分布」
- 用户3个月前购买过的品类 → 对季节性商品至关重要（需要长期记忆）
- 用户昨天随机浏览的内容 → 可能是噪声（需要遗忘）

Mamba 的选择性遗忘直接解决这个问题，而 Transformer 的注意力机制无法区分"值得记住的历史"和"噪声行为"。

**工程落地（MambaRec方案）：**
- 用户行为序列分段压缩（短/中/长期各段轻量Transformer提取摘要）
- 段级摘要输入 Mamba 主干（保留跨时间段长程依赖）
- 兴趣快照缓存（128维×20时间槽），推理时只加载快照
- 延迟：280ms → 45ms，CTR +6.8%，季节性品类 GMV +23%

**面试官点评：** 考察 SSM 基础理论与工程结合。高分要点：① O(n²) vs O(n) 的本质区别 ② Δ_t 参数化的选择机制（不只说"更快的Transformer"）③ 推荐场景的业务动机（稀疏有价值信号）④ 加分：O(1) 推理显存 vs O(n) KV Cache 的对比

### Q2 | 难度: 中级 | 延迟CVR标签建模

**Q: 广告CVR预估中的延迟标签问题是什么？请描述生存分析（Survival Analysis）如何建模转化延迟，以及如何设计样本重加权策略解决系统性偏差？**

**A:**

**直接结论：** 延迟标签问题是广告CVR建模中被系统性低估的陷阱——固定窗口策略将大量真实正样本标记为负，造成CVR被低估10-15%。生存分析将"是否转化"和"何时转化"分开建模，在样本到达时给出置信加权，替代粗暴的二值化处理。

**延迟标签问题的本质：**
```
用户点击广告
  │
  ├──── 30分钟内购买 ── 标记为正样本（1）✓
  │
  └──── 30分钟后才购买（42%的转化！）── 错误标记为负样本（0）✗
         ├─ 1-24小时后转化（23%）
         └─ 24小时以上（如家电/汽车，决策周期长）
```

**生存分析框架：**

定义"生存函数" S(t) = P(转化发生在时刻 t 之后 | 已发生转化)
及风险函数（Hazard） h(t) = P(在 t 时刻转化 | 已点击且 t 前未转化)

对每个（广告主类别 × 用户类型 × 品类）组合，用 Weibull 分布或 Cox 比例风险模型拟合：

```python
# Weibull 参数化生存模型
S(t; λ, k) = exp(-(t/λ)^k)    # λ=scale, k=shape
h(t; λ, k) = (k/λ)(t/λ)^(k-1) # 风险率

# 对数据拟合：最大化观测转化时间的似然
L = Π_{i: converted} h(t_i) × S(t_i)   # 已转化样本
  × Π_{j: not yet}    S(w_j)            # 窗口内未转化（右截尾）
```

**样本重加权策略：**

```python
def get_sample_weight(t_elapsed, window=30*60):
    # 窗口内已转化：置信权重=1.0
    if t_elapsed < window and converted:
        return 1.0
    # 延迟样本：按生存概率校正
    elif t_elapsed < 24*3600:
        # w = P(conversion | t_elapsed, context) 
        # 用拟合的生存函数估计当前等待到这里仍未转化的样本中真正不转化的比例
        prob_future_convert = 1 - S(24*3600) / S(t_elapsed)
        return prob_future_convert  # 软标签(0~1)
    # 超长延迟：离线队列批量修正
    else:
        return "deferred_to_offline_queue"
```

**CVR 校正层：**

在模型输出头加入实时修正因子 ε(t_post_launch)：
```
CVR_corrected = CVR_model × (1 + ε(t))
```
其中 ε(t) 在大促开始初期（模型样本少、延迟多）为正，随时间收敛。

**工业效果：** CVR LogLoss 提升 12.3%，出价偏差 15% → 3.2%，平台广告收入 +11.4%

**与其他方案对比：**

| 方案 | 核心思想 | 优点 | 缺点 |
|------|----------|------|------|
| 固定窗口（基线）| 窗口外=负样本 | 简单 | 系统性低估 10-15% |
| ES-DFM（Elapsed-time DFM）| 对每个曝光建独立延迟模型 | 精度高 | 模型数量多，维护成本高 |
| 生存分析（推荐）| 参数化延迟分布+软标签 | 统计严谨，工程可行 | 需要历史转化时间数据 |
| Bi-LSTM延迟预测 | 序列建模每条广告的延迟模式 | 捕捉非参数分布 | 训练复杂，延迟高 |

**面试官点评：** 考察对广告系统训练数据陷阱的深度理解。高分要点：① 明确给出延迟比例数字（42%超30分钟）② 生存函数的数学定义（右截尾的处理）③ 样本重加权的具体公式 ④ 加分：区分"排序性偏差"（AUC不变）和"校准性偏差"（出价影响）

### Q3 | 难度: 中级 | ESMM 解决 CVR 的 SSB 和 DS 问题

**Q: CVR 预估中存在两个核心问题：样本空间偏差（SSB）和数据稀疏（DS）。请解释这两个问题的根本原因，以及 ESMM（Entire Space Multi-task Model）如何通过全空间建模和参数共享同时解决这两个问题？**

**A:**

**直接结论：** SSB 源于 CVR 只在"点击"子空间训练但要在"曝光"全空间预测；DS 源于转化样本极其稀少（CTR≈1%，CVR≈1%，pCTCVR≈0.01%）。ESMM 通过建模 pCTCVR = pCTR × pCVR，将 CVR 的训练空间扩展到全曝光空间，并以 CTR 任务共享 embedding 参数解决稀疏问题。

**SSB（Sample Space Bias）根本原因：**

```
理想：CVR 模型应学习 P(conversion | exposure, x)
实际：CVR 模型只能从「有点击的样本」学，即 P(conversion | click, x)

训练空间：点击集合（impression 的 1%）
推理空间：全曝光（100%）

训练 ≠ 推理空间 → 分布偏移
```

**DS（Data Sparsity）根本原因：**

```
全量曝光样本：10亿/天
有点击的样本：1000万/天（1%）
有转化的样本：10万/天（0.01%）

CVR 模型只能用 10万 正样本学习，而特征空间维度可能是 10^9
```

**ESMM 架构与数学推导：**

```
                    曝光空间 (全量)
                          │
            ┌─────────────┴─────────────┐
            │ CTR Tower               │ CVR Tower
            │ P(click|x)              │ P(cvr|click,x)
            │ [embedding共享]          │ [embedding共享]
            └─────────┬───────────────┘
                      │ element-wise multiply
                      ↓
               P(ctcvr|x) = P(click|x) × P(cvr|click,x)
               
Loss = L_CTR(click, P(click|x)) + L_CTCVR(conversion, P(ctcvr|x))
```

**关键设计决策：**

1. **全空间标签**：CTCVR=1 当且仅当"有点击且有转化"，这个标签在全曝光空间都可定义，解决 SSB。
2. **乘法结构**：通过 pCTCVR = pCTR × pCVR 的分解，CVR Tower 的梯度通过 CTCVR Loss 反传，训练空间隐式扩展到全曝光。
3. **Embedding 共享**：CTR 有大量点击样本，CVR 通过共享底层 embedding 间接学习冷门用户/物品的表示，解决 DS。

**与基线方法对比：**

| 方法 | SSB 处理 | DS 处理 | AUC | 缺点 |
|------|---------|---------|-----|------|
| 独立 CVR 模型 | ❌ 只在点击空间 | ❌ 样本稀疏 | 低 | 系统性偏差 |
| 端到端 pCTCVR 模型 | ✅ 全空间 | 部分✅ | 中 | 无法独立控制 CVR |
| ESMM | ✅ 全空间 | ✅ embedding共享 | 高 | CVR 推理需配合 CTR（乘法结构） |
| ESCM²（进阶）| ✅ | ✅ + Counterfactual正则化 | 更高 | 实现复杂 |

**面试官点评：** 考察多任务学习解决训练偏差的深度理解。高分要点：① SSB vs DS 的区分（两个独立问题）② ESMM 数学推导（乘法分解的必要性）③ Loss 设计（作用在全曝光 CTCVR 上而非点击空间 CVR）④ 加分：ESCM² 的 Counterfactual 正则化扩展，或提到 ESMM 的局限（假设 CTR 和 CVR 独立）

### Q4 | 难度: 高级 | 搜索-广告联合排序系统设计

**Q: 某综合平台搜索结果页同时包含自然结果和竞价广告，两套系统独立运行导致用户体验和商业化的矛盾。请设计一套搜索-广告联合排序（STAR）系统，包括：统一打分框架、动态广告插入机制、用户满意度保底约束，以及如何评估系统效果？**

**A:**

**直接结论：** 搜索-广告的矛盾本质是"统一公平竞争"问题——广告不应"特权插队"而应"赢得位置"。核心设计是统一内容质量打分 + 效用最大化混排 + 满意度保底约束，让相关性高的广告获得展示权，低质量广告被自然淘汰。

**系统架构设计：**

```
┌──────────────────────────────────────────────────────┐
│               Query 输入                              │
└──────────────┬──────────────────────┬────────────────┘
               │                      │
    ┌──────────▼──────────┐  ┌────────▼────────────┐
    │  自然检索引擎        │  │  广告竞价引擎        │
    │  Top-K 自然结果      │  │  Top-M 候选广告      │
    └──────────┬──────────┘  └────────┬────────────┘
               │                      │
               └──────────┬───────────┘
                           │ 候选池合并
              ┌────────────▼─────────────────┐
              │    统一相关性打分层            │
              │  BERT Cross-Encoder           │
              │  output: quality_score (0-1)  │
              └────────────┬─────────────────┘
                           │
              ┌────────────▼─────────────────┐
              │    效用最大化混排引擎          │
              │  score_doc = quality          │
              │  score_ad  = quality × eCPM_w │
              └────────────┬─────────────────┘
                           │
              ┌────────────▼─────────────────┐
              │    满意度保底约束层            │
              │  满意度预测 ≥ baseline(85%)   │
              └────────────┬─────────────────┘
                           │
              ┌────────────▼─────────────────┐
              │         最终排序结果          │
              └──────────────────────────────┘
```

**Layer 1 — 统一相关性打分：**

```python
# 同一套 BERT Cross-Encoder 处理自然结果和广告
quality_score = CrossEncoder(query, content)  # 0~1

# 自然结果：直接使用 quality_score
doc_score = quality_score

# 广告：在质量分基础上乘以出价调整因子（保证质量门槛）
# eCPM_normalized = eCPM / max_eCPM_in_pool
ad_score = quality_score × (1 + α × eCPM_normalized)  # α=0.3，避免出价压制质量
```

**Layer 2 — 效用最大化混排：**

页面整体效用函数：
```
U(π) = β₁ × Σ quality_score_i 
      + β₂ × Σ eCPM_i × is_ad_i 
      - β₃ × Diversity_Penalty(π)
      
s.t. 广告数量 ≤ K_max（避免广告轰炸）
     相邻广告间距 ≥ 2（广告布局规则）
```

贪心序列构建（NP-hard 问题的近似解）：

for position in range(1, page_size+1):
    # 动态广告插入：只有广告质量超过阈值才能进入前列

- 特征：页面广告比例、广告质量分分布、广告相关性
- 标签：用户会话级满意度信号（深度浏览、无立即返回搜索）
- 推理：若预测满意度 < 基线 85%，强制替换最低质量广告为自然结果

**评估框架（双轨制）：**

| 指标维度 | 核心指标 | 采集方式 |
|---------|---------|---------|
| 用户体验 | NPS、会话退出率、搜索重复率 | 用户反馈 + 行为日志 |
| 商业化 | 广告 eCPM、点击深度、广告主 CTR | 广告系统日志 |

| 系统 | P99延迟、吞吐量 | 链路监控 |

**实验设计：** 双层 A/B（用户层随机 + 广告主层随机），至少持续 2 周，评估用户 NPS 和平台总收益的联合优化效果。

### Q5 | 难度: 开放 | 多渠道归因与预算分配

**Q: 某品牌在信息流、搜索、DSP展示、直播带货四类渠道投放广告。产品经理建议用"末次触点归因"，数据科学家建议用"Shapley值归因"。请从博弈论视角解释两种方案的本质差异，以及 Shapley 值在多渠道归因中的优势与落地挑战？**

**A:**

**直接结论：** 末次触点归因是一种任意约定，缺乏理论支撑，系统性地奖励"收割者"惩罚"播种者"；Shapley 值来自合作博弈论的公理体系，是唯一同时满足效率性、对称性、哑元性和可加性的公平归因方案，但需要枚举所有子集组合（指数复杂度），工业上用 Monte Carlo 近似。

**博弈论视角：**

归因问题是一个**合作博弈**：多个渠道（参与者）合作完成一次转化（创造价值），如何公平分配转化功劳（分配利益）？

**末次触点归因（直觉规则，非博弈论）：**
```
路径：品牌信息流曝光 → DSP展示 → 搜索点击 → 转化
归因：搜索 = 100%，其他 = 0%
```

违反了博弈论公理：
- 违反**对称性**（品牌投资和DSP投资的贡献被强制设为0）
- 违反**效率性**（只有最后一步被"算数"）

**Shapley 值的四条公理：**

| 公理 | 含义 | 对归因的意义 |
|------|------|------------|
| 效率性 | 所有渠道贡献之和 = 总转化价值 | 不浪费，不重复计算 |
| 对称性 | 贡献相同的渠道获相同归因 | 不歧视任何渠道 |
| 哑元性 | 不产生边际贡献的渠道归因=0 | 懒惰渠道不被奖励 |
| 可加性 | 多次活动的归因可叠加 | 大促/日常可分别计算 |

**Shapley 值的计算：**

```python
# 渠道 i 的 Shapley 值 = 该渠道加入所有可能联盟时的平均边际贡献
φ_i = Σ_{S⊆N\{i}} [|S|!(|N|-|S|-1)! / |N|!] × [v(S∪{i}) - v(S)]

# 其中 v(S) = 渠道子集 S 的转化贡献函数（通过历史数据估计）
# N = 所有渠道的集合

# 工业近似（Monte Carlo）：
for _ in range(100000):
    shuffle(channels)  # 随机排列渠道顺序
    for i, channel in enumerate(shuffled_channels):
        marginal = v(prefix_up_to_i) - v(prefix_up_to_{i-1})
        shapley[channel] += marginal
shapley = {k: v/100000 for k, v in shapley.items()}
```

**归因结果对比（示例：4渠道路径）：**

| 渠道 | 末次触点 | 首次触点 | 线性归因(均分) | Shapley值（推荐） |
|------|---------|---------|-------------|----------------|
| 品牌信息流（种草） | 0% | 100% | 25% | 28%（上游高价值） |
| DSP展示（强化）| 0% | 0% | 25% | 21% |
| 搜索（收割）| 100% | 0% | 25% | 35%（仍是最高） |
| 直播（辅助）| 0% | 0% | 25% | 16% |

**Shapley 归因的落地挑战：**

1. **计算复杂度**：4 个渠道 = 2⁴=16 个子集，工业上 20+ 渠道 = 2²⁰ = 100万子集 → 需要 Monte Carlo 近似
2. **v(S) 的估计**：需要足够的历史数据（含每个渠道子集的转化观测）→ 冷启动困难
3. **用户触点对齐**：跨渠道、跨设备的用户 ID 打通是数据基础（无打通则归因无意义）
4. **归因窗口选择**：不同品类（快消 vs 家电）的最优归因窗口不同（7天 vs 90天）
5. **反事实验证**：关闭某渠道做增量实验，验证 Shapley 归因是否真实反映渠道价值

**业务影响（某零售品牌实际数据）：**
- 品牌曝光渠道预算 12% → 28%
- 新用户引入量 +41%
- 整体营销 ROI +22%
- 搜索预算降低 22% 但转化量仅下降 6%（证明搜索的边际价值被高估）

**面试官点评：** 考察统计思维 + 博弈论基础 + 业务洞察。高分要点：① Shapley 值的四条公理（背下来，面试高频）② 末次触点的系统性危害（"奖励收割者"的商业后果）③ Monte Carlo 近似的必要性 ④ 加分：讨论 Shapley 的局限——用历史相关性代理因果，并非真正的增量实验（DID/PSM 才能证明真实增量）

### Q1｜初级｜O1 Embedder的核心思想是什么？与传统query扩展有何区别？

**A:**

**直接结论：** O1 Embedder在生成query embedding之前先进行"内部推理"，推理过程不修改原始query文本，而是通过Prepend思维链来增强编码表示。

**原理：**
传统query扩展是"先扩展文字，再编码"：
```
原始query → 同义词/扩展词 → 扩展后query → Encoder → embedding
```
O1 Embedder是"先内部推理，再编码"：

$$
e_q = \text{Encoder}(\text{Think}(q) \oplus q)
$$

`Think(q)` 是内部生成的推理文本（识别隐含意图、消歧义、关联概念），`⊕` 是拼接，整体输入Encoder后得到更丰富的embedding。

**关键区别：**
| 维度 | 传统query扩展 | O1 Embedder |
|------|----------|---------|
| 修改query文本 | ✅ 是 | ❌ 否（内部推理） |
| 对检索索引影响 | 需要重建 | 完全兼容现有ANN |
| 泛化能力 | 依赖同义词表 | 端到端学习推理 |
| 延迟 | 低 | 略高（推理链生成） |

**示例：** "比铝轻的常见金属" → 推理链识别出"需要比较金属密度，铝密度2.7 g/cm³，候选：镁、锂" → 这段推理Prepend后编码，embedding在"轻金属材料"空间更准确聚集。

**面试官点评：** 考察检索增强与推理能力融合的理解。高分要点：① 推理在编码前而非后 ② document侧完全不变（兼容所有ANN） ③ 加分：说明适用场景（知识型查询，不适合高频简单query因为延迟代价大）

### Q2｜中级｜DCN-V2的低秩分解如何在保证表达能力的同时压缩参数？

**A:**

**直接结论：** 将Cross层的 $d×d$ 权重矩阵分解为 $d×r$ 和 $r×d$ 两个矩阵的乘积，参数量从 $O(d^2)$ 降到 $O(d \times r)$，$r \ll d$，同时保留了特征高阶交叉的能力。

**原理：**

DCN-V2的Cross层：

$$
x_{l+1} = x_0 \odot (W_l x_l + b_l) + x_l
$$

当特征维度 $d=10000$ 时，$W_l \in \mathbb{R}^{d \times d}$ 有 $10^8$ 参数，不可承受。

低秩分解：$W = UV^T$，$U \in \mathbb{R}^{d \times r}$，$V \in \mathbb{R}^{d \times r}$，参数量 $2dr$。

当 $r=128$，$d=10000$ 时：
- 全矩阵：$10^8$ 参数
- 低秩：$2 \times 10000 \times 128 = 256K$ 参数（**节省390倍**）

**为什么低秩仍能保留表达力？**  
特征交叉的有效信息维度通常远低于原始维度。真实世界的特征相关性矩阵往往是低秩的（大部分特征交叉集中在少数主方向上）。

**与其他方法对比：**
| 方法 | 参数量 | 高阶交叉 | 可解释性 |
|------|--------|---------|---------|
| FM/FFM | $O(d)$ | 仅二阶 | 高 |
| DCN-V1 | $O(d)$ | 多阶但受限 | 中 |
| DCN-V2 全矩阵 | $O(d^2)$ | 强 | 低 |
| **DCN-V2 低秩** | $O(dr)$ | 接近全矩阵 | 低 |

**面试官点评：** 考察特征交叉工程能力。高分要点：① 矩阵分解的数学推导 ② 低秩假设的现实依据（特征相关性矩阵天然低秩） ③ 加分：讨论混合专家DCN-V2（多个低秩矩阵的Mixture）

### Q3｜中级｜生成式CTR（DGenCTR）与传统判别式CTR的范式差异，以及各自适用场景？

**A:**

**直接结论：** 判别式CTR建模条件概率 $P(y|x)$，直接预测点击；生成式CTR建模联合分布 $P(x, y)$ 或条件生成 $P(x|y)$，通过特征重建理解数据分布，泛化性更强但延迟更高。

**原理对比：**

**判别式（DeepFM/DIN/DCN）：**
```
用户画像 + 物品特征 → MLP → sigmoid → CTR分数
```
直接对齐业务目标，训练快，线上延迟低（<5ms）。

**生成式（DGenCTR，离散扩散）：**

$$
p_\theta(x_0 | x_t, c) = \prod_{i} p_\theta(x_0^{(i)} | x_t, c)
$$

将所有特征统一token化，用离散扩散模型重建特征。点击信号作为条件 $c$，通过建模"点击时的特征分布"来推断CTR。

**核心差异总结：**

| 维度 | 判别式CTR | 生成式CTR（DGenCTR） |
|------|---------|--------------|
| 建模目标 | $P(y|x)$ | $P(x_0|x_t, c)$ 联合分布 |
| 稀疏特征处理 | 依赖Embedding查表 | 统一token化，天然处理稀疏 |
| 冷启动 | 弱（依赖历史） | 较强（分布泛化） |
| 延迟 | 低（5-10ms） | 高（自适应步数，10-50ms） |
| 训练复杂度 | 低 | 高 |
| 特征扩展 | 需重新训练 | 更易泛化到新特征 |

**适用场景：**
- 判别式：大流量核心场景，延迟敏感，特征稳定
- 生成式：冷启动商品/用户、长尾场景、多模态特征融合

**与MTGR的联系：** 美团MTGR用decoder-only架构统一12+场景，本质也是生成式思路，但将"生成"定义为"生成推荐序列"而非"生成特征分布"。

**面试官点评：** 考察推荐系统范式演进的前沿理解。高分要点：① 两种范式的概率图差异 ② 生成式的实际工程代价 ③ 加分：生成式CTR目前在工业界还处于探索阶段，能说出落地挑战（推理延迟、不可解释性）是加分项

### Q4｜高级｜系统设计：设计一个支持FPA（第一价格拍卖）的智能竞价系统，如何结合GBS（生成式出价）和Bid2X（竞价环境预测）？

**A:**

**直接结论：** FPA系统的核心挑战是"出多少钱才是最优"——需要同时建模用户转化概率（价值侧）、竞争环境分布（市场侧）、以及出价不确定性（风险侧）。GBS负责建模出价分布，Bid2X负责预测竞争动态，两者串联形成完整决策链。

**整体架构：**

```
用户请求
    ↓
[价值预测层]
  CTR/CVR模型 → 预期价值 eCPM = CTR × CVR × 出价上限
    ↓
[竞争环境感知层 - Bid2X]
  历史市价分布 + 时序趋势 → 预测当前竞价密度 P(win|bid)
    ↓
[出价分布生成层 - GBS]
  P(w|x) = Σ π_k(x) · N(w|μ_k, σ_k²) → 采样最优出价
    ↓
[Bid Shading决策]
  最优出价 = argmax_b [eCPM · P(win|b) - E[支出|win, b]]
    ↓
[实时竞价]
```

**核心公式：**

GBS生成出价分布（混合高斯）：

$$
p(w | x) = \sum_{k=1}^{K} \pi_k(x) \cdot \mathcal{N}(w | \mu_k(x), \sigma_k^2(x))
$$

FPA最优出价（Bid Shading）：

$$
b^* = \arg\max_b \left[ v \cdot P(\text{win}|b) - b \cdot P(\text{win}|b) \right]
$$

其中 $v$ 是广告主的真实价值（eCPM），$P(\text{win}|b)$ 由Bid2X预测。

**关键工程设计：**

1. **实时性（<5ms决策）：**  
   - GBS的高斯混合参数离线训练，在线只做采样（O(K)，K通常3-5）  
   - Bid2X的竞争密度预测做周期性更新（分钟级）而非每次请求都预测  

2. **稳定性（防止竞价崩溃）：**  
   - 设置bid floor和bid ceiling  
   - 加入出价平滑（EMA），防止单次大幅震荡  

3. **预算控制：**  
   - ROI约束：调整 $\pi_k$ 权重，保证每日花费不超预算  
   - Pacing机制：实时花费速率反馈到出价系数  

4. **A/B验证：**  
   - bucket隔离（避免竞价池污染）  
   - 核心指标：eCPM、真实ROI、预算达成率、竞赢率

**与GSP系统的迁移差异：**
| 维度 | GSP | FPA |
|------|-----|-----|
| 出价策略 | 出真实价值即可 | 需要Shading（否则亏损） |
| 模型依赖 | CTR模型 | CTR + 竞价分布模型 |
| 系统复杂度 | 低 | 高 |

**面试官点评：** 考察广告系统架构设计能力 + 博弈论基础。高分要点：① FPA vs GSP的竞价策略本质差异 ② GBS解决了什么问题（量化出价不确定性，而非点估计） ③ Bid2X的实时性方案（离线/在线分离） ④ 加分：讨论多广告主同时Shading导致的均衡问题（纳什均衡）

### Q5｜开放｜将"推理能力"注入搜广推系统是2026年最热趋势，你认为最大的工程挑战是什么？怎么解决？

**A:**

**我的判断：推理注入的最大工程挑战是「延迟-质量的不可调和矛盾」。**

**问题的本质：**

搜广推系统的SLA极严格（召回<10ms，排序<50ms，广告<5ms），而推理链生成天然慢（即使是1.5B小模型，生成100 token也需要30-50ms）。这不是优化问题，是数量级的物理瓶颈。

**当前的三类解决路径：**

**路径1：蒸馏-推理能力迁移（DEAR范式）**
```
离线：GPT-4生成推理链（ranking rationale）
蒸馏：1.5B模型学习推理模式
在线：1.5B模型直接输出结果（推理已内化，无需显式生成Chain-of-Thought）
```
优点：线上延迟接近普通小模型；缺点：蒸馏结果上界受限于teacher

**路径2：异步推理（不挡在线链路）**
```
离线/准实时：对query/item做推理标注，结果写入Feature Store
在线：直接查表（<1ms），推理结果作为特征输入模型
```
典型：LLM Embedding的对齐融合方案。适合query/item侧的静态分析，不适合动态用户行为。

**路径3：早退机制（LimRank范式）**
```
简单样本（高置信度）：1-2步推理即退出
复杂样本（低置信度）：完整推理链
```
优点：动态balance延迟与质量；缺点：需要可靠的置信度估计

**我的判断（第二大挑战）：推理质量的一致性**

小模型推理链常出现幻觉，在排序任务中若推理链出错（如把相关doc判为不相关），下游会放大错误。工业上需要建立推理链质量的自动评估机制（比较推理链结果和精确标注的一致率），设置安全fallback。

**未来方向：**
- Speculative Reasoning（Speculative Decoding + 推理压缩）
- 查询感知的动态推理深度（简单query不推理，复杂query深推理）
- 端侧推理（减少网络延迟）

**面试官点评：** 考察系统性工程思维 + 前沿跟踪能力。高分要点：① 明确说出延迟是核心矛盾（而不是模糊说"挑战很多"） ② 三种解法都能说出trade-off ③ 加分：提到蒸馏的上界问题、推理幻觉的风险处理、以及自适应推理深度的思路

### Q1｜初级｜什么是 Differentiable Semantic ID？它解决了生成式推荐中的什么问题？

**A:**

**直接结论：** Differentiable Semantic ID（可微语义ID）是一种将 item 映射到离散语义码字（codebook token序列）的表示方法，使得生成式推荐模型可以用自回归方式"生成"推荐结果，同时保证 item 的语义相似性在 token 空间中可学习、可微分。

**原理：**

传统生成式推荐（如 P5 / TIGER）使用 item ID 的整数编码，存在两大问题：
1. **语义鸿沟**：ID 序号是任意的（item 1 和 item 2 在语义上可能毫无关系），模型无法从 token 的相邻关系中学习语义
2. **不可微**：码本离散化（VQ-VAE 的 argmax）导致梯度无法回传到编码器，item 表示和生成目标解耦

Differentiable Semantic ID 的核心改进：
```
传统方案：item emb → VQ (argmax) → token 序列（梯度断裂）
新方案：  item emb → Gumbel-Softmax / Straight-Through → token 序列（梯度连通）
```

通过 Gumbel-Softmax 近似离散化，允许梯度从生成损失（预测下一个 token）反传到 item embedding，使得语义相似的 item 在 codebook 空间中也会学习到相近的 token 序列。

**示例：**  
"iPhone 15" 和 "iPhone 14" 在语义空间相似，经过可微 Semantic ID 后，其 token 序列的前缀会高度重合（如都以 `[101, 23]` 开头），让自回归模型在 beam search 时能更高效地探索语义相似 item。

**与其他方法对比：**
| 方法 | 语义性 | 可微性 | 生成效率 |
|------|--------|--------|----------|
| 随机 Item ID | ❌ | ❌ | 低（大词表） |
| RQ-VAE（TIGER） | ✅ 离线固定 | ❌ | 中 |
| Differentiable Semantic ID | ✅ 端到端学习 | ✅ | 高 |

**面试官点评：** 考察生成式推荐的核心范式理解。高分要点：① 指出 ID 表示的语义鸿沟问题 ② 理解 Gumbel-Softmax 解决离散化不可微问题的原理 ③ 加分：举出 beam search 时语义前缀重合带来的检索加速效果

### Q2｜中级｜BundleNet 联合广告的神经捆绑竞价机制是什么？解决了什么工程难题？

**A:**

**直接结论：** BundleNet 是一种将多个广告主的广告"捆绑"成组合广告单元，通过神经网络联合优化捆绑组合收益的竞价机制，解决了传统逐条竞价（per-slot auction）无法最大化广告组合展示效益的问题。

**背景问题：**

在 Feed 流中连续展示 2-3 条广告（Bundle）时，传统独立拍卖存在：
1. **互补/替代效应被忽略**：A 广告和 B 广告一起展示时点击率可能高于单独展示（互补），或某两类广告并排会互相蚕食（替代）
2. **收入次优**：贪心地按单广告最高 bid 选出广告，不等于 bundle 整体收益最高

**BundleNet 的解法：**

```
输入：候选广告集合 {a_1, a_2, ..., a_n}
枚举（或采样）可行 bundle：B_k = {a_i, a_j, ...}
神经网络打分：f(B_k) = 预测该 bundle 整体 eCPM（联合建模广告间交互）
拍卖机制：选取 f(B_k) 最高的 bundle，并按 VCG / Myerson 计费
```

**关键工程设计：**
- **Bundle 枚举爆炸**：n 条候选 → C(n,k) 组合数指数增长。解法：先用粗排筛到 Top-20，再枚举 C(20,3)=1140 个三元组，可接受
- **联合交互建模**：bundle 内广告用 self-attention 建模交互，比独立打分多出 bundle-level 特征（类目重叠度、价格区间相似性）
- **VCG 计费保证激励相容**：每个广告主按其对 bundle 的边际贡献定价，防止广告主虚报出价

**与独立竞价对比：**
| 维度 | 传统逐条拍卖 | BundleNet |
|------|-------------|-----------|
| 广告间交互 | ❌ 独立假设 | ✅ 联合建模 |
| 收益最大化 | 局部最优 | 全局更优 |
| 计算复杂度 | O(n) | O(C(n,k)) |
| 激励相容 | GSP近似 | VCG严格保证 |

**面试官点评：** 考察广告拍卖机制 + 组合优化工程设计。高分要点：① 说清楚 Bundle 收益 ≠ 单广告收益之和（交互效应）② 枚举爆炸的工程解法 ③ 加分：VCG 计费的激励相容原理（说清"边际贡献"就行）④ ICML 2025 背景说明能跟踪前沿

### Q3｜中级｜Rank1 和 REARANK 分别代表了测试时计算（Test-Time Compute）在搜索重排中的什么思路？

**A:**

**直接结论：**
- **Rank1**：在**重排阶段**动态分配推理计算预算（Test-Time Compute Scaling），让模型对难度高的排序对投入更多"思考步骤"
- **REARANK**：将重排任务建模为 **RL 强化的推理 Agent**，模型显式生成排序推理链（Chain-of-Thought），用 RL 优化最终 NDCG

**Rank1 的核心机制：**

```
输入：Query + Top-K 候选文档
步骤：
  1. 模型先输出"推理预算"（难易度评估）
  2. 简单 query（高相关度候选）→ 1-2 步推理直接输出排序
  3. 困难 query（相关度模糊）→ 多步推理（compare → reason → rank）
  4. 推理步数受 budget token 约束，硬截断
```

优点：平均延迟可控（简单 query 快，复杂 query 精准）；缺点：budget 预估本身需要额外模型

**REARANK 的核心机制：**

```
训练阶段：
  用 PPO 训练 LLM 排序器
  奖励 = NDCG@10（最终排序质量），不是 token 级别监督
  模型学会生成「推理链 + 排序结果」的联合输出

推理阶段：
  输入 query + 候选文档
  模型显式输出：<think>文档A比B更相关因为...</think>[排序结果]
```

优点：推理链可解释，RL 直接对 IR 指标优化；缺点：PPO 训练不稳定，需要大量对比数据

**两者对比：**
| | Rank1 | REARANK |
|-|-------|---------|
| 核心思想 | 动态计算预算分配 | RL直接优化IR指标 |
| 推理可见性 | 隐式（步数控制） | 显式 CoT |
| 训练方式 | SFT + 预算回归 | PPO（稀疏奖励） |
| 延迟可控性 | ✅ 预算硬约束 | ⚠️ CoT长度不固定 |
| 可解释性 | ❌ | ✅ |

**面试官点评：** 考察 LLM-for-Ranking 前沿 + Test-Time Compute 的工业理解。高分要点：① 区分两者的本质区别（预算控制 vs RL优化目标）② 说出各自的工程 trade-off ③ 加分：讨论 CoT 在 reranking 的幻觉风险（模型可能生成看起来合理但错误的推理链）

### Q4｜高级｜系统设计：为电商平台设计一个基于 MTORL（多任务离线RL）的推荐系统，需要支持 GMV、点击率、复购率三个业务目标，如何架构？

**A:**

**系统目标明确化：**

单纯优化 CTR 会导致"标题党"内容泛滥，优化 GMV 会忽视用户体验，复购率需要跨会话的长序列建模。MTORL 的价值：通过离线 RL 在三个目标之间做全局策略优化，而非手工加权。

**整体架构：**

```
数据层
  用户行为日志（点击/购买/复购标记）→ Replay Buffer
  奖励设计：r = w1·CTR + w2·GMV_normalized + w3·复购信号（7日内再购）

策略模型（MTORL核心）
  Backbone: Transformer 序列编码（捕捉长序列兴趣）
  多头输出：
    ├── CTR Head → pCTR
    ├── CVR Head → pCVR（结合 ESMM 全空间估计）
    └── Policy Head → 当前状态下 Top-K 选品的价值函数 Q(s, a)
  
  离线RL训练：Conservative Q-Learning（CQL）
    - 离线数据分布偏移问题：CQL 惩罚 OOD（分布外）动作的 Q 值过高估计
    - Batch Size：大批次（4096）减少方差

排序层
  在线打分：Q(s, a) = α·pCTR + β·pCVR + γ·V_longterm（长期价值）
  约束：GMV 约束（单次推荐的期望 GMV ≥ floor）
  Pareto 前沿选品：不同用户偏好映射到不同 (α, β, γ) 配置

工程实现关键点
  1. 复购信号延迟（7日）：用生存分析软标签（类似延迟CVR）解决右截尾
  2. Replay Buffer 管理：按行为时间戳采样，避免近期行为过采样（时间衰减权重）
  3. 多目标 Pareto 优化：当三个目标冲突时，不手工调权，而是维护 Pareto 前沿策略池，AB 测试决定上线哪个
  4. 安全部署：先用 off-policy 评估（IS 重要性采样）验证策略收益，再在 5% 流量上实验
```

**与传统多目标排序对比：**

| | 手工加权多目标 | MTORL |
|-|--------------|-------|
| 目标冲突处理 | 手工调 w（艰难经验） | CQL 学习全局最优 |
| 长期收益 | ❌ 局部贪心 | ✅ Q函数近似长期价值 |
| 分布偏移 | 不处理 | CQL 保守约束 |
| 可解释性 | 高 | 低（需要 Q 值可视化） |
| 迭代周期 | 快（改权重即可） | 慢（需要重训RL策略） |

**落地建议：**
- Phase 1：先用 MTORL 做离线实验（off-policy eval），校验 Q 函数的 OOD 保守性
- Phase 2：在低流量（1%）验证在线收益，特别观察复购率 7-30 天延迟指标
- Phase 3：稳定后扩量，用 Pareto 策略替换手工加权系统

**面试官点评：** 考察多目标推荐系统设计 + RL 工业落地能力。高分要点：① 点出离线RL的分布偏移问题并说出 CQL 解法 ② 复购信号延迟的软标签处理 ③ Pareto 前沿替代手工加权（体现系统思维）④ 加分：off-policy evaluation 的 IS 估计，以及为什么 RL 策略要保守部署（safety）

### Q5｜开放｜From RAG to Memory — LLM 的「非参数持续学习」是什么范式转变？你认为它会如何影响推荐系统的长期用户建模？

**A:**

**范式转变的本质：**

传统 LLM 的知识存储方式：
- **参数记忆（Parametric Memory）**：知识烘焙在权重里，更新需要微调，成本高
- **RAG 检索增强**：知识存在外部文档库，推理时检索，但文档是静态的（无法自我更新）

"From RAG to Memory" 代表的新范式：
```
RAG（检索）→ Memory（记忆）
区别：Memory 是动态的、有结构的、可被 LLM 自己写入/更新的知识存储
LLM 不仅"读"外部知识，还能"写"经验到 Memory Bank，形成持续学习闭环
```

**具体机制（类比人类记忆）：**
- **工作记忆（短期）**：当前 context window，容量有限
- **情节记忆（中期）**：关键对话/事件存入向量数据库（如 LanceDB），按需检索
- **语义记忆（长期）**：压缩提炼后存为知识图谱节点，跨会话持久化
- **程序记忆（技能）**：反复用过的推理模式蒸馏进参数（LoRA 增量微调）

**对推荐系统长期用户建模的影响：**

现有推荐系统的用户记忆：
- 向量化 User Embedding（参数记忆，更新慢）
- 行为序列窗口（工作记忆，窗口有限，通常 50-200 items）
- 长期兴趣摘要（离线计算，延迟高）

"From RAG to Memory" 带来的可能变革：
```
未来推荐系统用户建模：
  实时行为 → Working Memory（Context）
  ↓（异步压缩）
  关键兴趣转变事件 → Episodic Memory（向量库，精确检索）
  ↓（周期蒸馏）
  稳定兴趣画像 → Semantic Memory（知识图谱：用户 → 偏好节点）
  
推理时：
  Query → 检索 Episodic Memory（"上周买了什么"）
        + 读取 Semantic Memory（"长期是科技品牌偏好者"）
        + 实时行为 Context
        → 融合生成个性化排序
```

**我的判断：三大机会 + 两大挑战**

机会：
1. **超长用户历史建模**：不再受 context window 限制，能检索 3 年前的行为模式
2. **跨域迁移**：用户在电商的购买记忆可以迁移到内容推荐（跨 App 的 Memory）
3. **可解释性**：Memory 是结构化的，可以对用户说"推荐此商品因为你3个月前看过类似的"

挑战：
1. **隐私风险**：Memory Bank 存储完整行为历史，比传统 embedding 更敏感，GDPR 合规难
2. **记忆遗忘策略**：什么值得保留？过期兴趣如何衰减？无差别存储会导致记忆"污染"

**面试官点评：** 考察对 LLM 新范式的前沿跟踪 + 工业推荐的迁移思考。高分要点：① 清楚区分 RAG 和 Memory 的本质差别（写入能力 + 动态更新）② 类比人类多层记忆系统（工作/情节/语义）③ 结合推荐系统现有问题（行为序列窗口限制）提出具体改进方向 ④ 加分：主动提出隐私/遗忘这两个工程挑战，体现系统落地思维

### Q1：SpargeAttention 和 FlashAttention 的核心区别是什么？ | 难度: 初

**A：**

**直接结论：** FlashAttention 优化的是内存访问，SpargeAttention 优化的是计算量本身。

**原理：**
- FlashAttention：把 QK^T 计算分块放入 SRAM，减少 HBM 读写次数（IO-bound 优化）。但仍然计算所有 Q×K 的点积，只是更高效地读写。
- SpargeAttention：动态判断注意力块的重要性 `I(B_ij) = max_{q,k∈B_ij} q^T·k / √d`，跳过低重要性块（稀疏计算）。真正减少了 FLOPs。

**示例：** 处理 100K token 序列时，FlashAttention 仍是 O(N²) 的 FLOP 开销；SpargeAttention 在实际注意力稀疏（常见于长文本）时可降至 O(N·k)，实测加速 1.8–2.5×。

**对比其他方案：**
| 方案 | 优化维度 | 缺点 |
|------|---------|------|
| FlashAttention | IO-bound | 仍是 O(N²) FLOPs |
| Linformer / Performer | 近似计算，改变机制 | 精度有损失 |
| SpargeAttention | 动态稀疏，即插即用 | 实际加速依赖稀疏度 |

**面试官点评：** 考察对 Attention 优化方向的分层理解。高分要点：① 区分 IO-bound 和 compute-bound 两个瓶颈 ② 说出块级稀疏判断机制 ③ 知道"即插即用"是工程价值（不需要改模型结构）

### Q2：推荐系统的 Scaling Laws 和 LLM Scaling Laws 有什么本质区别？ | 难度: 中

**A：**

**直接结论：** LLM Scaling Laws 是参数量驱动的平滑幂律；推荐系统的 Scaling Laws 被稀疏 Embedding 打断，存在阈值效应。

**原理（基于 Meta Kunlun 的发现）：**

LLM 的 Chinchilla Law：`L(N, D) = E + A/N^α + B/D^β`，参数量 N 和数据量 D 有平滑的幂律关系。

推荐系统的差异点：
1. **稀疏性打断幂律**：Item/User ID 的 Embedding 参数量巨大，但大量参数极少被激活（长尾物品），导致有效参数量≠总参数量
2. **离散特征 vs 连续特征**：语言是连续语义空间，推荐输入是离散稀疏特征，Scaling 机制本质不同
3. **MFU 提升是突破口**：Kunlun 通过将 MFU 从 17% 提升到 37%（算力利用率），才使 Scaling Laws 成立——即必须先解决算力利用率问题，再谈参数量缩放

**示例：** Kunlun 发现当 Embedding 查询效率低下时，增大模型参数反而降低效益；只有在高 MFU 前提下，参数量与效果才有稳定的幂律关系。

**面试官点评：** 考察前沿论文深度理解 + 跨系统迁移思维。高分要点：① 说出"稀疏Embedding是推荐Scaling的核心障碍" ② MFU的概念（Model FLOPs Utilization）③ 对比LLM参数量和推荐系统参数量的"质"的差异

### Q3：Causal Bidding 如何通过反事实推理消除竞价偏差？ | 难度: 中

**A：**

**直接结论：** 传统竞价学习 confounded by 历史出价策略（observational bias），Causal Bidding 通过反事实估计"如果用不同出价，ROI 会如何变化"，消除混淆偏差。

**原理：**

传统方法的问题：
```
历史数据: 出价高 → 赢得竞价 → 产生转化 → 正样本
         出价低 → 输掉竞价 → 无转化数据 → 系统学不到低出价的真实效果
```
这导致模型 overfit 到历史出价分布，不知道"如果出价更低，ROI会更高"。

Causal Bidding 的做法：
1. 学习潜在结果模型 `Y(b) = f(x, b)` —— 在给定特征 x 下，不同出价 b 对应的转化/ROI
2. 反事实估计：`CATE(x) = E[Y(b_high) - Y(b_low) | x]` —— 在 x 条件下，加价的边际收益
3. 最优出价 = 边际收益刚好等于竞价成本的出价点

**示例：** 长尾广告主（预算少、历史数据稀疏）最受益，因为他们的历史数据偏差最大。Meta 在此类广告主上 ROI 提升 18%。

**对比 RL 出价：** RL 出价建模序列决策，适合预算全局优化；Causal Bidding 建模单次竞价的因果效应，适合精确单次 ROI 优化。两者互补。

**面试官点评：** 考察因果推断在工业系统的应用，高分要点：① 清楚描述 observational bias 的产生机制 ② 解释反事实估计 vs 相关性估计的区别 ③ 说出长尾广告主是核心受益群体（说明场景理解）

### Q5：如果你是推荐系统 Scaling 的 PM，你会如何决策：先优化算力利用率（MFU），还是先增大模型参数量？ | 难度: 开放

**A：**

**结论：先 MFU，再参数量。** 原因是二者是乘法关系，MFU 低时增大参数量是"在漏水的桶里加水"。

**论证框架：**

1. **资金效率视角（ROI 分析）：**
   - MFU 从 17% → 37%（Kunlun 做到了），相当于算力成本砍半，而模型效果不变
   - 相同成本下，MFU 提升的边际收益 >> 参数量加倍
   - 推荐系统的 Scaling Laws 研究表明：只有 MFU 超过阈值，参数量和效果的幂律关系才成立

2. **技术路径视角：**
   - MFU 低的根因：稀疏 Embedding 查询效率差（内存带宽瓶颈）、批次大小受限、通信开销大
   - 这些是基础工程问题，不解决则参数量增大只会放大瓶颈
   - 类比：CPU 利用率 20% 时先买更多 CPU 是浪费

3. **实验驱动视角：**
   - 先用小模型验证 MFU 优化效果 → 建立 Scaling 曲线 → 有把握后扩参数量
   - 避免盲目 Scale 导致的资源浪费（大公司有反例）

4. **时机视角：**
   - MFU 优化是团队基础能力，先做，收益可持续
   - 参数量 Scale 有明显边际递减，等有 Scaling Laws 数据支撑再做决策

**开放补充：** 但如果竞争对手已经在大规模 Scale 且建立了先发优势，可能需要并行推进——MFU 优化和增参同时做，但设定 MFU 里程碑作为 Scale 的前置条件。

**面试官点评：** 考察工程直觉 + 系统思维 + 决策框架。高分要点：① 不直接选边，先建分析框架 ② 提出"MFU是Scaling Laws成立的前提条件"（这是Kunlun论文的核心洞见） ③ 用ROI/边际收益语言说话（体现PM/工程负责人思维） ④ 提出并行推进的现实策略
