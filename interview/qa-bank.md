### Q1 | 难度: 初级 | Mamba SSM 基础

**Q: 请解释 Mamba（选择性状态空间模型）相比 Transformer 的核心优势，以及为什么它特别适合推荐系统的超长用户行为序列建模？**

**A:**

**直接结论：** Mamba 以 O(n) 线性复杂度替代 Transformer 的 O(n²) 注意力计算，其「选择性遗忘」机制与推荐场景「用户历史稀疏有价值信号」的本质天然契合，是超长序列建模从工程不可行到生产可用的关键跨越。

**Mamba 核心机制 — 选择性状态空间（SSM）：**

传统 SSM 的状态转移方程：
```
h_t = A·h_{t-1} + B·x_t     # 状态更新（h 为隐状态）
y_t = C·h_t                  # 输出
```

Mamba 的改进：让 A、B、C **依赖于输入 x_t**（Selection Mechanism）：
```
Δ_t, B_t, C_t = Linear(x_t)   # 参数动态化
h_t = exp(Δ_t A)·h_{t-1} + Δ_t·B_t·x_t
```

- **Δ_t 大**：模型"专注"于当前输入，大量更新隐状态
- **Δ_t 小**：模型"遗忘"当前输入，保持历史状态不变

**与 Transformer 的对比：**

| 维度 | Transformer | Mamba |
|------|------------|-------|
| 计算复杂度 | O(n²) 注意力矩阵 | O(n) 顺序扫描 |
| 序列长度上限 | ~4096（显存瓶颈） | 理论无限 |
| 历史信息利用 | 全局注意力（平等对待所有位置） | 选择性记忆（动态权衡保留/遗忘） |
| 并行训练 | 完全并行 | 并行扫描（Parallel Scan） |
| 推理显存 | O(n) KV Cache | O(1) 固定大小状态 |
| 适合场景 | 短序列语义理解 | 长序列时序建模 |

**推荐场景的天然契合：**

用户行为序列特征：「大部分不相关，少量关键信号稀疏分布」
- 用户3个月前购买过的品类 → 对季节性商品至关重要（需要长期记忆）
- 用户昨天随机浏览的内容 → 可能是噪声（需要遗忘）

Mamba 的选择性遗忘直接解决这个问题，而 Transformer 的注意力机制无法区分"值得记住的历史"和"噪声行为"。

**工程落地（MambaRec方案）：**
- 用户行为序列分段压缩（短/中/长期各段轻量Transformer提取摘要）
- 段级摘要输入 Mamba 主干（保留跨时间段长程依赖）
- 兴趣快照缓存（128维×20时间槽），推理时只加载快照
- 延迟：280ms → 45ms，CTR +6.8%，季节性品类 GMV +23%

**面试官点评：** 考察 SSM 基础理论与工程结合。高分要点：① O(n²) vs O(n) 的本质区别 ② Δ_t 参数化的选择机制（不只说"更快的Transformer"）③ 推荐场景的业务动机（稀疏有价值信号）④ 加分：O(1) 推理显存 vs O(n) KV Cache 的对比

### Q2 | 难度: 中级 | 延迟CVR标签建模

**Q: 广告CVR预估中的延迟标签问题是什么？请描述生存分析（Survival Analysis）如何建模转化延迟，以及如何设计样本重加权策略解决系统性偏差？**

**A:**

**直接结论：** 延迟标签问题是广告CVR建模中被系统性低估的陷阱——固定窗口策略将大量真实正样本标记为负，造成CVR被低估10-15%。生存分析将"是否转化"和"何时转化"分开建模，在样本到达时给出置信加权，替代粗暴的二值化处理。

**延迟标签问题的本质：**
```
用户点击广告
  │
  ├──── 30分钟内购买 ── 标记为正样本（1）✓
  │
  └──── 30分钟后才购买（42%的转化！）── 错误标记为负样本（0）✗
         ├─ 1-24小时后转化（23%）
         └─ 24小时以上（如家电/汽车，决策周期长）
```

**生存分析框架：**

定义"生存函数" S(t) = P(转化发生在时刻 t 之后 | 已发生转化)
及风险函数（Hazard） h(t) = P(在 t 时刻转化 | 已点击且 t 前未转化)

对每个（广告主类别 × 用户类型 × 品类）组合，用 Weibull 分布或 Cox 比例风险模型拟合：

```python
# Weibull 参数化生存模型
S(t; λ, k) = exp(-(t/λ)^k)    # λ=scale, k=shape
h(t; λ, k) = (k/λ)(t/λ)^(k-1) # 风险率

# 对数据拟合：最大化观测转化时间的似然
L = Π_{i: converted} h(t_i) × S(t_i)   # 已转化样本
  × Π_{j: not yet}    S(w_j)            # 窗口内未转化（右截尾）
```

**样本重加权策略：**

```python
def get_sample_weight(t_elapsed, window=30*60):
    # 窗口内已转化：置信权重=1.0
    if t_elapsed < window and converted:
        return 1.0
    # 延迟样本：按生存概率校正
    elif t_elapsed < 24*3600:
        # w = P(conversion | t_elapsed, context) 
        # 用拟合的生存函数估计当前等待到这里仍未转化的样本中真正不转化的比例
        prob_future_convert = 1 - S(24*3600) / S(t_elapsed)
        return prob_future_convert  # 软标签(0~1)
    # 超长延迟：离线队列批量修正
    else:
        return "deferred_to_offline_queue"
```

**CVR 校正层：**

在模型输出头加入实时修正因子 ε(t_post_launch)：
```
CVR_corrected = CVR_model × (1 + ε(t))
```
其中 ε(t) 在大促开始初期（模型样本少、延迟多）为正，随时间收敛。

**工业效果：** CVR LogLoss 提升 12.3%，出价偏差 15% → 3.2%，平台广告收入 +11.4%

**与其他方案对比：**

| 方案 | 核心思想 | 优点 | 缺点 |
|------|----------|------|------|
| 固定窗口（基线）| 窗口外=负样本 | 简单 | 系统性低估 10-15% |
| ES-DFM（Elapsed-time DFM）| 对每个曝光建独立延迟模型 | 精度高 | 模型数量多，维护成本高 |
| 生存分析（推荐）| 参数化延迟分布+软标签 | 统计严谨，工程可行 | 需要历史转化时间数据 |
| Bi-LSTM延迟预测 | 序列建模每条广告的延迟模式 | 捕捉非参数分布 | 训练复杂，延迟高 |

**面试官点评：** 考察对广告系统训练数据陷阱的深度理解。高分要点：① 明确给出延迟比例数字（42%超30分钟）② 生存函数的数学定义（右截尾的处理）③ 样本重加权的具体公式 ④ 加分：区分"排序性偏差"（AUC不变）和"校准性偏差"（出价影响）

### Q3 | 难度: 中级 | ESMM 解决 CVR 的 SSB 和 DS 问题

**Q: CVR 预估中存在两个核心问题：样本空间偏差（SSB）和数据稀疏（DS）。请解释这两个问题的根本原因，以及 ESMM（Entire Space Multi-task Model）如何通过全空间建模和参数共享同时解决这两个问题？**

**A:**

**直接结论：** SSB 源于 CVR 只在"点击"子空间训练但要在"曝光"全空间预测；DS 源于转化样本极其稀少（CTR≈1%，CVR≈1%，pCTCVR≈0.01%）。ESMM 通过建模 pCTCVR = pCTR × pCVR，将 CVR 的训练空间扩展到全曝光空间，并以 CTR 任务共享 embedding 参数解决稀疏问题。

**SSB（Sample Space Bias）根本原因：**

```
理想：CVR 模型应学习 P(conversion | exposure, x)
实际：CVR 模型只能从「有点击的样本」学，即 P(conversion | click, x)

训练空间：点击集合（impression 的 1%）
推理空间：全曝光（100%）

训练 ≠ 推理空间 → 分布偏移
```

**DS（Data Sparsity）根本原因：**

```
全量曝光样本：10亿/天
有点击的样本：1000万/天（1%）
有转化的样本：10万/天（0.01%）

CVR 模型只能用 10万 正样本学习，而特征空间维度可能是 10^9
```

**ESMM 架构与数学推导：**

```
                    曝光空间 (全量)
                          │
            ┌─────────────┴─────────────┐
            │ CTR Tower               │ CVR Tower
            │ P(click|x)              │ P(cvr|click,x)
            │ [embedding共享]          │ [embedding共享]
            └─────────┬───────────────┘
                      │ element-wise multiply
                      ↓
               P(ctcvr|x) = P(click|x) × P(cvr|click,x)
               
Loss = L_CTR(click, P(click|x)) + L_CTCVR(conversion, P(ctcvr|x))
```

**关键设计决策：**

1. **全空间标签**：CTCVR=1 当且仅当"有点击且有转化"，这个标签在全曝光空间都可定义，解决 SSB。
2. **乘法结构**：通过 pCTCVR = pCTR × pCVR 的分解，CVR Tower 的梯度通过 CTCVR Loss 反传，训练空间隐式扩展到全曝光。
3. **Embedding 共享**：CTR 有大量点击样本，CVR 通过共享底层 embedding 间接学习冷门用户/物品的表示，解决 DS。

**与基线方法对比：**

| 方法 | SSB 处理 | DS 处理 | AUC | 缺点 |
|------|---------|---------|-----|------|
| 独立 CVR 模型 | ❌ 只在点击空间 | ❌ 样本稀疏 | 低 | 系统性偏差 |
| 端到端 pCTCVR 模型 | ✅ 全空间 | 部分✅ | 中 | 无法独立控制 CVR |
| ESMM | ✅ 全空间 | ✅ embedding共享 | 高 | CVR 推理需配合 CTR（乘法结构） |
| ESCM²（进阶）| ✅ | ✅ + Counterfactual正则化 | 更高 | 实现复杂 |

**面试官点评：** 考察多任务学习解决训练偏差的深度理解。高分要点：① SSB vs DS 的区分（两个独立问题）② ESMM 数学推导（乘法分解的必要性）③ Loss 设计（作用在全曝光 CTCVR 上而非点击空间 CVR）④ 加分：ESCM² 的 Counterfactual 正则化扩展，或提到 ESMM 的局限（假设 CTR 和 CVR 独立）

### Q4 | 难度: 高级 | 搜索-广告联合排序系统设计

**Q: 某综合平台搜索结果页同时包含自然结果和竞价广告，两套系统独立运行导致用户体验和商业化的矛盾。请设计一套搜索-广告联合排序（STAR）系统，包括：统一打分框架、动态广告插入机制、用户满意度保底约束，以及如何评估系统效果？**

**A:**

**直接结论：** 搜索-广告的矛盾本质是"统一公平竞争"问题——广告不应"特权插队"而应"赢得位置"。核心设计是统一内容质量打分 + 效用最大化混排 + 满意度保底约束，让相关性高的广告获得展示权，低质量广告被自然淘汰。

**系统架构设计：**

```
┌──────────────────────────────────────────────────────┐
│               Query 输入                              │
└──────────────┬──────────────────────┬────────────────┘
               │                      │
    ┌──────────▼──────────┐  ┌────────▼────────────┐
    │  自然检索引擎        │  │  广告竞价引擎        │
    │  Top-K 自然结果      │  │  Top-M 候选广告      │
    └──────────┬──────────┘  └────────┬────────────┘
               │                      │
               └──────────┬───────────┘
                           │ 候选池合并
              ┌────────────▼─────────────────┐
              │    统一相关性打分层            │
              │  BERT Cross-Encoder           │
              │  output: quality_score (0-1)  │
              └────────────┬─────────────────┘
                           │
              ┌────────────▼─────────────────┐
              │    效用最大化混排引擎          │
              │  score_doc = quality          │
              │  score_ad  = quality × eCPM_w │
              └────────────┬─────────────────┘
                           │
              ┌────────────▼─────────────────┐
              │    满意度保底约束层            │
              │  满意度预测 ≥ baseline(85%)   │
              └────────────┬─────────────────┘
                           │
              ┌────────────▼─────────────────┐
              │         最终排序结果          │
              └──────────────────────────────┘
```

**Layer 1 — 统一相关性打分：**

```python
# 同一套 BERT Cross-Encoder 处理自然结果和广告
quality_score = CrossEncoder(query, content)  # 0~1

# 自然结果：直接使用 quality_score
doc_score = quality_score

# 广告：在质量分基础上乘以出价调整因子（保证质量门槛）
# eCPM_normalized = eCPM / max_eCPM_in_pool
ad_score = quality_score × (1 + α × eCPM_normalized)  # α=0.3，避免出价压制质量
```

**Layer 2 — 效用最大化混排：**

页面整体效用函数：
```
U(π) = β₁ × Σ quality_score_i 
      + β₂ × Σ eCPM_i × is_ad_i 
      - β₃ × Diversity_Penalty(π)
      
s.t. 广告数量 ≤ K_max（避免广告轰炸）
     相邻广告间距 ≥ 2（广告布局规则）
```

贪心序列构建（NP-hard 问题的近似解）：
```python
for position in range(1, page_size+1):
    # 动态广告插入：只有广告质量超过阈值才能进入前列
```

**Layer 3 — 满意度保底约束：**

训练满意度预测器（Satisfaction Predictor）：
- 特征：页面广告比例、广告质量分分布、广告相关性
- 标签：用户会话级满意度信号（深度浏览、无立即返回搜索）
- 推理：若预测满意度 < 基线 85%，强制替换最低质量广告为自然结果

**评估框架（双轨制）：**

| 指标维度 | 核心指标 | 采集方式 |
|---------|---------|---------|
| 用户体验 | NPS、会话退出率、搜索重复率 | 用户反馈 + 行为日志 |
| 商业化 | 广告 eCPM、点击深度、广告主 CTR | 广告系统日志 |
| 质量 | 广告相关性@k（人工评估）、广告 CTR | 人工标注 + 系统日志 |
| 系统 | P99延迟、吞吐量 | 链路监控 |

**实验设计：** 双层 A/B（用户层随机 + 广告主层随机），至少持续 2 周，评估用户 NPS 和平台总收益的联合优化效果。

### Q5 | 难度: 开放 | 多渠道归因与预算分配

**Q: 某品牌在信息流、搜索、DSP展示、直播带货四类渠道投放广告。产品经理建议用"末次触点归因"，数据科学家建议用"Shapley值归因"。请从博弈论视角解释两种方案的本质差异，以及 Shapley 值在多渠道归因中的优势与落地挑战？**

**A:**

**直接结论：** 末次触点归因是一种任意约定，缺乏理论支撑，系统性地奖励"收割者"惩罚"播种者"；Shapley 值来自合作博弈论的公理体系，是唯一同时满足效率性、对称性、哑元性和可加性的公平归因方案，但需要枚举所有子集组合（指数复杂度），工业上用 Monte Carlo 近似。

**博弈论视角：**

归因问题是一个**合作博弈**：多个渠道（参与者）合作完成一次转化（创造价值），如何公平分配转化功劳（分配利益）？

**末次触点归因（直觉规则，非博弈论）：**
```
路径：品牌信息流曝光 → DSP展示 → 搜索点击 → 转化
归因：搜索 = 100%，其他 = 0%
```

违反了博弈论公理：
- 违反**对称性**（品牌投资和DSP投资的贡献被强制设为0）
- 违反**效率性**（只有最后一步被"算数"）

**Shapley 值的四条公理：**

| 公理 | 含义 | 对归因的意义 |
|------|------|------------|
| 效率性 | 所有渠道贡献之和 = 总转化价值 | 不浪费，不重复计算 |
| 对称性 | 贡献相同的渠道获相同归因 | 不歧视任何渠道 |
| 哑元性 | 不产生边际贡献的渠道归因=0 | 懒惰渠道不被奖励 |
| 可加性 | 多次活动的归因可叠加 | 大促/日常可分别计算 |

**Shapley 值的计算：**

```python
# 渠道 i 的 Shapley 值 = 该渠道加入所有可能联盟时的平均边际贡献
φ_i = Σ_{S⊆N\{i}} [|S|!(|N|-|S|-1)! / |N|!] × [v(S∪{i}) - v(S)]

# 其中 v(S) = 渠道子集 S 的转化贡献函数（通过历史数据估计）
# N = 所有渠道的集合

# 工业近似（Monte Carlo）：
for _ in range(100000):
    shuffle(channels)  # 随机排列渠道顺序
    for i, channel in enumerate(shuffled_channels):
        marginal = v(prefix_up_to_i) - v(prefix_up_to_{i-1})
        shapley[channel] += marginal
shapley = {k: v/100000 for k, v in shapley.items()}
```

**归因结果对比（示例：4渠道路径）：**

| 渠道 | 末次触点 | 首次触点 | 线性归因(均分) | Shapley值（推荐） |
|------|---------|---------|-------------|----------------|
| 品牌信息流（种草） | 0% | 100% | 25% | 28%（上游高价值） |
| DSP展示（强化）| 0% | 0% | 25% | 21% |
| 搜索（收割）| 100% | 0% | 25% | 35%（仍是最高） |
| 直播（辅助）| 0% | 0% | 25% | 16% |

**Shapley 归因的落地挑战：**

1. **计算复杂度**：4 个渠道 = 2⁴=16 个子集，工业上 20+ 渠道 = 2²⁰ = 100万子集 → 需要 Monte Carlo 近似
2. **v(S) 的估计**：需要足够的历史数据（含每个渠道子集的转化观测）→ 冷启动困难
3. **用户触点对齐**：跨渠道、跨设备的用户 ID 打通是数据基础（无打通则归因无意义）
4. **归因窗口选择**：不同品类（快消 vs 家电）的最优归因窗口不同（7天 vs 90天）
5. **反事实验证**：关闭某渠道做增量实验，验证 Shapley 归因是否真实反映渠道价值

**业务影响（某零售品牌实际数据）：**
- 品牌曝光渠道预算 12% → 28%
- 新用户引入量 +41%
- 整体营销 ROI +22%
- 搜索预算降低 22% 但转化量仅下降 6%（证明搜索的边际价值被高估）

**面试官点评：** 考察统计思维 + 博弈论基础 + 业务洞察。高分要点：① Shapley 值的四条公理（背下来，面试高频）② 末次触点的系统性危害（"奖励收割者"的商业后果）③ Monte Carlo 近似的必要性 ④ 加分：讨论 Shapley 的局限——用历史相关性代理因果，并非真正的增量实验（DID/PSM 才能证明真实增量）
