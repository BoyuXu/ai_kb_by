## 🔍 搜索系统（16题）

### 入门级 ★★★

**S1. BM25 和 TF-IDF 的核心区别是什么？**
- 考点：词频饱和、长度归一化、k1/b 参数调优
- 高频指数：⭐⭐⭐⭐⭐
- 参考资料：search/01_search_ranking.md、knowledge_graph.md §搜索词汇检索
- 快速答题框架：
  - BM25 核心公式中的两个创新：①TF 饱和（避免高频词过度影响）②长度归一化（避免长文档偏差）
  - k1=1.2（默认）控制词频上限，b=0.75（默认）控制长度惩罚强度
  - 工程意义：BM25 是快速检索的标准，混合检索的基准线

**S2. Dense Retrieval（DPR）相比 BM25 的优势和劣势是什么？**
- 考点：双塔模型、向量检索、召回率 vs 延迟权衡
- 高频指数：⭐⭐⭐⭐⭐
- 参考资料：knowledge_graph.md §搜索语义检索、synthesis/20260320_hybrid_retrieval_evolution.md
- 快速答题框架：
  - 优势：长尾 query、口语化 query、跨语言查询都比 BM25 好
  - 劣势：精确词匹配（SKU、品牌）不如 BM25，需要 ANN 索引维护开销大
  - 工程：线上一定是混合使用（BM25 + Dense），RRF 融合

**S3. Query 理解在搜索中的作用是什么？包括哪些模块？**
- 考点：意图识别、实体链接、纠错、改写
- 高频指数：⭐⭐⭐⭐
- 参考资料：interview/deep-questions-search-ads-rec.md Q2、Q23~Q26
- 快速答题框架：
  - 第一层：拼写纠错（编辑距离或语言模型）
  - 第二层：分词和实体识别（精确匹配先过一轮）
  - 第三层：意图分类（导航型/信息型/事务型）
  - 第四层：同义词扩展和 Query 改写

**S4. 位置偏差（Position Bias）在搜索排序里怎么解决？**
- 考点：IPS 校正、倾向分、counterfactual 学习
- 高频指数：⭐⭐⭐⭐
- 参考资料：synthesis/20260322_ad_bias_correction_trilogy.md（虽然是广告但原理相同）
- 快速答题框架：
  - 问题：排名靠前的结果天然点击率高，导致模型学歪
  - 解决：用倾向分（propensity score）校正，P = 1/排名位置（经验公式）
  - 样本加权：loss 时用 1/P 加权，让靠后的点击显得更有价值

**S5. 搜索结果多样性怎么控制？**
- 考点：MMR（最大边际相关性）、DPP、冗余惩罚
- 高频指数：⭐⭐⭐⭐
- 参考资料：interview/scenario-questions.md、knowledge_graph_complete.md §搜索重排多样性
- 快速答题框架：
  - 简单方案：贪心算法，每次选择与已有结果"最相关但最不冗余"的
  - 数学：MMR(i) = αRel(i) - (1-α)max_j Sim(i,j)，其中 j 是已选结果
  - 工业：DPP（行列式点过程）理论最优但计算复杂，通常贪心近似

**S6. 搜索系统的整体架构是什么？各层的延迟目标？**
- 考点：召回→粗排→精排→重排，QPS 和延迟预算
- 高频指数：⭐⭐⭐⭐
- 参考资料：interview/system-design.md、knowledge_graph_complete.md
- 快速答题框架：
  - 召回层（<5ms）：多路召回（BM25/Dense/规则）取 top-K（通常 200-500）
  - 粗排层（<10ms）：轻量级模型或规则快速初筛到 top-50
  - 精排层（<30ms）：深度模型精打细算，返回 top-10
  - 重排层（<5ms）：个性化、多样性、业务规则调整

### 中级 ★★★★

**S7. ColBERT 的 Late Interaction 为什么比 cross-encoder 更高效？**
- 考点：向量设计、索引结构、MaxSim 操作
- 高频指数：⭐⭐⭐⭐⭐
- 参考资料：synthesis/20260323_retrieval_triangle_dense_sparse_late.md、knowledge_graph.md
- 快速答题框架：
  - Cross-encoder：每个 query-doc 对都过一遍模型，O(n) latency（n=候选数）
  - Late Interaction：预先计算 doc 的多个向量，查询时只需做向量对齐，O(1) lookup + O(d) MaxSim（d=维度）
  - 折中点：精度略低于 cross-encoder，但速度快 100 倍
  - 工程：可做"精排 reranker"，在 top-100 候选上又过一遍

**S8. SPLADE-v3 为什么是稀疏检索的 SOTA，相比 v2 改进了什么？**
- 考点：稀疏神经检索、INT8 量化、蒸馏
- 高频指数：⭐⭐⭐⭐
- 参考资料：synthesis/20260321_sparse_dense_retrieval.md、knowledge_graph.md §神经稀疏检索
- 快速答题框架：
  - v3 改进：①双正则化（FLOP + Saturation），稀疏度控制更精细
  - ②MarginMSE + KL 联合蒸馏，学习 teacher（Dense 模型）的排序信号
  - ③INT8 量化感知训练，推理延迟 -40%
  - 指标：BEIR NDCG@10 = 0.546（vs BM25 0.427，vs Dense 0.490）

**S9. 搜索里的混合检索融合策略有哪些？**
- 考点：RRF、LTR、学习融合
- 高频指数：⭐⭐⭐⭐
- 参考资料：knowledge_graph.md §混合检索、synthesis/20260320_hybrid_retrieval_evolution.md
- 快速答题框架：
  - RRF（倒数排名融合）：不需要训练，两路得分独立，稳健（工业主流）
  - LTR 融合：用小数据集标注同步两路检索的候选，训练 pair-wise 模型
  - 三层漏斗：BM25 + Dense 召回 → ColBERT Late Interaction 精排 → LLM Reranker 顶层
  - 工程选型：BM25+Dense+RRF 最常用，如果有标注数据+时间就用 LTR

**S10. 搜索里的长尾 Query（低频查询）怎么优化？**
- 考点：数据稀疏、Query 改写、泛化性
- 高频指数：⭐⭐⭐⭐
- 参考资料：interview/deep-questions-search-ads-rec.md Q49
- 快速答题框架：
  - 数据问题：长尾 query 标注样本极少，排序模型学不好
  - 方案①：Query 改写把长尾改成常见 query（同义词扩展）
  - 方案②：LLM 改写（few-shot learning）
  - 方案③：用用户点击行为的隐含反馈进行学习（无标注）

**S11. 向量检索的 ANN 索引选型，FAISS vs HNSW vs 向量数据库？**
- 考点：索引数据结构、增量更新、内存/速度 tradeoff
- 高频指数：⭐⭐⭐⭐
- 参考资料：interview/deep-questions-search-ads-rec.md Q6
- 快速答题框架：
  - FAISS：离线批量检索快，支持 GPU，不支持增量（需重建）
  - HNSW：在线实时，支持增量，速度快，内存占用相对大
  - Milvus/Weaviate：云原生向量数据库，支持分布式、增量、多副本
  - 选型：Realtime 场景 → HNSW，大规模离线 → FAISS，云环境 → 向量数据库

**S12. 搜索系统的 A/B 测试设计，需要关注哪些 Metric？**
- 考点：因果推断、干扰效应、长期指标
- 高频指数：⭐⭐⭐⭐
- 参考资料：interview/ab_test_experiment_design.md
- 快速答题框架：
  - 短期指标：NDCG（质量）、Recall（覆盖）、CTR（参与度）
  - 长期指标：用户留存（7日、30日）、用户满意度（重复查询率）
  - 干扰效应：一个用户受到多个实验的影响，影响因果推断（需要分层随机化）
  - 显著性：样本量足够大（通常需要 >10000 用户*7 天）

### 高级 ★★★★★

**S13. Query-as-Anchor：搜索个性化的新范式是什么？**
- 考点：上下文用户表示、多粒度融合、场景感知
- 高频指数：⭐⭐⭐
- 参考资料：knowledge_graph.md §搜索个性化、synthesis/20260323_rec_sys_synthesis.md
- 快速答题框架：
  - 传统个性化：用用户历史作为特征，简单但容易 filter bubble
  - Query-as-Anchor：当前 query 作为 Attention Query，用户历史作为 K/V
  - 多粒度表示：会话级（最近 10 条查询）、中期（一周）、长期（一个月）
  - LLM 场景分类：购物/娱乐/工作，不同场景的历史相关性不同
  - 指标：+6.8% NDCG@10，冷启动用户 +12%

**S14. 生成式搜索（AI Overview）与传统搜索的技术差异？**
- 考点：RAG、检索-生成管道、Grounding
- 高频指数：⭐⭐⭐
- 参考资料：synthesis/20260320_llm_recommendation_retrieval.md、interview/deep-questions-search-ads-rec.md Q37
- 快速答题框架：
  - 管道：Query → 检索相关文档 → LLM 读取并生成回答 → 标注来源
  - 核心挑战：①检索要高精度（错了整个回答就错）②生成要 grounded（不能幻觉）③延迟（LLM 推理慢）
  - 工程方案：小模型做检索精排，大模型只负责生成，做并行化减延迟

**S15. 多模态搜索（图文混合）怎么做跨模态对齐？**
- 考点：多模态 embedding、对比学习、CLIP 风格
- 高频指数：⭐⭐
- 参考资料：synthesis/20260323_retrieval_triangle_dense_sparse_late.md
- 快速答题框架：
  - 思路：图片和文本映射到同一 embedding 空间，相关的靠近
  - 训练：用对比学习（Contrastive Learning），正样本对拉近，负样本对推远
  - 数据：需要图文对标注或用现成数据集（LAION）
  - 工程：向量维度（例如 768），检索用 ANN（FAISS）

**S16. 如何系统性评估搜索系统的质量降级？**
- 考点：监控 pipeline、逐层诊断、bad case 分析
- 高频指数：⭐⭐
- 参考资料：interview/deep-questions-search-ads-rec.md Q7
- 快速答题框架：
  - 第一步：确认 metric（NDCG 下跌）发生的时间点和涉及的 query 类型
  - 第二步：分层诊断（召回问题还是排序问题？）
  - 第三步：若是召回问题，查 index 或 query 分析；若是排序问题，比对特征打分
  - 工程工具：trace 系统，把某个 query-doc 的所有特征和中间结果都打印出来

## 🛒 推荐系统（17题）

### 入门级 ★★★

**R1. 推荐系统的漏斗架构是什么？各层的作用和 tradeoff？**
- 考点：召回→粗排→精排→重排，计算成本和精度递增
- 高频指数：⭐⭐⭐⭐⭐
- 参考资料：rec-sys/00_overview.md、knowledge_graph_complete.md
- 快速答题框架：
  - 召回（千+）：多路策略（协同过滤、双塔、热门等）快速生成候选
  - 粗排（百级）：轻量模型（决策树）快速初筛
  - 精排（十级）：深度模型精准预测（CTR、CVR）
  - 重排（最终）：多目标融合、多样性控制、业务规则
  - 设计原则：早期阶段快速过滤，后期精心调优

**R2. 协同过滤（CF）和内容过滤的优缺点比较？**
- 考点：User-based CF、Item-based CF、混合方法
- 高频指数：⭐⭐⭐⭐⭐
- 参考资料：rec-sys/00_overview.md、knowledge_graph_complete.md §推荐召回层
- 快速答题框架：
  - 协同过滤：不需要内容特征，历史行为就足够；但冷启动困难，数据稀疏性强
  - 内容过滤：对冷启动友好（可用初始特征），但需要人工特征工程
  - 混合：用协同过滤的人群相似度初始化，内容特征做补充
  - 工程：生产环境通常两个都有

**R3. DIN（Deep Interest Network）解决了什么问题？**
- 考点：注意力机制、序列建模、目标物品激活
- 高频指数：⭐⭐⭐⭐⭐
- 参考资料：knowledge_graph_complete.md §推荐精排、synthesis/20260322_ad_bias_correction_trilogy.md
- 快速答题框架：
  - 问题：用户历史很长（>100 条），简单平均 embedding 会稀释信号
  - 方案：用注意力机制，当前目标物品（query）激活相关的历史物品
  - 公式：Attention(Query, Keys) = softmax(w·relu(WQ + WK))
  - 效果：比 RNN 和简单 concat 都好，能捕捉用户的短期兴趣

**R4. 多任务学习（MMoE）为什么优于单任务模型？**
- 考点：共享底层、任务塔、expert 混合
- 高频指数：⭐⭐⭐⭐
- 参考资料：synthesis/card_001_multi_task_learning.md、knowledge_graph.md §推荐排序层
- 快速答题框架：
  - 单任务问题：CTR 和 CVR 单独训练，信号割裂
  - MMoE 架构：共享 embedding 层 → 多个 expert（各 128/256 neurons）→ 任务塔（CVR/CTR）
  - 门机制（Gating）：每个任务有独立的门网络，动态混合 experts，让任务自适应
  - 效果：CTR AUC +2~3%，CVR AUC +1~2%

**R5. 推荐中的冷启动问题怎么解决？**
- 考点：新用户、新物品、融合内容特征
- 高频指数：⭐⭐⭐⭐
- 参考资料：interview/deep-questions-recsys-qbank.md（长篇题库）
- 快速答题框架：
  - 新用户：用基础人口属性分类（年龄/性别）或引导问卷
  - 新物品：用内容特征（类目、标签）或用编辑推荐初期曝光
  - 混合策略：冷启动用规则/内容召回，有数据后逐步切到协同过滤
  - 工程：新物品有个"学习期"（1-7 天），线上特殊待遇

**R6. 推荐系统的实时性怎么保证？延迟在哪些环节？**
- 考点：特征获取、模型服务、在线缓存
- 高频指数：⭐⭐⭐⭐
- 参考资料：interview/system-design.md
- 快速答题框架：
  - 延迟源：特征库查询（50ms）> 模型推理（10ms）> 召回过滤（5ms）
  - 优化①：特征 cache 在内存（Redis）
  - 优化②：用轻量模型做在线推理（蒸馏、量化）
  - 优化③：多路召回并行执行
  - 目标：整体 P99 < 200ms（用户感知 1-2 秒内返回）

### 中级 ★★★★

**R7. 超长序列用户建模（超过 1000 条历史）怎么做？**
- 考点：SIM、GEMs、记忆网络、序列截断
- 高频指数：⭐⭐⭐⭐⭐
- 参考资料：knowledge_graph_complete.md §推荐长序列、synthesis/20260323_recommendation_scaling_law_wukong.md
- 快速答题框架：
  - 问题：RNN 无法处理 1000+ 长度序列（梯度消失/爆炸），Transformer O(n²) 显存
  - SIM（Sequential Interest Modeling）：两阶段，先用全局相似度过滤到 top-K，再用目标物品精细激活
  - GEMs（多流解码）：短期流（最近 100）+ 长期流（历史压缩）+ 类目流（特征)，离线预计算，在线 cross-attention
  - 指标：重度用户（>500 条）recall +15%，内存节省 4×

**R8. 生成式推荐的 Semantic ID 是什么，相比传统双塔的区别？**
- 考点：RQ-VAE、离散化、生成式范式
- 高频指数：⭐⭐⭐⭐⭐
- 参考资料：synthesis/20260321_semantic_id_generative_retrieval.md、knowledge_graph_complete.md
- 快速答题框架：
  - Semantic ID：用向量量化（RQ-VAE）把物品 embedding 离散化为整数序列（如 "42-15-68"）
  - 工作原理：推荐问题变成"生成合理的物品 ID 序列"，用 Transformer 自回归生成
  - 优势：冷启动天然支持，多样性好（生成多个候选），端到端学习
  - Spotify 落地：500 万+播客，召回 +25%，向量 size 从 256D 减到 4 bytes（8-16 个 token）

**R9. 推荐系统的多目标优化怎么做，软约束和硬约束的区别？**
- 考点：Pareto 优化、权重学习、MGDA 梯度平衡
- 高频指数：⭐⭐⭐⭐
- 参考资料：knowledge_graph_complete.md §重排多目标、synthesis/20260322_weekly_rec_ads_synthesis.md
- 快速答题框架：
  - 目标冲突：点击率 vs 多样性 vs 长期留存，无法同时最大化
  - 软约束：多个 loss 加权求和（λ1×CTR_loss + λ2×Diversity_loss），权重可调
  - 硬约束：约束条件（留存率 ≥ 90%），用 KKT 条件或 Lagrangian 对偶求解
  - 工程：权重动态调整（如果多样性掉了就提高权重），无需重训

**R10. 推荐的重排（Re-ranking）层需要做哪些事情？**
- 考点：多目标融合、规则调控、多样性、疲劳度
- 高频指数：⭐⭐⭐⭐
- 参考资料：knowledge_graph_complete.md §推荐重排、interview/scenario-questions.md
- 快速答题框架：
  - 多目标分数融合：eCPM = w1×CTR_pred + w2×CVR_pred + w3×LTV_pred
  - 多样性控制：相邻两个推荐物品的类目/属性差异 > 阈值
  - 新鲜度衰减：最近一周上线的物品优先级提高
  - 业务规则：排除禁品、地域限制、品牌竞争对手冲突
  - 疲劳度控制：同一用户最近 7 天看过的物品不再推荐

**R11. 推荐系统中的偏差问题有哪些，怎么治理？**
- 考点：曝光偏差（SSB）、交互偏差、反馈延迟偏差
- 高频指数：⭐⭐⭐⭐
- 参考资料：synthesis/20260322_ad_bias_correction_trilogy.md、knowledge_graph_complete.md
- 快速答题框架：
  - 曝光偏差（SSB）：只有被推荐的物品才有交互标签，ESMM 全空间多任务学习解决
  - 交互偏差：点击不等于满意（用户可能误点或找不到），用停留时长校正
  - 时间偏差：用户今天点击，三天后才转化，样本标签延迟问题
  - 治理方案：倾向分（propensity score）加权、因果推断、对抗去偏

**R12. 推荐系统的 A/B 测试怎么做，怎么处理干扰效应？**
- 考点：分流、显著性检验、长期效应
- 高频指数：⭐⭐⭐⭐
- 参考资料：interview/ab_test_experiment_design.md
- 快速答题框架：
  - 干扰效应（Network Effect）：一个用户的推荐结果影响他朋友的行为（社交网络）
  - 分流策略①：用用户 ID 的哈希固定分流，同一用户体验一致
  - 分流策略②：用聚类分流（地理位置/社交圈），减少干扰
  - 指标选择：点击率短期效应强，但长期可能有反向作用（用户倦怠），需要看 7/30 日留存

**R13. 推荐里的推理增强（Reasoning）是什么新方向？**
- 考点：CoT、思维链、冷启动利用先验
- 高频指数：⭐⭐⭐
- 参考资料：synthesis/20260323_rlvr_vs_rlhf_posttraining.md、knowledge_graph_complete.md
- 快速答题框架：
  - 传统推荐：直接从行为特征预测分数（一步到位）
  - 推理增强：LLM 先"思考"用户为什么需要这个物品，再生成推荐
  - 格式：<think>用户最近搜索了"跑步鞋"，有运动偏好，推荐 Nike 跑鞋</think> → 推荐 ID
  - 冷启动场景最有价值：行为数据少，但常识推理能弥补

### 高级 ★★★★★

**R14. Wukong（Meta 的推荐 Scaling Law）核心发现是什么？**
- 考点：参数配置、Embedding:MLP 比例、幂律规律
- 高频指数：⭐⭐⭐
- 参考资料：synthesis/20260323_recommendation_scaling_law_wukong.md、knowledge_graph_complete.md
- 快速答题框架：
  - 发现：推荐模型的最优参数配置 Embedding:MLP ≈ 10:1（vs LLM 的稠密矩阵）
  - 原因：推荐系统参数集中在 embedding table（极稀疏），而 LLM 是密集矩阵
  - 工程意义：设计推荐模型时，如果参数预算是 10B，分配约 10B embedding + 1B MLP
  - 幂律规律：性能 ∝ 参数量 ^ α，α ≈ 0.1（类似 LLM 的 Chinchilla 规律）

**R15. 推荐系统的线上线下 metric 一致性如何保证？**
- 考点：样本选择偏差、评估方法、线上 offline 反馈延迟
- 高频指数：⭐⭐
- 参考资料：interview/deep-questions-recsys-qbank.md、synthesis/20260322_weekly_rec_ads_synthesis.md
- 快速答题框架：
  - 线下评估通常用 holdout test set（历史数据分割）
  - 问题：线下用的是"已发生"的交互，而线上是"预测未来"的交互
  - 因果推断：必须用线上 A/B 测试验证，线下 AUC 提 2%，线上点击可能只提 0.5%
  - 根本解决：建立反馈循环，收集线上行为作为下一轮的训练数据

**R16. 推荐系统的实时反馈机制怎么设计？**
- 考点：点击日志、强化学习反馈、Bandit 问题
- 高频指数：⭐⭐
- 参考资料：interview/scenario-questions.md
- 快速答题框架：
  - 延迟问题：用户交互（点击/转化）需要秒级或分钟级反馈到模型
  - 架构：User Action → 消息队列 → 特征更新 → 模型重训
  - 强化学习角度：推荐是 contextual bandit 问题，每次推荐是一个臂
  - 工程方案：在线学习（incremental update）比全量重训快，但稳定性需要保证

**R17. 如何评估推荐系统中不同召回路的贡献度？**
- 考点：对照实验、移除法、特征重要性
- 高频指数：⭐⭐
- 参考资料：interview/system-design.md
- 快速答题框架：
  - 问题：多路召回融合后，无法看清每路的真实贡献（1+1 可能 >2 due to synergy）
  - 方法①：移除法（移除某路，测试效果差异）
  - 方法②：上下文模拟（模拟某路不可用，复现效果）
  - 方法③：特征重要性分解（如 Shapley value），精准定量
  - 工程用途：判断是否值得维护这路召回（成本 vs 收益）

## 📢 广告系统（15题）

### 入门级 ★★★

**A1. CTR 预估从 LR 到 DNN 再到 Transformer，每一步解决了什么？**
- 考点：特征交叉、表示学习、注意力机制
- 高频指数：⭐⭐⭐⭐⭐
- 参考资料：interview/mock-interview-ads.md Q1、knowledge_graph_complete.md
- 快速答题框架：
  - LR：线性模型，特征交叉要手工组合（成本高）
  - FM：自动二阶交叉，但高阶交叉还是不行
  - DNN：任意阶交叉，但低阶信号在高维空间被稀释
  - DeepFM：显式+隐式交叉结合，Wide 处理低阶，Deep 处理高阶
  - Transformer/Attention：动态特征相关性，不同样本的特征重要性不同

**A2. GSP 竞价和 VCG 竞价的区别，为什么行业在切 FPA（第一价格拍卖）？**
- 考点：拍卖理论、激励相容、策略出价
- 高频指数：⭐⭐⭐⭐⭐
- 参考资料：knowledge_graph_complete.md §广告竞价、interview/mock-interview-ads.md Q2
- 快速答题框架：
  - GSP：出价排序，按第二名价格付费（广义第二价格），工业标准但非 truthful
  - VCG：理论上 truthful（出真实价值最优），但计算复杂，在多位置场景实现困难
  - FPA：出价排序，按自己价格付费（第一价格），看似不公平，但工业发现它更稳健
  - 趋势：Google、Facebook、腾讯等都在切 FPA（需要智能出价系统支撑）

**A3. 广告系统的预算控制（Pacing）怎么做？**
- 考点：PID 控制、对偶梯度、流量不均
- 高频指数：⭐⭐⭐⭐⭐
- 参考资料：knowledge_graph_complete.md §广告预算、synthesis/20260320_ads_budget_pacing.md
- 快速答题框架：
  - 问题：流量不均匀（早/晚高峰不同），简单比例 throttle 会预算不平匀
  - Throttling：随机丢弃竞价，简单但粗糙
  - PID 控制：根据实际消耗速度 vs 目标速度偏差，实时调整参与率（主流方案）
  - 对偶方法：预算约束用 Lagrangian 乘子松弛，迭代求解（理论最优）
  - 工程：每小时更新一次目标速度，避免微秒级抖动

**A4. CTR 预估中的样本选择偏差（SSB）是什么，ESMM 怎么解决的？**
- 考点：全空间多任务、样本权重、协同学习
- 高频指数：⭐⭐⭐⭐
- 参考资料：knowledge_graph_complete.md §广告CTR、synthesis/20260322_ad_bias_correction_trilogy.md
- 快速答题框架：
  - 问题：CVR 标签只存在于被点击的样本，但实际需要预测"曝光后转化"
  - 数据集偏差：训练集 (clicked, converted) 的分布 ≠ 线上 (exposed, converted) 的分布
  - ESMM（Entire Space Multi-Task Model）：
    - p(CTCVR) = p(CTR) × p(CVR|click)，用全量曝光样本
    - 共享 embedding，CVR tower 仅用点击样本，CTR tower 用全量样本
    - 间接学习 CVR 的无偏估计
  - 效果：CVR AUC +3.4%，冷启动 +5.2%

**A5. 广告定向（Targeting）有哪些方式？隐私政策变化怎么应对？**
- 考点：人群定向、行为定向、Lookalike、第三方数据
- 高频指数：⭐⭐⭐⭐
- 参考资料：knowledge_graph_complete.md §广告定向、interview/deep-questions-search-ads-rec.md Q19~Q22
- 快速答题框架：
  - 人口属性：年龄、性别、地理位置（基础）
  - 行为定向：用户点击/购买历史（需要 Cookie）
  - Lookalike：相似人群扩展（种子用户 → 相似用户）
  - 隐私变化：iOS14+ 限制 IDFA → 转向一方数据（自有平台用户）+ 联邦学习 + 隐私计算
  - 新方向：Cohort（同群体）而非个人精准定向

### 中级 ★★★★

**A6. 多目标广告优化怎么做？如何平衡收入和用户体验？**
- 考点：Pareto 优化、约束优化、实时权重调整
- 高频指数：⭐⭐⭐⭐
- 参考资料：knowledge_graph_complete.md §广告多目标、synthesis/20260322_weekly_rec_ads_synthesis.md
- 快速答题框架：
  - 目标冲突：收入（CTR×Bid）vs 用户体验（关闭率低）vs 广告主 ROI
  - 软约束：Loss = λ1×CTR_loss + λ2×Quality_loss + λ3×UX_loss
  - 硬约束：用户负反馈比例 < 5%，否则强制约束
  - 权重调整：监控填充率，如果广告过多就减少权重，无需重训
  - 效果：次日留存 +2.3%，总收益（含 LTV）+4.8%，投诉 -18%

**A7. 自动出价（AutoBid）系统怎么用 RL 优化，核心的约束是什么？**
- 考点：Constrained MDP、Lagrangian 对偶、RL 出价
- 高频指数：⭐⭐⭐⭐⭐
- 参考资料：knowledge_graph_complete.md §广告AutoBid、synthesis/20260321_ad_bidding_evolution.md
- 快速答题框架：
  - MDP 框架：状态 = (剩余预算, 当前时段, 竞争强度)，动作 = 出价系数 k，奖励 = 转化数
  - 约束问题：不能超过预算、不能超过目标 CPA（约束 MDP）
  - 求解：Lagrangian 松弛，把硬约束转为乘子惩罚项
  - 训练：用历史竞拍数据建立 offline simulator，DDPG/A3C 训练
  - 部署：安全性考虑，先小流量灰度，监控 CPA 不超目标
  - 指标：转化量 +12.3%，CPA 达标率 78% → 94.2%，预算利用率 82% → 96.1%

**A8. 广告里的点击欺诈（Click Fraud）怎么检测和防护？**
- 考点：异常检测、规则引擎、机器学习模型
- 高频指数：⭐⭐⭐⭐
- 参考资料：interview/deep-questions-search-ads-rec.md Q47
- 快速答题框架：
  - 规则层：同一 IP 短时间内点击同一广告多次 → 拦截
  - 特征层：用户特征异常（不可能出现的地理跳跃）、设备异常（bot）
  - 模型层：训练二分类器，特征包括点击间隔、用户历史转化率、设备标签
  - 反欺诈价值：可以回收被欺诈消耗的预算（3-10%），商家很敏感
  - 工程：实时 hard block（确定欺诈）+ 离线 soft block（可疑，监控）

**A9. 广告系统中的多触点归因（Multi-touch Attribution）怎么做？**
- 考点：接触路径、归因模型、Data-driven vs 规则
- 高频指数：⭐⭐⭐⭐
- 参考资料：knowledge_graph_complete.md §广告归因、interview/deep-questions-search-ads-rec.md Q28
- 快速答题框架：
  - 问题：用户通常接触多个广告后才转化，该归因给谁？
  - 规则模型：
    - First Touch：所有功劳给第一个广告（获客归因）
    - Last Touch：所有功劳给最后一个（主流，但容易跟风）
    - Linear：平均分给所有广告
    - Time Decay：越近的广告权重越高
  - Data-driven：用转化用户的接触路径训练模型，预测每个 touch 的贡献
  - 工程：通常用 Shapley value 分解（理论最公平）

**A10. 广告创意优化怎么做，动态创意优化（DCO）的原理？**
- 考点：A/B 测试创意、多臂赌博、创意个性化
- 高频指数：⭐⭐⭐⭐
- 参考资料：knowledge_graph_complete.md §广告创意
- 快速答题框架：
  - 静态创意：广告主设计一套创意，所有用户看同一个
  - DCO：为不同用户动态组合创意元素（标题、图片、颜色）
  - 工作流：素材库（500 张图+1000 个文案）→ 组合生成（500×1000 个创意）→ 机器学习排序 → 个性化展示
  - Bandit 算法：Thompson Sampling 探索-利用，自动淘汰低效创意
  - LLM 应用：LLM 生成文案 → CTR 模型评分 → 筛选 top-K 投放
  - 效果：CTR +15-30%（取决于素材质量）

**A11. 广告系统中的偏差治理三部曲（位置偏差、SSB、兴趣坍塌）是什么？**

## 🧠 LLM 基础设施（8题）

### 入门级 ★★★

**L1. KV Cache 优化的核心思想是什么，有哪些方向？**
- 考点：显存瓶颈、量化、驱逐、注意力稀疏
- 高频指数：⭐⭐⭐⭐⭐
- 参考资料：knowledge_graph.md §KV Cache、synthesis/20260320_kv_cache_compression.md
- 快速答题框架：
  - 问题：KV Cache 是推理时的显存杀手，序列长度越长越严重
  - 量化：FP16 → INT8（-50% 显存），需要处理异常值
  - 驱逐：H2O 算法（Heavy Hitter Only），保留关键 token（Attention Sink）
  - 稀疏注意力：不是所有 token 都需要关注
  - 工程组合：三个方向正交，可叠加使用

**L2. FlashAttention 为什么比标准 Attention 快那么多？**
- 考点：IO-aware 算法、显存层次、分块计算
- 高频指数：⭐⭐⭐⭐⭐
- 参考资料：knowledge_graph.md §Attention计算、synthesis/20260321_flashattention3_llm_infra.md
- 快速答题框架：
  - 标准 Attention：QK^T 矩阵大小 O(n²)，爆显存
  - FlashAttention-1：分块加载，O(n) 显存（从 HBM → SRAM）
  - FA2 改进：支持 Ampere GPU，并行度更好
  - FA3（最新，H100 深度优化）：
    - Warp 专业化（Producer/Consumer 异步流水线）
    - Softmax 延迟隐藏在计算空隙
    - FP8 支持，120 TFLOP/s（vs BF16 的 75%）
  - 效果：端到端训练 -25% step time，显存节省 3~4×

**L3. MoE（混合专家）推理为什么比稠密模型慢？怎么优化？**
- 考点：Expert dispatch、通信开销、流水线
- 高频指数：⭐⭐⭐⭐⭐
- 参考资料：knowledge_graph.md §MoE推理、synthesis/20260320_moe_disaggregated_inference.md
- 快速答题框架：
  - 瓶颈：Dispatch（token 到 expert 的路由）本身成本高，跨 GPU 通信延迟大
  - 传统方案：Attention 和 FFN 串行，一个 expert 不能完全用满 GPU
  - MegaScale-Infer 优化：
    - 解耦 Attention 和 FFN（各用各的 expert set）
    - 乒乓流水线：一组 GPU 算 Attention，另一组算 FFN，隐藏延迟
    - M2N 通信库：针对 token dispatch 优化（零拷贝）
  - 效果：DeepSeek-V3、Qwen3 都是 MoE，推理速度虽慢但效果好

### 中级 ★★★★

**L4. GRPO vs PPO+RLHF：为什么 GRPO 更适合可验证任务？**
- 考点：RL 对齐、Critic 网络、组内相对优化
- 高频指数：⭐⭐⭐⭐
- 参考资料：synthesis/20260321_grpo_rl_alignment.md、knowledge_graph.md §LLM对齐
- 快速答题框架：
  - PPO+RLHF 流程：生成多个回答 → Reward 模型评分 → Critic 学习 → PPO 优化（显存 3x）
  - GRPO 改进：
    - 无 Critic：直接在生成的回答组内（G 个）做相对排序
    - 组内归一化：advantage = (score_i - mean) / std，消除 reward scale 问题
    - 显存节省 50%（不需要 Critic 网络）
  - 适用：数学题、代码（有明确正确答案），不适合开放生成
  - 案例：DeepSeek-R1-Zero 纯 GRPO 训练，自发产生 CoT

**L5. Speculative Decoding 怎么加速推理，关键是什么？**
- 考点：草稿模型、接受率、KV Cache 重用
- 高频指数：⭐⭐⭐⭐
- 参考资料：knowledge_graph.md §Speculative Decoding
- 快速答题框架：
  - 思路：小模型快速生成 K 个 token（草稿），大模型批量验证
  - 关键：草稿模型的分布要对齐大模型，不能偏太远
  - 对齐方法：LoRA 微调小模型，KL 蒸馏损失
  - 工程：如果验证成功，K 个 token 一起返回；失败则回退（无开销）
  - 指标：接受率 ~15% 提升，端到端加速 1.3x（旧版），2.5x（FA3 配合）

### 高级 ★★★★★

**L6. KV Cache 优化的"红线"是什么，为什么不能驱逐 Attention Sink？**
- 考点：Attention Sink 现象、开头 token 重要性、驱逐策略边界
- 高频指数：⭐⭐
- 参考资料：knowledge_graph.md §KV Cache优化、synthesis/20260323_kvcache_and_llm_inference_optimization.md
- 快速答题框架：
  - 现象：LLM 倾向于把很多 attention weight 压到开头的几个 token（[BOS], first few），称为 Attention Sink
  - 原因：可能是对 position encoding 的伪适应，或者是模型的固有行为
  - 驱逐陷阱：如果删除了 Attention Sink（开头 token），模型性能坍塌
  - 工程红线：驱逐策略（如 H2O）必须保留开头 token，不能动
  - 启示：H2O only 驱逐"中间"和"最近"的重要 token，开头是禁区

**L7. LLM 在搜广推各领域的赋能路径分别是什么？**
- 考点：LLM 应用场景、冷启动、多模态、推理
- 高频指数：⭐⭐⭐
- 参考资料：synthesis/20260320_llm_recommendation_retrieval.md、knowledge_graph_complete.md
- 快速答题框架：
  - 搜索：Query 改写 + Query 意图分类 + 文档重排（Cross-Encoder） + 生成式搜索（RAG）
  - 推荐：冷启动（常识推理补充）→ 召回（LLM 生成候选）→ 重排（LLM judge）→ 推理（CoT 解释）
  - 广告：创意生成（文案 + 图片）→ 质量评估（LLM judge）→ 出价建议（策略推荐）
  - 共同点：都在"补充不足的数据"或"提升决策质量"

**L8. LLM 推理优化的下一步方向是什么？**
- 考点：量化、蒸馏、边缘部署、多模态
- 高频指数：⭐⭐
- 参考资料：synthesis/20260322_llm_efficiency_trifecta.md、knowledge_graph.md
- 快速答题框架：
  - 量化方向：INT4/INT8 大规模部署，精度影响可控
  - 蒸馏：小模型替代大模型，成本 1/100（参数）但性能维持 80-90%
  - 边缘部署：手机/IoT 端部署 7B 级小模型，隐私更好
  - 多模态：LLM + Vision Transformer，端到端学习
  - 跨领域联动：KV Cache 优化 + 量化 + 蒸馏 + 混合精度，三管齐下
