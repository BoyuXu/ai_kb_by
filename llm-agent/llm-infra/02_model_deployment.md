## 一、大模型推理优化

### 1. 量化技术

#### 1.1 精度类型对比

| 精度 | 位数 | 存储/参数 | 适用场景 |
|------|------|-----------|----------|
| FP32 | 32位 | 4字节 | 训练阶段，需要高精度梯度更新 |
| FP16 | 16位 | 2字节 | 混合精度训练/推理，计算加速 |
| BF16 | 16位 | 2字节 | 更大动态范围，TPU常用 |
| INT8 | 8位 | 1字节 | 推理阶段，显著加速 |
| INT4 | 4位 | 0.5字节 | 极致压缩，边缘部署 |
| FP8 | 8位 | 1字节 | 训练和推理，平衡精度与性能 |

**FP16 vs BF16 对比**：
- **BF16**：1位符号 + 8位指数 + 7位尾数，动态范围与FP32相同，适合梯度计算
- **FP16**：1位符号 + 5位指数 + 10位尾数，精度更高但动态范围较小

#### 1.2 GPTQ 量化

**特点**：
- 一次性量化方法，无需重新训练
- 支持极端量化（2-3-4 bit），几乎不影响模型准确性
- 采用"Lazy Batch-Updates"方法，逐块压缩模型
- 支持175B参数级别的大模型量化

**优势**：
- 减小GPU内存需求，可在小显存GPU上运行大模型
- 提高GPU利用率，批量更新操作
- A100上约3.25倍性能提升，A6000上4.5倍提升

#### 1.3 AWQ 量化（Activation-aware Weight Quantization）

**原理**：
- 基于激活值分布挑选显著权重进行量化
- 不依赖反向传播或重建，保持模型泛化能力
- 属于训练后量化(PTQ)方法

**核心技术**：
- 计算scale系数tensor，对激活除以该tensor，对权重乘以该tensor
- 降低权重量化难度，可采用常规group量化

#### 1.4 量化方法选择指南

| 需求 | 推荐方法 |
|------|----------|
| 快速部署，精度损失小 | GPTQ (4-bit) |
| 保持泛化能力，不过拟合校准集 | AWQ |
| 极致压缩，边缘设备 | INT4/INT3 + GPTQ |
| 训练时量化 | QAT (Quantization-Aware Training) |
| 动态精度调整 | 混合精度量化 |

### 2. 剪枝技术

#### 2.1 剪枝粒度分类

1. **突触剪枝**：剪掉神经元之间不重要的连接（权重置0）
2. **神经元剪枝**：直接裁剪某个节点（某行某列置零）
3. **权重矩阵剪枝**：整个权重矩阵裁剪，保留最重要的部分

#### 2.2 剪枝时机

- **结构化剪枝**：移除整个卷积核或神经网络层
- **非结构化剪枝**：随机移除单个权重
- **静态剪枝**：训练后进行
- **动态剪枝**：训练过程中动态调整

#### 2.3 稀疏模型

**优势**：
- 高效计算：减少不必要的计算步骤
- 显存节省：参数和激活值稀疏，显存需求降低
- 模型可扩展性：更容易扩展到超大规模

**挑战**：
- 硬件支持有限：GPU主要优化稠密计算
- 过度稀疏化可能影响性能
- 训练复杂度增加

### 3. 模型蒸馏

#### 3.1 基本原理

使用大型模型（教师模型）的输出（软标签）训练小型模型（学生模型）：
- 软标签包含类别概率信息，比硬标签更丰富的信息
- 学生学习到更细致的决策边界
- 通过**温度参数(T)**调整软标签的平滑程度

#### 3.2 蒸馏步骤

1. **训练教师模型**：大型深度网络达到SOTA精度
2. **生成软标签**：用教师模型预测训练集，记录类别概率
3. **训练学生模型**：使用真实标签+软标签的加权和作为损失
4. **评估学生模型**：验证性能是否接近教师模型

#### 3.3 温度调整

$$
q_i = \frac{\exp(z_i/T)}{\sum_j \exp(z_j/T)}
$$

- 较高温度使概率分布更平滑，学生学到更细微差别
- 损失函数：交叉熵损失计算学生预测与软标签差异

#### 3.4 蒸馏优势

- **效率提升**：学生模型更小更快，适合资源受限环境
- **泛化能力**：通过学习软标签获得更好的泛化能力
- **多任务适用**：分类、检测、分割、NLP等广泛适用

## 三、PagedAttention & KV Cache管理

### 1. 为什么需要PagedAttention

LLM推理分为两个阶段：
1. **Prefill阶段**：整段prompt做forward计算，生成KV Cache
2. **Decode阶段**：逐token生成，复用KV Cache

**传统KV Cache问题**：
- 序列长度不确定，难以预先分配存储空间
- 显存碎片化严重
- 并行采样时无法共享prompt的KV Cache

### 2. PagedAttention原理

借鉴操作系统虚拟内存分页管理：
- 将KV Cache划分为固定大小的**块（block）**
- 块在显存中不必连续
- 通过**block table**将逻辑块映射到物理块
- 物理块按需分配，只有最后一个block可能浪费（<4%）

### 3. 内存共享机制

**并行采样优化**：
- 同一个prompt生成多个输出序列时共享KV Cache
- 通过block table映射到同一物理块
- 引用计数跟踪，实现"写时复制"(Copy-on-Write)
- **效果**：显存占用减少55%，吞吐量提升2.2x

### 4. vLLM调度器与Block分配

**调度器三个队列**：
- `waiting`：用户刚输入的序列
- `running`：正在推理的序列
- `swapped`：显存不足时换出的序列

**Block分配器**：
- `UncachedBlockAllocator`：正常token的KV Cache分配
- `CachedBlockAllocator`：带前缀缓存的分配器，支持LRU驱逐

## 四、连续批处理（Continuous Batching）

### 1. 传统批处理（Static Batching）问题

- 一个batch输入后，batch size保持不变直到最后一个输出结束
- 不同输入输出长度差异大，造成大量空白等待
- GPU利用率低

### 2. 连续批处理原理

- 当一个请求生成结束后，立即插入新请求
- Batch size动态变化
- 减少等待时间，提高吞吐量

**效果对比**：
- vLLM使用continuous batching获得**23倍**吞吐量提升
- 效果优于naive batching **8倍**

### 3. 动态批处理（Dynamic Batching）

Triton Inference Server支持：
- 将一个或多个推理请求合并成一个批次
- 最大化吞吐量
- 支持多模型并发推理

## 五、推测解码（Speculative Decoding）

### 1. 基本原理

使用小型草稿模型（Draft Model）预测多个token，然后用大模型并行验证：
- 草稿模型快速生成候选token序列
- 目标大模型并行验证多个候选token
- 接受则继续，拒绝则回退重新生成

### 2. 核心思想

- 利用小模型的推理速度优势
- 大模型并行验证减少总推理步数
- 保持大模型输出质量的同时加速推理

### 3. 效果

- 在某些场景下可实现**2-3倍**加速
- 特别适合需要快速响应的交互式应用
- 草稿模型选择是关键（通常小3-5倍的同类模型）

## 六、多卡部署

### 1. 并行策略对比

| 并行方式 | 拆分对象 | 通信开销 | 显存优化 | 适用场景 | 设备依赖 |
|----------|----------|----------|----------|----------|----------|
| **数据并行(DP)** | 训练数据 | 低（梯度all-reduce） | 不减少模型显存 | 数据量大、模型可单卡容纳 | 无限制 |
| **张量并行(TP)** | 单层模型权重 | 高（层内all-reduce） | 减少单卡参数存储 | 单层过大（如Attention层） | 需同节点高带宽 |
| **流水线并行(PP)** | 模型层 | 中（层间点对点） | 减少单卡层数存储 | 模型层数多（百层Transformer） | 可跨节点 |

### 2. 通俗理解（汽车工厂类比）

假设训练12层Transformer，使用4台GPU：

- **数据并行**：4个工厂各复制全套生产线，分别生产不同车型，每天同步设计图
- **流水线并行**：生产线拆成4段（每GPU负责3层），依次处理
- **张量并行**：单个工序由4个工人协作完成，各自负责部分工作

**实际组合（PTD-P策略）**：TP解决单层显存问题 + PP扩展层数 + DP提升数据量

### 3. 多卡推理框架支持

**vLLM多卡部署**：
```python
llm = LLM(model="model_path", tensor_parallel_size=4)
```

**TensorRT-LLM多卡部署**：
- 支持单机多卡（NCCL）
- 支持多机多卡
- 支持张量并行+流水线并行组合

**DeepSpeed-Inference**：
- 内存优化、计算优化、通信优化
- 支持多GPU横向扩展
- 单卡可推理数百亿参数模型

### 4. 选型建议

| 场景 | 推荐方案 |
|------|----------|
| 模型可单卡容纳 | 数据并行 |
| 单层参数过大 | 张量并行 |
| 层数过多 | 流水线并行 |
| 超大规模模型（>100B） | TP + PP + DP 组合 |
| 超长序列 | Ring Attention + 多卡 |

## 七、补充：关键优化技术

### 1. FlashAttention

**核心思想**：IO感知的精确注意力计算
- 避免频繁从HBM读写N×N注意力矩阵
- 利用高速SRAM进行分块计算
- 显存复杂度从O(N²)降低到O(N)

**实现方式**：
1. 将输入序列划分为大小为B的小块
2. 加载Q、K、V到高速缓存
3. 在块内计算缩放点积注意力
4. 累积结果，避免中间结果全局存储

### 2. MQA & GQA

**MHA (Multi-Head Attention)**：QKV三部分有相同数量的头

**MQA (Multi-Query Attention)**：
- Q保持多头，K和V只有单个头
- 所有Q头共享一组KV头
- 提升30%-40%吞吐量

**GQA (Group-Query Attention)**：
- MHA和MQA的折衷方案
- 分组共享KV头
- 平衡性能和精度

**加速原理**：
1. 降低从内存读取的数据量，减少计算单元等待时间
2. KV Cache变小，可加大batch size

### 3. 推理性能评估指标

| 指标 | 说明 | 重要性 |
|------|------|--------|
| **Throughput** | 总吞吐（output tokens/seconds） | 公共服务核心指标 |
| **TTFT** | Time to First Token，首包延迟 | 流式输出体验关键 |
| **Time per output token** | 每个token生成时间 | 影响流畅度 |
| **Latency** | 完整请求处理时间 | 实时性要求 |
| **QPS** | 每秒处理请求数 | 服务能力指标 |

## 八、面试常见问题

### Q1: 为什么选择某种量化方法？

**回答要点**：
- 精度要求：QAT > GPTQ > AWQ > PTQ
- 部署速度：PTQ最快，无需重新训练
- 模型规模：超大模型用GPTQ/AWQ，小模型可考虑INT8
- 硬件支持：不同硬件对不同精度支持不同

### Q2: vLLM的PagedAttention与传统KV Cache的区别？

**回答要点**：
- 传统：预分配连续显存，浪费严重，无法共享
- PagedAttention：分块管理，非连续存储，按需分配
- 支持内存共享和写时复制，显存效率提升55%

### Q3: 如何选择并行策略？

**回答要点**：
- 先判断瓶颈：是单层太大（TP）还是层数太多（PP）还是数据太多（DP）
- 实际场景通常是组合使用
- 考虑通信成本：TP需要高带宽（NVLink），PP可跨节点

### Q4: 推理框架选型考虑因素？

**回答要点**：
- 硬件环境：是否NVIDIA GPU（TensorRT-LLM）
- 并发需求：高并发选vLLM，低延迟选TensorRT-LLM
- 易用性：Ollama最简单，TGI生态最好
- 模型类型：是否支持目标模型架构
