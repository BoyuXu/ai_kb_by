# Adaptive Iterative RAG for Multi-Hop Reasoning
> 来源：arXiv | 领域：LLM基础设施 | 学习日期：20260327

## 问题定义
标准RAG（Retrieval-Augmented Generation）只做一次检索，对需要多跳推理的复杂问题效果差：
1. **单跳局限**：问题"谁是X公司CEO的母校校长？"需要先找X公司CEO，再找其母校，再找校长
2. **检索浪费**：简单问题不需要检索（LLM已知答案），但每次都检索浪费资源
3. **检索质量下降**：多跳问题的中间查询难以构建（第一跳的答案未知）

**目标**：自适应迭代RAG，动态判断是否需要检索，并通过多轮迭代完成复杂推理，MultiHop QA +15% EM（Exact Match）。

## 核心方法与创新点

### 1. 检索必要性判断
在每步生成前，LLM判断是否需要检索：
```
<NEED_RETRIEVAL>: 当前问题超出LLM知识范围，或需要最新信息
<NO_RETRIEVAL>: 基于已有知识和检索结果可以直接回答
```

通过在训练数据中添加这两个特殊token，让LLM学会自主决策。

### 2. 迭代检索-生成循环
```
Loop:
  1. LLM生成中间推理步骤（"X公司的CEO是张三"）
  2. 判断是否需要下一跳检索
  3. 构建子查询（"张三的母校是哪所大学？"）
  4. 检索相关文档
  5. 基于新检索结果继续推理
Until 最终答案生成
```

最大迭代轮数 T_max（通常3-5轮），防止无限循环。

### 3. 检索质量自评估
LLM评估检索结果是否与问题相关：

$$
\text{ISREL}(d, q) \in \{0, 1\}
$$

不相关文档被过滤，防止噪声干扰推理。

### 4. 反思机制（Self-Reflection）
生成答案后，LLM反思答案的可信度：

$$
\text{ISSUP}(\hat{a}, d_{retrieved}) \in \{\text{Fully Supported}, \text{Partially}, \text{Not Supported}\}
$$

不支持的答案触发重新检索。

## 实验结论
- **MultiHop QA（HotpotQA）**：EM +15%（相比标准RAG）
- **简单问题**：检索次数减少40%（不必要检索被跳过），延迟降低
- **FEVER（事实验证）**：准确率 +8%（反思机制的效果）
- 平均检索轮数：2.3轮（对比固定3轮节省约24%检索调用）

## 工程落地要点
1. **检索延迟累积**：每次检索增加100-300ms，多轮检索总延迟可达1s+，需要设置最大轮数
2. **缓存中间检索结果**：相同子查询的检索结果可以缓存，避免重复检索
3. **检索失败处理**：某轮检索未找到相关文档时，需要改写子查询或回退
4. **成本控制**：迭代RAG的LLM Token消耗是标准RAG的2-5x，需要在效果和成本间权衡
5. **流式输出**：迭代推理过程可以流式输出中间步骤，改善用户体验

## 面试考点
Q1: 自适应检索（判断是否需要检索）的核心技术是什么？
A: 核心是让LLM学会元认知（Metacognition）—了解自己知道什么、不知道什么。技术实现：(1)Special Token方式：训练时在需要检索的位置插入<RETRIEVE>特殊token，让LLM在推理时自主预测是否生成这个token；(2)Confidence-based：LLM输出当前答案的置信度，低于阈值时触发检索；(3)Classifier方式：训练独立分类器判断是否需要检索。Special Token方式效果最好，但需要专门训练数据。

Q2: 多跳推理中的子查询构建有什么挑战？
A: (1)依赖关系：第二跳查询依赖第一跳的结果（"张三的母校"中，"张三"来自第一跳），实体链接错误会传播；(2)查询多样性：同一知识需要从不同角度检索，单一子查询可能遗漏；(3)查询退化：多轮后子查询越来越短或越来越通用，检索质量下降；(4)停止条件：何时判断已有足够信息可以生成最终答案，避免过度检索。

Q3: RAG的知识截止日期（Knowledge Cutoff）问题如何彻底解决？
A: 完全解决很难，但可以缓解：(1)实时检索：对时效性强的问题（股价、新闻），接入实时检索API；(2)检索新鲜度排序：在检索结果排序中加入文档时间权重；(3)日期感知提示：在Prompt中注入当前日期，让LLM意识到时间差；(4)混合记忆：LLM参数化知识（静态）+实时检索（动态）；(5)持续学习：定期用新数据fine-tune LLM（成本高，适合关键业务场景）。
