# Recurrent Drafter for Fast Speculative Decoding
> 来源：工业研究论文（Apple Research）| 日期：20260319

## 问题定义
投机解码（Speculative Decoding）用小型Draft模型生成候选token序列，大模型并行验证，显著加速推理。但现有Draft模型（独立小LM）参数量大、内存占用高，且与主模型独立训练导致分布偏差。Recurrent Drafter提出用循环网络（RNN结构）替代小LM作为Draft模型，大幅降低开销。

## 核心方法与创新点
1. **循环Draft模型**：用轻量RNN（而非Transformer）作为Draft网络，隐状态压缩历史信息，参数量远小于小型LM
2. **依赖解耦**：Draft模型的隐状态由主模型的KV Cache驱动初始化，不需要独立的前缀计算
3. **树形验证**：Draft阶段生成Token树（多个候选分支），主模型并行验证所有分支，选择最长匹配前缀
4. **联合训练**：Draft模型与主模型联合训练（或用主模型蒸馏Draft），缩小分布差距，提升验证通过率
5. **自适应推测长度**：根据当前验证通过率动态调整每次Draft生成的token数量

## 实验结论
- 在LLaMA-2-7B上，推理速度提升2.5x-3x（优于独立Draft模型的2x）
- Draft模型参数量<主模型的1%，几乎不增加内存占用
- 验证通过率（Acceptance Rate）约70%-80%，优于标准Draft LM

## 工程落地要点
- 适合推理密集、延迟敏感的服务（对话/代码生成）
- Draft模型可与主模型共享部分层（前几层），进一步减少Draft计算量
- 树形Draft需要管理多个并行假设的KV Cache，需专门内存管理
- vLLM、TGI等推理框架已集成投机解码，建议直接使用

## 面试考点
**Q: 投机解码（Speculative Decoding）的基本原理？**
A: 小模型快速生成γ个候选token，大模型并行验证（一次前向传播验证γ个token），若候选与大模型分布一致则接受，否则拒绝并从大模型分布重采样。期望加速比≈1/(1-接受率×γ/(γ+1))。

**Q: 投机解码的加速比受什么因素影响？**
A: （1）验证通过率（α）：越高越快；（2）Draft长度（γ）：越长潜在收益越大但风险越高；（3）大/小模型速度比：差距越大收益越大；（4）任务特性：重复性强的任务通过率更高。

**Q: KV Cache在推理加速中的作用？**
A: KV Cache存储已生成token的Key-Value矩阵，避免重复计算。自回归生成中，每次只需计算新token的注意力，KV Cache减少90%+的重复计算，是现代LLM推理的标配。
