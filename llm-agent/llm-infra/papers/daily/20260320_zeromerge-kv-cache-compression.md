# ZeroMerge: Dynamic Zero-Shot KV Cache Compression

> 来源：https://arxiv.org/abs/2503.10714 | 日期：20260320 | 领域：llm-infra

## 问题定义

LLM长上下文推理面临KV Cache线性增长和Attention计算二次复杂度的双重瓶颈：
- 以Llama2-7B为例，32K token长度时KV Cache占用32GB内存，超过模型本身的26GB
- 现有解决方案存在局限：token剪枝造成不可逆信息损失，特征合并需要昂贵的参数重训练
- 急需一种**零样本、无需重训练、可严格控内存**的KV Cache管理策略

## 核心方法与创新点

### 1. 三分区动态预算分配 (B = Bc + Br + Bp)

| 分区 | 功能 | 策略 |
|------|------|------|
| **Bc (Context)** | 保留关键上下文 | 按token贡献分数保留Top-Bc |
| **Bp (Proximity)** | 捕获局部注意力模式 | 保留最近Bp个token |
| **Br (Residual)** | 吸收被驱逐token信息 | 相似性驱动的动态合并 |

### 2. 多维度Token重要性评估

贡献分数采用**几何衰减积分**：
```
s_t^(T) = λ * s_t^(T-1) + a_t^(T)   (λ ∈ [0,1] 为动量因子)
```
- 融合历史贡献、时空特征、数据分布三个维度
- 支持在线O(1)更新，无额外计算开销

### 3. 残差Token合并机制

当token被驱逐时，不是直接丢弃，而是合并到Residual分区：
1. **选择最兼容槽位**：r̂ = argmax(k_r^T · k_t)
2. **动量融合更新**：k_r̂ ← (w_r̂·k_r̂ + k_t)/(w_r̂+1)
3. **融合计数器**：w_r̂ += 1

### 4. 补偿注意力评分 (核心创新)

修正合并token的注意力偏差：
```
â_t^(T) = exp(q_T^T·k_t/√d + α·log w_t) / Σ[exp(...)]
```
- **理论保证** (Theorem 3.1)：未压缩token的注意力分数不会被合并token淹没
- 通过`log w_t`项补偿表示偏移，保持注意力质量守恒

### 5. 零样本适配优势
- 无需训练/微调，即插即用
- 兼容Llama、Falcon、Mistral等主流架构
- 支持任意压缩比动态调整

## 实验结论

### 主要结果 (LLaMA2-7B on LongBench)

| 指标 | 结果 |
|------|------|
| **压缩比** | 20:1 (保留5% KV Cache) |
| **性能保持** | 与完整模型相当 |
| **吞吐量提升** | 54K token上下文下提升3倍 |
| **OOM消除** | 极端长上下文不再内存溢出 |

### 与基线对比

| 方法 | 是否需要训练 | 压缩比 | 性能损失 |
|------|-------------|--------|----------|
| Full KV | - | 1:1 | 0% |
| H2O | 否 | ~10:1 | 中等 |
| LESS | **是** | ~15:1 | 低 |
| **ZeroMerge** | **否** | **20:1** | **极低** |

### 关键发现
- 40K token长度下吞吐量翻倍
- 5% KV Cache预算下性能与完整模型持平
- 跨任务域泛化能力强于现有学习方法

## 面试考点

**Q1: ZeroMerge如何解决传统Token剪枝的"不可逆信息损失"问题？**

A: 传统方法(如H2O、StreamingLLM)直接丢弃低分token，导致注意力分布漂移。ZeroMerge引入Residual分区(Br)，将被驱逐的token通过相似性驱动合并保留其信息，配合补偿注意力评分机制，使合并token的表示贡献可追溯到原始token，实现"软删除"而非"硬剪枝"。

**Q2: 补偿注意力评分中 α·log(w_t) 的数学原理是什么？**

A: 合并token的key向量是多个原始token的加权平均(式7)，这导致其表示与原始分布存在偏差。Theorem 3.1证明：通过添加与融合次数对数成正比的偏置项，可确保未压缩token的注意力分数不小于原始分数，防止合并token淹没关键上下文。

**Q3: ZeroMerge的"零样本"特性对工程部署有何实际价值？**

A: (1) 无需离线训练或微调，降低部署门槛；(2) 同一模型可服务不同压缩比需求的任务；(3) 模型更新时无需重新训练压缩策略；(4) 可动态调整预算适应负载变化，无需重启服务。

**Q4: 三分区预算(Bc/Br/Bp)的设计依据是什么？**

A: 基于对注意力模式的观察：(1) 初始token具有Attention Sink特性需保留(Bc)；(2) 近期token对局部语义建模关键(Bp)；(3) 中间token信息可通过相似性聚类压缩(Br)。这种设计覆盖了长序列推理的完整语义需求。

**Q5: 与LESS等需要训练的合并方法相比，ZeroMerge的trade-off是什么？**

A: ZeroMerge牺牲了少量压缩效率换取通用性——在特定任务/模型上，LESS可能有更好的性能-压缩比曲线；但ZeroMerge无需训练即可跨模型/任务/压缩比通用，且无需维护辅助网络参数，在多样性部署场景下总拥有成本更低。
