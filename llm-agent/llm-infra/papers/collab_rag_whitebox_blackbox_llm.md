# Collab-RAG: Boosting RAG for Complex QA via White-Box and Black-Box LLM Collaboration

> 来源：arxiv | 领域：llm-infra | 学习日期：20260328

## 问题定义

RAG 系统在多跳问答（Multi-hop QA）上表现不佳，主要挑战：
1. **检索不精准**：复杂问题直接检索往往召回不相关上下文
2. **推理能力不足**：黑盒大模型（GPT-4 等）能力强但无法微调；白盒小模型可训练但能力弱
3. **训练信号稀缺**：如何在不访问黑盒 LLM 内部的情况下提升小模型？

## 核心方法与创新点

### 1. 协作架构
- **白盒 SLM（Small LM，可训练）**：负责复杂查询分解（decompose complex query into sub-questions）
- **黑盒 LLM（不可训练，仅 API 访问）**：负责基于检索到的上下文进行推理和回答

### 2. 相互增强机制
$$\text{SLM}(q) \rightarrow \{q_1, q_2, ..., q_k\} \xrightarrow{\text{Retrieval}} \{d_1, ..., d_k\} \xrightarrow{\text{LLM}} a$$

- SLM 将复杂问题分解为简单子问题 → 提升检索准确率
- 黑盒 LLM 对 SLM 的分解质量提供反馈信号 → 训练 SLM 改进分解能力

### 3. 训练流程（无需 frontier LLM 蒸馏）
1. **初始化**：用基础 SFT 训练 SLM 的查询分解能力
2. **反馈循环**：黑盒 LLM 的最终答案质量作为 SLM 训练的远程监督信号
3. **迭代优化**：SLM 分解更准 → 检索更好 → LLM 回答更准 → 反馈信号更强

### 4. 核心优势
- 仅使用 affordable 的黑盒 LLM（无需 GPT-4 级别蒸馏）
- 3B SLM 经训练后在查询分解上超越冻结的 32B LLM

## 实验结论

在五个多跳 QA 数据集上的评估：

| 方法 | 平均提升 vs. 黑盒-only |
|------|----------------------|
| Collab-RAG vs. GPT-4o-only | +14.2% |
| Collab-RAG vs. 最佳 SFT baseline | +1.8% ~ +14.2% |
| 3B SLM (trained) vs. 32B LLM (frozen) | SLM 分解能力更强 |

- 跨黑盒 LLM 的强泛化性：在 GPT-4, Claude, Llama 上均有效

## 工程落地要点

1. **查询分解是 Multi-hop RAG 的关键**：原始复杂问题直接检索效果差，分解后的子问题检索精度大幅提升
2. **白盒+黑盒协作模式**：可训练的小模型做分解/路由，大模型做最终推理，成本效益最优
3. **远程监督**：LLM 的最终答案质量作为 SLM 的训练信号，无需额外标注
4. **部署模式**：SLM 本地部署（低延迟），LLM 云端 API 调用（高能力），兼顾速度和准确率
5. **适用场景**：知识密集型多跳问答、复杂客服系统、企业知识库问答

## 面试考点

**Q1: Collab-RAG 如何解决 Multi-hop RAG 的检索不准问题？**

A: 训练一个白盒 SLM 将复杂多跳问题分解为简单的单跳子问题，每个子问题独立检索，获取精准的支撑文档，最后由黑盒 LLM 整合推理。

**Q2: 黑盒 LLM 的反馈信号如何训练白盒 SLM？**

A: 将黑盒 LLM 在多跳 QA 上的最终答案正确与否作为远程监督信号，训练 SLM 学习产生更好的问题分解策略，无需访问 LLM 内部参数。

**Q3: 为什么 3B SLM 在查询分解上能超越冻结的 32B LLM？**

A: 因为 3B SLM 专门针对查询分解任务进行了微调，而 32B LLM 是通用模型未专门优化此任务。专精 > 通才，专项任务微调的小模型优于未调优的大模型。

**Q4: 这种 White-Box + Black-Box 协作模式有哪些工程上的实际优势？**

A: (1) 成本优化：小模型本地运行处理前处理任务；(2) 可控性：白盒 SLM 可监控调试；(3) 可扩展：替换不同黑盒 LLM 只需重新训练 SLM；(4) 延迟：子问题并行检索减少总体延迟。
