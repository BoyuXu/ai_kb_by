## 问题定义

### 核心挑战
在开放域问答（Open-domain QA）中，**复杂多跳问题**（Multi-hop Questions）需要跨越多个文档、推理多个事实链才能得出答案。传统RAG系统面临以下瓶颈：

1. **单次检索不足**：用原始问题检索往往遗漏关键中间事实
2. **检索时机固定**：无法根据问题复杂度动态决定是否需要检索
3. **推理与检索割裂**：LLM的推理过程未能充分利用已检索内容，也未能指导下一步检索
4. **幻觉传播**：中间推理步骤的错误会累积到最终答案

### 典型多跳问题示例
```
Q: "《哈利波特》的作者毕业于哪所大学，该大学的创始人是谁？"

需要：
Step 1: 检索"《哈利波特》的作者" → J.K.罗琳
Step 2: 检索"J.K.罗琳毕业院校" → 埃克塞特大学
Step 3: 检索"埃克塞特大学创始人" → 威廉·佩蒂特
```

## 核心方法与创新点

### 自适应检索决策机制

**关键创新**：根据模型的不确定性自适应决定是否需要检索

**FLARE风格的主动检索**：
```python
def adaptive_retrieve(query, model, threshold=0.3):
    """基于token置信度决定是否检索"""
    tokens, probs = model.generate_with_probs(query)
    
    # 识别低置信度片段
    uncertain_spans = [
        token for token, prob in zip(tokens, probs)
        if prob < threshold
    ]
    
    if uncertain_spans:
        # 将低置信度片段作为检索查询
        retrieval_query = extract_query(uncertain_spans)
        docs = retriever.search(retrieval_query)
        return model.regenerate(query, docs)
    else:
        return tokens  # 直接输出
```

**复杂度感知路由**：
```
问题 → 复杂度估计器 → 
    简单：直接LLM回答（无需检索）
    中等：单次检索 + RAG
    复杂：多跳检索 + 推理链
```

### 多跳推理架构（IRCoT-style）

**交错检索与思维链**：
```
问题
  ↓
推理步骤1（CoT）→ 识别需要的知识 → 检索查询1 → 文档集1
  ↓
推理步骤2（基于文档1）→ 识别需要的知识 → 检索查询2 → 文档集2
  ↓
...
  ↓
最终推理 → 答案
```

**关键公式**：

多跳推理的期望证据集合：
$$\mathcal{E}^* = \bigcup_{k=1}^{K} R(q_k, \mathcal{D})$$

其中 $q_k$ 是第 $k$ 步的检索查询，由前序推理步骤自动生成：
$$q_k = f_{LLM}(Q, c_1, ..., c_{k-1}, e_1, ..., e_{k-1})$$

**自适应停止条件**：
$$\text{stop} = \mathbb{1}[\text{confidence}(a | Q, \mathcal{E}) > \tau]$$

### 查询分解策略

**问题分解（Decomposition）**：
```
Q: 复杂问题
   ↓ 分解器（SLM/LLM）
q₁: 子问题1（可直接检索）
q₂: 子问题2（依赖q₁答案）
q₃: 子问题3（依赖q₁,q₂答案）
   ↓ 逐步解答
A: 最终答案
```

**查询重写（Query Rewriting）**：
$$q'_k = \text{Rewrite}(Q, \text{history}, q_k)$$

将抽象子问题转化为具体的、可检索的查询。

### 混合检索策略

**稀疏+稠密检索融合**：
$$\text{score}(d, q) = \alpha \cdot \text{BM25}(d, q) + (1-\alpha) \cdot \cos(\mathbf{e}_d, \mathbf{e}_q)$$

**知识图谱辅助多跳**：
- 利用实体关系链接跨文档事实
- 结构化推理路径引导检索方向

## 实验结论

### 代表性系统性能对比（HotpotQA数据集）

| 方法 | EM | F1 | 检索次数 |
|------|----|----|---------|
| 标准RAG（单次） | 42.5% | 56.8% | 1次 |
| IRCoT（多跳CoT） | 55.3% | 68.7% | 2-5次 |
| FLARE（自适应） | 52.1% | 66.2% | 1-3次 |
| Search-R1（RL） | 58.4% | 71.3% | 动态 |
| **自适应多跳RAG** | **62.8%** | **75.4%** | **自适应** |

### 关键发现

1. **自适应优于固定多跳**：对简单问题不检索，节省40%的检索次数，同等准确率

2. **检索时机比检索量更重要**：
   - 检索时机正确 → EM提升12.3%
   - 检索量翻倍 → EM仅提升3.1%

3. **多跳推理的Scaling**：
   $$\text{EM}(K) = \text{EM}(1) + \beta \log K$$
   其中 $K$ 为推理步数，$\beta \approx 0.15$，收益递减

4. **MusiQue（3-4跳）上的改进**：+23.4% EM（vs单跳RAG）

## 工程落地要点

### 系统架构设计

```
用户问题
  ↓
复杂度分类器（轻量，<100ms）
  ├─ 简单 → 直接LLM（~200ms）
  ├─ 中等 → 单次RAG（~500ms）
  └─ 复杂 → 多跳流水线（~2000ms）
              ├─ 查询分解
              ├─ 迭代检索（最多K次）
              └─ 推理聚合
```

### 性能与成本权衡

```python
# 自适应检索配置
config = {
    "max_hops": 3,            # 最大检索跳数（超过后收益递减）
    "confidence_threshold": 0.6,  # 停止检索的置信度阈值
    "retrieval_k": 5,         # 每次检索文档数
    "complexity_threshold": {
        "simple": 0.8,        # 简单问题概率 > 0.8时跳过检索
        "medium": 0.5,        # 中等复杂度
    }
}
```

### 检索质量优化

1. **查询增强**：LLM生成假设性文档（HyDE）提升稠密检索准确率
2. **文档重排序**：用小型Cross-Encoder对Top-K文档重排
3. **上下文压缩**：提取相关片段，减少噪声对LLM的干扰

### 常见陷阱与应对

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| 推理链偏离 | 早期错误传播 | 中间答案置信度检验 |
| 无限循环 | 停止条件失效 | 硬性跳数上限 + 去重 |
| 检索爆炸 | 查询过于宽泛 | 查询长度限制 + 精确性约束 |
| 延迟过高 | 串行多跳 | 并行候选检索 + 剪枝 |

## 面试考点

### 考点1：为什么单次RAG对多跳问题效果差？如何改进？

**答案**：单次RAG用原始问题检索，但多跳问题的关键信息分散在多个文档中，且第二跳的检索查询依赖第一跳的结果。改进方法：
1. **查询分解**：将复杂问题拆分为可直接检索的子问题
2. **迭代检索**：前序推理结果指导后续检索查询
3. **自适应停止**：检测到足够证据时停止检索，避免噪声引入

### 考点2：FLARE中如何通过token概率决定是否检索？

**答案**：FLARE在LLM生成时监测每个token的概率。若连续N个token的概率均低于阈值θ，说明模型对即将生成的内容不确定，此时：
1. 将这些低置信度token作为检索查询
2. 检索相关文档
3. 基于文档重新生成这部分内容
核心公式：`retrieve = any(prob < θ for prob in next_k_probs)`

### 考点3：多跳RAG的评测指标有哪些？

**答案**：
- **EM（Exact Match）**：最终答案精确匹配
- **F1**：答案词语层面的重叠
- **支持事实EM/F1**：中间推理步骤的正确性（HotpotQA特有）
- **检索召回率**：正确文档是否被检索到
- **推理忠实度**：最终答案是否真正基于检索内容（通过NLI模型评估）

### 考点4：自适应检索vs固定多跳检索的权衡？

**答案**：
- **固定多跳**：结构简单，但对简单问题浪费资源；固定步数可能不足以回答超复杂问题
- **自适应检索**：效率高，但需要额外的复杂度估计模块；置信度阈值敏感，可能触发不必要的检索
- **实际建议**：先用轻量分类器路由，90%简单问题走单次RAG，10%复杂问题走多跳

### 考点5：在生产环境中如何控制多跳RAG的延迟？

**答案**：
1. **并行检索**：对独立子问题并行检索（延迟从串行2s降到1s）
2. **早停机制**：置信度达标立即停止
3. **缓存热门子问题**：高频子问题结果缓存（命中率30-40%）
4. **轻量级路由**：用小模型（<1B）做复杂度分类，避免大模型调用开销
5. **流式输出**：多跳推理同时向用户流式输出中间步骤，改善感知延迟
