# ReasonIR: Training Retrievers for Reasoning Tasks

> 来源：arXiv 2025 | 领域：llm-infra | 学习日期：20260404

## 问题定义

传统 Dense Retrieval 在**推理密集型**任务中表现差：
- 数学推理：需要检索推理步骤相关文档，而非事实
- 多跳推理：需要检索多个文档并推理其关联
- 代码辅助推理：需要检索类似问题的解法推理路径

问题本质：检索器只学会了语义相似，未学会推理相关性。

$$\text{sim}(q, d) \approx \text{semantic\_sim}(q, d) \neq \text{reasoning\_relevance}(q, d)$$

## 核心方法与创新点

**ReasonIR** 训练推理感知检索器：

1. **推理路径合成数据**：
   - 从数学/代码/逻辑推理数据集出发
   - 用 LLM 生成「推理相关文档」（展示推理步骤的参考资料）
   - 而非事实性文档（维基百科段落）

2. **推理感知对比学习**：
   - Positive：与推理步骤相关的文档（能帮助推导答案）
   - Negative：语义相似但推理无关的文档（干扰项）
   
$$\mathcal{L}_{\text{reason}} = -\log \frac{e^{s(q, d^+)/\tau}}{e^{s(q, d^+)/\tau} + \sum e^{s(q, d^-_{\text{reason-hard}})/\tau}}$$

3. **推理步骤对齐**：
   - 检索器的查询 = 推理链中的某一步（而非原始问题）
   - 多跳：每步推理生成新查询，检索支撑该步的证据

4. **自蒸馏增强**：
   - 用已训练的 ReasonIR 检索到的文档再次增强训练数据
   - 迭代提升推理检索质量

## 实验结论

- BRIGHT（推理检索 benchmark）: **+8.3%** NDCG@10 vs 最强 Embedding 基线
- 数学推理（MATH-RAG）: 正确率 **+11.7%**
- 多跳推理（MuSiQue）: F1 **+9.2%**
- 通用检索（BEIR）: 无性能损失（+0.3%）

## 工程落地要点

- 推理步骤查询生成：可在推理时动态生成，也可预计算
- 多跳检索：建议最多 3 跳（避免误差累积）
- 与 Search-o1 配合：ReasonIR 提供高质量推理感知检索，Search-o1 进行推理-检索交织
- 训练数据：至少需要 10k 推理相关 (query, doc) 对

## 面试考点

1. **Q**: 传统检索和推理感知检索的区别？  
   **A**: 传统检索：语义相似（BM25/Dense）；推理感知检索：检索能帮助推导答案的文档，即使语义不相似（如检索数学定理支撑某推理步骤）。

2. **Q**: 多跳推理如何与检索结合？  
   **A**: 迭代检索：问题 → 检索第一跳文档 → 更新推理状态 → 基于新状态生成新查询 → 检索第二跳 → ... → 得出答案。

3. **Q**: 推理密集型检索的主要挑战？  
   **A**: 训练数据稀缺（推理标注昂贵）；查询与文档的推理相关性难以自动标注；多跳错误累积（第一跳错误导致后续全错）。
