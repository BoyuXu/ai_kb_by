# Chain of Agents: Large Language Models Collaborating on Long-Context Tasks

> 来源：arXiv 2402.01620 | 领域：llm-infra | 学习日期：20260402

## 问题定义

LLM 的上下文窗口虽然不断扩大，但在超长文本任务中仍面临注意力稀释、推理质量下降和计算成本高昂的问题。Chain of Agents 提出无需训练的多 Agent 链式协作框架，通过接力处理解决超长文本任务。

## 核心方法与创新点

1. **链式协作**：将长文本分段分配给多个 Agent，每个 Agent 处理一段并将关键信息传递给下一个
2. **无需训练**：纯 prompt engineering 方案，不需要对 LLM 进行微调
3. **信息压缩**：每个 Agent 在传递时压缩当前段的关键信息，避免信息爆炸
4. **质量控制**：设计验证 Agent 检查链式传递中的信息一致性

## 实验结论

- 在 10 万 token 文本的问答任务上，准确率比单 Agent 提升 25%
- 信息传递的压缩比约 10:1，关键信息保留率 > 90%
- 总成本与单次长上下文调用相当，但质量显著更好

## 工程落地要点

- **分段策略**：按语义边界（段落/章节）分段优于固定长度分段
- **传递信息格式**：结构化的 JSON 比自由文本传递更可靠
- **错误传播**：链式传递可能积累错误，需要定期校验

## 面试考点

1. **Q：和 RAG 处理长文本有什么区别？**
   A：RAG 是检索相关片段，可能丢失全局上下文；Chain of Agents 按序处理全文，保持信息的完整性和连贯性。
2. **Q：链式传递的信息损失如何控制？**
   A：通过结构化信息格式、冗余传递和验证 Agent 三重机制控制信息损失。
3. **Q：多 Agent 协作的开销？**
   A：总 token 消耗约为单次长上下文的 1.2-1.5 倍（包括传递信息的开销），但可以使用更便宜的短上下文模型。
