# Agent 失败模式：踩坑记录与解法

> 来源：工业界 Agent 系统的常见问题（非学术survey，偏工程实践）

## 失败模式 1：工具调用的级联失败

**现象**：Agent 在第5步调用了一个 API，返回了空结果，但 Agent 没有处理这个异常，继续用空结果做后续推理，导致后续8步全部基于错误假设。

**根因**：LLM 的错误处理能力弱——看到工具返回空，倾向于"脑补"一个结果而不是报错重试。

**解法**：
1. **工具返回值校验**：每次工具调用后检查返回格式，异常时显式告知 LLM
2. **强制重试提示**：`系统：工具返回为空，请重新思考查询条件后重试（最多3次）`
3. **Fail-fast**：设置关键步骤的 early termination，宁可失败也不基于错误数据继续

---

## 失败模式 2：目标漂移（Goal Drift）

**现象**：多步任务中，Agent 在执行子任务时"忘记"了最终目标，把子任务当成了终极目标。

**例子**：任务是"帮我找一个便宜的去东京的机票"，Agent 先搜索机票，然后发现有个很便宜的酒店，就开始推荐酒店，忘记了原始目标是机票。

**根因**：每步 prompt 中原始目标的比重被中间步骤信息稀释。

**解法**：
1. **目标锚定**：每步 prompt 开头重复原始目标（简短版）
2. **约束列表**：把不能做的事也明确列出 `["不要推荐酒店", "只关注机票价格"]`
3. **Reflection Step**：每5步做一次"回顾"，检查当前状态是否与目标一致

## 失败模式 3：幻觉工具调用

**现象**：LLM 调用了一个不存在的工具，或者给工具传了根本不存在的参数。

**例子**：`call_tool(name="search_database", query="...", sort_by="relevance")` — 但 search_database 根本没有 sort_by 参数。

**根因**：LLM 在预测工具调用时，把训练数据中见过的 API pattern "幻觉"出来。

**解法**：
1. **Schema 验证**：工具调用前做 JSON Schema 校验，不符合就让 LLM 重试
2. **Tool 描述精简**：工具说明越简洁越好，避免 LLM 被误导（不要列"可以但不推荐"的参数）
3. **Function Calling 强制格式**：用 OpenAI function calling / Claude tool use 的结构化输出，不允许自由文本调用

---

## 失败模式 4：Context 溢出导致记忆丢失

**现象**：长任务执行到一半，历史 context 已满，LLM "忘记"了前面的工作，开始重复已做过的步骤。

**解法**（按优先级）：
1. **阶段性总结**：每 N 步让 LLM 输出当前进度摘要，丢弃详细历史，保留摘要
2. **外部记忆**：把中间结果写入文件/数据库，不放在 context 里
3. **分解任务**：把长任务分解成独立的短任务，每个短任务独立执行

---

## 失败模式 5：过度思考导致超时

**现象**：CoT 模型（o1、R1 类）在简单任务上也花大量 token 做推理，导致延迟极高、成本翻倍。

**例子**：用 o1 回答"今天几号"——o1 会先推理日期格式、时区、闰年规则……

**解法**：
1. **任务分类路由**：简单任务（查表/格式化）用普通 LLM，复杂推理才用 o1/R1
2. **最大思考 token 限制**：设置 `max_thinking_tokens` 或 `budget_tokens`（Claude 3.5+ 支持）
3. **早停条件**：如果中间步骤已经产生了明确答案，不要继续思考

---

## 失败概率与任务步数的关系

$$
P(\text{成功}) = p^n
$$

其中 $p$ 是单步成功率，$n$ 是步数。

| 单步成功率 | 5步任务 | 10步任务 | 20步任务 |
|-----------|---------|---------|---------|
| 95% | 77% | 60% | 36% |
| 99% | 95% | 90% | 82% |
| 99.9% | 99.5% | 99% | 98% |

**结论**：要让20步任务成功率>95%，单步成功率需要>99.7%。这是为什么 Agent 系统需要极其严格的错误处理。
