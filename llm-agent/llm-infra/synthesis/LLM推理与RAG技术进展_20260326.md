## 📐 核心公式

### 1. GRPO（Group Relative Policy Optimization）

$$
\mathcal{L}_\text{GRPO} = -\sum_{i=1}^{G} \hat{A}_i \log \pi_\theta(o_i|q)
$$

$$
\hat{A}_i = \frac{r_i - \mu_r}{\sigma_r}, \quad \mu_r = \frac{1}{G}\sum_{j=1}^G r_j
$$

其中 $G$ 个样本组内相对奖励，不需要 Critic 网络。

### 2. DPO（Direct Preference Optimization）

$$
\mathcal{L}_\text{DPO} = -\log \sigma\left(\beta \log \frac{\pi_\theta(y_w|x)}{\pi_\text{ref}(y_w|x)} - \beta \log \frac{\pi_\theta(y_l|x)}{\pi_\text{ref}(y_l|x)}\right)
$$

### 3. LoRA 权重分解

$$
h = Wx + \Delta Wx = Wx + \frac{\alpha}{r} BAx
$$

$$
B \in \mathbb{R}^{d \times r}, A \in \mathbb{R}^{r \times k}, \quad r \ll \min(d,k)
$$

### 4. 投机解码接受率

$$
\alpha_i = \min\left(1, \frac{P_\text{target}(x_i|x_{<i})}{P_\text{draft}(x_i|x_{<i})}\right)
$$

期望加速比：$\text{speedup} \approx \frac{1 - \alpha^{\gamma+1}}{(1-\alpha)(1 + 1/\gamma)}$

### 5. 自适应 RAG 停止条件

$$
\text{retrieve\_again} = \mathbb{1}\left[\text{Conf}(a|q, \mathcal{C}) < \theta\right]
$$

## 🎓 Q&A 面试精华

**Q1: DeepSeek-R1 的最大技术突破是什么？**
A: 无需推理过程标注的纯 RL 激励：只用答案正确性奖励，模型自发习得长推理链、自我反思、回溯等推理策略。这打破了"教推理需要大量 CoT 标注"的惯例，将强化学习正式引入推理能力培养的主流方法论。AIME 2024 从 9% 提升到 80%。

**Q2: Qwen3 混合推理模式（thinking/non-thinking）的设计哲学？**
A: 务实的"按需推理"：简单对话不需要推理，强制 thinking 浪费 token；复杂数学需要 reasoning，没有 thinking 准确率低。混合模式让用户通过 API 参数/System Prompt 按场景控制，比维护两个独立模型更经济。Qwen3-235B-A22B 的 MoE 设计也体现同样理念：大容量但低激活成本。

**Q3: LIMO（817样本超越10万样本）挑战了什么假设？**
A: 挑战了"数据规模假设"：传统认为推理需要大量 SFT 数据（>10万）。LIMO 证明：对于预训练强的模型（Qwen2.5-32B），推理能力已内化，SFT 数据的作用是"格式激活"而非"知识教授"。少量（数百）高质量、难度适中、步骤完整的样本足以激活潜在推理能力。

**Q4: Agentic RAG 相比 Naive RAG 的核心价值和风险？**
A: 核心价值：处理复杂多跳问题（Naive RAG 单次检索无法覆盖所有相关信息）；自适应（简单问题不检索，复杂问题多次）；工具扩展（代码执行/API）。主要风险：延迟（多轮迭代 × LLM 调用）；成本（10-50x）；可靠性（错误级联）；幻觉传播（错误中间结果）。

**Q5: Collab-RAG 中"小模型负责检索辅助，大模型负责最终推理"的分工依据？**
A: 任务难度分析：查询改写/分解 = 理解局部结构，7B 微调后足够胜任，速度快成本低。最终综合推理 = 多文档信息整合+复杂推理，需要大模型的广泛知识和推理能力。这种分工减少 60% API 成本，是"能力-成本优化"的工程实践。

**Q6: LoRA 微调在推荐系统中的最佳实践？**
A: ①r=16 作为起点（电商推荐）②target_modules 包含 q_proj/v_proj（Attention）和 gate_proj（FFN）③数据格式：将推荐任务文本化（"用户历史: item1, item2... 预测下一个: ?"）④QLoRA 节省显存（4-bit Base + FP16 LoRA，单 A100 可训练 7B-13B）⑤多场景：每场景独立 LoRA，共享 Base，按场景路由。

**Q7: 投机解码为什么适合代码生成而不适合随机对话？**
A: 代码高度可预测：函数名/关键字/缩进等遵循严格语法，小模型猜对率高（>80%），加速比 4-6x。随机对话：下一个词可以是任何词，小模型猜对率低（<50%），频繁拒绝导致加速比仅 1.5-2x。核心：加速比 = f(接受率 α, 猜测长度 γ)，任务可预测性决定 α。

**Q8: 多跳 RAG 中 IRCoT（交替推理+检索）的工程实现要点？**
A: 关键设计：①每步生成一个推理句后立即触发检索（不等整段推理完成）②检索 query = 当前推理句（精准定向）③新检索结果注入 Context（实时扩充上下文）④循环直到推理完整或达到最大步数。实现：Streaming Generation + 推理句边界检测 + 并行检索（减少延迟）。

**Q9: GRPO vs PPO，工业落地哪个更常用？**
A: GRPO 更常用于推理任务：无 Critic 网络（显存节省 ~50%），实现简单稳定，DeepSeek-R1 验证有效。PPO 更通用：有 Critic 提供精确 Value 估计，适合奖励信号复杂/连续的场景（如对话质量优化）。趋势：推理任务用 GRPO（轻量），RLHF 对话对齐用 PPO/DPO 混合。

**Q10: 如何选择 LLM 推理加速方案（投机解码/量化/蒸馏）？**
A: 场景驱动选择：①延迟敏感（streaming chat）→ 投机解码（无质量损失）②资源受限（单卡部署）→ 量化（INT8/INT4，2-4x 压缩）③吞吐量优先（batch inference）→ Continuous Batching + Flash Attention ④质量优先（生产高价值请求）→ 蒸馏（teacher→student，保持输出分布接近）。通常组合使用：量化 Base + 投机解码 + Continuous Batching。
