# 测试时计算扩展与推理时对齐（Test-Time Compute Scaling & Inference-Time Alignment）

> 综合日期：2026-04-11 | 涵盖论文：Scaling LLM Test-Time Compute (Snell et al., 2024), PITA (Bobbili et al., 2025)
> 领域：LLM 推理优化 | 类型：综合分析

---

## 核心问题

传统 LLM 改进路径是"训练时扩展"——更大模型、更多数据、更长训练。但训练成本已趋近天花板。**能否在推理时（test-time）投入更多计算来提升性能？** 这条路线的核心挑战是：如何最优分配推理时计算资源？

---

## 🆚 训练时扩展 vs 推理时扩展

| 维度 | 训练时扩展 | 推理时扩展（Test-Time Compute） |
|------|-----------|-------------------------------|
| 资源投入时机 | 训练阶段（一次性） | 每次推理（按需） |
| 成本模式 | 固定大成本 | 按难度弹性分配 |
| 扩展瓶颈 | 数据量 + 算力 + 训练稳定性 | 验证器质量 + 搜索效率 |
| 代表方法 | Scaling Laws, 更大模型 | Best-of-N, Beam Search, Self-Refinement |
| 适用场景 | 通用能力提升 | 特定难题的精准突破 |

## Paper 1: Scaling LLM Test-Time Compute Optimally

> arxiv 2408.03314 | Snell, Lee, Xu, Kumar (Google DeepMind) | 2024

### 问题定义

给定固定的推理计算预算，如何最优分配 test-time compute 以最大化 LLM 在困难问题上的表现？

### 核心方法

研究了两种 test-time compute 扩展机制：

**机制 1：搜索 + 过程奖励模型（PRM）验证**
- 采样 N 个候选答案，用 Process Reward Model（PRM）对每一步打分
- PRM 产生每一步的 reward-to-go 估计，支持 tree search
- 搜索策略：Best-of-N weighted（独立采样选最优）、Beam Search（BFS-V，逐步搜索）

**机制 2：自适应分布修正（Adaptive Revision）**
- 在推理时根据 prompt 动态调整模型输出分布
- 模型对自己的初始回答进行迭代修正

**Compute-Optimal 策略（核心创新）**：
- 不同难度的问题，最优策略不同
- 简单问题：少量采样 + Best-of-N 就够
- 困难问题：Beam Search + PRM 逐步验证更有效
- 动态根据问题难度分配 test-time compute

### 关键公式

$$
\text{Compute-Optimal: } \pi^*(q) = \arg\max_{\pi \in \Pi} \; \mathbb{E}_{y \sim \pi(q)} [R(y, q)] \quad \text{s.t. } C(\pi) \leq B
$$

其中 $q$ 是问题，$\pi$ 是推理策略（搜索方法 + 计算量分配），$R$ 是正确性奖励，$B$ 是计算预算。

### 实验亮点

- Compute-optimal 策略比 Best-of-N baseline 效率提升 **4x 以上**
- 在 FLOPs-matched 评估中，小模型 + 充足 test-time compute **超越 14x 更大的模型**
- 结论：对于中等难度问题，test-time compute scaling 比 model scaling 更高效

### 面试考点

**Q: Test-time compute scaling 的两种核心机制？**
30秒答案：(1) 搜索+验证：采样多个候选，用 PRM 逐步打分选最优；(2) 自适应修正：模型迭代修正自己的回答。关键是 compute-optimal 策略——根据问题难度动态切换方法和计算量。

**Q: PRM 比 ORM 好在哪？**
30秒答案：ORM 只看最终答案对不对，PRM 评估每一步的正确性，支持 tree search 在错误步骤剪枝。PRM 的 per-step 分数是 reward-to-go 的估计。

**Q: 为什么小模型 + test-time compute 能超越大模型？**
30秒答案：大模型的优势是"一次推理就大概率对"，但如果允许小模型多次尝试并用验证器选最优，在总 FLOPs 相同的情况下，小模型的多次尝试覆盖了更大的解空间。前提是问题不能太难（小模型至少有 non-trivial 成功率）。

## Paper 2: PITA — 推理时偏好引导对齐

> arxiv 2507.20067 | Bobbili, Dinesha, Narasimha, Shakkottai (Texas A&M) | 2025

### 问题定义

现有 LLM 后训练对齐方法（RLHF/DPO/GRPO）都需要在训练阶段完成对齐。但：(1) 不同用户有不同偏好，训练时无法穷尽；(2) 偏好会随时间变化；(3) RLHF 依赖的 Reward Model 训练不稳定。能否在推理时直接根据偏好调整输出？

### 核心方法

PITA 训练一个小型 **Preference Guidance Policy**，在推理时修改 frozen LLM 的 token 概率分布，无需修改 LLM 权重。

$$
p_{\text{aligned}}(x_t | x_{<t}) = p_{\text{LLM}}(x_t | x_{<t}) \cdot g_\phi(x_t | x_{<t}, \text{pref})
$$

其中 $g_\phi$ 是小型 guidance policy，$\text{pref}$ 是用户偏好信号。

### 关键创新

1. **无需 Reward Model**：直接从偏好信号学习 guidance policy，绕过 RM 训练不稳定的问题
2. **LLM 权重不动**：guidance policy 是外挂的小模型，不影响 LLM 本身能力
3. **实时可切换**：不同用户/任务可以用不同的 guidance policy，无需重新训练 LLM
4. **参数高效**：guidance policy 参数量远小于 LLM 本身

### 实验亮点

- 在数学推理和情感分类任务上有效对齐 LLM 输出
- 计算开销远小于完整 RLHF/DPO 训练
- 可与不同 base LLM 组合使用

### 面试考点

**Q: PITA 和 RLHF/DPO 的核心区别？**
30秒答案：RLHF/DPO 在训练时修改模型权重做对齐，PITA 在推理时用外挂小模型调整 token 概率。好处：LLM 不动（保留通用能力），可实时切换偏好，不需要 RM。

**Q: Inference-time alignment 的应用场景？**
30秒答案：(1) 多用户个性化：同一 LLM 对不同用户展示不同风格；(2) 快速适应新偏好：不用重新训练就能调整；(3) A/B 测试：同一模型用不同 guidance policy 对比效果。


## 核心洞察

1. **Test-time compute 是"穷人的 scaling"**：不需要训练更大模型，只需在推理时多花计算就能获得显著提升，但需要好的验证器
2. **Compute-optimal 的关键是"按难度分配"**：简单问题不值得多搜索，困难问题值得投入大量 test-time compute
3. **PRM 是 test-time scaling 的瓶颈**：搜索效果上限取决于 PRM 的质量，训练好的 PRM 本身就很难
4. **推理时对齐是个性化的关键**：训练时无法预见所有用户偏好，推理时动态调整是更灵活的方案
5. **训练时 vs 推理时不是互斥的**：最优方案是训练时学通用能力 + 推理时做个性化/难题攻克
