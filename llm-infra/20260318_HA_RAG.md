# HA-RAG：热度感知的 RAG 加速混合精度与数据放置

> 来源：HA-RAG: Hotness-Aware RAG Acceleration via Mixed Precision and Data Placement | 日期：20260318 | 领域：llm-infra

## 问题定义

RAG 系统的知识库文档存在明显的**访问热度不均匀**现象：少量文档（热文档）被频繁检索，大量文档（冷文档）极少被访问（幂律分布）。传统 RAG 对所有文档使用统一的 embedding 精度（FP16 或 FP32）和统一的存储层（GPU 显存/CPU 内存/SSD）。这导致：
- 冷文档占用昂贵的 GPU 显存，浪费资源。
- 热文档放在 CPU/SSD 时访问延迟高。
- 统一精度忽略了冷文档可以用低精度存储。

## 核心方法与创新点

1. **文档热度感知**：实时统计每个文档的访问频率（滑动窗口计数），按热度分为三层：
   - 热（Hot）：最近 1 小时访问 > N 次。
   - 温（Warm）：最近 24 小时有访问。
   - 冷（Cold）：长期未访问。

2. **混合精度存储**：
   - 热文档 embedding：FP16，存在 GPU 显存（快速检索，精度优先）。
   - 温文档 embedding：INT8 量化，存在 CPU 内存（平衡速度和内存）。
   - 冷文档 embedding：INT4 量化，存在 SSD（最大化容量）。

3. **动态迁移**：当冷/温文档被频繁访问后，自动提升到更高热度层，重新量化为更高精度并迁移到更高层存储。反向迁移按 LRU 策略触发。

4. **量化精度对检索的影响补偿**：INT4 量化会引入误差，用 re-scoring（用 FP16 对 INT4 top-K 结果重新打分）补偿精度损失。

## 实验结论

- 相比全 FP16 存储，HA-RAG 在相同 GPU 显存下可容纳 4-8x 更多文档。
- 平均检索延迟降低约 35%（热文档命中率高，GPU 直接返回）。
- 答案质量（EM/F1）相比 FP16 基线损失 < 1%（re-scoring 有效补偿 INT4 误差）。
- 量化误差分析：INT8 对检索结果影响极小（< 0.1%），INT4 在 re-scoring 后影响也在 0.5% 内。

## 工程落地要点

- **热度统计粒度**：文档 chunk 级别比文档级别更精细（一篇长文档的不同段落热度可能差异大），但管理开销更高。
- **迁移成本**：文档从 SSD 迁移到 GPU 显存需要量化/反量化，迁移频率过高会成为开销；建议设置迁移阈值（访问 M 次才触发）。
- **量化方案选择**：GPTQ、AWQ 等量化方法对 embedding 量化效果不同（embedding 量化与模型权重量化方法不完全相同），需实验确认最优方案。
- **多 GPU 场景**：热文档可以在多 GPU 之间复制，避免单 GPU 成为热点瓶颈。

## 面试考点

**Q: RAG 系统中的主要延迟来源有哪些？**
- 检索延迟（ANN 搜索）；文档获取延迟（存储 I/O）；LLM Prefill 延迟（处理检索到的文档）；LLM Decode 延迟（生成答案）。HA-RAG 优化前两项。

**Q: 什么是向量量化？INT8 量化如何影响检索精度？**
- 将 FP32/FP16 embedding 的每个浮点数压缩为 INT8 整数（对称量化：scale = max_abs / 127）；余弦相似度计算用量化值近似，在 k >> 1 时 top-K 召回率损失通常 < 1%。

**Q: 如何在 GPU 显存不足时扩大 RAG 知识库容量？**
- 层级存储（GPU/CPU/SSD）；向量量化（INT8/INT4）压缩 embedding；IVFPQ 索引（product quantization）；选择性加载（只加载高频文档到 GPU）。
