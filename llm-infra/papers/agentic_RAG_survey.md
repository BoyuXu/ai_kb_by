# Agentic RAG Survey: From Static Retrieval to Autonomous Reasoning Agents
> 来源：arXiv综述 | 领域：LLM基础设施 | 学习日期：20260327

## 问题定义
RAG（检索增强生成）从早期的"检索一次、生成一次"进化到"自主推理、多轮检索"的Agentic范式。
本综述梳理Agentic RAG的5种主要类型：
1. **Self-RAG**：LLM自主决策是否检索、评估检索质量
2. **Corrective-RAG**：发现生成错误时自动修正
3. **Adaptive-RAG**：根据问题复杂度选择检索策略
4. **Iterative-RAG**：多轮迭代检索完成复杂推理
5. **Graph-RAG**：基于知识图谱的结构化检索

## 核心方法与创新点

### 1. Self-RAG：自我反思的检索
引入特殊token进行自我评估：
- `[ISREL]`：检索结果是否相关
- `[ISSUP]`：生成内容是否有检索支持
- `[ISUSE]`：最终答案是否有用

训练时将这些评估token加入训练数据，推理时LLM自动生成评估信号：
$$P(response, reflect) = P(response | context) \cdot P(reflect | response, context)$$

### 2. Corrective-RAG：自动修正
检测生成错误后重新检索：
```
生成答案 → 事实验证器检查 → 发现不一致 → 
  改写Query → 重新检索 → 基于新信息修正答案
```

事实验证器：独立的小型NLI（自然语言推理）模型，判断答案与检索文档是否一致。

### 3. Adaptive-RAG
根据问题复杂度分级处理：
- **Level 0（无检索）**：简单事实问题，LLM直接回答
- **Level 1（单次检索）**：直接回答类问题，检索一次
- **Level 2（迭代检索）**：复杂推理问题，多轮检索

复杂度判断：轻量级分类器（<1B参数），实时判断问题复杂度。

### 4. Iterative-RAG：多跳推理
每轮生成的中间结论作为下一轮检索的查询：
$$q_{t+1} = f_{LLM}(q_{original}, d_t, c_t)$$
其中 $c_t$ 是第t轮的中间结论，$d_t$ 是第t轮的检索文档。

### 5. Graph-RAG：知识图谱增强
将文档解析为知识图谱（实体+关系），利用图结构进行检索：
- **图构建**：LLM从文档中提取(主体, 关系, 客体)三元组
- **图检索**：基于图结构的路径检索，能处理多跳关系
- **社区摘要**：对相关节点的社区生成摘要，减少检索噪声

Graph-RAG适合结构化知识（组织关系、法规体系、科学知识图谱）。

## 5种Agentic RAG对比

| 类型 | 核心机制 | 适用场景 | 额外成本 |
|------|---------|---------|---------|
| Self-RAG | 自我反思token | 通用问答，需要可靠性 | 低（训练成本） |
| Corrective-RAG | 错误检测+修正 | 高准确性要求场景 | 中（NLI模型） |
| Adaptive-RAG | 复杂度路由 | 混合复杂度问题 | 低（分类器轻量） |
| Iterative-RAG | 多轮检索 | 多跳推理，复杂逻辑 | 高（多次LLM调用） |
| Graph-RAG | 知识图谱 | 结构化知识，关系推理 | 高（图构建成本） |

## 实验结论
- **Self-RAG**：在TriviaQA等问答基准上比标准RAG提升+5~8%准确率
- **Graph-RAG**：在企业知识库问答上比向量检索提升+25%（结构化知识场景）
- **Adaptive-RAG**：在延迟上节省40%（简单问题不检索），效果损失<1%
- 联合使用多种策略：效果最好但成本最高

## 工程落地要点
1. **策略选择原则**：先用Adaptive-RAG路由，简单问题用无检索，复杂问题用Iterative
2. **Graph-RAG的图构建**：一次性离线构建，在线检索。图更新频率视文档变化速度而定
3. **Self-RAG的部署**：需要专门fine-tune的模型，不能直接用通用LLM
4. **成本监控**：Agentic RAG的token消耗是标准RAG的2-10x，需要实时成本监控
5. **回退策略**：当迭代次数超过上限时，以当前最优答案返回，不能无限循环

## 面试考点
Q1: Self-RAG和Corrective-RAG有什么区别？
A: Self-RAG：在生成过程中插入反思token，LLM在生成时实时评估是否需要检索、检索结果是否相关、生成是否有支持，整个反思是LLM生成的一部分（同一个模型）。Corrective-RAG：生成答案后，用独立的验证模块（NLI分类器）检查事实性，发现错误后重新检索并修正。区别：Self-RAG是"边生成边反思"（在线），Corrective-RAG是"生成后检查"（后处理）。Self-RAG需要专门训练，Corrective-RAG可以在任意LLM上直接部署。

Q2: Graph-RAG相比向量检索（DPR）在什么场景下有明显优势？
A: Graph-RAG的优势场景：(1)关系推理："A公司的母公司是谁？其CEO有什么经历？"——需要沿图的边进行多跳推理；(2)全局摘要："描述某个研究领域的整体现状"——需要整合多个节点的信息；(3)结构化查询："哪些公司属于半导体行业且市值>100亿？"——图的结构化特征比向量检索更适合。向量检索的优势：语义相似性查询、大规模文档检索（亿级）、无需结构化预处理。

Q3: 如何评估Agentic RAG系统的效果？
A: (1)准确率：最终答案的正确率（Exact Match/F1）；(2)检索质量：检索文档的相关性（NDCG@K）；(3)推理合理性：推理链的逻辑连贯性（人工评估或LLM-as-Judge）；(4)效率：平均检索轮数、token消耗、端到端延迟；(5)鲁棒性：对噪声文档、误导性检索结果的鲁棒性；(6)成本效益：效果提升/额外成本比。
